## 定義

機能要求とは、ソフトウェアが提供すべき観測可能な振る舞いを表す要求である。言い換えると、「ソフトウェアが何をするか」を述べる要求である。

## 何を書く場所か

利用者や外部システムから観測できる振る舞い、業務規則、業務手続きを書く。

## 書くべきこと

- 方針: 常に守られるべき業務規則
- プロセス: 実行される業務手続き
- 入力、出力、状態変化
- 例外時の振る舞い
- 受け入れ基準へのトレース

## 見分け方

理想的なコンピュータがあってもその要求を書く必要があるなら、機能要求である可能性が高い。

## 書き方

構造化自然言語とは、自然言語を使いながら、書き方に制約を加える方法である。機能要求では、次の要素を明示する。

- いつ: きっかけ、時点、イベント
- 誰が / 何が: 主体、アクター、システム
- 何をする: 観測可能な動作
- どの条件で: 条件、例外、制約

例:

```text
注文が出荷されたとき、システムは請求書を作成する。ただし、注文条件が前払いの場合を除く。
```

この文では、きっかけ、主体、動作、例外条件が明示されている。

ユースケース仕様、ユーザーストーリー、決定表も構造化自然言語として使える。ユーザーストーリーは、次の形式で利用者視点の価値を表す。

```text
<役割> として、<能力> がほしい。なぜなら、<便益> を得たいからである。
```

| 安定性 | 変更される可能性を示す | TBD |
| 変更履歴 | いつ、誰が、なぜ変更したかを残す | TBD |

### トレースメモ

- すべての機能要求は、少なくとも1つの受け入れ基準へ対応付ける。
