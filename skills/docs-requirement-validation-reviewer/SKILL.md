# Docs Requirement Validation Reviewer

Use this skill while writing each requirement, not only after a document is complete.

## Definition

要求妥当性確認とは、現在理解されている要求が、利害関係者の真のニーズを表しているかを確認する活動である。英語では Requirements Validation と呼ばれる。

The key question is not only whether the requirement is written correctly, but whether building it would solve the real problem. A grammatically correct requirement is not valid if it does not address the user's actual need.

## Review Timing

Perform a lightweight review each time a requirement is added or materially changed.

Do not wait for a final requirements document review if the issue can be caught while writing.

## Review Viewpoints

- 顧客・利用者: 自分たちのニーズが正しく表されているか。
- 要求の専門家: 文書が明確で、標準に沿っているか。
- 設計者・開発者: 設計・実装に必要な情報があるか。
- テスト担当: テスト可能か、受け入れ基準へ落とせるか。

## Validation Questions

Ask these questions while writing:

- 必要性: この要求は現時点で必要か。不要な要求ではないか。
- 十分性: 必要な要求を十分に含んでいるか。
- 理解容易性: 利害関係者が同じ意味に読めるか。
- 一貫性: 他の要求と矛盾していないか。
- 適合性: 標準、契約、制約に合っているか。
- 実現可能性: 設計・実装の前提として使えるか。
- 検証可能性: 受け入れ基準やテストに変換できるか。
- ニーズ適合: これを作れば本当に問題が解決するか。

## Outcome

If the review finds a problem, either revise the requirement immediately or record the open issue in the requirement's conflict, dependency, or change history attributes.
