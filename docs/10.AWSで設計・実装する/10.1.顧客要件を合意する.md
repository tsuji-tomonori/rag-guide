# 10.1 最小要件を合意する

最初の構築では、顧客と決める事項も最小限にします。
大規模な非機能要件表を作るのではなく、「何を登録し、何を質問し、何が返れば成功か」を一枚にまとめます。

## 10.1.1 この段階で決める八項目

表10-1は、初期構築で決める要件、顧客への質問、今回採用する値、判断の根拠を対応付けています。

**表10-1　最小RAGで決める要件と初期値**

| ID | 決めること | 顧客に確認する質問 | このハンズオンの初期値 | 根拠となる章節 |
|---|---|---|---|---|
| H01 | 利用目的 | どの一種類の質問に答えたいですか | 架空製品のサポート条件への質問 | [1.4 RAGが向くケース](../1.RAGとは何か/1.4.RAGが向くケース.md)、[2.7 最初に作る構成](../2.アーキテクチャ/2.7.最初に作る構成と育て方.md) |

まず一つの固定構成を動かし、入力と出力を確認可能にすることを優先します。

## 10.1.2 顧客と埋める最小シート

実際の顧客データへ置き換える前に、次の項目を一緒に埋めます。

```yaml
purpose: "製品サポート担当者が受付条件を確認する"
representative_question: "スタンダードプランの受付時間と問い合わせ方法は？"
expected_answer: "平日09:00-17:00 JST、Webフォームとメール"
expected_source: "support-policy.md"
source_owner: "製品サポート部門"
source_files_max: 20
allowed_data: "機密情報と個人情報を含まない検証用文書"
aws_region: "ap-northeast-1"
retrieval_results: 5
completion:
  ingestion_failed_documents: 0
  expected_source_rank_max: 1
  citation_required: true
```

顧客固有の初期構築でも、文書は3件から20件程度、一つの業務、一つの代表質問に絞ります。
範囲を広げる前に、この一問で検索と生成を分けて確認できるようにします。

## 10.1.3 AWSリソースへの対応

表10-2の対応だけを使います。

**表10-2　要件とAWSリソースの最小対応**

| 要件 | AWSリソース | 設定 |
|---|---|---|
| H02、H03 | Amazon S3 | 東京リージョン、Block Public Access有効、検証用Markdownのみ |
| H04 | Amazon Bedrock Knowledge Bases | Default parser、Default chunking |
| H04 | Amazon Titan Text Embeddings V2 | 1024次元、float |
| H04、H05 | Amazon S3 Vectors | Quick createしたベクトルバケットとインデックス |
| H05、H06 | Bedrock Agent Runtime `Retrieve` | Semantic search、5件 |
| H07 | Bedrock Agent Runtime `RetrieveAndGenerate` | Claude Haiku 4.5、回答と引用を返す |

`RetrieveAndGenerate`を使うため、本章ではManaged Knowledge Baseではなく、S3 Vectorsをベクトルストアにする標準Knowledge Baseを使います。

## 10.1.4 この段階では決めないこと

次の事項は、最小構成が動いた後に別の設計作業として扱います。

- 認証方式と文書ACL
- 部門、テナント、環境の分離
- 応答時間、可用性、同時実行数
- ログ保持、監査、アラーム
- 顧客管理鍵と閉域接続
- reranker、hybrid search、metadata filter
- API Gateway、Lambda、Web UI
- 自動同期、CI/CD、Infrastructure as Code

本番データを使う場合は、これらを未決定のまま進めてはいけません。
本章は、架空データを使ってRAGの四工程を確認する技術ハンズオンです。

## 10.1.5 設計へ進む条件

次の三点を確認したら、10.2へ進みます。

1. 代表質問、期待回答、期待する根拠文書が一組あります。
2. 登録する文書に、機密情報、個人情報、顧客の本番情報が含まれていません。
3. AWS上に作成したリソースへ費用が発生し得ることと、終了時に削除することを理解しています。
