# 10.3 ハンズオン：設計を一枚にまとめる

AWSを操作する前に、作るものを一枚へ固定します。
この作業は10分程度です。

## 10.3.1 最初に知る六つの用語

本章から読み始めた場合は、次の意味だけを押さえてください。

| 用語 | このハンズオンでの意味 |
|---|---|
| 原文 | S3へ置くMarkdownファイル |
| chunk | 検索しやすい長さへ分けた原文の一部 |
| embedding | 文の意味を表す数値vector |
| vector index | 質問に近いembeddingを探す索引 |
| score | 質問とchunkの関連度を検索器が表した値 |
| citation | 生成された回答から、利用した原文へ戻るための参照 |

RAGは、モデルへ文書を学習させ直す仕組みではありません。
質問時に関係するchunkを検索し、その内容を生成モデルへ渡して回答を作ります。

## 10.3.2 Step 1：代表質問を固定する

今回の代表質問は次のとおりです。

```text
スタンダードプランのサポート受付時間と問い合わせ方法を教えてください。
```

期待する回答は次の内容です。

```text
受付時間は平日09:00から17:00までの日本標準時です。
Webフォームまたはメールで問い合わせます。
```

期待する根拠文書は`support-policy.md`です。

質問、期待回答、期待文書を先に固定する理由は、検索失敗と生成失敗を区別するためです。

- `support-policy.md`が上位にない場合は、検索前処理または検索の問題です。
- 上位にあり、本文も正しいのに回答が違う場合は、生成の問題です。

## 10.3.3 Step 2：入力文書を固定する

付属する文書は、すべて架空の製品「Example Support Cloud」に関するものです。

| ファイル | 主題 | 代表質問での期待 |
|---|---|---|
| `product-overview.md` | 製品機能とプラン概要 | 補助的または下位 |
| `support-policy.md` | 受付時間、問い合わせ方法、初回応答 | 1位 |
| `security-guide.md` | 暗号化とバックアップ | 代表質問では下位 |

意図的に主題の異なる文書を用意することで、検索順位が質問に応じて変わることを確認できます。

顧客文書へ置き換える場合も、最初は次を満たす文書だけにします。

- UTF-8のテキストまたはMarkdown
- 3件から20件
- 所有者が分かる
- 期待回答を人が原文で確認できる
- 機密情報、個人情報、認証情報を含まない

## 10.3.4 Step 3：構成値を固定する

次の値を変更せずに一度動かします。

```yaml
lab_name: rag-four-elements
region: ap-northeast-1
source_prefix: source/
parser: default
chunking: default
embedding_model: amazon.titan-embed-text-v2:0
embedding_dimensions: 1024
embedding_type: float
vector_store: s3_vectors_quick_create
search_type: semantic
number_of_results: 5
generation_model: anthropic.claude-haiku-4-5-20251001-v1:0
```

初回からchunk size、embedding、検索件数を同時に変更すると、結果が変わった理由を説明できません。
本章では比較実験をせず、固定した一経路が動くことだけを確認します。

## 10.3.5 Step 4：リソース名を決める

AWS内で見つけやすいよう、次の名前を使います。

| リソース | 名前 |
|---|---|
| S3原文バケット | `rag-four-elements-<account-id>-<suffix>` |
| Knowledge Base | `rag-four-elements-kb` |
| data source | `rag-four-elements-source` |
| service role | コンソールの自動生成名 |
| S3 vector bucket | Quick createされた名前を記録 |
| S3 vector index | Quick createされた名前を記録 |

S3 bucket名は全AWSアカウントで一意である必要があります。
アカウントIDと時刻をsuffixへ入れ、他の利用者と重複しない名前にします。

## 10.3.6 Step 5：実行環境を確認する

必要なものは次のとおりです。

- 課金が許可された検証用AWSアカウント
- root userではないIAM identity
- S3、S3 Vectors、Amazon Bedrock Knowledge Bases、model invocationを利用できる権限
- service roleを自動作成するためのIAM role作成・PassRole権限
- AWS CLI v2
- Python 3.11以降
- `jq`（API応答の一部表示に使用。なくても全JSONは確認可能）
- インターネットへ接続できる端末、またはAWS CloudShell

組織のAWSアカウントでは、必要な権限を管理者へ依頼してください。
権限不足を避けるために、個人判断でAdministratorAccessを付与しません。

端末で次を実行し、利用するidentityとregionを確認します。

```bash
aws --version
aws sts get-caller-identity
export AWS_REGION=ap-northeast-1
aws configure set region "$AWS_REGION"
```

`Account`と`Arn`が、ハンズオンで使う検証環境のものか確認します。
別のアカウントであれば、この先へ進みません。

- [ ] 期待回答を原文から確認しました。
- [ ] 期待する1位の文書を決めました。
- [ ] 検証用文書に機密情報と個人情報がありません。
- [ ] 東京リージョンを使います。
- [ ] 作成するAWSリソースと、費用が発生し得ることを理解しました。

- [ ] AWS CLIが正しい検証アカウントを指しています。
