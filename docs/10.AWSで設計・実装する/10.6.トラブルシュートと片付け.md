# 10.6 トラブルシュートして片付ける

最後に、四工程の完了を証拠で確認し、作成したAWSリソースを削除します。

## 10.6.1 四工程の完了表

**表10-5　RAGの四工程と確認証拠**

| 工程 | 実行したこと | 完了を証明する結果 |
|---|---|---|
| 検索前処理 | S3へ3文書を配置し、Knowledge Baseを同期 | 同期`COMPLETE`、処理3件、失敗0件 |
| 検索 | `Retrieve`を実行 | rank、score、本文、S3 URIを持つ結果が返る |
| 検索後処理 | 上位5件へ制限し、順位と本文を確認 | 代表質問で`support-policy.md`が1位 |
| 生成 | `RetrieveAndGenerate`を実行 | 正しい回答、引用、session IDが返る |

最終的な完了条件は、コマンドが終了コード0になったことだけではありません。
期待文書、期待回答、引用先が一致していることまで確認します。

## 10.6.2 よくあるエラー

### `KB_IDが未設定です`

Knowledge Base IDを環境変数へ設定します。

```bash
export KB_ID=実際のKnowledge-Base-ID
```

Knowledge Base名やdata source IDではありません。

### `ResourceNotFoundException`

最初にregionとIDを確認します。

```bash
printf 'region=%s kb=%s\n' "$AWS_REGION" "$KB_ID"
aws bedrock-agent get-knowledge-base \
  --knowledge-base-id "$KB_ID" \
  --region "$AWS_REGION"
```

東京以外で作ったKnowledge Baseを、東京のendpointから呼び出すことはできません。

### `AccessDeniedException`

二つのroleを区別します。

- terminalのIAM identity：`Retrieve`、`RetrieveAndGenerate`、generation modelの利用権限が必要です。
- Knowledge Base service role：S3原文、embedding model、S3 vector indexの利用権限が必要です。

エラーのoperation名とresource ARNを管理者へ伝えます。
ハンズオンを通すために権限境界を無条件で広げません。

### `ValidationException`とManaged Knowledge Baseの記述が出る

`RetrieveAndGenerate`はManaged Knowledge Baseでは利用できません。
10.4の手順で、S3 data sourceとS3 Vectorsを使う標準Knowledge Baseを作り直します。

### API operationが見つからない

仮想環境を有効にし、boto3を更新します。

```bash
source .venv-rag-handson/bin/activate
python -m pip install --upgrade boto3
python -c 'import boto3; print(boto3.__version__)'
```

OSに最初から入っている古いboto3と混在していないか確認します。

### 検索結果が0件

次の順に確認します。

1. S3の`source/`に3ファイルがあります。
2. data sourceのS3 URIが同じprefixを指しています。
3. 最新の同期statusが`COMPLETE`です。
4. 同期statisticsの失敗件数が0です。
5. Knowledge BaseとAPIのregionが同じです。

文書をアップロードしただけではindexは更新されません。
S3を変更した後は、data sourceをもう一度同期します。

### 期待文書が1位にならない

次を確認します。

- 質問を本文どおりコピーしたか
- `support-policy.md`がS3にあるか
- 更新後に同期したか
- 1位の本文に受付時間が実際に含まれていないか

scoreのしきい値を先に調整しません。
正しい原文が取り込まれているかを最初に確認します。

### 回答は返るが引用がない

`generation.citations`を確認し、空なら次を調べます。

1. `Retrieve`の上位本文だけで質問へ答えられるか
2. generation model ARNが東京リージョンの有効なモデルか
3. `RetrieveAndGenerate`応答を付属クライアントが加工する前のAPI仕様に変更がないか

回答文が正しく見えても、引用で原文へ戻れなければ本章の完了条件を満たしません。

## 10.6.3 AWSリソースを削除する

AWSリソースには費用が発生し得ます。
結果を確認したら、次の順で削除します。

### 1. Knowledge Baseを削除する

Amazon Bedrock consoleで`rag-four-elements-kb`を選び、削除します。
削除対象のIDが、記録した`KB_ID`と一致することを確認してください。

### 2. S3 vector indexとvector bucketを削除する

Quick createされたvector bucket名とindex名を環境変数へ設定します。

```bash
export VECTOR_BUCKET=YOUR_VECTOR_BUCKET_NAME
export VECTOR_INDEX=YOUR_VECTOR_INDEX_NAME
```

削除前に対象を確認します。

```bash
aws s3vectors get-index \
  --vector-bucket-name "$VECTOR_BUCKET" \
  --index-name "$VECTOR_INDEX" \
  --region "$AWS_REGION"
```

名前が記録と一致した場合だけ、indexを先に削除します。

```bash
aws s3vectors delete-index \
  --vector-bucket-name "$VECTOR_BUCKET" \
  --index-name "$VECTOR_INDEX" \
  --region "$AWS_REGION"
```

次にvector bucketを削除します。

```bash
aws s3vectors delete-vector-bucket \
  --vector-bucket-name "$VECTOR_BUCKET" \
  --region "$AWS_REGION"
```

bucket policyが残っているというエラーの場合は、対象名を再確認してからpolicyを削除し、bucket削除を再実行します。

```bash
aws s3vectors delete-vector-bucket-policy \
  --vector-bucket-name "$VECTOR_BUCKET" \
  --region "$AWS_REGION"
```

vector indexの削除方法は、[S3 Vectorsのindex削除](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-vectors-index-delete.html)でも確認できます。

### 3. S3原文バケットを削除する

対象名を表示します。

```bash
echo "$DOC_BUCKET"
aws s3 ls "s3://${DOC_BUCKET}/" --recursive
```

ハンズオンで作ったbucketだと確認してから、objectとbucketを削除します。

```bash
aws s3 rm "s3://${DOC_BUCKET}/" --recursive
aws s3api delete-bucket \
  --bucket "$DOC_BUCKET" \
  --region "$AWS_REGION"
```

### 4. 自動作成されたservice roleを確認する

コンソールが作成したKnowledge Base専用roleが残っている場合があります。
IAM consoleで、記録したrole名と一致し、他のリソースから使われていないことを確認してから削除します。

role名が不明、または他用途との共有が疑われる場合は削除せず、AWS管理者へ確認します。

## 10.6.4 削除確認

次のすべてを確認します。

- [ ] `rag-four-elements-kb`が存在しません。
- [ ] Quick createしたS3 vector indexが存在しません。
- [ ] Quick createしたS3 vector bucketが存在しません。
- [ ] `DOC_BUCKET`が存在しません。
- [ ] 専用service roleを削除するか、残した理由を記録しました。
- [ ] `/tmp`のAPI結果以外に、検証データを複製していません。

ここまでで、AWS上の最小RAGハンズオンは終了です。
本番化する場合は、この構成へそのまま利用者を接続せず、第7章の評価と第8章の安全設計を別途行います。
