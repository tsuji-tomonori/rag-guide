# 9.4 Federated RAG

## 9.4.1. federated化が必要な条件

Federated化は、単一indexが大きいから行うのではなく、sourceごとにACL、schema、更新頻度、検索器、引用形式が異なるときに必要になる。部門文書、tenant別KB、SQL、graph、外部APIを一つへ無理に正規化すると、権限と正本の意味が失われる。Modular RAGのrouting発想を使い、統合indexとfederated構成を、運用owner、latency、重複、cross-source質問で比較する。分離の理由をsource catalogへ明記する。

**参照論文**
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 9.4.2. source catalogとretrieval contract

各sourceを検索可能なserviceとして扱い、source ID、owner、trust、freshness、ACL、対応query、latency、costをcatalog化する。retrieverの入力query・filterと、出力candidate・score・citation payloadを共通contractへ揃える一方、固有の意味は消さない。たとえばSQL rowとPDF chunkは同じevidence interfaceへ載せてもlocationが違う。schema versionとerror codeを定め、source停止・stale・permission-deniedをrouterが区別できるようにする。

**参照論文**
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)

## 9.4.3. routing

Routingにはrule、classifier、embedding、確率model、LLMを使える。query中の「在庫」はAPI、「規程」は文書KBというruleは説明しやすく、複雑な自然文はclassifierが補う。top-1で速さを取り、確信が低い場合だけtop-2へfan-outする。Adaptive-RAGの段階選択を参考に、source数、timeout、fallbackをbudget化する。権限のないsourceを候補に含めず、routerが知らないsourceの存在を回答へ漏らさない。

**参照論文**
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)

## 9.4.4. per-source retrieval

Federated構成でも各sourceに同じtop-kを配る必要はない。規程KBはBM25＋dense、売上DBはSQL、関係dataはgraph、最新statusはAPIを使い、sourceのprecisionとlatencyに応じてcandidate budgetを割り当てる。document retrievalのscoreとSQLの結果確度を同じ尺度にせず、まずsource内で品質を確定する。各retrieverのfilter、timeout、snapshotを個別traceし、source固有failureを中央routerの失敗へ混ぜない。

**参照論文**
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)
- [P069] [RAT-SQL: Relation-Aware Schema Encoding and Linking for Text-to-SQL Parsers](https://arxiv.org/abs/1911.04942)

## 9.4.5. result fusion

異なるsourceの結果はraw scoreを直接足さず、RRF、校正済みscore、source weightを比較する。RRFは順位だけで統合できるため異種retrieverに使いやすいが、source trustや鮮度は別制約として残す。document ID、version、sourceを保持したままduplicateとmirrorを統合し、一sourceの大量候補がcontextを独占しないquotaを置く。fusion後もどのsource・queryが候補を押し上げたかを記録し、rerankへ渡す。

**参照論文**
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P106] [An Analysis of Fusion Functions for Hybrid Retrieval](https://arxiv.org/abs/2210.11934)

## 9.4.6. 矛盾・鮮度・信頼順位

複数sourceが違う値を返したら、score順で決着させない。authority、effective date、version、原文か生成summaryか、適用scopeを比較し、正式な後継版、source固有例外、未解決衝突を分ける。external Webの新情報が社内正式規程を自動上書きしないようpolicyを定める。矛盾が残れば複数見解と確認先を提示するか回答を保留し、source ownerへ通知する。鮮度と信頼をfusion重みへ隠さず明示する。

**参照論文**
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)
- [P052] [FEVER: a Large-scale Dataset for Fact Extraction and VERification](https://aclanthology.org/N18-1074/)

## 9.4.7. ACLとtenant isolation

認可はrouter前と各retriever内の二段で行う。最初に利用可能sourceをidentityから絞り、その後source固有のrow・document ACLを適用する。service accountの広い権限だけで全tenantへ検索させず、user delegationとserver-side policyを使う。権限外sourceの名前、候補件数、拒否理由も利用者向けtraceへ出さない。cache key、fusion、fallbackがtenant境界を保持するかをnegative testし、source停止時にも別tenantへ迂回しない。

**参照論文**
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)
- [P144] [Curator: Efficient Indexing for Multi-Tenant Vector Databases](https://arxiv.org/abs/2401.07119)

## 9.4.8. citationとprovenance

Federated回答のcitationは、PDF page、DB row、API response、graph pathを共通UIへ変換しつつ、元のlocationを失わない。claimからsource、document／record、実行query、snapshot、時刻へ遡れるpayloadを持つ。複数sourceの一致を一つの引用へ潰さず、独立した裏付けかmirrorかを示す。SQLやAPIでは実行条件とdata freshnessを表示し、再実行で値が変わる場合に回答時点のresult hashを保存する。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P128] [Measuring Attribution in Natural Language Generation Models](https://arxiv.org/abs/2112.12870)
