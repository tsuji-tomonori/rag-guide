# 9.3 Conversational RAG

## 9.3.1. 会話型RAG固有の失敗

会話型RAGでは、単発QAにない失敗が起きる。「それ」が何を指すか、話題が変わった後に以前の対象へ戻ったか、前turnの期間条件を継ぐべきかを誤ると、検索は流暢でも別問題を解く。QReCCやConversational Dense Retrievalは、会話queryの解決と検索を独立評価する必要を示す。本節ではcoreference、省略、topic shift、topic return、誤継承をfixture化し、turn単位だけでなくsession全体の成功を測る。

**参照論文**
- [P113] [Open-Domain Question Answering Goes Conversational via Question Rewriting](https://arxiv.org/abs/2010.04898)
- [P063] [Few-Shot Conversational Dense Retrieval](https://arxiv.org/abs/2105.04166)

## 9.3.2. 会話状態の分離

会話の情報を一つのmemoryへ混ぜず、retrievalに必要な対象・用語、generationに必要な形式・制約、長期的な嗜好を分ける。raw transcriptをそのままqueryへ連結すると、古い話題や過去の誤回答が検索を汚す。Query Resolutionは現在turnに必要な履歴を選ぶ考え方を示す。各stateへsource turn、confidence、期限を持たせ、業務文書KBと利用者memoryを別namespaceにする。会話要約も正本ではなく再検索の補助として扱う。

**参照論文**
- [P114] [Query Resolution for Conversational Search with Limited Supervision](https://arxiv.org/abs/2005.11723)
- [P063] [Few-Shot Conversational Dense Retrieval](https://arxiv.org/abs/2105.04166)

## 9.3.3. standalone query rewriting

Standalone query rewritingでは、指示語の参照先を解き、製品、期間、地域、否定条件を補い、検索に不要な挨拶や過去話題を除く。CONQRRはrewriteを自然さではなく下流retrieval rewardへ最適化し、会話検索の改善を狙った。実装ではoriginalとrewriteを両方保存し、必須slotの保持を検査する。曖昧な参照を一意に捏造せず、複数queryかclarificationへ分岐し、rewriteなしのbaselineも残す。

**参照論文**
- [P062] [CONQRR: Conversational Query Rewriting for Retrieval with Reinforcement Learning](https://arxiv.org/abs/2112.08558)
- [P113] [Open-Domain Question Answering Goes Conversational via Question Rewriting](https://arxiv.org/abs/2010.04898)

## 9.3.4. contextual dense retrieval

Contextual dense retrievalは、明示的なrewriteを作らず、会話履歴をquery encoderへ直接取り込む。Few-Shot Conversational Dense Retrievalは、rewrite型とhistory-aware retrieverを比較し、少量dataでの適応を扱った。利点は省略を内部表現で捉えられることだが、既存indexとの互換、学習data、trace説明が重くなる。汎用retriever＋rewriteで不足するfailureが再現できる場合だけ会話専用encoderを検討し、単発queryの回帰を測る。

**参照論文**
- [P063] [Few-Shot Conversational Dense Retrieval](https://arxiv.org/abs/2105.04166)
- [P113] [Open-Domain Question Answering Goes Conversational via Question Rewriting](https://arxiv.org/abs/2010.04898)

## 9.3.5. short-term / long-term memory

Short-term memoryはsession内の発話と一時条件、long-term memoryは許可された嗜好や継続設定を扱う。業務事実や規程をmemoryへ保存して正本代わりにすると、更新・引用・権限を失う。session transcript、会話要約、preference、user-provided factを別namespaceにし、sourceと更新日を付ける。長期化する情報は明示同意と削除手段を要求し、検索時にはauthoritative KBとmemoryの役割を分けてcontextへ提示する。

**参照論文**
- [P114] [Query Resolution for Conversational Search with Limited Supervision](https://arxiv.org/abs/2005.11723)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 9.3.6. session・tenant・privacy

Conversation stateはuser、tenant、sessionへ正しく束縛し、共有cacheや非同期処理で混ざらないようにする。保存同意、retention、削除、PII masking、権限変更の伝播を設計し、退職・契約終了後のmemory参照をnegative testする。Membership inferenceやRAG extractionの脅威はmemoryにも及ぶため、存在を示唆する拒否や反復queryを監視する。session IDをclient任せにせずbackendでidentityと照合し、cross-tenant履歴を一件でも許さない。

**参照論文**
- [P151] [Generating Is Believing: Membership Inference Attacks against Retrieval-Augmented Generation](https://arxiv.org/abs/2406.19234)
- [P157] [RAG-Thief: Scalable Extraction of Private Data from Retrieval-Augmented Generation Applications with Agent-based Attacks](https://arxiv.org/abs/2411.14110)

## 9.3.7. retrievalとgeneration

検索にはrewrite済みqueryを使い、回答には必要な会話制約とoriginal wordingを戻す。これによりretriever向けの明確さと、利用者意図に沿った応答を両立する。過去回答のcitationを再利用する場合も、source版、有効日、ACLを現時点で再確認する。会話中にtopicが変わったら古いevidenceを自動継承せず、新しいtraceを開始する。検索失敗と履歴解決失敗を別labelにし、clarification後の再検索が正しい対象へ移ったか評価する。

**参照論文**
- [P062] [CONQRR: Conversational Query Rewriting for Retrieval with Reinforcement Learning](https://arxiv.org/abs/2112.08558)
- [P114] [Query Resolution for Conversational Search with Limited Supervision](https://arxiv.org/abs/2005.11723)
