# 9.3 Conversational RAG

**この節の目的:** 会話中の省略、指示語、topic shift、過去条件を検索可能なqueryへ変換し、sessionとmemoryを安全に扱う設計を示す。

## 9.3.1. 会話型RAG固有の失敗
  - 指示語、省略、topic shift、topic return、前turnの制約、誤った履歴継承を例示する。
  - 単発QAと異なる評価単位を定義する。
## 9.3.2. 会話状態の分離
  - retrievalに必要な履歴、generationに必要な制約、長期的なユーザー嗜好を分ける。
  - raw transcriptを無条件に検索queryやpromptへ連結しない。
## 9.3.3. standalone query rewriting
  - coreference解決、期間・対象・条件の補完、不要履歴の除去を行う。
  - original queryとrewritten queryの両方をtraceに保存する。
## 9.3.4. contextual dense retrieval
  - query rewritingとの違い、教師queryの作り方、retriever再学習のコストを説明する。
  - 既存retrieverを維持するか、会話専用retrieverを持つかを比較する。
## 9.3.5. short-term / long-term memory
  - session内履歴、会話要約、ユーザー嗜好、業務事実を別namespaceにする。
  - 業務文書KBとmemoryを混同せず、memoryを正本にしない。
## 9.3.6. session・tenant・privacy
  - user_id、tenant_id、session_idの対応をアプリ側で保証する。
  - 保存同意、retention、削除、PII masking、権限変更反映を設計する。
## 9.3.7. retrievalとgeneration
  - rewrite queryで検索し、回答時には必要な会話制約だけを追加する。
  - 過去回答の引用を再利用する場合は、元sourceの有効性を再確認する。
