# 9.7 Hierarchical RAG / RAPTOR

## 9.7.1. 適用する質問

階層検索が必要かは質問の粒度で決まる。局所事実、section要約、文書全体、corpus全体、cross-document multi-hopを分け、FAQ中心ならflat chunkで十分な場合も示す。RAPTORはleafをcluster・要約してtreeを作り、長文の異なる抽象度を検索する。導入前に、baselineが失敗するのが局所hit不足なのか全体把握不足なのかを確認し、概要質問sliceで階層化の増分を測る。

**参照論文**
- [P068] [RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval](https://arxiv.org/abs/2401.18059)
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)

## 9.7.2. hierarchyの単位

Hierarchyにはleaf chunk、proposition、section summary、document summary、cluster／community summaryがある。上位nodeほど広い傾向を表す一方、具体的な条件と少数意見が失われる。各nodeへlevel、children、source document、page、spanを持たせ、summaryからleafへ戻れるprovenanceを必須にする。検索単位、LLMへ渡す単位、citation単位を分け、上位summaryだけを最終根拠として断定しない。

**参照論文**
- [P068] [RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval](https://arxiv.org/abs/2401.18059)
- [P078] [ArchRAG: Attributed Community-based Hierarchical Retrieval-Augmented Generation](https://arxiv.org/abs/2502.09891)

## 9.7.3. offline tree構築

Offline treeはparse、leaf生成、embedding、clustering、summary生成、parent link作成の順に構築する。RAPTORは再帰的なclusteringとabstractive summaryで文書を多層表現し、query時にtreeから関連nodeを選ぶ。cluster algorithm、summary model・prompt、random seedをversion化し、同じsource snapshotから再構築できるようにする。summary生成失敗をbuild成功へ混ぜず、leaf coverageとancestor参照の整合をquality gateにする。

**参照論文**
- [P068] [RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval](https://arxiv.org/abs/2401.18059)
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)

## 9.7.4. retrieval方式

検索方式には全levelを同時検索するcollapsed tree、level別検索、top-down navigation、leaf-first parent expansionがある。概要質問は上位node、具体質問はleafを優先し、粒度classifierが不確実ならmulti-level fusionを使う。RAPTORのtree retrievalと通常parent-childを区別し、候補数、token、latencyを比較する。上位summaryが高scoreでも、回答時には関連leafを展開して具体的根拠を確認する。

**参照論文**
- [P068] [RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval](https://arxiv.org/abs/2401.18059)
- [P078] [ArchRAG: Attributed Community-based Hierarchical Retrieval-Augmented Generation](https://arxiv.org/abs/2502.09891)

## 9.7.5. index-time enrichment

Index-time enrichmentではsummary、synthetic question、bridge fact、entity linkをofflineに付与し、online検索を軽くする。EnrichIndexはLLM生成metadataを事前追加して検索品質を高め、online tokenを大幅に削減したと報告する。IndexRAGはcross-document推論用のbridge factsをindex時に作り、single-pass retrievalを狙う。代償はbuild費用と誤りの固定化である。生成物を原文と分け、version・support span・再計算条件を持たせる。

**参照論文**
- [P081] [EnrichIndex: Using LLMs to Enrich Retrieval Indices Offline](https://arxiv.org/abs/2504.03598)
- [P082] [IndexRAG: Bridging Facts for Cross-Document Reasoning at Index Time](https://arxiv.org/abs/2603.16415)

## 9.7.6. update・delete・rollback

一文書の更新でも、そのleafだけでなく所属clusterと全ancestor summaryが影響を受ける。依存graphから再生成範囲を求め、削除文書の情報が上位summaryへ残らないことを検査する。新treeを別indexでbuildし、golden queryとsummary diffを通してから切り替える。cluster再編でnode IDが変わる場合はcitationと評価labelの移行を管理し、旧treeへ即時rollbackできる期間を設ける。

**参照論文**
- [P068] [RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval](https://arxiv.org/abs/2401.18059)
- [P093] [FreshDiskANN: A Fast and Accurate Graph-Based ANN Index for Streaming Similarity Search](https://arxiv.org/abs/2105.09613)

## 9.7.7. summary errorとprovenance

Summaryは検索補助であって原典ではない。hallucination、過度な抽象化、条件・少数意見の消失をclaim-levelで検出し、各文をsupporting leafへ結ぶ。RAPTORの利点を享受しても、回答citationはsummaryだけで終わらせず元page・spanへ着地させる。上位nodeとleafが矛盾する場合はleafを優先してsummaryを再生成し、model変更時のdriftを回帰評価する。summary trustを原文と同じにしない。

**参照論文**
- [P068] [RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval](https://arxiv.org/abs/2401.18059)
- [P128] [Measuring Attribution in Natural Language Generation Models](https://arxiv.org/abs/2112.12870)
