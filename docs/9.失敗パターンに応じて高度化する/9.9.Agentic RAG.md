# 9.9 Agentic RAG

**この節の目的:** 固定pipelineでは解けないmulti-step調査やtool利用に対して、計画・検索・検証・実行を制約付きloopとして設計し、安全境界を示す。

## 9.9.1. agentic化する条件
  - multi-step、検索先が動的、外部計算 / API必要、途中結果で計画変更が必要な場合に限定する。
  - 固定workflowや通常RAGで十分なqueryをagentへ送らない。
## 9.9.2. 基本loop
  - plan→retrieve / tool→observe→verify→replan→answerを示す。
  - 各stepの停止条件と失敗時fallbackを定義する。
## 9.9.3. reflection / planner / executor / verifier
  - reflectionは「根拠十分性」「未解決論点」「次action」を返すstructured judgeとして実装する。
  - planner、executor、verifierの責務、入力schema、出力schema、共有stateを分ける。
  - 自由な自然言語planより、JSON schemaで制約したplanを優先する。
## 9.9.4. retrieval as a tool
  - query decomposition、iterative retrieval、source routing、evidence sufficiency判定を扱う。
  - 同じ検索を繰り返すloopを検知する。
## 9.9.5. tool registry
  - tool name、description、input / output schema、permission、side effect、timeout、retry、costを定義する。
  - read toolとwrite toolを分離する。
## 9.9.6. 実行制約
  - max_steps、max_tool_calls、timeout、token / monetary budget、allowed tool sequenceを設定する。
  - write系はhuman approvalとidempotency keyを必須にする。
## 9.9.7. identity・authorization・session
  - ユーザー権限を検索とtool callへ伝播する。
  - service credentialだけで全data / APIへアクセスさせない。
## 9.9.8. prompt injectionとuntrusted observation
  - retrieved document、email、web、tool outputを命令ではなくuntrusted dataとして隔離する。
  - ユーザー目的から外れるtool callをpolicy engineで拒否する。
## 9.9.9. observability
  - plan、tool input / output、retrieved evidence、decision、approval、final answerをtrace_idで結ぶ。
  - 内部思考の全文保存ではなく、監査に必要な構造化decision logを残す。
## 9.9.11. multi-agent collaboration
  - 検索、SQL、graph、検証などの役割分担と、責務重複・矛盾・loopのリスクを示す。
  - まずplanner-executor-verifier型を採用し、複数agent化は必要性を評価してから行う。
