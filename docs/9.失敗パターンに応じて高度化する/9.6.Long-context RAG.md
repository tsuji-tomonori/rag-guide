# 9.6 Long-context RAG

## 9.6.1. long-contextとRAGの関係

Long-contextはRAGの反対ではない。全文を読ませる範囲が広がっても、どの文書を選ぶか、権限、鮮度、citationは残る。RAG、単一文書全文、選別済み複数文書をlong readerへ渡すhybridを比較する。Lost in the Middleは長い入力で重要情報の位置による性能差を示し、context windowの公称長と実効利用を区別した。検索単位とreader budgetを変える技術として位置づけ、corpus全投入を既定にしない。

**参照論文**
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)
- [P075] [Retrieval Augmented Generation or Long-Context LLMs? A Comprehensive Study and Hybrid Approach](https://arxiv.org/abs/2407.16833)

## 9.6.2. 選択基準

選択基準はmodelの最大token数ではなく、corpus size、質問scope、更新頻度、ACL、citation粒度、latency、costである。単一契約全体の要約ならlong-contextが自然だが、数万文書から該当版を探すならretrievalが必要になる。RAGとLCの比較研究は、資源が十分ならLCが高品質な場合が多い一方、RAGが大幅に安価であることを報告した。qualityだけでなくtaskごとの予算とprivacyを含めて経路を決める。

**参照論文**
- [P075] [Retrieval Augmented Generation or Long-Context LLMs? A Comprehensive Study and Hybrid Approach](https://arxiv.org/abs/2407.16833)
- [P133] [RULER: What’s the Real Context Size of Your Long-Context Language Models?](https://arxiv.org/abs/2404.06654)

## 9.6.3. retrieval unitの拡張

Long readerを使うと、retrieval unitを小chunkだけでなくparent section、document、関連document groupへ広げられる。小unitでhitを見つけてparentを展開する方法なら、検索precisionと読解文脈を分担できる。ただし大きなdocument内で根拠が埋もれる場合は、内部再検索やsection indexが必要になる。Dense Xの粒度比較を参考に、事実質問、要約、multi-hopで最適単位を分け、document全体を渡すだけで改善したと結論づけない。

**参照論文**
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)
- [P068] [RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval](https://arxiv.org/abs/2401.18059)

## 9.6.4. context selection

Context windowが大きくても、候補選別は省かない。dedup、MMR、rerank、parent-child展開、conflict filteringを先に行い、top-kの件数ではなくevidence coverageを測る。RAGGEDはretrieval depthを増やすとreaderのnoise robustness次第で性能が上下することを示した。関連documentを広く読むqueryと、一点の事実を探すqueryでbudgetを変え、低信頼・権限外文書をlong contextへ入れない。

**参照論文**
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)
- [P038] [The Use of MMR, Diversity-Based Reranking for Reordering Documents and Producing Summaries](https://dl.acm.org/doi/10.1145/290941.291025)

## 9.6.5. compressionとpacking

広いcontextをそのまま連結せず、抽出型・要約型compressionとpackingで論点を整理する。LongLLMLinguaはquery-aware圧縮と文書再配置により長文利用を改善しようとしたが、数値や例外の欠落を評価する必要がある。重要spanとquote mappingを保ち、論理順・時系列・authority順をqueryに応じて選ぶ。要約はcitation sourceにせず原文へ着地させ、圧縮率ではなく支持情報の保持をrelease条件にする。

**参照論文**
- [P037] [LongLLMLingua: Accelerating and Enhancing LLMs in Long Context Scenarios via Prompt Compression](https://arxiv.org/abs/2310.06839)
- [P036] [LLMLingua: Compressing Prompts for Accelerated Inference of Large Language Models](https://arxiv.org/abs/2310.05736)

## 9.6.6. position bias対策

Position biasは、根拠を中央へ置いたときの見落としとして試験する。Lost in the Middleは先頭・末尾より中央で性能が低下する傾向を示し、RULERはmulti-needleやaggregationで実効context能力を測った。同じevidenceを位置だけ変えるstress testを作り、model・context長別に利用率を比較する。重要根拠を前方へ置く工夫は使えるが、順序だけに依存せずcontextを短くし、複製でtokenを浪費しない。

**参照論文**
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)
- [P133] [RULER: What’s the Real Context Size of Your Long-Context Language Models?](https://arxiv.org/abs/2404.06654)
- [P120] [Attention Sorting Combats Recency Bias In Long Context Language Models](https://arxiv.org/abs/2310.01427)

## 9.6.7. RAG / LC / hybrid routing

Routingは通常RAG、long-context、RAG＋long readerの三経路から始める。短い事実質問は通常RAG、単一文書全体はLC、複数文書の比較・統合はhybridというruleをbaselineにし、query complexityとbudgetで調整する。Self-Routeはmodel自身の回答可能性を利用してRAG／LCを選び、LC相当の品質を低costで狙った。router誤りをslice別に測り、timeout時は根拠なし経路でなく通常RAGへ戻す。

**参照論文**
- [P075] [Retrieval Augmented Generation or Long-Context LLMs? A Comprehensive Study and Hybrid Approach](https://arxiv.org/abs/2407.16833)
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)

## 9.6.8. securityとprivacy

Long-contextは不要文書までLLMへ送るため、漏えいと攻撃面を拡大しやすい。ACL・PII filterを文書選択前に適用し、低信頼sourceやprompt injectionを含むpageを一括投入しない。長いcontextを全文ログへ保存すると監査dataも膨らむため、snapshot IDとredacted spanを使い分ける。providerへ送るtoken量、region、retentionを評価し、context windowの余裕をdata minimization放棄の理由にしない。

**参照論文**
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)
- [P141] [Analyzing Leakage of Personally Identifiable Information in Language Models](https://arxiv.org/abs/2302.00539)
- [P151] [Generating Is Believing: Membership Inference Attacks against Retrieval-Augmented Generation](https://arxiv.org/abs/2406.19234)
