# 9.5 Structured Data RAG

**この節の目的:** 数値、件数、在庫、状態、日付などを文書chunkから推測せず、DB・表・APIを正本として安全に取得し、LLMは問い合わせ生成と説明に限定する設計を示す。

## 9.5.1. 文書RAGで扱わないデータ
  - 最新値、集計、状態、トランザクション結果をvector化した文書から答えない原則を示す。
  - structured sourceを正本とし、LLMはquery生成と説明を担う。
## 9.5.2. 対象sourceの分類
  - RDB、warehouse、CSV / table、BI semantic layer、REST / GraphQL API、knowledge graphを分ける。
  - read consistency、更新頻度、権限、監査要件を整理する。
## 9.5.3. schema理解とsemantic layer
  - business term、table、column、join、unit、time zone、effective dateをmetadata化する。
  - ユーザー用語とschemaの対応をschema linkingで解く。
## 9.5.4. text-to-SQL pipeline
  - source routing→schema selection→question decomposition→SQL generation→validation→execution→result explanationの順にする。
  - 生成SQLと実行結果をtraceへ保存する。
## 9.5.5. SQL安全化
  - read-only、allowed tables / columns、row limit、timeout、cost limit、parameterization、tenant predicateを強制する。
  - DDL / DML、任意function、外部network accessを禁止または別承認にする。
## 9.5.6. table QAとAPI tool
  - SQLが不要な静的表はcell selection / aggregationを検討する。
  - 最新状態や業務ロジックはAPI toolを正本として呼ぶ。
## 9.5.7. 結果のgrounding
  - 回答内の数値をresult row / cellへ紐づける。
  - query実行時刻、data freshness、unit、filter条件を表示する。
## 9.5.8. 失敗とfallback
  - ambiguous column、複数解釈、empty result、timeout、権限不足時の応答を定義する。
  - 危険または不確実なqueryは実行せず確認要求へ分岐する。
