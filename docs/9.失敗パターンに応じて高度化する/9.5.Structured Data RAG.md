# 9.5 Structured Data RAG

## 9.5.1. 文書RAGで扱わないデータ

在庫、残高、件数、最新status、集計値を文書chunkから推測してはいけない。vector化したsnapshotは取り込み時点のコピーであり、transactionの正本ではない。Structured Data RAGではDB・APIを直接参照し、LLMの役割をschema選択、query生成、結果説明へ限定する。MRKLが専門moduleへ処理をrouteするように、計算や状態確認を文書検索から分離する。RAGは定義・手順・背景を補い、現在値そのものはstructured sourceへ委ねる。

**参照論文**
- [P072] [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445)
- [P069] [RAT-SQL: Relation-Aware Schema Encoding and Linking for Text-to-SQL Parsers](https://arxiv.org/abs/1911.04942)

## 9.5.2. 対象sourceの分類

Structured sourceも一括りにしない。RDBはjoinとtransaction、warehouseは分析集計、CSV・tableは静的data、BI semantic layerはbusiness定義、REST／GraphQLは業務logic、knowledge graphは関係探索を担う。各sourceについてread consistency、更新頻度、権限、query cost、監査を整理する。数値がどこで確定するかをsource of truthとして記録し、同じ質問を文書indexとDBの双方へ投げて多数決しない。

**参照論文**
- [P069] [RAT-SQL: Relation-Aware Schema Encoding and Linking for Text-to-SQL Parsers](https://arxiv.org/abs/1911.04942)
- [P072] [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445)

## 9.5.3. schema理解とsemantic layer

Natural languageから安全なqueryを作るには、table名だけでなくbusiness term、column、join、unit、timezone、有効日をsemantic layerへ記述する。RAT-SQLはrelation-aware schema encodingとschema linkingで、質問語とschema要素の対応を強化した。社内用語と物理column名を辞書・ontologyで結び、同名fieldや複数join pathの曖昧さを明示する。schema変更をversion化し、LLMへ全DBを見せず質問に関係するsubsetだけを渡す。

**参照論文**
- [P069] [RAT-SQL: Relation-Aware Schema Encoding and Linking for Text-to-SQL Parsers](https://arxiv.org/abs/1911.04942)
- [P064] [DIN-SQL: Decomposed In-Context Learning of Text-to-SQL with Self-Correction](https://arxiv.org/abs/2304.11015)

## 9.5.4. text-to-SQL pipeline

Text-to-SQLは一度の生成で実行せず、source routing、schema selection、question decomposition、SQL generation、static validation、execution、result explanationへ分ける。DIN-SQLは複雑質問をsub-problemへ分解し、schema linkingとself-correctionを組み合わせた。生成SQL、選択schema、parameter、実行plan、resultをtraceへ保存し、説明文の数値がresult rowと一致するか再検査する。曖昧な質問は勝手なjoinを作らず確認へ戻す。

**参照論文**
- [P064] [DIN-SQL: Decomposed In-Context Learning of Text-to-SQL with Self-Correction](https://arxiv.org/abs/2304.11015)
- [P069] [RAT-SQL: Relation-Aware Schema Encoding and Linking for Text-to-SQL Parsers](https://arxiv.org/abs/1911.04942)

## 9.5.5. SQL安全化

生成SQLにはread-only、allowed table・column、row limit、timeout、cost limit、tenant predicateを強制する。DDL、DML、任意function、外部network accessをdefault denyとし、user文字列はparameter化する。SQL parserとpolicyでASTを検査し、LLMの「安全です」という自己評価へ頼らない。実行credentialは最小権限とし、query planがfull scanやcross-tenant joinになる場合は停止する。write操作は別workflowと人間承認へ分離する。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)

## 9.5.6. table QAとAPI tool

静的な小表なら、SQLを生成せず必要cellとheaderを選ぶtable QAが適する。一方、在庫statusや料金計算のように業務logicがAPIにある場合は、API toolを正本として呼ぶ。table extractionの誤りを考慮し、row・column対応とunitを保持する。APIではrequest parameter、response field、実行時刻をevidenceへ変換し、エラーやpartial resultを通常値と混同しない。どちらもLLMの暗算ではなく、構造化結果をgroundingに使う。

**参照論文**
- [P100] [PubTables-1M: Towards comprehensive table extraction from unstructured documents](https://arxiv.org/abs/2110.00061)
- [P072] [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445)

## 9.5.7. 結果のgrounding

回答内の数値をSQL文ではなく、実際のresult row、cell、API fieldへ結び付ける。citationには実行時刻、data freshness、unit、filter、aggregationを含め、同じ値でも対象期間が違うことを示す。複数rowを合計した場合は計算式とsource setを保存し、丸め規則を明記する。自然言語説明が「全社」を「国内」へ過剰一般化していないかclaim checkerで検査し、resultが空なら0と解釈しない。

**参照論文**
- [P128] [Measuring Attribution in Natural Language Generation Models](https://arxiv.org/abs/2112.12870)
- [P045] [FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation](https://arxiv.org/abs/2305.14251)

## 9.5.8. 失敗とfallback

Structured queryの失敗には、ambiguous column、複数schema解釈、empty result、timeout、permission-denied、stale replicaがある。それぞれ確認質問、別source、再試行、拒否へ分岐し、危険なSQLを「結果が欲しい」から実行しない。validation失敗時に文書RAGの推測へfallbackせず、説明可能な範囲だけ返す。query costやtimeoutを越えた場合は集計粒度を勝手に変えず、利用者へ制約を示す。失敗SQLと期待動作を回帰caseへ残す。

**参照論文**
- [P064] [DIN-SQL: Decomposed In-Context Learning of Text-to-SQL with Self-Correction](https://arxiv.org/abs/2304.11015)
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
