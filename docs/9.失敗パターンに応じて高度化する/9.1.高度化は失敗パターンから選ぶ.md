# 9.1 高度化は失敗パターンから選ぶ

## 9.1.1. 高度化の前提

高度化へ進む前に、比較可能なProduction Baselineを固定する。hybrid検索、rerank、citation、refusal、traceを備え、golden datasetで既知failureを再現できることが前提である。RAG surveyのNaive／Advanced／Modular分類は、componentを増やすこと自体が成熟ではないと読める。新技術の採用ではなく、baselineで観測した失敗を解く変更として目的を定義し、品質・latency・cost・riskの初期値を残す。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)

## 9.1.2. 失敗を観測するための入力

高度化の判断材料は最終回答だけでは足りない。original・rewrite query、retrieved chunk、rerank score、packed context、answer、citation、latency、costをtraceで結び、offline評価、本番sample、人手feedbackを別sourceとして扱う。RAGCheckerはclaim単位でretrievalとgenerationを診断し、同じ誤答でも原因を分ける。本番の一例を印象で一般化せず、frequency、severity、再現性を確認してから手法選定へ進む。

**参照論文**
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)

## 9.1.3. 失敗パターンの分類

失敗を検索漏れ、検索noise、根拠矛盾、会話文脈不足、構造化data誤読、長文全体把握不足、multi-hop不足、tool必要、図表未対応へ分ける。さらに権限違反、根拠なし断定、回答不能失敗は高度化の候補ではなく、まず安全側で停止すべき欠陥である。RGB benchmarkの四能力やRAGCheckerのtaxonomyを使い、最初にgoldが失われたstageを特定する。表面的な症状ではなく再現可能なfailure classへ落とす。

**参照論文**
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)

## 9.1.4. 失敗から手法への対応付け

手法はfailureとの対応から選ぶ。会話省略にはConversational RAG、最新値・集計にはStructured Data、全体傾向にはHierarchical／GraphRAG、図表にはMultimodal、外部操作にはAgenticを候補にする。複数手法が当てはまるときは、固定pipelineの小さな変更で解けるものを先に試す。CRAGのようなcorrective flowも、通常retrievalが不十分なqueryへ限定する。対応表には期待効果だけでなく新たなcost、非決定性、安全riskを併記する。

**参照論文**
- [P079] [Corrective Retrieval Augmented Generation](https://arxiv.org/abs/2401.15884)
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)

## 9.1.5. routingと段階的エスカレーション

全queryを高度な経路へ送らず、no-retrieval、通常RAG、rerank強化、iterative retrieval、long-context、agentic flowへ段階的にescalateする。Adaptive-RAGは質問複雑度で検索なし・単発・反復を選び、Self-RouteはRAGとlong-contextを使い分けた。routing条件にquery complexity、情報源、risk tier、budgetを用い、各段階でevidence sufficiencyを確認する。確信が低いときの上位経路と、時間切れ時の安全な停止を定める。

**参照論文**
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)
- [P075] [Retrieval Augmented Generation or Long-Context LLMs? A Comprehensive Study and Hybrid Approach](https://arxiv.org/abs/2407.16833)

## 9.1.7. 高度化のアンチパターン

典型的な失敗は、GraphRAGやagentを先に入れbaselineとの差を測れなくすること、全queryを同じ高価なflowへ送ること、最終回答だけでretrievalとgenerationを混同することである。Modular RAGは部品を組み替える自由を示すが、interface、評価、rollbackがなければ複雑性だけが増える。一変更ずつablationし、改善しないmoduleは外す。手法名を成果物にせず、解消したfailureと残った副作用をrelease reportへ記録する。

**参照論文**
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
