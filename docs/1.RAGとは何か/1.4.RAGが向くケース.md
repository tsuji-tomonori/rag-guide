# 1.4 RAGが向くケース

## 参考論文一覧

| 優先度 | 論文 | この節で使う論点 | 主に読む部分 |
|---|---|---|---|
| 必読 | [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252) — Petroni et al., 2020 | open-domain QA、fact checking、dialogue、slot filling等を「外部知識が必要なタスク」として整理する枠組み | Abstract、task taxonomy、provenance評価 |
| 必読 | [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401) — Lewis et al., 2020 | knowledge-intensive QA、fact verification、generationにおけるRAGの基本的な適用範囲 | Abstract、実験タスク |
| 推奨 | [Leveraging Passage Retrieval with Generative Models for Open Domain Question Answering](https://arxiv.org/abs/2007.01282) — Izacard & Grave, 2020/2021 | 複数文書を統合する質問でretrievalとgenerationを組み合わせる価値 | Abstract、multi-passage結果 |
| 推奨 | [WebGPT: Browser-assisted question-answering with human feedback](https://arxiv.org/abs/2112.09332) — Nakano et al., 2021 | 調査型QAで検索、情報収集、回答、引用を一体化する代表例 | Abstract、システム概要、引用付き回答評価 |
| 推奨 | [Teaching language models to support answers with verified quotes](https://arxiv.org/abs/2203.11147) — Menick et al., 2022 | 根拠を提示し、不確実な場合にabstainするQAの設計 | Abstract、abstention分析 |
| 補助 | [RAFT: Adapting Language Model to Domain Specific RAG](https://arxiv.org/abs/2403.10131) — Zhang et al., 2024 | 専門ドメインで、正しい根拠とdistractorを見分けながら回答する課題 | Abstract、domain-specific evaluation |
| 補助 | [RepoCoder: Repository-Level Code Completion Through Iterative Retrieval and Generation](https://arxiv.org/abs/2303.12570) — Zhang et al., 2023 | コードリポジトリ内の分散した文脈を検索し、生成へ利用する代表例 | Abstract、iterative retrieval-generation概要 |

- **最初に適合条件を示す**
  - 上記の条件を5〜6項目のチェックリストとして置く。
  - 「文書が多いからRAG」ではなく、「外部根拠を検索して自然言語で統合する必要があるからRAG」と説明する。
- **社内文書・規程QA**
  - 正本: 規程、手順書、FAQ、社内Wiki、通達。
  - RAGの価値: 最新版と適用条件を検索し、該当箇所を引用する。
  - 注意: 部門・役職・地域ごとのACLと有効期限。
- **製品サポート・製品仕様QA**
  - 正本: マニュアル、リリースノート、既知問題、互換性表、サポート履歴。
  - RAGの価値: バージョンや型番を跨いで回避策を統合する。
  - 注意: 型番・エラーコードのkeyword retrievalと、意味検索の併用。
- **契約・規程レビュー支援**
  - 正本: 契約本文、条項集、社内基準、過去レビュー。
  - RAGの価値: 条項、例外、関連規程を並べ、根拠付きでレビューを補助する。
  - 注意: 最終法的判断は人間が行う。版、管轄、契約主体をmetadataで管理する。
- **障害対応支援**
  - 正本: runbook、postmortem、チケット、監視アラート説明、既知回避策。
  - RAGの価値: 症状から類似障害と手順を検索し、確認順を整理する。
  - 注意: 実環境の状態確認や操作はtool/API側へ分離する。
- **研究・技術文書QA**
  - 正本: 論文、技術報告、標準仕様、設計資料。
  - RAGの価値: 複数文献の比較、根拠箇所の提示、関連研究の整理。
  - 注意: 引用の主張支持、版、査読状況、撤回・訂正情報。
- **コードベースQA・開発支援**
  - 正本: source code、README、ADR、API仕様、issue、PR。
  - RAGの価値: 複数ファイルに分散した定義、利用箇所、設計意図を探索する。
  - 注意: branch、commit、generated code、dependency versionを追跡する。
- **ユースケース選定表を置く**
  - 列: ユースケース、正本、質問例、必要なretrieval、必要なcitation、主なリスク。
- **PoC対象の選び方を示す**
  - 1部門、1製品、1文書種別に限定する。
  - 正本と期待根拠を定義できる質問から始める。
  - 高頻度だが低〜中リスクの調査業務を優先する。
  - 似て見えても、答えがDB更新、計算、承認、最終判断にある場合はRAG単独に向かないと示す。
