- **問題を一つの図で俯瞰する**
  - 「LLM内部知識だけに依存する場合の問題」から「外部知識を検索することで改善できる点」への対応図を置く。
  - 各問題について、`症状 -> RAGによる改善 -> 残るリスク` の順で説明する。
- **知識の鮮度**
  - 症状: 変更後の規程、仕様、障害情報をモデルが知らない。
  - 改善: 外部知識源とindexを更新し、最新文書を検索対象にする。
  - 残るリスク: 同期遅延、旧版混入、有効期限metadataの欠落。
- **組織・ドメイン固有知識**
  - 症状: 社内Wiki、契約、顧客設定、コード、チケットが学習データにない。
  - 改善: private corpusを権限制御付きで検索対象にする。
  - 残るリスク: data sourceの権威性、ACL漏れ、重複・矛盾。
- **根拠追跡性とprovenance**
  - 症状: 正しそうでも、どの資料のどの箇所に基づくか確認できない。
  - 改善: document、page、section、chunk、versionを回答へ紐づける。
  - 残るリスク: citationが主張を実際には支えていない、引用範囲が粗い。
- **回答可能性と回答保留**
  - 症状: 根拠がなくても流暢な回答を生成する。
  - 改善: 検索結果の有無、根拠充足、矛盾を判定してrefusalへ分岐する。
  - 残るリスク: negative rejectionが不安定、閾値が業務ごとに異なる。
- **知識更新コスト**
  - 症状: 事実が変わるたびに再学習するのは遅く、検証も重い。
  - 改善: 文書とindexを更新し、モデル本体と知識ライフサイクルを分離する。
  - 残るリスク: parser、chunk、embedding変更時の再indexと回帰評価。
- **ロングテール知識**
  - 症状: 頻度の低い製品、例外規定、過去障害、顧客固有条件をモデルが安定して再現できない。
  - 改善: 必要時に該当文書を検索し、その場で回答へ使う。
  - 残るリスク: query表現と文書表現が離れている場合の検索漏れ。
- **RAGが解決しない問題を明記する**
  - 悪い原文、誤った表、古い版、権限設計不備はRAGで自動修正されない。
  - LLMが根拠を読み違える可能性も残る。
  - よって目的は「誤りをゼロにすること」ではなく「根拠を確認し、失敗を分解して改善できること」とする。
