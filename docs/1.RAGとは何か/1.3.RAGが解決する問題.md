# 1.3 RAGが解決する問題

## 1.3.1. 問題を一つの図で俯瞰する

章の冒頭では、問題を技術名の羅列ではなく対応関係として見せる。左側に「古い知識」「社内資料を知らない」「出典を示せない」「根拠がなくても答える」を置き、右側に外部index、ACL付きcorpus、provenance、abstentionを配置する。中央には、RAGで改善しても同期遅延や誤引用が残ることを記す。RGBがnoise robustnessやnegative rejectionを独立能力として測った点を使い、「導入すれば解消」ではなく「失敗を分解可能にする」図へ導く。

**参照論文**
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)

## 1.3.2. 知識の鮮度

規程や障害情報が改定された直後、モデルの内部知識を再学習するより、正本とindexを更新する方が速い。REALMは外部記憶をモジュール化し、RAGはその検索結果を生成へ接続した。ただし鮮度は文書の更新日だけでは決まらず、取得、解析、再index、公開までの遅延に分解される。本項では、最新版と質問時点で有効だった版を区別し、同期遅延、旧版混在、有効期間metadata欠損を残存リスクとして追跡する。 ここでいう「最新」は単なる更新日時ではなく、正本性と施行時点を満たすことだと定義する。

**参照論文**
- [P008] [REALM: Retrieval-Augmented Language Model Pre-Training](https://arxiv.org/abs/2002.08909)
- [P009] [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)

## 1.3.3. 組織・ドメイン固有知識

社内Wiki、契約、顧客設定、コードは、一般モデルの学習対象に含まれないか、含まれても利用権限を表現できない。KILTが多様なknowledge-intensive taskを共通知識源とprovenanceで扱ったように、RAGでは組織固有corpusを明示的な検索対象へする。価値は「非公開情報を読める」ことだけではなく、tenantやroleで候補を限定できる点にある。同時に、owner不明、重複、矛盾、ACL漏れはモデルで補正できないため、データ統治を前提に置く。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 1.3.4. 根拠追跡性とprovenance

もっともらしい回答を監査可能な回答へ変える鍵がprovenanceである。KILTはタスク正解だけでなく、根拠となるWikipedia pageを正しく特定できたかを評価対象にした。ALCEはさらに、長文回答の各claimをcitationが実際に支持するかを測る。本項ではdocument、page、section、chunk、versionを回答へ紐づけ、引用ラベルの存在ではなく、原文の該当箇所まで辿れることを要件化する。粗い引用や主張と無関係な出典は失敗として残す。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)

## 1.3.5. 回答可能性と回答保留

答えがない質問へ流暢に答えることは、正解問題での誤答とは別の故障である。GopherCiteは不確かな質問をabstainすると評価が向上することを示し、RGBはnegative rejectionを独立能力として扱った。そこで、検索結果なし、関連文書はあるが支持不足、根拠間の矛盾、権限不足を分け、それぞれ再検索、確認質問、併記、拒否へ結ぶ。閾値は万能ではないため、業務リスク別にfalse answerとfalse refusalの費用を測る。

**参照論文**
- [P076] [Teaching language models to support answers with verified quotes](https://arxiv.org/abs/2203.11147)
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)

## 1.3.6. 知識更新コスト

事実が変わるたびにモデルを再学習すると、データ作成、学習、回帰試験、配布が一体となり、変更単位が大きくなる。REALMが示した外部知識のモジュール化は、生成能力と知識ライフサイクルを分離する設計根拠になる。文書追加や失効はindex更新で扱い、モデル更新は表現・推論能力の改善へ限定できる。ただしparser、chunker、embeddingの変更では再indexと再評価が必要になるため、RAGを「更新無料」と誤解させず、費用の移動として説明する。

**参照論文**
- [P008] [REALM: Retrieval-Augmented Language Model Pre-Training](https://arxiv.org/abs/2002.08909)
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)

## 1.3.7. ロングテール知識

頻度の低い型番、例外条項、過去障害は、モデル規模を増やしても安定して再現されるとは限らない。DPRは質問とpassageの意味対応を学習することで語彙不一致を補ったが、希少な識別子ではBM25が優位な例も報告した。ロングテール知識には、必要時に該当資料を呼び出すRAGが適する一方、dense検索だけへ寄せないことが重要である。exact match、field検索、hybridを組み合わせ、検索漏れをquery slice別に測る流れを示す。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)

## 1.3.8. RAGが解決しない問題を明記する

RAGは誤った原文を真実へ変換しない。古い表、OCRの列崩れ、非公式メモ、権限設計不備が検索されれば、生成器はそれをもっともらしく統合し得る。RAGGEDは検索件数を増やしても、readerがnoiseに弱ければ性能が下がると示した。したがって目的は誤りをゼロにすることではなく、データ、検索、context、生成のどこで失敗したかを特定し、引用と保留で被害を抑えることだと締める。 原文品質を直すdata governanceと、根拠利用を直すRAG改善を分けることで、責任の所在も明確になる。

**参照論文**
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)

## 1.3.9. まとめ表を置く

まとめ表では、各問題を一行の効能へ縮めない。「知識の鮮度」なら外部index、freshness指標、同期遅延を並べ、「根拠追跡」ならcitation機構、support判定、誤引用を並べる。列を問題、追加する機構、観測指標、残存リスクとすることで、RAGが何を改善し、何を新たに運用対象へ持ち込むかが同時に見える。次のユースケース選定では、この表を要件チェックへ転用できる形にする。 効能だけでなく観測方法を並べるため、導入後に改善の有無を判定できる表になる。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)
