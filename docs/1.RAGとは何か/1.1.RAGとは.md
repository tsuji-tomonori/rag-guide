# 1.1 RAGとは

## 参考論文一覧

| 優先度 | 論文 | この節で使う論点 | 主に読む部分 |
|---|---|---|---|
| 必読 | [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401) — Lewis et al., 2020 | RAGという名称と、parametric memoryである生成モデルとnon-parametric memoryである外部indexを組み合わせる基本定義 | Abstract、Introduction、モデル概要 |
| 必読 | [REALM: Retrieval-Augmented Language Model Pre-Training](https://arxiv.org/abs/2002.08909) — Guu et al., 2020 | 外部retrieverを、更新可能でモジュール化された知識記憶として使う発想 | Abstract、Introduction |
| 推奨 | [Language Models as Knowledge Bases?](https://aclanthology.org/D19-1250/) — Petroni et al., 2019 | LLMのパラメータ内に事実知識が保持されるという、parametric memoryの背景 | Abstract、Introduction、結論 |
| 推奨 | [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997) — Gao et al., 2023 | RAGをretrieval・augmentation・generationからなるシステムとして整理するための全体地図 | taxonomy、Naive/Advanced/Modular RAGの整理 |
| 補助 | [Leveraging Passage Retrieval with Generative Models for Open Domain Question Answering](https://arxiv.org/abs/2007.01282) — Izacard & Grave, 2020/2021 | generatorが複数passageの根拠を統合できること。RAGを単一文書の貼り付けに限定しない説明 | Abstract、Introduction、multi-passage実験 |

- **導入: 一文定義を先に置く**
  - 「RAGは、質問に関係する外部知識を検索し、その検索結果を根拠としてLLMが回答を生成する方式」と定義する。
  - 最初の段落ではvector、embedding、rerankerなどの実装語を出しすぎない。
- **RAGの入力と出力を明確にする**
  - 入力はユーザー質問と検索対象の知識源。
  - 中間成果物は検索された根拠候補。
  - 出力は根拠に基づく回答、引用、または根拠不足時の回答保留。
- **parametric memoryとnon-parametric memoryを説明する**
  - parametric memory: 学習済みモデルのパラメータに保持された知識。
  - non-parametric memory: 文書indexや検索可能な外部知識源。
  - RAGでは両者を競合させるのではなく、役割分担させる。
- **最小パイプラインを示す**
  - `Question -> Retrieve -> Augment -> Generate -> Cite or Refuse`
- **「根拠に基づく」の意味を限定する**
  - 検索結果がプロンプトに含まれるだけではgroundingを保証しない。
  - 回答中の主張が根拠に支えられ、出典を確認できることまでを目標に含める。
- **RAGではないものを短く示す**
  - 単なる全文検索はRAGではない。
  - 長い文書を無選別にプロンプトへ詰めるだけでもない。
  - fine-tuningやtool実行と同義ではない。
- **RAGを単一処理ではなくシステムとして位置づける**
  - 実務ではdata source、検索、根拠整形、citation、評価、権限、運用までが設計対象になる。
  - この節では全体像だけを示し、具体的な構成は2章以降へ送る。
  - 同じ質問をLLM単体、検索システム、RAGがどう処理するかを比較すると予告する。
