# 1.1 RAGとは

## 1.1.1. 導入：一文定義を先に置く

読者が最初に掴むべき像は、RAGを「検索機能付きチャット」と呼ぶことではなく、回答前に外部知識を取得し、その内容を条件として生成する方式だと理解することである。LewisらのRAGは生成モデルのparametric memoryと検索可能なnon-parametric memoryを結び、REALMは外部記憶を更新可能な部品として扱った。ここではこの二点を一文に圧縮した後、検索語やベクトルの説明へ急がず、何を根拠に答えたかを後から確認できる設計だと示す。

**参照論文**
- [P009] [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)
- [P008] [REALM: Retrieval-Augmented Language Model Pre-Training](https://arxiv.org/abs/2002.08909)

## 1.1.2. RAGの入力と出力を明確にする

質問を受け取ってから回答を返すまでを、曖昧な「AI処理」で包まない。入力は利用者の質問、検索対象、権限、基準時点であり、中間成果物は出典を持つ根拠候補、出力は回答・引用・回答保留のいずれかになる。FiDが複数passageを統合できることを示した一方、引用付き生成研究は根拠が存在するだけでは足りないと教える。したがって本項では、各段階の受け渡しを契約として描き、どこで失敗したか追える形にする。 さらに、根拠不足を正常な出力として扱うことで、無理な生成を例外ではなく設計済みの分岐にできる。

**参照論文**
- [P006] [Leveraging Passage Retrieval with Generative Models for Open Domain Question Answering](https://arxiv.org/abs/2007.01282)
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)

## 1.1.3. parametric memoryとnon-parametric memoryを説明する

Petroniらは言語モデルのパラメータに事実知識が蓄えられることを示したが、その知識は更新時点も出典も直接管理できない。REALMとRAGは、文書indexを外部記憶として併置し、生成能力と知識の保管場所を分ける道を開いた。この対比は「内部知識を捨てて検索へ置き換える」という話ではない。文章化や一般化はparametric memory、鮮度・組織固有情報・証跡はnon-parametric memoryが担う、という役割分担として説明する。

**参照論文**
- [P005] [Language Models as Knowledge Bases?](https://arxiv.org/abs/1909.01066)
- [P008] [REALM: Retrieval-Augmented Language Model Pre-Training](https://arxiv.org/abs/2002.08909)
- [P009] [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)

## 1.1.4. 最小パイプラインを示す

最小構成は `Question → Retrieve → Augment → Generate → Cite or Refuse` と置く。ただし矢印を並べるだけでは、retrievalで正解根拠が取れなかった失敗と、generatorが根拠を誤読した失敗が同じに見える。RAG surveyの整理を借り、取得、根拠の選別・配置、生成を別工程として短く注釈する。最後に引用または保留を置くことで、流暢な文章を返すことより、確認可能な回答を返すことが終点だと読者に印象づける。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P009] [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)

## 1.1.5. 「根拠に基づく」の意味を限定する

「検索結果をpromptへ入れた」ことと「回答が根拠に基づく」ことは同義ではない。ALCEはcitation correctnessとcompletenessを分け、引用先が主張を本当に支えるか、根拠が必要な主張に引用が付くかを別々に測った。この知見を起点に、groundingを、主張と出典箇所を対応づけ、根拠外の補完を抑え、不足時は断定しない状態として定義する。見栄えのよい脚注だけを成功と数えないことが、本項の結論になる。

**参照論文**
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)

## 1.1.6. RAGではないものを短く示す

境界を明確にするため、似ているが別の仕組みを反例から示す。全文検索は文書を返すだけならRAGではなく、長い文書を無選別にcontextへ詰める方法も取得・選別の設計を欠く。fine-tuningは主に振る舞いや表現を変え、tool useは計算や業務操作を実行する。Lewisらの定義に戻れば、外部知識の取得と、その根拠に条件づけられた生成が核である。何をRAGと呼ばないかを先に区切ることで、後章の技術を混同せず読める。

**参照論文**
- [P009] [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)
- [P015] [Toolformer: Language Models Can Teach Themselves to Use Tools](https://arxiv.org/abs/2302.04761)

## 1.1.7. RAGを単一処理ではなくシステムとして位置づける

一つのretrieverとLLMを接続した時点を完成とはしない。GaoらのsurveyはRAGをretrieval・augmentation・generationの連鎖として整理し、Modular RAGはroutingやverificationまで再構成可能な部品として捉え直した。実務ではさらに、正本管理、解析、ACL、版、引用、評価、監視が回答品質を左右する。本項は、RAGを単一モデルではなく、知識を安全に更新し、検索し、検証可能な形で使うシステムとして位置づけ、2章以降の設計対象を予告する。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)
