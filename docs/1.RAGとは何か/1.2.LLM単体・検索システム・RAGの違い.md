# 1.2 LLM単体・検索システム・RAGの違い

## 参考論文一覧

- **比較のための共通質問を一つ置く**
  - 例: 「製品Xで障害Yが起きたときの最新版の回避策は何か。根拠も示してほしい」
  - 三方式が同じ質問へどう応答するかを先に示す。
- **LLM単体を説明する**
  - 強み: 自然な文章、要約、一般化、対話、形式変換。
  - 弱み: 学習時点以降の更新、非公開知識、正確な出典、権限制御。
  - 典型的失敗: 流暢だが根拠を確認できない回答、古い仕様の混入。
- **検索システム単体を説明する**
  - 強み: 関連文書の提示、原文確認、更新反映、フィルタ、権限連携。
  - 弱み: 複数文書の統合、要約、比較、対話的な説明は別実装が必要。
  - sparse検索とdense検索の違いは概略だけ触れ、4章へ送る。
- **RAGを説明する**
  - 検索が候補根拠を取り出し、LLMが質問に合わせて統合・説明する。
  - 回答と引用を対応づけ、根拠不足時には回答を保留する設計が可能。
  - 検索と生成の両方の失敗モードを持つことも明記する。
- **三者比較表を置く**
  - 比較軸: 主な出力、知識の鮮度、社内知識、根拠提示、文章統合、更新方法、権限制御、代表的失敗。
  - 「どれが常に優れているか」ではなく、目的別の選択として示す。
- **fine-tuning、long-context、tool useとの違いを予告する**
  - fine-tuning: 振る舞いや形式を学習させる手段。頻繁に変わる事実の正本管理とは別。
  - long-context: 与えられた長文を読む手段。どの文書を選ぶか、更新、権限、コストは別問題。
  - tool use: 検索だけでなく計算や業務操作を実行する仕組み。
  - 三者比較から、RAGが具体的にどの業務問題を改善するのかへ進む。
