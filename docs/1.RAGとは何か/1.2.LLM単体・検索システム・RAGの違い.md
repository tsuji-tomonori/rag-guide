# 1.2 LLM単体・検索システム・RAGの違い

LLM単体、検索システム、RAGは、同じ質問へ答えるために利用できますが、参照する知識と返す出力が異なります。
三つの方式を優劣だけで比べるのではなく、必要な出力と失敗時の調査方法から選ぶ必要があります。

## 1.2.1 共通の質問による比較

ここでは、三つの方式へ次の質問を入力する場面を考えます。

> 製品Xのバージョン4.2でエラーE101が発生しました。
> 最新版の回避策と、その根拠を教えてください。

この質問には、自然な説明だけでなく、製品、バージョン、エラーコード、情報の鮮度、出典という条件があります。
条件を一つに固定すると、各方式が利用する情報源と、誤答時に点検できる箇所の違いを明確にできます。

LLM単体は、学習済み知識からE101に関係しそうな説明を生成します。
検索システムは、E101やバージョン4.2に一致する文書と原文の箇所を返します。
RAGは、検索で得た文書を使って質問に合わせた回答を生成し、参照元を提示します。

## 1.2.2 LLM単体

LLM単体は、質問の意図を読み取り、自然な説明、要約、言い換え、形式変換を行うことに適しています。
[外部文書を使わない質問応答（closed-book QA）の研究](https://arxiv.org/abs/2002.08910)では、モデルの規模が大きくなるにつれて多くの質問へ答えられることが示されました。
この結果は、LLMが学習済みパラメーター内に一定の事実知識を保持することを示します。

一方で、学習後に公開されたバージョン4.2の回避策や、非公開の社内情報は、モデル内部に存在しない可能性があります。
内部知識の出典、版、更新日時を質問ごとに指定することもできません。
モデルがE101について流暢な説明を返しても、その説明が製品Xのバージョン4.2へ適用できるかは別に確認する必要があります。

LLM単体が不適切なのは、すべての質問ではありません。
一般的な概念の説明、文章の校正、入力済み文章の要約など、外部の最新事実を必要としない処理では有効です。
判断基準は文章が生成できるかではなく、回答に必要な知識をモデル内部だけで管理できるかです。

## 1.2.3 検索システム

検索システムは、質問に関係する文書や文書内の箇所を順位付けして返します。
利用者は原文を読み、版や適用条件を直接確認できます。
インデックスを更新すれば、新しい文書をモデルの再学習なしで検索対象へ追加できます。

キーワード検索で広く使われるBM25は、検索語の出現頻度、語の希少性、文書長などを用いて関連度を計算します。
[BM25の確率的検索モデル](https://doi.org/10.1561/1500000019)は、エラーコード、型番、条文番号のように文字列そのものが重要な検索で有効です。
文書の版や製品名などの付加情報（メタデータ）による絞り込みを組み合わせれば、製品X、バージョン4.2、公開中の資料だけに範囲を限定できます。

検索システムは、通常、複数文書の内容を一つの回答へ統合しません。
回避策がマニュアル、リリースノート、既知問題へ分散している場合、利用者が各文書を開いて関係を判断します。
原文を利用者自身が読めることは欠点ではなく、生成結果が疑わしい場合にも検証手段を残す性質です。

## 1.2.4 RAG

RAGは、検索システムが取得した根拠候補をLLMへ渡し、質問に合わせた回答へ統合します。
先ほどの質問では、バージョン4.2に適用される回避策を複数の資料から選び、実施条件と参照元を一つの回答にまとめられます。
利用者は生成された説明から原文へ戻り、手順や例外を確認できます。

[Dense Passage Retrieval（DPR）](https://arxiv.org/abs/2004.04906)は、質問と文書をベクトルへ変換し、語が一致しなくても意味の近い箇所を検索する方法を示しました。
[FiD](https://arxiv.org/abs/2007.01282)は、複数の取得文書を生成器で統合する能力を高めました。
RAGでは、キーワード検索と意味検索を組み合わせ、取得した複数の根拠をLLMが説明へ変換できます。

ただし、RAGは検索と生成の両方の失敗を引き受けます。
正しい回避策が検索候補へ入らなければ、生成器はその情報を利用できません。
正しい資料が候補へ入っても、LLMが別バージョンの条件を混ぜたり、否定を読み違えたりする可能性があります。
そのため、検索結果と生成結果を別々に記録し、引用の正しさと回答保留を評価します。

## 1.2.5 三つの方式の比較

表1-1は、同じ比較軸について三つの方式を横に並べています。
一つの行を左から右へ読むと、出力、知識の更新方法、失敗時の修正箇所が方式ごとにどう違うかを比較できます。

**表1-1　LLM単体、検索システム、RAGの比較**

| 比較軸 | LLM単体 | 検索システム | RAG |
|---|---|---|---|
| 主な出力 | 生成された文章 | 文書または該当箇所の一覧 | 根拠を統合した文章と参照元 |
| 利用する知識 | 学習済みパラメーター | 外部の検索インデックス | 学習済みパラメーターと外部知識 |
| 知識の更新 | モデル更新や再学習に依存します | 文書とインデックスを更新します | 文書とインデックスを更新します |
| 非公開情報 | 学習していなければ利用できません | 権限内の文書を検索できます | 権限内の文書を検索し、回答へ利用できます |
| 原文の確認 | 回答だけでは困難です | 検索結果から確認できます | 引用設計があれば確認できます |
| 複数資料の統合 | 資料を入力すれば可能です | 通常は利用者が行います | 取得した根拠をLLMが統合します |
| 代表的な失敗 | 古い知識、根拠のない補完 | 検索漏れ、順位不良 | 検索漏れ、根拠の誤読、誤引用 |
| 主な修正箇所 | プロンプトまたはモデル | 文書、質問、検索設定 | 文書、検索、文脈、生成の各工程 |

[KILT](https://arxiv.org/abs/2009.02252)は、回答の正しさと、回答に必要な根拠を取得できたかを分けて評価しました。
同じ考え方を使うと、文章が自然かという軸だけで三方式を比較せず、出典、更新、権限、失敗時の追跡可能性を含めて選べます。

検索結果そのものが成果物であり、利用者が原文を読む業務では、検索システムが適しています。
外部事実を必要とせず、与えた文章の変換が目的であれば、LLM単体で十分な場合があります。
外部根拠を探し、複数資料を統合した説明と参照元を返す必要がある場合は、RAGが候補です。

## 1.2.6 ファインチューニング、長いコンテキスト、ツール利用

RAGの近隣技術は、何を変更する手段かで区別できます。

- **ファインチューニング**は、モデルの振る舞い、出力形式、専門語の使い方などを学習させます。
- **長いコンテキスト**は、一回の入力でLLMが読める情報量を増やします。
- **ツール利用**は、計算、検索API、データベース照会、業務操作などを外部システムへ実行させます。
- **RAG**は、質問に必要な外部知識を選び、その内容を回答生成へ利用します。

長いコンテキストへすべての資料を入れられる場合でも、情報を正しく利用できるとは限りません。
また、文書選択、アクセス権、更新、入力費用は、コンテキスト長とは別の設計課題です。
長いコンテキストとRAGの比較は、1.5.7で詳しく扱います。

ツール利用は、文書に書かれた説明ではなく、現在値や計算結果が正解となる質問に適しています。
例えば、現在の在庫数は在庫APIから取得し、在庫の数え方は規程文書をRAGで検索できます。
これらの技術は排他的な選択肢ではなく、正解の所在と処理の責任に応じて組み合わせます。
