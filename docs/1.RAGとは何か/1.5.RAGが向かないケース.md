# 1.5 RAGが向かないケース

## 参考論文一覧

| 優先度 | 論文 | この節で使う論点 | 主に読む部分 |
|---|---|---|---|
| 必読 | [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172) — Liu et al., 2023 | 長いcontextへ全文を入れれば必ず解けるわけではなく、根拠位置とノイズに依存すること | Abstract、position experiments、結論 |
| 必読 | [Retrieval Augmented Generation or Long-Context LLMs? A Comprehensive Study and Hybrid Approach](https://arxiv.org/abs/2407.16833) — Li et al., 2024 | RAGとlong-contextの品質・コストの比較、およびquery routingという選択肢 | Abstract、比較結果、Self-Route |
| 推奨 | [Toolformer: Language Models Can Teach Themselves to Use Tools](https://arxiv.org/abs/2302.04761) — Schick et al., 2023 | 計算、検索、カレンダー等は、モデルに記憶させるより外部toolを呼ぶべき場合があること | Abstract、tool examples |
| 推奨 | [ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629) — Yao et al., 2022/2023 | 情報取得や業務操作を含む課題では、reasoningとactionを分離する必要 | Abstract、QAとinteractive taskの比較 |
| 推奨 | [RAT-SQL: Relation-Aware Schema Encoding and Linking for Text-to-SQL Parsers](https://arxiv.org/abs/1911.04942) — Wang et al., 2019/2020 | 数値集計や最新状態を文書chunkから推測せず、構造化DBをqueryする設計 | Abstract、schema linking |
| 補助 | [DIN-SQL: Decomposed In-Context Learning of Text-to-SQL with Self-Correction](https://arxiv.org/abs/2304.11015) — Pourreza & Rafiei, 2023 | 複雑なDB質問を分解し、SQLで正本へ問い合わせる方式 | Abstract、decomposition、self-correction |
| 補助 | [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445) — Karpas et al., 2022 | 検索、計算、業務処理を専門moduleへroutingする考え方 | Abstract、architecture proposal |

- **「正解はどこにあるか」から判断する原則を置く**
  - 文書内にある: RAG候補。
  - DB・APIの現在状態にある: structured queryやtool候補。
  - 計算で求める: calculator、code execution、rule engine候補。
  - 承認・責任判断にある: workflow、人間レビュー候補。
  - モデルの振る舞い・文体にある: prompt設計やfine-tuning候補。
- **状態変更・トランザクションを主目的とするケース**
  - 例: 決済、在庫更新、権限変更、設定変更、チケット承認。
  - RAGは手順や根拠を説明できるが、実行は認証・認可・冪等性・監査を持つAPI/workflowへ委ねる。
  - 書き込み操作ではhuman approvalやpolicy enforcementを入れる。
- **厳密な数値計算・集計**
  - 例: 売上合計、最新在庫、料金計算、SLA達成率、日付計算。
  - 文書chunkの類似検索とLLMの暗算に依存しない。
  - DB、SQL、calculator、分析基盤を正本とし、RAGは定義や説明の補助に使う。
- **リアルタイム状態の問い合わせ**
  - 例: 現在の注文状態、監視値、アカウント残高、稼働中version。
  - static indexでは更新遅延が生じるため、APIやDBを直接参照する。
  - RAGは関連手順、既知問題、用語説明を付加する役割に限定できる。
- **高リスクな最終判断**
  - 例: 医療診断、法的結論、信用判断、人事処分、安全制御。
  - RAGは資料検索と論点整理に使えても、最終決定者にはしない。
  - authority、説明責任、異議申立て、監査、専門家レビューを要件化する。
- **信頼できる外部資料が存在しないケース**
  - 正本がない、文書が古い、所有者不明、矛盾が放置されている場合。
  - RAG導入より先にknowledge governanceを整備する。
- **小さく固定された文書を一度だけ読むケース**
  - 対象文書が少なくcontext windowへ安全に収まり、更新や権限分離が不要ならlong-contextで十分な可能性がある。
  - ただし根拠位置、コスト、再現性は評価する。
- **知識ではなく振る舞いを変えたいケース**
  - 出力形式、用語、文体、分類規則などはpromptやfine-tuningが主手段になる場合がある。
  - 頻繁に変わる事実をfine-tuningだけで管理しない。
- **RAGと代替手段の判断表を置く**
  - 列: 問題、正解の所在、第一候補、RAGの役割、必要な安全機構。
- **複合構成を説明する**
  - `RAGで手順を検索 -> APIで状態確認 -> ruleで判定 -> 人間が承認` のように、RAGを一部品として使う。
  - RAGが、closed-book QA、外部検索、根拠付き生成、長文context、適応的検索という研究課題からどのように発展したかを予告する。
