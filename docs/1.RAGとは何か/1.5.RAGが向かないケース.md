# 1.5 RAGが向かないケース

## 1.5.1. 「正解はどこにあるか」から判断する原則を置く

採用判断の出発点を「RAGでできるか」ではなく「正解がどこに存在するか」に置く。文書の記述にあるなら検索候補、DBやAPIの現在値ならstructured query、計算で得るならcodeやrule、承認にあるならworkflowと人間判断が第一候補になる。MRKLは言語モデルを専門moduleへroutingするsystems approachを提示した。この考えを用い、RAGを万能入口にせず、正本と責任主体に合わせて経路を選ぶ。

**参照論文**
- [P072] [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445)

## 1.5.2. 状態変更・トランザクションを主目的とするケース

決済、在庫更新、権限変更は、正しい説明を返すだけでは完了しない。認証、認可、冪等性、監査、rollbackを備えたAPIが状態を変える必要がある。ReActはreasoningとactionを交互に進める構成を示したが、これはLLMの提案を無条件に実行してよいという意味ではない。RAGは手順やpolicyを検索する補助にとどめ、write操作はserver-side policyと承認を通す。文書内の命令から操作へ直結させない境界を明記する。

**参照論文**
- [P012] [ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)
- [P015] [Toolformer: Language Models Can Teach Themselves to Use Tools](https://arxiv.org/abs/2302.04761)

## 1.5.3. 厳密な数値計算・集計

売上合計や料金計算を、類似chunkとLLMの暗算へ任せると、行の欠落、単位違い、更新時点のずれを検証しにくい。RAT-SQLはschemaと関係を明示的に扱い、DIN-SQLは複雑な質問を分解してSQL生成を改善した。したがって数値の正本はDB、semantic layer、calculatorに置き、RAGは項目定義や計算規則を説明する役割へ限定する。生成SQL、filter、実行時刻、結果行を証跡として残す構成を予告する。

**参照論文**
- [P069] [RAT-SQL: Relation-Aware Schema Encoding and Linking for Text-to-SQL Parsers](https://arxiv.org/abs/1911.04942)
- [P064] [DIN-SQL: Decomposed In-Context Learning of Text-to-SQL with Self-Correction](https://arxiv.org/abs/2304.11015)

## 1.5.4. リアルタイム状態の問い合わせ

「いまの在庫」「現在の注文状態」は、昨日作ったindexが正しくても答えられない。static corpusではsource更新から検索可能になるまで必ず遅延があるため、APIやDBをquery時に参照する。RAGは関連マニュアルや障害情報を添える補助線として使えるが、最新値そのものを文書snapshotから推測しない。回答には取得時刻とdata freshnessを明示し、API停止時は古い値で埋めず、利用不能として返す。

**参照論文**
- [P072] [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445)
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)

## 1.5.5. 高リスクな最終判断

医療診断、法的結論、信用判断、人事処分では、根拠が見つかったことと決定権を持つことを分けなければならない。引用付き回答は専門家の確認を速めるが、GopherCiteのTruthfulQA分析が示す通り、証拠で支持された主張が必ず真とは限らない。RAGの役割を資料探索と論点整理へ限定し、authority、説明責任、異議申立て、人間reviewを要件にする。高リスクの最終判断は自動確定しない。 専門家へ渡す画面には結論だけでなく、反対根拠と未確認事項も同時に表示する。

**参照論文**
- [P076] [Teaching language models to support answers with verified quotes](https://arxiv.org/abs/2203.11147)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 1.5.6. 信頼できる外部資料が存在しないケース

正本がなく、ownerも版も不明なcorpusへRAGを載せると、検索の速さだけが上がり、矛盾は隠れやすくなる。Datasheets for Datasetsが収集目的、構成、保守、制約の文書化を求めたように、まず資料の責任主体と採否基準を整えるべきである。retrieverやpromptを調整しても、誤った原文は修復できない。owner不明資料は隔離し、正式化・廃止・期限付き利用の判断を先に行う。 検索基盤より先に資料統治を整える判断も、RAG採用可否の重要な結論として扱う。

**参照論文**
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)
- [P091] [Everyone wants to do the model work, not the data work: Data Cascades in High-Stakes AI](https://dl.acm.org/doi/10.1145/3411764.3445518)

## 1.5.7. 小さく固定された文書を一度だけ読むケース

数ページの固定文書を一度読むだけなら、検索indexを構築する利益より運用負担が大きい場合がある。Long-contextとRAGの比較研究では、十分な資源があればlong-contextが平均性能で優位な一方、RAGはcostで有利だった。対象がcontext windowへ安全に収まり、更新、権限分離、再利用が不要なら全文読解をbaselineにする。ただし重要情報の位置による見落とし、token費用、引用精度は実測し、二択でなくhybridも残す。

**参照論文**
- [P075] [Retrieval Augmented Generation or Long-Context LLMs? A Comprehensive Study and Hybrid Approach](https://arxiv.org/abs/2407.16833)
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 1.5.8. 知識ではなく振る舞いを変えたいケース

出力をJSONに揃えたい、社内用語を使わせたい、分類規則を安定させたい――これらは知識の所在より振る舞いの問題である。prompt、schema validation、必要ならfine-tuningが主手段になる。RAGを加えても、検索文書が形式遵守を教えるとは限らず、逆にnoiseを増やす。頻繁に変わる事実は外部知識へ、繰り返し求める表現・手順はモデルやpromptへ、という変更頻度と責務の違いで選ぶ。 事実更新と応答作法を別の変更経路へ置けば、fine-tuningのたびに知識まで再検証する混乱を避けられる。

**参照論文**
- [P066] [RAFT: Adapting Language Model to Domain Specific RAG](https://arxiv.org/abs/2403.10131)
- [P009] [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)

## 1.5.9. RAGと代替手段の判断表を置く

判断表は問題の表面ではなく、正解の所在を軸にする。行に「最新値」「計算」「文書根拠」「行動」「振る舞い」「最終判断」を置き、第一候補をAPI、calculator、RAG、workflow、fine-tuning、人間reviewへ振り分ける。さらにRAGが補助として担える説明や引用と、必要な安全機構を併記する。MRKLのmodule routingを実務向けに翻訳し、単一技術を選ぶ表ではなく、複合構成を組むための設計表にする。

**参照論文**
- [P072] [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445)

## 1.5.10. 複合構成を説明する

現実の業務は一方式で閉じない。たとえばRAGで手順を探し、APIで現在状態を取得し、rule engineで条件を判定し、人が承認してから更新する。この順序なら、文書知識、事実確認、決定、実行の責任が分かれる。ReActやMRKLは複数moduleを連携させる発想を与えるが、各境界にはtimeout、権限、監査、停止条件が必要である。本項は複合構成を高度化ではなく、正解の所在に忠実な基本設計として示す。 各moduleの出力を根拠として次へ渡すことで、複合処理でも一つ一つの判断を監査できる。

**参照論文**
- [P012] [ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)
- [P072] [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445)
