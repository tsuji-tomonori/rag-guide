# 2.2 Retrieval

## 2.2.1. Retrievalの定義と成功条件

Retrievalの使命は回答を完成させることではなく、必要な根拠をtop-k候補へ入れることである。DPRの評価がTop-20やTop-100の回収率を重視したのは、readerが見ていない根拠を生成段階で救えないためだ。入力は正規化query、identity、時点、対象corpus、出力はrank、score、metadata、取得経路を持つcandidate setとする。precisionの最終調整は後段へ任せ、ここではall-evidence recallと権限境界を成功条件に置く。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)

## 2.2.2. Query理解と正規化

会話の「それ」や社内略語をそのまま検索すると、retrieverへ必要な条件が届かない。Conversational query研究は、履歴全体を連結するより、参照先と制約を補ってstandalone queryへ解決する方法を検討してきた。Unicode、表記ゆれ、製品名、期間を整える一方、型番や条文番号は破壊的に修正しない。元query、補完した情報、参照turnをtraceへ残し、書き換えが意味を変えた失敗を独立評価する。

**参照論文**
- [P114] [Query Resolution for Conversational Search with Limited Supervision](https://arxiv.org/abs/2005.11723)
- [P063] [Few-Shot Conversational Dense Retrieval](https://arxiv.org/abs/2105.04166)

## 2.2.3. Sparse retrieval

Sparse retrievalは、語彙が一致したという観測可能な根拠で候補を集める。BM25は希少語をIDFで重くし、term frequencyの飽和と文書長を補正するため、型番、エラーコード、条文番号、固有名詞に強い。反面、表現が違えば意味が同じでも取り逃がす。exact match、phrase、fielded searchをbaselineに置き、tokenizerと辞書のversionを管理することで、dense検索だけでは見えにくい検索漏れを補完する。

**参照論文**
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)

## 2.2.4. Dense retrieval

Dense retrievalはqueryとchunkを学習済み空間へ写し、単語が重ならなくても意味の近い候補を探す。DPRは単純なdual-encoderでもin-batch negativeとBM25由来hard negativeにより、多くのopen-domain QAでBM25を上回った。ただし希少識別子、数値、否定は苦手で、ANNは近似によるrecall低下も持つ。model、言語、domain、metric、filter selectivityを一体で評価し、similarity scoreを正答確率と解釈しない。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P026] [Approximate Nearest Neighbor Negative Contrastive Learning for Dense Text Retrieval](https://arxiv.org/abs/2007.00808)

## 2.2.5. Hybrid retrievalとfusion

Hybrid retrievalは二つのscoreを雑に足す処理ではない。BM25のexactnessとdenseのsemantic coverageを別々に取得し、chunk identityを揃えてから統合する。RRFは各listの順位だけを使うため、BM25 scoreとcosineの尺度差へ依存しにくい。一方、queryやcorpusによって単独方式が勝る場合もある。sparse、dense、hybridを同じgold setで比較し、どのretrieverが候補を運んだかをprovenanceとして残す。

**参照論文**
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P027] [BEIR: A Heterogeneous Benchmark for Zero-shot Evaluation of Information Retrieval Models](https://arxiv.org/abs/2104.08663)

## 2.2.6. Query expansion、decomposition、routing

初回queryで根拠が足りない場合だけ、検索表現や検索先を広げる。HyDEは仮想文書のembeddingでzero-shot dense retrievalを改善し得るが、生成した固有名詞に引かれる危険がある。multi-query、decomposition、文書・SQL・graph・imageへのroutingを、同じ「query変換」で一括りにせず目的別に分ける。追加LLM callのcostとlatency、意味変質を測り、最大query数、検索回数、停止条件を検索計画へ含める。

**参照論文**
- [P030] [Precise Zero-Shot Dense Retrieval without Relevance Labels](https://arxiv.org/abs/2212.10496)
- [P110] [Interleaving Retrieval with Chain-of-Thought Reasoning for Knowledge-Intensive Multi-Step Questions](https://arxiv.org/abs/2212.10509)
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)

## 2.2.7. Metadata / ACL filtering

権限filterはretrieval後の表示処理ではなく、候補集合を作る条件である。tenant、group、version、effective date、source typeをqueryとともに渡し、権限外chunkをpromptやtraceへ入れない。pre-filterは安全だが狭い集合でANN recallが落ちる場合があり、post-filterはtop-k不足と漏えい面を持つ。ACORNのようなpredicate-aware searchを含め、selectivity別にrecallとlatencyを測る。filter式とpolicy versionもtraceへ残す。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)
