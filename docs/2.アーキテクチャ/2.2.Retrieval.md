# 2.2 Retrieval

## この節の役割

- ユーザー質問に対して、回答に必要な根拠候補を取得する工程を定義する。
- sparse、dense、hybrid、query処理、routing、filteringの位置関係を示す。
- Retrievalの目的を「最終回答を作ること」ではなく、「必要な根拠を候補集合へ入れること」とする。

## 2.2.1. Retrievalの定義と成功条件

- 入力を正規化済みquery、ユーザー権限、検索対象、時点条件とする。
- 出力をrank付きの候補chunkと、そのscore、metadata、取得経路とする。
- 成功条件を「正解根拠がtop-k候補へ入ること」とする。
- precisionの最終調整はPost-retrievalが担うことを明示する。

## 2.2.2. Query理解と正規化

- 会話中の省略を補うstandalone query化を説明する。
- 略語、表記ゆれ、誤字、製品名、期間、対象部門を正規化する。
- 元queryを消さず、rewrite前後をtraceへ残す。
- 書き換えが意味を変える失敗を、評価対象として挙げる。

## 2.2.3. Sparse retrieval

- exact match、BM25、fielded searchを位置付ける。
- 型番、エラーコード、条文番号、固有名詞に強いことを示す。
- synonymや表記ゆれだけでは意味類似を十分に扱えない限界を説明する。

## 2.2.4. Dense retrieval

- queryとchunkを同じvector空間で比較する基本形を示す。
- 同義語や自然言語の言い換えに強い一方、exact identifierを落とす場合があると説明する。
- embedding model、言語、domain、distance metricが結果へ影響する。
- ANNによる近似検索では、速度とrecallのトレードオフがある。

## 2.2.5. Hybrid retrievalとfusion

- sparseとdenseを並列実行し、候補を統合する流れを示す。
- score normalization、weighted fusion、RRFを区別する。
- 同一chunkの重複を統合し、どのretrieverが取得したかを残す。
- hybridは常に優れるという断定を避け、業務queryで評価する。

## 2.2.6. Query expansion、decomposition、routing

- multi-query、HyDE、Query2docをquery expansionの例として示す。
- 複雑な質問ではsub-questionへ分解する。
- 文書、SQL、Graph、API、画像など、検索対象をroutingする。
- 追加のLLM呼び出しによるlatency、cost、誤書き換えをトレードオフとして示す。

## 2.2.7. Metadata / ACL filtering

- tenant、group、version、effective date、source typeで候補を絞る。
- 権限filterは生成後ではなく、検索前または検索時に適用する。
- pre-filterとpost-filterでrecallやlatencyが変わることを示す。
- filter条件そのものをtraceへ残す。
