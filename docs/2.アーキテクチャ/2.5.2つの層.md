# 2.5 2つの層

## 2.5.1. なぜ層を分けるのか

OfflineとOnlineを分ける理由は、batchという製品名を付けるためではない。質問前に確定できる解析・embeddingと、identityや会話に依存するquery時判断では、再現性、latency、cost、権限の責務が異なる。Dataflow Modelがcorrectness、latency、costを分けてデータ処理を捉えたように、層分けを障害切り分け、team ownership、release、監査の境界として使う。処理配置を変えられる設計余地も残す。

**参照論文**
- [P014] [The Dataflow Model](https://doi.org/10.14778/2824032.2824076)

## 2.5.2. Offline Layer

Offline Layerはsource documentと変更eventを受け、parse、normalize、chunk、embed、index build、quality checkを行う。成果物はversion付きindex、chunk manifest、ingestion reportであり、単なる保存先ではない。SLOにはfreshness、削除反映、再構築時間、失敗率を置く。EnrichIndexのようにLLM reasoningを取り込み時へ移す方法はonline costを下げ得るが、誤りをindexへ固定し、corpus更新時の再計算を増やす点も合わせて示す。

**参照論文**
- [P081] [EnrichIndex: Using LLMs to Enrich Retrieval Indices Offline](https://arxiv.org/abs/2504.03598)

## 2.5.3. Online Layer

Online Layerではuser query、identity、session、request policyを受け、認可、query理解、検索、rerank、packing、生成、citationを期限内に実行する。ここで使う情報は質問ごとに変わるため、offlineへ前倒ししすぎられない。出力はanswerまたはrefusal、citations、trace IDとし、p50/p95/p99 latency、availability、cost/query、filter correctnessをSLOへ置く。最新API値や会話文脈もこの層で取得する。

**参照論文**
- [P014] [The Dataflow Model](https://doi.org/10.14778/2824032.2824076)
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)

## 2.5.4. 2層と4工程の対応

四工程との対応は厳密な一対一ではない。Pre-retrievalは主にOffline、Retrieval・Post-retrieval・Generationは主にOnlineへ置くが、evaluation、version、policyは両者を横断する。query-independentなsummaryやentity linkは事前計算でき、query-dependentなdecompositionやverificationはonlineに残る。Huangらの四分類と工学的な二層を重ね、分類名より処理の依存情報を見て配置する原則を示す。

**参照論文**
- [P077] [A Survey on Retrieval-Augmented Text Generation for Large Language Models](https://arxiv.org/abs/2404.10981)

## 2.5.5. Offlineへ寄せる設計

online latencyが厳しいとき、文書summary、entity、synthetic query、bridge factをOfflineへ寄せる選択がある。IndexRAGは共有entityからbridging factをindex時に作り、multi-hop QAを単発検索へ近づけた。EnrichIndexも暗黙の関連性をofflineに明示化し、online tokenを大幅に削減した。一方、生成誤りの固定化、storage増、更新時再計算が生じる。効果をquery latencyだけでなくindex freshnessと再構築費用で評価する。

**参照論文**
- [P082] [IndexRAG: Bridging Facts for Cross-Document Reasoning at Index Time](https://arxiv.org/abs/2603.16415)
- [P081] [EnrichIndex: Using LLMs to Enrich Retrieval Indices Offline](https://arxiv.org/abs/2504.03598)

## 2.5.6. Onlineへ残す設計

identity、現在時刻、会話文脈、request-specific filterは質問が来るまで決まらないためOnlineに残す。Adaptive-RAGのような複雑度判定や、根拠不足時の追加検索もquery依存である。online処理を増やすほど品質を調整しやすい反面、latency、cost、非決定性が増える。何を残すかは機能一覧でなく、事前計算すると意味や権限が古くならないかで判断し、deadlineとfallbackを契約にする。

**参照論文**
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)

## 2.5.7. 層間の契約

二層の間には暗黙の共有DBではなく、明示的な契約を置く。Offlineはindex schema、metadata schema、version、freshness、ACL表現を公開し、Onlineは使用version、filter、query transformationをtraceへ残す。schema変更時にはblue/greenで切り替え、互換性がなければ旧routeへrollbackする。権限外本文はcontextだけでなくdebug logにも流さない。層間契約があることで、同じqueryを異なるindexへ再現評価できる。

**参照論文**
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)

## 2.5.8. 分類上の注意

Offline/OnlineはRAG研究で唯一の標準taxonomyではないと明記する。BatchとOffline、Real-timeとOnlineを完全な同義語にすると、incremental ingestionやstreaming updateの居場所がなくなる。Dataflow Modelがevent timeやtriggerを通じてbatchとstreamを統一的に捉えたように、中間形を認める。本書では責務、SLO、release境界を説明する便宜的な層分けとして使い、製品の名称へ設計を従属させない。

**参照論文**
- [P014] [The Dataflow Model](https://doi.org/10.14778/2824032.2824076)
