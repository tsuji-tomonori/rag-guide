# 2.1 Pre-retrieval

## 2.1.1. Pre-retrievalの定義

Pre-retrievalは、質問が来る前に原文を検索可能な知識へ変える工程である。HuangらのsurveyがRAGをpre-retrieval、retrieval、post-retrieval、generationへ分けた理由は、上流の欠損を後段のLLMが原則として回復できないからだ。入力には原文書、source metadata、ACL、更新・削除eventを置き、出力にはversion付きchunk、embedding、sparse/vector index、ingestion reportを置く。成功を「job終了」ではなく、根拠が再現可能に検索・引用できる状態と定義する。

**参照論文**
- [P077] [A Survey on Retrieval-Augmented Text Generation for Large Language Models](https://arxiv.org/abs/2404.10981)

## 2.1.2. データソースと正本

検索対象を増やす前に、どの資料を正本と呼ぶかを決める。Datasheetsの発想を取り入れ、owner、更新頻度、有効期間、利用制限、既知の欠陥をsource単位で記録する。正式規程、参考資料、下書き、失効文書が同列にindex化されれば、関連度の高い誤根拠が生まれる。したがって同内容の重複、版違い、矛盾を取り込み前に可視化し、「大量に入った」ことを「知識が整った」ことと混同しない。 正本の優先順位は検索後にLLMへ判断させず、source metadataと運用policyとして事前に確定する。

**参照論文**
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)
- [P091] [Everyone wants to do the model work, not the data work: Data Cascades in High-Stakes AI](https://dl.acm.org/doi/10.1145/3411764.3445518)

## 2.1.3. 文書解析と正規化

PDFを文字列へ変換しただけでは、列、見出し、表、図と本文の関係が消える。LayoutLMv3はtextとimage patchを統一的にmaskし、layoutを含む表現が文書理解の多様な課題に効くことを示した。解析ではpage、bbox、reading order、table cell、caption、code blockを保持し、headerやnavigationの反復noiseを除く。parser versionとconfidenceを残し、誤解析をretrieval failureと切り分けて再処理できる形にする。

**参照論文**
- [P083] [LayoutLMv3: Pre-training for Document AI with Unified Text and Image Masking](https://arxiv.org/abs/2204.08387)

## 2.1.4. Chunkingと検索単位

chunkingはtoken数をそろえる作業ではなく、何を一件の検索候補として比較するかを決める。Dense X Retrievalはdocument、passage、sentence、propositionの選択がretrievalとQAを大きく変え、細粒度のpropositionが計算予算下で有利な場面を示した。一方、細かすぎる単位は条件や主語を失う。検索単位、LLMへ渡す単位、citation単位を分け、fixed、見出し、semantic、parent-child、table-awareを文書種別ごとに評価する。

**参照論文**
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)
- [P025] [TextTiling: Segmenting Text into Multi-paragraph Subtopic Passages](https://aclanthology.org/J97-1003/)

## 2.1.5. MetadataとACL

embeddingは意味の近さを表すが、誰が閲覧できるか、いつ有効か、どの版かを表さない。その不足を埋めるcontrol情報がmetadataとACLである。document_id、chunk_id、source、page、section、version、effective periodをprovenanceとして持ち、tenant、role、groupは検索時filterへ使える粒度でchunkへ継承する。欠損した権限属性を推測で補わずdeny-by-defaultとし、権限変更がactive indexへ反映される時間も要件に含める。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)

## 2.1.6. Embeddingとindex構築

Sentence-BERTは文を独立にencodeして類似度計算を高速化し、HNSWは近似近傍探索をgraph構造で高速化した。しかしmodel、dimension、normalization、distance metricが不一致なら、同じvector storeでもrankingは意味を失う。sparse indexとvector indexの役割を分け、embeddingとANN設定を一つのindex契約として固定する。recall、latency、memoryを同時に測り、parser・chunkerを含むversion bundleで再構築可能にする。

**参照論文**
- [P024] [Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks](https://arxiv.org/abs/1908.10084)
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)
