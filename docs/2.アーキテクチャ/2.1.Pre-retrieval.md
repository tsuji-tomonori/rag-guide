# 2.1 検索前処理（Pre-retrieval）

検索前処理は、質問が届く前に、原資料を検索可能な知識へ変える工程です。
資料の選定、解析、分割、メタデータ付与、埋め込み生成、インデックス構築を行います。
この工程で失われた情報を、質問時のLLMが原文なしで回復することはできません。

## 2.1.1 入力、処理、出力

[RAGのテキスト生成に関するサーベイ](https://arxiv.org/abs/2404.10981)は、RAGの処理をpre-retrieval、retrieval、post-retrieval、generationへ分けて整理しています。
検索前処理の入力は、原文書、情報源のメタデータ、アクセス制御リスト（Access Control List：ACL）、更新・削除イベントです。

主な処理は次のとおりです。

1. 検索対象とする情報源を選びます。
2. 文書から本文、見出し、表、図などの構造を取り出します。
3. 検索に用いる単位へ文書を分割します。
4. 出典、版、有効期間、ACLなどのメタデータを付与します。
5. キーワード検索用と意味検索用のインデックスを構築します。
6. 件数、欠損、権限、検索可能性を検査して公開します。

出力は、版を持つ検索単位、埋め込みベクトル、検索インデックス、取り込み結果の報告です。
一まとまりの処理単位（ジョブ）が終了したことだけを成功と見なしてはいけません。
正しい根拠を検索でき、回答から原文の版と箇所へ戻れる状態になっていることが成功条件です。

図2-2は、情報源から検索インデックスまでの準備工程を左から右へ示します。
上段の七工程を追い、下段の品質検査、版管理、ACL継承が全工程にまたがることを確認します。
途中の成果物に問題があれば、右端の検索可能な状態へ公開しません。

## 2.1.2 情報源と正本

検索対象を増やす前に、どの資料を**正本（source of truth）**として扱うかを決めます。
正式な規程、参考資料、下書き、失効文書を同じ優先度で登録すると、質問とよく似た誤った根拠が上位に現れる可能性があります。
LLMへ資料の権威性を推測させず、取り込み前に優先関係を定めます。

[Datasheets for Datasets](https://arxiv.org/abs/1803.09010)は、データセットの目的、構成、収集過程、利用条件、保守などを文書化する枠組みを提案しました。
RAGの情報源にも、所有者、目的、更新頻度、有効期間、利用制限、既知の欠陥を記録します。
同じ内容の重複、版違い、派生資料、矛盾を可視化し、どの資料を優先するかを規則にします。

[Data Cascades](https://dl.acm.org/doi/10.1145/3411764.3445518)は、高リスクなAIシステムで、上流のデータ問題が下流へ連鎖する事例を調査しました。
RAGでも、所有者不明の文書や更新されない資料を大量に登録すると、検索器とLLMの調整だけでは解決できない問題が生じます。
登録件数ではなく、正本性、更新責任、利用条件が明確な資料の割合を確認します。

## 2.1.3 文書解析と正規化

PDFやWebページから文字列だけを抽出すると、読み順、見出し階層、表の行列、図と説明文の関係が失われる場合があります。
二段組みのPDFで左右の列が混ざると、存在しない文章が検索単位として作られます。
表の見出しを失うと、数値が何を表すか分からなくなります。

[LayoutLMv3](https://arxiv.org/abs/2204.08387)は、文書のテキストと画像領域を統合的に事前学習し、レイアウトを含む複数の文書理解タスクで評価されました。
実務の解析でも、ページ、座標、読み順、表セル、キャプション、コードブロックなど、回答と引用に必要な構造を保持します。
ヘッダー、フッター、ナビゲーションなど、各ページで繰り返される文字列は本文と区別します。

正規化では、文字コード、改行、空白、箇条書きなどを揃えます。
ただし、型番、記号、数値、コードの空白を意味が変わる形で変更してはいけません。
解析器の版、警告、信頼度、失敗理由を保存し、検索失敗と解析失敗を区別して再処理できるようにします。

## 2.1.4 検索単位への分割

**チャンク分割（chunking）**は、文書を検索候補となる単位へ分割する処理です。
分割してできた各単位を**チャンク**と呼びます。
単に同じトークン数へ揃える作業ではなく、何を一件の候補として比較するかを決めます。
粒度は、文書をどの大きさの単位で扱うかを表します。

[Dense X Retrieval](https://arxiv.org/abs/2312.06648)は、文書、段落、文、命題という検索粒度を比較し、評価した条件で細かな命題単位が有効な場面を報告しました。
粒度を細かくすると、質問に直接関係する文を検索しやすくなります。
一方で、主語、前提条件、例外、表見出しを別のチャンクへ分けると、取得した箇所だけでは意味を判断できません。

文書種別に応じて、一定長、見出し単位、意味的な話題単位、親子構造、表単位などを比較します。
[TextTiling](https://aclanthology.org/J97-1003/)は、語彙のまとまりから複数段落の話題境界を検出する初期の方法です。
意味的な分割も、節番号や表構造のような明示的な構造を無視せずに利用します。

検索単位、LLMへ渡す単位、引用単位は同じである必要はありません。
短い子チャンクで検索し、周囲を含む親節をLLMへ渡し、主張を支える文だけを引用できます。
三つの単位を識別子で結び、どの変換を行ったか追跡します。

## 2.1.5 メタデータとACL

埋め込みベクトルは文章の意味的な近さを表しますが、文書の所有者、版、有効期間、閲覧権限は表しません。
これらを明示する情報がメタデータとACLです。

来歴を追跡するため、少なくとも文書ID、チャンクID、情報源、ページ、節、版、有効期間を保持します。
[KILT](https://arxiv.org/abs/2009.02252)がタスクの正解と根拠ページを結び付けたように、検索単位から原文へ戻れる識別子が必要です。
検索の絞り込みには、文書種別、製品、地域、言語、公開状態などを利用できます。

契約組織などの分離単位であるテナント、利用者、役割、グループなどの権限属性は、文書から各チャンクへ継承します。
[Cedar](https://arxiv.org/abs/2403.04651)は、認可方針を明示的に記述し、解析できる言語として設計されました。
RAGでも、権限条件を関連度スコアへ混ぜず、検索候補へ入る前の必須フィルター（Hard filter）として扱います。

権限属性が欠けた場合は、公開資料だと推測せず、既定で拒否します。
文書の権限が剥奪された場合に、稼働中のインデックス、再利用のための一時保存（キャッシュ）、旧版から消えるまでの時間も要件として測ります。

## 2.1.6 埋め込みとインデックス

意味検索では、文章を数値ベクトルへ変換する**埋め込みモデル（embedding model）**を利用します。
[Sentence-BERT](https://arxiv.org/abs/1908.10084)は、文を独立してベクトル化し、文同士の類似度計算を高速に行える構成を提案しました。
検索時には、質問とチャンクを同じモデルと前処理でベクトル化し、定めた距離尺度で比較します。

ベクトル数が多い場合は、近似最近傍探索（Approximate Nearest Neighbor：ANN）を利用できます。
[HNSW](https://arxiv.org/abs/1603.09320)は、階層化した近傍グラフを探索するANN手法です。
近似によって検索時間を短縮できますが、真の近傍を一部取り逃がす可能性があるため、完全探索との再現率を測ります。

埋め込みモデル、次元数、正規化、距離尺度、ANN設定は、一つのインデックス契約として版を固定します。
質問と文書で異なるモデルや正規化を使うと、距離の意味が失われます。
[MTEB](https://arxiv.org/abs/2210.07316)のような共通評価は候補選定に役立ちますが、最終判断は対象言語、専門用語、文書、質問で行います。

キーワード検索用の転置インデックスと、意味検索用のベクトルインデックスは目的が異なります。
両方を構築する場合は、同じチャンクIDと版へ結び付けます。
文書解析器、分割器、埋め込みモデル、インデックス設定をまとめて記録し、同じ入力から再構築できるようにします。

資料準備の設計と公開前の検査は、第3章で詳しく扱います。
