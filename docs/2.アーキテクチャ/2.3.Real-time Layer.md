# 2.3 Real-time Layer

## この節の役割

- ユーザー質問に対して、回答に必要な根拠候補を取得する工程を定義する。
- sparse、dense、hybrid、query処理、routing、filteringの位置関係を示す。
- Retrievalの目的を「最終回答を作ること」ではなく、「必要な根拠を候補集合へ入れること」とする。

## 2.2.1. Retrievalの定義と成功条件

- 入力を正規化済みquery、ユーザー権限、検索対象、時点条件とする。
- 出力をrank付きの候補chunkと、そのscore、metadata、取得経路とする。
- 成功条件を「正解根拠がtop-k候補へ入ること」とする。
- precisionの最終調整はPost-retrievalが担うことを明示する。

## 2.2.2. Query理解と正規化

- 会話中の省略を補うstandalone query化を説明する。
- 略語、表記ゆれ、誤字、製品名、期間、対象部門を正規化する。
- 元queryを消さず、rewrite前後をtraceへ残す。
- 書き換えが意味を変える失敗を、評価対象として挙げる。

## 2.2.3. Sparse retrieval

- exact match、BM25、fielded searchを位置付ける。
- 型番、エラーコード、条文番号、固有名詞に強いことを示す。
- synonymや表記ゆれだけでは意味類似を十分に扱えない限界を説明する。

## 2.2.4. Dense retrieval

- queryとchunkを同じvector空間で比較する基本形を示す。
- 同義語や自然言語の言い換えに強い一方、exact identifierを落とす場合があると説明する。
- embedding model、言語、domain、distance metricが結果へ影響する。
- ANNによる近似検索では、速度とrecallのトレードオフがある。

## 2.2.5. Hybrid retrievalとfusion

- sparseとdenseを並列実行し、候補を統合する流れを示す。
- score normalization、weighted fusion、RRFを区別する。
- 同一chunkの重複を統合し、どのretrieverが取得したかを残す。
- hybridは常に優れるという断定を避け、業務queryで評価する。

## 2.2.6. Query expansion、decomposition、routing

- multi-query、HyDE、Query2docをquery expansionの例として示す。
- 複雑な質問ではsub-questionへ分解する。
- 文書、SQL、Graph、API、画像など、検索対象をroutingする。
- 追加のLLM呼び出しによるlatency、cost、誤書き換えをトレードオフとして示す。

## 2.2.7. Metadata / ACL filtering

- tenant、group、version、effective date、source typeで候補を絞る。
- 権限filterは生成後ではなく、検索前または検索時に適用する。
- pre-filterとpost-filterでrecallやlatencyが変わることを示す。
- filter条件そのものをtraceへ残す。

## この節の役割

- Retrievalが返した候補を、LLMが安全かつ効率的に使えるevidence setへ変換する工程を定義する。
- reranking、filtering、deduplication、diversity、compression、context packingの順序と役割を示す。
- 「上位k件をそのままpromptへ貼る」構成の問題点を説明する。

## 2.3.1. Post-retrievalの定義

- 入力をrank付きcandidate set、query、ACL、token budgetとする。
- 出力をcitation可能なevidence setとする。
- 目的を「広く集めた候補から、少数で高品質な根拠を作ること」とする。
- retrieval scoreだけで最終contextを決めないことを示す。

## 2.3.2. Reranking

- cross-encoder、sequence-to-sequence、LLM rerankerを概観する。
- first-stageでrecall、rerankerでprecisionを担う二段階構成を説明する。
- rerank対象件数がlatencyとcostへ直結する。
- reranker scoreも正解確率ではなく、並べ替えの相対指標とする。

## 2.3.3. Filtering

- ACL、effective date、document status、source trust、parse qualityで除外する。
- 古い版と最新版が混在する場合の優先規則を定義する。
- 低品質chunk、空chunk、引用できないsourceを除く。
- filter理由をtraceへ残し、検索漏れと区別できるようにする。

## 2.3.4. DeduplicationとDiversity

- 同一chunk、重複文書、near-duplicateを区別する。
- 同一documentからcontextが埋まる偏りを抑える。
- MMRなどでrelevanceとnoveltyを調整する。
- 多様性を上げすぎて主要根拠が下がる副作用も示す。

## 2.3.5. Context compression

- 抽出型、要約型、token-level圧縮を区別する。
- 圧縮前の原文とcitation対応を保持する。
- 数値、否定、条件、例外、表の列関係が失われやすいことを示す。
- 圧縮導入前後でfaithfulnessとcitation accuracyを評価する。

## 2.3.6. Context packing

- token budget内でevidenceを並べる処理と定義する。
- relevance、論理順、文書順、source priority、conflictを考慮する。
- chunkごとにcitation ID、title、page、section、versionを付ける。
- 根拠を先頭・末尾へ置く工夫だけに頼らず、context自体を短く保つ。

## 2.3.7. Evidence setの契約

- supporting evidence、conflicting evidence、background informationを区別する。
- generatorに渡す本文と、UIへ返すcitation metadataを分ける。
- answerability判定に必要な「根拠量」と「根拠の一貫性」を持たせる。
- 元文書へ戻れるprovenanceを保持する。

## この節の役割

- evidence setを使って、根拠に基づく回答、引用、または安全な拒否を生成する工程を定義する。
- grounded prompt、回答形式、answerability、citation、verificationを一続きの設計として示す。
- 「根拠だけで答えて」と書くだけでは、faithfulnessを保証できないことを説明する。

## 2.4.1. Generationの定義

- 入力をquery、evidence set、回答policy、出力schemaとする。
- 出力をanswer、citations、answerability status、warning / refusal reasonとする。
- generatorの役割を、根拠の要約・統合・説明と定義する。
- evidenceにない事実を補うことは、通常のRAG回答では許可しない。

## 2.4.2. Grounded prompt

- system instruction、user query、retrieved evidenceを明確に分離する。
- evidence内の指示文を命令ではなくデータとして扱う。
- 推測禁止、根拠不足時の拒否、矛盾時の扱いを明示する。
- citation IDの使用方法と、引用なしclaimの扱いを指定する。

## 2.4.3. 回答形式

- Markdown、JSON、表、手順、比較など、用途別schemaを定義する。
- 回答本文とcitation payloadを分離する。
- 構造化出力では、schema validationと再試行条件を決める。
- 長さ、対象読者、専門用語、日付表現もprompt契約に含める。

## 2.4.4. Citation設計

- document、chunk、page、section、source URL、versionをcitationへ紐づける。
- claim単位または文単位で、どのevidenceがsupportするかを示す。
- citation correctnessとcitation completenessを分けて評価する。
- 引用ラベルだけ正しく、本文内容が根拠と一致しない失敗を説明する。

## 2.4.5. AnswerabilityとRefusal

- 十分な根拠がある、根拠がない、矛盾している、権限外、範囲外を区別する。
- 不足時に一般知識で埋めず、確認できない理由を返す。
- 追加質問で解消できる曖昧さと、知識ベースに情報がない場合を分ける。
- refusal correctnessを通常の回答品質と同等に評価する。

## 2.4.6. 複数根拠の統合

- 複数文書の一致点、差分、時系列、例外条件を整理する。
- 新しい版を優先する規則と、複数見解を併記する規則を分ける。
- 数値や条件を統合する際、出典ごとのscopeを失わない。
- multi-hop reasoningでは、各推論段階を支える根拠を追跡する。

## 2.4.7. Verification

- rule-based validation、claim-evidence判定、LLM judge、人手reviewを区別する。
- unsupported claim、誤citation、format violation、policy violationを検査する。
- verifier自身の誤判定を考慮し、golden setで校正する。
- 高リスク領域では回答を自動確定せず、人間の承認へ送る。

## この節の役割

- RAGの実装責務をOffline LayerとOnline Layerへ分ける理由を説明する。
- この区分を研究上の唯一のtaxonomyではなく、再現性、latency、cost、権限、運用を整理する工学的な層分けとして扱う。
- どの処理を事前計算し、どの処理をquery時に行うかという設計判断を示す。

## 2.5.1. なぜ層を分けるのか

- ユーザー要求前にできる処理と、要求後にしか決められない処理を分ける。
- Offlineは再現性と更新可能性、Onlineはlatencyと権限制御を主目的とする。
- 障害時に、index側とrequest path側を切り分けやすくする。
- team ownership、release、監査、cost計測の境界としても使う。

## 2.5.2. Offline Layer

- 入力: source document、metadata、ACL、更新・削除イベント。
- 処理: ingestion、parse、normalize、chunk、embed、index build、quality check。
- 出力: version付きindex、chunk manifest、ingestion report。
- SLO: freshness、ingestion成功率、削除反映時間、再構築時間。

## 2.5.3. Online Layer

- 入力: user query、identity、session context、request policy。
- 処理: auth、query理解、retrieval、filter、rerank、packing、generation、citation、trace。
- 出力: answer / refusal、citations、trace ID。
- SLO: p50 / p95 / p99 latency、availability、cost/query、権限filter correctness。

## 2.5.4. 2層と4工程の対応

- Pre-retrievalは主にOffline Layerへ置く。
- Retrieval、Post-retrieval、Generationは主にOnline Layerへ置く。
- evaluation、version、policyは両層を横断する。
- query-independentな要約や関係抽出はOfflineへ、query-dependentな判断はOnlineへ置く原則を示す。

## 2.5.5. Offlineへ寄せる設計

- 文書要約、entity抽出、synthetic query、bridge fact、階層indexを例示する。
- online latencyとtoken costを下げられる可能性を示す。
- corpus更新時の再計算、storage増加、生成物の誤り固定化を代償として示す。
- EnrichIndexやIndexRAGは代表例であり、適用条件を評価して使うとする。

## 2.5.6. Onlineへ残す設計

- ユーザー権限、最新query、会話文脈、request-specific filteringを挙げる。
- real-time DB / APIの値、時点依存情報はonline取得が必要とする。
- adaptive retrieval、query decomposition、verificationはqueryごとに必要性が変わる。
- online処理を増やすほどlatency、cost、非決定性が増える。

## 2.5.7. 層間の契約

- Offlineが提供するindex schema、metadata schema、version、freshnessを定義する。
- Onlineは使用したindex versionとfilter条件をtraceへ残す。
- schema互換性、blue-green index切替、rollbackを設計する。
- 権限外データはOnlineのcontextだけでなくtraceにも残さない。

## 2.5.8. 分類上の注意

- Offline / Onlineは主要surveyの標準的な唯一の分類ではないと明記する。
- BatchとOffline、Real-timeとOnlineを完全な同義語として扱わない。
- incremental ingestionやstreaming updateのような中間形を認める。
- 本書では実装責務を説明する便宜的な層分けとして使う。
