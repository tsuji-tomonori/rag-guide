# 2.3 Post-retrieval

## 2.3.1. Post-retrievalの定義

Retrievalが広く集めた候補を、LLMが読める少数のevidenceへ変えるのがPost-retrievalである。入力はrank付きcandidate、query、ACL、token budget、出力は原文へ戻れるevidence setとする。RAGGEDが示す通り、検索深度を増やしてもreaderがnoiseに弱ければ性能は落ちる。retrieval scoreの上位をそのまま貼らず、再順位付け、filter、重複排除、圧縮、packingを分離し、どの工程で根拠が消えたか追える構造にする。

**参照論文**
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)
- [P077] [A Survey on Retrieval-Augmented Text Generation for Large Language Models](https://arxiv.org/abs/2404.10981)

## 2.3.2. Reranking

first-stage retrieverはrecallを優先するため、上位には意味が近いだけの候補も混ざる。BERT cross-encoderはqueryとpassageを同時に読み、独立embeddingより精密な関連判断を行う一方、候補数に比例して計算が増える。monoT5やLLM listwiseも選択肢だが、cost、順序依存、再現性が異なる。何件をrerankし何件を残すかを評価変数にし、timeout時はfirst-stage順位へ戻す。

**参照論文**
- [P040] [Passage Re-ranking with BERT](https://arxiv.org/abs/1901.04085)
- [P039] [Document Ranking with a Pretrained Sequence-to-Sequence Model](https://arxiv.org/abs/2003.06713)
- [P084] [Zero-Shot Listwise Document Reranking with a Large Language Model](https://arxiv.org/abs/2305.02156)

## 2.3.3. Filtering

Filteringでは「関連しない」だけでなく「使ってはいけない」を除く。ACL、失効、draft、低いsource trust、parse failureはhardな除外理由になり、古い版と現行版の競合は単純削除せず後の矛盾処理へ送る。除外reasonを残さないと、retrieval missとfilter誤除去を区別できない。安全条件をmodel scoreへ混ぜずruleで強制し、softなfreshnessやproduct適合だけをranking調整へ使う。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P060] [PoisonedRAG: Knowledge Corruption Attacks to Retrieval-Augmented Generation of Large Language Models](https://arxiv.org/abs/2402.07867)

## 2.3.4. DeduplicationとDiversity

同じ文書のoverlap chunkがcontextを占めると、見かけのtop-kは増えても論点は増えない。MMRはrelevanceと選択済み文書に対するnoveltyを同時に考え、検索結果や要約の多様性を制御した。exact duplicate、near duplicate、隣接chunk、version familyを分け、完全重複は早期に落とす。document capやsource capを使いつつ、多様性を上げすぎて低信頼sourceや主要根拠を失う副作用も評価する。

**参照論文**
- [P038] [The Use of MMR, Diversity-Based Reranking for Reordering Documents and Producing Summaries](https://dl.acm.org/doi/10.1145/290941.291025)

## 2.3.5. Context compression

Context compressionは候補数を減らすのではなく、候補内部から必要部分を短くする。RECOMPは抽出型・要約型compressorと、不要ならcontextを空にするselective augmentationを提案した。原文の数値、否定、条件、表headerを失うと、短くなってもevidenceとして壊れる。抽出を基本とし、要約を使う場合はsupporting spanへ逆参照を持たせ、圧縮前後のfaithfulnessとcitation accuracyを測る。

**参照論文**
- [P042] [RECOMP: Improving Retrieval-Augmented LMs with Compression and Selective Augmentation](https://arxiv.org/abs/2310.04408)
- [P036] [LLMLingua: Compressing Prompts for Accelerated Inference of Large Language Models](https://arxiv.org/abs/2310.05736)

## 2.3.6. Context packing

packingはtoken budgetの残りへ上位文書を詰めるだけの処理ではない。質問の論点ごとに必要根拠を配分し、authority、時系列、論理順、conflictを考えて並べる。Lost in the Middleは長文の中央にある情報が利用されにくいことを示したが、位置だけをいじる前にcontext自体を短くするべきである。citation ID、title、page、versionを各blockへ付け、文や表の途中でtruncateしない。

**参照論文**
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 2.3.7. Evidence setの契約

Evidence setはgeneratorへ渡す単なる長文ではなく、役割を持つ構造体にする。supporting、conflicting、background、outdatedを区別し、本文とUI用citation metadataを分離する。KILTのprovenanceとALCEのclaim-citation対応を踏まえ、各evidenceをdocument、page、span、versionへ解決可能にする。さらに、回答可能性を判断できるだけの根拠量と一貫性を持たせ、不足や矛盾を生成側へ隠さず伝える。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)
