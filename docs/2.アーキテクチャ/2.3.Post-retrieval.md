# 2.3 検索後処理（Post-retrieval）

検索後処理は、検索器が広く集めた候補を、LLMが回答に利用できる根拠集合へ変える工程です。
再順位付け、除外、重複排除、圧縮、配置を行い、LLMへ一度に渡せる入力（コンテキスト）へ必要な根拠を残します。

## 2.3.1 入力、出力、成功条件

入力は、順位とメタデータを持つ検索候補、利用者の質問、ACL、利用できるトークン数です。
出力は、原文へ戻れる情報を持つ**根拠集合（evidence set）**です。
候補から何を残し、何を除いたかを工程ごとに記録します。

検索順位の上位をそのままLLMへ渡すだけでは、必要な根拠が使われるとは限りません。
[RAGGED](https://arxiv.org/abs/2403.09040)は、取得する文書数と生成器のノイズ耐性の組合せがRAGの性能へ影響することを分析しました。
候補を増やして検索再現率が上がっても、無関係な情報によって最終回答が悪化する場合があります。

検索後処理では、必要な根拠を残すことと、不要または利用禁止の情報を除くことの両方が成功条件です。
処理を一つのLLM呼び出しへまとめず、再順位付け、絞り込み、重複排除、圧縮、配置を分けると、根拠が消えた箇所を特定できます。

## 2.3.2 再順位付け

第一段階の検索器は、必要な根拠を候補へ入れるために再現率を優先します。
そのため、上位候補にも、質問と語や話題が近いだけで、回答を支えない文書が含まれます。
**再順位付け（reranking）**は、候補をより精密に比較し、LLMへ渡す順序を決めます。

[BERTによる文書断片の再順位付け](https://arxiv.org/abs/1901.04085)は、質問と文書を同時に入力して比較するクロスエンコーダーを検索順位付けへ適用しました。
質問と文書を独立してベクトル化する密検索より、両者の語の対応や条件を詳しく評価できます。
一方で、候補ごとにモデルを実行するため、計算量は候補数に応じて増えます。

[monoT5](https://arxiv.org/abs/2003.06713)のような系列変換モデルや、[LLMによるリスト単位の再順位付け](https://arxiv.org/abs/2305.02156)も候補です。
方式によって、一件ずつ採点するか、候補同士を比較するか、結果が入力順に影響されるかが異なります。
再順位付けする件数と残す件数を評価変数にし、品質、応答時間、費用を測ります。

再順位付けがタイムアウトした場合の処理も決めます。
安全要件を満たす候補に限り、第一段階の順位へ戻すか、回答を保留します。

## 2.3.3 候補の絞り込み

候補の絞り込みは、質問との関連性が低い候補だけでなく、回答に利用してはいけない候補を除きます。
ACL違反、失効、下書き、解析失敗、隔離中の情報源は、スコアに関係なく除外する**強制条件**です。
安全条件をモデルの関連度スコアへ混ぜると、高スコアの文書が条件を上書きする危険があります。

情報源の信頼度、製品への適合度、鮮度などは、用途によって強制条件または順位調整として扱います。
現行版だけを答える質問では旧版を除外できますが、変更履歴を問う質問では旧版も必要です。
古い版と現行版が矛盾する場合、単純に一方を削除せず、矛盾情報として後段へ渡すこともあります。

[PoisonedRAG](https://arxiv.org/abs/2402.07867)は、攻撃者が作成した文書を検索対象へ混入させ、標的となる質問への回答を操作する攻撃を示しました。
関連度が高いことは、情報源が信頼できることを意味しません。
情報源の採用、公開状態、信頼度を明示的な規則で扱います。

除外時には、候補IDと理由を記録します。
理由がなければ、検索器が候補を取得できなかったのか、絞り込みが誤って除いたのかを区別できません。

## 2.3.4 重複排除と多様性

重なりを持たせて文書を分割すると、同じ文章を含む隣接チャンクが複数取得されます。
同じ内容がコンテキストを占めると、候補件数は多く見えても、回答に使える論点は増えません。
完全重複、ほぼ同じ内容、隣接チャンク、版違いを区別して処理します。

完全に同じチャンクは、安定したIDや内容ハッシュで早期にまとめます。
隣接チャンクは、文や表を重複させずに結合できるかを確認します。
版違いでは、最新版を残すだけでよい質問と、差分を保持すべき質問を分けます。

[Maximal Marginal Relevance（MMR）](https://dl.acm.org/doi/10.1145/290941.291025)は、質問への関連性と、すでに選んだ候補に対する新規性を同時に考慮します。
同じ情報源だけで根拠集合が埋まることを防ぎ、異なる論点を含められます。
ただし、多様性を強くしすぎると、主要な根拠を落とし、関連性の低い情報源を残す可能性があります。

一文書または一情報源から残す最大件数を設定する場合は、複数箇所が必要な質問を阻害しないか評価します。
重複率だけでなく、必要な根拠をすべて保持した割合を測ります。

## 2.3.5 コンテキスト圧縮

**コンテキスト圧縮（context compression）**は、候補文書の中から質問に必要な部分を取り出し、LLMへ渡す情報量を減らす処理です。
候補数を減らす再順位付けとは異なり、一つの候補内部を短くします。

[RECOMP](https://arxiv.org/abs/2310.04408)は、抽出型と要約型の圧縮器、および検索結果が不要な場合にコンテキストを使わない選択的拡張を提案しました。
抽出型は原文の一部をそのまま残すため、引用との対応を保ちやすい方法です。
要約型は大きく短縮できますが、元の表現にない補完や条件の脱落が生じる可能性があります。

数値、単位、否定、例外、適用条件、表の見出しを失うと、短くても根拠として利用できません。
原則として引用に必要な原文範囲を残します。
要約を用いる場合は、要約中の主張から元の文へ戻れる対応を保持します。

評価では、圧縮率だけでなく、圧縮前後で必要な事実が保持されたか、引用が原文を支持するかを測ります。
[LLMLingua](https://arxiv.org/abs/2310.05736)のようなトークン単位の圧縮でも、識別子や構文を失った場合の影響を対象データで確認します。

## 2.3.6 コンテキスト配置

**コンテキスト配置（context packing）**は、選んだ根拠をトークン上限内に収め、LLMが利用しやすい順序で並べる処理です。
検索順位の高い文書から残り容量へ詰めるだけでは、質問の一部に必要な根拠が入らない場合があります。

複数の論点がある質問では、論点ごとに必要な根拠の枠を確保します。
文書の権威性、時系列、手順の順序、根拠間の矛盾も配置に反映します。
各ブロックへ引用ID、文書名、ページ、節、版を付け、LLMが情報源の境界を識別できるようにします。

[Lost in the Middle](https://arxiv.org/abs/2307.03172)は、長い入力の中央にある情報の利用性能が低下する傾向を報告しました。
必要な根拠を先頭や末尾へ移す工夫は候補ですが、位置調整の前に不要なコンテキストを減らします。
文や表の途中で切り捨てず、切り捨てた範囲と理由を記録します。

## 2.3.7 根拠集合の契約

根拠集合は、複数の文章を連結した文字列ではなく、各根拠の役割と来歴を持つ構造化データです。
回答を直接支持する根拠、矛盾する根拠、背景説明、失効した根拠を区別します。
本文と、画面表示に使う文書名・リンクなどの引用情報も分離します。

各根拠は、文書、版、ページ、節、原文範囲へ解決できる必要があります。
[KILT](https://arxiv.org/abs/2009.02252)の来歴評価と、[ALCE](https://arxiv.org/abs/2305.14627)の主張と引用の対応は、この契約を評価する基礎です。

生成器へは、質問へ答えるのに十分な根拠があるか、矛盾があるか、どの条件が未確認かを伝えます。
検索後処理で不足を隠すと、生成器は流暢な文章で空白を埋める可能性があります。
根拠不足と矛盾を明示することが、生成工程での適切な回答保留につながります。
