# 2.2 Batch Layer

## この節の役割

- ユーザーから質問を受ける前に、文書を検索可能な知識表現へ変換する工程を定義する。
- データソース、解析、chunking、metadata / ACL、embedding、indexが、後段の検索品質の上限を決めることを示す。

## 2.1.1. Pre-retrievalの定義

- 「質問が来る前に知識ベースを準備する工程」と一文で定義する。
- 入力を原文書、source metadata、ACL、更新・削除イベントとする。
- 出力をversion付きのchunk、embedding、sparse / vector index、ingestion reportとする。
- 後段のLLMが壊れた解析結果や欠落した根拠を原則として回復できないことを示す。

## 2.1.2. データソースと正本

- authoritative source、参考情報、下書き、期限切れ文書を区別する。
- 文書所有者、更新頻度、有効期間、削除要件、引用可否を定義する。
- 同内容の重複、版違い、矛盾文書を取り込み前に扱う必要性を示す。
- 「大量に入れること」と「検索可能な知識を作ること」は同義ではないと説明する。

## 2.1.3. 文書解析と正規化

- PDF、HTML、Markdown、Office、表、画像、OCRを例示する。
- 見出し、ページ、段組み、表セル、図キャプション、コードブロックを保持する。
- header / footer、navigation、重複テキストなどを除去する。
- parser versionとparse confidenceを残し、再処理可能にする。

## 2.1.4. Chunkingと検索単位

- fixed-size、段落、見出し階層、semantic、parent-child、table-awareを位置付ける。
- 検索単位、LLMへ渡す単位、citation単位は同一とは限らないと説明する。
- 小さすぎるchunkの文脈欠落と、大きすぎるchunkの検索ノイズを対比する。
- chunk境界変更はindex全体の検索分布を変えるため、再評価が必要とする。

## 2.1.5. MetadataとACL

- `document_id`、`chunk_id`、`source_uri`、`page`、`section`、`version`、`effective_from/to`を基本項目として挙げる。
- tenant、role、groupなど、検索時filterに使うACLをchunkへ継承する。
- filter用metadataと表示・監査用metadataを区別する。
- 権限変更や削除がindexへ反映されるまでの時間を運用要件に含める。

## 2.1.6. Embeddingとindex構築

- sparse indexとvector indexの役割を簡潔に示す。
- embedding model、dimension、正規化、distance metricをindex契約として固定する。
- ANN indexにはrecall、latency、memoryのトレードオフがあることを示す。
- parser、chunker、embedding、index設定をversion bundleとして追跡する。

## この節の役割

- RAGの実装責務をOffline LayerとOnline Layerへ分ける理由を説明する。
- この区分を研究上の唯一のtaxonomyではなく、再現性、latency、cost、権限、運用を整理する工学的な層分けとして扱う。
- どの処理を事前計算し、どの処理をquery時に行うかという設計判断を示す。

## 2.5.1. なぜ層を分けるのか

- ユーザー要求前にできる処理と、要求後にしか決められない処理を分ける。
- Offlineは再現性と更新可能性、Onlineはlatencyと権限制御を主目的とする。
- 障害時に、index側とrequest path側を切り分けやすくする。
- team ownership、release、監査、cost計測の境界としても使う。

## 2.5.2. Offline Layer

- 入力: source document、metadata、ACL、更新・削除イベント。
- 処理: ingestion、parse、normalize、chunk、embed、index build、quality check。
- 出力: version付きindex、chunk manifest、ingestion report。
- SLO: freshness、ingestion成功率、削除反映時間、再構築時間。

## 2.5.3. Online Layer

- 入力: user query、identity、session context、request policy。
- 処理: auth、query理解、retrieval、filter、rerank、packing、generation、citation、trace。
- 出力: answer / refusal、citations、trace ID。
- SLO: p50 / p95 / p99 latency、availability、cost/query、権限filter correctness。

## 2.5.4. 2層と4工程の対応

- Pre-retrievalは主にOffline Layerへ置く。
- Retrieval、Post-retrieval、Generationは主にOnline Layerへ置く。
- evaluation、version、policyは両層を横断する。
- query-independentな要約や関係抽出はOfflineへ、query-dependentな判断はOnlineへ置く原則を示す。

## 2.5.5. Offlineへ寄せる設計

- 文書要約、entity抽出、synthetic query、bridge fact、階層indexを例示する。
- online latencyとtoken costを下げられる可能性を示す。
- corpus更新時の再計算、storage増加、生成物の誤り固定化を代償として示す。
- EnrichIndexやIndexRAGは代表例であり、適用条件を評価して使うとする。

## 2.5.6. Onlineへ残す設計

- ユーザー権限、最新query、会話文脈、request-specific filteringを挙げる。
- real-time DB / APIの値、時点依存情報はonline取得が必要とする。
- adaptive retrieval、query decomposition、verificationはqueryごとに必要性が変わる。
- online処理を増やすほどlatency、cost、非決定性が増える。

## 2.5.7. 層間の契約

- Offlineが提供するindex schema、metadata schema、version、freshnessを定義する。
- Onlineは使用したindex versionとfilter条件をtraceへ残す。
- schema互換性、blue-green index切替、rollbackを設計する。
- 権限外データはOnlineのcontextだけでなくtraceにも残さない。

## 2.5.8. 分類上の注意

- Offline / Onlineは主要surveyの標準的な唯一の分類ではないと明記する。
- BatchとOffline、Real-timeとOnlineを完全な同義語として扱わない。
- incremental ingestionやstreaming updateのような中間形を認める。
- 本書では実装責務を説明する便宜的な層分けとして使う。
