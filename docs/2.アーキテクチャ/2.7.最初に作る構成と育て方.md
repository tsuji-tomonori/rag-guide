# 2.7 最初に作る構成と育て方

## 2.7.1. 最初の構成：小さなAdvanced RAG

最初の本番候補は、1製品・1部門・1業務に限定した小さなAdvanced RAGとする。Offlineにparser、chunk、metadata/ACL、BM25、embedding、index versionを置き、Onlineにauth、hybrid retrieval、fusion、rerank、dedup、packingを置く。生成にはcitationとrefusal、運用にはgolden set、trace、rollbackを含める。最小とは部品数が少ないことではなく、正解根拠、権限、答えなしを診断できる下限だと定義する。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)

## 2.7.2. 育て方：失敗パターン駆動で拡張する

育て方は製品roadmapではなくfailure logから始める。正解根拠が候補にないならchunk、query、filterを調べ、候補にはあるが落ちるならrerank、contextにあるのに誤答するならpromptやverificationを見る。RAGCheckerのような細粒度診断を使い、変更を一つずつ試す。query、candidate、context、answer、citationを同じtraceで結び、改善した指標だけでなく追加latency、cost、権限への影響を記録する。

**参照論文**
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)

## 2.7.3. 導入順序の目安

導入順序はToy、Scoped Baseline、Production Baseline、Controlled Optimization、Selective Advancedの五段階で示す。早期には正本とACL、hybrid、citation、refusalを整え、次にrerank、評価、index lifecycleを安定させる。query rewritingやcompressionは回帰試験の後で追加し、Graph、SQL、agent、multimodalは該当failureへ限定する。段階を上げる条件と戻す条件を同時に書き、複雑さを不可逆な成長にしない。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)

## 2.7.4. リリース判定

release判定は平均accuracy一つで決めない。golden setでbaseline以上、negative queryで正しいrefusal、権限caseで漏えいゼロ、p95 latencyとcost/queryがSLO内であることを確認する。index、prompt、model、policyを一つのrelease manifestへ固定し、旧versionへの切替手順を実演する。RAGGEDが示すreaderのnoise感度も考慮し、top-k増加による見かけのrecall改善が最終回答を悪化させていないかを見る。

**参照論文**
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)

## 2.7.5. 本節のまとめ

本節の結論は、最初から最先端構成を作ることではない。正本、権限、引用、回答保留、traceを備えた小さな固定pipelineを先に安定させ、その失敗だけを高度な経路へ送る。BM25、dense、RRF、rerankという基礎部品も、業務queryで比較して残す。成熟度はmodule数ではなく、変更前後を再現し、危険なcaseを止め、不要な複雑さを戻せる運用能力で測る。次章から各部品を詳細化する。 したがって本章の到達点は完成形ではなく、比較可能で撤回可能な最初の運用単位を得ることである。
