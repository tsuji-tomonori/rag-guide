# 2.7 最初に作る構成と育て方

## この節の役割

- 最初の本番候補を「小さく限定したAdvanced RAG」として具体化する。
- 初期構成に含める要素と、後から失敗パターンに応じて追加する要素を分ける。
- 現行構成に合わせ、`2.7.1 最初の構成`、`2.7.2 失敗パターン駆動`、`2.7.3 導入順序`の三部でまとめる。

## 2.7.1. 最初の構成: 小さなAdvanced RAG

- 対象を1製品、1部門、1業務、1〜数種類のauthoritative sourceへ限定する。
- read-only QAを先に成立させ、toolによる更新操作は含めない。
- Offline側に、parser、chunker、metadata / ACL、embedding、BM25 index、vector index、index versionを置く。
- Online側に、auth / ACL、最小限のquery normalization、hybrid retrieval、fusion、reranker、dedup、context packingを置く。
- Generationに、grounded prompt、citation、refusalを含める。
- 運用に、golden set、trace、latency / cost監視、rollbackを含める。
- 「最小」をcomponent数の少なさではなく、本番事故を診断・抑制できる最小要件と定義する。


- Data: 正本と対象範囲を限定する。
- Parsing: 見出し、ページ、表、コードをできるだけ保持する。
- Metadata / ACL: version、effective date、tenant、groupを持つ。
- Retrieval: BM25とdenseを併用する。
- Fusion / Rerank: 候補回収と精密順位付けを分ける。
- Post-retrieval: filter、dedup、packingでcontextを整える。
- Generation: 根拠付き回答または拒否を返す。
- Evaluation / Operation: 変更前後を比較し、原因を追跡できるようにする。

## 2.7.2. 育て方: 失敗パターン駆動で拡張する

- まずProduction Baselineを固定し、query、retrieved chunks、context、answer、citationを記録する。
- 失敗をRetrieval、Post-retrieval、Generation、Data、Policyへ分類する。
- 一度に複数の高度化手法を入れず、1変更ずつ評価する。
- 改善指標だけでなく、latency、cost、権限、監査への影響を測る。


- 正解根拠が候補へ入らない
  - chunking、query、index、filterを確認する。
  - query rewriting、expansion、HyDE、retriever変更を比較する。
- 型番、条文番号、固有名詞を落とす
  - BM25、fielded retrieval、exact-match fieldを強化する。
- 意味的には近いが正解でない候補が多い
  - reranker、source trust、metadata-aware retrievalを追加する。
- contextが重複やノイズで埋まる
  - dedup、MMR、compression、context packingを改善する。
- 根拠はあるがLLMが誤読する
  - prompt、evidence order、claim-level verificationを評価する。
- 答えがない質問に答える
  - negative dataset、answerability判定、refusal policyを強化する。
- 複数文書を統合できない
  - query decomposition、iterative retrieval、Graph / hierarchical retrievalを検討する。
- 図表や画像が根拠になる
  - layout-aware parser、table index、multimodal retrievalを検討する。
- DB / APIの最新値が必要
  - structured retrievalまたはread-only toolへroutingする。
- 業務操作が必要
  - Agentic RAGへ進む前にtool権限、承認、停止条件を設計する。

## 2.7.3. 導入順序の目安

- Stage 0: Toy RAG
  - 小規模文書で概念を確認する。
  - 学習・demoに限定し、本番候補とは呼ばない。
- Stage 1: Scoped Baseline
  - authoritative sourceを絞る。
  - parser、chunker、metadata、ACLを整える。
  - BM25 + dense、citation、refusal、traceを入れる。
- Stage 2: Production Baseline
  - hybrid retrieval、reranker、dedup、packingを安定化する。
  - retrieval / generation / end-to-endを分離評価する。
  - index lifecycle、rollback、監視、security testを入れる。
- Stage 3: Controlled Optimization
  - query rewriting、compression、verificationを評価で追加する。
  - quality、latency、costのSLOを定める。
  - driftと本番feedbackを監視する。
- Stage 4: Selective Advanced RAG
  - multi-hopにはGraph / hierarchical RAGを選ぶ。
  - DB / APIにはstructured retrieval、業務操作にはAgentic RAGを選ぶ。
  - 図表・画像にはMultimodal RAGを選ぶ。
  - 根拠利用行動が弱い場合のみretrieval-aware tuningを検討する。

## 2.7.4. リリース判定

- golden setでbaseline以上の品質を確認する。
- negative queryと権限ケースで安全側の挙動を確認する。
- p95 latency、error rate、cost/queryがSLO内であることを確認する。
- index、prompt、model、policyをversion bundleとして固定する。
- rollback先と切替手順を準備してから公開する。

## 2.7.5. 本節のまとめ

- 最初から高度なRAGを作るのではなく、小さなAdvanced RAGを安定させる。
- 高度化の判断は、手法の人気ではなく観測された失敗から行う。
- 成熟度は、正しい根拠、権限制御、拒否、評価、監査を継続できるかで測る。
