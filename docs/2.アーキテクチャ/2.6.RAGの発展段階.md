# 2.6 RAGの発展段階

## この節の役割

- Naive RAG、Advanced RAG、Modular RAGを、単なる流行語ではなく責務・制御点・運用難度の違いとして説明する。
- 本番候補をNaive RAGより堅く、無制限なagentic構成より単純な固定pipelineとして位置付ける。
- 高度化を、観測された失敗に対して必要なmoduleを追加する過程として示す。

## 2.6.1. 発展段階を使う目的

- 成熟度を「導入した手法の数」ではなく、必要な制御点と運用能力で捉える。
- 各段階の入力、処理、評価、監査、失敗時fallbackを比較する。
- 高度な段階ほど常に優れるわけではないと明記する。
- queryの一部だけを高度なflowへroutingする混合設計を認める。

## 2.6.2. Naive RAG

- `question → retrieve top-k → prompt → LLM → answer` を最小形として示す。
- 学習、demo、狭いPoCで有用であることを認める。
- ACL、version、rerank、citation、refusal、評価、traceが不足しやすいと説明する。
- Naive RAGを本番の完成形として扱わない。

## 2.6.3. Advanced RAG

- query normalization、hybrid retrieval、filter、rerank、dedup、packingを追加する。
- grounded generation、citation、refusal、traceを標準経路にする。
- 各段階を個別に評価できる固定pipelineとして構成する。
- 多くの社内文書QAで、最初の本番候補になる段階とする。

## 2.6.4. Modular RAG

- router、planner、memory、tool、Graph、SQL、multimodal、verifierをmoduleとして示す。
- linear、conditional、branching、loopingのcontrol flowを説明する。
- moduleのinterface、timeout、retry、停止条件、権限を設計対象にする。
- queryごとに必要なmoduleだけを使うことを基本にする。

## 2.6.5. Adaptive / Corrective / Reflective RAG

- 検索が必要か、根拠が十分か、再検索すべきかを判定する構成を示す。
- Self-RAG、CRAG、FLAREを代表例として位置付ける。
- judgeやreflection自体が誤る可能性を説明する。
- loop回数上限、latency budget、fallbackを必須要件とする。

## 2.6.6. Agentic RAGとの境界

- read-onlyな知識参照と、API・DB・業務操作を伴うactionを分ける。
- tool実行には追加の認証、認可、policy、人間承認が必要とする。
- 単純FAQをagent loopへ入れず、必要なqueryだけroutingする。
- 成功条件を回答品質だけでなく、業務task完了と安全性で測る。

## 2.6.7. 段階比較

- component数、latency、cost、非決定性、評価難度、監査難度を比較する。
- NaiveからAdvancedへ進む条件を、本番要件の不足として示す。
- AdvancedからModularへ進む条件を、固定pipelineで解けない失敗として示す。
- 元へ戻せるfallback pathとrelease単位を定義する。

## 2.6.8. 導入原則

- まずAdvanced RAGの固定baselineを作る。
- 失敗データを分類し、1つずつmoduleを追加する。
- module追加前後を同じ評価セットで比較する。
- 改善が確認できなければmoduleを外す。
