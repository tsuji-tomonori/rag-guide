# 2.6 RAGの発展段階

## 2.6.1. 発展段階を使う目的

Naive、Advanced、Modularという段階は、導入した技術の数を競う成熟度表ではない。Gaoらのsurveyを基礎に、各段階でどの制御点、評価、監査、fallbackが必要になるかを比較する。高度なflowほどcostと非決定性も増すため、queryの一部だけを上位段階へrouteしてよい。読者が「流行の手法を足す」のではなく、固定pipelineで再現する失敗に応じてmoduleを追加する判断軸を得ることを目的にする。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)

## 2.6.2. Naive RAG

Naive RAGは `question → top-k retrieval → prompt → LLM` という学習に適した最小形である。Lewisらの原型を理解するには有用だが、実務ではACL、版、rerank、citation、refusal、traceが抜けやすい。失敗してもretrievalかgenerationか分からず、権限外文書を取得後に隠す危険もある。toy demoや狭いPoCとして価値を認めつつ、本番の完成形と呼ばないこと、次段階へ進む不足条件を具体化する。

**参照論文**
- [P009] [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)

## 2.6.3. Advanced RAG

Advanced RAGは派手なagent loopではなく、固定pipelineを堅くする段階である。query normalization、hybrid retrieval、hard filter、rerank、dedup、packingを分け、grounded generation、citation、refusal、traceを標準経路にする。各stageを同じgold setで比較できるため、改善と回帰を切り分けやすい。社内文書QAの最初の本番候補として、機能の少なさではなく、事故を診断し抑制できる最小構成だと位置づける。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)

## 2.6.4. Modular RAG

Modular RAGの核心は部品が多いことではなく、router、planner、retriever、SQL、graph、verifierを条件付きで組み替えるcontrol flowにある。Modular RAG論文はlinearだけでなくbranching、looping、fusionを整理した。moduleごとにinput/output schema、timeout、retry、権限、停止条件を定義し、単純queryでは不要なmoduleを通さない。interfaceが曖昧なままagent化すると、性能差より責任境界の消失が先に問題になる。

**参照論文**
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)

## 2.6.5. Adaptive / Corrective / Reflective RAG

Adaptive、Corrective、Reflective RAGは、検索が必要か、取得根拠が十分か、再検索すべきかを途中で判断する。Self-RAGはreflection token、CRAGはlightweight retrieval evaluator、FLAREは生成途中の低confidenceを再検索の契機にした。共通する利点は難しいqueryだけ処理を増やせる点で、共通する弱点はjudge自体も誤る点にある。loop上限、latency budget、停止理由、固定baselineへのfallbackを必須にする。

**参照論文**
- [P013] [Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection](https://arxiv.org/abs/2310.11511)
- [P079] [Corrective Retrieval Augmented Generation](https://arxiv.org/abs/2401.15884)
- [P046] [Active Retrieval Augmented Generation](https://arxiv.org/abs/2305.06983)

## 2.6.6. Agentic RAGとの境界

read-onlyな知識参照と、APIやDBを通じたactionの間に境界を引く。ReActはreasoning、action、observationの反復でinteractive taskを改善したが、tool実行は回答生成より強い権限と副作用を持つ。単純FAQをagent loopへ送らず、計算や業務操作が必要なqueryだけrouteする。成功を回答品質だけでなくtask completion、認可、重複実行なし、停止条件遵守で測り、write操作には人間承認を加える。

**参照論文**
- [P012] [ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)
- [P015] [Toolformer: Language Models Can Teach Themselves to Use Tools](https://arxiv.org/abs/2302.04761)

## 2.6.7. 段階比較

段階比較表では、component数ではなく、latency、cost、非決定性、評価粒度、監査難度、権限面を並べる。NaiveからAdvancedへ進む条件は本番要件の欠落、AdvancedからModularへ進む条件は固定flowで再現するfailureとする。たとえばmulti-hop不足だけをGraph経路へ送り、全queryを複雑化しない。各段階にrollback先とrelease単位を持たせ、改善が一部sliceの悪化を隠していないか確認する。

**参照論文**
- [P010] [Retrieval-Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2312.10997)
- [P011] [Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks](https://arxiv.org/abs/2407.21059)
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)

## 2.6.8. 導入原則

導入原則は「まず動くもの」ではなく「まず比較できるもの」を作ることである。Advanced RAGの固定baselineをversion化し、failureをdata、retrieval、post-retrieval、generation、policyへ分類する。その上で一度に一moduleだけ追加し、同じdatasetで品質、latency、cost、安全性を比較する。改善が再現しなければ外し、難しいqueryへ限定する。技術の新しさではなく、失敗を減らした証拠が構成を残す条件になる。

**参照論文**
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)
