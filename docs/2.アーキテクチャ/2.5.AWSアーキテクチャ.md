# 2.5 AWSアーキテクチャ

## この節の役割

- RAGの実装責務をOffline LayerとOnline Layerへ分ける理由を説明する。
- この区分を研究上の唯一のtaxonomyではなく、再現性、latency、cost、権限、運用を整理する工学的な層分けとして扱う。
- どの処理を事前計算し、どの処理をquery時に行うかという設計判断を示す。

## 2.5.1. なぜ層を分けるのか

- ユーザー要求前にできる処理と、要求後にしか決められない処理を分ける。
- Offlineは再現性と更新可能性、Onlineはlatencyと権限制御を主目的とする。
- 障害時に、index側とrequest path側を切り分けやすくする。
- team ownership、release、監査、cost計測の境界としても使う。

## 2.5.2. Offline Layer

- 入力: source document、metadata、ACL、更新・削除イベント。
- 処理: ingestion、parse、normalize、chunk、embed、index build、quality check。
- 出力: version付きindex、chunk manifest、ingestion report。
- SLO: freshness、ingestion成功率、削除反映時間、再構築時間。

## 2.5.3. Online Layer

- 入力: user query、identity、session context、request policy。
- 処理: auth、query理解、retrieval、filter、rerank、packing、generation、citation、trace。
- 出力: answer / refusal、citations、trace ID。
- SLO: p50 / p95 / p99 latency、availability、cost/query、権限filter correctness。

## 2.5.4. 2層と4工程の対応

- Pre-retrievalは主にOffline Layerへ置く。
- Retrieval、Post-retrieval、Generationは主にOnline Layerへ置く。
- evaluation、version、policyは両層を横断する。
- query-independentな要約や関係抽出はOfflineへ、query-dependentな判断はOnlineへ置く原則を示す。

## 2.5.5. Offlineへ寄せる設計

- 文書要約、entity抽出、synthetic query、bridge fact、階層indexを例示する。
- online latencyとtoken costを下げられる可能性を示す。
- corpus更新時の再計算、storage増加、生成物の誤り固定化を代償として示す。
- EnrichIndexやIndexRAGは代表例であり、適用条件を評価して使うとする。

## 2.5.6. Onlineへ残す設計

- ユーザー権限、最新query、会話文脈、request-specific filteringを挙げる。
- real-time DB / APIの値、時点依存情報はonline取得が必要とする。
- adaptive retrieval、query decomposition、verificationはqueryごとに必要性が変わる。
- online処理を増やすほどlatency、cost、非決定性が増える。

## 2.5.7. 層間の契約

- Offlineが提供するindex schema、metadata schema、version、freshnessを定義する。
- Onlineは使用したindex versionとfilter条件をtraceへ残す。
- schema互換性、blue-green index切替、rollbackを設計する。
- 権限外データはOnlineのcontextだけでなくtraceにも残さない。

## 2.5.8. 分類上の注意

- Offline / Onlineは主要surveyの標準的な唯一の分類ではないと明記する。
- BatchとOffline、Real-timeとOnlineを完全な同義語として扱わない。
- incremental ingestionやstreaming updateのような中間形を認める。
- 本書では実装責務を説明する便宜的な層分けとして使う。

## 2.7.1. 最初の構成: 小さなAdvanced RAG

- 対象を1製品、1部門、1業務、1〜数種類のauthoritative sourceへ限定する。
- read-only QAを先に成立させ、toolによる更新操作は含めない。
- Offline側に、parser、chunker、metadata / ACL、embedding、BM25 index、vector index、index versionを置く。
- Online側に、auth / ACL、最小限のquery normalization、hybrid retrieval、fusion、reranker、dedup、context packingを置く。
- Generationに、grounded prompt、citation、refusalを含める。
- 運用に、golden set、trace、latency / cost監視、rollbackを含める。
- 「最小」をcomponent数の少なさではなく、本番事故を診断・抑制できる最小要件と定義する。

## 2.7.2. 育て方: 失敗パターン駆動で拡張する

- まずProduction Baselineを固定し、query、retrieved chunks、context、answer、citationを記録する。
- 失敗をRetrieval、Post-retrieval、Generation、Data、Policyへ分類する。
- 一度に複数の高度化手法を入れず、1変更ずつ評価する。
- 改善指標だけでなく、latency、cost、権限、監査への影響を測る。

## 2.7.3. 導入順序の目安

- Stage 0: Toy RAG
- 小規模文書で概念を確認する。
- 学習・demoに限定し、本番候補とは呼ばない。
- Stage 1: Scoped Baseline
- authoritative sourceを絞る。
- parser、chunker、metadata、ACLを整える。
- BM25 + dense、citation、refusal、traceを入れる。
- Stage 2: Production Baseline
- hybrid retrieval、reranker、dedup、packingを安定化する。
- retrieval / generation / end-to-endを分離評価する。
- index lifecycle、rollback、監視、security testを入れる。
- Stage 3: Controlled Optimization
- query rewriting、compression、verificationを評価で追加する。
- quality、latency、costのSLOを定める。
- driftと本番feedbackを監視する。
- Stage 4: Selective Advanced RAG
- multi-hopにはGraph / hierarchical RAGを選ぶ。
- DB / APIにはstructured retrieval、業務操作にはAgentic RAGを選ぶ。
- 図表・画像にはMultimodal RAGを選ぶ。
- 根拠利用行動が弱い場合のみretrieval-aware tuningを検討する。

## 2.7.4. リリース判定

- golden setでbaseline以上の品質を確認する。
- negative queryと権限ケースで安全側の挙動を確認する。
- p95 latency、error rate、cost/queryがSLO内であることを確認する。
- index、prompt、model、policyをversion bundleとして固定する。
- rollback先と切替手順を準備してから公開する。

## 2.7.5. 本節のまとめ

- 最初から高度なRAGを作るのではなく、小さなAdvanced RAGを安定させる。
- 高度化の判断は、手法の人気ではなく観測された失敗から行う。
- 成熟度は、正しい根拠、権限制御、拒否、評価、監査を継続できるかで測る。
