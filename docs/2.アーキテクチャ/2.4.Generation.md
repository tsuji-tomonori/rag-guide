# 2.4 Generation

## 2.4.1. Generationの定義

Generationはevidence setを要約・統合・説明し、回答、引用、または安全な拒否へ変換する工程である。RAGとFiDは検索根拠を条件として生成する基本を示したが、evidenceにない事実までモデル内部知識で補えば通常のgrounded QAから外れる。入力をquery、evidence、policy、output schemaに固定し、出力へanswerability statusとwarningを含める。文章の流暢さではなく、claimが根拠へ対応することを成功条件にする。

**参照論文**
- [P009] [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)
- [P006] [Leveraging Passage Retrieval with Generative Models for Open Domain Question Answering](https://arxiv.org/abs/2007.01282)

## 2.4.2. Grounded prompt

Grounded promptでは命令と資料の境界を明示する。system policy、user query、retrieved evidenceを別blockに置き、文書中の「前の指示を無視せよ」を命令として扱わない。Indirect prompt injection研究は、外部dataがtoolや出力を乗っ取る危険を実証した。推測禁止、根拠不足時の保留、矛盾時の併記、citation IDの使い方を契約化し、promptだけでretrieval missを修復できないことも明記する。

**参照論文**
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)

## 2.4.3. 回答形式

回答形式は見た目ではなく利用目的から逆算する。FAQなら短い結論とclaim直後のcitation、手順なら前提・順序・警告、機械連携ならJSON schemaを選ぶ。構造化出力ではparse可能性、必須field、再試行条件を決め、回答本文とcitation payloadを分ける。長さ、対象読者、用語、日付表現もversion化された契約へ含めることで、model変更時に内容品質とformat failureを別々に評価できる。

**参照論文**
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)
- [P053] [G-Eval: NLG Evaluation using GPT-4 with Better Human Alignment](https://arxiv.org/abs/2303.16634)

## 2.4.4. Citation設計

citationは文末へsource名を飾る処理ではない。ALCEはcitation correctnessとcompletenessを分離し、主張を支えない引用と、引用が欠けた主張を別の失敗として扱った。document、chunk、page、section、versionを短いcitation IDへ束ね、claim単位でsupport関係を持たせる。UIでは原文を開けるようにし、内部ではquote spanとsnapshotまで追跡する。ラベルが正しくても本文と根拠が一致しないcaseを必ず検査する。

**参照論文**
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)

## 2.4.5. AnswerabilityとRefusal

回答可能性は二値ではなく、十分な根拠、部分的根拠、矛盾、権限外、範囲外、曖昧に分ける。GopherCiteは不確かな質問でabstainすることが品質を上げると報告し、SQuAD 2.0はunanswerable questionを評価へ導入した。追加質問で解消できる曖昧さと、KBに情報がない状態を区別し、不足を一般知識で埋めない。false refusalも測り、業務riskに応じた閾値を設ける。 拒否文にも確認できた範囲と不足理由を含めれば、安全性を保ちながら次の行動を案内できる。

**参照論文**
- [P076] [Teaching language models to support answers with verified quotes](https://arxiv.org/abs/2203.11147)
- [P139] [Know What You Don't Know: Unanswerable Questions for SQuAD](https://arxiv.org/abs/1806.03822)

## 2.4.6. 複数根拠の統合

複数根拠の統合では、文書を一つの平均的説明へ溶かさない。FiDは多数passageをdecoderで融合できることを示したが、版、scope、時系列が異なる情報を混ぜる危険は残る。一致点、差分、例外をsourceごとに保ち、最新を優先する規則と複数見解を併記する規則を分ける。multi-hopでは各推論段階を支える根拠を追跡し、数値や条件のscopeが統合時に消えないようにする。 統合の価値は差を消すことではなく、差が生じた条件を質問へ対応づけて説明する点にある。

**参照論文**
- [P006] [Leveraging Passage Retrieval with Generative Models for Open Domain Question Answering](https://arxiv.org/abs/2007.01282)

## 2.4.7. Verification

生成後のverificationは、同じモデルへ「正しいですか」と聞く一工程ではない。Chain-of-Verificationはdraftと検証質問、独立回答、最終修正を分け、FActScoreは長文をatomic factへ分解して支持を測る。ruleでschema、数値、citation IDを検査し、claim-evidence判定、LLM judge、人手reviewを段階化する。verifierにも誤判定があるためgold setで校正し、高リスクでは自動確定せず承認へ送る。

**参照論文**
- [P044] [Chain-of-Verification Reduces Hallucination in Large Language Models](https://arxiv.org/abs/2309.11495)
- [P045] [FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation](https://arxiv.org/abs/2305.14251)
