# 8.2 プライバシーと機密情報

## 8.2.1. データ分類とdata flow

Privacy設計は、どこに機密情報が流れるかを描くことから始まる。public、internal、confidential、restricted、PII、credentialを分類し、sourceからparser、chunk、embedding、index、retrieval、LLM、answer、memory、logまで追う。Datasheetsがデータの収集・用途・保守を文書化するように、各保存先のowner、region、subprocessor、retentionを台帳化する。同じ文字列でも本文保存、embedding化、外部送信、画面表示で許可条件を分ける。

**参照論文**
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)
- [P086] [Data Statements for Natural Language Processing: Toward Mitigating System Bias and Enabling Better Science](https://aclanthology.org/Q18-1041/)

## 8.2.2. Privacy threat model

Privacyの攻撃者は外部者だけではない。閲覧権限を持つ利用者の過剰抽出、内部不正、connector侵害、provider設定ミスを分け、membership inference、KB extraction、memorization、side channel、log leakageを考える。RAG-Thiefはblack-boxな反復queryからprivate knowledgeを抽出し得ることを示し、Generating Is Believingは応答から文書の所属を推定する脅威を扱った。回答本文だけでなく、拒否やcitationの有無も漏えい信号として評価する。

**参照論文**
- [P157] [RAG-Thief: Scalable Extraction of Private Data from Retrieval-Augmented Generation Applications with Agent-based Attacks](https://arxiv.org/abs/2411.14110)
- [P151] [Generating Is Believing: Membership Inference Attacks against Retrieval-Augmented Generation](https://arxiv.org/abs/2406.19234)
- [P153] [Membership Inference Attacks Against Machine Learning Models](https://arxiv.org/abs/1610.05820)

## 8.2.3. Data minimizationと目的制限

最も確実な保護は、不要な情報を入れないことである。RAGの目的に必要な文書、field、期間だけを取り込み、embeddingへcredentialや不要PIIを含めない。生成、品質評価、analytics、model improvementを別目的として定義し、ある目的で収集したqueryを無条件に二次利用しない。Data Statementsが利用背景を明示するように、対象言語・地域・利用者と除外範囲を記録する。minimization後の検索品質低下も測り、必要性を根拠付きで見直す。

**参照論文**
- [P086] [Data Statements for Natural Language Processing: Toward Mitigating System Bias and Enabling Better Science](https://aclanthology.org/Q18-1041/)
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)

## 8.2.4. Redaction・tokenization・表示制御

Redactionは一箇所で済まない。ingestion前に不要PIIを除き、retrieval後に利用者権限へ応じてmaskし、generation後に表示・citation previewを再検査する。復元可能tokenizationは業務照合に使えるが鍵管理が必要で、不可逆redactionは分析用途に向く。PII leakage研究が示す抽出patternをtestへ取り込み、本文を隠してもsource titleや抜粋から個人情報が露出しないか確認する。mask後の数字・文脈が誤回答を生まないことも評価する。

**参照論文**
- [P141] [Analyzing Leakage of Personally Identifiable Information in Language Models](https://arxiv.org/abs/2302.00539)
- [P147] [Extracting Training Data from Large Language Models](https://arxiv.org/abs/2012.07805)

## 8.2.5. 暗号化・network・provider境界

暗号化はat rest、in transitだけでなく、backup、temporary file、vector store、model providerへの経路まで対象にする。customer-managed key、private endpoint、region、provider retentionをデータ分類に応じて選ぶ。ただし暗号化は、権限を誤った利用者やprompt injectionから平文を守らない。外部modelへ送るcontextを最小化し、network egressとsubprocessorを台帳化する。provider境界を越える前後でredactionと監査を行い、鍵ローテーションと復旧も試験する。

**参照論文**
- [P141] [Analyzing Leakage of Personally Identifiable Information in Language Models](https://arxiv.org/abs/2302.00539)
- [P145] [Deep Learning with Differential Privacy](https://arxiv.org/abs/1607.00133)

## 8.2.7. Retention・削除・法的保全

Source削除は元ファイルだけで終わらず、chunk、embedding、index、cache、memory、log、backupへ伝播する。通常retentionとlegal holdを分け、保全中は削除不能である理由と閲覧制限を記録する。削除完了には各artifactのtombstone、再index、cache purge、backup期限を証跡として残す。trainingや評価setへ二次利用した場合も対象に含める。期限をSLO化し、deleted-but-searchable canaryで実際に検索不能になったかを確認する。

**参照論文**
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)

## 8.2.8. Privacy testと残余リスク

Privacy controlは導入しただけでなく、攻撃とutilityの両面から試す。canary PII、membership inference probe、反復抽出、rate-limit回避、citation previewを隔離環境で実行する。Differential privacyはformal guaranteeを与えるが、RAGの検索・回答全体へ単純適用できるわけではなく、精度と複雑性の代償がある。redaction前後のrecall、faithfulness、latencyを比較し、対応できた攻撃、未対応、残余リスク、利用者への注意を明記する。
