# 8.7 知識更新と運用

## 8.7.1. Lifecycle対象とowner

Knowledge lifecycleの対象を一枚の「index」と呼ばず、source document、parsed block、chunk、embedding、sparse／vector index、cacheへ分ける。各artifactにowner、source of truth、更新頻度、削除責任を割り当て、data change、pipeline change、model change、policy changeを別typeとして扱う。Hidden Technical Debtがdata dependencyとconfiguration debtを指摘するように、誰も所有しない派生物が長期障害を生む。変更通知とSLOをartifact間の契約にする。

**参照論文**
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)

## 8.7.2. Identity・version・manifest

更新後も同じ文書を追えるstable document IDと、入力・設定から再現できるchunk IDを設計する。manifestにはsource snapshot、content hash、parser、chunker、embedding model・dimension・metric、metadata schema、index build codeを記録する。Delta Lakeのtransaction logやtime travelがデータ版を再現するように、RAGでもrelease IDから全artifactを辿れるようにする。URLや配列順を主キーにせず、同じ設定から再buildできるかを定期試験する。

**参照論文**
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)

## 8.7.3. Ingestion方式

Scheduled batchは単純で再確認しやすく、incremental syncは差分だけを処理し、CDC／event-drivenは鮮度を縮める。どれを選んでもidempotency、ordering、late event、dead-letter、reconciliationが必要である。DataflowとMillWheelはbatch・streamを共通のstateとevent timeで考え、遅延や再実行を扱う枠組みを示した。失敗文書を全体成功へ混ぜず、partial successを可視化し、定期full scanでevent取りこぼしを検出する。

**参照論文**
- [P014] [The Dataflow Model](https://doi.org/10.14778/2824032.2824076)
- [P098] [MillWheel: Fault-Tolerant Stream Processing at Internet Scale](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)
- [P092] [FlumeJava: Easy, Efficient Data-Parallel Pipelines](https://research.google/pubs/flumejava-easy-efficient-data-parallel-pipelines/)

## 8.7.4. Blue/green・canary・rollback

新indexをactiveへ上書きせず、別versionでbuildしてgolden setとshadow trafficを通す。canary比率、切替条件、rollback条件、旧index保持期間を先に決め、model・prompt・policyとの互換組合せをrelease manifestへ固定する。切替後に異常が出たらaliasを戻せるが、旧indexのACLや削除状態も現行policyと一致しなければならない。blue／greenは可用性だけでなく、比較可能性と安全な撤回の仕組みとして使う。

**参照論文**
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)

## 8.7.5. Freshnessとtemporality

Freshnessは文書の更新時刻ではなく、source変更が検索・回答へ反映されるまでのend-to-end lagで測る。published、effective-from/to、supersedes、query as-ofを別fieldにし、最新文書と質問時点で有効な文書を使い分ける。StreamingQAやFreshLLMsは時間とともに増える知識への適応をQAで評価し、Temporality研究はquery timeをretrievalへ組み込む。authoritative sourceと有効期間を併用し、時計ずれも監視する。

**参照論文**
- [P135] [StreamingQA: A Benchmark for Adaptation to New Knowledge over Time in Question Answering Models](https://arxiv.org/abs/2205.11388)
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)
- [P152] [It's About Time: Incorporating Temporality in Retrieval Augmented Language Models](https://arxiv.org/abs/2401.13222)

## 8.7.6. Delete・ACL propagation

削除や権限剥奪は追加更新より厳しいSLOを持つ。sourceからindex、cache、memory、replicaへtombstoneを伝播し、hard delete、compaction、rebuildの完了を確認する。FreshDiskANNやSPFreshは更新可能な大規模ANNの性能を扱うが、セキュリティ上の削除保証はアプリ側の検証も必要である。deleted IDとrevoked principalをcanary queryで探し、旧index・backupの保持期限まで追う。反映遅延を安全指標として公開する。

**参照論文**
- [P093] [FreshDiskANN: A Fast and Accurate Graph-Based ANN Index for Streaming Similarity Search](https://arxiv.org/abs/2105.09613)
- [P159] [SPFresh: Incremental In-Place Update for Billion-Scale Vector Search](https://arxiv.org/abs/2410.14452)

## 8.7.7. Drift monitoring

Driftはquery、corpus、metadata、embedding score、zero-result、answerability、citation、feedbackに現れる。Dataset shift研究が示すように、分布変化の検知は品質劣化そのものではないため、alert後にgolden probeとhuman sampleを実行する。modelやindex versionを跨いだraw scoreは比較せず、rank overlap、slice metric、canary queryを使う。製品改称のようなbenign changeと、旧版急増や検索漏れのharmful driftを分け、対応ownerへ送る。

**参照論文**
- [P148] [Failing Loudly: An Empirical Study of Methods for Detecting Dataset Shift](https://arxiv.org/abs/1810.11953)
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)

## 8.7.8. Regression・runbook・postmortem

変更ごとにablationとregressionを行い、stale answer、missing delete、corrupt index、ingestion backlog、schema mismatchのrunbookを用意する。runbookには検知signal、切り分けquery、停止・rollback、復旧確認、通知先を含める。Hidden Technical Debtがmonitoring debtを警告するように、未知の障害を人の記憶へ依存させない。復旧後は原因、検知遅延、影響範囲、再発防止をpostmortemへ残し、再現caseをsuiteへ追加する。

**参照論文**
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)
