# 8.7 知識更新と運用

## この節の役割

文書追加・更新・削除、権限変更、parser/chunker/embedding変更を、再現可能なindex lifecycleとして管理する。freshness、drift、blue/green、rollback、regression test、incident responseを一つの運用loopへまとめる。

- source、parsed block、chunk、embedding、indexのidentityとversionを設計できる。
- batch、incremental、event-driven updateを使い分けられる。
- blue/green、canary、rollbackとrelease gateを定義できる。
- freshness、削除反映、ACL反映、driftを監視できる。
- 変更時の回帰評価と障害runbookを作れる。

## 参考論文一覧

1. **[StreamingQA: A Benchmark for Adaptation to New Knowledge over Time in Question Answering Models](https://arxiv.org/abs/2205.11388)** — Adam Liška et al., 2022
   - 新しい知識を追加したときのQA適応を、単なるingestion件数ではなく回答能力で評価する。

2. **[FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)** — Tu Vu et al., 2023
   - fast-changing knowledgeやfalse-premise questionを含むfreshness評価と、最新根拠の順序・量を考える。

3. **[It's About Time: Incorporating Temporality in Retrieval Augmented Language Models](https://arxiv.org/abs/2401.13222)** — Anoushka Gade and Jorjeta Jetcheva, 2024
   - query time、published time、effective periodをretrievalへ取り込む設計の参考にする。

4. **[FreshDiskANN: A Fast and Accurate Graph-Based ANN Index for Streaming Similarity Search](https://arxiv.org/abs/2105.09613)** — Aditi Singh et al., 2021
   - 更新を受けるANN indexで、search recall、update性能、保守コストを両立する論点を理解する。

1. **[SPFresh: Incremental In-Place Update for Billion-Scale Vector Search](https://arxiv.org/abs/2410.14452)** — Yuming Xu et al., 2024
   - 大規模vector indexのincremental updateを扱い、full rebuildとのtrade-offを説明する発展資料にする。

2. **[Failing Loudly: An Empirical Study of Methods for Detecting Dataset Shift](https://arxiv.org/abs/1810.11953)** — Stephan Rabanser et al., 2018
   - query/corpus分布のshiftを検知する基礎。ただしshift検知と品質劣化を区別する。

3. **[Hidden Technical Debt in Machine Learning Systems](https://proceedings.neurips.cc/paper/2015/hash/86df7dcfd896fcaf2674f757a2463eba-Abstract.html)** — D. Sculley et al., 2015
   - data dependency、configuration debt、monitoring debtをRAG運用へ対応づけ、変更管理を後付けにしない根拠にする。

## 8.7.1. Lifecycle対象とowner
  - source document、parsed block、chunk、embedding、sparse/vector index、cacheを区別する。
  - 各artifactのowner、source of truth、更新頻度、削除責任を定義する。
  - data change、pipeline change、model change、policy changeを別change typeにする。

## 8.7.2. Identity・version・manifest
  - stable document_id、reproducible chunk_id、content hashを設計する。
  - source snapshot、parser、chunker、embedding、dimension、metric、metadata schemaをmanifestへ記録する。
  - 同じ入力と設定から同じartifactを再構築できるようにする。

## 8.7.3. Ingestion方式
  - scheduled batch、incremental sync、CDC/event-driven、manual publishを比較する。
  - idempotency、dedup、ordering、late event、dead-letter queue、再処理を設計する。
  - 失敗文書を全体成功に混ぜず、partial successを可視化する。

## 8.7.4. Blue/green・canary・rollback
  - 新indexを別versionで構築し、golden setとshadow trafficで評価する。
  - 切替条件、canary比率、rollback条件、旧index保持期間を決める。
  - model/prompt/policyとindexの互換組合せをrelease manifestへ固定する。

## 8.7.5. Freshnessとtemporality
  - source updateからsearchableになるまでをingestion lagとして測る。
  - published_at、effective_from/to、supersedes、as_of queryを扱う。
  - 最新文書が常に正しいとは限らないため、authoritative sourceと有効期間を併用する。

## 8.7.6. Delete・ACL propagation
  - 削除、権限剥奪、契約終了をindex、cache、memory、replicaへ伝播する。
  - tombstone、hard delete、rebuildの使い分けと完了確認を定める。
  - revocation/deletion SLOを安全性要件として扱う。

## 8.7.7. Drift monitoring
  - query、corpus、metadata、embedding score、zero result、answerability、citation、feedbackを監視する。
  - versionを跨いだscore値を直接比較せず、rank overlapやgolden probesを併用する。
  - shift alert後に品質testを実行し、benign changeとharmful driftを分ける。

## 8.7.8. Regression・runbook・postmortem
  - parser、chunker、embedding、index、prompt、model変更ごとにablationと回帰testを行う。
  - stale answer、missing delete、corrupt index、ingestion backlog、schema mismatchのrunbookを用意する。
  - 復旧後に原因、検知遅延、影響範囲、再発防止を記録する。
