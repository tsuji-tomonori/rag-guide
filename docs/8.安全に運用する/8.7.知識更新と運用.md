# 8.7 知識更新と運用

知識更新では、情報源の変更を検索結果へ正しく反映し、必要なら安全に元の状態へ戻します。
文書だけでなく、その文書から作られた解析結果、チャンク、埋め込み、インデックス、キャッシュを一つのライフサイクルとして管理します。

## 8.7.1 管理対象と責任者

管理対象を一つのインデックスとしてまとめず、情報源文書、解析要素、チャンク、埋め込み、疎インデックス、ベクトルインデックス、キャッシュへ分けます。
それぞれに責任者、正本、更新頻度、削除責任、復旧方法を割り当てます。

情報の変更、処理プログラムの変更、モデルの変更、認可方針の変更は、影響範囲が異なるため別の変更種類として記録します。
[Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)は、データ依存関係や設定の複雑さが長期的な負債になることを指摘しています。

派生成果物を誰も所有しない状態にしません。
前工程から後工程へ渡す項目、通知条件、反映期限を成果物間の契約として定めます。

## 8.7.2 識別子、版、マニフェスト

文書の保存場所や題名が変わっても同じ文書を追える、安定した文書IDを設計します。
チャンクIDは、文書ID、文書版、分割設定、位置から再現できる形式にします。

マニフェストには、情報源のスナップショット、内容のハッシュ、解析器、チャンク分割器、埋め込みモデル、次元、距離尺度、メタデータスキーマ、構築コードの版を記録します。
[Delta Lake](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)がトランザクションログから過去の状態を参照できるように、RAGでもリリースIDから全成果物をたどれるようにします。

変更されやすいURLや配列の順番を主キーにしません。
保存したマニフェストから同じインデックスを再構築できるかを定期的に試験します。

## 8.7.3 取り込み方式

決まった時刻に全体または一部を処理する定期バッチは、構成が単純で再確認しやすい方式です。
差分同期は変更分だけを処理し、変更データキャプチャやイベント駆動方式は反映遅延を短くできます。

どの方式でも、同じイベントを再処理しても結果が重複しない性質、イベントの順序、遅れて届くイベント、失敗イベントの隔離、情報源との定期照合が必要です。
[Dataflow Model](https://doi.org/10.14778/2824032.2824076)と[MillWheel](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)は、イベント時刻、状態、再処理を含むデータ処理の考え方を示しています。

一部の文書だけ失敗した状態を全体成功として扱いません。
失敗文書、失敗工程、再試行回数を可視化し、定期的な全件走査でイベントの取りこぼしを検出します。

## 8.7.4 青緑切り替え、少量公開、切り戻し

新しいインデックスで公開中のものを直接上書きせず、別の版として構築します。
固定評価データと、利用者へ返答しない並行要求で品質と性能を確認します。

少量公開で試す本番要求の割合、切り替え条件、切り戻し条件、旧版の保持期間を事前に決めます。
モデル、プロンプト、認可方針との互換性が確認できた組み合わせをリリースマニフェストへ記録します。

異常があれば参照先を旧版へ戻します。
ただし、旧インデックスにも現在のアクセス制御と削除状態を適用しなければ、切り戻しによって情報漏えいが再発します。
青緑切り替えを、可用性だけでなく比較と撤回の仕組みとして使います。

## 8.7.5 鮮度と時間情報

鮮度は文書の更新日時だけではなく、情報源の変更が検索と回答へ反映されるまでの全体遅延で測ります。
公開日時、有効開始日、有効終了日、置き換えた版、質問の基準日時を別々の項目にします。

最新の文書と、質問時点で有効だった文書は同じとは限りません。
[StreamingQA](https://arxiv.org/abs/2205.11388)と[FreshLLMs](https://arxiv.org/abs/2310.03214)は、時間とともに変わる知識へ質問応答システムを適応させる課題を扱っています。

情報源の信頼度と有効期間を組み合わせ、単に日付が新しい文書を優先しません。
サービス間の時計のずれも監視し、時刻比較に使う基準を統一します。

## 8.7.6 削除と権限変更の反映

削除と権限剥奪は、一般的な追加更新より短い反映期限を持たせます。
削除済みを示す記録を、情報源からインデックス、キャッシュ、記憶、複製先へ伝えます。

論理削除、物理削除、圧縮処理、再構築のどの段階まで完了したかを追跡します。
[FreshDiskANN](https://arxiv.org/abs/2105.09613)や[SPFresh](https://arxiv.org/abs/2410.14452)は大規模ベクトルインデックスの更新を扱いますが、権限上の削除保証はアプリケーション側でも検証する必要があります。

削除済み文書IDと権限を失った利用者を固定質問で検索し、取得不能であることを確認します。
旧インデックスとバックアップの保持期限まで追い、反映遅延を安全性の指標として監視します。

## 8.7.7 変化の監視

運用中の分布変化をドリフトと呼びます。
質問、コーパス、メタデータ、検索順位、0件検索、回答可能性、引用、利用者の反応に変化が現れます。

[Failing Loudly](https://arxiv.org/abs/1810.11953)がデータ分布変化の検知方法を比較するように、統計的な変化は調査のきっかけになります。
変化そのものを品質劣化と断定せず、固定質問と人が確認した標本で影響を確かめます。

モデルやインデックスの版が違う検索スコアを直接比較しません。
順位の重なり、スライス別指標、固定した監視質問を使います。
製品名変更のような正常な変化と、旧版急増や検索漏れの有害な変化を分け、担当者へ通知します。

## 8.7.8 回帰試験、手順書、事後検証

変更ごとに構成要素を分けた比較と回帰試験を行います。
古い回答、削除漏れ、破損したインデックス、取り込みの滞留、スキーマ不一致について運用手順書を用意します。

手順書には、検知信号、切り分け用の質問、停止条件、切り戻し、復旧確認、連絡先を含めます。
担当者の記憶だけに依存せず、定期訓練で手順が現在の構成に合うか確認します。

復旧後は、原因、検知までの時間、影響範囲、防御と復旧の成否、再発防止を事後検証へ残します。
障害を再現する事例を評価項目群へ追加し、次回の公開前に必ず実行します。
