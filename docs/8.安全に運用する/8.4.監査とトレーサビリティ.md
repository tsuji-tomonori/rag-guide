# 8.4 監査とトレーサビリティ

## 8.4.1. 監査目的と利用者

監査ログは誰のためかで必要情報が変わる。developerは候補とprompt、SREはlatencyとretry、securityはidentityとdeny、complianceは承認・版、利用者は出典と訂正履歴を求める。目的ごとに粒度、保存期間、閲覧権限を定め、全contextを無期限保存することを監査性と混同しない。ALCEやAISのattributionを品質証跡へ使いつつ、機密本文はredacted copyやhashで代替する。

**参照論文**
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)
- [P128] [Measuring Attribution in Natural Language Generation Models](https://arxiv.org/abs/2112.12870)

## 8.4.2. End-to-end traceモデル

一回答を再構成できるよう、trace IDの下にauth、rewrite、retrieve、rerank、pack、generate、verify、tool、respondのspanを置く。request、session、tenant、actor、parent spanを区別し、retry、fallback、timeout、refusal、人間承認をeventとして残す。RAGCheckerのようなstage別診断は、この連鎖があって初めて機能する。分散service間でclockとIDを統一し、別requestのcontextを誤結合しない。

**参照論文**
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
- [P055] [RAGAS: Automated Evaluation of Retrieval Augmented Generation](https://arxiv.org/abs/2309.15217)

## 8.4.3. Retrieval evidence

Retrieval証跡にはoriginal・rewrite query、search type、filter、top-k、raw score、rank、index versionを含める。取得chunkのdocument・chunk ID、page、section、version、有効日、trustを残せば、後から当時の候補を再現できる。ただし権限外候補の本文をログへ書かず、deny件数とpolicy decisionだけにする。KILTのprovenance思想を検索stageへ拡張し、「根拠がなかった」と「filterで使えなかった」を区別する。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)

## 8.4.4. Claim・citation・provenance

回答をclaimへ分け、各claimをsupporting chunkとcitationへ結ぶ。監査ではcitation precision・recall、unsupported、contradiction、修正履歴を確認し、脚注があることだけを成功としない。RARRはclaimごとに調査・改訂する流れを示し、ALCEは引用品質を独立評価した。sourceが後日更新・削除されても、回答時のsnapshot IDとhashを識別できるようにし、表示URLと証拠保存先を分ける。

**参照論文**
- [P048] [RARR: Researching and Revising What Language Models Say, Using Language Models](https://arxiv.org/abs/2210.08726)
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)

## 8.4.5. Version lineage

同じqueryでもmodel、prompt、guardrail、retriever、reranker、embedding、parser、chunker、index、policyが違えば結果は別systemである。これらをrelease IDへ束ね、evaluation report、feature flag、A/B variant、fallback先もmanifest化する。Model Cardsの考え方をRAG全体へ広げ、用途・制約・評価sliceをversionごとに記す。incident時に当時の組合せを再現できない変更は、少なくとも入出力artifactとprovider revisionを保存する。

**参照論文**
- [P154] [Model Cards for Model Reporting](https://arxiv.org/abs/1810.03993)
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)

## 8.4.6. Log securityとprivacy

ログは監査資産であると同時に高密度な機密情報である。prompt、context、answerの全文、redacted copy、hash、metadata-onlyを目的別に使い分け、暗号化、append-only、access audit、tamper evidence、retention、legal holdを定義する。support、analytics、model improvementへの二次利用は別権限・同意にする。PII leakage研究を踏まえ、検索本文だけでなくqueryと拒否理由にも個人情報が含まれる前提でmaskする。

**参照論文**
- [P141] [Analyzing Leakage of Personally Identifiable Information in Language Models](https://arxiv.org/abs/2302.00539)
- [P143] [Closing the AI Accountability Gap: Defining an End-to-End Framework for Internal Algorithmic Auditing](https://arxiv.org/abs/2001.00973)

## 8.4.8. インシデントと監査レビュー

インシデント時は、影響request、source、actor、version、tool actionを時系列で再構成する。evidence exportにはchain of custody、閲覧記録、hashを付け、調査中の複製も管理する。内部algorithmic audit研究が提案するend-to-end processのように、設計・release・運用の証跡をつなぐ。定期reviewでは未使用ログの削除、controlの有効性、権限棚卸し、残余リスクを確認し、保存量の増加を監査成熟度と取り違えない。

**参照論文**
- [P143] [Closing the AI Accountability Gap: Defining an End-to-End Framework for Internal Algorithmic Auditing](https://arxiv.org/abs/2001.00973)
- [P154] [Model Cards for Model Reporting](https://arxiv.org/abs/1810.03993)
