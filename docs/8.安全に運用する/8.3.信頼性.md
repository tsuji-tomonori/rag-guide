# 8.3 信頼性

## 参考論文一覧

   - 複数componentを直列・並列に呼ぶRAGでは、個々のrare latencyがend-to-end tailへ増幅することを理解する。

2. **[Efficient and robust approximate nearest neighbor search using Hierarchical Navigable Small World graphs](https://arxiv.org/abs/1603.09320)** — Yu. A. Malkov and D. A. Yashunin, 2016
   - HNSWのrecall、latency、memoryをindex parameterで調整する基礎として使う。

3. **[Billion-scale similarity search with GPUs](https://arxiv.org/abs/1702.08734)** — Jeff Johnson, Matthijs Douze, and Hervé Jégou, 2017
   - 大規模vector searchの計算・memory・GPU利用を理解し、FAISS系実装の背景にする。

4. **[RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)** — Jennifer Hsia et al., 2024
   - context量・質・retrieverとreader robustnessの関係を調べ、性能改善が回答品質を悪化させないようにする。

1. **[Efficient Memory Management for Large Language Model Serving with PagedAttention](https://arxiv.org/abs/2309.06180)** — Woosuk Kwon et al., 2023
   - LLM servingのthroughputとKV cache管理を理解し、self-hosted modelのcapacity設計へつなぐ。

2. **[LLMLingua: Compressing Prompts for Accelerated Inference of Large Language Models](https://arxiv.org/abs/2310.05736)** — Huiqiang Jiang et al., 2023
   - prompt/context圧縮でlatencyとtoken costを下げる方向を示す一方、根拠欠落を評価する必要がある。

## 8.5.1. SLI・SLO・error budget
- time to first token、time to final answer、retrieval latency、tool latency、availabilityを定義する。
- interactive QA、batch report、agentic workflowで別SLOを持つ。
- 品質劣化を伴うdegraded responseを成功扱いにする条件を明記する。

## 8.5.2. End-to-end latency budget
- auth、rewrite、retrieve、rerank、pack、LLM、verify、toolへbudgetを配分する。
- fan-out、multi-query、multi-source、retryがcritical pathへ与える影響を示す。
- p50だけでなくp95/p99、cold start、large queryを測る。

## 8.5.3. Retrieval/index性能
- corpus規模、filter selectivity、embedding dimension、top-k、distance metricを整理する。
- HNSWのm、ef_construction、ef_searchなどをrecallとlatencyでsweepする。
- ACL filterを外して高速化することは許容しない。

## 8.5.4. Rerank・context・LLM性能
- first-stage候補数、rerank後件数、context token、output tokenを調整する。
- batching、parallelism、streaming、prompt cache、model sizeを比較する。
- compressionやsmall model利用時にfaithfulnessとcitationを回帰評価する。

## 8.5.5. Capacity・throughput・quota
- 同時実行、QPS、token/s、queue depth、rate limit、burstを見積もる。
- tenant別quotaとnoisy neighbor対策を設計する。
- provider quota、vector store capacity、tool backendの最小値をsystem capacityとして扱う。

## 8.5.6. Timeout・retry・circuit breaker
- 各dependencyにdeadlineを渡し、無制限retryを避ける。
- idempotent operationだけを安全に再試行し、jitterと上限を設定する。
- 継続障害時はcircuit breakerとbackpressureを使う。

## 8.5.7. Fallbackとdegraded mode
- reranker失敗時のfirst-stage fallback、model fallback、tool unavailable時のrefusalを定義する。
- 根拠なしでLLM単体回答へ落とす場合は用途を限定し、明示表示する。
- 安全性・認可・citationを外すfallbackは禁止する。

## 8.5.8. 性能・可用性テスト
- load、stress、soak、spike、failure injection、region/dependency outageを試す。
- large document、low-selectivity filter、multi-hop、長いconversationをworst caseに含める。
- recovery time、data loss、duplicate action、stale cacheを検証する。

## 8.7.1. Lifecycle対象とowner
- source document、parsed block、chunk、embedding、sparse/vector index、cacheを区別する。
- 各artifactのowner、source of truth、更新頻度、削除責任を定義する。
- data change、pipeline change、model change、policy changeを別change typeにする。

## 8.7.2. Identity・version・manifest
- stable document_id、reproducible chunk_id、content hashを設計する。
- source snapshot、parser、chunker、embedding、dimension、metric、metadata schemaをmanifestへ記録する。
- 同じ入力と設定から同じartifactを再構築できるようにする。

## 8.7.3. Ingestion方式
- scheduled batch、incremental sync、CDC/event-driven、manual publishを比較する。
- idempotency、dedup、ordering、late event、dead-letter queue、再処理を設計する。
- 失敗文書を全体成功に混ぜず、partial successを可視化する。

## 8.7.4. Blue/green・canary・rollback
- 新indexを別versionで構築し、golden setとshadow trafficで評価する。
- 切替条件、canary比率、rollback条件、旧index保持期間を決める。
- model/prompt/policyとindexの互換組合せをrelease manifestへ固定する。

## 8.7.5. Freshnessとtemporality
- source updateからsearchableになるまでをingestion lagとして測る。
- published_at、effective_from/to、supersedes、as_of queryを扱う。
- 最新文書が常に正しいとは限らないため、authoritative sourceと有効期間を併用する。

## 8.7.6. Delete・ACL propagation
- 削除、権限剥奪、契約終了をindex、cache、memory、replicaへ伝播する。
- tombstone、hard delete、rebuildの使い分けと完了確認を定める。
- revocation/deletion SLOを安全性要件として扱う。

## 8.7.7. Drift monitoring
- query、corpus、metadata、embedding score、zero result、answerability、citation、feedbackを監視する。
- versionを跨いだscore値を直接比較せず、rank overlapやgolden probesを併用する。
- shift alert後に品質testを実行し、benign changeとharmful driftを分ける。

## 8.7.8. Regression・runbook・postmortem
- parser、chunker、embedding、index、prompt、model変更ごとにablationと回帰testを行う。
- stale answer、missing delete、corrupt index、ingestion backlog、schema mismatchのrunbookを用意する。
- 復旧後に原因、検知遅延、影響範囲、再発防止を記録する。
