# 8.3 Prompt injection・data poisoning対策

## この節の役割

ユーザー入力、外部文書、Web、メール、チケット、memoryなどに埋め込まれた悪意ある命令と、knowledge baseへ注入されたpoison文書を別の攻撃として整理し、ingestionからretrieval、generation、tool実行までの層状防御を示す。

- direct/indirect prompt injection、jailbreak、data poisoning、backdoorを区別できる。
- source compromise、retriever manipulation、context instruction、tool misuseの攻撃連鎖を説明できる。
- source trust、sanitization、structured context、server-side policy、human approvalを組み合わせられる。
- 攻撃成功率だけでなくutility低下と誤検知を含む評価を設計できる。
- 検知後の隔離、再index、credential rotation、影響調査のrunbookを作れる。

## 参考論文一覧

1. **[Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)** — Kai Greshake et al., 2023
   - 外部から取得したdataがinstructionとして解釈され、情報窃取やtool操作へつながる攻撃面を体系的に理解する。

2. **[Prompt Injection attack against LLM-integrated Applications](https://arxiv.org/abs/2306.05499)** — Yi Liu et al., 2023
   - 実アプリケーションでのprompt injectionと攻撃構成を把握し、単純なdelimiter防御の限界を説明する。

3. **[Benchmarking and Defending Against Indirect Prompt Injection Attacks on Large Language Models](https://arxiv.org/abs/2312.14197)** — Jingwei Yi et al., 2023
   - BIPIAを通じて間接攻撃を評価し、boundary awarenessなどの防御を比較する。

4. **[PoisonedRAG: Knowledge Corruption Attacks to Retrieval-Augmented Generation of Large Language Models](https://arxiv.org/abs/2402.07867)** — Wei Zou et al., 2024
   - 少数のmalicious textが特定queryの回答を誘導し得るため、KBを新しいattack surfaceとして扱う。

1. **[StruQ: Defending Against Prompt Injection with Structured Queries](https://arxiv.org/abs/2402.06363)** — Sizhe Chen et al., 2024
   - instructionとdataを構造的に分離する方向を示す。アプリ側のpolicyや認可の代替ではない点も記す。

2. **[The Instruction Hierarchy: Training LLMs to Prioritize Privileged Instructions](https://arxiv.org/abs/2404.13208)** — Eric Wallace et al., 2024
   - system/developer/user/retrieved dataの優先順位をモデルに学習させる防御を理解する。

## 8.3.1. 用語と攻撃者能力
  - direct injection、indirect injection、jailbreak、data poisoning、backdoorを定義する。
  - 攻撃者をquery送信者、文書投稿者、connector侵害者、index管理者に分ける。
  - 目標を回答改変、機密抽出、policy回避、tool misuse、DoSへ分類する。

## 8.3.2. RAG攻撃面の分解
  - source、connector、parser/OCR、metadata、embedding、retriever、context、memory、toolを列挙する。
  - malicious contentがどのqueryで上位取得されるかを含めて攻撃連鎖を描く。
  - 画像、不可視文字、HTML、metadata、添付ファイルも対象にする。

## 8.3.3. Ingestion時の防御
  - source allowlist、owner approval、署名/checksum、trust level、malware/content scanを適用する。
  - 新規・更新文書をquarantineし、既知の危険patternと異常なretrieval誘導語を検査する。
  - 低信頼sourceを最終回答やtool actionの根拠に使わない。

## 8.3.4. Retrieval・context境界
  - retrieved textを命令ではなくuntrusted dataとして明示する。
  - contextをdocument単位で囲み、source、trust、citation IDを付与する。
  - instructionらしい文を検出・除外する場合は、検知漏れと正当文書の誤除外を測る。

## 8.3.5. Generation・tool containment
  - model出力を実行命令として直接使わず、schema validationとallowlistを通す。
  - write action、credential access、外部送信にはserver-side authorizationとhuman approvalを置く。
  - network egress、tool scope、最大step、timeout、budgetを制限する。

## 8.3.6. Guardrailとmodel-side defense
  - prompt attack filter、instruction hierarchy、structured query、classifierを多層で使う。
  - 単一guardrailで完全防御できると主張しない。
  - defense導入によるanswer quality、latency、false positiveを併記する。

## 8.3.8. 検知・封じ込め・復旧
  - 疑わしいsource/chunkを隔離し、影響queryと回答をtraceから特定する。
  - index rollback、再ingestion、cache purge、credential rotation、利用者通知の条件を定める。
  - 攻撃経路、検知遅延、残余リスクをpostmortemへ残す。
