# 8.3 Prompt injection・data poisoning対策

## 8.3.1. 用語と攻撃者能力

攻撃名を混同すると防御点もずれる。direct injectionはuser prompt、indirect injectionは取得data、jailbreakはpolicy回避、data poisoningはknowledge base改ざん、backdoorは特定triggerで挙動を変える攻撃である。攻撃者をquery送信者、文書投稿者、connector侵害者、index管理者へ分け、目標を回答改変、機密抽出、tool misuse、DoSに分類する。GreshakeらとPoisonedRAGの脅威モデルを並べ、入力経路と永続性の違いを示す。

**参照論文**
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)
- [P060] [PoisonedRAG: Knowledge Corruption Attacks to Retrieval-Augmented Generation of Large Language Models](https://arxiv.org/abs/2402.07867)
- [P155] [Prompt Injection attack against LLM-integrated Applications](https://arxiv.org/abs/2306.05499)

## 8.3.2. RAG攻撃面の分解

RAGのattack surfaceはprompt欄だけではない。source、connector、parser/OCR、metadata、embedding、retriever、context、memory、toolへ攻撃が連鎖する。悪意ある文書が特定queryで上位に来るよう語を埋め、本文の命令で外部送信を促す、といった経路を図示する。HTMLの不可視文字、画像内文字、metadata、添付、tool outputも対象にする。各stageで攻撃者が書けるfieldと信頼されるfieldを分け、境界を一つのguardrailへ集約しない。

**参照論文**
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)
- [P142] [Benchmarking and Defending Against Indirect Prompt Injection Attacks on Large Language Models](https://arxiv.org/abs/2312.14197)
- [P060] [PoisonedRAG: Knowledge Corruption Attacks to Retrieval-Augmented Generation of Large Language Models](https://arxiv.org/abs/2402.07867)

## 8.3.3. Ingestion時の防御

Poison文書を検索後に見抜くより、knowledge baseへ入る前に門を設ける。source allowlist、owner approval、署名・checksum、trust level、malware／content scanを適用し、新規・更新文書をquarantineする。PoisonedRAGは少数の攻撃textでもtarget queryを誘導できると示したため、異常なkeyword反復、類似文書急増、retrieval誘導文を検査する。低信頼sourceはtool actionや高risk結論の根拠にせず、公開前canaryで検索挙動を確認する。

**参照論文**
- [P060] [PoisonedRAG: Knowledge Corruption Attacks to Retrieval-Augmented Generation of Large Language Models](https://arxiv.org/abs/2402.07867)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 8.3.4. Retrieval・context境界

取得本文はsystem instructionと同じ文字列でも、命令ではなくuntrusted dataである。documentごとにdelimiter、source、trust、citation IDを付け、instructionらしいspanを検知する場合も原文との対応を残す。StruQはinstructionとdataを構造的に分離する防御を提案し、BIPIAは間接攻撃に対するboundary awareness等を比較した。ただし検知器のfalse positiveで正当な手順書を消す危険があるため、attack successと業務回答品質を同時に測る。

**参照論文**
- [P160] [StruQ: Defending Against Prompt Injection with Structured Queries](https://arxiv.org/abs/2402.06363)
- [P142] [Benchmarking and Defending Against Indirect Prompt Injection Attacks on Large Language Models](https://arxiv.org/abs/2312.14197)

## 8.3.5. Generation・tool containment

Model出力をそのままshell、SQL、APIへ渡さない。tool callはschema validation、allowlist、server-side authorization、scope制限を通し、write、credential access、外部送信には人間承認を置く。ReAct型loopでは一つの悪意あるobservationが後続actionを変え得るため、最大step、timeout、budget、network egressを制約する。tool resultもuntrusted dataとして再入力し、利用者目的から外れるactionをpolicy engineが拒否する。

**参照論文**
- [P012] [ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)

## 8.3.6. Guardrailとmodel-side defense

Prompt attack filter、classifier、structured query、instruction hierarchyは相補的に使う。Instruction Hierarchyはprivileged instructionを優先するようmodelを訓練し、StruQは入力構造を分けるが、どちらも誤認可やpoisoned sourceを自動解決しない。単一guardrailへ完全防御を期待せず、ingestion、retrieval、generation、toolの各層でcontrolを置く。防御ごとにattack success、false positive、answer quality、latencyを測り、shadowからblockへ段階導入する。

**参照論文**
- [P161] [The Instruction Hierarchy: Training LLMs to Prioritize Privileged Instructions](https://arxiv.org/abs/2404.13208)
- [P160] [StruQ: Defending Against Prompt Injection with Structured Queries](https://arxiv.org/abs/2402.06363)
- [P142] [Benchmarking and Defending Against Indirect Prompt Injection Attacks on Large Language Models](https://arxiv.org/abs/2312.14197)

## 8.3.8. 検知・封じ込め・復旧

攻撃を検知したら、回答を止めるだけでなく影響範囲を封じる。疑わしいsource・chunkをquarantineし、traceから取得したquery、生成回答、tool actionを特定する。index rollback、再ingestion、cache purge、credential rotation、利用者通知の条件をrunbook化する。Poison文書がsummaryやmemoryへ派生していれば、元chunk削除だけでは残る。検知遅延、漏れた情報、残存artifact、再発防止をpostmortemへ記録し、攻撃caseを回帰suiteへ追加する。

- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)
