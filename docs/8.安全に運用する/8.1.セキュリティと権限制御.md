# 8.1 セキュリティと権限制御

## 8.1.1. 対象範囲と用語

RAGのsecurity対象は回答文だけではない。source、chunk、embedding、index、prompt、memory、log、tool credentialをassetとして列挙し、end user、文書owner、operator、service、外部provider、攻撃者をactorとして分ける。SaltzerとSchroederのleast privilege、complete mediation、fail-safe defaultsを基礎に、認証は誰か、認可は何を許すか、ACLはresourceごとの許可、tenant isolationは境界維持と定義する。用語を揃えてから脅威図を作る。

**参照論文**
- [P156] [Protection of Information in Computer Systems](https://doi.org/10.1109/PROC.1975.9939)
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)

## 8.1.2. Identityとtrust boundary

User、tenant、service、sessionのidentityは同じではない。clientが送るtenant IDやroleを信用せず、認証基盤で検証したclaimからbackendが検索条件を作る。API gateway、application、retriever、vector store、LLM、tool gatewayの間にtrust boundaryを引き、どこでidentityが変換・委任されるかを示す。間接prompt injectionの研究が示すように、LLMは信頼境界を判断する認可器には向かない。各境界でserver-side validationを繰り返す。

**参照論文**
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)
- [P156] [Protection of Information in Computer Systems](https://doi.org/10.1109/PROC.1975.9939)

## 8.1.3. 認可モデルの選択

固定的な職務権限はRBAC、文書の地域・機密区分・契約・時点はABAC、個別共有はACL、所有・所属関係を辿る場合はReBACが向く。SandhuらのRBACはrole hierarchyとconstraintを体系化し、Cedarはprincipal／action／resource／contextを明示するpolicyへ広げる。実装では方式を混ぜても、deny優先、継承、例外、break-glassの最終規則を一つにする。policy storeと検索用metadataの同期責任も分ける。

**参照論文**
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)

## 8.1.4. Retrieval時のACL適用

権限外chunkはtop-kへ取ってから隠すのではなく、検索前または探索中に候補集合から除く。tenant、allowed group、classification、有効期間をhard filterとし、filter後のrecall低下はindex設計で解く。Curatorはmulti-tenant vector databaseでtenant filterと検索効率を両立するindex構造を研究し、ACLを外して速度を稼ぐ必要がない方向を示す。取得本文がLLM、trace、cacheへ入る前に認可を完了し、0件時も権限を緩めない。

**参照論文**
- [P144] [Curator: Efficient Indexing for Multi-Tenant Vector Databases](https://arxiv.org/abs/2401.07119)
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)

## 8.1.5. Tenant・session isolation

Tenant隔離はindexだけでなく、conversation history、cache、memory、trace、非同期jobにも及ぶ。session再利用やretryで別利用者のcontextが混ざるcaseを、IDORや権限昇格と同様にnegative testする。RAG-Thiefのような反復queryによるprivate KB抽出は、正規利用者でも過剰探索できることを示す。cache keyへtenant・user・permission versionを含め、sessionとidentityの結び付きをbackendが保証し、異常な探索量を監視する。

**参照論文**
- [P157] [RAG-Thief: Scalable Extraction of Private Data from Retrieval-Augmented Generation Applications with Agent-based Attacks](https://arxiv.org/abs/2411.14110)
- [P151] [Generating Is Believing: Membership Inference Attacks against Retrieval-Augmented Generation](https://arxiv.org/abs/2406.19234)

## 8.1.6. Toolとactionの認可

LLMがtoolを選べても、実行権限を与えられたわけではない。readとwriteを分け、副作用のある決済、送信、権限変更には再認証、人間承認、idempotencyを要求する。Indirect Prompt Injectionは外部文書の指示がtool actionへ波及する危険を示したため、server-side policyがactor、resource、action、scopeを再判定する。tool inputとdecision、resultを監査し、service credential一つで全resourceへ到達させない。

**参照論文**
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)

## 8.1.7. 権限変更・削除の反映

認可は付与時だけでなく、剥奪がいつ効くかで安全性が決まる。group変更、退職、契約終了、文書公開範囲変更をindex、cache、memory、replicaへ伝播し、revocation latencyをSLOにする。policy versionとindex versionを対応づけ、古いindexをblue/greenで残す期間も同じ境界を守る。削除eventの取りこぼしを定期reconciliationで検出し、権限変更直後のcanary queryで漏えいがないか確認する。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P098] [MillWheel: Fault-Tolerant Stream Processing at Internet Scale](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)

## 8.1.8. 検証項目と運用指標

Security運用ではunauthorized retrieval rateを0とし、filter accuracy、over-block率、revocation latency、deny理由、抽出試行を監視する。権限別golden dataset、tenant横断、退職者、時限権限、tool misuseをCIへ入れる。RAG membership inference研究は本文を返さなくても応答差からKB内の存在を推測できる可能性を示すため、拒否文やhit有無の露出も検査する。異常query量にはrate limitとabuse detectionを組み合わせる。

- [P151] [Generating Is Believing: Membership Inference Attacks against Retrieval-Augmented Generation](https://arxiv.org/abs/2406.19234)
- [P157] [RAG-Thief: Scalable Extraction of Private Data from Retrieval-Augmented Generation Applications with Agent-based Attacks](https://arxiv.org/abs/2411.14110)
