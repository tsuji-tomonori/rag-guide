# 8.2 セキュリティ

## 参考論文一覧

1. **[Protection of Information in Computer Systems](https://doi.org/10.1109/PROC.1975.9939)** — Jerome H. Saltzer and Michael D. Schroeder, 1975
   - least privilege、complete mediation、fail-safe defaultsなど、RAGの認可境界を設計する基礎原則として使う。

2. **[Role-Based Access Control Models](https://doi.org/10.1109/2.485845)** — Ravi S. Sandhu et al., 1996
   - role、hierarchy、constraintを整理し、部門・職務・管理者権限をRAGのmetadata/ACLへ写像する基礎にする。

3. **[Curator: Efficient Indexing for Multi-Tenant Vector Databases](https://arxiv.org/abs/2401.07119)** — Yicheng Jin et al., 2024
   - multi-tenant filterとvector searchの効率を両立する論点を把握する。認可policyそのものではなく、pre-filter実装の性能面に使う。

4. **[RAG-Thief: Scalable Extraction of Private Data from Retrieval-Augmented Generation Applications with Agent-based Attacks](https://arxiv.org/abs/2411.14110)** — Changyue Jiang et al., 2024
   - black-box queryの反復によりprivate KBが抽出され得ることを踏まえ、rate limit、出力制御、abuse detectionを認可設計へ加える。

1. **[Generating Is Believing: Membership Inference Attacks against Retrieval-Augmented Generation](https://arxiv.org/abs/2406.19234)** — Yuying Li et al., 2024
   - 本文を直接返さなくても、応答挙動からKB内文書の存在が推定される可能性を脅威モデルへ含める。

2. **[Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)** — Kai Greshake et al., 2023
   - retrieved dataがtool実行へ影響する場合、LLM判断とserver-side authorizationを分離すべき理由を補強する。

## 8.1.1. 対象範囲と用語
- 認証、認可、ACL、RBAC、ABAC、tenant isolationを区別する。
- assetをsource文書、chunk、embedding、index、prompt、answer、memory、log、tool credentialに分ける。
- actorをend user、document owner、operator、service、external model provider、attackerに分ける。

## 8.1.2. Identityとtrust boundary
- user identity、tenant identity、service identity、session identityの関係を定義する。
- API Gateway、application backend、retriever、vector store、LLM、tool gatewayの境界を図示する。
- clientから渡されたtenant_idやroleを無条件に信用しない。

## 8.1.3. 認可モデルの選択
- 固定的な職務権限にはRBAC、文書属性・地域・契約・時点条件にはABACを使う。
- allow/denyの優先順位、継承、例外、break-glassを明記する。
- authoritativeなpolicy storeと、検索用metadataの同期責務を分ける。

## 8.1.4. Retrieval時のACL適用
- 権限外chunkをtop-k取得後に隠すのではなく、検索前または検索時に除外する。
- tenant_id、allowed_group_ids、confidentiality、effective dateをhard filterとして扱う。
- filter適用後のrecall低下とlatencyを評価し、安全性を下げずにindex設計を調整する。

## 8.1.5. Tenant・session isolation
- conversation history、cache、memory、traceをtenant/user/sessionへ正しく束縛する。
- session再利用、共有cache、非同期job、retry時のcontext混入をテストする。
- cross-tenant query、IDOR、権限昇格をnegative testへ含める。

## 8.1.6. Toolとactionの認可
- LLMはtool候補を提案できても、実行可否はserver-side policyで判定する。
- read toolとwrite toolを分離し、副作用のある操作には承認・再認証・idempotencyを入れる。
- tool input、resource scope、actor、decision、resultを監査可能に残す。

## 8.1.7. 権限変更・削除の反映
- group変更、退職、契約終了、文書公開範囲変更をindex/cache/memoryへ伝播する。
- revocation latencyをSLO化し、古いindexやcacheからの漏えいを検証する。
- policy versionとindex versionの対応を記録する。

## 8.1.8. 検証項目と運用指標
- unauthorized retrieval rateは0を要求し、filter accuracyとrevocation latencyを測る。
- 権限別golden dataset、tenant横断テスト、tool misuseテストをCIへ入れる。
- 認可エラー、deny理由、異常な探索量、抽出試行を監視する。

## 8.2.1. データ分類とdata flow
- public、internal、confidential、restricted、PII、credentialを定義する。
- sourceからparser、chunk、embedding、index、retrieval、LLM、answer、memory、logまでのflowを図示する。
- 各保存先のowner、region、subprocessor、retentionを一覧化する。

## 8.2.2. Privacy threat model
- 閲覧権限のある利用者による過剰抽出、外部攻撃者、内部不正、provider側の誤設定を分ける。
- membership inference、KB extraction、memorization、side channel、log leakageを扱う。
- 質問・引用・拒否応答自体が情報を漏らす場合を含める。

## 8.2.3. Data minimizationと目的制限
- RAGで本当に必要な文書・field・期間だけを取り込む。
- embeddingへ不要なPIIやcredentialを入れない。
- 生成、評価、analytics、model improvementの利用目的を分離し、二次利用条件を明記する。

## 8.2.4. Redaction・tokenization・表示制御
- ingestion前、retrieval後、generation後のどこでmaskingするかを決める。
- 復元可能tokenizationと不可逆redactionを使い分ける。
- citationやsource previewから原文PIIが露出しないか確認する。

## 8.2.5. 暗号化・network・provider境界
- at rest、in transit、backup、temporary file、vector storeを対象にする。
- customer-managed key、private endpoint、region制約、provider data retentionを整理する。
- 暗号化は誤認可やprompt injectionを防がないことを明記する。

## 8.2.7. Retention・削除・法的保全
- source削除をchunk、embedding、index、cache、memory、log、backupへ伝播する。
- 通常retentionとlegal holdを分け、削除不能な例外を明示する。
- 削除完了を検証するevidenceと期限を定義する。

## 8.2.8. Privacy testと残余リスク
- canary PII、membership inference probe、extraction試験、rate-limit回避試験を行う。
- privacy control導入前後でretrieval recall、faithfulness、latencyを比較する。
- ゼロリスクを主張せず、想定攻撃・未対応攻撃・利用上の注意を残す。

## 8.3.1. 用語と攻撃者能力
- direct injection、indirect injection、jailbreak、data poisoning、backdoorを定義する。
- 攻撃者をquery送信者、文書投稿者、connector侵害者、index管理者に分ける。
- 目標を回答改変、機密抽出、policy回避、tool misuse、DoSへ分類する。

## 8.3.2. RAG攻撃面の分解
- source、connector、parser/OCR、metadata、embedding、retriever、context、memory、toolを列挙する。
- malicious contentがどのqueryで上位取得されるかを含めて攻撃連鎖を描く。
- 画像、不可視文字、HTML、metadata、添付ファイルも対象にする。

## 8.3.3. Ingestion時の防御
- source allowlist、owner approval、署名/checksum、trust level、malware/content scanを適用する。
- 新規・更新文書をquarantineし、既知の危険patternと異常なretrieval誘導語を検査する。
- 低信頼sourceを最終回答やtool actionの根拠に使わない。

## 8.3.4. Retrieval・context境界
- retrieved textを命令ではなくuntrusted dataとして明示する。
- contextをdocument単位で囲み、source、trust、citation IDを付与する。
- instructionらしい文を検出・除外する場合は、検知漏れと正当文書の誤除外を測る。

## 8.3.5. Generation・tool containment
- model出力を実行命令として直接使わず、schema validationとallowlistを通す。
- write action、credential access、外部送信にはserver-side authorizationとhuman approvalを置く。
- network egress、tool scope、最大step、timeout、budgetを制限する。

## 8.3.6. Guardrailとmodel-side defense
- prompt attack filter、instruction hierarchy、structured query、classifierを多層で使う。
- 単一guardrailで完全防御できると主張しない。
- defense導入によるanswer quality、latency、false positiveを併記する。

## 8.3.7. Adversarial evaluation
- clean/attackの対になったdatasetを作り、attack success rateとutilityを測る。
- query、document、image、memory、toolを跨ぐ攻撃を含める。
- model、prompt、index、guardrail更新ごとに回帰実行する。

## 8.3.8. 検知・封じ込め・復旧
- 疑わしいsource/chunkを隔離し、影響queryと回答をtraceから特定する。
- index rollback、再ingestion、cache purge、credential rotation、利用者通知の条件を定める。
- 攻撃経路、検知遅延、残余リスクをpostmortemへ残す。
