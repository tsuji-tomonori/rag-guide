# 8.6 コスト管理

## 8.6.1. Cost taxonomy

RAGの費用はLLM tokenだけではない。ingestion、parser/OCR、embedding、index storage、query、rerank、tool、memory、observabilityを分け、固定費、変動費、更新費、incident費へ分類する。FrugalGPTがmodel cascadeでcost–qualityを扱うように、どのcomponentがどのrequestで発生したかをtraceとinvoice tagで結ぶ。無料枠や一時的な割引を設計前提にせず、full reindexやログ保持など平常query以外の費用も予算へ含める。

**参照論文**
- [P150] [FrugalGPT: How to Use Large Language Models While Reducing Cost and Improving Performance](https://arxiv.org/abs/2305.05176)
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)

## 8.6.2. Unit economics

平均月額だけでは、どの利用が高いか分からない。cost/query、session、answerable query、successful taskを算出し、tenant、feature、route、model、document type、risk tierでsliceする。失敗して再質問が増える安価な回答はtask単価が高くなり得る。p95 costと異常な長文出力も監視し、品質・latencyと同じcase IDで比較する。単価削減をKPIにすると拒否やcitation省略へ偏るため、task success当たり費用を主軸にする。

**参照論文**
- [P150] [FrugalGPT: How to Use Large Language Models While Reducing Cost and Improving Performance](https://arxiv.org/abs/2305.05176)
- [P158] [RouteLLM: Learning to Route LLMs with Preference Data](https://arxiv.org/abs/2406.18665)

## 8.6.3. Batch Layerの費用

Batch費用はcorpus量だけでなく、変更の仕方で決まる。full reindexとincremental update、embedding dimension、複製index、backup、parser再処理を分けて見積もる。Matryoshka表現はdimensionと資源のtrade-offを示すが、任意modelのvectorを安全に短縮できるとは限らない。parser・chunker・embedding変更による全件再計算をrelease計画へ入れ、content hash cacheで未変更chunkを再利用する。blue/green期間の二重storageも忘れない。

**参照論文**
- [P019] [Matryoshka Representation Learning](https://arxiv.org/abs/2205.13147)
- [P093] [FreshDiskANN: A Fast and Accurate Graph-Based ANN Index for Streaming Similarity Search](https://arxiv.org/abs/2105.09613)
- [P159] [SPFresh: Incremental In-Place Update for Billion-Scale Vector Search](https://arxiv.org/abs/2410.14452)

## 8.6.4. Retrieval・rerankの費用

Online検索ではtop-k、multi-query数、KB数、rerank候補数が掛け算で費用を増やす。まず単純queryを低cost routeへ、multi-hopや低confidenceだけを高recall・高rerankへ送る。RAGGEDがcontext増加で常に品質が上がらないと示すため、追加候補一件当たりのrecall・answer改善を測る。cacheにはtenant、ACL、index versionをkeyとして漏えいを防ぎ、hit率だけでなくstale・権限誤用率も監視する。

**参照論文**
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)

## 8.6.5. Context・generationの費用

Generation費用はinstruction、evidence、history、answerのtoken内訳で見る。dedup、extractive compression、selective augmentation、short answerを順に試し、重要根拠を削らず入力を減らす。RECOMPは不要contextを渡さない選択、LLMLinguaはtoken-level圧縮を提案したが、追加model callの費用とfaithfulness低下も考える。providerのprompt cacheやinput discountは条件を明記し、価格変更に依存しないtoken・call数も基礎指標として保存する。

**参照論文**
- [P042] [RECOMP: Improving Retrieval-Augmented LMs with Compression and Selective Augmentation](https://arxiv.org/abs/2310.04408)
- [P036] [LLMLingua: Compressing Prompts for Accelerated Inference of Large Language Models](https://arxiv.org/abs/2310.05736)

## 8.6.6. Model routingとcascade

全queryを最大modelへ送る代わりに、small／large model、RAG／long-context、通常／agentic flowをrouteする。RouteLLMはpreference dataからstrong・weak modelを選び、品質を保ちながらcostを削減する方法を示した。routerの誤りで高risk質問が弱い経路へ落ちないようminimum routeを設定し、confidenceが低ければ上位へescalateする。routing前後の品質、費用、latencyをslice別に測り、model名ではなくtask条件で選ぶ。

**参照論文**
- [P158] [RouteLLM: Learning to Route LLMs with Preference Data](https://arxiv.org/abs/2406.18665)
- [P150] [FrugalGPT: How to Use Large Language Models While Reducing Cost and Improving Performance](https://arxiv.org/abs/2305.05176)
- [P075] [Retrieval Augmented Generation or Long-Context LLMs? A Comprehensive Study and Hybrid Approach](https://arxiv.org/abs/2407.16833)

## 8.6.8. Budget guardrailと改善サイクル

Budget guardrailは事後の請求確認ではなく、実行前・実行中の制御である。tenant・feature別quota、alert、hard limit、approvalを設定し、looping agent、過剰tool call、長大output、再index連発を検知する。上限到達時の挙動を回答短縮、低cost route、安全な停止へ分け、認可やcitationを省かない。定期ablationでquality、p95 latency、costを並べ、Pareto frontierから外れたcomponentを外す。例外予算には期限とownerを付ける。

**参照論文**
- [P150] [FrugalGPT: How to Use Large Language Models While Reducing Cost and Improving Performance](https://arxiv.org/abs/2305.05176)
- [P158] [RouteLLM: Learning to Route LLMs with Preference Data](https://arxiv.org/abs/2406.18665)
