# 8.8 ガバナンス

## 8.8.1. Governanceの対象と原則

Governanceはmodel利用規約だけでなく、data、retriever、index、prompt、tool、memory、evaluation、operationを対象にする。accuracyに加えてsecurity、privacy、fairness、transparency、accountability、costを扱い、技術controlと組織processの両方を必要条件とする。Trustworthy RAG surveyが横断的なrisk taxonomyを整理する一方、Sociotechnical fairness研究はcomponentだけを見て制度・利用者影響を落とす危険を指摘する。system境界を業務まで広げる。

**参照論文**
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)
- [P149] [Fairness and Abstraction in Sociotechnical Systems](https://doi.org/10.1145/3287560.3287598)

## 8.8.2. Stakeholder・RACI・意思決定権

意思決定の空白を避けるため、business owner、document owner、data steward、security、privacy、legal、SRE、model owner、auditorをRACIへ割り当てる。approve、operate、monitor、investigate、retireの各局面で最終責任者を一人にする。内部algorithmic auditの枠組みが示すように、accountabilityは報告書ではなく開発から運用までの役割と証跡の連鎖である。例外承認とrisk acceptanceには署名者、期限、再審査条件を付ける。

**参照論文**
- [P143] [Closing the AI Accountability Gap: Defining an End-to-End Framework for Internal Algorithmic Auditing](https://arxiv.org/abs/2001.00973)
- [P154] [Model Cards for Model Reporting](https://arxiv.org/abs/1810.03993)

## 8.8.3. Source governance

Source governanceではauthoritative、informal、externalを分類し、owner、trust、更新頻度、利用・引用可否を持つ。connector追加、文書公開、poison疑い、source廃止にapproval flowを設ける。DatasheetsとData Statementsを応用し、corpusの目的、coverage、known gaps、同意、retention、地域・言語偏りを記録する。文書数の多さを充実度と見なさず、正本不明・owner不明のsourceは隔離する。source変更が評価setへ与える影響もreviewする。

**参照論文**
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)
- [P086] [Data Statements for Natural Language Processing: Toward Mitigating System Bias and Enabling Better Science](https://aclanthology.org/Q18-1041/)

## 8.8.4. Use policyとrisk tier

Use caseをimpact、reversibility、data sensitivity、automation level、affected populationでrisk tierへ分類する。低risk FAQと、医療・法務・人事・権限変更では、citation、verification、人間承認、logging、testingの強度を変える。Model Cardsがintended useとlimitationsを明記するように、RAGを最終判断へ使わない範囲を明文化する。高risk queryを費用の安い経路へ自動routingせず、責任ある人へescalateする条件をpolicy化する。

**参照論文**
- [P154] [Model Cards for Model Reporting](https://arxiv.org/abs/1810.03993)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 8.8.5. 文書化とrelease evidence

Releaseの根拠を口頭説明にせず、system card、architecture、threat model、corpus datasheet、evaluation report、known limitations、runbookとして残す。model、prompt、index、policy versionと承認decisionを結び、どのslice・攻撃を試したかを示す。Model CardsとDatasheetsはmodel・dataの報告枠組みを与えるが、RAGでは検索・生成・toolの組合せへ拡張する。利用者向け説明と内部証跡は粒度を分けても、能力と制約を矛盾させない。

**参照論文**
- [P154] [Model Cards for Model Reporting](https://arxiv.org/abs/1810.03993)
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)

## 8.8.6. Change managementとrelease gate

新source、model、embedding、parser、tool、memory policyをchange typeとして登録し、risk assessment、security・privacy review、golden set、canary、rollback準備をrelease gateにする。内部監査研究が求めるend-to-end evidenceを、pull requestから本番versionまで追えるようにする。緊急変更では安全な最小修正を優先し、事後reviewと期限付き例外を必須にする。同時に複数componentを変えず、責任とrollback単位を小さく保つ。

**参照論文**
- [P143] [Closing the AI Accountability Gap: Defining an End-to-End Framework for Internal Algorithmic Auditing](https://arxiv.org/abs/2001.00973)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)

## 8.8.7. Human oversight・escalation・appeal

Human oversightは「最後に人が見る」という飾りではない。低信頼、未解決矛盾、権限不明、高impact actionを自動でreview queueへ送り、reviewerに根拠、version、uncertainty、変更履歴を提示する。承認・却下理由を構造化し、同じcaseが再発したときに参照できるようにする。利用者には誤り・不適切回答・source問題を報告し、訂正状況を追うappeal経路を用意する。人へ送るだけで責任を曖昧にせず、最終decision ownerを定める。

**参照論文**
- [P143] [Closing the AI Accountability Gap: Defining an End-to-End Framework for Internal Algorithmic Auditing](https://arxiv.org/abs/2001.00973)
- [P149] [Fairness and Abstraction in Sociotechnical Systems](https://doi.org/10.1145/3287560.3287598)

## 8.8.8. Fairness・coverage・定期review

定期reviewでは、language、region、department、document type、user group別にretrieval、answer、refusal、latencyをsliceする。Data Statementsがデータの社会的・言語的背景を記すように、corpusに存在しない声や古いpolicyが結果へ与える影響を確認する。KPI達成だけでなくincident、complaint、over-refusal、source gap、環境・費用、残余riskを議題にする。改善不能・利用価値低下なら縮小や廃止も選び、運用継続を既定路線にしない。

- [P086] [Data Statements for Natural Language Processing: Toward Mitigating System Bias and Enabling Better Science](https://aclanthology.org/Q18-1041/)
- [P149] [Fairness and Abstraction in Sociotechnical Systems](https://doi.org/10.1145/3287560.3287598)
