# 8.5 性能と可用性

## 8.5.1. SLI・SLO・error budget

性能目標は平均応答時間一つでは表せない。time-to-first-token、final answer、retrieval、tool latency、availabilityをSLIとし、interactive QA、batch report、agentic workflowで別SLOを持つ。The Tail at Scaleは多数componentを組み合わせるほどrare latencyがend-to-end tailへ増幅することを示した。p50だけでなくp95・p99を追い、品質を下げたdegraded responseを成功と数える条件も明記する。error budgetは安全controlを外す口実にしない。

**参照論文**
- [P162] [The Tail at Scale](https://research.google/pubs/the-tail-at-scale/)

## 8.5.2. End-to-end latency budget

End-to-end SLOからauth、rewrite、retrieve、rerank、pack、LLM、verify、toolへdeadlineを配る。multi-queryのfan-outやretryは個別平均が短くてもcritical pathを伸ばし、最遅dependencyが全体を決める。Tail at Scaleの知見を踏まえ、parallel callのp99とcold start、長いqueryを含めて測る。各stageへ残り時間を伝播し、予算切れなら安全なfallbackか回答保留へ進む。後段へ時間を残さない過剰検索を防ぐ。

**参照論文**
- [P162] [The Tail at Scale](https://research.google/pubs/the-tail-at-scale/)
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)

## 8.5.3. Retrieval/index性能

Retrieval性能はcorpus規模、filter selectivity、dimension、top-k、metric、ANN parameterの組合せで決まる。HNSWでは`M`、`efConstruction`、`efSearch`をsweepし、recall、latency、memoryを同時に見る。FAISSのbillion-scale研究がGPUによる高速化を示しても、転送・容量・費用はworkload依存である。ACL filterを外したbenchmarkは本番性能ではない。権限条件を含むquery分布でp95と候補不足を測る。

**参照論文**
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)
- [P016] [Billion-scale similarity search with GPUs](https://arxiv.org/abs/1702.08734)
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)

## 8.5.4. Rerank・context・LLM性能

Rerank候補数、context token、output token、model sizeは一つの直列budgetを共有する。候補を増やせばreranker時間と入力tokenが増え、圧縮を強めれば根拠を失う。LLMLinguaはprompt圧縮で推論を高速化する可能性を示し、PagedAttentionはKV cache管理でserving throughputを高める。batching、parallelism、streaming、prompt cacheを比較し、small model・compression導入時はfaithfulnessとcitationを必ず回帰評価する。

**参照論文**
- [P036] [LLMLingua: Compressing Prompts for Accelerated Inference of Large Language Models](https://arxiv.org/abs/2310.05736)
- [P146] [Efficient Memory Management for Large Language Model Serving with PagedAttention](https://arxiv.org/abs/2309.06180)

## 8.5.5. Capacity・throughput・quota

CapacityはLLMのQPSだけでなく、vector store、provider quota、tool backend、queueの最小能力で決まる。同時実行、token/s、queue depth、burst、rate limitをtenant別に見積もり、noisy neighborを抑えるquotaを設ける。Agentic flowでは一requestが複数tool callを生むため、query数ではなくstep数分布も使う。通常、peak、障害時fallbackの需要を分け、backpressureで下流を守る。容量超過時に認可・citationを省く経路は作らない。

**参照論文**
- [P162] [The Tail at Scale](https://research.google/pubs/the-tail-at-scale/)
- [P146] [Efficient Memory Management for Large Language Model Serving with PagedAttention](https://arxiv.org/abs/2309.06180)

## 8.5.6. Timeout・retry・circuit breaker

Timeoutとretryは障害を隠すのではなく、連鎖を止める仕組みである。全dependencyへ絶対deadlineを渡し、idempotentなreadだけをjitter付きで限定再試行する。write toolはidempotency keyと結果照合なしに再実行しない。継続障害ではcircuit breakerを開き、queueへbackpressureをかける。retry stormがtail latencyとcostを増やすため、回数より残りbudgetで判断し、失敗理由を利用者へ安全に伝える。

**参照論文**
- [P162] [The Tail at Scale](https://research.google/pubs/the-tail-at-scale/)
- [P098] [MillWheel: Fault-Tolerant Stream Processing at Internet Scale](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)

## 8.5.7. Fallbackとdegraded mode

Fallbackは「何でも答える」経路ではない。reranker障害なら認可済みfirst-stage順位、large model障害なら評価済みsmall model、tool unavailableなら現状を明示して拒否する。根拠を取得できないときLLM内部知識だけへ落とす場合は、低risk用途に限定し表示を変える。安全性、ACL、citationを外すfallbackは禁止する。各degraded modeの期待品質とSLOを事前評価し、通常経路へ戻る条件とstale cache許容時間を定義する。

**参照論文**
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)
- [P075] [Retrieval Augmented Generation or Long-Context LLMs? A Comprehensive Study and Hybrid Approach](https://arxiv.org/abs/2407.16833)

## 8.5.8. 性能・可用性テスト

性能試験はsteady loadだけでなく、stress、soak、spike、failure injection、region・dependency outageを含める。大文書、低selectivity filter、multi-hop、長会話、tool timeoutをworst caseにし、recovery time、data loss、duplicate action、stale cacheを確認する。RAGGEDが検索深度とreader安定性の関係を示したように、負荷時の候補削減が回答品質をどう変えるかも測る。障害注入は隔離環境とcanaryから始める。

**参照論文**
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)
- [P162] [The Tail at Scale](https://research.google/pubs/the-tail-at-scale/)
