## 3.7.1. Index構築とライフサイクルの役割
- indexを、検索可能なデータ構造とそのmanifestの組と定義する。
- sparseでは転置index、denseではvector ANN indexを使う。
- index構築は単なるuploadではなく、再現可能なbuild工程だと説明する。

## 3.7.2. exact searchとANN
- brute-force exact k-NNをbaselineとして説明する。
- corpus規模、query量、latency SLOによりANNが必要になる条件を示す。
- ANNは正解保証を緩めて速度を得るため、recall評価が必須である。
- build time、memory、update、filterも方式選定へ含める。

## 3.7.3. HNSW
- small-world graphと階層探索の直感を図解する。
- `M`、`efConstruction`、`efSearch`の役割を説明する。
- 値を上げた場合のrecall、latency、memory、build timeの変化を示す。
- metadata filterとの組み合わせで候補不足が起きる可能性を扱う。

## 3.7.4. IVFとProduct Quantization
- coarse quantizerで候補領域を絞るIVFを説明する。
- PQでvectorを圧縮し、メモリ・I/Oを削減する。
- `nlist`、`nprobe`、code sizeを評価パラメータとして示す。
- 圧縮誤差とreranking用original vector保持の選択肢を示す。

## 3.7.5. GPU・disk-based ANN
- GPUはbuild/searchを高速化できるが、転送、容量、費用を伴う。
- DiskANNを例に、SSDとRAMの分担を説明する。
- workloadが低頻度・大規模保管か、高頻度・低latencyかで選ぶ。
- hardware benchmarkは自社vector分布とqueryで測る。

## 3.7.6. index schema
- vector dimension、metric、normalization、field mappingを固定する。
- text、title、heading、metadata、ACLをどのfieldへ置くか決める。
- filterable/non-filterable metadataを分ける。
- schema変更が再indexを必要とする条件を明記する。

## 3.7.7. index manifestと再現性
- `index_version`、source snapshot、parser version、chunker version、embedding modelを記録する。
- build code、configuration、random seed、build日時、件数、hashを保存する。
- 文書件数、chunk件数、vector件数が一致するか検査する。
- manifestから同じindexを再構築できるようにする。

## 3.7.8. 初回build pipeline
- validate、deduplicate、embed、bulk load、optimize、verifyの段階を分ける。
- partial failure、retry、checkpoint、idempotencyを設計する。
- build中indexを本番queryから隔離する。
- sample queryとgolden setでsmoke testしてから公開する。

## 3.7.9. incremental updateとfreshness
- append、document replacement、metadata-only updateを分ける。
- insert/deleteがANN graphへ与える影響を説明する。
- tombstone、compaction、rebuildの閾値を決める。
- source更新から検索反映までのfreshness SLOを監視する。

## 3.7.10. deletion propagationとACL変更
- 文書削除、失効、権限剥奪を検索結果から確実に消す。
- cache、replica、旧indexにも削除が反映されることを確認する。
- deletion lagをセキュリティ指標として扱う。
- soft deleteとphysical deleteの期限を定義する。

## 3.7.11. blue/green indexとrollback
- 新indexを別名で構築し、評価後にalias/routerを切り替える。
- cutover前後でquery shadowingまたはA/B評価を行う。
- 旧indexを一定期間保持し、即時rollbackできるようにする。
- embedding空間やschemaが異なるindexを混在検索しない。

## 3.7.12. consistencyとcloud-native vector DB
- write path、read path、WAL、snapshot、MVCCの役割を概説する。
- strong consistencyが必要な権限変更と、eventual consistencyを許容する追加文書を分ける。
- replica lag、segment compaction、failure recoveryを運用要件にする。
- vector DBの製品機能を、indexアルゴリズムと混同しない。

## 3.7.14. observability
- index size、active version、freshness、failed ingestion、embedding errorを監視する。
- deleted-but-searchable件数、ACL propagation lagを監視する。
- recallを直接オンライン監視できないため、canary queryを定期実行する。
- index build、cutover、rollbackを監査ログへ残す。

## 3.8.1. Batch品質チェックの位置付け
- ingestion jobの技術的成功と、公開可能な知識indexの完成を区別する。
- quality gateを`pass / warn / quarantine / fail`に分ける。
- 全件検査とsample検査の対象を定義する。

## 3.8.2. manifest整合性
- source snapshot、parser version、chunker version、embedding model、index versionを確認する。
- document件数、chunk件数、vector件数、index登録件数の対応を検査する。
- build config、schema version、実行時刻、code revision、hashを保存する。

## 3.8.3. source・文書レベル検査
- owner不明、版不明、更新日不明、機密区分不明を検知する。
- 失効文書、draft、削除済み文書、取得対象外形式の混入を検知する。
- 取得予定件数と実取得件数の差分を確認する。

## 3.8.4. parse品質
- parse成功率、空出力率、極端に短い出力、文字化け、重複ページを検査する。
- heading、reading order、page、table、caption、bboxの保持率をsample評価する。
- OCR confidenceやlayout confidenceの低い文書を隔離する。

## 3.8.5. chunk品質
- token長の分布、極小・極大chunk、overlap率、重複率を測る。
- headingだけ、footerだけ、ページ番号だけのchunkを検知する。
- FAQのQ/A分離、表の行列崩壊、コードの途中分断など文書種別固有のruleを持つ。
- parent-child参照や元block参照の欠損を検査する。

## 3.8.6. metadata完全性
- `document_id`、`chunk_id`、source、version、page、section、languageを検査する。
- `effective_from/to`、document status、source trustの型と整合性を確認する。
- filterable metadataのkey数・型・値サイズがindex制約内か確認する。

## 3.8.7. ACL品質とnegative test
- tenant、group、role、classificationの欠損をdeny-by-defaultで扱う。
- 文書ACLからchunk ACLへの継承数を検査する。
- 代表ユーザーでtenant横断、権限剥奪、期限切れ、NULL属性をnegative testする。
- active indexだけでなくstaging・cache・旧indexにも権限外chunkが残らないか確認する。

## 3.8.8. 重複・競合・鮮度
- exact duplicateとnear duplicateを分けて測る。
- 同一文書の複数版が同時にactiveになっていないか確認する。
- source更新時刻、ingested_at、indexed_at、published_atからfreshness lagを測る。
- 矛盾sourceは自動統合せず、優先規則または隔離対象にする。

## 3.8.9. embedding・vector検査
- dimension、dtype、normalization、NaN/Inf、zero vectorを検査する。
- documentとqueryでmodel revision・prefix・metricが一致しているか確認する。
- 同一content hashで不要なvector重複がないか確認する。
- 代表queryのnearest neighborを人手またはgold labelでspot checkする。

## 3.8.10. index検査
- active version、schema、metric、dimension、document countをmanifestと照合する。
- tombstone、削除済みID、ACL変更済みIDが検索結果に出ないか確認する。
- canary queryで空結果、異常なscore分布、特定sourceへの偏りを検知する。
- exact searchとの比較sampleでANN recall低下を確認する。

## 3.8.11. golden documentと回帰試験
- PDF、表、スキャン、長文、FAQ、複数言語など代表文書を固定する。
- parser・chunker更新前後でblock、chunk、metadata、引用位置をdiffする。
- expected failureもfixture化し、誤って公開されないことを試験する。

## 3.8.12. publish判定と隔離
- security-critical項目は1件でもfail、品質項目は閾値と影響範囲で判定する。
- 隔離文書を除いたpartial publishの可否を明文化する。
- approver、例外期限、rollback条件、監査ログを定義する。

- Batch gate通過後にrecall@k、MRR、nDCG、citation accuracyを測る。
- parse/chunk/metadataの変更とretrieval差分を結び付ける。
- quality gateの閾値が実際のRAG品質に寄与しているか定期的に見直す。
