# 3.1 データソース棚卸し

データソースの棚卸しでは、RAGへ登録できそうな資料を列挙するだけでなく、正本、所有者、版、利用条件、更新方法を確定します。
検索器は、登録されていない正しい資料を取得できず、誤った原文を正しい内容へ変換することもできません。

## 3.1.1 棚卸しが検索品質を決める理由

同じ手順について、正式な規程、部門FAQ、個人メモ、旧版の配布コピーが存在する場面を考えます。
すべてを区別せずに登録すると、質問と最も似た個人メモが、正式規程より上位に現れる可能性があります。
高性能な検索モデルでも、資料の正本性や現在の有効性を、文章の類似度だけから判断できません。

[Data Cascades](https://dl.acm.org/doi/10.1145/3411764.3445518)は、高リスクなAIシステムで、データの収集・管理上の問題が下流へ連鎖する事例を調査しました。
RAGでも、所有者不明、収集範囲不明、アクセス権不明という上流の問題は、検索や生成の調整では直しにくくなります。

棚卸しの初期成果は、資料数ではありません。
正本、適用範囲、所有者、更新責任、利用条件を確認できた情報源の割合です。
結果を情報源台帳へ残すと、誤答が原文に起因する場合に、文書所有者とともに修正できます。

## 3.1.2 情報源台帳

情報源台帳には、名称と保存場所だけでなく、取得・検索・引用・削除の判断に必要な情報を記録します。

表3-1は、台帳項目を識別、責任、範囲、更新、利用、品質、運用に分けています。
左列で確認したい目的を選び、右列の項目を台帳に記録できているかを確認します。

**表3-1　情報源台帳に記録する項目**

| 分類 | 主な項目 |
|---|---|
| 識別 | 情報源ID、名称、保存場所、形式、言語 |
| 責任 | 所有者、データ管理者、公開承認者、問い合わせ先 |
| 範囲 | 収集目的、対象部門、地域、期間、文書件数 |
| 更新 | 更新方式、頻度、版の表現、削除の通知方法 |
| 利用 | 機密区分、ライセンス、引用可否、ログ保存可否 |
| 品質 | 既知の欠陥、欠損、重複、正本との関係 |
| 運用 | 保持期間、削除期限、障害時の復旧方法 |

[Datasheets for Datasets](https://arxiv.org/abs/1803.09010)は、データセットの作成目的、構成、収集、利用、保守などを記録する枠組みを提案しました。
この考え方を情報源へ適用すると、登録後に「この資料は何のために集めたのか」「誰が更新するのか」を追跡できます。

Wiki、FAQ、チケット、データベース、画像を同じ文書種別として扱ってはいけません。
取得方法、更新方法、検索方法、引用可能な範囲が異なるためです。
情報源の追加時には、台帳の変更、情報源へ接続する取り込み用プログラム（コネクター）の設定、承認記録を結び付けます。

## 3.1.3 正本と権威性

**正本（source of truth）**は、特定の事実や規則について、組織が優先して参照する情報源です。
同じ事実を複数の情報源が記述している場合、検索の関連度スコアで正本を決めてはいけません。

情報源は、正式、参考、派生、外部などに分類できます。
例えば、正式規程を最優先とし、部門FAQは規程の説明、生成要約は検索補助として扱います。
優先関係をメタデータと運用方針へ記録し、LLMの多数決に任せません。

権威性と引用可否は別の属性です。
内部の判定には使える正本でも、契約や機密性の理由で回答画面へ原文を表示できない場合があります。
逆に、公開資料でも、組織の正式な判断根拠としては使えない場合があります。

矛盾が解消しない場合の確認先と所有者を台帳へ結び付けます。
回答時に自動で一方を選べない場合は、矛盾を示して確認へ回します。

## 3.1.4 採用・除外基準

情報源の採用は、ファイルを読み取れるかだけで判断しません。
正確性、完全性、鮮度、権限、引用可能性、機密性、ライセンス、保守責任を確認します。
下書き、失効文書、空ファイル、解析不能な資料、機械生成ノイズは除外または隔離の候補です。

[Data Statements](https://aclanthology.org/Q18-1041/)は、自然言語データについて、言語、地域、話者、収集条件などを明示することを提案しました。
特定地域や特定顧客だけの資料を、全社・全顧客向けの正本と誤認しないために、対象範囲を記録します。

自動採用できる情報源と、所有者の承認を必要とする情報源を分けます。
例外的に採用する場合は、理由、承認者、期限、再確認日を記録します。
採否理由を残すと、後から必要になった資料について、当時の判断を再検討できます。

## 3.1.5 版、有効期間、鮮度

「最新版」は、一つの更新日時だけでは表せません。
少なくとも、公開日時、更新日時、適用開始日、適用終了日、取り込み日時を分けます。
未来に施行される規程は公開済みでも、現在の回答に適用できない場合があります。

「現在有効な規程」と「2024年4月時点で有効だった規程」では、選ぶ版が異なります。
質問時点を検索条件へ含め、過去版を保持するか、失効後に隔離するかを決めます。
[時間を考慮したRAGの研究](https://arxiv.org/abs/2401.13222)も、時間情報を検索と生成へ取り込む必要性を扱っています。

[FreshLLMs](https://arxiv.org/abs/2310.03214)は、変化の速い知識を含む質問を評価し、検索拡張を利用しました。
必要な鮮度は、情報の種類によって異なります。
障害情報は数分、規程は施行日前、研究資料は版更新後など、情報源ごとに許容遅延を定めます。

情報源の変更から稼働中インデックスへ反映されるまでの時間を測ります。
文書の日時が新しくても、取り込みが止まっていれば回答は古いままです。

## 3.1.6 重複、派生、同一性

URLやファイル名が異なっても、内容が同じ配布コピーが存在します。
同じ内容が検索上位を占めると、候補件数は増えても、異なる根拠は増えません。
一方で、旧版と新版には差分の意味があるため、単純に一つへまとめてはいけません。

正規文書、版、複製、派生成果物の関係を管理します。
原文のバイト列を短い値へ変換したハッシュと、空白などを正規化した内容ハッシュを使い分けます。
[文書の類似性と包含関係の研究](https://ieeexplore.ieee.org/document/666900)で用いられた、連続する語や文字のまとまりを照合するシングリングなどは、近い文書の検出に利用できます。

同じ本文でも、所有者、有効期間、ACLが異なれば、同じ管理対象とは限りません。
重複をまとめる場合も、元の情報源、版、権限、引用先を失わないようにします。

## 3.1.7 静的文書と動的データ

規程やマニュアルは、更新間隔を管理して文書インデックスへ登録できます。
在庫、残高、注文状態、現在の障害状況は、取得した直後から変わる可能性があります。
動的な事実を文書スナップショットだけから回答すると、古い値を現在値として示す危険があります。

[MRKL Systems](https://arxiv.org/abs/2205.00445)は、言語モデルから検索、計算、データベースなどの専門モジュールへ処理を振り分ける考え方を示しました。
静的な知識は文書検索、動的な事実はデータベースまたはAPIへ送ります。

スナップショットを検索対象にする場合は、取得時刻、更新周期、許容される遅延を明示します。
画面と回答にも基準時刻を示し、現在値だと誤認させません。
同じ項目を文書とAPIの二つから回答しないよう、正本と振り分け先を台帳へ記録します。

## 3.1.8 所有者と変更責任

文書の所有者、データ管理者、公開承認者、問い合わせ先は、同じ人物や部署とは限りません。
版更新、誤記訂正、削除、ACL変更について、誰が決定し、誰が実施し、誰が承認するかを分けます。

所有者へ連絡できない場合の処理も必要です。
所有者不明の情報源を自動取り込みせず、隔離します。
業務上必要な例外利用には、利用範囲、承認者、失効期限を付けます。

[内部アルゴリズム監査の枠組み](https://arxiv.org/abs/2001.00973)は、システムの企画から廃止までの全期間を通じた説明責任を扱いました。
RAGの誤答をモデル担当だけの問題にせず、原文の所有者、取り込み担当、検索・生成担当へ適切に戻す経路を作ります。

## 3.1.9 機密区分と利用条件

「公開」「社内」「機密」「厳秘」という分類だけでは、RAGで何をしてよいか判断できません。
取得、保存、解析、インデックス化、検索、回答への引用、ログ保存、評価利用の可否を目的別に定めます。

個人情報、契約上の秘密、越境移転、著作権、ライセンスなどの条件を情報源台帳へ記録します。
原文は保存できても外部モデルへ送信できない、検索結果には使えても回答画面へ引用できないという条件もあります。

分類や利用条件が不明な資料を、低リスクと推測してはいけません。
安全側に隔離し、所有者と管理部門が判断します。
情報の一部を隠す処理（マスキング）の後に利用できる場合は、除去対象、変換方法、復元可能性、再識別リスクを別に評価します。

## 3.1.10 小さな本番基準

最初の検索対象となる文書集合（コーパス）は、網羅性より検証可能性を優先します。
一つの製品、一つの部門、一つの業務、一つの文書種別など、人が対象資料を確認できる範囲へ絞ります。

対象内の正本と、登録してはいけない資料を列挙します。
所有者、版、有効期間、ACL、引用位置が揃うかを全件または統計的に十分な範囲で確認します。
対象外の情報源と、次に追加する条件も明文化します。

[ML Test Score](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)は、学習済みモデルだけでなく、データとパイプラインを含む本番準備を評価する観点を示しました。
RAGでも、小さな基準で正解根拠と失敗を固定してから資料を追加すると、変更による品質差を測れます。

対象範囲を限定することは、単なる機能不足ではありません。
資料の追加が検索、引用、権限へ与えた影響を比較できる条件を作る設計です。
