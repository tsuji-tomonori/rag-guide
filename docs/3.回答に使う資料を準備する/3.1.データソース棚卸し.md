# 3.1 データソース棚卸し

## 3.1.1. 棚卸しが検索品質の上限を決める

棚卸しは事務作業ではなく、検索品質の上限を決める設計工程である。Data Cascadesが示すように、owner不明や収集条件の曖昧さは下流で増幅し、model調整では直しにくい。古い版、配布コピー、非公式メモ、権限不明文書が混ざった具体例から始め、なぜ高性能retrieverでも誤根拠を選ぶかを説明する。資料数より、正本、適用範囲、更新責任を確定した割合を初期成果とする。 棚卸し結果をsource台帳として残せば、後の誤答を文書所有者とともに修正できる。

**参照論文**
- [P091] [Everyone wants to do the model work, not the data work: Data Cascades in High-Stakes AI](https://dl.acm.org/doi/10.1145/3411764.3445518)
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)

## 3.1.2. データソース台帳の最小項目

データソース台帳は名称一覧では足りない。Datasheets for Datasetsにならい、source_id、owner、保存場所、言語、件数に加え、収集目的、更新方式、保持期間、削除要件、既知の欠陥を記録する。Wiki、FAQ、チケット、DB、画像を同じ文書種別へ押し込まず、取得・検索・引用の可否を分ける。台帳をconnector設定とrelease判定の起点にし、source追加時に誰が何を承認したか追えるようにする。

**参照論文**
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)

## 3.1.3. source of truthと権威性

同じ事実を複数sourceが語るとき、関連度scoreで正本を決めてはいけない。authoritative、informal、derived、externalを区分し、正式規程、部門FAQ、生成要約の優先関係を事前に定める。source trustと回答で引用可能かは別属性にし、非公開の正本を内部判定には使えてもUIへ表示できない場合を扱う。矛盾時の最終確認先とownerを台帳へ結び、LLMの多数決を避ける。 同じ内容でも正式性と引用可否は別属性とし、関連度scoreへ権威性を埋め込まない。

**参照論文**
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)

## 3.1.4. 採用・除外基準

採用基準は「読める形式か」より広い。正確性、完全性、鮮度、権限、引用可能性、機密性、ライセンスで評価し、draft、失効文書、空ファイル、機械生成noiseを除外候補にする。Data Statementsが言語・地域・収集条件の明示を求めた視点を取り入れ、特定地域だけの資料を全社向け正本と誤認しない。自動採用できるsourceと、owner承認が必要なsourceを分け、例外には期限を付ける。 採否理由を記録することで、除外された資料が後から必要になった場合も判断を再検討できる。

**参照論文**
- [P086] [Data Statements for Natural Language Processing: Toward Mitigating System Bias and Enabling Better Science](https://aclanthology.org/Q18-1041/)
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)

## 3.1.5. 版・有効期間・鮮度

「最新版」は一つの日付fieldでは表せない。published_at、updated_at、effective_from、effective_to、ingested_atを分け、現在有効な版と質問時点で有効だった版を別の検索条件にする。FreshLLMsがfast-changing knowledgeを評価したように、鮮度は回答対象ごとに変わる。stale判定、未来施行、失効文書の隔離、過去版の保管方針を決め、source更新からactive index反映までのlagを測る。

**参照論文**
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)
- [P152] [It's About Time: Incorporating Temporality in Retrieval Augmented Language Models](https://arxiv.org/abs/2401.13222)

## 3.1.6. 重複・派生・同一性

ファイル名やURLが違っても内容が同じ配布コピーは、top-kを占有して多様性を損なう。一方、旧版と新版は差分に意味があるため、単純なcontent hashで統合してはいけない。canonical document、version、mirror、derived artifactの関係を持ち、raw hashと正規化hashを使い分ける。owner、有効期間、ACLが違えば同一本文でも別の管理対象とし、重複排除でprovenanceを失わない。

**参照論文**
- [P130] [On the Resemblance and Containment of Documents](https://ieeexplore.ieee.org/document/666900)
- [P122] [Deduplicating Training Data Makes Language Models Better](https://arxiv.org/abs/2107.06499)

## 3.1.7. 静的文書と動的データの境界

規程やマニュアルは文書indexへ向くが、在庫、残高、障害状態は取得した瞬間に古くなり得る。MRKLのsystems approachを借り、静的知識はretriever、動的事実はDB/APIへroutingする。snapshotをindex化する場合は取得時刻、更新周期、許容lagを明示し、最新値として見せない。文書RAGとstructured routeの境界を台帳に記し、同じ項目を二つの正本から回答しないようにする。

**参照論文**
- [P072] [MRKL Systems: A modular, neuro-symbolic architecture that combines large language models, external knowledge sources and discrete reasoning](https://arxiv.org/abs/2205.00445)
- [P135] [StreamingQA: A Benchmark for Adaptation to New Knowledge over Time in Question Answering Models](https://arxiv.org/abs/2205.11388)

## 3.1.8. 所有者と変更責任

文書owner、data steward、公開承認者、問い合わせ先は同一とは限らない。版更新、誤記訂正、削除、ACL変更の責任主体を分け、連絡不能時の扱いを決める。Datasheetsが保守責任を文書化するのは、データが作成時点で完結しないからである。owner不明sourceは自動取り込みせずquarantineし、期限付き例外だけで利用する。RAGの誤答をmodel teamだけへ返さず、原文の改善loopへつなげる。

**参照論文**
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)
- [P143] [Closing the AI Accountability Gap: Defining an End-to-End Framework for Internal Algorithmic Auditing](https://arxiv.org/abs/2001.00973)

## 3.1.9. 機密区分と利用条件

public、internal、confidential、restrictedという分類を付けるだけでは不十分である。取得可能、index化可能、検索可能、回答で引用可能、ログ保存可能を目的別に判断する。Data Statementsの利用条件とTrustworthy RAGのprivacy観点を組み合わせ、個人情報、契約、越境、ライセンスをsource台帳へ残す。分類不明は低リスク扱いせず安全側に隔離し、redaction後に使える範囲も別途定義する。

**参照論文**
- [P086] [Data Statements for Natural Language Processing: Toward Mitigating System Bias and Enabling Better Science](https://aclanthology.org/Q18-1041/)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 3.1.10. 小さく始めるProduction Baseline

最初のcorpusは網羅性より検証可能性を優先する。1製品、1部門、1業務、1文書種別へ絞り、gold sourceと除外sourceを人が全件確認できる規模にする。ここで、owner、版、ACL、引用位置が揃うかを試し、対象外sourceと追加条件を明文化する。小さなbaselineで正解根拠とfailureを固定してから拡張すれば、資料追加による品質変化を測れる。最初から全社検索を目標にしない。 範囲を限定することは機能不足ではなく、資料追加による因果を測れる実験条件を作る行為である。

**参照論文**
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)
