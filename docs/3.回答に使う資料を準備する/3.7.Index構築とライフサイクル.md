# 3.7 インデックスの構築と運用

インデックスは、ベクトルや語を保存する場所だけではありません。
検索可能なデータ構造、設定、元データ、版を記録した構成一覧（マニフェスト）を一つの成果物として管理します。
構築、更新、削除、切り替え、復旧までを一続きの運用として扱います。

## 3.7.1 インデックスの役割

キーワード検索では、語から文書へ対応する転置インデックスを利用します。
意味検索では、埋め込みベクトルから近い候補を探すために、完全探索または近似最近傍探索（ANN）のデータ構造を利用します。
ハイブリッド検索では、二つのインデックスが同じ文書版とチャンクIDを参照する必要があります。

構築は、データをアップロードする一工程ではありません。
入力検査、登録、最適化、検証、公開へ分けます。
元の情報源スナップショットから、同じ設定と件数で再構築できるようにします。

稼働中のインデックスへ更新を追加するだけでは、削除、権限変更、圧縮、性能劣化を管理できません。
どの版が検索に使われ、どの変更が反映済みかを追跡できる状態を作ります。

## 3.7.2 完全探索とANN

**完全探索（exact search）**は、質問ベクトルをすべての文書ベクトルと比較します。
データ量が小さい場合や評価用の一部データでは、正しい近傍を得る基準として利用できます。
計算量はベクトル数に比例するため、規模と要求数が増えると応答時間と計算資源が増えます。

ANNは、探索範囲を絞り、真の近傍を一部見落とす可能性と引き換えに検索を高速化します。
必要かどうかは、コーパス件数、同時要求数、応答時間目標、利用できるメモリから決めます。
ANNを使うこと自体を最適化と見なさず、完全探索と比較して近傍再現率を測ります。

方式選定では、質問時の応答時間だけでなく、構築時間、メモリ、保存容量、更新、削除、フィルターを比較します。
一部の標本で完全探索を継続し、ANN設定変更による品質差を測れる参照経路を残します。

## 3.7.3 HNSW

**Hierarchical Navigable Small World（HNSW）**は、近いベクトルをグラフで結び、階層的に探索するANN手法です。
[HNSWの論文](https://arxiv.org/abs/1603.09320)は、階層化した小世界グラフによる近傍探索を提案しました。

代表的な設定には、ノード当たりの接続数 `M`、構築時の探索幅 `efConstruction`、質問時の探索幅 `efSearch` があります。
値を大きくすると近傍再現率が上がる場合がありますが、構築時間、メモリ、応答時間も増えます。
設定間の関係は実装に依存するため、製品の既定値を無条件に採用しません。

[LuceneのHNSWをAnseriniへ統合した研究](https://arxiv.org/abs/2304.12139)は、LuceneとFaissについて、検索品質、構築時間、処理量、インデックス容量を比較しました。
この比較では質問や文書をベクトルへ変換する時間を含まないため、実務ではベクトル化から検索までの応答時間も別に測ります。

メタデータフィルターで候補が少なくなると、グラフ上で条件に合う近傍へ到達しにくくなる場合があります。
[ACORN](https://arxiv.org/abs/2403.04871)のようにフィルター条件を考慮する探索も候補です。
[NaviX](https://arxiv.org/abs/2506.23397)は、フィルターを満たす候補の割合に加え、その候補がベクトル空間のどこへ集まるかに応じて探索方法を切り替えました。
ただし、切り替え値は評価データに基づく設定であり、更新と削除は主な評価対象ではありません。
フィルター後に残る割合と候補の分布ごとに、再現率、応答時間、更新時の維持費用を測ります。

## 3.7.4 IVFと積量子化（Product Quantization）

**Inverted File Index（IVF）**は、ベクトルを粗い中心へ割り当て、質問時に近い領域だけを探索します。
中心数や探索する領域数によって、速度と再現率が変わります。

**Product Quantization（PQ）**は、ベクトルを部分ベクトルへ分け、各部分を短いコードで表して保存量と距離計算を減らします。
[Product Quantization](https://doi.org/10.1109/TPAMI.2010.57)は、大規模な近傍探索のための圧縮方法を提案しました。
圧縮によって元ベクトルとの距離へ誤差が入ります。

IVFでは中心数と探索領域数、PQではコード長などを変えて評価します。
圧縮ベクトルで候補を取得し、元ベクトルで再順位付けする構成もあります。
完全探索との再現率差、保存容量、応答時間、引用に必要な上位順位の変化を測ります。

## 3.7.5 GPUとディスクを使うANN

GPUは大量のベクトル距離計算を並列化でき、インデックス構築や検索を高速化できる場合があります。
[FaissのGPU近傍探索](https://arxiv.org/abs/1702.08734)は、10億規模のベクトル検索を扱いました。
GPUメモリ、CPUとの転送、装置費用、障害時の代替経路が制約です。

[DiskANN](https://www.microsoft.com/en-us/research/publication/diskann-fast-accurate-billion-point-nearest-neighbor-search-on-a-single-node/)は、SSDとメモリを組み合わせた大規模ANNを扱います。
頻度は低いが規模が大きい検索では、すべてのベクトルを高価なメモリへ置かない方式が候補です。

公開ベンチマークの値だけで方式を選びません。
対象ベクトルの分布、フィルター、要求数、目標応答時間で測ります。
計算機の障害時における復旧、再構築、機能を限定した運転も比較します。

## 3.7.6 インデックススキーマ

インデックススキーマには、ベクトル次元数、距離尺度、正規化、本文項目、見出し、ACL、絞り込みに使えるメタデータを定義します。
キーワード検索では、分かち書き、辞書、項目ごとの重みも含めます。

質問エンコーダーと文書ベクトルが、同じ埋め込み版と前処理を利用することを検査します。
距離尺度や次元数が違う場合、登録できても検索結果に意味はありません。

値の種類が多い項目や、大きいメタデータを無制限に検索用の保存先へ入れません。
検索、画面表示、監査の保存先を分けます。
スキーマ変更が既存検索と互換か、全件再インデックスを必要とするかを定めます。

製品の既定値を暗黙に使わず、設定値と選択理由をマニフェストへ残します。

## 3.7.7 マニフェストと再現性

インデックス名に作成日を入れるだけでは、同じ成果物を再構築できません。
次の情報をマニフェストへ保存します。

- インデックス版とスキーマ版
- 情報源スナップショットと対象件数
- 解析器、分割器、埋め込みモデルの版
- 次元数、正規化、距離尺度、ANN設定
- 構築コード、設定、実行環境、必要な乱数の初期値
- 文書、チャンク、ベクトル、登録済み件数
- 各段階のハッシュ、警告、隔離件数

文書、チャンク、ベクトル、インデックス登録件数の差を説明できる必要があります。
例えば、埋め込みに失敗したチャンクは隔離一覧と一致させます。

[Hidden Technical Debt](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)が構成管理の負債を論じたように、管理画面で手動変更した設定もマニフェストへ取り込みます。
コード管理外の値がある場合は、再現性が未達であることを公開判定へ伝えます。

## 3.7.8 初回構築

初回構築は、入力検査、重複処理、埋め込み、一括登録、最適化、検証を段階に分けます。
各段階にチェックポイントと冪等性を持たせ、途中失敗から安全に再開します。

構築中インデックスは、利用者の検索経路から隔離します。
文書・チャンク・ベクトル件数、代表的な近傍、ACLの否定試験、正解質問を検査してから公開します。
失敗文書を黙って落とさず、隔離して情報源と影響範囲を示します。

大規模登録の速度だけを成功基準にしません。
どの情報源が欠け、どの質問へ影響するかを再現できることを優先します。

## 3.7.9 増分更新と鮮度

増分更新では、新規追加、文書の置換、メタデータのみの変更、削除を区別します。
変更種別に応じて、チャンク再生成、再埋め込み、メタデータ更新、削除を実行します。

グラフ型ANNへ追加と削除を繰り返すと、検索品質と性能が初回構築時から変化する場合があります。
論理削除の蓄積、再配置、圧縮、全件再構築を行うしきい値を設けます。
[FreshDiskANN](https://arxiv.org/abs/2105.09613)と[SPFresh](https://arxiv.org/abs/2410.14452)は、継続的な追加・削除を伴う大規模ベクトル検索を扱いました。

情報源の更新から検索可能になるまでの時間、削除済みだが検索できる件数、ANN再現率、応答時間を監視します。
更新処理が速いことだけでなく、検索品質と削除の正しさを維持していることで評価します。

## 3.7.10 削除とACLの変更

削除、失効、ACL剥奪は、稼働中インデックスだけでなく、待機中インデックス、複製、キャッシュ、旧インデックスへ反映させます。
検索結果、生成コンテキスト、引用画面から消えたことを、権限の異なる試験主体で確認します。

検索から直ちに除外する論理削除と、保存媒体から消去する物理削除の期限を分けます。
ロールバック先にも同じ削除情報を適用し、旧インデックスへ戻したときに文書が復活しないようにします。

追加文書の反映に短い遅延を許容できる場合でも、権限剥奪にはより短い反映時間が必要な場合があります。
変更種別ごとにサービス目標を分け、削除・剥奪の遅延をセキュリティ指標として扱います。

## 3.7.11 青緑切り替えとロールバック

新しいインデックスは別名で構築し、旧版と並行して保持します。
同じ質問を回答へ影響しない形で新旧へ実行し、正解集合、ACL、応答時間、エラーを比較します。
合格後に別名または検索の参照先を新インデックスへ切り替えます。

異なる埋め込み空間やスキーマの候補を、同じ順位一覧へ無計画に混在させません。
候補を統合する必要がある場合は、対応する検索器と統合規則を版管理します。

旧インデックスを定めた期間保持し、品質、応答時間、エラーが基準を外れた場合に戻します。
ロールバックで古いACLや削除済み文書を復活させないよう、データ変更と処理リリースを別々に管理します。

## 3.7.12 分散型ベクトルデータベースの整合性

複数の計算資源へデータを分散するベクトルデータベースは、ANNアルゴリズムだけでなく、書き込み経路、ログ、データ区画、スナップショット、複製、圧縮を含みます。
[Manu](https://arxiv.org/abs/2206.13843)は、継続的な更新とベクトル検索を扱う分散型システムを示しました。

どの変更を、いつ、どの複製で検索できるかを定めます。
新規文書には、短い遅延の後ですべての複製が同じ状態になる結果整合性を許容しても、権限剥奪や削除には強い境界を要求する場合があります。
一つの「整合性」という値ではなく、変更種別と利用者リスクで決めます。

複製の遅延、保存区画（セグメント）の統合・圧縮、障害復旧、失われる可能性のある更新範囲を運用要件へ含めます。
製品名を選ぶ前に、必要な整合性モデル、復旧時間、復旧時点を定義します。

## 3.7.13 容量計画と性能試験

容量計画では、現在の件数だけでなく、文書増加、再構築中の新旧併存、複製、削除待ちのデータを含めます。
ベクトル本体の概算容量は `N × d × b` で求められます。
`N` はベクトル数、`d` は次元数、`b` は一要素当たりのバイト数です。
実際には、グラフや量子化の索引、ID、メタデータ、ログ、複製にも容量が必要なため、対象製品で実測します。

性能試験では、本番に近い次の条件を用意します。

1. 文書数、ベクトル分布、メタデータ、ACLの選択率を本番の見込みに合わせます。
2. 同時質問数、上位取得件数、更新・削除の頻度、質問の長短を組み合わせます。
3. データがメモリにある状態と、ディスクから読み込む状態を分けて測ります。
4. 初回構築、増分更新、全件再構築、複製障害、復旧、青緑切り替えも試験します。

[Faissの研究](https://arxiv.org/abs/1702.08734)と[DiskANN](https://www.microsoft.com/en-us/research/publication/diskann-fast-accurate-billion-point-nearest-neighbor-search-on-a-single-node/)は、使用する計算資源と保存方式によって、大規模近傍探索の速度と容量が変わることを示しました。
公開値をそのまま採用せず、対象の計算機、ベクトル、絞り込み条件で再測定します。

公開前に、完全探索と比べた近傍再現率、50・95・99パーセンタイルの応答時間、処理量、資源使用量、更新反映時間、復旧時間の上限を定めます。
増加予測と再構築に必要な余裕を含めて基準を満たす場合だけ公開します。
基準を外れた場合は、索引設定、データ分割、複製数、計算資源を一つずつ変え、同じ試験を繰り返します。

## 3.7.14 観測可能性

インデックス容量、稼働版、鮮度、取り込み失敗、埋め込みエラーを監視します。
削除済みIDの残存、ACL反映遅延、スコア分布、検索結果ゼロの割合、情報源ごとの候補比率も確認します。

本番では各質問の真の検索再現率を直接観測できません。
固定した試験質問を定期実行し、期待する文書が取得されるかを確認します。
一部の標本では、ANNと完全探索の候補を比較します。

構築、切り替え、ロールバックを監査イベントとして記録します。
異常を検知したときに、情報源、解析、埋め込み、ANN、フィルターのどこで変化したかを分けられる指標を持ちます。
