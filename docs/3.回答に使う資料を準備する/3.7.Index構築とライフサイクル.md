# 3.7 Index構築とライフサイクル

## 3.7.1. Index構築とライフサイクルの役割

indexはvectorの保管箱ではなく、検索可能なdata structureと、その由来を示すmanifestの組である。sparseでは転置index、denseではANNを使い、schema、metric、filter、versionを固定する。構築はuploadではなく、validate、load、optimize、verify、publishの再現可能なbuild工程とする。source snapshotから同じ件数と設定で再構築でき、削除とACL変更を追跡できて初めてlifecycleを持つindexになる。

**参照論文**
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)

## 3.7.2. exact searchとANN

小規模corpusではbrute-force exact k-NNをbaselineにできる。ANNは一部の近傍を見落とす代わりに速度を得るため、必要になる条件をcorpus size、QPS、latency SLOから決める。exact sampleと比較してrecallを測り、build time、memory、update、filter性能も方式選定へ含める。近似を使うこと自体を最適化と見なさず、品質差を数値化できるreference pathを残す。

**参照論文**
- [P016] [Billion-scale similarity search with GPUs](https://arxiv.org/abs/1702.08734)
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)

## 3.7.3. HNSW

HNSWはsmall-world graphを階層的に探索し、高recallと低latencyを両立する代表的ANNである。Mはnodeの接続数、efConstructionはbuild時、efSearchはquery時の探索幅へ効く。値を上げれば一般にrecallは増すが、memory、build time、latencyも増える。metadata filterで有効候補が疎になるとgraph探索が難しくなるため、selectivity別にsweepし、全queryへ高設定を固定しない。

**参照論文**
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)

## 3.7.4. IVFとProduct Quantization

IVFはcoarse centroidで探索領域を絞り、Product Quantizationはsubvectorをcodeへ圧縮してmemoryとI/Oを減らす。nlist、nprobe、code sizeを変えると速度と圧縮誤差が動く。JégouらのPQは大規模近傍探索の基礎を与えたが、citationに必要な精密rankingまで保証しない。圧縮vectorで候補を取り、original vectorでrerankする選択を含め、exact baselineとのrecall差を測る。

**参照論文**
- [P023] [Product Quantization for Nearest Neighbor Search](https://doi.org/10.1109/TPAMI.2010.57)
- [P016] [Billion-scale similarity search with GPUs](https://arxiv.org/abs/1702.08734)

## 3.7.5. GPU・disk-based ANN

GPUはvector距離計算とbuildを高速化できるが、device memory、転送、費用が制約になる。FAISSはbillion-scale similarity searchをGPUで実装し、DiskANNはSSDとRAMを組み合わせて単一nodeで大規模検索を実現した。高頻度・低latencyか、低頻度・巨大保管かでarchitectureを分ける。vendor benchmarkではなく自社vector分布、filter、QPSで測り、hardware障害時のrebuildとfallbackも含めて選ぶ。

**参照論文**
- [P016] [Billion-scale similarity search with GPUs](https://arxiv.org/abs/1702.08734)
- [P089] [DiskANN: Fast Accurate Billion-point Nearest Neighbor Search on a Single Node](https://www.microsoft.com/en-us/research/publication/diskann-fast-accurate-billion-point-nearest-neighbor-search-on-a-single-node/)

## 3.7.6. index schema

index schemaではdimension、metric、normalization、text field、title、ACL、filterable metadataを固定する。高cardinality fieldや大きいmetadataを無制限に入れず、検索、表示、監査の保存先を分ける。schema変更がonline互換か再index必須かを表にし、query encoderとdocument vectorの契約を検査する。製品機能のdefaultを暗黙に使わず、manifestへ値と理由を残す。

**参照論文**
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.7.7. index manifestと再現性

再現性はindex名の日付だけでは得られない。index_version、source snapshot、parser、chunker、embedding model、dimension、metric、build code、config、random seed、件数、hashをmanifestへ保存する。document、chunk、vector、登録件数の対応を検査し、同じartifactを再構築できる手順を持つ。Hidden Technical Debtがconfiguration debtを指摘したように、dashboard外の手動設定もrelease manifestへ取り込む。

**参照論文**
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)

## 3.7.8. 初回build pipeline

初回buildはvalidate、deduplicate、embed、bulk load、optimize、verifyを分離し、checkpointとidempotencyを持たせる。build中indexはquery pathから隔離し、件数整合、sample nearest neighbor、ACL negative、golden queryを通してから公開する。失敗文書を黙って落とさずquarantineし、partial publishの条件を明記する。大規模loadの速さより、どのsourceが欠けたかを再現できることを優先する。

**参照論文**
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)

## 3.7.9. incremental updateとfreshness

incremental updateではappend、document replacement、metadata-onlyを分ける。ANN graphへinsert/deleteが蓄積すると品質と性能が変わるため、tombstone、compaction、full rebuildの閾値を持つ。FreshDiskANNはstreaming insert/delete/searchを扱い、鮮度と検索精度の両立を設計対象にした。source更新から検索反映までのlag、deleted-but-searchable件数、graph劣化を監視し、更新速度だけで成功を判断しない。

**参照論文**
- [P093] [FreshDiskANN: A Fast and Accurate Graph-Based ANN Index for Streaming Similarity Search](https://arxiv.org/abs/2105.09613)
- [P159] [SPFresh: Incremental In-Place Update for Billion-Scale Vector Search](https://arxiv.org/abs/2410.14452)

## 3.7.10. deletion propagationとACL変更

削除、失効、ACL剥奪はactive indexだけでなく、cache、replica、staging、旧indexへ伝播する必要がある。soft deleteとphysical deleteの期限を分け、検索結果から消えたことをcanary principalで確認する。deletion lagをsecurity metricとし、rollback先にも同じtombstoneを適用する。追加eventのeventual consistencyは許容できても、権限剥奪にはより強い整合性を要求する、と変更種別でSLOを分ける。

**参照論文**
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)

## 3.7.11. blue/green indexとrollback

新indexは別名で構築し、query shadowing、golden set、ACL testを通した後にaliasを切り替える。embedding空間やschemaが違うindexを同じrank listへ無計画に混在させない。旧indexを一定期間保持し、quality、latency、errorがgateを外れたら即時rollbackする。ただしrollbackで古いACLや削除済み文書を復活させないよう、data changeとpipeline releaseを分離したmanifestを持つ。

**参照論文**
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)

## 3.7.12. consistencyとcloud-native vector DB

cloud-native vector DBはANN algorithmだけでなく、write path、WAL、segment、snapshot、replica、MVCCを含む。Manuはbinlogやdelta consistencyを使い、検索と継続更新を分離した。追加文書はeventual consistencyでよくても、権限剥奪や削除は強い境界が必要になる。replica lag、compaction、failure recoveryを運用要件へ入れ、製品名を選ぶ前に必要なconsistency modelとrecovery pointを定義する。

**参照論文**
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)

## 3.7.14. observability

index size、active version、freshness、failed ingestion、embedding errorだけでなく、削除済みIDの残存、ACL propagation lag、score分布を監視する。onlineでtrue recallを直接観測しにくいため、固定canary queryを定期実行し、exact sampleとの比較を残す。build、cutover、rollbackを監査eventへし、versionごとのlatencyとzero-result率を追う。異常を検知しても原因がdataかANNかを切り分けられるmetric設計にする。

- [P093] [FreshDiskANN: A Fast and Accurate Graph-Based ANN Index for Streaming Similarity Search](https://arxiv.org/abs/2105.09613)
