# 3.3 文書解析と正規化

文書解析は、ファイルから文字を取り出すだけの処理ではありません。
見出し、段落、表、図、ページ位置を保ち、検索と引用に利用できる共通構造へ変換します。

## 3.3.1 解析と正規化の役割

PDFから文字列を取得できても、二段組みの左右が混ざり、表の列が失われていれば、検索用の根拠としては壊れています。
解析の目的は、原文を検索でき、取得した箇所から元のページへ戻れる構造へ変換することです。
文字の有無だけでなく、読み順、見出し階層、ページ位置、要素の種類を扱います。

[LayoutLMv3](https://arxiv.org/abs/2204.08387)は、テキストと画像領域を統合して事前学習し、複数の文書理解タスクで評価されました。
[PubLayNet](https://arxiv.org/abs/1908.07836)は、文書画像中の本文、見出し、表、図などを検出する大規模データセットを提供しました。
これらの研究は、文書理解に文字以外のレイアウト情報が必要なことを示します。

解析誤りは、後続の分割、埋め込み、検索、引用へ連鎖します。
空文字や文字化けに加え、読み順の逆転、表見出しの欠落、原文位置へ戻れない状態も失敗として扱います。

## 3.3.2 入力形式と失敗モード

拡張子が同じPDFでも、内部構造は異なります。
ソフトウェアから出力したPDFには文字層がある一方、スキャンPDFはページ全体が画像です。
画像と文字層が重なったPDF、暗号化PDF、回転ページ、多段組み、縦書きもあります。

HTML、Word、Excel、スライドも、保持すべき構造が異なります。
HTMLでは文書の構造を表すDocument Object Model（DOM）とリンク、Wordでは見出し・表・変更履歴、Excelではシート・セル・数式、スライドでは要素位置と発表順を扱います。
すべてを一つの「文書」クラスへ押し込まず、形式と内容の特徴を記録します。

[DocLayNet](https://arxiv.org/abs/2206.01062)は、財務、法律、科学、マニュアルなど複数分野の文書レイアウトを人手で注釈しました。
学術論文だけで評価した解析器が、帳票や規程でも同じ性能を持つとは限りません。
画像率、表密度、言語、ページ数、回転、暗号化などから文書クラスを作り、既定解析器の対象外を早期に検出します。

## 3.3.3 共通文書スキーマ

後続処理を特定の解析器へ依存させないため、解析結果を**共通文書スキーマ（canonical document schema）**へ変換します。
文書、ページ、ブロック、表、画像、リンクなどの階層を持たせます。

各ブロックには、種類、本文、ページ番号、ページ内座標、読み順、解析信頼度を記録します。
表には行、列、セル、結合関係、見出しを持たせます。
画像にはキャプションと周辺本文への関係を持たせます。

[LayoutLM](https://arxiv.org/abs/1912.13318)は、文字とページ上の二次元座標を組み合わせ、帳票理解などを評価しました。
この結果は、文字だけでなく位置も保持する設計を支えます。
ただし、共通文書スキーマ自体はLayoutLMの出力形式ではなく、解析器を交換できるようにする本書の運用設計です。

元ファイル、解析器の生出力、正規化後の表現を別々に保存します。
チャンクからブロック、ページ、座標へ戻れる安定IDを用意します。
解析器を交換しても、後続の分割器と引用画面が同じ契約を利用できるようにします。

## 3.3.4 レイアウトと読み順

二段組みの右列を左列へ挟むと、原文にはない文章が生成されます。
各ページのヘッダーやフッターを本文へ挟むと、検索候補へ反復ノイズが入ります。
図のキャプションを別の図へ結び付けると、引用の意味も変わります。

本文、見出し、リスト、表、図、キャプション、脚注、ヘッダー、フッターを領域として識別します。
文字層とページ画像を照合し、ページ内の読み順を復元します。
ページをまたぐ段落や表については、継続条件を明示します。

[PubLayNet](https://arxiv.org/abs/1908.07836)と[DocLayNet](https://arxiv.org/abs/2206.01062)は、レイアウト要素の検出をクラス別に評価できる基盤です。
利用するモデルの学習分野と対象文書の差を確認し、自社資料の代表標本で見出し、表、図、読み順を評価します。
ページ座標は引用画面で該当箇所を強調するために保持します。

## 3.3.5 OCRとOCRを使わない方式

文字層がないページには、光学文字認識（Optical Character Recognition：OCR）が必要です。
一方で、正しい文字層がある文書へOCRを適用し直すと、文字、記号、数値が劣化する場合があります。
まずページ単位でOCRの要否を判定します。

OCR結果には、文字、単語、行ごとの信頼度と座標を保持します。
信頼度が低いページ、表が多いページ、手書きを含むページは、別の解析器または人の確認へ振り分けます。
平均信頼度だけでなく、型番や金額など業務上重要な箇所の認識誤りを調べます。

[Donut](https://arxiv.org/abs/2111.15664)は、OCR結果を中間入力とせず、文書画像から構造化された文字列を生成する方法を提案しました。
OCRを使わない方式は、OCR誤りの連鎖を避けられる可能性があります。
一方で、生成出力の制御、計算費用、再現性、原画像との位置対応を対象文書で確認する必要があります。

一つの方式で全文書を置き換えず、文書クラスごとにOCR、レイアウト解析、画像からの直接理解を比較します。

## 3.3.6 表、図、数式、コード

表を単純な改行テキストへ変換すると、値がどの行見出しと列見出しに対応するか分からなくなります。
表題、行・列見出し、結合セル、脚注、単位を保持します。
検索表示用のMarkdownと、項目名と値を組にして表せるJSON形式などを併存させます。

[PubTables-1M](https://arxiv.org/abs/2110.00061)は、表領域の検出、表構造の認識、セルの対応を一続きの課題として扱いました。
セル内の文字だけでなく、行と列の関係を復元する必要があります。
引用は元ページ、表ID、セル範囲へ戻れるようにします。

図は、画像だけでなく、図番号、キャプション、本文からの参照、周辺説明を結び付けます。
数式は記号、添字、分数、式番号を保持します。
コードは空白、行頭の字下げ（インデント）、改行、ファイル名、言語を保持し、通常文章の正規化規則を適用しません。

[Nougat](https://arxiv.org/abs/2308.13418)は、学術文書画像を、見出しや数式などの構造を記号で表すマークアップ形式へ変換し、数式を含む文書解析を扱いました。
出力された数式や構造が原文と一致するかを、文字列一致だけでなく表示と意味の両面から確認します。

## 3.3.7 テキスト正規化

正規化は、検索しやすい表記へ揃える処理です。
文字の内部表現を揃えるUnicode正規化、全角・半角、改行、行末ハイフン、連続空白、ページごとに繰り返す上部見出しなどを対象にできます。
ただし、検索しやすさのために原文の意味を変えてはいけません。

URL、製品コード、条文番号、数値、数式、コードの空白と記号は、一般文章と別の規則で扱います。
例えば、`A-01` を `A01` へ変えると、別の型番と一致する可能性があります。
表の空白やコードのインデントを削除すると構造が変わります。

原文と正規化済みテキストを別項目へ保存します。
検索は正規化済み表現を利用でき、引用は原文を優先します。
正規化規則を版管理し、代表例と失敗例で変更前後を比較します。

## 3.3.8 文書構造の復元

検出したブロックを順番に並べるだけでは、章、節、リスト、脚注の関係は復元されません。
見出し階層、リストの入れ子、節の経路、本文から脚注や図表への参照を明示します。

HTMLとMarkdownでは元の構造を利用し、Office文書では見出しスタイル、コメント、変更履歴、非表示要素の採用方針を決めます。
ページをまたぐ段落、表、脚注を結合する場合は、文法、罫線、見出し、継続記号などの条件を使います。

構造は、見出し文字列を本文へ付け足すだけでなく、型を持つ親子・参照関係として保存します。
推定した結合には信頼度を持たせ、誤結合を後から検出できるようにします。

## 3.3.9 解析器の振り分け

一つの高価な解析器をすべてのファイルへ使う必要はありません。
形式、文字層、画像率、表密度、言語、ページ数から、既定解析、レイアウト対応解析、OCR、画像理解へ振り分けます。

各経路には、適用条件、失敗条件、次に試す解析器を定めます。
例えば、文字層があり単段のPDFは軽量解析、スキャンPDFはOCR、複雑な帳票はレイアウト対応解析へ送ります。
解析信頼度が基準を下回る場合は、次の経路または人の確認へ進みます。

LayoutLMv3、Donut、Nougatは、対象と出力が異なります。
論文で報告された一つの指標だけで選ばず、文書クラスごとに文字、構造、表、位置、費用を比較します。
解析器変更は稼働中成果物へ上書きせず、別版として同じ文書を影響評価します。

## 3.3.10 品質検査

解析成功率は、処理が例外なく終了した割合にすぎません。
読み順の逆転、見出しの誤分類、表の崩壊、ページ対応のずれは、成功結果の中にも含まれます。

空出力、原文に比べて極端に短い出力、重複ページ、文字化け、異常な文字比率は全件を規則で検査します。
代表文書について、文字欠落、見出し階層、読み順、表構造、ページ位置、OCR信頼度を人手の正解と比較します。

文書形式とページ配置の種類別に結果を集計します。
全体平均が良くても、スキャン文書や縦書き文書だけが失敗している場合があります。
解析器の更新ごとに差分を保存し、改善した文書と悪化した文書を確認します。

## 3.3.11 チャンク分割への引き渡し条件

分割器へ渡す入力は、一続きのテキストではありません。
種類を持つブロック、見出し経路、ページ、表ID、座標、原文範囲、解析信頼度を渡します。
分割器はこれらを使って、見出し、表、リスト、コードを壊さない境界を選びます。

解析器が推定した境界を確定事実として扱わず、不確実性も契約へ含めます。
再解析でブロックが変わった場合に、影響するチャンク、埋め込み、引用を特定できる依存関係を持ちます。

この契約によって、解析段階で本文が壊れた失敗と、正しい構造を分割器が誤って切った失敗を区別できます。
チャンクから原ページへ戻れない場合は、インデックス公開前に止めます。
