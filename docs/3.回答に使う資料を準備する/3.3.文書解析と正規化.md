# 3.3 文書解析と正規化

## 3.3.1. 解析・正規化の役割

文書解析の目的を「文字を取れた」に置くと、表の列や見出し階層が消えても成功扱いになる。Document AI研究はtext、layout、imageを組み合わせることで理解性能が上がることを示してきた。そこで解析を、原文を検索・引用可能な構造へ変換する工程と定義する。parser品質がchunk、embedding、citationへ連鎖する図を置き、空文字や文字化けだけでなく、reading orderと位置対応の欠落を失敗に含める。

**参照論文**
- [P083] [LayoutLMv3: Pre-training for Document AI with Unified Text and Image Masking](https://arxiv.org/abs/2204.08387)
- [P099] [PubLayNet: largest dataset ever for document layout analysis](https://arxiv.org/abs/1908.07836)

## 3.3.2. 入力形式と失敗モードの分類

入力形式を拡張子だけで分類しない。born-digital PDF、scan、HTML、Word、Excel、slideでは、文字layer、画像比率、表密度、reading orderの失敗が異なる。多段組、縦書き、回転page、暗号化、巨大fileをdocument classとして記録し、default parserの対象外を早期検知する。DocLayNetが多様なbusiness documentでlayout差を示した点を踏まえ、代表一種類のPDFで全体品質を判断しない。

**参照論文**
- [P090] [DocLayNet: A Large Human-Annotated Dataset for Document-Layout Analysis](https://arxiv.org/abs/2206.01062)
- [P099] [PubLayNet: largest dataset ever for document layout analysis](https://arxiv.org/abs/1908.07836)

## 3.3.3. canonical document schema

canonical schemaは後段を特定parserへ依存させないために作る。document、page、block、table、image、linkの階層を設け、blockへtype、text、page、bbox、reading_order、confidenceを持たせる。元file、raw parser output、normalized representationを別保存し、chunkから原page座標へ戻るstable IDを用意する。LayoutLM系が利用する空間情報を消さず、parser交換時にも同じdownstream契約を維持する。

**参照論文**
- [P083] [LayoutLMv3: Pre-training for Document AI with Unified Text and Image Masking](https://arxiv.org/abs/2204.08387)
- [P018] [LayoutLM: Pre-training of Text and Layout for Document Image Understanding](https://arxiv.org/abs/1912.13318)

## 3.3.4. レイアウト解析と読み順

二段組の右列を左列へ混ぜたり、footerを本文へ挟んだりすると、意味の近い誤chunkが生まれる。PubLayNetとDocLayNetはheading、text、table、figureなどの領域検出をbenchmark化し、domainによるlayout差も明らかにした。heading、list、caption、footnoteを識別し、text layerとpage imageを照合してreading orderを復元する。modelの学習domainと自社文書の差をsample評価し、bboxをcitationへ残す。

**参照論文**
- [P099] [PubLayNet: largest dataset ever for document layout analysis](https://arxiv.org/abs/1908.07836)
- [P090] [DocLayNet: A Large Human-Annotated Dataset for Document-Layout Analysis](https://arxiv.org/abs/2206.01062)

## 3.3.5. OCRとOCR-free方式

text layerが信頼できる文書へOCRをかけ直すと、正しい文字を劣化させる場合がある。まずOCR要否を判定し、必要なpageだけ処理する。DonutはOCRを介さず画像から構造化出力を生成する別経路を示したが、計算cost、出力拘束、再現性の課題も持つ。文字・行confidenceを保持し、低confidence pageは別parserまたは人手へrouteする。OCR-freeを万能置換でなく文書class別の選択肢として扱う。

**参照論文**
- [P022] [OCR-free Document Understanding Transformer](https://arxiv.org/abs/2111.15664)

## 3.3.6. 表、図、数式、コードの扱い

表を改行textへ潰すと、値がどのheaderと交差するか分からなくなる。PubTables-1Mはtable detection、structure recognition、cell対応を一体の課題として扱った。caption、row/column header、merged cellを保ち、Markdown表示とcell JSONを併存させる。図はcaptionと周辺本文、数式とcodeは空白・記号・改行を保持する。検索用表現を作っても、citationは元page、table ID、cellへ戻せるようにする。

**参照論文**
- [P100] [PubTables-1M: Towards comprehensive table extraction from unstructured documents](https://arxiv.org/abs/2110.00061)
- [P021] [Nougat: Neural Optical Understanding for Academic Documents](https://arxiv.org/abs/2308.13418)

## 3.3.7. テキスト正規化

正規化は検索しやすさと原文忠実性の綱引きである。Unicode、全半角、改行、hyphenationを整え、header/footerや透かしの反復を除く一方、URL、製品code、条文番号、数値記号は壊さない。原文とnormalized textを別fieldへ置き、検索は後者、引用は前者を優先する。過剰な空白除去でcodeや表が変わるcaseをfixtureにし、正規化ruleのversion差を回帰評価する。 正規化前の原文を別fieldに保存し、引用では必ず元表記へ戻ることで、検索最適化と証拠保全を両立する。

**参照論文**
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.3.8. 構造の復元

blockを並べるだけでは文書構造は戻らない。heading hierarchy、list nesting、section pathを復元し、pageをまたぐ段落・表・脚注の結合ruleを決める。HTML/MarkdownではDOMを利用し、Officeではcomment、変更履歴、hidden elementの採否を明示する。Nougatが学術文書をmarkupへ変換する発想を参考に、構造を後付けの見出し文字列でなくtyped relationとして保存する。誤った結合にはconfidenceを残す。

**参照論文**
- [P021] [Nougat: Neural Optical Understanding for Academic Documents](https://arxiv.org/abs/2308.13418)

## 3.3.9. parser routing

一つの高価なparserを全fileへ適用するのではなく、形式、画像比率、表密度、言語、page数でrouteする。default、layout-aware、OCR、VLM系の順序とfallback条件を決め、文書classごとに品質とcostを比較する。LayoutLMv3、Donut、Nougatは得意領域が異なるため、論文上の最高値だけで選ばない。parser変更は同じactive indexへ上書きせず、新しいdata sourceまたはversionでshadow評価する。

**参照論文**
- [P083] [LayoutLMv3: Pre-training for Document AI with Unified Text and Image Masking](https://arxiv.org/abs/2204.08387)
- [P022] [OCR-free Document Understanding Transformer](https://arxiv.org/abs/2111.15664)
- [P021] [Nougat: Neural Optical Understanding for Academic Documents](https://arxiv.org/abs/2308.13418)

## 3.3.10. 品質検査

parse成功率だけでは、読み順の逆転や表崩壊を見逃す。gold documentを用意し、文字欠落、heading、page対応、table structure、bbox、OCR confidenceを文書種別別に評価する。空出力、極端に短いtext、重複page、文字化けは全件ruleで検知し、複雑なlayoutはsample人手確認する。DocLayNetのclass別評価を参考に、平均値でscan文書の失敗を隠さず、parser updateごとにdiffを残す。

**参照論文**
- [P090] [DocLayNet: A Large Human-Annotated Dataset for Document-Layout Analysis](https://arxiv.org/abs/2206.01062)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.3.11. Chunkingへの引き渡し契約

chunkerへ渡すのは一続きのtextではなく、type付きblock、heading path、page、table ID、bbox、confidenceである。parserが推定した境界を確定事実として扱わないよう、不確実性も契約へ含める。再解析でblockが変わったとき、影響するchunkだけを再生成できるdependencyを持つ。これによりchunkingの失敗と解析の失敗を分け、citationが元pageへ戻れないcaseをpublish前に止められる。

**参照論文**
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)
