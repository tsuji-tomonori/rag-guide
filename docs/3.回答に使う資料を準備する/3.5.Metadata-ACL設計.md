# 3.5 メタデータとACLの設計

メタデータは、文書の意味以外の条件を検索、引用、更新、削除へ伝える制御情報です。
アクセス制御リスト（Access Control List：ACL）は、どの主体がどの文書を利用できるかを表します。
埋め込みの類似度では、版、有効期間、権威性、権限を判断できません。

## 3.5.1 制御情報としてのメタデータ

メタデータには、来歴、版、時間、分類、信頼度、機密区分、ACLなどがあります。
検索フィルター、引用表示、更新処理、監査で同じ属性を利用します。
工程ごとに別名・別形式で持つと、検索時の版と引用時の版がずれる可能性があります。

[KILT](https://arxiv.org/abs/2009.02252)は、タスクの回答と根拠となるページを一体として評価しました。
RAGでも、文書の内容だけでなく、出所へ戻る情報を一次的な品質要件として扱います。

属性が欠けた場合の処理を項目ごとに定めます。
ACLや文書IDの欠損は公開禁止、表示用の説明文の欠損は警告など、業務影響に応じて区別します。
欠損値をLLMや既定値で推測して補いません。

## 3.5.2 安定ID

URLやファイル名は、移動、名称変更、再配布によって変わります。
一つの値だけを主キーにせず、情報源、文書、文書版、チャンクを別々に識別します。

- `source_id` は情報源を識別します。
- `document_id` は版をまたぐ論理的な文書を識別します。
- `document_version_id` は特定の文書版を識別します。
- `chunk_id` は特定の解析・分割結果中の検索単位を識別します。

内容ハッシュは、同一内容と差分を検出する補助に利用します。
同じ本文でもACLや有効期間が違えば、同じ利用条件とは限りません。

ここでいう「安定」は、文字列が永久に変わらないことではありません。
原文、版、分割方法が変わったときに、新しいIDを発行する規則が決定的であり、元文書との関係を追跡できることです。
ID変更が引用URL、キャッシュ、評価ラベルへ与える影響も管理します。

## 3.5.3 来歴と引用のメタデータ

引用に必要な情報は、文書名とURLだけではありません。
ページ、節、見出し経路、ページ内座標、原文範囲、文書版、情報源スナップショットを保持します。

情報源が提供した値と、解析器が推定した値を区別します。
例えば、PDF内の印刷ページ番号、PDFビューアー上のページ番号、解析器が数えたページ番号は異なる場合があります。
引用画面でどの番号を表示するかを定めます。

検索単位が短い子チャンクでも、生成には親節を渡し、引用は数文の原文範囲を表示できます。
子、親、原文範囲の関係を保存します。
[ALCE](https://arxiv.org/abs/2305.14627)が主張と引用の対応を評価したように、回答の各主張から支持範囲へ戻れる必要があります。

利用者へ公開する文書位置と、内部監査用の保存場所は分けられます。
文書が移動・更新されても、回答時に参照した版を識別できる情報を保持します。

## 3.5.4 版、時点、鮮度

作成、公開、更新、適用開始、適用終了、取り込みは異なる時刻です。
一つの `updated_at` へまとめると、公開済みだが未施行の版や、失効後に修正された版を正しく選べません。

質問の基準時点と文書の適用期間を比較します。
公開状態は、下書き、現在有効、失効、保管などに分けます。
最新版を常に選ぶのではなく、過去時点への質問では当時の版を選べるよう、版の前後関係を持ちます。

情報源自体の鮮度と、インデックスへ反映された鮮度を分けます。
情報源が更新されていても、取り込みが停止していればインデックスは古いままです。
[StreamingQA](https://arxiv.org/abs/2205.11388)や[FreshLLMs](https://arxiv.org/abs/2310.03214)は、時間とともに変化する知識への適応を評価しました。

日時はタイムゾーンを含む形式へ揃えます。
日付だけで施行する規程について、どの地域の一日の開始を基準にするかも決めます。

## 3.5.5 分類と振り分け

言語、部署、製品、地域、文書種別、話題は、検索先の振り分けとフィルターに利用できます。
自由入力のタグだけで運用すると、`Product-A`、`product_a`、旧製品名などが混在します。
管理語彙、値の所有者、スキーマ版を定めます。

フィルターに利用する属性と、画面表示だけに利用する属性を分けます。
検索エンジンで絞り込める型、配列、演算子を確認します。
値の種類が多い属性や、候補が少なくなる条件は、近似最近傍探索（Approximate Nearest Neighbor：ANN）の検索品質と応答時間へ影響するため、質問条件別に評価します。

分類モデルが推定した値には、推定元、モデル版、信頼度を持たせます。
利用者が指定した製品条件や認可条件と同じ強制フィルターへ、推定値を無条件に使いません。

## 3.5.6 情報源の信頼度と機密区分

情報源の信頼度と機密区分は、別の軸です。
正式だが厳秘の規程、公開されている未検証のフォーラム投稿、正式資料から生成した要約は、意味が近くても扱いが異なります。

情報源の位置付けを、正式、参考、派生、外部などに分類します。
機密区分を、公開、社内、機密、限定などに分類します。
二つの属性を組み合わせ、矛盾時の優先度、検索可否、引用可否、ログ保存可否を決めます。

[PoisonedRAG](https://arxiv.org/abs/2402.07867)は、質問に対して高い関連性を持つ悪意ある文書を混入させ、回答を操作する攻撃を示しました。
関連度が高いことは、信頼できる情報源であることを意味しません。
信頼度を検索後の飾りにせず、採用と回答の規則へ接続します。

## 3.5.7 認可モデル

認可方式は、ACL、ロールベース、属性ベース、関係ベースなどを組み合わせられます。

- **ACL**は、文書ごとに許可または拒否する主体を列挙します。
- **役割に基づくアクセス制御（Role-Based Access Control：RBAC）**は、利用者へ付与した役割に基づいて許可します。
- **属性に基づくアクセス制御（Attribute-Based Access Control：ABAC）**は、利用者、契約組織（テナント）、機密区分、部署、時刻などの属性を条件にします。
- **関係に基づくアクセス制御（Relationship-Based Access Control：ReBAC）**は、所有、所属、共有など、主体と資源の関係を条件にします。

[RBACモデル](https://doi.org/10.1109/2.485845)は、利用者、役割、権限の関係を体系化しました。
[Cedar](https://arxiv.org/abs/2403.04651)は、主体、操作、資源、状況を入力とする認可方針言語です。

複数の方式を利用しても、最終的な許可・拒否の契約を一つにします。
拒否優先、例外、緊急時アクセスの条件を明記します。
LLMへ権限判定を委ねず、決定的な認可サービスで判断します。

## 3.5.8 文書からチャンクへのACL継承

原則として、文書のACLをすべての子チャンクへ継承します。
ただし、添付ファイル、コメント、埋め込みオブジェクト、外部リンクは、親文書と別の権限を持つ場合があります。
例外を明示的な資源関係として表します。

親と子の権限を和集合にするか積集合にするかを曖昧にしてはいけません。
一方で許可されているから見せるのか、両方で許可されている場合だけ見せるのかによって結果が変わります。
業務規則として定義し、テストします。

ACLが不明なチャンクは、既定で拒否します。
文書から生成したチャンク件数とACL件数を照合します。
解析器や分割器の変更で権限属性が落ちないことを、権限外利用者の試験で確認します。

## 3.5.9 検索時の認可

最初に認証によって利用者を確定し、認可によって閲覧できる範囲を決めます。
その範囲を強制フィルターとして検索器へ渡します。
テナント条件は常に適用し、利用者の質問から上書きできないようにします。

グループと役割を質問時に展開する方法と、検索用属性として事前展開する方法があります。
事前展開は検索を速くできますが、権限剥奪時に多くのチャンクを更新する必要があります。
利用者・グループ数、変更頻度、剥奪の反映要件から選びます。

回答生成後のマスキングでは不十分です。
権限外本文がすでにプロンプト、キャッシュ、トレースへ入っているためです。
フィルターによってANN再現率が低下してもACLを緩めず、候補数の拡大、条件対応の検索、テナント別分割など、検索側で対処します。

## 3.5.10 方針をコードとして管理する

方針をコードとして管理する場合、主体、操作、資源、状況を入力し、許可・拒否、理由、方針版を出力します。
方針変更をコードレビュー、単体試験、権限外の否定試験へ通します。

検索エンジンのフィルター式と認可方針を同じ文字列へ混ぜません。
認証・認可から得た強制条件と、利用者が指定した製品・期間などの検索条件を、条件の入れ子関係を表す別々の構文木として作り、両方を満たす形で結合します。
利用者が入力したフィルターで強制条件を削除できない構造にします。

方針を解析できない場合や必要な属性がない場合は、許可せず拒否します。
例外的な承認には、理由、承認者、対象、期限を付けます。

## 3.5.11 更新、削除、権限剥奪

権限剥奪は、一つのメタデータ値を更新して完了する処理ではありません。
情報源、待機中・稼働中インデックス、複製、キャッシュ、セッション、旧インデックスへ変更を反映させます。

検索対象から削除済みであることを示す記録（墓標）と、保存媒体から物理的に削除する期限を分けます。
検索から直ちに除外し、保存期間や法的要件に従って後から物理削除できます。
切り替え前後の両インデックスへ同じ削除と権限変更を適用します。

剥奪からすべての検索経路へ反映されるまでの時間を測ります。
旧版へロールバックしても退職者や削除済みグループの権限が復活しないことを確認します。
反映に失敗した資源は、一時的に検索不能にします。

## 3.5.12 スキーマと保存先

ベクトルストアごとに、メタデータのキー数、値の大きさ、型、配列、フィルター演算子の制約があります。
メタデータスキーマへ版を付け、旧チャンクの移行方法を定めます。

検索フィルター用、画面表示用、内部監査用の属性は、同じ保存先である必要はありません。
機密な監査情報を検索結果へ返さないよう、目的に応じて分離します。

見出しや製品名を本文へ付けて埋め込むと、意味検索のスコアへ影響します。
別項目に置くと、検索エンジンの絞り込みや順位の引き上げ機能に依存します。
目的ごとに比較し、本文とメタデータの重複による順位変化を測ります。

スキーマ変更で再インデックスが必要な条件を明記します。
未知の値、NULL、旧型を安全側に扱い、暗黙の型変換で権限条件を失いません。

## 3.5.13 セキュリティ試験

正常な利用者が自分の文書を検索できることだけでは、認可を検証できません。
次の境界条件を試験データとして用意します。

- 別テナント、別部署、別グループの利用者
- 退職者、権限剥奪直後、期限切れ権限
- 属性の値がない状態（NULL）、旧スキーマ、大小文字が異なる値
- 複数グループへ所属する利用者
- 共有を解除した文書と旧インデックス
- 複数質問、候補数拡大、キャッシュ利用時の検索

権限外チャンクが、検索候補、プロンプト、回答、引用、ログへ一件も入らないことを確認します。
同時に、過剰な拒否によって正しい根拠が欠落した割合も別の指標で測ります。

拒否した文書の存在、件数、名称を回答から推測できないことも確認します。
試験結果には、利用者属性、方針版、インデックス版、検索条件を記録します。
