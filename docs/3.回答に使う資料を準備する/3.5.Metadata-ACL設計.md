# 3.5 Metadata / ACL設計

## 3.5.1. メタデータをRAGのcontrol planeと定義する

metadataはembeddingの付属情報ではなく、RAG全体のcontrol planeである。意味の近さだけでは、版、地域、権限、有効日、source trustを判定できない。検索filter、citation、更新、削除、監査へ同じ属性を通し、欠損時に除外すべきfieldと警告で済むfieldを分ける。KILTが回答とprovenanceを一体で評価した視点を取り入れ、metadata品質を検索精度と安全性の双方に効く一次要件として扱う。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 3.5.2. stable ID設計

URLやfilenameを主キーにすると、移動や再配布のたび同じ文書が別物になる。source_id、document_id、document_version_id、chunk_idを分離し、content hashは同一性判定の補助に使う。再解析・再chunking後も元文書と版へ戻れるrelationを残し、ID変更がcitationやgold labelへ与える影響を管理する。stableとは不変文字列ではなく、何が変わったときに新IDを発行するかが再現可能であることだと定義する。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P058] [Datasheets for Datasets](https://arxiv.org/abs/1803.09010)

## 3.5.3. provenanceとcitation metadata

citationに必要なmetadataはtitleとURLだけではない。page、section、heading path、bbox、source snapshot、versionを持ち、parserが推定した値とsourceが提供した値を区別する。検索単位がchild chunkでも、表示はparent section、引用は原文spanという対応を保存する。KILTのprovenanceを実装へ落とし、回答UIへ公開できるlocationと内部監査用storageを分け、source移動後も当時の根拠を識別できるようにする。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)

## 3.5.4. 版・時点・鮮度metadata

created、published、updated、effective、ingestedは異なる時間を表す。質問の基準時点とeffective periodを比較し、current、expired、draft、archivedを検索条件へ落とす。FreshLLMsやStreamingQAが知識の時間変化を評価したように、source freshnessとindex freshnessを別の指標として監視する。最新版を常に返すのではなく、過去時点の質問では旧版を選べるよう、version lineageとtimezone規則を持つ。

**参照論文**
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)
- [P135] [StreamingQA: A Benchmark for Adaptation to New Knowledge over Time in Question Answering Models](https://arxiv.org/abs/2205.11388)
- [P152] [It's About Time: Incorporating Temporality in Retrieval Augmented Language Models](https://arxiv.org/abs/2401.13222)

## 3.5.5. 分類・routing用metadata

language、department、product、region、document type、topicはroutingとfilterへ使うが、自由入力tagだけでは表記が漂流する。管理語彙、schema version、値のownerを設け、filterable fieldと表示専用fieldを分ける。cardinalityが極端に高い属性をANN filterへ入れると性能が変わるため、必要なquery sliceで評価する。分類modelが推定した値にはconfidenceを持たせ、利用者明示条件と同じhard filterへ無条件に昇格させない。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.5.6. source trustと機密区分

source trustと機密区分は別軸である。正式だがrestrictedな規程、公開されているが未検証のforum投稿、公式資料から生成したsummaryは、関連度が同じでも扱いが違う。authoritative、informal、derivedとpublic、internal、confidentialを組み合わせ、矛盾時の優先、引用可否、log保存可否へ接続する。PoisonedRAGが高関連度の悪意文書で回答を誘導できることを示したため、trustをscoreの後付けにしない。

**参照論文**
- [P060] [PoisonedRAG: Knowledge Corruption Attacks to Retrieval-Augmented Generation of Large Language Models](https://arxiv.org/abs/2402.07867)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 3.5.7. ACL、RBAC、ABAC、ReBACの使い分け

認可方式は名称を選ぶ問題ではなく、policyをどこへ置くかの設計である。ACLはresourceへ主体を列挙し、RBACはrole、ABACはtenant・classification・時点、ReBACは所属や所有relationを使う。SandhuらのRBACとCedarのprincipal/action/resource/contextモデルを並べ、複数方式を混ぜても最終allow/deny契約を一つにする。例外、deny優先、break-glassを明記し、LLMへ権限判定を委ねない。

**参照論文**
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)

## 3.5.8. 文書ACLからchunk ACLへの継承

文書ACLをchunkへ継承する際、単純copyだけでは添付、comment、埋め込み表の境界を誤る場合がある。原則は文書ACLの継承、例外は明示的なresource relationとして扱い、unionかintersectionかを曖昧にしない。ACL不明chunkはdeny-by-defaultとする。chunk生成後の件数と許可主体を照合し、parserやchunker変更で権限fieldが落ちないnegative testを用意する。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 3.5.9. retrieval時の認可

authenticationで利用者を確定し、authorizationで閲覧可能範囲を解決してからretrievalへ進む。tenant_idを常時hard filterにし、groupやroleはquery時展開かindex時materializeかを規模とrevocation要件で選ぶ。生成後maskingでは、権限外textがprompt、cache、traceへ既に入るため不十分である。filtered ANNでrecallが下がってもACLを緩めず、oversamplingやindex partitionを性能側の解として検討する。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P144] [Curator: Efficient Indexing for Multi-Tenant Vector Databases](https://arxiv.org/abs/2401.07119)

## 3.5.10. policy-as-codeと監査

policy-as-codeではprincipal、action、resource、contextを入力にし、decisionとreason、policy versionを返す。Cedarが安全で解析可能なauthorization languageを目指した点を参考に、allow/denyの理由を監査可能にし、policy変更をunit testとnegative testへ通す。検索filterのDSLと認可policyを混ぜず、認証由来条件とuser指定条件を別ASTとしてAND結合する。解析失敗はfail-closedにし、例外承認へ期限を持たせる。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)

## 3.5.11. 更新・削除・権限剥奪

権限剥奪はmetadata update一件で終わらない。source、staging、active index、replica、cache、session、旧indexへ変更を伝播し、race conditionを検証する。削除tombstoneとphysical deleteの期限、blue/green両側のpolicy整合を決める。revocation latencyをsecurity SLOとして測り、旧versionへrollbackしても退職者の権限が復活しないrelease contractを作る。失敗時は対象resourceを一時的に検索不能へ倒す。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P144] [Curator: Efficient Indexing for Multi-Tenant Vector Databases](https://arxiv.org/abs/2401.07119)

## 3.5.12. metadata制約とschema evolution

vector storeごとにmetadata key数、値size、型、配列、演算子の制約が異なる。schema versionを持ち、古いchunkのmigrationと、filter用途・表示用途・監査用途の保存先を分ける。metadataを本文へ埋め込めばsemantic scoreへ影響し、別fieldならfilter機能へ依存するため、目的別に比較する。schema変更で再indexが必要な条件を明示し、NULLや旧型を安全側へ扱う。

**参照論文**
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)

## 3.5.13. セキュリティ試験

security testでは正常ユーザーだけを試さない。tenant横断、group境界、退職者、時限権限、剥奪直後、NULL属性、大小文字、古いschemaをfixture化する。top-kやmulti-queryを増やしても権限外chunkが一件も候補へ入らないことを確認し、過剰制限によるgold欠落も別指標で測る。retrieved text、prompt、answer、logの全段階を追い、denyした文書の存在自体を応答へ漏らさない。

**参照論文**
- [P144] [Curator: Efficient Indexing for Multi-Tenant Vector Databases](https://arxiv.org/abs/2401.07119)
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)
