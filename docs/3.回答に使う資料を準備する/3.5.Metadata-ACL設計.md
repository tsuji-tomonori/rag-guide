# 3.5 Metadata / ACL設計

## この節の役割

- 検索、引用、版管理、監査、権限制御に必要なmetadataは何か。
- 文書とchunkを、更新後も追跡できるstable IDでどう識別するか。
- RBAC、ABAC、ACL、ReBACをどこで使い分けるか。
- 権限外chunkを、LLMへ渡る前に確実に除外するにはどうするか。

## 3.5.1. メタデータをRAGのcontrol planeと定義する
  - embeddingは意味の近さを表すが、権限・版・地域・有効日を保証しない。
  - metadataが検索、filter、引用、監査、更新、削除を横断することを示す。
  - metadata欠損を「品質低下」ではなく「安全側で除外すべき状態」とする項目を決める。

## 3.5.2. stable ID設計
  - `source_id`、`document_id`、`document_version_id`、`chunk_id`を分ける。
  - URL、ファイル名、配列順など変わりやすい値を主キーにしない。
  - 再解析・再chunking後も、元文書と版へ戻れる参照を残す。
  - content hashを重複検知と差分更新に使う。

## 3.5.3. provenanceとcitation metadata
  - title、source URL、page、section、heading path、bboxを持つ。
  - parserが作った情報とsourceが持つ情報を区別する。
  - 回答UIへ公開可能なURLと、内部監査用の保存場所を分ける。
  - 引用単位と検索単位が異なる場合の対応関係を保存する。

## 3.5.4. 版・時点・鮮度metadata
  - created、updated、published、effective、ingestedの日時を区別する。
  - current/expired/draft/archivedなどdocument statusを持つ。
  - 質問の基準時点と文書の有効期間を比較できる形にする。
  - index freshnessとsource freshnessを別指標で監視する。

## 3.5.5. 分類・routing用metadata
  - language、department、product、region、document type、topicを定義する。
  - 自由入力タグだけに依存せず、管理語彙とschema versionを持つ。
  - filterable metadataと表示専用metadataを分ける。
  - cardinalityが高すぎるfieldを安易にfilter indexへ入れない。

## 3.5.6. source trustと機密区分
  - authoritative/informal/derivedを区別する。
  - public/internal/confidential/restrictedなどの分類を持つ。
  - source trustは関連度scoreとは別に扱う。
  - 矛盾時の優先順位、引用可否、ログ保存可否へ接続する。

## 3.5.7. ACL、RBAC、ABAC、ReBACの使い分け
  - ACL: 文書・chunkに許可主体を直接列挙する。
  - RBAC: roleやgroupを介して権限を管理する。
  - ABAC: tenant、department、classification、region、時刻など属性で判定する。
  - ReBAC: 所属・所有・関係を辿って判定する。
  - 実装では複数方式を混ぜるため、最終判定契約を明示する。

## 3.5.8. 文書ACLからchunk ACLへの継承
  - 原則は文書ACLを全chunkへ継承する。
  - 付録、添付、コメント、埋め込み表など例外の扱いを定める。
  - ACLのunion/intersectionを曖昧にしない。
  - ACL不明のchunkはdeny-by-defaultにする。

## 3.5.9. retrieval時の認可
  - authentication、authorization、retrieval filterの順序を示す。
  - `tenant_id`を常時必須filterにする。
  - groupやroleの展開をquery時に行うか、index時にmaterializeするか比較する。
  - 生成後のmaskingではなく、候補集合へ入れない設計を強調する。

## 3.5.10. policy-as-codeと監査
  - principal、action、resource、contextを入力とする認可契約を定義する。
  - policy version、評価結果、適用時刻をtraceへ残す。
  - allow/denyの理由を監査できるようにする。
  - policy変更をunit testとnegative testで検証する。

## 3.5.11. 更新・削除・権限剥奪
  - source側ACL変更からindex反映までのSLOを決める。
  - 削除tombstone、再index、cache invalidationを扱う。
  - ACL変更中に古いindexが参照されるrace conditionを考慮する。
  - blue/green切替時に両indexで同じ認可境界を維持する。

## 3.5.12. metadata制約とschema evolution
  - key数、値サイズ、filter演算子、型、配列対応の制約を確認する。
  - schema versionを持ち、古いchunkのmigration方法を決める。
  - filter用途、表示用途、監査用途で保存先を分ける。
  - metadataを本文へ埋め込む方式と、別fieldで持つ方式を比較する。

## 3.5.13. セキュリティ試験
  - tenant横断、group境界、退職者、権限剥奪直後、時限権限を試す。
  - metadata欠損、大小文字、配列、NULL、古いschemaをnegative caseにする。
  - top-kを増やしても権限外chunkが混入しないことを確認する。
  - retrieved chunk、prompt、回答、ログの全段階で漏えいがないか検査する。
