# 3.4 Chunking設計

## 3.4.1. Chunkingを「検索単位の設計」と定義する

chunkingは保存fileを切る処理ではなく、retrieverが比較する知識単位を設計する作業である。Dense X Retrievalは粒度がretrievalとdownstream QAの双方を左右すると示した。保存単位、検索単位、LLMへ渡す単位を分け、小さい単位の高い根拠密度と、大きい単位の十分な文脈を両立させる。chunk sizeを唯一のparameterにせず、文書構造、citation、parent relationを含む検索表現として評価する。

**参照論文**
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)

## 3.4.2. 最小baseline：fixed-size + overlap

最初の比較点としてfixed-sizeとoverlapを置く。300〜800 token、10〜20% overlapは答えではなく、測定を始める範囲である。短いchunkは条件や主語を失い、長いchunkは複数話題を一vectorへ圧縮する。overlapを増やすと境界分断は減るが、index量と重複hitが増える。長さ分布、gold span coverage、context重複率を同時に測り、文書種別とembeddingごとにbaselineを固定する。

**参照論文**
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 3.4.3. 構造ベースchunking

規程や設計書では、意味境界の多くが見出しと段落に既に表現されている。heading、section、list、pageを境界に使い、heading pathを本文またはmetadataへ継承する。TextTilingのような統計的境界検出へ進む前に、原文構造を使う方が再現しやすい。ただしparserが見出しを誤ればchunkも崩れるため、構造ベースの改善をparser品質と切り離して評価しない。 文書構造を活用した分割は、利用者が原文で見慣れた節単位へcitationを返しやすい利点も持つ。

**参照論文**
- [P025] [TextTiling: Segmenting Text into Multi-paragraph Subtopic Passages](https://aclanthology.org/J97-1003/)
- [P083] [LayoutLMv3: Pre-training for Document AI with Unified Text and Image Masking](https://arxiv.org/abs/2204.08387)

## 3.4.4. semantic / topic-based chunking

固定長が話題の途中を切る文書では、語彙 cohesionやembedding類似度でtopic changeを探す。TextTilingはmulti-paragraph textをsubtopicへ分ける古典的な方法を示した。LLMで境界を決める方法は柔軟だが、cost、非決定性、model更新による揺れを持つ。semantic chunkingを導入するときは、同じgold setでfixed baselineとablationし、境界の自然さではなくretrieval recallとcitation精度で判断する。

**参照論文**
- [P025] [TextTiling: Segmenting Text into Multi-paragraph Subtopic Passages](https://aclanthology.org/J97-1003/)

## 3.4.5. 文書形式別chunking

文書形式ごとに壊してはいけない関係が違う。FAQは質問と回答、表はcaption・header・対象row、codeはsymbolと依存行、契約は条・項・但し書きを一体にする。slideは一枚完結と章の連続性を比較する。万能chunkerを作るより、canonical block typeへ規則を割り当てる。failure fixtureにはQ/A分離、table header欠落、function途中分断を含め、形式別にgold evidence coverageを測る。

**参照論文**
- [P100] [PubTables-1M: Towards comprehensive table extraction from unstructured documents](https://arxiv.org/abs/2110.00061)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.4.6. proposition単位の検索

sentenceより細かいpropositionは、一つのfactを自己完結した表現へ変え、根拠密度を高める。Dense X Retrievalではproposition-level indexingがpassageよりretrievalとQAを改善した。ただし分解modelが否定、条件、主語を誤れば、検索しやすい誤事実を作る。propositionを原文の代替にせず、source spanとparent chunkへ必ず逆参照させる。fact QAや条項検索へ限定し、生成した命題の忠実性も評価する。

**参照論文**
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)

## 3.4.7. parent-child / hierarchical chunking

小さいchildで正確にhitし、大きいparentをreaderへ渡すのがparent-child設計である。局所検索と文脈保持を分けられるが、parentを大きくしすぎるとnoiseとlost-in-the-middleが戻る。child 200〜400、parent 1,000〜2,000 tokenを初期比較点にし、置換後の重複とtoken量を測る。RAPTORの再帰的cluster summaryは単純な見出しparentと区別し、概要質問が多い場合の発展形として扱う。

**参照論文**
- [P068] [RAPTOR: Recursive Abstractive Processing for Tree-Organized Retrieval](https://arxiv.org/abs/2401.18059)
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 3.4.8. Late Chunkingと文脈付きembedding

通常はchunkごとにencodeするため、代名詞や省略主語が境界で失われる。Late Chunkingは長い文書を先にencodeし、後からchunk単位へpoolingすることで周辺contextをembeddingへ残す。期待できるのは局所textの書き換えではなく、表現時の文脈保持である。対応modelの最大長、計算量、index互換性を確認し、代名詞が多い文書sliceで通常方式と比較する。新手法を理由に全corpusを即時再embeddingしない。

**参照論文**
- [P095] [Late Chunking: Contextual Chunk Embeddings Using Long-Context Embedding Models](https://arxiv.org/abs/2409.04701)

## 3.4.9. stable IDとmetadata継承

chunk内容が同じでも、document版、chunker、ordinalが違えば同一性は変わる。`document_id + version + chunker_version + ordinal`のようなstable ruleを定め、content hashを差分更新と重複検知へ使う。page、heading path、ACL、source trustを継承し、parent-child、proposition、table cell間のrelationを保存する。IDが変わる変更ではcitation URLとevaluation goldのmigrationも計画し、黙って上書きしない。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)

## 3.4.11. 失敗パターンと診断

診断は平均chunk長を見るだけでは始まらない。正解が境界で割れる、titleや条件節が欠ける、overlap領域がtop-kを占める、表やcodeが途中で切れる、大きなparentへ根拠が埋もれる――それぞれ別のfailure codeにする。queryとgold spanから失敗箇所を可視化し、chunker、parser、retrieverの責任を分ける。対策を入れた後は、別sliceのrecallやcitationが悪化していないか確認する。

**参照論文**
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 3.4.12. versioningと切替

chunking変更はindex内の単位数とranking分布を変えるreleaseである。chunker_versionを付け、新方式は別indexへ構築し、golden queryとshadow trafficで比較してからblue/green切替する。旧IDへ依存するcitation、cache、evaluation labelを追跡し、rollback期間を確保する。境界が自然に見えることではなく、retrieval、reader利用、token cost、引用位置の総合で採否を決める。

**参照論文**
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)
