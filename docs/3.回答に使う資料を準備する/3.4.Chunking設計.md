# 3.4 Chunking設計

## この節の役割

- 文書のどの単位を検索対象にするか。
- 小さな検索粒度と、LLMが読むための十分な文脈をどう両立するか。
- 文書種別ごとに、固定長・構造・意味・階層chunkingをどう使い分けるか。
- chunking変更を、どの指標で評価し安全に切り替えるか。

## 3.4.1. Chunkingを「検索単位の設計」と定義する
  - 保存単位、検索単位、LLMへ渡す単位を区別する。
  - 短すぎる場合の文脈欠落と、長すぎる場合のノイズ・コストを示す。
  - chunkingは前処理の細部ではなく、retrieval品質の主要パラメータだと説明する。

## 3.4.2. 最小baseline: fixed-size + overlap
  - token基準と文字基準の違いを説明する。
  - 300〜800 tokens、overlap 10〜20%を「初期候補」として示す。
  - 数値を正解として固定せず、文書・質問・embedding別に評価する。
  - 重複chunkがindex量、検索結果の多様性、引用へ与える影響を示す。

## 3.4.3. 構造ベースchunking
  - heading、paragraph、list、section、pageを境界に使う。
  - heading pathをchunk本文またはmetadataへ継承する。
  - 規程、仕様書、設計書、マニュアルでの利点を示す。
  - parserの誤りがchunk境界へ伝播する点を明記する。

## 3.4.4. semantic / topic-based chunking
  - TextTilingを入口に、話題転換で分割する考え方を説明する。
  - embedding類似度やLLMで境界を決める方法を比較する。
  - 再現性、計算コスト、境界の揺れ、モデル変更の影響を示す。
  - 固定長baselineより本当に改善するかをablationで確認する。

## 3.4.5. 文書形式別chunking
  - FAQは質問と回答を同一chunkに保つ。
  - 表はcaption、header、関連行を壊さず保持する。
  - コードは関数・class・module境界と依存情報を使う。
  - 契約書・規程は条・項・号と見出し階層を使う。
  - スライドは1枚単位だけでなく、章や話題の連続性を考慮する。

## 3.4.6. proposition単位の検索
  - sentenceとpropositionの違いを説明する。
  - fact QAや条項検索で根拠密度を高められる可能性を示す。
  - 分解時の情報欠落、否定・条件・主語の誤変換をリスクとして扱う。
  - 元chunkへの逆参照を必須にする。

## 3.4.7. parent-child / hierarchical chunking
  - 小さなchildで検索し、大きなparentをLLMへ渡す構成を説明する。
  - child 200〜400 tokens、parent 1,000〜2,000 tokensを初期候補にする。
  - parentが大きすぎる場合のノイズとlost-in-the-middleを示す。
  - RAPTORの再帰要約treeは、通常のparent-childと区別して発展扱いにする。

## 3.4.8. Late Chunkingと文脈付きembedding
  - chunk-then-embedと、encode-then-poolの違いを図解する。
  - 代名詞、固有名詞、省略主語が多い文書で期待される利点を示す。
  - 対応embeddingモデル、最大長、計算量の制約を明記する。
  - 新しい手法なので、既存baselineとの比較を必須にする。

## 3.4.9. stable IDとmetadata継承
  - `document_id + document_version + chunker_version + ordinal`などの規則を定める。
  - page、heading path、ACL、source trustをchunkへ継承する。
  - chunk内容hashを持ち、差分更新・重複検出に使う。
  - parent-child、proposition、表セル間の参照関係を保存する。

## 3.4.11. 失敗パターンと診断
  - 正解がchunk境界で分断される。
  - titleや条件節が欠落し、本文だけでは意味が変わる。
  - 同じoverlap領域がtop-kを占有する。
  - 表やコードが分割され、対応関係が壊れる。
  - 大きなparentがノイズを増やし、回答根拠が埋もれる。

## 3.4.12. versioningと切替
  - `chunker_version`を持ち、既存indexへ上書きしない。
  - 新chunkingで別indexを作り、回帰評価後にblue/green切替する。
  - chunking変更がcitation URLや評価データへ与える影響を追跡する。
