# 3.2 Ingestion設計

## 3.2.1. Ingestionを状態を持つpipelineとして定義する

Ingestionはファイルcopyではなく状態を持つpipelineである。`discover→fetch→validate→parse→normalize→chunk→embed→load→verify→publish`をstageへ分け、各入力、出力、version、失敗状態を定義する。FlumeJavaがdata-parallel処理を再利用可能なstageとして構成した考え方を応用し、source別例外を共通契約へ閉じ込める。「job成功」と「検索可能なindex公開」を別statusにし、部分失敗を隠さない。

**参照論文**
- [P092] [FlumeJava: Easy, Efficient Data-Parallel Pipelines](https://research.google/pubs/flumejava-easy-efficient-data-parallel-pipelines/)
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)

## 3.2.2. source別ingestion契約

sourceごとにconnectorの癖は異なる。認証、pagination、rate limit、最大サイズ、削除表現、ACL表現をingestion契約へ書き、source側IDをcanonical schemaへ対応づける。取得不能時に前回snapshotを使うか、公開を止めるかもsourceのriskで決める。契約をコード外の暗黙知にせずversion化し、API仕様変更をschema driftとして検知する。ownerと復旧手順まで含めて初めてconnectorを本番部品と呼ぶ。

**参照論文**
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.2.3. full sync / incremental sync / event-driven update

初回構築やparser全面変更ではfull sync、通常更新ではcursorやCDCによるincremental sync、低遅延が必要ならevent-drivenを使う。ただしwebhookは取りこぼし得るため、定期reconciliationを残す。Dataflow Modelがbatchとstreamをcorrectness、latency、costの軸で捉えたように、方式を優劣で選ばない。更新量、source負荷、許容freshness、再処理能力を比較し、同じidempotent state modelで実装する。

**参照論文**
- [P014] [The Dataflow Model](https://doi.org/10.14778/2824032.2824076)
- [P098] [MillWheel: Fault-Tolerant Stream Processing at Internet Scale](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)

## 3.2.4. snapshotと処理範囲の固定

job実行中にsourceが変われば、同じ設定でも別indexができる。Delta Lakeのsnapshotとtransaction logの発想を借り、source_snapshot_id、cursor、取得開始・終了、件数、hashを固定する。parserやembeddingを比較するときは同じsnapshotを入力にし、差をpipeline変更へ帰属できるようにする。time travel可能な原文とmanifestを保持し、障害調査で「当時何を処理したか」を再現できる状態を作る。

**参照論文**
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)

## 3.2.5. 冪等性と重複排除

retry可能なpipelineでは、同じeventを二度受けることが正常系になる。`source_id + object_id + source_version`をidempotency keyにし、timeout後の再試行や部分成功でも二重登録しない。content hashは本文同一性の補助に使うが、ACLや版の差を消さない。MillWheelのexactly-once processingが依存するpersistent stateの考えを踏まえ、処理済み判定と出力commitを一貫した状態遷移として設計する。

**参照論文**
- [P098] [MillWheel: Fault-Tolerant Stream Processing at Internet Scale](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)

## 3.2.6. 差分判定と再処理範囲

すべての変更で全工程をやり直すと、費用とfreshnessが悪化する。本文変更、metadata-only、ACL、削除を区別し、parser変更ならparse以降、embedding変更ならvector以降というdependency graphを持つ。未変更chunkはcontent hashとmodel versionで再利用する一方、chunk境界が変わればstable IDの影響を追う。差分最適化は速さだけでなく、必要な再処理を省いて古いartifactを残さない正しさで評価する。

**参照論文**
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)
- [P159] [SPFresh: Incremental In-Place Update for Billion-Scale Vector Search](https://arxiv.org/abs/2410.14452)

## 3.2.7. retry、checkpoint、dead-letter

障害処理は無限retryで隠さない。transient errorとpermanent errorを分類し、exponential backoff、最大試行、checkpoint粒度をstageごとに決める。処理できない文書はdead-letterへ隔離し、error code、parser version、source snapshotを付けて再処理可能にする。MillWheelのpersistent stateとwatermarkの考えを参考に、長時間停止後もどこから再開し、遅延eventをどう扱うかを明文化する。

**参照論文**
- [P098] [MillWheel: Fault-Tolerant Stream Processing at Internet Scale](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)

## 3.2.8. backpressureとrate limit

source API、OCR、embedding service、vector storeは別々のquotaを持つ。最も遅いstageへ無制限に流すとqueueが膨らみ、online検索やprovider制限まで巻き込む。queue age、processing time、retry率からconcurrencyを調整し、backpressureで上流を抑える。Dataflow Modelのlatency/cost trade-offを使い、大量更新時はstagingへ積んで検証後にpublishする。throughput最大化よりfreshness SLOと安定性を優先する。

**参照論文**
- [P014] [The Dataflow Model](https://doi.org/10.14778/2824032.2824076)
- [P092] [FlumeJava: Easy, Efficient Data-Parallel Pipelines](https://research.google/pubs/flumejava-easy-efficient-data-parallel-pipelines/)

## 3.2.9. 削除・失効・権限変更の伝播

追加より削除と権限剥奪の方が安全上の優先度は高い。source delete、soft delete、版失効、ACL変更を別eventとして扱い、staging、active index、replica、cacheへ伝播したことを検証する。tombstoneだけ残して検索に出る、旧indexへrollbackすると権限が戻る、といった経路をnegative testにする。反映遅延をsecurity SLOとして測り、失敗時はfail-openでなく対象を隔離する。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 3.2.10. 状態遷移と部分成功

文書単位の状態とjob全体の状態を混ぜない。`discovered / fetched / parsed / chunked / embedded / indexed / published / failed / quarantined`を定義し、何件がどこで止まったかを示す。部分成功を常に失敗とも成功ともせず、security-criticalな欠損はpublish禁止、低影響の文書失敗は隔離して公開可能、とgateを分ける。欠損率と影響範囲を3.8の品質判定へ渡し、手動上書きには承認期限を付ける。

**参照論文**
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)

## 3.2.11. publishとblue/green切替

build中のindexを利用者へ見せず、manifest検査とcanary queryを通した後にaliasまたはrouterを切り替える。Delta Lakeのatomicなcommitとtime travelを設計比喩にし、公開を一つのtransactionとして扱う。旧indexの保持期間、rollback条件、schema互換性を決め、切替前後のqueryをshadow比較する。blue/greenは無停止化だけでなく、parser・chunk・embedding変更の品質差を安全に検証する仕組みである。

**参照論文**
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)
- [P097] [Manu: A Cloud Native Vector Database Management System](https://arxiv.org/abs/2206.13843)

## 3.2.12. Observabilityと運用指標

運用指標はjobの成功件数だけでは足りない。source別の発見件数、処理件数、失敗、retry、queue age、throughputに加え、source更新からactive indexまでのend-to-end freshnessを測る。stage spanへsnapshot、version、error codeを付ければ、connector遅延とembedding quotaを切り分けられる。Hidden Technical Debtがmonitoring debtを警告したように、例外処理を増やすたび観測可能性も更新し、silent data lossを防ぐ。

**参照論文**
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)
