# 3.2 取り込み設計

**取り込み（ingestion）**は、情報源から資料を取得し、検査、解析、分割、埋め込み、登録を経て、検索可能なインデックスとして公開する処理です。
ファイルのコピーではなく、各文書と処理全体の状態を持つパイプラインとして設計します。

## 3.2.1 状態を持つ処理の流れ

取り込みは、次の段階に分けられます。

`発見 → 取得 → 検査 → 解析 → 正規化 → 分割 → 埋め込み → 登録 → 検証 → 公開`

各段階について、入力、出力、使用した処理版、成功条件、失敗状態を定義します。
[FlumeJava](https://research.google/pubs/flumejava-easy-efficient-data-parallel-pipelines/)は、並列データ処理を再利用可能な変換段階として構成する考え方を示しました。
RAGでも、情報源固有の取得処理と、共通の解析・登録処理を契約で分離します。

処理ジョブが終了した状態と、利用者が検索できる状態を分けます。
一部の文書で解析に失敗していてもジョブ自体は終了できます。
何件がどの段階で止まったかを公開判定へ渡し、部分的な欠損を隠しません。

## 3.2.2 情報源ごとの取り込み契約

情報源ごとに、認証、ページ分割、一定時間内の要求数に設けられた上限（レート制限）、最大ファイルサイズ、更新・削除・ACLの表現が異なります。
これらを**取り込み契約**として記録します。

契約には、少なくとも次の項目を含めます。

- 接続先、認証方式、必要な権限
- 情報源側の文書IDと版の表現
- 全件取得と差分取得の方法
- 削除、失効、ACL変更の検出方法
- レート制限、タイムアウト、最大サイズ
- 取得不能時の扱いと復旧手順
- 情報源の所有者とコネクターの担当者

情報源側のIDを、RAG内部の標準スキーマへ対応付けます。
API仕様の変更で項目が消えたり型が変わったりした場合は、スキーマ変化として検出します。
[Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)がデータの異常とスキーマ変化を検出する枠組みを扱ったように、取得要求が成功したという通信結果だけで正常と判断しません。

取得不能時に前回のスナップショットを使うか、公開を止めるかは、情報源のリスクと鮮度要件から決めます。
暗黙の運用判断にせず、契約を版管理します。

## 3.2.3 全件、差分、イベント駆動の同期

初回構築、解析器の全面変更、データ不整合からの復旧では、全件同期を行います。
通常更新では、更新位置を示す値（カーソル）、変更履歴、データベースの変更を記録する仕組み（Change Data Capture：CDC）などを使った差分同期が候補です。
反映遅延を短くする場合は、更新イベントや、変更時に指定先へ通知する仕組み（Webhook）を利用できます。

イベント駆動だけに依存すると、送信失敗や順序入れ替わりによって変更を取り逃がす可能性があります。
定期的な全件照合または差分照合を残し、情報源とインデックスの件数・版を比較します。

[Dataflow Model](https://doi.org/10.14778/2824032.2824076)は、一括処理と継続処理を単純な優劣ではなく、正しさ、結果が利用可能になるまでの時間、費用から扱いました。
RAGでも、更新量、情報源への負荷、許容される鮮度、再処理能力から方式を選びます。
三方式は同じ文書状態と冪等性のモデルを共有します。

## 3.2.4 スナップショット

処理中に情報源が変化すると、同じ設定で実行しても異なるインデックスができます。
比較と障害調査のため、処理対象を**スナップショット**として固定します。

スナップショットID、差分カーソル、取得開始・終了時刻、対象件数、内容を短い値へ変換したハッシュを記録します。
解析器や埋め込みモデルを比較するときは、同じスナップショットを入力し、出力差の原因を処理変更に結び付けられるようにします。

[Delta Lake](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)は、トランザクションログとスナップショットによってデータの版と過去状態を扱います。
RAGの取り込みでも、当時の原文または再取得可能な参照とマニフェストを保持し、「その回答時に何が検索対象だったか」を再現します。

## 3.2.5 冪等性と重複実行

ネットワークのタイムアウト後に再試行すると、最初の処理が実際には成功している場合があります。
同じ更新イベントを複数回受け取ることもあります。
再試行可能なパイプラインでは、重複実行を異常ではなく想定済みの状態として扱います。

情報源ID、文書ID、情報源側の版を組み合わせたキーで、同じ入力を識別します。
同じキーを再処理しても、チャンクやベクトルを二重登録しないようにします。
内容ハッシュは本文の同一性を確認する補助ですが、ACLや版が違う文書を一つにまとめる根拠にはしません。

[MillWheel](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)は、永続状態と、処理済み位置を保存して再開点とするチェックポイントを用いたストリーム処理の耐障害性を扱いました。
処理済みの判定と出力の確定を別々に行うと不整合が生じるため、一貫した状態遷移として設計します。

## 3.2.6 差分判定と再処理範囲

すべての変更で全工程をやり直すと、費用が増え、更新反映が遅れます。
一方で、必要な再処理を省くと、古いチャンク、ベクトル、ACLが残ります。
変更の種類と依存関係から再処理範囲を決めます。

表3-2は、変更の種類と、やり直す必要がある主な処理を対応付けています。
左列で変更内容を選び、右列を起点に下流の成果物へ古い情報が残らない範囲まで再処理します。

**表3-2　変更の種類と主な再処理範囲**

| 変更 | 主な再処理範囲 |
|---|---|
| 原文本文 | 解析以降、または変更箇所以降 |
| メタデータのみ | メタデータ検査とインデックス更新 |
| ACL | 認可情報と検索インデックス、キャッシュ |
| 解析器 | 解析、分割、埋め込み、インデックス |
| 分割方法 | チャンクID、埋め込み、インデックス |
| 埋め込みモデル | ベクトル生成とベクトルインデックス |
| 文書削除 | 稼働中・待機中インデックスとキャッシュからの削除 |

変更していないチャンクは、内容ハッシュと処理版が一致する場合に再利用できます。
分割境界が変わる場合は、安定ID、引用リンク、評価データへの影響を確認します。
差分処理は速さだけでなく、古い成果物が残っていないことで評価します。

## 3.2.7 再試行、チェックポイント、隔離

障害を無限の再試行で隠してはいけません。
一時的な通信障害、レート制限、入力不正、未対応形式など、失敗を分類します。
一時的な障害には待ち時間を増やす再試行を用い、恒久的な入力不正は隔離します。

各段階に最大試行回数とチェックポイントを定めます。
長時間処理の途中で停止した場合、すべてを最初から実行せず、安全な段階から再開します。
再開時にも入力スナップショットと処理版が変わっていないことを確認します。

処理できない文書は、再処理を待つ領域（デッドレター）または隔離領域へ移します。
文書ID、エラーコード、情報源スナップショット、解析器の版、試行回数を付け、原因修正後に再処理できるようにします。
隔離した文書を黙って検索対象から欠落させず、件数と業務影響を公開判定へ渡します。

## 3.2.8 バックプレッシャーとレート制限

情報源API、OCR、埋め込みサービス、ベクトルストアは、それぞれ異なる処理能力と利用上限を持ちます。
遅い段階へ無制限に入力すると、処理待ちの列（キュー）が増え、更新遅延、費用、障害範囲が拡大します。

**バックプレッシャー**は、下流の処理能力に合わせて上流の取得量を抑える仕組みです。
キュー内の最古データの経過時間、処理時間、失敗率、再試行率を使って同時実行数を調整します。
オンライン検索と同じサービスを利用する場合は、取り込みが検索の容量を使い切らないよう資源を分けます。

大量更新時は、待機用インデックスへ段階的に登録し、検証後に公開します。
瞬間的な処理量を最大化するより、定めた鮮度と安定性を守ることを優先します。

## 3.2.9 削除、失効、権限変更

追加よりも、削除と権限剥奪を早く正確に反映する必要があります。
情報源での削除、論理削除、版の失効、ACL変更を別々のイベントとして扱います。

変更は、待機中インデックス、稼働中インデックス、複製、キャッシュ、旧版へ反映させます。
検索結果から消えたことだけでなく、生成用コンテキスト、引用画面、デバッグログへ残っていないことを確認します。
旧インデックスへ戻したときに、削除済み文書や古い権限が復活しないようにします。

反映までの時間をセキュリティ上のサービス目標として測ります。
反映に失敗した場合は、閲覧を許可したままにせず、対象文書または情報源を隔離します。

## 3.2.10 状態遷移と部分成功

文書単位とジョブ全体の状態を分けます。
文書には、発見済み、取得済み、解析済み、分割済み、埋め込み済み、登録済み、公開済み、失敗、隔離などの状態を持たせます。
ジョブには、処理中、検証待ち、公開可能、公開禁止、公開済みなどの状態を持たせます。

一部文書の失敗を、常にジョブ全体の失敗または成功と見なす必要はありません。
権限属性の欠損、正本の欠落、削除反映の失敗は、件数が少なくても公開を禁止します。
低影響の参考資料で解析に失敗した場合は、隔離と影響表示を条件に公開できる場合があります。

公開基準には、失敗件数だけでなく、文書種別、正本性、利用者、質問への影響を含めます。
手動で基準を上書きする場合は、理由、承認者、期限、再確認条件を記録します。

## 3.2.11 公開と切り替え

構築中のインデックスを利用者の検索へ直接追加してはいけません。
待機環境でマニフェスト、件数、権限、代表質問を検査し、合格後に検索の参照先または別名を新インデックスへ切り替えます。
旧版と新版を並べる方式を、青緑切り替え（blue/green deployment）と呼びます。

公開は、複数の設定を順番に変える作業ではなく、一つの版を選ぶ操作として扱います。
切り替え前には、同じ質問を新旧インデックスへ実行し、検索結果、回答、応答時間を比較します。
旧インデックスの保持期間、戻す条件、スキーマ互換性を定めます。

青緑切り替えは無停止化だけを目的としません。
解析、分割、埋め込みの変更による品質差を、利用者へ影響を与える前に確認し、安全に戻すための仕組みです。

## 3.2.12 観測と運用指標

ジョブの成功件数だけでは、取り込みの健全性を判断できません。
情報源別に、発見件数、取得件数、処理件数、失敗、再試行、隔離、削除を測ります。
各段階について、処理時間、キューの経過時間、処理量も記録します。

情報源の更新から稼働中インデックスへ反映されるまでの時間を、端から端までの鮮度として測ります。
段階ごとのトレースへ、スナップショット、処理版、エラーコードを付けると、取得遅延、OCR混雑、埋め込みの利用上限を区別できます。

[Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)は、機械学習（ML）システムで監視されない依存関係や設定が負債となることを論じました。
新しい例外経路や情報源を追加するときは、対応する状態、指標、警告も追加します。
データが黙って欠落する状態を、ジョブ成功として扱いません。
