# 3.2 Ingestion設計

## この節の役割

- full sync、incremental sync、event-driven updateをどう使い分けるか。
- 同じイベントやファイルを複数回受けても、重複登録しない仕組みをどう作るか。
- 失敗、部分成功、rate limit、長時間停止からどう再開するか。
- 更新、削除、権限剥奪を、どの期限でindexへ反映するか。

## 3.2.1. Ingestionを状態を持つpipelineとして定義する
  - `discover -> fetch -> validate -> parse -> normalize -> chunk -> enrich -> embed -> load -> verify -> publish`を示す。
  - 各stageの入力・出力・version・失敗状態を明示する。
  - 「ジョブ成功」と「検索可能なindex公開」を分ける。

## 3.2.2. source別ingestion契約
  - connector、認証、取得範囲、rate limit、pagination、最大サイズを定義する。
  - source側ID、更新時刻、削除表現、ACL表現をcanonical schemaへ対応づける。
  - 取得不能時に前回snapshotを継続利用するか、公開停止するかを決める。

## 3.2.3. full sync / incremental sync / event-driven update
  - 初回構築・大規模再処理ではfull syncを使う。
  - `updated_at`、cursor、change token、CDCでincremental syncを行う。
  - webhookやevent busを使う場合も、取りこぼし検出用の定期reconciliationを残す。
  - 方式ごとにfreshness、負荷、実装複雑性を比較する。

## 3.2.4. snapshotと処理範囲の固定
  - 同一job内でsourceが変化しても結果を再現できるようsnapshot境界を定義する。
  - `source_snapshot_id`、取得開始・終了時刻、cursor、件数、hashを記録する。
  - parser・chunker・embedding変更時は、入力snapshotを固定して比較する。

## 3.2.5. 冪等性と重複排除
  - `source_id + source_object_id + source_version`を冪等性keyの候補にする。
  - 同一イベントの再送、timeout後の再試行、部分成功でも二重登録しない。
  - content hashは補助に使い、版やACLの違いを消さない。

## 3.2.6. 差分判定と再処理範囲
  - 本文変更、metadata-only変更、ACL変更、削除を分ける。
  - parser変更、chunker変更、embedding変更ごとに必要な再処理stageを定義する。
  - dependency graphを使い、未変更chunkの再embeddingを避ける。

## 3.2.7. retry、checkpoint、dead-letter
  - transient errorとpermanent errorを分類する。
  - exponential backoff、最大試行回数、checkpoint粒度を定義する。
  - 失敗文書をdead-letterへ隔離し、job全体を無条件に成功扱いしない。
  - 再処理時に同じ失敗を無限反復しないため、error codeとparser versionを残す。

## 3.2.8. backpressureとrate limit
  - source API、parser、embedding API、vector storeの各quotaを別々に管理する。
  - queue長、処理時間、retry率からconcurrencyを制御する。
  - 大量更新でオンライン検索SLOを壊さないpublish方式を設計する。

## 3.2.9. 削除・失効・権限変更の伝播
  - source削除、soft delete、版失効、権限剥奪を別eventとして扱う。
  - cache、staging index、active index、replicaへ削除が伝播することを検証する。
  - ACL変更と削除の反映遅延をセキュリティSLOとして持つ。

## 3.2.10. 状態遷移と部分成功
  - `discovered / fetched / parsed / chunked / embedded / indexed / published / failed / quarantined`を定義する。
  - 文書単位とjob単位のstatusを分ける。
  - 部分成功時の公開条件と、欠損率の許容範囲を3.8のquality gateへ接続する。

## 3.2.11. publishとblue/green切替
  - build中のindexをonline queryから隔離する。
  - manifest検証とcanary query後にaliasまたはrouterを切り替える。
  - rollback対象、旧index保持期間、切替監査ログを決める。

## 3.2.12. Observabilityと運用指標
  - source別件数、処理件数、失敗件数、retry率、queue age、throughputを測る。
  - source更新からactive index反映までのend-to-end freshnessを測る。
  - stage別latencyとerror codeをtraceし、ボトルネックを切り分ける。
