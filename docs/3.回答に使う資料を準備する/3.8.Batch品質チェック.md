# 3.8 Batch品質チェック

## 3.8.1. Batch品質チェックの位置付け

ingestion jobが例外なく終わっても、公開可能なknowledge indexとは限らない。Batch quality gateをpass、warn、quarantine、failへ分け、security-critical項目と品質項目を別基準にする。Data Validation for MLがschemaと分布異常をpipelineへ組み込んだ考えをRAGへ移し、全件ruleとsample人手検査を使い分ける。品質チェックを後付けreportではなく、publishを許可する実行可能なtest suiteとして置く。

**参照論文**
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)

## 3.8.2. manifest整合性

最初にmanifestの内部整合を確かめる。source snapshot、parser、chunker、embedding、index schema、code revisionが一つのreleaseを指し、document、chunk、vector、index件数が説明可能に対応しているかを見る。hash、実行時刻、configを保存し、異なるrevisionが混ざるbuildをfailにする。ここで不整合を止めれば、後の検索評価で原因不明の欠損を追う費用を減らせる。

**参照論文**
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.8.3. source・文書レベル検査

source・文書levelでは、owner不明、版不明、更新日不明、機密区分不明を検知する。draft、失効、削除済み、対象外formatの混入と、取得予定件数との差分を確認する。Data Cascadesが示す上流問題の連鎖を避けるため、不明値を空欄のまま通さず、項目ごとにfail、quarantine、warnを決める。急な件数減少や特定sourceの消失もschemaは正しくても異常として扱う。 件数差の理由をsource側の削除、取得失敗、filter除外へ分解し、単なる総数比較で終えない。

**参照論文**
- [P091] [Everyone wants to do the model work, not the data work: Data Cascades in High-Stakes AI](https://dl.acm.org/doi/10.1145/3411764.3445518)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.8.4. parse品質

parse品質は成功statusではなく、内容と構造を検査する。空出力、極端な短さ、文字化け、重複pageを全件で探し、heading、reading order、table、caption、bboxは代表documentでgoldと比較する。OCR/layout confidenceの低いfileを隔離し、文書class別に率を出す。平均文字抽出率が高くても、数字や列対応を誤るscan契約書を見逃さないよう、downstream QA caseも併用する。

**参照論文**
- [P099] [PubLayNet: largest dataset ever for document layout analysis](https://arxiv.org/abs/1908.07836)
- [P090] [DocLayNet: A Large Human-Annotated Dataset for Document-Layout Analysis](https://arxiv.org/abs/2206.01062)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.8.5. chunk品質

chunk品質ではtoken長分布、極小・極大、overlap、duplicateを測るだけでなく、意味上の破損をrule化する。headingだけ、footerだけ、FAQのQ/A分離、table header欠落、code途中切断をdocument type別に検出する。parent-childと元block参照の欠損もfailにする。gold spanが一chunkへ収まる割合と、重複chunkがtop-kを占める率を合わせ、短さだけを最適化しない。

**参照論文**
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.8.6. metadata完全性

metadata完全性ではdocument_id、chunk_id、source、version、page、section、languageを必須として型と一意性を検査する。effective period、status、source trustの矛盾や、filterable fieldがstore制約を超えていないかも見る。単なるNULL率ではなく、citationに必要な欠損、ACLに必要な欠損、ranking補助の欠損へseverityを付ける。schema versionを跨ぐchunkが混在したらmigrationまたは隔離へ送る。

**参照論文**
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)

## 3.8.7. ACL品質とnegative test

ACL品質はallow caseよりnegative caseを重く見る。tenant、group、role、classificationの欠損をdeny-by-defaultとし、文書からchunkへの継承件数を照合する。代表principalでtenant横断、剥奪直後、期限切れ、NULL属性を検索し、activeだけでなくstaging、cache、旧indexにも漏えいがないか確かめる。ACL violationは平均品質へ混ぜず一件でもrelease failとし、過剰filterによるgold欠落は別の改善課題にする。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P144] [Curator: Efficient Indexing for Multi-Tenant Vector Databases](https://arxiv.org/abs/2401.07119)

## 3.8.8. 重複・競合・鮮度

exact duplicateとnear duplicate、同一文書の版競合を分けて測る。active versionが複数ある場合は単純統合せず、authorityとeffective periodを確認する。source updated_at、ingested_at、indexed_at、published_atからfreshness lagを計算し、特定sourceだけ古い状態を見つける。矛盾文書は自動多数決せず、優先規則がなければquarantineまたはconflict labelへ送る。

**参照論文**
- [P130] [On the Resemblance and Containment of Documents](https://ieeexplore.ieee.org/document/666900)
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)

## 3.8.9. embedding・vector検査

vectorはdimension、dtype、normalization、NaN、Inf、zero vectorを全件検査する。query/documentでmodel revision、prefix、metricが一致するかをmanifestと照合し、同じcontent hashの不要重複を探す。数値検査だけではsemantic failureを見逃すため、代表queryのnearest neighborをgoldまたは人手でspot checkする。embedding serviceの部分失敗で特定languageだけ欠けていないか、slice別件数も確認する。

**参照論文**
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.8.10. index検査

index検査ではactive version、schema、metric、dimension、件数をmanifestと照合する。tombstone、削除済みID、ACL変更済みIDが結果へ出ないことをcanaryで確かめ、zero resultや特定source偏重、score分布の急変を検知する。ANNはexact search sampleと比較してrecall低下を測る。buildが速くても、filter selectivityの高いcaseでgoldを落とすならpublishしない。

**参照論文**
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)

## 3.8.11. golden documentと回帰試験

golden documentは平均的なfileだけでなく、scan、table、長文、FAQ、multi-column、多言語、壊れやすい例を固定する。parser・chunker更新前後でblock、chunk、metadata、citation locationをdiffし、expected failureもfixtureにする。ML Test Scoreの継続testing思想を取り入れ、incidentで見つかった文書をsuiteへ追加する。成功例だけを増やさず、「公開されないべき文書」が今も隔離されるかを守る。

**参照論文**
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)

## 3.8.12. publish判定と隔離

publish gateはsecurity項目をzero tolerance、品質項目を閾値と影響範囲で判定する。隔離文書を除くpartial publishが許されるか、欠損が回答範囲へどう影響するかを明記する。approver、例外期限、rollback条件、監査logを残し、manual overrideを永続設定にしない。RAGCheckerのようなdownstream診断もcanaryへ加え、batch内部のscore改善が最終citationを悪化させていないか確認する。

**参照論文**
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
