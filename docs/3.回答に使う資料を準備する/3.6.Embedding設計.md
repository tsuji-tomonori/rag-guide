# 3.6 Embedding設計

## 3.6.1. Embeddingとベクトル空間の直感

embeddingはtextを固定長vectorへ写すが、その距離は学習された表現上の近さであり、真偽、権限、鮮度を意味しない。2次元図で直感を示した後、高次元空間をそのまま可視化したものではないと断る。queryとchunkを同一または対応する空間へ置き、similarityで候補を作る。semantic matchの強みと、exact identifierや否定の弱さを最初に並べ、hybridとmetadataが必要になる理由へつなぐ。

**参照論文**
- [P024] [Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks](https://arxiv.org/abs/1908.10084)
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)

## 3.6.2. bi-encoder / dual-encoder

bi-encoderはqueryとdocumentを独立にencodeできるため、document側をoffline計算して大量検索へ使える。Sentence-BERTはsentence pair比較を効率化し、DPRはquery encoderとpassage encoderをQA向けに学習した。性能を分けたのはarchitectureだけでなく、in-batch negativeとhard negativeの設計だった。cross-encoderより速い代わりに細かな相互作用を省くため、first-stage recallと後段rerankの役割分担を明確にする。

**参照論文**
- [P024] [Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks](https://arxiv.org/abs/1908.10084)
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)

## 3.6.3. similarity metricと正規化

cosine、inner product、Euclideanは同じscore名ではない。modelが想定するnormalizationとmetricをindex/queryで一致させ、L2 normalizationの有無をversion contractへ含める。正規化済みvectorではcosineとinner productが順位上近づく場合があるが、実装設定を推測で置き換えない。metric変更時は同じvectorを再利用できてもrankingが変わるため、golden queryでrecallとscore分布を再評価し、scoreを確率へ変換しない。

**参照論文**
- [P024] [Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks](https://arxiv.org/abs/1908.10084)
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)

## 3.6.5. domain適合

公開benchmarkの一位が社内略語、法令、code、日英混在で一位とは限らない。MTEBは多数taskと言語でembeddingを比較し、model間の優劣がtaskごとに変わることを示した。zero-shot modelをbaselineに置き、domain query、hard negative、cross-languageをslice評価してからfine-tuningを検討する。合成queryがtestへ漏れないよう分割し、製品体系や語彙の変化をembedding driftとして監視する。

**参照論文**
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)
- [P029] [Unsupervised Dense Information Retrieval with Contrastive Learning](https://arxiv.org/abs/2112.09118)
- [P096] [M3-Embedding: Multi-Linguality, Multi-Functionality, Multi-Granularity Text Embeddings Through Self-Knowledge Distillation](https://arxiv.org/abs/2402.03216)

## 3.6.6. embeddingが苦手な情報

型番、条文番号、URL、短い略語、数値は、意味空間で近い別物と混同されやすい。DPRの分析でもBM25が希少なkey phraseを正確に拾う例があり、denseだけが常に上位ではなかった。single-vectorは長いchunkの複数話題や表の細部も圧縮する。苦手なquery sliceを明示し、exact field、BM25、multi-vector、metadata filterへ役割を分ける。embedding modelを変える前に、問題が表現方式かchunkかを切り分ける。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)
- [P028] [ColBERT: Efficient and Effective Passage Search via Contextualized Late Interaction over BERT](https://arxiv.org/abs/2004.12832)

## 3.6.7. 文書長とchunkとの相互作用

chunkが長いほど一vectorに複数意味が混ざり、短いほど周辺contextを失う。chunkingとembeddingを同時に変えると、改善原因を特定できないため別々にablationする。titleやheadingを本文へ付ける案、別fieldにする案、Late Chunkingで長文contextをencodeしてからpoolingする案を比較する。代名詞や省略が多い文書、表、multi-topic文書をslice化し、retrievalだけでなくcitation単位の明瞭さも測る。

**参照論文**
- [P095] [Late Chunking: Contextual Chunk Embeddings Using Long-Context Embedding Models](https://arxiv.org/abs/2409.04701)
- [P080] [Dense X Retrieval: What Retrieval Granularity Should We Use?](https://arxiv.org/abs/2312.06648)

## 3.6.8. 次元数の設計

次元数は品質だけでなくstorage、memory、network、ANN latencyへ効く。Matryoshka Representation Learningは一つの表現のprefixで精度と資源を調整する学習を提案したが、任意modelのvectorを単純truncateしてよいとは限らない。256、512、1024などを別indexで比較し、recall、nDCG、latency、index sizeを同時に記録する。dimensionはschemaの一部として固定し、変更を再embedding releaseとして扱う。

**参照論文**
- [P019] [Matryoshka Representation Learning](https://arxiv.org/abs/2205.13147)

## 3.6.9. batching、cache、throughput

大量embeddingではbatch sizeを上げるだけで最適化しない。provider rate limit、GPU memory、retry、checkpointを測り、content hashとmodel versionをkeyにcacheする。未変更chunkを再計算せず、失敗chunkをquarantineしてindex欠損を検知する。throughputを上げてonline QPSやsource freshnessを壊さないようqueueを制御し、cost/document、error率、p95処理時間を観測する。cache hitでもACLやmetadata更新は別処理として反映する。

**参照論文**
- [P092] [FlumeJava: Easy, Efficient Data-Parallel Pipelines](https://research.google/pubs/flumejava-easy-efficient-data-parallel-pipelines/)
- [P094] [Hidden Technical Debt in Machine Learning Systems](https://papers.nips.cc/paper/5656-hidden-technical-debt-in-machine-learning-systems)

## 3.6.10. versioningと再embedding

embedding_model_id、revision、dimension、normalization、query/document prefix、tokenizationを一つのversion bundleへ置く。異なるmodelのvectorを同一空間へ混在させず、新modelは別indexへ全件生成する。MTEBと業務gold setで比較し、query shadowing後にaliasを切り替える。旧indexの保持期間とrollback条件を決め、model providerのsilent updateもrevision差として検知する。再embeddingを単なるbatch jobでなく検索分布のreleaseとして管理する。

**参照論文**
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)
- [P088] [Delta Lake: High-Performance ACID Table Storage over Cloud Object Stores](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)
