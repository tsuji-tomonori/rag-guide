# 3.3 文書解析

## 3.3.1. 解析・正規化の役割
- 目的を「文字抽出」ではなく「検索・引用可能な構造への変換」と定義する。
- parser品質がchunk境界、embedding、引用、表QAに連鎖することを示す。
- 解析失敗とretrieval失敗を区別できるようにする。

## 3.3.2. 入力形式と失敗モードの分類
- born-digital PDF、スキャンPDF、HTML、Markdown、Word、Excel、PowerPointを分ける。
- 画像、表、数式、コード、脚注、多段組、縦書き、回転ページを分類する。
- 暗号化、破損、フォント埋め込み、文字コード、巨大ファイルを例外として扱う。

## 3.3.3. canonical document schema
- document、page、block、table、image、linkの階層を定義する。
- blockに`type`、`text`、`page`、`bbox`、`reading_order`、`confidence`を持たせる。
- 元ファイル、parser出力、正規化済み出力を分離保存する。
- 後続chunkから元ページと座標へ戻れるstable IDを設計する。

## 3.3.4. レイアウト解析と読み順
- heading、paragraph、list、table、figure、caption、footnoteを識別する。
- 二段組、サイドバー、ヘッダー・フッターの読み順を復元する。
- レイアウトモデルの学習ドメインと自社文書との差を確認する。
- 文書画像とテキストレイヤーの両方を照合する方式を説明する。

## 3.3.5. OCRとOCR-free方式
- OCRが必要な条件と、不要な条件を判定する。
- 文字単位・行単位のconfidenceを保持する。
- 低confidenceページを隔離し、人手確認または別parserへroutingする。
- OCR-freeモデルの利点と、出力拘束・再現性・計算コストの限界を示す。

## 3.3.6. 表、図、数式、コードの扱い
- 表はcaption、header、row、column、merged cellを保持する。
- 表全体のMarkdown表現と、セル構造JSONを併存させる。
- 図はcaption、周辺本文、画像ID、bboxを紐づける。
- 数式・コードは空白、改行、記号を過度に正規化しない。

## 3.3.7. テキスト正規化
- Unicode正規化、改行、空白、ハイフネーション、全半角を扱う。
- header/footer、ページ番号、透かし、目次の反復ノイズを除去する。
- URL、メール、製品コード、条文番号など、検索に必要な表記は保持する。
- 原文と正規化文を別フィールドにし、引用では原文を優先する。

## 3.3.8. 構造の復元
- heading hierarchy、list nesting、section pathを復元する。
- pageを跨ぐ段落、表、脚注の結合規則を定める。
- HTMLやMarkdownではDOM・heading構造を最大限利用する。
- Office文書ではコメント、変更履歴、非表示要素の採否を決める。

## 3.3.9. parser routing
- 文書形式、画像比率、表密度、言語、ページ数でparserを振り分ける。
- default parser、高度parser、OCR、VLM系parserのfallback順を定義する。
- parser選択を固定せず、文書クラスごとに評価する。
- parser変更時は新しいdata sourceまたはindex versionへ分離する。

## 3.3.10. 品質検査
- parse成功率、文字欠落率、読み順、表構造、ページ対応を測る。
- 代表文書のgold parseを用意し、parser更新時に回帰試験する。
- 空文書、極端に短い出力、重複ページ、文字化けを自動検知する。
- `parser_version`、`ocr_confidence`、`layout_confidence`を記録する。

## 3.3.11. Chunkingへの引き渡し契約
- chunkerへ渡すblock種別、heading path、page、table IDを定義する。
- parserが不確かな境界をchunkerが確定事項として扱わないようconfidenceを渡す。
- 再解析時に、影響するchunkだけを再生成できる依存関係を保持する。
