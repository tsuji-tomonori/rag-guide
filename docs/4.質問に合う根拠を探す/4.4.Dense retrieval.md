# 4.4 Dense retrieval

## 4.4.1. Dense retrievalの直感

Dense retrievalは、質問とchunkを同じ意味空間へ写し、単語が一致しなくても近い内容を探す。DPRは「bad guy」と「villain」のような語彙差をdual encoderで越え、複数QAデータセットでBM25を大きく上回った。一方、類似度は真偽、権限、鮮度を表さない。したがってベクトルは候補生成の一信号と位置づけ、metadata filterと後段rerankで補う。二次元図は直感に限り、実際の高次元配置を過度に意味解釈しない。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P024] [Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks](https://arxiv.org/abs/1908.10084)

## 4.4.2. Bi-encoder / dual encoder

Bi-encoderの利点は、queryとdocumentを別々にencodeできる点にある。DPRではpassage側を事前計算してFAISSへ置き、質問時にはquery vectorだけを作って内積検索する。この分離により数千万passageを扱えるが、queryとcandidateを同時に読むcross-encoderより細かな対応は弱い。offlineで全documentを計算し、onlineで大量候補を高速回収する責務分担を示し、精密判定は5章のrerankerへ渡す。encoder、prefix、tokenizerの契約も明示する。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P024] [Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks](https://arxiv.org/abs/1908.10084)

## 4.4.3. 学習データとnegative

Dense retrieverの品質はmodel名より、何を負例として学ばせたかに強く左右される。DPRはin-batch negativesとBM25由来のhard negativeを組み合わせ、単純なdual encoderでも検索精度を大幅に高めた。ANCEはANN indexから難しい負例を継続的に採ることで学習を強化する。ただし旧版と新版、同義の別資料を誤ってnegativeにすれば、必要な近傍を遠ざける。random、hard、false negativeを区別し、document familyと版を考慮して教師対を作る。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P026] [Approximate Nearest Neighbor Negative Contrastive Learning for Dense Text Retrieval](https://arxiv.org/abs/2007.00808)

## 4.4.4. Embedding model選定

公開benchmarkの首位をそのまま採用せず、質問と言語の分布からmodelを選ぶ。MTEBはretrievalだけでなく分類や類似度など多数task・datasetを横断し、単一scoreではembeddingの適性を語れないことを示した。日本語のみ、日英混在、cross-language、専門略語を別sliceにし、最大長、dimension、license、throughputも比較する。E5のようにinstructionやquery/document prefixを要求するmodelでは、仕様通りの入力を揃えた上でgolden setを評価する。

**参照論文**
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)
- [P118] [Text Embeddings by Weakly-Supervised Contrastive Pre-training](https://arxiv.org/abs/2212.03533)
- [P096] [M3-Embedding: Multi-Linguality, Multi-Functionality, Multi-Granularity Text Embeddings Through Self-Knowledge Distillation](https://arxiv.org/abs/2402.03216)

## 4.4.5. Query・document encodeの整合性

document vectorをmodel Aで作り、queryをmodel Bで作れば、次元が同じでも比較可能とは限らない。さらに同一modelでも、query prefixの付け忘れ、正規化の不一致、truncation差で順位が変わる。E5はtask instructionを表現学習へ組み込み、入力形式が性能の一部であることを示す。model revision、tokenizer、prefix、max length、normalizationを一つのversion契約にし、query cache keyにも含める。index versionと組でtraceし、混在をbuild時に拒否する。

**参照論文**
- [P118] [Text Embeddings by Weakly-Supervised Contrastive Pre-training](https://arxiv.org/abs/2212.03533)
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)

## 4.4.6. Similarity metric

cosineは方向、dot productは方向と大きさ、L2は距離を測るが、どれが常に優れるわけではない。SBERTはcosineで文類似を扱い、DPRは内積によるMaximum Inner Product Searchを用いた。重要なのは学習時に想定されたmetricと、index・query時の正規化を一致させることである。scoreは確率ではなく相対的な近さなので、固定閾値を正答信頼度と呼ばない。model・corpusごとに順位品質を比較し、metric変更もrelease対象とする。

**参照論文**
- [P024] [Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks](https://arxiv.org/abs/1908.10084)
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)

## 4.4.7. ANN検索

全vectorとの厳密比較は基準になるが、corpusが大きいと遅い。HNSWは近傍graphを階層化し、入口から近いnodeへ移動して候補を絞ることで高速化する。`M`、`efConstruction`、`efSearch`を上げればrecallは高まりやすい反面、memory、build時間、latencyが増える。ANNは近似である以上、top-kを増やすだけでなくexact検索とのsample比較が必要である。ACL filterのselectivity別にも測り、速度のために認可を外さない。

**参照論文**
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)
- [P016] [Billion-scale similarity search with GPUs](https://arxiv.org/abs/1702.08734)

## 4.4.9. Late interaction / ColBERT

ColBERTは文書全体を一つのvectorへ圧縮せず、queryとdocumentのtoken vectorを保持し、各query tokenが最も近いdocument tokenを探すMaxSimでscoreを作る。これにより専門語と周辺文脈を細粒度に照合しつつ、document表現は事前計算できる。ColBERTv2はresidual compressionでindex負担を抑えたが、single-vectorよりstorageとquery計算は重い。長い技術文書や固有表現で改善が見える場合に限定して導入する。

**参照論文**
- [P028] [ColBERT: Efficient and Effective Passage Search via Contextualized Late Interaction over BERT](https://arxiv.org/abs/2004.12832)
- [P108] [ColBERTv2: Effective and Efficient Retrieval via Lightweight Late Interaction](https://arxiv.org/abs/2112.01488)

## 4.4.10. Multi-vector retrieval

一つのchunkに複数vectorを持たせると、title、sentence、table row、image regionのどこが質問へ反応したかを残せる。ColBERTのtoken-level表現はその極端な例で、BGE M3はdense、sparse、multi-vectorを一モデルで扱う方向を示した。hitしたvectorをdocument候補へ集約する際は、最大値、平均、quotaで順位が変わる。重複candidateを統合し、storage増加とtrace複雑化を測る。最終citationはvectorではなく元spanへ着地させる。

**参照論文**
- [P028] [ColBERT: Efficient and Effective Passage Search via Contextualized Late Interaction over BERT](https://arxiv.org/abs/2004.12832)
- [P096] [M3-Embedding: Multi-Linguality, Multi-Functionality, Multi-Granularity Text Embeddings Through Self-Knowledge Distillation](https://arxiv.org/abs/2402.03216)

## 4.4.11. 多言語・ドメイン適合

多言語modelを名乗っていても、日本語質問から英語仕様書を探す性能と、日本語同士の性能は同じではない。MTEBとM3-Embeddingは言語・task・粒度を分けた評価の必要性を示す。社内略語、製品コード、否定、数値を専用sliceにし、zero-shot baselineを先に固定する。domain tuningではhard negativeを使えるが、適応後はcorpus全件を再embeddingし、一般queryの回帰も確認する。翻訳経路を使う場合は原queryと訳文を両方traceする。

**参照論文**
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)
- [P096] [M3-Embedding: Multi-Linguality, Multi-Functionality, Multi-Granularity Text Embeddings Through Self-Knowledge Distillation](https://arxiv.org/abs/2402.03216)
- [P026] [Approximate Nearest Neighbor Negative Contrastive Learning for Dense Text Retrieval](https://arxiv.org/abs/2007.00808)

## 4.4.12. Metadata filterとの組み合わせ

vector近傍が意味的に正しくても、tenantや有効日が違えば利用できない。pre-filterは安全な集合内で探せるが、条件が厳しいとgraph探索が難しくなる。search-then-filterは実装しやすい反面、top-kが権限外候補で埋まり正解が残らない。ACORNはpredicateを探索へ組み込み、filtered ANNの性能課題に取り組む。実装ではfilter selectivityごとにrecallとlatencyを測り、oversamplingやexact fallbackを用意するが、ACL自体は決して緩和しない。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P129] [NaviX: A Native Vector Index Design for Graph DBMSs With Robust Predicate-Agnostic Search Performance](https://arxiv.org/abs/2506.23397)
