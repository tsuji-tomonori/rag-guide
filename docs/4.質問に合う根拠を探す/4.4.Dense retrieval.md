# 4.4 Dense retrieval

## この節の役割

queryとchunkをembeddingへ変換し、ベクトル類似度で検索する仕組みを扱う。モデル選定だけでなく、queryとdocumentのencode条件、ANN、index version、多言語、late interactionまでを一体で説明する。

Dense retrievalは言い換えや意味類似に強い一方、型番、数値、否定、権限、鮮度を自然には保証しない。sparse retrieval、metadata filter、rerankingと組み合わせて使用する。

## 4.4.1. Dense retrievalの直感
  - queryとchunkを同じベクトル空間へ写す。
  - 単語一致がなくても意味の近い候補を取得できる。
  - embedding similarityは事実の正しさ、権限、鮮度を保証しない。
  - 検索対象は3章で作成したchunk embeddingであることを確認する。

## 4.4.2. Bi-encoder / dual encoder
  - query encoderとdocument encoderを分けて説明する。
  - document embeddingをoffline計算できる利点を示す。
  - dot productまたはcosineで大量候補を高速に取る。
  - cross-encoderとの速度・精度差は5章へ接続する。

## 4.4.3. 学習データとnegative
  - positive pair
  - random negative
  - in-batch negative
  - hard negative
  - false negative
  - hard negative miningがdomain適合に与える効果とリスクを説明する。
  - version違いの同一文書をfalse negativeにしない。

## 4.4.4. Embedding model選定
  - 日本語のみ、日英混在、cross-lingualで別々に評価する。
  - query prefix、document prefix、instructionの有無をmodel仕様に合わせる。
  - general benchmarkだけでなく業務golden setを使う。
  - license、dimension、最大入力長、throughput、batch sizeを選定条件に含める。
  - modelを変更した場合は再embeddingが必要になる。

## 4.4.5. Query・document encodeの整合性
  - 同じmodel family、dimension、normalizationを使う。
  - query用prefixとdocument用prefixを取り違えない。
  - preprocessing、tokenization、truncation条件をversion化する。
  - `embedding_model_version`と`index_version`をtraceへ残す。
  - query vectorをcacheする場合はmodel versionをkeyに含める。

## 4.4.6. Similarity metric
  - cosine similarity
  - dot product
  - Euclidean distance
  - embedding正規化との関係
  - model・serviceの推奨と実データ評価の両方で決める。
  - scoreを確率や正答信頼度として解釈しない。

## 4.4.7. ANN検索
  - exact nearest neighborとapproximate nearest neighborを比較する。
  - HNSWの`m`、`ef_construction`、`ef_search`を説明する。
  - top-kを増やすとrecall、latency、noiseが増える。
  - index parameterは評価sweepで決める。
  - filtered ANNではfilter selectivity別にrecallを測る。

## 4.4.9. Late interaction / ColBERT
  - queryとdocumentをtoken-level vectorとして保持する。
  - MaxSimによるlate interactionの直感を説明する。
  - single-vectorより細粒度な語句対応を評価できる。
  - index size、memory、query計算量が増える。
  - technical document、長いchunk、固有表現が混在する文書で評価する。

## 4.4.10. Multi-vector retrieval
  - 一つのchunkまたはdocumentに複数vectorを持つ。
  - sentence、paragraph、token、title、image regionなどの粒度を示す。
  - vectorごとのhitをdocument candidateへ集約する。
  - 重複candidate、score aggregation、storageを扱う。
  - どのvectorがhitしたかをtraceとcitationへ残す。

## 4.4.11. 多言語・ドメイン適合
  - 日本語queryと英語文書のcross-language retrievalを評価する。
  - domain term、略語、固有名詞、否定、数値を専用sliceで測る。
  - fine-tuning前にzero-shot baselineを残す。
  - domain adaptation後は全corpusを再embeddingし、回帰評価する。

## 4.4.12. Metadata filterとの組み合わせ
  - tenant、ACL、active statusはvector検索前または検索時に適用する。
  - product、region、date、versionはquery confidenceに応じて適用する。
  - search-then-filterでtop-k不足が起きる問題を説明する。
  - filtered ANN、oversampling、fallbackを比較する。
