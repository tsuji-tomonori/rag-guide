# 4.2 Query変換

## この節の役割

正規化済みqueryのままでは正解根拠を回収できない場合に、検索用queryを積極的に変換する方法を扱う。目的は文章を自然にすることではなく、retrieval recallを改善することである。

Query変換は常時有効化せず、語彙不一致、短すぎる質問、会話省略、multi-hopなど、対象となる失敗を定義して適用する。元query、変換内容、検索結果を必ずtraceへ残す。

## 4.2.1. Query変換を使う条件
  - 語彙不一致が疑われる。
  - queryが短く、対象や条件が検索表現に現れていない。
  - 文書側が説明文で、query側が単語だけである。
  - 同義語、旧名称、専門語が多い。
  - 複数の事実を順番に調べる必要がある。
  - 初回検索のrecallまたはevidence sufficiencyが低い。
  - chunking、index freshness、ACL filterの問題をquery変換で隠さない。

## 4.2.2. Query rewriting
  - 元の意図と制約を保持したstandalone queryを生成する。
  - product、region、期間、否定条件、比較条件を明示する。
  - 元queryとrewrite queryの両方を検索するfallbackを用意する。
  - rewrite modelが知らないentityや条件を追加しないよう制約する。
  - rewrite前後の差分を保存する。

## 4.2.3. 決定的なQuery expansion
  - 同義語、略語、旧名称・新名称、専門語を辞書から追加する。
  - 日英表記、製品コード、社内用語を対応づける。
  - expansion語ごとに出典となる辞書versionを残す。
  - exact match語はoriginal formも保持する。
  - 上位概念・下位概念の追加はquery typeに応じて制限する。

## 4.2.4. LLMによるQuery expansion
  - 複数の言い換え候補を生成する。
  - original queryの必須条件を各queryへ継承する。
  - 生成数を2〜5件程度から評価する。
  - 類似queryをembeddingまたは文字列で重複除去する。
  - 生成queryを回答根拠として扱わない。

## 4.2.5. Multi-query retrieval
  - 一つの質問から異なる観点のqueryを作る。
  - queryごとのretrieverとtop-kを設定する。
  - query内の検索結果とquery間の検索結果の統合順序を決める。
  - union、RRF、weighted fusionを比較する。
  - query数×retriever数で候補数、レイテンシ、費用を管理する。

## 4.2.6. Query2doc
  - queryに対応しそうなpseudo-documentを生成する。
  - sparse retrievalでは文書に現れやすい語彙を追加する。
  - dense retrievalではquery表現を文書表現へ近づける。
  - pseudo-documentと実際に取得した文書を明確に区別する。
  - pseudo-document内の固有名詞が検索を誤誘導する失敗を評価する。

## 4.2.7. HyDE
  - 仮想的な回答文書を生成する。
  - 仮想文書のembeddingをquery vectorとして検索する。
  - 生成文の事実性ではなく、検索表現としての有効性を評価する。
  - 型番、条文、契約文言などexact matchが重要なqueryではsparseを併用する。
  - original query embeddingとのunionまたはfusionを比較する。

## 4.2.8. Query decomposition
  - 複雑な質問を独立したサブ質問へ分ける。
  - 時系列、地域、対象、条件、例外を分離する。
  - サブ質問間の依存関係をDAGまたは順序として表す。
  - 各サブ質問の検索結果と根拠を別々に保持する。
  - `max_subqueries`、`max_retrievals`、timeoutを設定する。

## 4.2.9. Interleaved retrieval
  - 最初に全サブ質問を作る方式と、途中の根拠から次queryを作る方式を比較する。
  - IRCoTのようにreasoning stepとretrievalを交互に行う。
  - 同じquery・同じ結果を繰り返すloopを検知する。
  - evidence chainとstop reasonをtraceへ残す。
  - 単純な質問を高コストflowへ送らない。

## 4.2.10. 意図保持と安全性
  - 利用者が指定した否定、期間、対象を削除しない。
  - ACLやtenant条件をquery変換へ委ねない。
  - query変換で推定したfilterをhard filterにする場合はconfidence閾値を持つ。
  - 外部Web検索へ拡張する場合はsource allowlistと監査を用意する。
  - 変換失敗時はoriginal queryへ戻す。
