# 4.2 Query変換

## 4.2.1. Query変換を使う条件

Query変換は検索の標準手順ではなく、語彙不一致という診断に対する処方である。Rewrite-Retrieve-Readは検索向けに質問を書き直す利点を示す一方、書換器が追加した語が検索を誤誘導する危険も残す。そこで、短すぎる質問、旧称・専門語、会話省略、multi-hop、初回recall不足を起動条件とし、index欠損やACL過剰制限は別問題として扱う。変換なしの結果をbaselineとして保存し、改善がないquery typeには適用しない。

**参照論文**
- [P115] [Query Rewriting for Retrieval-Augmented Large Language Models](https://arxiv.org/abs/2305.14283)
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)

## 4.2.2. Query rewriting

書き換えの仕事は質問を美文にすることではなく、検索に必要な対象と制約を露出させることにある。Rewrite-Retrieve-Readでは、LLM向けの元質問ではなくretriever向けqueryを生成し、検索結果をreaderへ渡す三段構成を採る。実務では製品、地域、期間、否定を固定し、未知の固有名詞を補わない制約を置く。元queryとrewriteを並行検索して候補を統合すれば、書換え失敗を緩和できる。差分、model、prompt、採用結果は監査用に残す。

**参照論文**
- [P115] [Query Rewriting for Retrieval-Augmented Large Language Models](https://arxiv.org/abs/2305.14283)
- [P062] [CONQRR: Conversational Query Rewriting for Retrieval with Reinforcement Learning](https://arxiv.org/abs/2112.08558)

## 4.2.3. 決定的なQuery expansion

略語辞書による展開は、LLM生成より地味だが再現性が高い。「S3V」を正式名へ、「旧製品名」を現名称へ結ぶとき、追加語の出典と版を説明できる。Document Expansionの研究は文書側へ想定queryを足す発想を示したが、query側でも管理語彙を使えば同じ語彙差を狭められる。exact identifierの原形は削らず、上位・下位概念の追加は質問型に応じて制限する。辞書更新をindex・評価versionと結び、過剰展開を回帰試験する。

**参照論文**
- [P109] [Document Expansion by Query Prediction](https://arxiv.org/abs/1904.08375)
- [P119] [Unsupervised Domain Adaptation for Sparse Retrieval by Filling Vocabulary and Word Frequency Gaps](https://arxiv.org/abs/2211.03988)

## 4.2.4. LLMによるQuery expansion

辞書にない言い換えを拾う場面では、LLMに異なる観点の検索文を少数生成させる。重要なのは数ではなく、各queryが元の必須条件を保持しつつ別の語彙面を開くことである。Query2docやRewrite-Retrieve-Readが示すように、生成表現はrecallを上げ得る一方、もっともらしい固有名詞を混ぜる。2〜5件から始め、相互に近いqueryは除き、生成文を証拠として扱わない。一つのqueryが候補を独占しないquotaも設定する。

**参照論文**
- [P031] [Query2doc: Query Expansion with Large Language Models](https://arxiv.org/abs/2303.07678)
- [P115] [Query Rewriting for Retrieval-Augmented Large Language Models](https://arxiv.org/abs/2305.14283)

## 4.2.5. Multi-query retrieval

一つの質問が複数の表現を持つなら、検索結果も一つの順位表に早々と潰さない。各queryをsparse・denseへ送り、まずquery内で候補を整理し、その後query間でunionやRRFを行う。RRFはscore尺度に依存せず順位を統合できるため、生成queryごとのscore分布が違う場合に扱いやすい。ただしquery数×retriever数×top-kで候補と費用が膨らむ。subqueryごとの目的、候補上限、重複統合、寄与queryを記録し、recall増分が小さい枝を切る。

**参照論文**
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P106] [An Analysis of Fusion Functions for Hybrid Retrieval](https://arxiv.org/abs/2210.11934)

## 4.2.6. Query2doc

Query2docは短い質問を、答えが載っていそうな擬似文書へ広げる。文書らしい語彙が増えるため、BM25では一致語を補い、denseではquery表現をpassage側の分布へ近づけられる。論文では複数retriever・benchmarkで改善を報告したが、擬似文書は事実ではなく検索用の仮説である。生成された人名や製品名を引用へ流用せず、原query検索とのfusionを保つ。どの語がhitへ寄与したかを追い、誤誘導の多いdomainでは辞書展開へ戻す。

**参照論文**
- [P031] [Query2doc: Query Expansion with Large Language Models](https://arxiv.org/abs/2303.07678)
- [P109] [Document Expansion by Query Prediction](https://arxiv.org/abs/1904.08375)

## 4.2.7. HyDE

HyDEは正解文書を直接予測せず、まず仮想的な回答文書を生成し、そのembedding近傍から実文書を探す。relevance labelなしでもqueryとdocumentの表現差を埋める点が特徴で、zero-shot dense retrievalの改善を狙う。しかし仮想文書が誤った事実を含んでも、検索表現として有効な場合がある一方、固有名詞や数値を誤ると逆効果になる。原query vectorとのunionを残し、型番・条文はsparse経路で保護し、query slice別に効果を測る。

**参照論文**
- [P030] [Precise Zero-Shot Dense Retrieval without Relevance Labels](https://arxiv.org/abs/2212.10496)
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)

## 4.2.8. Query decomposition

「A社とB社の2024年の変更点を比較し、影響を説明せよ」は一回の検索では論点が混ざる。分解では対象、時点、変更、影響をサブ質問へ切り、依存関係をDAGとして表す。IRCoTが示すように、multi-hopでは途中の証拠が次の問いを決めることもあるため、最初に全てを固定する方法と逐次生成を使い分ける。各subqueryの根拠を別管理し、統合時に欠けた枝を可視化する。最大件数と停止条件を決め、分解自体の過剰推論を防ぐ。

**参照論文**
- [P110] [Interleaving Retrieval with Chain-of-Thought Reasoning for Knowledge-Intensive Multi-Step Questions](https://arxiv.org/abs/2212.10509)
- [P105] [Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity](https://arxiv.org/abs/2403.14403)

## 4.2.9. Interleaved retrieval

Interleaved retrievalでは、検索と推論を交互に進める。「創業者は誰か」を調べた結果から、その人物の別会社を探すように、次queryは前のevidenceに依存する。IRCoTはこの循環でmulti-step QAを改善したが、途中の誤根拠が後続検索を連鎖的に歪める。したがって各stepで根拠十分性と新規性を判定し、同一query・同一document集合の反復を止める。evidence chain、生成query、採否理由、stop reasonを残し、単純質問はこの経路へ送らない。

**参照論文**
- [P110] [Interleaving Retrieval with Chain-of-Thought Reasoning for Knowledge-Intensive Multi-Step Questions](https://arxiv.org/abs/2212.10509)
- [P079] [Corrective Retrieval Augmented Generation](https://arxiv.org/abs/2401.15884)

## 4.2.10. 意図保持と安全性

変換器には「元質問を守る」という安全契約が必要である。利用者が指定した対象、否定、期間、版を必須slotとして固定し、生成queryが削除・反転していないか検査する。ACLとtenantは言語モデルへ委ねず、認証由来のfilter ASTとして常にAND結合する。外部Webへ拡張する場合もsource allowlistを越えさせない。変換がparse不能、意味差過大、低確信ならoriginalへ戻す。このfail-safe設計により、recall改善が権限や意図の逸脱を正当化しない。

**参照論文**
- [P115] [Query Rewriting for Retrieval-Augmented Large Language Models](https://arxiv.org/abs/2305.14283)
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
