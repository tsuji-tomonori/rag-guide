# 4.3 Sparse retrieval

## 4.3.1. Sparse retrievalの位置づけ

Sparse retrievalは古い方式だから残すのではない。語彙が一致した事実を直接手掛かりにできるため、型番、条文、API名のように一文字の差が意味を変える質問でdenseとは別の強みを持つ。BM25の確率的枠組みは、希少語の重みと文書長補正を組み合わせ、説明可能なbaselineを与える。一方、同義語や言い換えは拾いにくい。本節では優劣ではなく失敗モードを対比し、まずBM25を固定してからdense・hybridの増分を測る。

**参照論文**
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)
- [P027] [BEIR: A Heterogeneous Benchmark for Zero-shot Evaluation of Information Retrieval Models](https://arxiv.org/abs/2104.08663)

## 4.3.2. 転置index

転置indexは、文書から語を探すのではなく、語から出現chunkへ逆引きする。posting listには文書ID、頻度、位置、fieldが格納され、希少語でも全件走査せず候補を得られる。この単純さが高速性と監査性の源である。ただしtokenizerやanalyzerが変われば、同じ原文でもpostingが変わる。更新・削除・版切替をindexへ反映し、query側とdocument側の解析規則を一致させる。検索結果の異常をモデルではなくindex構造から説明できるようにする。

**参照論文**
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)
- [P107] [Anserini Gets Dense Retrieval: Integration of Lucene's HNSW Indexes](https://arxiv.org/abs/2304.12139)

## 4.3.3. Tokenizationと日本語

日本語では空白が単語境界を示さないため、形態素解析と文字n-gramで検索結果が大きく変わる。「障害対応手順」を一語にするか三語にするか、英数字やハイフンをどう残すかが型番検索を左右する。Sparse domain adaptation研究は、語彙と頻度のずれが検索性能を落とすことを示す。user dictionary、同義語、全半角処理を文書種別ごとに評価し、誤字補正はコードfieldへ適用しない。analyzer変更は再indexと回帰試験を伴う仕様変更として扱う。

**参照論文**
- [P119] [Unsupervised Domain Adaptation for Sparse Retrieval by Filling Vocabulary and Word Frequency Gaps](https://arxiv.org/abs/2211.03988)
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)

## 4.3.4. TF-IDF

TF-IDFは、文書内で頻出する語を重くし、コーパス全体でありふれた語を軽くする。この二つの重みから、「その文書を特徴づける語」を数値化する直感を得られる。ただし頻度が増えるほど線形に重要になるわけではなく、文書長の影響も残る。そこで実務の主baselineはBM25へ進むが、IDFを理解しておけば、希少なエラーコードが効く理由や、全資料に現れる社名が効かない理由を説明できる。意味や語順を直接扱わない限界も明示する。

**参照論文**
- [P117] [Term-weighting approaches in automatic text retrieval](https://doi.org/10.1016/0306-4573(88)90021-0)
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)

## 4.3.5. BM25

BM25はTFをそのまま増幅せず、同じ語の反復効果を飽和させ、長い文書が偶然多くの語を含む偏りを補正する。`k1`は頻度飽和、`b`は長さ補正を調整するが、最適値はchunk長分布と質問に依存する。短いchunk中心のRAGで既定値を盲信せず、phrase matchやminimum-should-matchと併せて評価する。Robertsonらの整理を数式の紹介で終わらせず、各parameter変更がどのqueryを上げ下げしたかcase単位で確認する。

**参照論文**
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)

## 4.3.6. Fielded retrieval / BM25F

本文、見出し、FAQ質問、製品コードを一つの袋へ混ぜると、重要度の異なる一致が同じscoreになる。Fielded retrievalではfield別に統計とboostを持ち、たとえばtitle一致をbody一致より重くする。ただしtitleだけを過剰評価すれば、本文が無関係な文書が上位を占める。ID・versionは専用fieldへ分け、metadata filterとの違いを保つ。BM25Fの発想を用い、fieldごとの寄与をtraceし、文書種別ごとにboostを校正する。

**参照論文**
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)
- [P107] [Anserini Gets Dense Retrieval: Integration of Lucene's HNSW Indexes](https://arxiv.org/abs/2304.12139)

## 4.3.7. Exact matchと検索演算子

検索演算子は、曖昧さを増やす前に確実な一致を取りに行く道具である。引用符付きphrase、prefix、numeric range、boolean条件を、query typeに応じて組み合わせる。たとえばエラーコードはexact field、規程名はphrase、日付はrangeで扱い、結果がなければ段階的に緩和する。fuzzy matchを最初から有効にすると別型番を拾うため、識別子では禁止する。どの演算子でhitしたかを残せば、意味検索とは異なる根拠で順位を説明できる。

**参照論文**
- [P034] [The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)

## 4.3.8. 語彙辞書とDocument expansion

利用者が使う語と文書が使う語の隔たりは、query辞書か文書拡張で埋められる。DocT5Queryは文書から想定queryを生成し、その語を転置indexへ加えることで、検索時の語彙不一致を事前に緩和した。原文と生成語を別fieldに保存すれば、引用内容を汚さず検索だけを広げられる。ただし誤った想定queryが恒久的なノイズになるため、生成model・prompt・辞書versionをindex manifestへ記録し、BM25 baselineとの差を確認する。

**参照論文**
- [P109] [Document Expansion by Query Prediction](https://arxiv.org/abs/1904.08375)
- [P111] [Learning Passage Impacts for Inverted Indexes](https://arxiv.org/abs/2104.12016)

## 4.3.9. Learned sparse retrieval

SPLADEはTransformerが語彙ごとの重みを予測し、原文にない関連語も疎な表現として展開する。転置indexの高速性を保ちながらsemantic matchingを取り込む点が、denseとの違いである。DeepImpactはpassage内termの影響を学習し、SPLADE v2はpoolingやdistillationで精度・効率を改善した。ただしindexが膨らみ、言語・domainの語彙差にも影響される。BM25を消すのではなく追加retrieverとして比較し、sparsity、latency、zero-shot性能を測る。

**参照論文**
- [P033] [SPLADE: Sparse Lexical and Expansion Model for First Stage Ranking](https://arxiv.org/abs/2107.05720)
- [P111] [Learning Passage Impacts for Inverted Indexes](https://arxiv.org/abs/2104.12016)
- [P116] [SPLADE v2](https://arxiv.org/abs/2109.10086)
- [P119] [Unsupervised Domain Adaptation for Sparse Retrieval by Filling Vocabulary and Word Frequency Gaps](https://arxiv.org/abs/2211.03988)

## 4.3.10. Metadata filterとの組み合わせ

語彙一致が高くても、権限外・失効・別製品なら候補にしてはいけない。tenant、ACL、active statusはhard filterとしてBM25順位より先に適用し、製品や地域は明示度に応じてfilterまたはboostへ分ける。filter後の集合が小さいとIDFや候補数の挙動も変わるため、selectivity別にrecallを測る。field boostで権限を代替せず、0件時もACLを緩めない。検索深度だけを増やすfallbackにし、適用条件をtraceへ残す。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)
