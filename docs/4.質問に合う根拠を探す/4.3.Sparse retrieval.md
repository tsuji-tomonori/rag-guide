# 4.3 Sparse retrieval

## この節の役割

語彙一致を中心とする検索を扱う。RAGではdense retrievalだけに寄せず、型番、エラーコード、条文番号、API名、固有名詞を確実に拾うための基礎として使う。

Sparse retrievalは高速で説明しやすい一方、同義語や意味的な言い換えには弱い。BM25を明確なbaselineとして用意し、辞書、field設計、learned sparse retrievalを段階的に評価する。

## 4.3.1. Sparse retrievalの位置づけ
  - 単語・token一致に基づく検索であることを説明する。
  - dense retrievalとの優劣ではなく、異なる失敗モードを持つことを示す。
  - 型番、エラーコード、URL、条文番号、製品名で強い理由を説明する。
  - 最初に構築すべき検索baselineとして位置づける。

## 4.3.2. 転置index
  - termからdocument・chunkへ引く構造を説明する。
  - document frequency、posting list、fieldを整理する。
  - chunk更新、削除、版切替がindexへ反映される必要を示す。
  - index analyzerとquery analyzerの整合性を扱う。

## 4.3.3. Tokenizationと日本語
  - 形態素解析、文字n-gram、複合語分割の選択肢を説明する。
  - 全角半角、英数字、ハイフン、製品コードの分割を扱う。
  - synonym辞書とuser dictionaryの更新運用を含める。
  - tokenizer変更はindex再構築を伴うことを明記する。
  - query時の誤字補正が型番を壊さないようにする。

## 4.3.4. TF-IDF
  - TFとIDFの直感を説明する。
  - 希少語を重くする利点を示す。
  - 語順や意味を直接扱えない限界を示す。
  - 本番の主方式というより、BM25理解の前提として位置づける。

## 4.3.5. BM25
  - TF saturationとdocument length normalizationを説明する。
  - `k1`と`b`の意味を示す。
  - parameterを固定値として断定せず、業務datasetで評価する。
  - exact match、phrase match、minimum should matchとの組み合わせを示す。
  - chunk length分布がscoreへ与える影響を確認する。

## 4.3.6. Fielded retrieval / BM25F
  - title、heading、body、tag、FAQ questionを別fieldとして扱う。
  - field boostの例を示す。
  - metadata filterとfield boostを混同しない。
  - titleだけの過剰boostによる誤ランキングを注意点にする。
  - ID、code、versionは専用fieldへ分離する。

## 4.3.7. Exact matchと検索演算子
  - quoted phrase
  - prefix
  - fuzzy match
  - typo tolerance
  - code・ID専用field
  - date・numeric range
  - boolean condition
  - query typeに応じて厳密一致と曖昧一致を切り替える。

## 4.3.8. 語彙辞書とDocument expansion
  - 略語、旧名称、同義語を辞書で管理する。
  - DocT5Queryで文書側に想定queryを追加する考え方を説明する。
  - expansion textと原文を区別して保存する。
  - expansionが誤った語を追加するリスクを扱う。
  - 辞書versionとindex versionを対応づける。

## 4.3.9. Learned sparse retrieval
  - DeepImpact、SPLADE、SPLADE v2を比較する。
  - 転置indexの運用性を保ちながら意味的な語彙拡張を行う点を説明する。
  - modelとtokenizerの言語・ドメイン適合性を扱う。
  - index size、sparsity、推論コストを測る。
  - BM25を置換するか、hybridの追加retrieverにするかを評価する。

## 4.3.10. Metadata filterとの組み合わせ
  - ACL、tenant、active statusを検索前または検索時に適用する。
  - product、region、date、versionをquery条件として扱う。
  - field boostとhard filterを分離する。
  - filter適用で候補数が不足する場合のsearch depthを評価する。
