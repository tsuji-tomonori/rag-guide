# 4.1 Query理解と検索計画

## この節の役割

利用者の入力を検索器へ渡す前に、質問の意味、会話文脈、固有表現、期間、版、権限、検索先を整理し、どのretrieverを何件・何回使うかを決める。

ここでは、queryの意味をできるだけ変えずに理解・正規化する処理と、検索計画の作成を扱う。query rewriting、expansion、HyDE、Query2docなど、検索表現を積極的に変える処理は4.2で扱う。

## 4.1.1. Query理解と検索計画を分ける理由
- Query理解は「何を聞いているか」を確定する処理とする。
- 検索計画は「どこを、どの方式で、どの条件・予算で探すか」を決める処理とする。
- 変換後queryだけでなく、元queryと利用者が明示した制約を保持する。
- 生成モデルが推定した条件と、利用者が指定した条件を区別する。

## 4.1.2. 入力情報
- `original_query`
- 直近の会話履歴と会話要約
- `user_id`、`tenant_id`、group、role
- query時点とtimezone
- 利用可能なcorpus、index、tool、structured data source
- latency budget、cost budget、最大検索回数

## 4.1.3. 会話文脈の解決
- 「それ」「前のプラン」「この製品」の参照先を特定する。
- 会話履歴を丸ごとqueryへ連結せず、検索に必要な条件だけを補う。
- 参照したturnと補完内容をtraceへ残す。
- 複数候補がある場合は曖昧性を保持し、必要なら複数計画を作る。
- 過去の回答内容を正本の事実として扱わず、再検索可能にする。

## 4.1.4. 表記・語彙の正規化
- Unicode、全角半角、大文字小文字、空白、記号を正規化する。
- 製品略称、旧名称・新名称、社内略語、日英表記を辞書で対応づける。
- 型番、API名、条文番号、エラーコードは破壊的に分割・修正しない。
- 誤字補正は候補を保持し、元の文字列を捨てない。
- 日本語の分かち書きがsparse retrievalへ与える影響を明記する。

## 4.1.5. Query typeの分類
- exact identifier query
- natural-language semantic query
- 比較・一覧・要約query
- multi-hop query
- 時点指定・版指定query
- 会話依存query
- 数値集計・最新状態query
- 画像・表・graphを必要とするquery
- 検索不要の一般会話

## 4.1.6. Entity・期間・版・否定条件の抽出
- product、region、language、document type、versionを抽出する。
- `effective_from`、`effective_to`、基準時点を抽出する。
- 「除く」「以前」「以降」「最新版ではない」などの否定・比較条件を保持する。
- 抽出値ごとにconfidenceと根拠spanを付ける。
- query文字列に残す情報とmetadata filterに渡す情報を分ける。

## 4.1.7. ACLとmetadata条件
- `tenant_id`、user、group、roleから閲覧可能範囲を解決する。
- ACL、deleted、inactiveはhard filterとする。
- product、region、language、dateは確度に応じてhard filterまたはsoft constraintにする。
- 権限条件をLLMに推定させず、認証・認可情報から付与する。
- cache key、trace、evaluation logにもtenant・permission境界を反映する。
- 権限外chunkを取得後に隠す設計を避ける。

## 4.1.8. Retrieval routing
- exact identifierが中心ならsparseを優先する。
- 自然文の意味一致が中心ならdenseを優先する。
- 複合queryや一般的な文書QAではhybridを標準候補にする。
- 数値集計や最新状態はDB、SQL、APIなどのstructured routeへ送る。
- 関係探索や全体傾向はgraph・hierarchical routeを候補にする。
- queryの意味が十分に明確なら、不要なquery変換を行わない。
- 検索不要と判定した理由もtraceへ残す。

## 4.1.9. 検索計画
- 使用するretrieverと実行順序を決める。
- retrieverごとの`top_k`と全体candidate budgetを決める。
- metadata filterとACL filterを確定する。
- query変換を使うか、何件生成するかを決める。
- rerankへ渡す最大件数を決める。
- retry条件、別retrieverへのfallback、停止条件を決める。
- 最大step、timeout、token、costを明示する。

## 4.1.10. Active / adaptive retrieval
- 初回検索結果のscore、retriever間一致、候補数を確認する。
- 根拠不足時のみrewrite、search depth変更、別retrieverを試す。
- multi-hop queryではサブ質問ごとに検索する。
- 生成途中で追加根拠が必要な方式は発展として扱う。
- 無制限loopを避け、最大検索回数と停止理由を必須にする。
