# 4.5 Hybrid retrieval

## 4.5.1. Hybrid retrievalを使う理由

企業の質問には「ERR-1042」の厳密一致と「ログインできない原因」の意味一致が同居する。BM25とdenseの比較研究では、datasetやqueryの作られ方によって勝者が入れ替わり、どちらか一方を万能とみなせない。Hybrid retrievalは二方式を競わせるのではなく、異なる候補生成器として補完させる設計である。sparse、dense、hybridを同一gold setで比較し、hybridの増分がないsliceでは単純な経路を残す。複雑化そのものを品質向上と見なさない。

**参照論文**
- [P027] [BEIR: A Heterogeneous Benchmark for Zero-shot Evaluation of Information Retrieval Models](https://arxiv.org/abs/2104.08663)
- [P112] [Leveraging Semantic and Lexical Matching to Improve the Recall of Document Retrieval Systems](https://arxiv.org/abs/2010.01195)

## 4.5.2. 基本pipeline

Hybridの流れは、二つのscoreを足すところから始まらない。まず同じ認可・時点条件の下でsparseとdenseを独立実行し、chunk IDとversionで候補を同定する。次にhard filterを再確認し、重複をまとめ、rank fusionまたは校正済みscoreで統合してからrerankerへ渡す。Anseriniのdense統合は、Luceneの語彙検索とHNSWを同じ再現可能な基盤で扱う例を示した。各段階の入力順位を残し、候補消失の場所を追跡する。

**参照論文**
- [P107] [Anserini Gets Dense Retrieval: Integration of Lucene's HNSW Indexes](https://arxiv.org/abs/2304.12139)
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)

## 4.5.3. Hard filterの適用

権限と有効性はranking featureではなく、候補集合の境界である。tenant、allowed principal、active status、effective period、deleted/revokedをfusion前に強制し、どれほど高いBM25・cosine scoreでも越えさせない。ACORNのようなfiltered vector searchは、predicateを近傍探索中に扱うことで性能とfilterを両立しようとするが、認可policy自体を決めるものではない。権限外本文をpromptだけでなくtraceやcacheにも残さず、deny件数のみ監査する。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)

## 4.5.4. Soft constraintとboost

product、region、language、freshnessは、質問中で明示されたか推定されたかによって扱いを変える。明示条件はfilter、推定条件はboostや別検索にすれば、誤推定で正解を消しにくい。source trustもrelevanceとは別軸で、信頼度の低い文書が意味的に近いから採用される事態を防ぐ。soft constraintの重みはscore fusionへ混ぜる前に意味を定義し、版違いを同時に返す場合はcurrent、historical、exceptionの優先規則を設ける。

**参照論文**
- [P106] [An Analysis of Fusion Functions for Hybrid Retrieval](https://arxiv.org/abs/2210.11934)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 4.5.5. Candidate identityと重複

二つのretrieverが同じchunkを返しても、二件の裏付けにはならない。`document_id + version + chunk_id`を基本identityとし、parent-childやmirror sourceの関係も保持して統合する。一方、言語版や適用地域が違う文書をcontent similarityだけで同一視してはいけない。完全重複、near duplicate、同一familyの別版を区別し、同一documentからの上限を設ける。どの経路が候補を発見したかは消さず、後の診断とfusion寄与分析に残す。

**参照論文**
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P122] [Deduplicating Training Data Makes Language Models Better](https://arxiv.org/abs/2107.06499)

## 4.5.6. RRF

RRFは各ranked listでの順位だけを使い、`1/(k+rank)`を合計する。BM25 scoreとcosine similarityの尺度をそろえる必要がなく、一方で上位に現れた候補を安定して押し上げられる。原論文は複数rankerの単純なfusionが個別手法やCondorcet法を上回る場面を示した。ただし定数`k`と各retrieverのsearch depthで結果は変わる。query間fusionにも使えるが、候補を返さなかった経路の扱いと寄与内訳を明記する。

**参照論文**
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)

## 4.5.7. Score normalization / weighted fusion

scoreを使うfusionでは、BM25の非負で幅広い値と、cosineの狭い範囲を未校正のまま加算しない。min-max、z-score、rank normalization、convex combinationは分布への感度が異なる。Hybrid fusionの分析は、正規化と重みの選択がdatasetによって大きく変わることを示す。training labelがあれば重みを学習できるが、modelやindex更新後には再校正が必要である。raw score、変換後score、特徴別寄与を保存し、driftを監視する。

**参照論文**
- [P106] [An Analysis of Fusion Functions for Hybrid Retrieval](https://arxiv.org/abs/2210.11934)
- [P112] [Leveraging Semantic and Lexical Matching to Improve the Recall of Document Retrieval Systems](https://arxiv.org/abs/2010.01195)

## 4.5.8. Query-dependent fusion

質問の形が分かるなら、fusion比率も固定する必要はない。型番や条文はsparseへ比重を置き、説明的な自然文はdenseへ寄せ、混合queryでは両者に最低quotaを与える。BEIRが示したdomainごとの強弱は、query-dependentな選択の根拠になる。ただしrouterの誤判定が候補を失わせるため、確信が低い場合はscore学習より頑健なRRFへ戻す。選んだ経路、重み、分類根拠をtraceし、slice別にrouter自体を評価する。

**参照論文**
- [P027] [BEIR: A Heterogeneous Benchmark for Zero-shot Evaluation of Information Retrieval Models](https://arxiv.org/abs/2104.08663)
- [P106] [An Analysis of Fusion Functions for Hybrid Retrieval](https://arxiv.org/abs/2210.11934)

## 4.5.9. Multi-queryとの組み合わせ

三つの生成queryを二つのretrieverへ送れば、六本の順位表が生まれる。ここで全結果を一括fusionすると、似たqueryが同じ文書を何度も押し上げ、一つの誤生成queryが候補を支配し得る。まずquery内でsparse/denseを統合し、次にquery間でquota付きfusionを行うなど順序を決める。類似queryと重複candidateを早めに削り、`query数×retriever数×top-k`をbudgetとして監視する。各候補に発見queryを紐づけ、不要な枝を評価で落とす。

**参照論文**
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P031] [Query2doc: Query Expansion with Large Language Models](https://arxiv.org/abs/2303.07678)

## 4.5.10. Pre-filter / post-filter / filtered ANN

filter-then-searchは認可済み集合内で探すため安全だが、小さなsubsetではANN graphの接続性が弱くなる。search-then-filterは探索しやすい一方、権限外候補を取得しtop-k不足を招く。ACORNはpredicate-awareな探索でこの二律背反を緩和し、NaviXもselectivity変動へ頑健な設計を論じる。実装選択はcorpus規模、filter選択率、QPSで決め、post-filterならoversamplingと不足時retryを設ける。ただし本文露出前に必ず認可を完了する。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P129] [NaviX: A Native Vector Index Design for Graph DBMSs With Robust Predicate-Agnostic Search Performance](https://arxiv.org/abs/2506.23397)

## 4.5.11. Rerankとの境界

Fusionとrerankは役割が異なる。Fusionは複数の順位表を一つにまとめるが、candidate本文とqueryの細かな意味関係を新たに読むわけではない。Cross-encoder rerankerはqueryと各候補を同時に入力し、first-stageの順序を精密に見直す。したがって、まずsparse/denseでrecallを確保し、fusion後の20〜100件をrerankして5〜10件へ絞る。ACLはrerankerへ委ねず、timeout時はfusion順位へ戻しても根拠なし生成には進まない。

**参照論文**
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P040] [Passage Re-ranking with BERT](https://arxiv.org/abs/1901.04085)

## 4.5.12. Trace仕様

Hybrid検索を改善するには、最終順位だけでなく候補が歩いた経路を記録する。retrieverごとのquery、raw score、rank、適用filter、重複統合、fusion方式とparameter、query依存重み、選択結果、latency、fallbackを一つのtraceへ結ぶ。Anseriniの再現性重視の統合が示すように、同じindexと設定を再現できなければ比較は成立しない。candidate IDを全stageで固定し、「取得されなかった」「filterで落ちた」「fusionで下がった」を区別する。

**参照論文**
- [P107] [Anserini Gets Dense Retrieval: Integration of Lucene's HNSW Indexes](https://arxiv.org/abs/2304.12139)
- [P106] [An Analysis of Fusion Functions for Hybrid Retrieval](https://arxiv.org/abs/2210.11934)
