# 4.5 Hybrid retrieval

## この節の役割

sparse retrievalとdense retrievalを並列または段階的に実行し、候補を統合する設計を扱う。単に二つのscoreを足すのではなく、filter、score scale、重複、search depth、query種別、rerankとの境界を明確にする。

企業RAGでは、固有名詞・型番・条文番号と自然文の意味検索が同じsystemに混在するため、hybrid retrievalは有力な標準構成となる。ただし、hybrid化だけで必ず品質が上がるわけではなく、sparse単体・dense単体との比較が必要である。

## 4.5.1. Hybrid retrievalを使う理由
  - BM25はexact match、denseはsemantic matchに強い。
  - 一方だけでは業務queryの異なる種類を安定して扱えない。
  - hybridは異なるcandidate generatorを組み合わせる方法と説明する。
  - sparseのみ、denseのみ、hybridを同じdatasetで比較する。

## 4.5.2. 基本pipeline
  - 4.1で作成した検索計画とfilterを受け取る。
  - normalized queryまたは4.2の複数queryを各retrieverへ送る。
  - sparseとdenseから独立にtop-kを取得する。
  - document_id、chunk_id、versionを使って候補を同定する。
  - filter、重複除去、fusionを行う。
  - first-stage候補を5章のrerankerへ渡す。

## 4.5.3. Hard filterの適用
  - `tenant_id`
  - `allowed_user_ids` / `allowed_group_ids`
  - `status=active`
  - `effective_from <= query_time`
  - `effective_to > query_time`
  - deleted、revoked、archived
  - 権限と利用可否はfusion scoreより前に強制する。
  - 権限外候補をprompt、trace、cacheへ残さない。

## 4.5.4. Soft constraintとboost
  - product、region、language、source type、freshnessを扱う。
  - 明示指定はfilter、推定条件はboostまたは別検索として扱う。
  - source trustとrelevanceを別の特徴量として管理する。
  - boostでACLやtenant境界を代替しない。
  - 版違いの文書を同時に返す場合の優先順位を決める。

## 4.5.5. Candidate identityと重複
  - 同一chunkが両retrieverから返る場合のscoreを統合する。
  - parent-child chunkの同一性を扱う。
  - 同一documentから候補が偏りすぎないよう上限を設ける。
  - source、version、languageの違いを誤って同一視しない。
  - near-duplicateと完全重複を区別する。

## 4.5.6. RRF
  - scoreではなく順位を使う。
  - `score(d) = Σ_i 1 / (k + rank_i(d))`を示す。
  - score calibrationが不要な利点を説明する。
  - `k`とretrieverごとのsearch depthを評価する。
  - 候補に現れなかったretrieverの扱いを明示する。
  - query間fusionにも利用できる。

## 4.5.7. Score normalization / weighted fusion
  - min-max、z-score、rank normalizationを整理する。
  - BM25 scoreとcosine similarityを直接加算しない。
  - sparse重み、dense重み、title boost、freshnessを分離する。
  - training dataがある場合は重みを学習する。
  - score分布driftをmonitorする。
  - modelまたはindex変更時に再校正する。

## 4.5.8. Query-dependent fusion
  - 型番、エラーコード、条文番号ではsparseを高くする。
  - 自然文、要約的query、言い換えではdenseを高くする。
  - mixed queryではminimum quotaを両retrieverへ割り当てる。
  - classifierが不確実な場合はRRFへfallbackする。
  - route、重み、選択理由をtraceへ記録する。

## 4.5.9. Multi-queryとの組み合わせ
  - queryごとにsparseとdenseを使う場合の候補膨張を示す。
  - query内fusionとquery間fusionの順序を決める。
  - 類似queryと重複candidateを早期に削る。
  - query数×retriever数×top-kでcandidate budgetを管理する。
  - 一つの生成queryが全候補を支配しないよう上限を持つ。

## 4.5.10. Pre-filter / post-filter / filtered ANN
  - filter-then-searchは安全だが、候補集合が小さい場合のrecallを評価する。
  - search-then-filterはtop-k不足と権限外candidate取得の問題がある。
  - filtered ANNは探索中にpredicateを考慮する。
  - filter selectivity、corpus規模、QPS、latencyで方式を選ぶ。
  - post-filterを使う場合はoversamplingと候補不足時のretryを定義する。

## 4.5.11. Rerankとの境界
  - fusionは複数ranked listの統合である。
  - rerankはqueryとcandidateの関係を高精度modelで再評価する。
  - fusion後20〜100件、rerank後5〜10件などを評価で決める。
  - reranker失敗時はfusion結果へfallbackする。
  - ACLとhard filterをrerankerへ委ねない。

## 4.5.12. Trace仕様
  - retrieverごとのquery、rank、raw score
  - applied filters
  - candidate identity
  - duplicate merge結果
  - fusion methodとparameter
  - fusion score
  - query-dependent weights
  - selected candidates
  - latencyとcost
  - fallback・retry理由
