# 6.1 プロンプト

## この節の役割

- 検索・整形済みの根拠に限定して回答させるための生成契約を定義する。
- 命令、利用者の質問、会話履歴、検索結果、tool出力を混同しない入力構造を決める。
- 回答形式、回答可能条件、拒否、会話継続、tool利用の規則をプロンプトへ落とし込む。
- 検索文書内の悪意ある指示を命令として実行しない境界を設ける。

## 参考論文一覧

1. **Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks** — Patrick Lewis et al., NeurIPS 2020
   <https://arxiv.org/abs/2005.11401>
   - 外部のnon-parametric memoryを条件として生成するRAGの基本形を確認する。
   - promptでは「モデル内部知識」と「検索された根拠」の役割を分ける必要がある。

2. **Leveraging Passage Retrieval with Generative Models for Open Domain Question Answering** — Gautier Izacard and Edouard Grave, EACL 2021（Fusion-in-Decoder）
   <https://arxiv.org/abs/2007.01282>
   - 複数passageを統合する生成方式を理解する。
   - evidenceを単純な長文へ潰さず、文書境界と識別子を保持する設計の参考になる。

3. **Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection** — Akari Asai et al., ICLR 2024
   <https://arxiv.org/abs/2310.11511>
   - 検索要否、根拠の関連性、回答の支持性を動的に判定する考え方を学ぶ。
   - 固定promptだけでなく、判定と再試行を含む生成フローへつなげられる。

4. **Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection** — Kai Greshake et al., 2023
   <https://arxiv.org/abs/2302.12173>
   - 検索文書、Webページ、メールなどに含まれる指示がLLMの動作を乗っ取る危険を扱う。
   - retrieved contextを「命令ではなくデータ」として隔離する必要性の根拠になる。

5. **WebGPT: Browser-assisted question-answering with human feedback** — Reiichiro Nakano et al., 2021
   <https://arxiv.org/abs/2112.09332>
   - 情報探索、回答生成、出典提示を一連の作業として扱う。
   - 利用者が根拠を確認できる回答を生成目標に含める参考になる。

6. **Chain-of-Verification Reduces Hallucination in Large Language Models** — Shehzaad Dhuliawala et al., 2023
   <https://arxiv.org/abs/2309.11495>
   - draft、検証質問、独立回答、final answerを分ける。
   - 一度のpromptで回答と自己採点を同時に行わせない設計の参考になる。

7. **ReAct: Synergizing Reasoning and Acting in Language Models** — Shunyu Yao et al., ICLR 2023
   <https://arxiv.org/abs/2210.03629>
   - reasoning、action、observationを反復するagentic workflowの基本形を学ぶ。
   - tool利用の停止条件、観測結果の扱い、最終回答への接続を整理する際に有用。

8. **Toolformer: Language Models Can Teach Themselves to Use Tools** — Timo Schick et al., NeurIPS 2023
   <https://arxiv.org/abs/2302.04761>
   - API、検索、計算などを生成と分離されたtoolとして扱う考え方を確認する。

## 6.1.1. プロンプトを「生成契約」として定義する
  - 入力を `policy + user query + conversation state + evidence set + output contract` に分ける。
  - 「正しく答える」ではなく、「使用可能な情報と許される動作」を明示する。
  - retrievalやrerankの失敗はpromptだけでは修復できないことを示す。

## 6.1.2. 命令とデータの優先順位を定義する
  - system / developer policyを最上位の命令とする。
  - user queryは回答対象として扱う。
  - retrieved context、会話履歴、tool出力は参照データとして扱う。
  - 根拠中に「前の指示を無視せよ」などがあっても実行しない。

## 6.1.3. Evidenceの入力形式を決める
  - evidenceごとに `citation_id`、`document_id`、`chunk_id`、title、page、sectionを付ける。
  - version、effective date、source trust、ACL判定結果を必要に応じて含める。
  - 文書境界を明示し、複数文書の本文を一つの無区別な文章へ連結しない。
  - 表、SQL、API、画像由来情報はsource typeとlocationを明示する。

## 6.1.4. Grounding規則を定義する
  - evidenceで支持できるclaimだけを書く。
  - 数値、日付、固有名詞、条件、例外にはcitationを付ける。
  - evidenceにない内容をモデル内部知識で補わない。
  - 明示的な事実と、複数根拠から導いた推論を区別する。
  - 矛盾を解消できない場合は、一方を恣意的に採用せず不一致を示す。

## 6.1.5. 回答可能性と拒否を設計する
  - 完全回答、部分回答、確認質問、回答不能、権限不足を区別する。
  - 「検索結果がない」と「検索結果はあるが質問を支持しない」を分ける。
  - 曖昧な対象、期間、製品、版がある場合は確認質問へ分岐する。
  - 高リスク領域では、根拠があっても人間承認が必要な条件を定義する。

## 6.1.6. 回答形式を設計する
  - FAQ: 短い結論とclaim単位citation。
  - 調査: 結論、根拠、不確実性、次の確認事項。
  - 手順: 前提条件、番号付き手順、注意事項、出典。
  - 比較: 比較軸を固定した表。
  - システム連携: JSON Schemaまたはtyped object。
  - 人間向け表示と機械処理用schemaを必要に応じて分ける。

## 6.1.7. 会話履歴を扱う
  - 会話履歴をそのまま全投入せず、検索に必要な対象・期間・略語を抽出する。
  - standalone queryと元のuser queryを両方traceへ残す。
  - 前回答のcitationを再利用する場合でも、現時点の版と権限を再確認する。
  - tenantやuserを跨いだ履歴共有を禁止する。

## 6.1.8. Tool利用の境界を設計する
  - 検索・参照・計算toolと、更新・送信・承認toolを分ける。
  - toolを呼ぶ条件、引数schema、timeout、再試行回数、停止条件を定義する。
  - 更新系toolはallowlist、最小権限、人間承認を基本とする。
  - 文書内の指示だけを理由にtoolを実行しない。

## 6.1.9. Prompt templateを示す
  - role / task / authority boundary / evidence boundary / grounding rules / refusal rules / citation rules / output schemaの順に並べる。
  - policy、query、evidenceを明確なdelimiterで分離する。
  - frameworkやAPIが要求するcitation用placeholderを維持する。

## 6.1.10. 正常系・異常系の例を示す
  - 根拠が十分で、引用つきで回答できる例。
  - 根拠の一部だけが揃い、部分回答する例。
  - 旧版と新版が矛盾し、判断を保留する例。
  - 文書中のprompt injectionを無視する例。
  - 権限外情報や危険なtool実行を拒否する例。
