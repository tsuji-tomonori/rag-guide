# 6.2 根拠

根拠付き回答では、回答中の各主張を、利用可能な根拠の原文範囲へ対応付けます。
引用IDが表示されることだけでなく、原文が主張を支持し、根拠を必要とする主張が引用で覆われていることを検証します。

## 6.2.1 根拠集合

検索後処理の出力を、単なる入力用文字列ではなく、項目と型を定めた根拠集合として受け取ります。
各項目に本文、情報源、位置、版、日付、信頼度、アクセス制御リスト（Access Control List：ACL）の判定を持たせます。

支持、反対、背景、旧版、ツール結果など、根拠の役割を区別します。
一つの主張を複数の原文が共同で支える場合は、根拠集合として結び付けます。

[FEVER](https://aclanthology.org/N18-1074/)は、主張を支持・反証する根拠を扱い、[KILT](https://arxiv.org/abs/2009.02252)は情報源の出所を課題の評価へ組み込みました。
生成器へ渡す最小情報と、監査・画面表示に保持する情報を分け、同じ安定IDで結びます。

## 6.2.2 引用IDと情報源

引用IDは、回答中で根拠を短く参照するための鍵です。
根拠そのものではありません。
内部では、文書、チャンク、ページ、節、URL、原文範囲へたどれる必要があります。

表ではセルと行・列見出し、画像ではページ座標、外部システムとの連携窓口（API）では要求時刻と応答項目を表します。
表示用の脚注番号と、内部の安定IDを分けます。

文書が更新・移動された後も、回答時点に保存した複製（スナップショット）へ戻れる情報を保持します。
同じ引用IDが別の本文を指す状態を作りません。

## 6.2.3 引用する範囲の細かさ

段落末に一つの引用を置くと、段落中のどの文を支えるか曖昧になります。
数値、日付、固有名、条件、例外、比較など、検証可能な主張の直後に引用を置きます。

一つの主張を複数情報源が共同で支える場合は、引用の集合を示します。
同内容の別掲載（ミラー）を独立した裏付けとして数えません。

[Attributable to Identified Sources](https://arxiv.org/abs/2112.12870)は、生成内容を特定の情報源に結び付けられるかを評価する枠組みを扱いました。
情報源名だけでなく、主張を支持する原文範囲を示します。

一般的な接続表現や構造見出しに過剰に引用を付けると可読性が下がります。
主張の抽出と連動し、外部根拠を必要とする主張を定義します。

## 6.2.4 引用の正しさ

引用の正しさは、引用先が話題に関連するかではなく、対応する主張を実際に支持するかを表します。
原文の対象、値、単位、否定、時制、適用範囲を主張と比較します。

判定は、少なくとも次の三つに分けます。

- `SUPPORTS` は主張を支持します。
- `REFUTES` は主張と矛盾します。
- `NOT_ENOUGH_INFO` は関連しますが主張を確定できません。

文書が正しくても、引用範囲や版が違えば引用は不正確です。
文字列一致だけでなく、言い換え、数値、条件を検査します。
[ALCE](https://arxiv.org/abs/2305.14627)は、引用の正しさを独立した評価軸としました。

判定器を人が確認した正解集合で調整します。
高リスクな主張と判定器が不確かな主張は、人の確認へ送ります。

## 6.2.5 引用の網羅性

正しい引用が数件あっても、重要な主張の多くが無引用なら、回答全体を検証できません。
引用の網羅性は、外部根拠を必要とする主張が適切な引用で覆われた割合です。

回答を、それ以上分けると意味が変わる最小の主張へ分け、引用の有無と支持を確認します。
一つの引用を、支持しない複数の主張へ使い回しません。

引用漏れがある場合は、再検索して根拠を補うか、支持されない主張を削除・限定します。
既存の引用を近い話題という理由だけで付けません。
検証対象から除く一般論・見出しの規則も固定します。

## 6.2.6 主張の抽出

長い回答全体に一つの正誤ラベルを付けず、検証可能な最小の主張へ分けます。
数値、日付、固有名、因果、比較、条件、例外、手順を別の主張として扱います。

[FActScore](https://arxiv.org/abs/2305.14251)は、長文をこれ以上分けにくい小さな事実へ分解し、それぞれの知識源による支持を測りました。
RAGでも、主張IDと引用IDの対応を保存します。

見出し、接続表現、利用者への案内のうち、外部事実を含まないものは検証対象の主張から除外できます。
一文に複数の検証可能な主張がある場合は、分割するか、それぞれの支持を確認します。

主張単位で検証すると、誤った部分だけを修正し、支持済みの部分を残せます。

## 6.2.7 支持関係の分類

支持判定に、RAG固有の時間・権限状態を加えます。

- `SUPPORTS` は現在の質問条件で主張を支持します。
- `REFUTES` は主張と矛盾します。
- `NOT_ENOUGH_INFO` は十分な情報を持ちません。
- `OUTDATED` は内容が旧版にのみ支えられています。
- `SUPERSEDED` は正式な後継版に置き換えられています。
- `POLICY_BLOCKED` は権限・利用条件により回答へ使えません。

旧版だけが支持する主張は、事実内容が過去に正しくても、現在回答には使えない場合があります。
権限外の文書は、内部で支持関係を判定できても、回答や引用に出してはいけません。

判定理由と使用した原文範囲を保存します。
検索関連度スコアを支持判定に変換しません。

## 6.2.8 版、鮮度、矛盾

複数情報源の不一致を、更新日時だけで解決しません。
版のつながり、有効期間、所管部門、情報源の信頼度、適用対象を比較します。
質問時点に有効で、質問の対象へ適用される正式文書を選びます。

同じ公式性・優先度・時点・対象で矛盾が残る場合は、両方を示し、LLMの多数決を禁止します。
回答に不確実性、不一致の要点、確認先を反映します。

[FreshLLMs](https://arxiv.org/abs/2310.03214)は、変化の速い知識、誤った前提、検索結果の順序などを含む評価を扱いました。
新しい情報を取得できることと、その情報を正しく優先・利用できることを分けて確認します。

## 6.2.9 検証手段

一つの検証器ですべてを判断しません。
対象に応じて、次の手段を組み合わせます。

表6-2は、左列の検証対象に対して、まず使う手段を右列に示します。
上から処理順に並んだ表ではありません。
IDや数値は規則で確かめ、文章の支持関係は言語モデルで補助し、高リスク判断は権限を持つ人が確認します。

**表6-2　検証対象と主な検証手段**

| 検証対象 | 主な手段 |
|---|---|
| 引用ID、出力形式 | 同じ入力なら同じ結果になる規則 |
| 原文範囲の存在 | 文字列・位置照合 |
| 数値、日付、単位 | 構造化比較 |
| 支持・矛盾 | 自然言語推論（NLI）モデルまたはLLMを使う判定器 |
| 複数根拠の統合 | LLMを使う判定器と規則 |
| 計算・現在値 | データベース問い合わせ（SQL）、外部システムとの連携（API）、検証済みコード |
| 高リスク判断 | 権限を持つ人 |

[Chain-of-Verification](https://arxiv.org/abs/2309.11495)は、下書きと検証質問、独立回答、最終修正を分けました。
検証モデルも誤るため、同じ正解集合で、正しいと判定した中で実際に正しい割合（適合率）と、正しい対象を拾えた割合（再現率）を測ります。
互いの判定を無条件に正しいと見なしません。

## 6.2.10 検証後の分岐

検証結果をスコアとして記録するだけでなく、次の処理に結び付けます。

表6-3は、左列の検証結果から右列の次処理を選ぶ対応表です。
軽微な修正でも新しい主張が生まれる可能性があるため、修正後は一行目へ直接進まず、もう一度検証します。

**表6-3　検証結果と次の処理**

| 検証結果 | 処理 |
|---|---|
| すべての主張が支持されています | 回答を確定します |
| 一部の主張だけが支持されています | 支持部分だけの回答へ限定します |
| 軽微な値・表現誤りがあります | 原文へ合わせて修正し、再検証します |
| 根拠が不足しています | 再検索または確認質問を行います |
| 重大な矛盾があります | 回答を保留し、人へ確認します |
| 方針違反があります | 出力せず、必要な監査処理を行います |

[RARR](https://arxiv.org/abs/2210.08726)は、生成済み文章を調査し、根拠を付けながら改訂する処理を示しました。
修正によって新しい主張が生まれる可能性があるため、修正版を再検証します。
最大修正回数と停止条件を定め、元回答と修正版の履歴を保存します。

## 6.2.11 表示と監査

利用者画面と内部監査では、必要な情報量が異なります。
画面には、読みやすい脚注、文書名、ページ、短い原文抜粋を示します。
引用を選ぶと、権限内で原文の前後を確認できるようにします。

内部の処理記録には、質問、検索と再順位付けの経路、主張、引用、支持判定、修正履歴、各構成版を保存します。
再現に必要な情報を保持しつつ、権限外本文、不要な個人情報、秘密を監査目的で丸ごと保存しません。

表示用データと再現用成果物を安定IDで結びます。
情報源の削除・保持期限が来た場合に、過去回答と回答時点に保存した原文をどう扱うかを、法令・契約・監査要件に従って定めます。
