# 7.4 Retrieval評価

## 7.4.1. Retrieval評価の入出力を固定する

Retrieval比較の入力を固定しなければ、query rewriteやcorpus更新の差がretriever差へ混ざる。original query、会話文脈、rewrite、filter、基準時点、corpus snapshotを保存し、candidateごとにdocument・chunk ID、rank、raw score、retrieverを記録する。BEIRが統一corpus・query・qrelsで手法を比較したように、実験境界を明確にする。cache hitやfallbackも含め、同じ入力を再実行して順位差を説明できる形にする。

**参照論文**
- [P027] [BEIR: A Heterogeneous Benchmark for Zero-shot Evaluation of Information Retrieval Models](https://arxiv.org/abs/2104.08663)
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)

## 7.4.2. Gold relevanceを定義する

Gold relevanceは「回答文字列を含むchunk」と同義ではない。DPRのTop-k評価は大規模比較に有用だが、偶然同じ文字列を含む文書も正解になり得る。そこでdocument、chunk、sentence、claimのどこをgoldにするかを定め、primary、alternative、supporting evidenceを区別する。binary relevanceは回収確認に、graded relevanceはnDCGに使う。複数annotatorでscopeと時点を確認し、chunking変更後も元source spanからgoldを再対応づける。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P051] [Cumulated Gain-based Evaluation of IR Techniques](https://doi.org/10.1145/582415.582418)

## 7.4.3. Candidate recallを測る

Candidate recallは、生成器が正解へ到達できる上限を測る。recall@5、10、20、50を並べ、候補数を増やしたときの限界効用とlatencyを同時に見る。DPRはTop-20とTop-100でBM25との差を示し、retriever改善がQA精度へつながることを実証した。multi-hopでは一つのgoldが入るhit rateだけでは足りず、必要な全evidenceが揃うall-evidence recallを使う。権限で利用不能な文書はgold集合から適切に除く。

**参照論文**
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P110] [Interleaving Retrieval with Chain-of-Thought Reasoning for Knowledge-Intensive Multi-Step Questions](https://arxiv.org/abs/2212.10509)

## 7.4.4. Ranking qualityを測る

正解が候補に入っていても、50位では後段へ届かない。MRRは最初の正解、nDCGはgraded relevanceを順位割引付きで、precision@kは上位の密度を測る。nDCGの原典が示す累積利得の考え方を使い、主要根拠と補助根拠に異なるgainを与える。複数正解、同点、隣接重複chunkの扱いをevaluation codeで固定し、metric実装差で結果が変わらないようtest fixtureを用意する。

**参照論文**
- [P051] [Cumulated Gain-based Evaluation of IR Techniques](https://doi.org/10.1145/582415.582418)
- [P038] [The Use of MMR, Diversity-Based Reranking for Reordering Documents and Producing Summaries](https://dl.acm.org/doi/10.1145/290941.291025)

## 7.4.5. Query slice別に比較する

Retrieverの弱点はqueryの種類ごとに現れる。型番、条文、固有名詞、自然文、略語、誤字、multi-hop、日本語、英語、cross-language、head・long-tailへsliceし、平均の裏を読む。BEIRではBM25とdenseの強弱がdatasetごとに変わり、zero-shotの一般化が単一benchmarkでは判断できない。各sliceのsample数とconfidence intervalを示し、重要だが少数のqueryを全体平均で消さない。

**参照論文**
- [P027] [BEIR: A Heterogeneous Benchmark for Zero-shot Evaluation of Information Retrieval Models](https://arxiv.org/abs/2104.08663)
- [P020] [MTEB: Massive Text Embedding Benchmark](https://arxiv.org/abs/2210.07316)

## 7.4.6. Retrieverを比較する

BM25、dense、learned sparse、hybridを同じgold setと候補budgetで比較する。SPLADEは語彙展開を疎indexへ取り込み、DPRは意味表現を使い、RRFは順位表を統合するため、改善理由が異なる。score fusionとrank fusionも分け、zero-shotとdomain tuningを別条件にする。winner一つを選ぶだけでなく、どのsliceで補完関係があるかを見てroutingやhybridを設計する。index size、build時間、QPSも同じ表へ加える。

**参照論文**
- [P033] [SPLADE: Sparse Lexical and Expansion Model for First Stage Ranking](https://arxiv.org/abs/2107.05720)
- [P001] [Dense Passage Retrieval for Open-Domain Question Answering](https://arxiv.org/abs/2004.04906)
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P027] [BEIR: A Heterogeneous Benchmark for Zero-shot Evaluation of Information Retrieval Models](https://arxiv.org/abs/2104.08663)

## 7.4.7. Query処理を評価する

Query処理の評価では、retrieverを固定してnormalization、rewrite、expansion、HyDE、decompositionを一つずつon/offする。HyDEがzero-shot denseを改善しても、型番へ架空語を足すcaseでは悪化し得る。会話ではstandalone queryが参照対象を正しく補ったかを人手labelで確認する。検索scoreだけでなく、元意図の保持、追加・削除した条件、candidate overlapを測り、rewrite failureをretriever failureへ数えない。

**参照論文**
- [P030] [Precise Zero-Shot Dense Retrieval without Relevance Labels](https://arxiv.org/abs/2212.10496)
- [P115] [Query Rewriting for Retrieval-Augmented Large Language Models](https://arxiv.org/abs/2305.14283)
- [P062] [CONQRR: Conversational Query Rewriting for Retrieval with Reinforcement Learning](https://arxiv.org/abs/2112.08558)

## 7.4.8. Metadata／ACL filterを評価する

Metadata／ACL filterは、漏えい防止とrecall維持の二面で評価する。tenant、role、date、status、version、source typeを個別・組合せで試し、権限外混入率と過剰filterによるgold欠落率を出す。ACORNがfilter selectivityとANN性能の関係を扱うように、条件の厳しさ別に候補数とlatencyも測る。ACL違反はzero tolerance、製品・言語などsoft条件の誤除外は品質指標として分離する。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)

## 7.4.9. 性能とcostを評価する

検索品質の比較には、retrieval stageのp50、p95、p99、timeout、cost/queryを添える。top-k、multi-query数、fusion、ANN parameterを増やせばrecallは上がり得るが、tail latencyと後段costも増える。HNSWの`efSearch` sweepやfilter selectivity別load testを行い、quality–latency曲線を描く。平均速度だけで高価なqueryを隠さず、tenant・query type別に計測する。SLO超過時のfallbackが品質をどう変えるかも含める。

**参照論文**
- [P017] [Efficient and robust approximate nearest neighbor search using HNSW graphs](https://arxiv.org/abs/1603.09320)
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)

## 7.4.10. 統計的に比較する

同じcaseに対するbaselineとcandidateの差をpairedに扱う。平均nDCG差だけでなく、改善件数、悪化件数、重大caseの変化、bootstrap confidence intervalを示す。少数の極端なqueryが平均を動かしていないか分布を見る。複数parameterを探索した場合はtest setへの選択biasを避け、devで選びholdoutで確認する。重要sliceには非劣化の下限を置き、統計的有意でも業務上小さい差を過大評価しない。

**参照論文**
- [P050] [ARES: An Automated Evaluation Framework for Retrieval-Augmented Generation Systems](https://arxiv.org/abs/2311.09476)
- [P140] [Practical Guide to Controlled Experiments on the Web](https://doi.org/10.1145/1281192.1281295)

## 7.4.11. Failureを分類する

Retrieval missを一つの箱へ入れず、corpus absence、stale index、parse・chunk欠損、embedding mismatch、rewrite逸脱、filter過剰、rank不足へ分類する。正解文書がcorpusにないcaseをmodel tuningへ回しても直らない。traceから候補生成前後を辿り、最初にgoldが消えたstageをfailure reasonとする。RAGCheckerの分解思想を使い、generationが誤ったcaseでも、根拠が候補に入っていたかを先に確認する。分類ごとにownerと修正手段を割り当てる。

**参照論文**
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
- [P087] [Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)
