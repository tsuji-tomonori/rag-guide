# 7.1 評価指標

評価指標は、システムの状態を数値や判定結果として表す物差しです。
一つの総合点だけでは失敗箇所を特定できないため、RAGでは工程、評価単位、利用場面を分けて指標を設計します。

## 7.1.1 評価対象と評価単位

最初に、何を一件として数えるかを決めます。
文書を取得できたか、正解箇所を含むチャンクを取得できたか、回答中の主張を根拠が支えているか、利用者が業務を完了できたかは、それぞれ別の評価です。

表7-1は、評価する一件の大きさを行に、確認対象と業務例を列に示します。
上の行ほど個別の資料に近く、下の行ほど利用者の一連の行動に近い単位です。

**表7-1　RAGで使う主な評価単位**

| 評価単位 | 確認する対象 | 例 |
|---|---|---|
| 文書 | 情報源となるファイルやページ | 正しい版の規程を取得したか |
| チャンク | 検索用に分割した文章 | 回答に必要な段落を含むか |
| スパン | 文書内の連続した文字範囲 | 対象となる条件文を保持したか |
| 主張 | 回答中の検証可能な記述 | 金額の上限を根拠が支えるか |
| 回答 | 一回の質問に対する出力 | 質問へ正しく答えたか |
| セッション | 一連の対話 | 再質問を含めて問題を解決したか |
| タスク | 利用者が達成したい業務 | 申請可否を判断できたか |

[KILT](https://arxiv.org/abs/2009.02252)は、回答の正確さと情報源を示す来歴を組み合わせて知識集約タスクを評価します。
この考え方をRAGへ適用すると、答えが偶然合っていた場合と、正しい根拠に基づいて答えた場合を区別できます。

正解データには、評価単位、文書の版、位置、許容する別解を記録します。
文書更新でチャンクIDが変わっても追跡できるように、元文書のIDと位置も残します。

## 7.1.2 検索前の指標

検索前評価は、質問を受け取る前に知識基盤が利用可能な状態かを測ります。
上流で欠けた情報は、検索や生成の調整では取り戻せません。

少なくとも、次の指標を監視します。

- 取り込み対象に対する成功件数の割合
- 原文と抽出結果の文字、表、見出し、読み順の一致
- メタデータとアクセス制御情報の欠損件数
- 埋め込み生成とインデックス登録の失敗件数
- 情報源の更新から検索可能になるまでの遅延
- 更新と削除が反映された文書の割合

[Data Validation for Machine Learning](https://research.google/pubs/data-validation-for-machine-learning/)が機械学習パイプラインの入口でスキーマや分布の異常を検出するように、RAGでも公開前に知識成果物を検査します。
アクセス制御リストの欠落のような安全性に関わる不具合は、平均値ではなく許容件数ゼロの条件として扱います。

## 7.1.3 検索の指標

検索では、「必要な根拠を候補へ入れられたか」と「候補の上位へ置けたか」を分けて測ります。
これまで「検索再現率」「必要根拠の回収率」と表現した値は、ここで定義するRecall@k系の指標です。
正解を含む文書やチャンクを関連ありとみなし、質問ごとに順位を評価します。

表7-2は、行ごとに検索指標を、列ごとに測る内容と用途を示します。
`k`は、検索結果の上位何件までを見るかを表します。

**表7-2　検索結果の主な評価指標**

| 指標 | 測る内容 | 主な用途 |
|---|---|---|
| Recall@k | 正解集合のうち上位k件で回収した割合 | 根拠の取りこぼし確認 |
| Precision@k | 上位k件のうち正解である割合 | 候補のノイズ確認 |
| MRR | 最初の正解順位の逆数の平均 | 最初の有用な根拠までの近さ |
| nDCG@k | 関連度と順位を反映した利得 | 複数段階の関連度がある検索 |

Recall@kは、質問`q`の正解集合を`Gq`、上位k件を`Rk(q)`とすると、`|Gq ∩ Rk(q)| / |Gq|`で求めます。
[DPR](https://arxiv.org/abs/2004.04906)も、読解器へ渡す前の検索性能を、上位k件で正解を取得できた割合で測っています。

複数の根拠を組み合わせる質問では、一件でも当たれば成功とする指標だけでは不十分です。
必要な根拠をすべて回収した質問の割合である全根拠回収率を加えます。
権限のない文書が候補に入っていないことは、関連度とは別のアクセス制御指標で確認します。

## 7.1.4 検索後処理の指標

検索後処理は候補の並べ替え、重複除去、圧縮、配置を行うため、ノイズを減らす一方で正解も失う可能性があります。
各処理の前後で同じ質問を追跡し、どの段階で根拠が消えたかを確認します。

主な指標には、再順位付け前後のMRRとnDCG、正解根拠保持率、重複率、文書多様性、圧縮率、正解スパン保持率があります。
最終コンテキスト内の位置も記録すると、根拠を保持したままモデルが見落とした場合を区別できます。

[RECOMP](https://arxiv.org/abs/2310.04408)は、検索文書の圧縮をトークン削減量だけでなく下流タスクの性能とともに評価しています。
同様に、短いコンテキストを作れたことだけを成功とはみなさず、回答に必要な情報が残ったことを条件にします。

## 7.1.5 生成の指標

生成評価では、回答が正しいことと、提示された根拠に忠実であることを別々に測ります。
外部知識として正しい記述でも、与えた根拠から確認できなければ、根拠性の評価では支持されていない主張です。

主な評価軸は次のとおりです。

- 正確性は、質問に対する結論や値が正しいかを測ります。
- 根拠忠実性は、回答中の検証可能な主張をコンテキストが支えるかを測ります。
- 引用精度は、付けた引用が直前の主張を実際に支える割合を測ります。
- 引用再現率は、根拠を必要とする主張のうち引用が付いた割合を測ります。
- 完全性は、質問に必要な論点を欠かしていないかを測ります。
- 関連性は、不要な説明を増やさず質問へ答えたかを測ります。
- 形式準拠率は、指定したJSONや見出し構造を守った割合を測ります。
- 支持なし主張率は、根拠で検証できない主張が占める割合を測ります。

[ALCE](https://arxiv.org/abs/2305.14627)は長文回答について引用の正しさと網羅性を評価し、[FActScore](https://arxiv.org/abs/2305.14251)は長文を検証可能な原子的事実へ分解します。
長い回答を一括で採点せず、主張単位へ分解すると、どの文に根拠不足があるかを診断できます。

根拠がない質問への拒否、曖昧な質問への確認、矛盾した資料の明示も評価対象に含めます。
回答率を上げることが、常に品質向上を意味するわけではありません。

## 7.1.6 端から端までの指標

端から端までの評価は、RAG全体が利用者の目的を達成したかを測ります。
検索指標が高くても、回答が分かりにくい、処理が遅い、費用が高いといった理由で業務に使えない場合があります。

品質については、タスク完了率、部分完了率、適切な担当者への引き継ぎ率、再質問率を記録します。
システム特性については、最初のトークンまでの時間、95パーセンタイル遅延、タイムアウト率、一質問当たりの費用を併記します。
95パーセンタイル遅延は、処理時間が短い順に並べたときに95%の要求が収まる値です。

[RAG評価のサーベイ](https://arxiv.org/abs/2405.07437)も、検索、生成、RAGシステムの評価を区別して整理しています。
情報源を開いた回数や回答のコピー回数は利用状況の補助信号にはなりますが、正しさの直接的な証拠としては扱いません。

## 7.1.7 条件別の指標

全体平均だけを見ると、件数が少なくても業務上重大な失敗が隠れます。
データを意味のある条件で分けた集合をスライスと呼び、全体値と同じ指標をスライスごとに確認します。

たとえば、質問形式、複数根拠の要否、OCR文書、言語、テナント、役割、回答可能性、事故時の重大度で分けます。
型番のような短い識別子と自然文でも、検索方式の得意不得意は変わります。
[BEIR](https://arxiv.org/abs/2104.08663)が異なる検索領域を横断して評価した結果からも、あるデータセットで優れた検索器が別の領域でも同じように優れるとは限らないことが分かります。

運用上重要なスライスは評価前に登録し、最低基準を設定します。
結果を見た後だけ都合のよい切り分けを増やすと、偶然の差を改善と誤認しやすくなります。

## 7.1.8 指標の集約

複数の結果を集約するときは、重み付けの意味を明示します。
マクロ平均は各スライスの値を同じ重みで平均し、マイクロ平均は各評価例を同じ重みで集計するため、件数の多いスライスの影響が大きくなります。
業務影響を表す重大度の重みを使う場合は、重みの根拠と変更履歴を残します。

回答可能な質問が多いデータセットでは、回答率の平均だけでは不適切な回答拒否を捉えられません。
回答可能と回答不能を分けて結果を示します。

総合点をリリース判断に使う場合も、元の指標とスライス別結果を保持します。
アクセス制御違反や危険なツール実行は平均へ混ぜず、一件でも不合格となる独立した安全条件にします。

## 7.1.9 不確実性

評価値は、選んだ質問や評価者によって変動します。
結果には評価件数、信頼区間、評価者間一致率を添え、小さな点差を直ちに改善とは判断しません。

[ARES](https://arxiv.org/abs/2311.09476)は、少量の人手ラベルと自動評価器を組み合わせ、RAG評価値の信頼区間を推定します。
変更前後で同じ質問を使う場合は、各質問の差を対応させたブートストラップ法などを使うと、質問の難易度差によるばらつきを抑えられます。

平均差に加えて、改善した件数、変わらない件数、悪化した件数を示します。
重要スライスの件数が少なく信頼区間が広い場合は、結論を保留して追加データを集めます。

## 7.1.10 合格条件

合格条件は、変更後に全体平均が上がったかだけでは決めません。
改善対象、非劣化条件、重要スライスの下限、安全条件、遅延上限、費用上限を別々に定め、すべてを満たすことを要求します。

表7-3は、検索方式を変更するときの判定例です。
行ごとに独立した条件を置き、改善目標だけでなく、品質、安全性、性能、費用の全条件を満たしたかを読みます。

**表7-3　検索方式を変更するときの合格条件の例**

| 条件 | 判定例 |
|---|---|
| 改善目標 | 型番スライスのRecall@10が基準値以上 |
| 非劣化 | 全体の正確性が許容幅を超えて低下しない |
| 最低品質 | 重大度が高い各スライスが下限以上 |
| 安全性 | 権限外文書の取得が0件 |
| 性能 | 95パーセンタイル遅延が上限以内 |
| 費用 | 一質問当たりの費用が予算以内 |

[ML Test Score](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)が本番準備度を複数領域へ分けるのと同様に、RAGのリリースも複数条件の論理積で判断します。
重大な失敗を多数の容易な質問で相殺してはいけません。
例外を承認する場合は、責任者、期限、利用範囲、ロールバック条件を記録します。
