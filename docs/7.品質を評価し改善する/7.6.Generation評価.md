# 7.6 Generation評価

## 7.6.1. Generation条件を固定する

Generationだけを比較するには、queryとfinal contextを固定し、model・prompt以外を動かさない。temperature、max tokens、output schema、model revisionを記録し、同じevidence setでbaselineとcandidateを生成する。RAGASやALCEの指標を使っても、取得根拠が違えば生成能力の差とは言えない。非決定性を扱うため複数runを行う場合はseed・sampling条件をそろえ、平均だけでなく分散と重大failureの再現率を報告する。

**参照論文**
- [P055] [RAGAS: Automated Evaluation of Retrieval Augmented Generation](https://arxiv.org/abs/2309.15217)
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)

## 7.6.2. Answer correctnessを評価する

Answer correctnessはexact matchだけでは測れない。必須のkey fact、数値、日付、条件、手順をrubric化し、許容表現・数値誤差・禁止claimを定める。reference answerが唯一解でない調査質問では、claim coverageと重大誤りを人手で判定する。FActScoreのatomic fact評価を応用し、長い回答の一部正解を全体正解へ丸めない。gold自体も文書更新で古くなるため、source versionと一緒に定期reviewする。

**参照論文**
- [P045] [FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation](https://arxiv.org/abs/2305.14251)
- [P136] [HAGRID: A Human-LLM Collaborative Dataset for Generative Information-Seeking with Attribution](https://arxiv.org/abs/2307.16883)

## 7.6.3. Claimへ分解する

回答をclaimへ分解すると、どの部分が失敗したかが見える。数値、日付、固有名詞、因果、比較、条件、例外、手順を個別のatomic claimにし、それぞれをsupported、contradicted、NEIへ分類する。FActScoreは長文伝記を原子的事実へ分け、細粒度のprecisionを測った。RAGではclaim severityも持たせ、中心結論の誤りと補助説明の誤りを区別する。分解器の見落としは人手sampleで校正する。

**参照論文**
- [P045] [FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation](https://arxiv.org/abs/2305.14251)
- [P057] [RefChecker: Reference-based Fine-grained Hallucination Checker and Benchmark for Large Language Models](https://arxiv.org/abs/2405.14486)

## 7.6.4. Faithfulnessを評価する

Faithfulnessは回答が世の中で正しいかではなく、与えたcontextに支えられているかを問う。contextにない補足、根拠の過剰一般化、条件・否定の取り落としをclaim単位で検出する。RAGASはfaithfulnessをRAG評価の中心に置き、RAGTruthはunsupported・contradictory hallucinationを具体化した。一般常識として正しい文もgroundedness上は別扱いにし、重大unsupported claim件数と全claim率を分けて報告する。

**参照論文**
- [P055] [RAGAS: Automated Evaluation of Retrieval Augmented Generation](https://arxiv.org/abs/2309.15217)
- [P131] [RAGTruth: A Hallucination Corpus for Developing Trustworthy Retrieval-Augmented Language Models](https://arxiv.org/abs/2401.00396)

## 7.6.5. Context utilizationを評価する

正解根拠がcontextに存在するのに回答が使わない場合、問題はretrievalではなくgenerationにある。claimとgold evidenceの対応を作り、必要spanの利用、見落とし、不要evidenceへの依存を分ける。RAGCheckerはcontext utilizationとhallucinationを細粒度に測り、retrieverとgeneratorの責任を切り分ける。根拠位置、context長、noise量を変えるablationでreaderの弱点を確認し、prompt変更とmodel変更を別々に比較する。

**参照論文**
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)

## 7.6.6. Citation precisionを評価する

Citation precisionでは、付いた脚注のうち何件が直前claimを本当に支えるかを見る。正しいdocumentでもspanが違う、旧版、関連するだけ、反証するcaseを別labelにする。quoteを表示する場合は原文一致をruleで確認し、言い換えはNLI・人手で判定する。ALCEとAISが示すように、出典の存在とattributionは同じではない。重大claimの誤引用を件数で示し、平均precisionが高くても看過しない。

**参照論文**
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)
- [P128] [Measuring Attribution in Natural Language Generation Models](https://arxiv.org/abs/2112.12870)

## 7.6.7. Citation recall／completenessを評価する

Citation recall／completenessは、根拠を必要とするclaimのうち何件に適切な引用があるかを測る。複数文書の統合claimでは一件のcitationだけで十分とは限らず、必要なevidence setが揃うかを見る。ALCEはcorrectnessとcompletenessを分離し、長文回答の引用漏れを評価した。一方、一般的な接続文へ大量の脚注を付ける行為は改善ではない。claim extractionと連動し、過剰引用も補助指標として記録する。

**参照論文**
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)

## 7.6.8. Answer completeness／relevanceを評価する

Completenessは質問の必須論点を満たすか、relevanceは余計な話へ逸れないかを測る。前提、例外、比較軸、手順の欠落をrubric化し、冗長さとは別項目にする。短い回答が重要条件を落としていても高評価にせず、長い回答が全論点を含んでも無関係な説明で埋まればrelevanceを下げる。G-Eval型のrubricを使う場合はanchor exampleを用意し、人手評価との一致を確認する。 必須論点と不要部分を別々に数えることで、単なる長文化を完全性の改善と誤認しない。

**参照論文**
- [P053] [G-Eval: NLG Evaluation using GPT-4 with Better Human Alignment](https://arxiv.org/abs/2303.16634)
- [P136] [HAGRID: A Human-LLM Collaborative Dataset for Generative Information-Seeking with Attribution](https://arxiv.org/abs/2307.16883)

## 7.6.9. 矛盾と鮮度を評価する

矛盾・鮮度caseでは、単一の正答文字列より行動を評価する。新旧版があるとき有効期間へ従うか、反対証拠が同等なら断定せず併記するか、source priorityを説明できるかを見る。FreshLLMsやStreamingQAの時間変化queryを参考に、現在・過去・未来施行のscenarioを作る。最新日付を機械的に選ぶ回答、古い根拠だけのclaim、矛盾を隠した要約をfailureとして分ける。 この評価では、正しい保留も成功として数え、無理な一意回答だけを高得点にしない。

**参照論文**
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)
- [P135] [StreamingQA: A Benchmark for Adaptation to New Knowledge over Time in Question Answering Models](https://arxiv.org/abs/2205.11388)

## 7.6.10. Answerability／Refusalを評価する

Answerability評価では、answerable、unanswerable、ambiguous、permission-denied、out-of-scopeを混ぜない。SQuAD 2.0は答えのない質問を加え、modelが回答可能性を判定する課題を明確化した。RAGではcorrect refusal、false refusal、unsafe answer、適切なclarificationを区別する。権限外caseでは文書の存在自体を示唆しないことも検査し、回答率を上げるための過剰断定と、安全側に倒れすぎる過剰拒否の両方を測る。

**参照論文**
- [P139] [Know What You Don't Know: Unanswerable Questions for SQuAD](https://arxiv.org/abs/1806.03822)
- [P076] [Teaching language models to support answers with verified quotes](https://arxiv.org/abs/2203.11147)

## 7.6.11. Noise／Counterfactual robustnessを評価する

Noise robustnessは無関係文書が混ざっても正解を使えるか、counterfactual robustnessはmodel内部知識とretrieved evidenceが食い違うとき指定根拠を扱えるかを見る。RGB benchmarkはnoise、negative rejection、information integration、counterfactualを四能力として分けた。noise件数と位置を段階化し、誤った高score文書やもっともらしい反証も加える。単にmodelが内部知識へ固執した場合と、悪い根拠へ盲従した場合を別failureにする。

**参照論文**
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)

## 7.6.12. Format adherenceを評価する

内容が正しくてもJSONをparseできなければ業務連携では失敗である。JSON Schema、Markdown表、必須field、citation ID、列数、enumをdeterministicに検査し、parse failureと内容品質を分ける。再試行で形式だけ直せる場合と、再生成で内容が変わる場合も記録する。tool引数では型、範囲、禁止fieldを厳格にし、自然言語judgeへ任せない。format adherenceを高めるprompt変更がfaithfulnessを損なわないかも確認する。

**参照論文**
- [P102] [The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction](https://research.google/pubs/the-ml-test-score-a-rubric-for-ml-production-readiness-and-technical-debt-reduction/)
- [P015] [Toolformer: Language Models Can Teach Themselves to Use Tools](https://arxiv.org/abs/2302.04761)

## 7.6.13. 評価器を校正する

評価器はrule、NLI、claim checker、LLM judge、人手を適材適所で組み合わせる。RefCheckerはclaimより細かなtriplet単位でsupportを確認し、LLM judge研究はposition・verbosity biasを明らかにした。人手gold setでprecision、recall、混同行列を測り、言語、domain、回答長ごとの誤差を見る。judge modelやpromptを変えたら過去scoreと直接比較せず再校正し、評価器自身のversionとfailure sampleを管理する。

**参照論文**
- [P057] [RefChecker: Reference-based Fine-grained Hallucination Checker and Benchmark for Large Language Models](https://arxiv.org/abs/2405.14486)
- [P054] [Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena](https://arxiv.org/abs/2306.05685)

## 7.6.14. Failure sampleを保存する

低scoreを集計表へ埋めず、再現可能なfailure sampleとして保存する。unsupported、contradiction、citation mismatch、omission、over-refusal、format failureへ分類し、query、context、answer、判定理由、model・prompt versionをcase IDで結ぶ。RAGTruthのような実例corpusは検出器開発に役立つ。個人情報をredactしつつ、原因を失わない最小artifactを残し、修正後もcaseを回帰suiteから削除しない。頻度とseverityで改善順を決める。

**参照論文**
- [P131] [RAGTruth: A Hallucination Corpus for Developing Trustworthy Retrieval-Augmented Language Models](https://arxiv.org/abs/2401.00396)
- [P056] [RAGChecker: A Fine-grained Framework for Diagnosing Retrieval-Augmented Generation](https://arxiv.org/abs/2408.08067)
