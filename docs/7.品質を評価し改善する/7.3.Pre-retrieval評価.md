# 7.3 検索前の評価

検索前の評価は、情報源を取得してから検索用インデックスを公開するまでの品質を確認します。
入力と設定を固定し、取り込み、解析、分割、メタデータ、埋め込み、インデックスを順番に検査します。

## 7.3.1 評価対象のスナップショット

入力文書が比較中に変わると、解析器やチャンク分割の変更による差を判別できません。
評価に使う情報源のある時点の状態を複製し、変更しないよう固定したものをスナップショットと呼びます。

文書IDと版、取得日時、解析器、チャンク分割器、埋め込みモデル、インデックス設定を一つのマニフェストへ記録します。
マニフェストは、評価対象を再構築するために必要な版と設定の一覧です。

[Delta Lake](https://www.vldb.org/pvldb/vol13/p3411-armbrust.pdf)がスナップショットと過去時点の参照によって再現可能なデータ処理を支えるように、RAGでも入力時点を保存します。
変更後は別のインデックスを構築して比較し、合格後に参照先を切り替えます。
旧インデックスを一定期間保持すると、不具合時にすぐ戻せます。

## 7.3.2 取り込み

取り込みの成功は、処理が終了したことではなく、予定した文書が正しい状態で検索可能になったことで判定します。
作成、更新、削除、復元の各イベントについて、成功率、再試行回数、重複件数、欠落件数、反映時間を測ります。

情報源の文書数とインデックス側の文書数を照合しますが、件数一致だけでは十分ではありません。
文書ID、版、内容のハッシュを対応付け、古い版の残存や別文書への置換も検出します。

[MillWheel](https://www.vldb.org/pvldb/vol6/p1033-akidau.pdf)は、障害からの回復と永続的な処理状態を備えたストリーム処理を説明しています。
RAGでも、再試行によって同じ文書が二重登録されず、途中から再開しても文書を失わないことを試験します。
一部の文書だけ失敗した状態を全体成功として隠さず、文書ID、失敗工程、再処理の可否を報告します。

## 7.3.3 解析器とOCR

解析器はファイルから文字、見出し、表などの構造を取り出す処理です。
画像化された文字を認識する光学文字認識を、OCRと呼びます。

文字の一致率に加えて、読み順、見出し階層、ページ番号、箇条書き、表、図の説明、ページ上の位置が保持されたかを確認します。
[PubLayNet](https://arxiv.org/abs/1908.07836)と[DocLayNet](https://arxiv.org/abs/2206.01062)は、文書画像内の要素を人手で分類したデータセットであり、文書種別によってレイアウトが異なることを示しています。

代表文書には、人が確認した正しい解析結果を用意します。
解析器を更新するたびに、要素単位の追加、削除、順序変更を比較します。
OCRでは、数字、単位、記号、固有名詞、表の列対応を独立したスライスにします。

レイアウト検出の点数が高くても、回答に必要な表の行や引用位置が崩れる場合があります。
正解スパンを検索し、元ページの位置へ戻れるかまで検査します。

## 7.3.4 チャンク分割

チャンク分割は、平均トークン数ではなく、回答に必要なまとまりを壊さず検索できるかで評価します。
正解スパンが一つのチャンクへ収まる割合、境界で分断された割合、重複部分の割合、見出しをまたいだ割合を測ります。
表、コード、条件と例外が別々になっていないことも確認します。

[Dense X Retrieval](https://arxiv.org/abs/2312.06648)は、文書断片、文、命題という検索粒度を比較し、適した粒度が課題によって変わることを報告しています。
固定長、意味的な境界、親子チャンク、階層型の各方式を同じ質問で比較します。

検索のRecallだけでなく、最終コンテキストの正解保持率、回答の完全性、引用位置の分かりやすさまで追跡します。
細かい分割で検索が当たりやすくなっても、条件や文脈を失うなら採用しません。

## 7.3.5 メタデータ

メタデータは、文書本文以外の識別、絞り込み、引用に使う情報です。
文書ID、チャンクID、題名、ページ、節、版、有効日、情報源URLについて、欠損、型の不一致、重複、親子関係の不整合を測ります。

[Datasheets for Datasets](https://arxiv.org/abs/1803.09010)がデータセットの作成目的、構成、収集過程、利用上の注意を文書化するように、情報源ごとに所有者、収集条件、更新頻度、制約を記録します。
文書の属性が各チャンクへ正しく継承されたことを、固定した試験データで確認します。

欠損値を安全側で検索対象から除外すると、権限漏えいは防げても正解まで消える場合があります。
欠損件数と、欠損により正解根拠が除外された件数を別々に報告します。

## 7.3.6 アクセス制御

アクセス制御の評価には、取得を許可する事例と、境界を越えようとする不許可事例の両方が必要です。
テナント、役割、グループ、文書権限を組み合わせ、実際の利用者として検索を実行します。

権限外チャンクが候補へ入った割合と、許可された正解チャンクを過剰に除外した割合を分けて測ります。
[Role-Based Access Control Models](https://doi.org/10.1109/2.485845)が扱う役割階層や制約も試験条件へ反映します。

権限変更、退職、公開期限切れの直後から検索結果へ反映されるまでの時間を測ります。
公開中のインデックスだけでなく、準備中のインデックス、キャッシュ、旧版にも同じ試験を行います。
一件の情報漏えいでもリリースを止める安全条件とし、平均精度で相殺しません。

## 7.3.7 埋め込み

埋め込みモデルを比較するときは、モデル名だけでなく、版、ベクトル次元、正規化、入力接頭辞、距離尺度を一組として固定します。
[MTEB](https://arxiv.org/abs/2210.07316)では、埋め込みモデルの順位がタスクやデータセットによって変わります。

日本語、英語、言語混在、言語をまたぐ検索、専門語、意味が近い誤答候補を分けて測ります。
一般的なベンチマークは候補選定の入口に使い、最終判断には業務データのRecallとnDCGを使います。

ゼロベクトル、非数、次元の混在、空文字入力を構築時に検出します。
検索品質とともに、保存容量、生成時間、処理件数、API費用も比較します。
モデルや入力形式を変えた場合は、古いベクトルと混在させず全対象を再計算します。

## 7.3.8 インデックス構築

インデックス構築では、文書、チャンク、ベクトル、メタデータの件数をマニフェストと照合します。
古いベクトル、削除済み文書、重複ベクトル、一部だけ読み込まれた状態を検出します。

近似最近傍検索は、高速化の代わりに真の上位候補を取りこぼす場合があります。
無作為に選んだ質問で全件比較による厳密検索と結果を比べ、近似検索のRecallを測ります。

[HNSW](https://arxiv.org/abs/1603.09320)では、グラフの接続数や検索時の探索幅がメモリ、遅延、Recallのバランスを変えます。
設定ごとに構築時間、保存容量、更新性能、フィルター適用時の品質も測ります。
公開前に固定した監視質問を実行し、インデックス作成の完了だけを品質保証とはみなしません。

## 7.3.9 下流工程への影響

上流の指標が改善しても、最終回答が改善するとは限りません。
レイアウトを細かく認識した結果チャンクが断片化したり、短い命題を検索した結果例外条件が失われたりするためです。

解析器、チャンク分割器、埋め込みモデルを一要素ずつ変更します。
そのうえで、検索Recall、最終コンテキストの正解保持率、回答の正確性、引用精度を連続して測ります。
[RAGChecker](https://arxiv.org/abs/2408.08067)のように検索と生成を細分化すると、上流の改善が新しい生成上の失敗を生んでいないか確認できます。

局所指標の改善だけで公開せず、重要な下流指標が許容幅を超えて悪化しないことを合格条件にします。

## 7.3.10 回帰試験用文書

回帰試験には、通常の文書だけでなく、表、スキャン画像、数式、長文、多段組、多言語、破損したPDFを固定して置きます。
本番障害の原因になった文書は、修正後も評価対象から削除しません。

文書種別とレイアウト要素の一覧を作り、試験範囲の偏りを可視化します。
各文書について、解析要素、チャンク、メタデータ、引用位置を成果物として保存します。
変更前後の差分を文書単位で表示できるようにすると、最終得点が変わる前の構造的な破損も発見できます。
