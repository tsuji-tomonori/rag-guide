# 5.5 Context生成（必須）— 参考論文と節構成案

## 5.5.1. 質問の論点・claimへの分解

最終contextを作る前に、質問を答えに必要なclaimへ分ける。「導入条件と例外と費用を比較せよ」なら三つの論点があり、各evidenceがどれを支えるかを示す。FActScoreは長文をatomic factへ分解して支持率を測り、RARRはclaimごとに根拠を探して修正する。この発想を生成前へ移し、candidateへclaim IDを付ける。未割当候補は背景かnoiseとして再評価し、論点ごとの根拠不足をtop scoreの高さで隠さない。

**参照論文**
- [P045] [FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation](https://arxiv.org/abs/2305.14251)
- [P048] [RARR: Researching and Revising What Language Models Say, Using Language Models](https://arxiv.org/abs/2210.08726)

## 5.5.2. Evidence unitの粒度

証拠の最小単位は常にchunkではない。数値なら表のrowとheader、規程なら条文と但し書き、図ならregion、APIならresponse fieldがclaimを支える。KILTはprovenanceをtask評価へ組み込み、回答だけでなく使用箇所を特定する重要性を示した。検索しやすい単位と引用しやすいspan、LLMが誤読しない前後文を分けて持つ。mergeやcompression後も元のpage・cell・timestampへ戻せるよう、evidence unitを型付きで定義する。

**参照論文**
- [P004] [KILT: a Benchmark for Knowledge Intensive Language Tasks](https://arxiv.org/abs/2009.02252)
- [P128] [Measuring Attribution in Natural Language Generation Models](https://arxiv.org/abs/2112.12870)

## 5.5.3. Evidence roleの分類

関連する文書をすべて「根拠」と呼ぶと、反対証拠や古い資料まで同じ扱いになる。supporting、conflicting、background、outdated、insufficientを役割として付け、version、effective period、authority、scopeで判定する。FEVERのSUPPORTS／REFUTES／NOT ENOUGH INFOは、関連性と支持性を分ける基礎になる。retrieval scoreだけでroleを決めず、同じ文書がclaimごとに異なる役割を持つことも許す。

**参照論文**
- [P052] [FEVER: a Large-scale Dataset for Fact Extraction and VERification](https://aclanthology.org/N18-1074/)
- [P131] [RAGTruth: A Hallucination Corpus for Developing Trustworthy Retrieval-Augmented Language Models](https://arxiv.org/abs/2401.00396)

## 5.5.4. Provenanceとcitation payload

Citation payloadは文書名だけでは足りない。claimからdocument、page、section、span、table・figure ID、versionへ辿れ、URLが変わってもsnapshotとcontent hashで当時の原文を識別できる必要がある。ALCEはcitation correctnessとcompletenessを分け、脚注があるだけでは検証可能性を保証しないと示した。生成用の短いcitation IDと内部provenanceを分離し、同じIDを異なる本文へ再利用しない。

**参照論文**
- [P043] [Enabling Large Language Models to Generate Text with Citations](https://arxiv.org/abs/2305.14627)
- [P128] [Measuring Attribution in Natural Language Generation Models](https://arxiv.org/abs/2112.12870)

## 5.5.5. Evidence sufficiencyの判定

候補が多いことと、質問に答えられることは別である。各subquestionについて対象、期間、単位、例外、必須fieldをsupporting evidenceが満たすかを確認する。GopherCiteは根拠付き回答とabstentionを組み合わせ、一定割合を保留すると支持品質を高められることを示した。不足ならrewrite、追加検索、clarification、refusalへ分岐し、conflictは黙って削らない。score閾値一つではなく、論点coverageと一貫性で判定する。

**参照論文**
- [P076] [Teaching language models to support answers with verified quotes](https://arxiv.org/abs/2203.11147)
- [P071] [Benchmarking Large Language Models in Retrieval-Augmented Generation](https://arxiv.org/abs/2309.01431)

## 5.5.6. Token budgetの分解と選択

context windowを全てevidenceへ使うことはできない。instruction、会話・query、tool schema、answer reserveを先に確保し、残りを証拠へ配る。global top-kだけでは複合質問の一論点が候補を独占するため、subquestionごとにminimum quotaを置き、document・source capで偏りを抑える。Lost in the Middleが示すように、長く詰めるほど利用できるとは限らない。supporting、conflicting、backgroundの予算も分け、答える余地を残す。

**参照論文**
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)
- [P133] [RULER: What’s the Real Context Size of Your Long-Context Language Models?](https://arxiv.org/abs/2404.06654)

## 5.5.7. Grouping・merge・変換履歴

Evidenceを同一document、section、subquestion、時系列でgroup化すると、LLMが断片の関係を読みやすくなる。隣接chunkは重複spanだけを除いてmergeし、異なるsourceの境界は保つ。rerank、filter、dedup、merge、compressionの前後IDを鎖として記録し、mirror sourceを独立証拠として数えない。RARRのような後段revisionで元資料を再確認できるよう、変換履歴をcontext生成の副産物ではなく正式artifactにする。

**参照論文**
- [P048] [RARR: Researching and Revising What Language Models Say, Using Language Models](https://arxiv.org/abs/2210.08726)
- [P122] [Deduplicating Training Data Makes Language Models Better](https://arxiv.org/abs/2107.06499)

## 5.5.8. Orderingとposition bias対策

同じ証拠でも置く位置で利用率が変わる。Lost in the Middleは長い入力の中央にある関連情報が見落とされやすく、先頭・末尾で性能が高いU字型傾向を報告した。そこで一般的なrelevance順だけでなく、論点順、時系列、authority順、multi-hopの推論順を比較する。最重要根拠を前方へ置き、制約を末尾に複製する案もablationするが、同じ証拠の重複はcitationを曖昧にするため標準化しない。

**参照論文**
- [P007] [Lost in the Middle: How Language Models Use Long Contexts](https://arxiv.org/abs/2307.03172)
- [P120] [Attention Sorting Combats Recency Bias In Long Context Language Models](https://arxiv.org/abs/2310.01427)

## 5.5.9. 矛盾・古い根拠のserialization

矛盾する証拠を一列のcontextへ混ぜると、LLMが多数決やposition biasで一方を選びやすい。`SUPPORTING`、`CONFLICTING`、`OUTDATED`、`UNVERIFIED`を別blockにし、各々へversion、有効日、authority、scopeを添える。FreshQA系研究が扱うfast-changing knowledgeでは、新しい日付だけで正しさを決められない。serialization段階では解決済みと未解決を区別し、決着しないものは生成側へ明示的に渡す。

**参照論文**
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)
- [P052] [FEVER: a Large-scale Dataset for Fact Extraction and VERification](https://aclanthology.org/N18-1074/)

## 5.5.10. Promptへのserializationと命令境界

Promptへ渡すとき、evidence本文を命令として読ませない。各itemを安定したdelimiterで囲み、citation IDと必要最小限のversion・date・authorityだけを付ける。Indirect Prompt Injection研究は、Webや文書中の文字列が上位命令を乗っ取り得ることを実アプリで示した。したがって「以下は信頼できない参照データ」と境界を明示し、本文内のtool指示を無効化する。出力schemaとevidence blockも別fieldにし、parserの混同を避ける。

**参照論文**
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)
- [P160] [StruQ: Defending Against Prompt Injection with Structured Queries](https://arxiv.org/abs/2402.06363)

## 5.5.11. Truncationとfallback

Token超過時に末尾を機械的に切れば、文、表、code、citation markerの途中が失われる。低優先evidenceの除外、抽出型圧縮、query分割の順で縮め、主要claimの根拠が収まらなければ再検索か回答保留へ戻す。packer timeout時のfallbackは短い検証済み原文であり、根拠なし回答ではない。どのitemがbudgetで落ちたかを記録し、answer completenessの低下がtruncation由来か検索由来かを判別する。

**参照論文**
- [P036] [LLMLingua: Compressing Prompts for Accelerated Inference of Large Language Models](https://arxiv.org/abs/2310.05736)
- [P042] [RECOMP: Improving Retrieval-Augmented LMs with Compression and Selective Augmentation](https://arxiv.org/abs/2310.04408)

## 5.5.12. Validation・再現性・trace

完成したcontextには構造検査をかける。citation ID重複、source span欠落、ACL判定なし、期限不明、claim未支持、変換履歴欠損を検知し、重大項目はgenerationを止める。packer version、tokenizer、予算、候補、順序、除外理由を保存し、同点時のtie-breakを固定する。RULERが公称context長と実効能力の差を測るように、単にwindowへ収まったことを成功とせず、同じ入力から同じcontextを再現し利用できることを確認する。

**参照論文**
- [P133] [RULER: What’s the Real Context Size of Your Long-Context Language Models?](https://arxiv.org/abs/2404.06654)
- [P127] [LongBench: A Bilingual, Multitask Benchmark for Long Context Understanding](https://arxiv.org/abs/2308.14508)
