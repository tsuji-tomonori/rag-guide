## 5.5.1. 質問の論点・claimへの分解
  - complex queryをsubquestion、expected facts、制約条件へ分解する。
  - 各evidenceへ `claim_ids` を付ける。
  - 一つのclaimを複数sourceが支える場合と、複数claimを一つのsourceが支える場合を許容する。
  - claim未割当の候補をbackgroundまたはnoiseとして再評価する。

## 5.5.2. Evidence unitの粒度
  - document、chunk、paragraph、sentence、table row、figure region、API responseの粒度を比較する。
  - atomic claimを支える最小spanと、誤読を防ぐ前後文・見出しを分けて保持する。
  - 表ではrow/column header、規程では条番号と但し書き、codeではsymbolと依存行を一体で扱う。
  - merge・compression後も元のchunkとspanへ戻れるようにする。

## 5.5.3. Evidence roleの分類
  - supporting: 主張を直接支える。
  - conflicting: 主張と反対または異なる値を示す。
  - background: 理解には役立つが主張を直接支えない。
  - outdated: 過去には有効だが基準時点では失効している。
  - insufficient: 関連はあるが回答に必要な情報が不足している。
  - roleをretrieval scoreだけで決めず、version、effective period、authority、scopeを使う。

## 5.5.4. Provenanceとcitation payload
  - document IDだけでなく、page、section、span、table/figure ID、versionを保持する。
  - source URLが変わる場合に備え、content hashとsnapshot IDを持つ。
  - citationを回答文末の飾りではなく、claimから原文へ辿れるlinkとして設計する。
  - generated summaryではなく、identified sourceへattributionできることを要件にする。
  - 同じcitation IDを異なる本文へ再利用しない。

## 5.5.5. Evidence sufficiencyの判定
  - 各subquestionにsupporting evidenceがあるか確認する。
  - 対象、期間、単位、例外条件、必須fieldが揃っているか判定する。
  - top scoreが高いことと、回答に十分なことを区別する。
  - 不足時はquery rewrite、追加retrieval、clarification、refusalへ分岐する。
  - conflicting evidenceを黙って削除せず、5.6の処理へ渡す。

## 5.5.6. Token budgetの分解と選択
  - `context window = instructions + conversation/query + evidence + tool/schema + answer reserve` と分解する。
  - answer reserveを先に確保し、残りをevidenceへ割り当てる。
  - global top-kだけでなく、subquestionごとのminimum quotaを持たせる。
  - per-document / per-source capで偏りを抑える。
  - supporting、conflicting、backgroundへ明示的にbudgetを配分する。

## 5.5.7. Grouping・merge・変換履歴
  - 同一document、section、subquestion、時系列でgroup化する。
  - 隣接chunkは重複spanを除いてmergeする。
  - rerank、filter、dedup、merge、compressionの各処理と前後ID対応を記録する。
  - cross-document evidenceはsource境界を保ち、同一原典のmirrorを独立した裏付けとして数えない。

## 5.5.8. Orderingとposition bias対策
  - relevance順、論点順、時系列順、authority順、source内文書順を比較する。
  - multi-hopでは推論順、規程や版管理では現行根拠を先に置く。
  - 最重要根拠を先頭寄りに置き、重要な制約・拒否条件を末尾寄りに置く案を評価する。
  - 候補順を入れ替えるablationで順序依存を測る。
  - 同じ根拠の重複配置はtoken浪費とcitation曖昧化を招くため、標準手法にしない。

## 5.5.9. 矛盾・古い根拠のserialization
  - `SUPPORTING`、`CONFLICTING`、`OUTDATED`、`UNVERIFIED` を別blockにする。
  - 各blockへversion、effective date、authority、scopeを表示する。
  - 「多数決」「score最大」で矛盾を解消しないようprompt境界を定義する。
  - 詳細な優先順位・回答保留・escalationは5.6へ接続する。

## 5.5.10. Promptへのserializationと命令境界
  - evidenceごとに安定したdelimiterとcitation IDを付ける。
  - metadataをすべて渡さず、生成判断に必要なversion、date、authority、scopeだけを含める。
  - retrieved textを「命令」ではなく「信頼できないデータ」として囲う。
  - candidate本文に含まれる指示をsystem/developer instructionより優先させない。
  - JSONやMarkdownなど出力schemaとevidence blockを混同しない。

## 5.5.11. Truncationとfallback
  - 文、表、code block、citation markerの途中で切らない。
  - budget不足時は、低優先evidenceの削除、extractive compression、query分割の順に検討する。
  - 重要evidenceが収まらない場合は、根拠不足として再検索または回答保留にする。
  - packer timeout時は、検証済みの短い上位原文へfallbackし、根拠なし生成へはfallbackしない。

## 5.5.12. Validation・再現性・trace
  - citation ID重複、source span欠落、ACL判定欠落、期限不明、変換履歴欠落を検出する。
  - claimにsupporting evidenceがない場合はgenerationを停止または制限する。
  - packer version、tokenizer、budget、選択候補、順序、除外理由を記録する。
  - 同一入力で同じcontextを再現できる決定的なtie-break規則を持つ。
