## 5.1.1. Hard filterとsoft filter
  - hard filter: `tenant_id`、ACL、機密区分、削除状態、法的保留、`effective_from/to`、公開状態。
  - soft filter / boost: product、region、language、document_type、freshness、source_trust。
  - ACLやtenant条件をsoft boostにしてはいけない理由を書く。
  - user queryから推定した条件と、認証基盤から得た条件を分ける。

## 5.1.2. Metadata schema
  - 最低限のfieldを、`document_id`、`chunk_id`、`tenant_id`、`allowed_principals`、`classification`、`status`、`version`、`effective_from/to`、`source_type`、`source_trust`、`language` とする。
  - null、未設定、複数値、timezone、版番号の比較規則を定義する。
  - chunk-level metadataとdocument-level metadataの継承規則を書く。
  - metadata欠損時は安全側に除外するfieldと、検索品質のため警告だけにするfieldを分ける。

## 5.1.3. Filterを適用する場所
  - ingestion時: 不正source、期限切れ、削除済み文書をindexへ入れない。
  - retrieval前: tenant、user/group、機密区分から検索可能集合を決める。
  - in-index: vector searchとpredicateを同時に処理する。
  - post-retrieval: metadata整合性の再検証、soft条件の調整、異常候補の隔離を行う。
  - post-filterだけに依存すると、top-kを権限外候補が占有しrecallが落ちること、promptやtraceへ漏れる危険があることを書く。

## 5.1.4. Filter selectivityと検索品質
  - selectivityが高い・低い場合でANN graph探索の難しさが変わることを説明する。
  - `search-then-filter`、`filter-then-search`、predicate-aware hybrid searchを比較する。
  - 結果が0件の場合に、ACL条件を緩和せず、semantic条件だけを段階的に緩和する。
  - filter条件ごとに候補数とrecallをtraceする。

## 5.1.5. 時間・版・状態のfilter
  - `published_at`、`updated_at`、`effective_from`、`effective_to`を区別する。
  - 最新版だけを返す質問と、過去時点を問う質問を分ける。
  - superseded文書、draft、archived、deletedを明確に扱う。
  - 版の競合は単純除外せず、5.6の矛盾処理へ渡す条件を定義する。

## 5.1.6. Source trustと攻撃耐性
  - authoritative、reviewed、user-generated、external webなどのtrust tierを定義する。
  - 高関連度でも低信頼sourceを自動採用しない。
  - retrieved text内の命令をfilter条件やtool命令として解釈しない。
  - poisoning検知、異常な重複、急増したsource、署名不一致を隔離対象にする。

## 5.1.7. Filter expressionの安全性
  - user inputをそのままfilter DSLやSQLへ連結しない。
  - allowlistされたfieldとoperatorだけを使う。
  - 認証由来のpolicy条件とユーザー指定条件を別のASTとして構築し、最終的にAND結合する。
  - filter解析失敗時はfail-openではなくfail-closedにする。

## 5.1.8. Observabilityと監査
  - principal、tenant、適用policy version、filter式、除外理由、候補数をtraceする。
  - 権限外文書の本文はログへ残さず、document IDと除外reasonのみを必要最小限記録する。
  - 権限変更がindex・cache・sessionへ反映されるSLOを定義する。
