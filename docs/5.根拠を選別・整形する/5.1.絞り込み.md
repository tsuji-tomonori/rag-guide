# 5.1 絞り込み（Filtering）（必須）— 参考論文と節構成案

## 5.1.1. Hard filterとsoft filter

Filterには、違反すれば絶対に候補へ入れない条件と、順位を調整する条件がある。tenant、ACL、機密区分、削除、有効期間はhard filterであり、product、language、freshness、source trustは状況に応じたsoft filterである。ACORNが扱うpredicate-aware searchは前者を効率よく探索へ組み込む技術だが、両者の意味を混同すると認可がscore次第になる。利用者が明示した条件とモデル推定も分け、hard条件はfail-closedで処理する。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)

## 5.1.2. Metadata schema

Filterの精度はmetadataの語彙と型で決まる。document・chunk ID、tenant、allowed principals、classification、status、version、有効期間、source type、trust、languageを最小schemaとして定義し、NULLや複数値の意味まで決める。Cedarはprincipal、action、resource、contextを明示するpolicy言語により、認可判断をアプリの曖昧なif文から分離する。文書属性のchunk継承規則を固定し、欠損時に除外するfieldと警告に留めるfieldを区別する。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P101] [Role-Based Access Control Models](https://doi.org/10.1109/2.485845)

## 5.1.3. Filterを適用する場所

Filterは一度だけ実行する処理ではない。ingestionでは失効・不正sourceを隔離し、retrieval前にはidentityから検索可能集合を作り、index内ではpredicateと近傍探索を同時に行い、取得後にはmetadata整合性を再確認する。post-filterだけでは権限外candidateがtop-kを埋め、本文がtraceへ漏れる。反対にingestion時だけでは後日の権限変更に追随できない。各層の目的を分担し、同じdeny規則がどこで保証されるかを表にする。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)

## 5.1.4. Filter selectivityと検索品質

Filter selectivityは「条件に合う文書の割合」であり、ANNの難しさを変える。対象が全体の1%しかないと、通常のgraph探索は近いが条件外のnodeばかり辿り、必要件数を得られないことがある。ACORNやNaviXはpredicateを考慮する探索でこの問題へ取り組む。実装ではselectivity帯ごとにcandidate数、recall、latencyを測り、0件でもACLを緩和しない。緩められるのはsemanticなsoft条件だけであり、その順序を明示する。

**参照論文**
- [P104] [ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data](https://arxiv.org/abs/2403.04871)
- [P129] [NaviX: A Native Vector Index Design for Graph DBMSs With Robust Predicate-Agnostic Search Performance](https://arxiv.org/abs/2506.23397)

## 5.1.5. 時間・版・状態のfilter

「最新」と「2023年当時」は同じfilterではない。published、updated、effective-from/toを分け、質問の基準時点に有効だった版を選ぶ。draft、archived、deleted、supersededも状態として扱うが、過去時点質問では旧版を消してはいけない。FreshLLMsやStreamingQAは時間とともに変わる知識をQAで評価し、単に新しい文書を上位に置くだけでは不十分だと示す。版競合が解けなければ除外せず、矛盾処理へ渡す。

**参照論文**
- [P124] [FreshLLMs: Refreshing Large Language Models with Search Engine Augmentation](https://arxiv.org/abs/2310.03214)
- [P135] [StreamingQA: A Benchmark for Adaptation to New Knowledge over Time in Question Answering Models](https://arxiv.org/abs/2205.11388)

## 5.1.6. Source trustと攻撃耐性

関連度の高い文書が安全とは限らない。PoisonedRAGは少数の攻撃文書をknowledge baseへ挿入し、特定queryの生成を誘導できることを示した。そこでsourceをauthoritative、reviewed、user-generated、externalへ分け、低信頼sourceを高risk回答やtool actionの唯一根拠にしない。急増した類似文書、署名不一致、異常なkeyword密度を隔離し、本文中の命令をfilterやtool指示として解釈しない。trustとrelevanceは別の軸で管理する。

**参照論文**
- [P060] [PoisonedRAG: Knowledge Corruption Attacks to Retrieval-Augmented Generation of Large Language Models](https://arxiv.org/abs/2402.07867)
- [P103] [Towards Trustworthy Retrieval Augmented Generation for Large Language Models: A Survey](https://arxiv.org/abs/2502.06872)

## 5.1.7. Filter expressionの安全性

利用者の文字列をfilter DSLやSQLへ直接連結すると、検索条件そのものが注入面になる。許可fieldとoperatorをallowlistし、値は型検査・parameter化する。認証由来のtenant・ACL条件と、質問から抽出した製品・期間条件を別ASTとして構築し、最終段でAND結合する。Cedarのようなpolicy-as-codeは判断の入力と規則を明確にするが、parse failure時の挙動も必要である。失敗はfail-openにせず、検索停止または安全な最小集合へ落とす。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)

## 5.1.8. Observabilityと監査

後から「なぜこの文書が消えたか」を答えられるよう、filterを観測対象にする。principal、tenant、policy version、評価した式、候補数、除外reason、権限変更時刻をtraceへ残す。ただし権限外本文までログへ保存すれば監査が漏えい源になるため、IDとdecision metadataに限定する。revocationがindex、cache、sessionへ届く時間をSLOにし、0件や候補急減を品質異常として監視する。認可失敗と検索漏れを同じzero-resultに埋めない。

**参照論文**
- [P085] [Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization](https://arxiv.org/abs/2403.04651)
- [P134] [Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use](https://arxiv.org/abs/2605.05287)
