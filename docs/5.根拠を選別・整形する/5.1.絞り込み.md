# 5.1 絞り込み

絞り込み（Filtering）は、検索候補から、利用できない文書や質問の条件に合わない文書を除く処理です。
権限・削除・有効期間のように必ず守る条件と、順位を調整する条件を分けます。

## 5.1.1 必須フィルターと順位調整条件

**必須フィルター（Hard filter）**は、条件に違反する候補を結果へ入れない強制条件です。
利用組織を分ける単位（テナント）、アクセス制御リスト（Access Control List：ACL）、機密区分、削除状態、質問時点での有効性などがあります。
スコアが高くても条件を上書きできません。

**順位調整条件（Soft filter）**は、候補を完全に除外せず、質問への適合度に応じて順位を調整する条件です。
利用者が明示していない製品・言語の推定、鮮度、情報源の信頼度などが候補になります。

同じ属性でも、質問によって扱いが変わります。
「製品Aだけ」と利用者が明示した場合は必須フィルターにでき、モデルが製品Aだと推定しただけなら順位の引き上げまたは別検索にします。
認可条件は推定値として扱わず、認証済み情報から導きます。

## 5.1.2 メタデータの項目設計

フィルターの正しさは、メタデータの型、値、欠損時の意味に依存します。
文書・チャンクID、テナント、許可主体、機密区分、状態、版、有効期間、情報源種別、信頼度、言語を定義します。

項目に値がない状態（NULL）の場合、条件なしを意味するのか、未知を意味するのかを決めます。
ACL、テナント、削除状態の未知値は、既定で拒否します。
表示用の補足属性が欠けた場合は、警告で扱えることがあります。

文書の属性をチャンクへ継承する規則を固定します。
添付、コメント、埋め込み資料が親文書と異なるACLを持つ場合は、例外関係として明示します。
項目設計の版を保存し、旧版・新版本文が混在する検索を検査します。

## 5.1.3 フィルターを適用する場所

フィルターは一度だけ実行する処理ではありません。
各段階で目的が異なります。

表5-1は、左列の処理段階と、右列で行う検査の目的を対応付けます。
上から処理順に読みます。
同じ拒否規則でも、取り込み時の隔離と、質問時の最新権限確認では役割が異なります。

**表5-1　フィルターを適用する段階と目的**

| 段階 | 主な目的 |
|---|---|
| 取り込み | 失効、対象外、不正な情報源を隔離します |
| 検索前 | 利用者の認可済み候補集合を作ります |
| インデックス探索中 | 条件に合う候補を効率的に探索します |
| 検索後 | メタデータ整合性と状態を再確認します |
| 根拠集合の生成前 | 利用禁止候補が残っていないことを最終確認します |

検索後フィルターだけでは、権限外の候補が検索上位を占め、本文が中間データや処理記録へ入る可能性があります。
取り込み時のフィルターだけでは、後日の権限変更や失効へ対応できません。
同じ拒否規則がどの段階で強制されるかを設計し、二重化した検査の責任を明確にします。

## 5.1.4 フィルターの選択率

**フィルターの選択率（Filter selectivity）**は、全候補のうち条件に合う割合です。
対象が全体の1パーセントの場合、通常の近似最近傍（Approximate Nearest Neighbor：ANN）グラフは条件外の近傍を多くたどり、必要件数を取得できないことがあります。

[ACORN](https://arxiv.org/abs/2403.04871)は、条件式を近傍探索へ組み込み、条件付きANN検索の性能課題を扱いました。
条件の選択率を帯に分け、候補件数、必要根拠の回収率、応答時間を測ります。

結果がゼロ件でもACLを緩めません。
検索深度、候補を必要数より多めに取得する方法、条件に対応したANN検索、インデックス分割、完全探索への代替を検討します。
緩和できるのは、業務上の順位調整条件と定義した意味条件だけです。

[フィルターを考慮した距離を学習するワークショップ公開論文](https://openreview.net/forum?id=dILIRHcYvC)は、複数条件の下で少ない距離計算でも候補を拾いやすくする方法を扱いました。
ただし、強い既存方式との比較、実際の応答時間、保存容量の評価は限られています。
研究結果だけで方式を固定せず、フィルターの選択率ごとに完全探索との一致と処理時間を測ります。

## 5.1.5 時間、版、状態

「最新」と「2023年当時」では、選ぶ文書版が異なります。
公開、更新、適用開始、適用終了、取り込みの時刻を分け、質問の基準時点と比較します。

下書き、現在有効、失効、削除、後継版ありという状態を持たせます。
現在の質問では旧版を除外できますが、過去時点の質問では当時の版が必要です。
旧版を単純に削除せず、版の前後関係を保持します。

同じ時点に複数の有効版があり、優先規則で解決できない場合は、一方をフィルターで捨てません。
矛盾・版競合として後段へ渡します。

## 5.1.6 情報源の信頼度と攻撃耐性

質問との関連度が高い文書でも、安全または正しいとは限りません。
[PoisonedRAG](https://arxiv.org/abs/2402.07867)は、攻撃文書を知識ベースへ挿入し、特定の質問への回答を操作する攻撃を示しました。

情報源を、正式、レビュー済み、利用者生成、外部、派生などに分類します。
低信頼の情報源を、高リスク回答や外部操作の唯一の根拠にしません。
信頼度と関連度を別の属性として管理します。

短期間に急増した類似文書、署名や所有者の不一致、異常な検索語の反復を検知して隔離します。
文書中の命令を、フィルター条件やツールへの指示として解釈しません。

## 5.1.7 フィルター式の安全性

利用者の文字列をフィルター式やデータベース問い合わせ文（SQL）へそのまま連結すると、条件注入の危険があります。
利用を許可する項目と演算子を列挙し、値の型を検査して、問い合わせ文とは別の値として渡します。

認証・認可由来のテナントとACL条件、質問から抽出した製品・期間条件を、条件を木状に表す別々の構文木として作ります。
最後に論理積で結合し、利用者入力が認可条件を削除・上書きできない構造にします。

フィルター式を解析できない場合は、候補の利用を許可しません。
検索を停止するか、認可済みの最小集合へ限定します。
例外処理には対象、理由、承認者、期限を付けます。

## 5.1.8 観測と監査

後から「なぜこの文書が候補から消えたか」を説明できるよう、フィルターの判断を処理記録へ残します。
主体、テナント、方針の版、評価した条件、フィルター前後の候補数、除外理由を記録します。

権限外本文を監査記録へ保存すると、記録自体が漏えい源になります。
文書ID、判断、理由など、調査に必要なメタデータに限定し、監査記録にも閲覧権限と保持期間を設けます。

権限剥奪がインデックス、一時保存、利用中の会話状態へ届くまでの時間を測ります。
検索結果ゼロを一つの指標へまとめず、認可によるゼロ件、条件不一致、検索漏れ、情報源欠損を区別します。
