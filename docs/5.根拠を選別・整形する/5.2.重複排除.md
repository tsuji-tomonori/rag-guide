## 5.2.1. 重複の分類
- exact duplicate: 正規化後の本文が同一。
- near duplicate: 表記・ヘッダー・フッター・改行・軽微な編集だけが異なる。
- adjacent overlap: sliding-window chunkの共有部分が大きい。
- cross-format duplicate: 同じ内容のPDF、HTML、Markdown、Office版。
- mirrored source: 複製サイトや部門別コピー。
- version family: 旧版と新版。内容差があるため、無条件にduplicate扱いしない。

## 5.2.2. 正規化とexact dedup
- Unicode正規化、空白・改行、ページ番号、定型footer、tracking parameterの扱いを定義する。
- raw hashとnormalized content hashを両方保持する。
- table、数値、否定語を過剰に正規化しない。
- exact duplicateを代表文書へ統合するとき、全source URLとversion情報をprovenanceとして残す。

## 5.2.3. Near-duplicate検出
- character / token shingle + Jaccard + MinHashを説明する。
- SimHash + Hamming distanceを説明する。
- embedding cosine similarityは意味的類似も拾うため、duplicate判定に単独使用しない。
- 高recallなfingerprint候補生成と、原文比較による精密判定の二段階にする。
- thresholdは文書種別別に評価する。

## 5.2.4. 隣接chunkのmerge
- 同一document、連続page/section、overlap率、見出しpathでmerge条件を決める。
- merge後のtoken上限と、元chunk spanの対応表を保持する。
- 離れた箇所を無理に結合しない。
- parent-child retrievalでは、同じparentへ置換されたchild候補を一つにまとめる。

## 5.2.5. Version-aware dedup
- `document_family_id` と `version_id` を分ける。
- 完全に同じ本文の再配布版は統合できるが、規定値や日付が異なる版は統合しない。
- current queryの基準時点に応じてactive versionを選ぶ。
- 複数active versionやauthority競合は5.6へ渡す。

## 5.2.6. Diversity制御
- per-document cap、per-source cap、per-section capを説明する。
- MMRで `query relevance` と `selected evidenceとの非類似性` を両立する。
- multi-part questionでは、subquestionごとのquotaを持たせる。
- source diversityを増やしすぎると低品質sourceが入るため、source trustを制約にする。
- 多様性は「異なる言い換え」ではなく「新しい根拠・論点」を増やすことを目的にする。

## 5.2.7. Dedupとrerankingの順序
- exact duplicateはrerank前に落として計算量を減らす。
- semantic near-duplicateとdiversityはrerank scoreを使って選ぶ。
- 同一文書の隣接chunkをmergeする場合、merge前scoreの集約規則を定義する。
- dedup前後の候補ID対応をtraceへ残す。

## 5.2.8. Citationとprovenance
- 代表chunkだけを残しても、元source一覧を失わない。
- mergeされたevidenceには複数のsource spanを紐づける。
- citation UIでは、代表sourceと「同内容の別source」を区別する。
- generated summaryをdedupの代表本文として引用しない。
