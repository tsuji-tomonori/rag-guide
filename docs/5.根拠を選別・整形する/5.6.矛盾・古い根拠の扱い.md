## 5.6.1. 矛盾の分類
  - temporal conflict: 旧版と新版、改定前後。
  - authority conflict: 本社規程と部門メモ、公式仕様とFAQ。
  - scope conflict: 地域、製品、契約、tenant、例外条件が異なる。
  - factual conflict: 同じscope・時点なのに値や主張が異なる。
  - parsing conflict: OCR、表列、単位、日付抽出の誤り。
  - adversarial conflict: poisoning、偽source、retrieved prompt injection。

## 5.6.2. 時間metadataの扱い
  - `published_at`、`updated_at`、`effective_from`、`effective_to`、`retrieved_at`を分ける。
  - queryから基準時点を抽出し、未指定なら「現在時点」など明示的なdefaultを使う。
  - timezone、inclusive/exclusive boundary、未来施行文書を定義する。
  - point-in-time queryでは旧版を削除せず、時点に合う版を検索する。

## 5.6.3. Version lineage
  - `document_family_id`、`version_id`、`supersedes`、`superseded_by`を管理する。
  - semver、日付版、改訂番号などの比較規則を文書種別ごとに定義する。
  - 新版が旧版の全内容を置き換えるのか、一部だけ改定するのかを区別する。
  - lineage不明な文書を自動的に最新版扱いしない。

## 5.6.4. Authorityと適用scope
  - source authority、document owner、approval status、policy tierを定義する。
  - より新しい非公式メモが、古い正式規程を自動的に上書きしない。
  - global ruleよりscope-specific exceptionが優先される条件を書く。
  - authority同等の矛盾は、人間ownerへescalationする。

## 5.6.5. 矛盾検出
  - metadata ruleで時間・版・scopeの衝突を先に検出する。
  - claimを正規化し、support / refute / NEIを判定する。
  - 数値、単位、条件、否定、例外を抽出して比較する。
  - LLM/NLI判定は補助とし、重要判断はruleとsource metadataで拘束する。

## 5.6.6. 解決アクション
  - authoritativeな後継版が明確: 旧版をoutdatedとして分離する。
  - scope違い: 質問のscopeに適用される根拠を選び、他方を補足へ回す。
  - 根拠不足: query rewrite、追加検索、clarificationへ分岐する。
  - 同等authorityの未解決矛盾: 両方を提示し、確定回答を避ける。
  - poisoning疑い: candidateを隔離し、回答へ使わず、security eventを発行する。

## 5.6.7. LLMへ渡す形式
  - `CURRENT`, `OUTDATED`, `CONFLICTING`, `UNVERIFIED` のlabelを付ける。
  - 各evidenceへversion、effective date、authorityを明示する。
  - 「多数決で決める」「scoreが高い方を採用する」といった誤った解決をpromptで禁止する。
  - 解決不能時の定型応答と確認先を定義する。

## 5.6.8. 生成後の検証
  - 回答claimがcurrent evidenceに支えられているか確認する。
  - outdated evidenceだけに基づくclaimを検出する。
  - evidenceと反対のclaim、sourceにない統合claimを検出する。
  - unsupported / contradictory claimは修正または回答保留にする。
