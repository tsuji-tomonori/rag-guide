# 5.6 矛盾・古い根拠の扱い

候補同士の内容が違う場合、関連度の高い方や新しい方を機械的に残すと、質問の時点・対象に合う根拠を失う可能性があります。
矛盾の原因を分類し、版、情報源の公式性と優先度、適用範囲、本文から解決方法を選びます。

## 5.6.1 矛盾の分類

矛盾は、少なくとも次の種類へ分けます。

- **時間的矛盾**は、旧版と新版の違いです。
- **情報源の優先関係による矛盾**は、正式規程と非公式メモなどの違いです。
- **適用範囲の矛盾**は、地域、契約、製品版などの条件差です。
- **事実矛盾**は、同じ時点・対象・単位で値や主張が違う状態です。
- **解析矛盾**は、画像中の文字を読む光学文字認識（Optical Character Recognition：OCR）、表の列、単位の読み違いです。
- **攻撃・汚染**は、悪意ある文書が作る衝突です。

[FEVER](https://aclanthology.org/N18-1074/)の支持、反証、情報不足という分類は、主張と根拠の関係を分ける基礎になります。
RAGでは、メタデータだけで解ける版・範囲の違いと、本文比較が必要な事実矛盾をさらに区別します。

分類結果によって、優先、併記、追加検索、隔離、人への確認を選びます。

## 5.6.2 時間メタデータ

文書の日付には、公開、更新、適用開始・終了、取得があります。
互いに代用できません。
質問の基準時点で有効だった版を選び、現在の最新版だけを返しません。

質問に時点がない場合の既定値を定めます。
通常は現在時点ですが、会計期間、契約開始日、事故発生日など、業務上の基準がある場合は明示します。
標準時との時差（タイムゾーン）、境界時刻を含む・含まない、未来施行の扱いも決めます。

[StreamingQA](https://arxiv.org/abs/2205.11388)は、時刻付きの知識源と新しい知識への適応を評価しました。
過去版を単純に削除せず、時点指定の検索へ利用します。

## 5.6.3 版のつながり

版番号だけでは、どの文書がどれを置き換えたか分かりません。
文書系列ID、文書版ID、前版・後継版の関係を持ちます。
全面改定、一部改定、再配布、訂正を区別します。

版番号の三つの数値で変更の大きさを表すセマンティックバージョニング（Semantic Versioning）、日付、改訂番号など、版の比較方法は文書種別ごとに異なります。
更新日時だけで最新版を決めると、後から保存した非公式コピーが正式版より新しく見える場合があります。

更新時に、祖先関係、有効期間、承認状態を検査します。
版のつながりが分岐・循環・欠損している場合は、自動統合せず所有者へ確認します。

## 5.6.4 情報源の公式性・優先度と適用範囲

日付が新しい文書が、常に優先されるとは限りません。
正式な全社規程と、昨日書かれた個人メモが矛盾する場合は、正式規程を優先します。
一方で、全社規程へ正式な地域例外がある場合は、質問の地域へ適合する例外を使います。

情報源の所有者、承認状態、方針の優先関係、対象地域・製品・契約をメタデータとして記録します。
質問の対象と比較し、同じ条件の根拠だけを競合として扱います。

同じ公式性・優先度、時点、適用範囲で主張が違う場合は、検索スコアや文書件数で決めません。
両方を未解決の矛盾として保持し、責任者へ確認します。

## 5.6.5 矛盾検出

矛盾検出を一回のLLM判定だけへ任せません。
最初に、版、有効期間、適用範囲、単位、状態を、同じ入力なら同じ結果になる規則で比較します。
次に、主張を対象、関係、値、条件へ分け、支持、反証、情報不足を判定します。

数値差、単位、否定、条件、例外を構造的に抽出します。
文と文の支持・矛盾を判定する自然言語推論モデルやLLMは、言い換えと複雑な文章の判定を補助します。
判定器にも誤りがあるため、人が確認した矛盾集合で見逃しと誤検出を測ります。

[RAGTruth](https://arxiv.org/abs/2401.00396)は、取得した入力文脈に対して支持されない生成と矛盾する生成の例を収集しました。
関連文書があることだけでは、矛盾を正しく処理できたことになりません。

## 5.6.6 解決動作

解決処理は矛盾の種類から選びます。

表5-2は、左列で確認した状態に対し、右列の処理を選ぶ対応表です。
上から優先順位を示す表ではありません。
解析誤りや汚染の疑いがある場合は、日付だけで版を選ばず、原資料の確認または隔離を先に行います。

**表5-2　矛盾の状態と処理**

| 状態 | 主な処理 |
|---|---|
| 正式な後継版が明確です | 現行版を主根拠とし、旧版として区別します |
| 適用範囲が違います | 質問の範囲に合う根拠を選びます |
| 情報が不足しています | 質問変換、追加検索、確認質問を行います |
| 同等の根拠が矛盾します | 両方を示し、確定回答を保留します |
| 解析誤りの疑いがあります | 原画像・表を再解析し、人が確認します |
| 汚染の疑いがあります | 文書を隔離し、セキュリティ事象として扱います |

[Corrective RAG](https://arxiv.org/abs/2401.15884)は、取得文書を評価し、必要に応じて検索結果を補正する経路を提案しました。
追加検索が有効な状態と、隔離が必要な攻撃疑いを区別します。

## 5.6.7 LLMへ渡す形式

矛盾する根拠を一続きの文章として渡しません。
`CURRENT`（現行）、`OUTDATED`（旧版）、`CONFLICTING`（矛盾）、`UNVERIFIED`（未確認）などの区画へ分けます。
各ブロックへ文書版、有効期間、情報源の公式性と優先度、適用範囲、引用IDを付けます。

プロンプトでは、件数が多い方や検索スコアが高い方を真と見なさないようにします。
優先規則はLLMが作らず、文書を管理する組織の方針から与えます。

解決できない場合の回答には、不一致の要点、両方の根拠、確認できた範囲、確認先を含めます。
LLMの役割を、与えられた根拠の説明へ限定します。

## 5.6.8 生成後の検証

回答を主張単位へ分け、現在有効な根拠が各主張を支持するかを確認します。
旧版だけに支えられた主張、反対根拠を無視した断定、複数情報源から根拠なく合成した値を検出します。

[RefChecker](https://arxiv.org/abs/2405.14486)は、参照に基づき、根拠に支えられない生成（ハルシネーション）を細かく検査する方法を扱いました。
支持なしと矛盾を別の失敗として記録します。

軽微な表現誤りは根拠へ合わせて修正し、重大または解消不能な矛盾では回答を保留します。
検証結果、修正前後、利用した根拠を処理記録へ残します。
発見した矛盾を、変更前の品質を保つ試験（回帰試験）へ追加し、同じ誤りの再発を防ぎます。
