## 5.3.1. Rank fusionとrerankingの違い
- RRF、weighted score fusion、learned fusionを整理する。
- RRFは文書本文を深く読まず、各retrieverの順位を統合する方式であると説明する。
- BM25 scoreとcosine similarityを未校正のまま加算しない理由を書く。
- hybrid retrievalの候補統合と、cross-encoderによる意味的再判定を別工程として図示する。

## 5.3.2. Cross-encoder reranker
- queryとcandidateを同一Transformerへ入力する仕組みを説明する。
- bi-encoderとの精度・レイテンシ・index再利用性の違いを比較する。
- title、heading、本文、日付など、rerankerへ渡す入力fieldを定義する。
- 長すぎるcandidateのtruncationが関連箇所を落とす失敗を扱う。
- 多言語・日本語では公開ランキングだけでなく業務データで評価する必要性を書く。

## 5.3.3. Pointwise / Pairwise / Listwise
- pointwise: 各候補を独立にscore化する。
- pairwise: 候補対の優劣を学習または判定する。
- listwise: 候補集合全体を見て並べ替える。
- 候補数増加時の計算量、順序依存、再現性を比較表にする。

## 5.3.4. monoT5 / RankT5
- relevance tokenのlogitをranking scoreとして読む方法を説明する。
- classification lossとranking lossの目的差を説明する。
- domain fine-tuningを行う場合のpositive / hard negative作成方法を扱う。
- 教師ラベルの偏りと、古いindexから作ったnegativeによる分布ずれを注意点にする。

## 5.3.5. LLM reranker
- zero-shot listwise、pairwise prompting、sliding-window rerankingを整理する。
- candidate本文に含まれる命令をデータとして扱い、rerankerへのprompt injectionを防ぐ。
- position biasを測るため、候補順を入れ替える評価を入れる。
- LLM出力が不正なID列、重複ID、欠落IDを返す場合のvalidationを定義する。
- 高コストなLLM rerankerは、複雑queryだけにroutingする案を示す。

## 5.3.6. 本番パイプラインの設計
- `retrieve N -> fuse -> hard filter validation -> rerank -> select K` の流れを示す。
- 初期候補数 `N` と最終件数 `K` を、固定値ではなく評価対象として扱う。
- batching、cache、timeout、同時実行数、model fallbackを設計項目にする。
- reranker timeout時はfirst-stage順位へfallbackし、根拠なし生成へはfallbackしない。
- score、model version、入力候補順、出力順位をtraceへ保存する。

## 5.3.8. 選定ガイド
- 学習データなし・低コスト: RRF + 公開cross-encoder。
- 十分なrelevance labelあり: domain-tuned cross-encoderまたはRankT5。
- 少数候補の複雑な意味判断: LLM listwiseを限定利用。
- rerank効果が小さい場合: first-stage recall、chunk粒度、metadata filterを先に見直す。
