# 5.3 順位付け（Reranking）（必須）— 参考論文と節構成案

## 5.3.1. Rank fusionとrerankingの違い

Rank fusionは複数retrieverの順位をまとめ、rerankingは候補本文を読み直してqueryとの適合を再判定する。RRFはscore尺度をそろえず順位だけで統合できるが、文中の否定や条件までは理解しない。BERT rerankerはqueryとpassageを同時入力し、その細部を評価する。この境界を曖昧にすると、fusion parameter調整で意味誤判定を直そうとしてしまう。候補生成、fusion、精密順位付けを別stageとして図示し、各段階の改善指標を分ける。

**参照論文**
- [P032] [Reciprocal Rank Fusion Outperforms Condorcet and Individual Rank Learning Methods](https://dl.acm.org/doi/10.1145/1571941.1572114)
- [P040] [Passage Re-ranking with BERT](https://arxiv.org/abs/1901.04085)

## 5.3.2. Cross-encoder reranker

Cross-encoderはqueryとcandidateを一つのTransformerへ通し、token間の相互作用を見ながら関連度を判定する。BERT passage rerankingはfirst-stage候補へこの高精度判定を適用し、全corpusを読む負担を避けた。代償は候補数に比例する推論時間である。title、heading、本文、日付をどこまで入力するかを決め、truncationで答えの箇所を落とさない。日本語・専門domainでは公開benchmarkより業務goldで精度とp95 latencyを測る。

**参照論文**
- [P040] [Passage Re-ranking with BERT](https://arxiv.org/abs/1901.04085)

## 5.3.3. Pointwise / Pairwise / Listwise

Pointwiseは候補を一件ずつ採点し、pairwiseは二件の優劣を比べ、listwiseは候補集合全体を並べる。独立採点はbatchしやすいが相対順位を直接学ばず、pairwiseは比較数が増え、listwiseは位置や入力長に敏感になる。Zero-shot listwise研究はLLMに候補ID列を直接出させ、教師なしでも強い再順位付けを示した一方、順序biasと窓分割の問題を持つ。候補数、再現性、costに応じて方式を選び、出力検証を変える。

**参照論文**
- [P084] [Zero-Shot Listwise Document Reranking with a Large Language Model](https://arxiv.org/abs/2305.02156)
- [P132] [RankT5: Fine-Tuning T5 for Text Ranking with Ranking Losses](https://arxiv.org/abs/2210.10634)

## 5.3.4. monoT5 / RankT5

monoT5は「関連」「非関連」に対応するtokenを生成し、そのlogitをscoreとして読む。RankT5は分類lossではなくranking lossを導入し、順位そのものを学習目標へ近づけた。domain tuningではpositiveとhard negativeの質が重要で、古いindexから作った負例や、別版の正解文書をnegativeにすると歪む。学習前の公開modelをbaselineとして残し、label分布、query slice、model versionを固定する。改善がretrieval候補の偏りに依存していないかも確認する。

**参照論文**
- [P039] [Document Ranking with a Pretrained Sequence-to-Sequence Model](https://arxiv.org/abs/2003.06713)
- [P132] [RankT5: Fine-Tuning T5 for Text Ranking with Ranking Losses](https://arxiv.org/abs/2210.10634)

## 5.3.5. LLM reranker

LLM rerankerは自然言語で複雑な関連性を判断できるが、候補本文に埋め込まれた命令も読む。したがってcandidateをuntrusted dataとして囲み、IDだけを順位出力させ、重複・欠落・未知IDをschema validationで拒否する。Zero-shot listwise方式では候補順を反転してposition biasを測り、長いlistはsliding windowで扱う。費用と非決定性が大きいため、契約比較やmulti-hopなど難しいqueryへ限定し、通常はcross-encoderへ戻す。

**参照論文**
- [P084] [Zero-Shot Listwise Document Reranking with a Large Language Model](https://arxiv.org/abs/2305.02156)
- [P047] [Not what you've signed up for: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection](https://arxiv.org/abs/2302.12173)

## 5.3.6. 本番パイプラインの設計

本番の標準形は `retrieve N → fusion → hard-filter再確認 → rerank → select K` とする。Nを増やせば正解候補は入りやすいが、reranker callとtimeoutが増える。batching、cache、同時実行、model fallbackを設け、入力順とscoreを保存する。rerankerが落ちた場合は認可済みfirst-stage順位へ戻せるが、根拠集合が空ならLLM単体へは進まない。N・Kは固定の慣習ではなく、recall、nDCG、latency、costのParetoで決める。

**参照論文**
- [P040] [Passage Re-ranking with BERT](https://arxiv.org/abs/1901.04085)
- [P073] [RAGGED: Towards Informed Design of Retrieval Augmented Generation Systems](https://arxiv.org/abs/2403.09040)

## 5.3.8. 選定ガイド

学習labelがなければRRFと汎用cross-encoderから始め、十分なrelevance判定が蓄積したらdomain-tuned modelやRankT5を比較する。少数候補の複雑判断に限ってLLM listwiseを使う。選定はmodelの新しさではなく、rerank前後のgold rank、重要sliceの誤落ち、p95 latency、costで行う。効果が小さい場合はrerankerを大型化する前に、first-stage recall、chunk粒度、filter過剰制限を調べる。正解が候補にない問題は順位付けでは救えない。

**参照論文**
- [P040] [Passage Re-ranking with BERT](https://arxiv.org/abs/1901.04085)
- [P132] [RankT5: Fine-Tuning T5 for Text Ranking with Ranking Losses](https://arxiv.org/abs/2210.10634)
- [P084] [Zero-Shot Listwise Document Reranking with a Large Language Model](https://arxiv.org/abs/2305.02156)
