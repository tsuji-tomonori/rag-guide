# P136：HAGRID: A Human-LLM Collaborative Dataset for Generative Information-Seeking with Attribution

## 1. はじめに

本論文は、引用付き生成検索の研究に利用できる公開データセット **HAGRID** を提案する。HAGRIDは *Human-in-the-loop Attributable Generative Retrieval for Information-seeking Dataset* の略であり、新しい検索モデルそのものではなく、検索質問、関連文書、LLM生成回答、文中引用、人手評価を組み合わせたデータセットである。2023年7月にarXivへ投稿され、著者はEhsan Kamallooら5名である。([arXiv][1])

## 2. 背景

LLMを利用した生成検索では、検索結果を自然文で提示できる一方、回答が根拠文書に支持されていない、いわゆるハルシネーションが問題になる。このため、回答中の主張と根拠文書を引用によって対応付ける「attribution」が重要になる。

しかし、当時の引用付き生成検索システムの多くはブラックボックスで、学習データも非公開だった。類似研究であるALCEは既存QAデータから検索器を用いて根拠文書を推定し、回答や引用の品質も自動評価する。これに対してHAGRIDは、既に人手で関連性が判定されたMIRACLの文章を利用し、生成回答の有用性と引用の妥当性を人間が評価する点を特徴としている。([ar5iv][2])

## 3. 提案手法

HAGRIDのタスクは、質問 (Q) と、それに関連する複数の文章断片 (S) を入力し、各文末に根拠文章の番号を付けた回答 (A) を生成することである。

データ作成は次の流れで行われる。

**根拠文章の準備**
英語版MIRACLから、質問と人手で関連ありと判定されたWikipedia文章を取得する。論文ではこれらの文章を「Quotes」と呼ぶ。

**LLMによる回答生成**
質問とすべてのQuotesを `gpt-3.5-turbo-0301` に入力し、IEEE形式の `[1]`、`[2]` などの引用を含む短い回答をゼロショット生成する。指定形式に従わない出力は正規表現による後処理で除外される。

**人手アノテーション**
生成回答を文単位に分割し、以下の2基準で二値評価する。

| 基準              | 判定内容                      | 回答全体の集約規則                    |
| --------------- | ------------------------- | ---------------------------- |
| Informativeness | 質問に対して正確かつ有用な情報を提供しているか   | 1文でもinformativeなら回答全体をYes    |
| Attributability | 文中の事実的主張が引用先の文章に完全に支持されるか | 全文がattributableな場合だけ回答全体をYes |

引用がない文は、後続の引用付き文とまとめて評価される。複数の引用が付いた文では、すべての引用文書がその文を十分支持する必要がある。公開データには回答全体だけでなく、文単位のラベルも収録されている。

## 4. 実験方法

HAGRIDの主要な統計は以下の通りである。

| 項目               | Train |   Dev |
| ---------------- | ----: | ----: |
| 質問数              | 1,922 |   716 |
| 1質問当たりの平均Quotes数 |   2.7 |   2.9 |
| 生成回答数            | 3,214 | 1,318 |
| 1質問当たりの平均回答数     |   1.7 |   1.8 |
| 引用総数             | 6,577 | 3,305 |
| 1回答当たりの平均引用数     |   2.0 |   2.5 |

アノテーションには、1年以上の経験を持つ4名の専門アノテータが参加した。時給は15.2米ドルで、総作業時間は約1,400時間である。Informativenessは全回答に付与された一方、コストの大きいAttributabilityはTrainの754回答、Devの1,157回答にのみ付与された。著者によれば、Attributabilityの判定にはInformativenessの3～5倍の時間が必要だった。

モデル間の性能比較は行われていない。分析対象は主に、ラベル分布、Quotes数、回答と根拠文のBERTScore、CoverageとDensityによる抽出性、引用位置の頻度である。

## 5. 結果

人手評価の結果は以下の通りである。

| 分割    | 基準           |        Yes |       No |
| ----- | ------------ | ---------: | -------: |
| Train | Informative  | 2,704（84%） | 510（16%） |
| Train | Attributable |   547（73%） | 207（27%） |
| Dev   | Informative  | 1,179（90%） | 139（10%） |
| Dev   | Attributable |   826（71%） | 331（29%） |

したがって、GPT-3.5の回答は多くの場合質問に有用な回答を返すが、引用先が回答を完全には支持していない例が約3割存在する。

追加分析から、次の傾向が示された。

* Quotes数が増えるほど、uninformativeまたはunattributableと判定される回答が増える。
* BERTScoreによる回答と根拠文の類似度は、正例と負例の中央値の差が小さく、単純な意味類似度だけではAttributabilityを十分判定できない。
* Coverageは主に0.5～1.0に分布し、回答は根拠文の表現をかなりコピーする抽出的な傾向を持つ。
* Quotesが5件未満の場合は各文章が比較的均等に引用されるが、Quotesが5件を超えると上位3件への引用が偏る。これはLLMの位置バイアスを示唆するが、根拠文の順序を入れ替える対照実験は行われていない。([ar5iv][2])

## 6. 考察

HAGRIDの最大の長所は、検索質問、根拠文章、生成回答、引用、文単位の人手ラベルを一体化し、Apache 2.0で公開した点である。MIRACLの検索コーパスを基盤としているため、検索器と引用付き生成器を組み合わせる研究にも利用しやすい。また、自動評価だけに依存せず、引用が実際に主張を支持するかを人間が確認した点にも価値がある。([GitHub][3])

一方、データ品質の検証には重要な不足がある。4名のアノテータを用いたことは説明されているが、同じ例を複数人が評価した割合、アノテータ間一致度、意見不一致時の裁定方法は報告されていない。特にAttributabilityは判断が難しいため、κ係数などの一致度がないことは大きな弱点である。

AttributabilityのラベルはTrainの約24%にしか付いておらず、その754回答がどのように選ばれたかも明確でない。無作為抽出でなければ、73%という比率や学習データに選択バイアスが入り得る。また、Informativenessは「1文でも有用ならYes」という緩いOR条件、Attributabilityは「全ての文が支持される場合だけYes」という厳しいAND条件で集約されており、両者の割合を直接比較することはできない。

さらに、生成時には正解の関連文章だけが与えられている。実際のRAGでは無関係な検索結果や不完全な根拠が混入するため、本データだけでは検索誤りに対する頑健性を評価できない。形式違反の回答を除外した件数も報告されておらず、報告されたGPT-3.5の品質は「引用形式を守った回答」に条件付けられている可能性がある。ベースラインモデル、独立したテスト分割、検索から生成までのend-to-end評価も提示されていない。

特に注意すべきなのは、結論の数値的不整合である。Table 3ではuninformative率がTrain 16%、Dev 10%であり、全体でも
[
(510+139)/(3214+1318)=14.3%
]
となる。しかし結論では「約40%がnot informative」と記述されている。少なくとも回答レベルの集計とは一致せず、修正または別集計であることの説明が必要である。

## 7. まとめ

HAGRIDは、MIRACLの関連文章をGPT-3.5に与えて引用付き回答を生成し、その有用性と根拠への帰属可能性を人間が評価したデータセットである。約4,500件の回答と約9,900件の引用を含み、引用付きRAGや生成検索の学習・分析用資源として有用である。

主な知見は、GPT-3.5の回答の約84～90%がinformativeである一方、完全にattributableな回答は約71～73%にとどまること、根拠文章が増えるほど回答品質が低下し、引用が前方の文章に偏ることである。ただし、アノテーション一致度、部分的なラベル付与、ベースライン不足、実運用に近いノイズ文書の欠如、論文内の数値不整合には注意が必要である。

## 8. 総評

**公開上の査読状況：** arXivには2023年7月31日提出のv1のみが掲載されており、確認できる査読付き採録先や公式の採否結果は記載されていない。したがって、公開情報上はarXivプレプリントとして扱うのが適切である。([arXiv][1])

**本回答における査読判定：Weak Accept／条件付き採択（6/10）**

データセットの着眼点、公開性、文単位の人手ラベル、RAG研究への実用性は高く評価できる。一方で、アノテータ間一致度がないこと、Attributabilityラベルの選択過程が不明なこと、実験用ベースラインがないこと、正解文書のみを入力していること、結論の「40%」という重大な数値不整合が、強い採択判定を妨げる。

採択条件としては、数値不整合の修正、形式違反出力とラベル対象の抽出手順の開示、アノテータ間一致度の報告、検索ノイズを含むベースライン実験の追加が必要である。
