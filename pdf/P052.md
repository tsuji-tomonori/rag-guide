# FEVER: a Large-scale Dataset for Fact Extraction and VERification

**著者:** James Thorne, Andreas Vlachos, Christos Christodoulopoulos, Arpit Mittal
**発表:** NAACL-HLT 2018 Long Papers

## 1. はじめに

本論文は、テキスト情報源に基づく事実検証のための大規模データセット **FEVER（Fact Extraction and VERification）** を提案する。

FEVERはWikipediaから作成した **185,445件の主張**を収録し、各主張を次の3クラスに分類する。

* **SUPPORTED:** Wikipediaの記述によって支持できる
* **REFUTED:** Wikipediaの記述によって反証できる
* **NOT ENOUGH INFO（NEI）:** Wikipediaだけでは支持・反証する情報が不足している

SUPPORTEDとREFUTEDについては、ラベルだけでなく、その判断を正当化するWikipedia中の文も証拠として付与されている。したがって課題は単なる3値分類ではなく、文書検索、証拠文選択、自然言語推論を組み合わせたオープンドメインの事実検証として設計されている。([ACL Anthology][1])

## 2. 背景

従来の自然言語推論では、主張を判定するための前提文があらかじめ与えられる。一方、実際の事実検証では、大量の文書から関連文書と証拠を自動的に探す必要がある。

質問応答も関連するが、質問には通常、答えを検索するための手掛かりが含まれる。事実検証では、特に誤った主張において、反証に必要な正しい情報が主張中に含まれていない場合がある。

当時の事実検証データセットには、主に次の問題があった。

1. 主張数が数百から数千程度と小さい
2. 判断理由や証拠が機械可読な形で付与されていない
3. 検証対象文書があらかじめ与えられ、検索能力を評価できない
4. 発言者情報などからラベルを予測できても、根拠に基づく検証にはなっていない

例えばFake News Challengeは多数の主張・記事ペアを収録していたが、固有の主張数は約300件であり、検証対象の記事もシステムに与えられていた。FEVERは、主張生成から文書検索、証拠抽出、判定までを一つの課題として扱う点に新規性がある。([ACL Anthology][1])

## 3. 提案手法

### データセット構築

FEVERの構築は、**主張生成**と**主張ラベリング**の2段階で行われる。

### 3.1 主張生成

2017年6月版Wikipediaのうち、アクセス数の多いページとそのリンク先を含む約50,000ページの導入部分から文を抽出する。

アノテータは、与えられたWikipedia文から一つの事実を表す短い主張を作成する。元の文を単純に言い換えるだけでは課題が容易になるため、リンクされたエンティティのWikipedia情報も参照できる「辞書」を与え、制御された範囲で複雑な主張を作成させている。

さらに、元の主張に対して以下の変形を行う。

* 言い換え
* 否定
* 類似したエンティティ・関係への置換
* 非類似のエンティティ・関係への置換
* 一般化
* 特殊化

これらには、真偽を変えない変形と、真偽や検証可能性を変える変形の両方が含まれる。生成された主張の平均長は9.4トークンである。([ACL Anthology][1])

### 3.2 主張ラベリングと証拠付与

別のアノテータが主張をSUPPORTED、REFUTED、NEIのいずれかに分類する。SUPPORTEDとREFUTEDの場合は、判断に必要なWikipedia文も選択する。

主張生成時の元文そのものはラベリング担当者に知らされないため、単なる元文の再発見ではなく、主張に対して妥当な証拠を探索する必要がある。

証拠には、次のような複雑な例も含まれる。

* 31.75%では複数の文が証拠候補となる
* 16.82%では複数文を組み合わせる推論が必要
* 12.15%では複数のWikipediaページにまたがる証拠が必要

なお、このラベルは世界における絶対的な真偽ではなく、使用したWikipediaスナップショットに照らして支持・反証できるかを表す。そのため著者らはTRUE/FALSEではなく、SUPPORTED/REFUTEDという名称を採用している。([ACL Anthology][1])

### 3.3 ベースラインシステム

論文では、データセットの難易度を測るため、次の3段階のパイプラインを構築する。

**文書検索**
DrQAの検索器を使用し、主張とWikipedia文書のunigram・bigram TF-IDF類似度から上位 (k) 文書を検索する。

**証拠文選択**
検索された文書内の各文を主張とのTF-IDF類似度で順位付けし、上位 (l) 文を選択する。

**含意判定**
選択した証拠を連結し、主張をSUPPORTED、REFUTED、NEIに分類する。比較対象としてMLPを使用し、主要モデルにはDecomposable Attentionを採用する。

NEIには正解証拠文が存在しないため、訓練用の疑似証拠として次の2種類を比較する。

* **NEARESTP:** 検索上位ページから関連性はあるが不十分な文を取得
* **RANDOMS:** Wikipedia全体から文をランダム取得

NEARESTPは実際の検索誤りに近い難しい負例を生成する方法である。([ACL Anthology][1])

## 4. 実験方法

データは、主張生成元のWikipediaページが複数の分割にまたがらないように分割されている。これにより、同じページから作られた類似主張が訓練用と評価用の両方に入ることを防いでいる。

| 分割       | SUPPORTED | REFUTED |    NEI |      合計 |
| -------- | --------: | ------: | -----: | ------: |
| 訓練       |    80,035 |  29,775 | 35,639 | 145,449 |
| 開発       |     3,333 |   3,333 |  3,333 |   9,999 |
| テスト      |     3,333 |   3,333 |  3,333 |   9,999 |
| 共有タスク用予約 |     6,666 |   6,666 |  6,666 |  19,998 |

評価では、単純なラベル正解率に加えて、SUPPORTEDまたはREFUTEDと予測した場合、完全な正解証拠集合を取得できたときだけ正解とする厳しい指標を使用する。

証拠検索については文単位のPrecision、Recall、F1も測定する。複数文を必要とする主張では、必要な文をすべて取得しなければ完全な証拠とは認められない。

各構成要素のボトルネックを特定するため、正解文書や正解証拠を与えるオラクル実験も実施している。最終パイプラインでは文書数 (k=5)、証拠文数 (l=5) を使用する。([ACL Anthology][1])

## 5. 結果

### 5.1 アノテーション品質

主張の4%、7,506件を5人で再アノテーションした結果、ラベルのFleissのκは **0.6841** であった。

専門アノテータがWikipedia全体を詳しく調査して作成した証拠集合と比較すると、通常アノテータの証拠選択は以下の性能を示した。

* Precision: **95.42%**
* Recall: **72.36%**

著者が227例を直接確認した評価では、91.2%が正しくアノテーションされていた。主な問題はラベル自体よりも、選択された証拠が結論を十分に正当化していないことであった。([ACL Anthology][1])

### 5.2 文書・証拠検索

開発データで上位5文書を検索した場合、SUPPORTEDまたはREFUTEDの主張について、完全な証拠を含む文書群を取得できた割合は **55.30%** である。文書検索後に文選択を行うと、完全な証拠を取得できる割合はDrQA方式で **44.22%** まで低下した。

これは、TF-IDFによる文選択が関連文を絞り込む一方で、複数文推論に必要な文を除外してしまうことを示す。([ACL Anthology][1])

### 5.3 含意判定

正解証拠文を与えるオラクル設定では、Decomposable Attentionが高い分類性能を示した。

| モデル・NEI訓練方法    | 正解証拠使用時の精度 |
| -------------- | ---------: |
| MLP / NEARESTP |     65.13% |
| MLP / RANDOMS  |     73.81% |
| DA / NEARESTP  |     80.82% |
| DA / RANDOMS   |     88.00% |
| SNLIで事前学習したDA  |     38.54% |

RANDOMSはオラクル設定では高いが、ランダム文が主張と明らかに無関係であるため、NEI判定を過度に容易にしている。実際のパイプラインでは、検索器が返す「関連しているが判定には不十分な文」に対応できず、NEARESTPより低い性能となる。([ACL Anthology][1])

### 5.4 完全パイプライン

開発データでは、Decomposable AttentionとNEARESTPを組み合わせた構成が最良となった。

| 評価方法    | 開発データ精度 |
| ------- | ------: |
| ラベルのみ   |  52.09% |
| 正解証拠も必須 |  32.57% |

テストデータにおける最終結果は次の通りである。

| 指標            |         結果 |
| ------------- | ---------: |
| ラベルのみの正解率     | **50.91%** |
| 正解証拠を要求する正解率  | **31.87%** |
| 証拠検索Precision | **10.79%** |
| 証拠検索Recall    | **45.89%** |
| 証拠検索F1        | **17.47%** |

ラベルだけを評価すると約51%である一方、証拠まで要求すると約32%に低下する。したがって、最大の課題は最終的な含意分類だけではなく、完全かつ適切な証拠を検索することである。([ACL Anthology][1])

### 5.5 エラー分析

誤りを含む961件を人手で分析したところ、以下の傾向が確認された。

* 58.27%では検索器が正しい証拠を一つも取得できなかった
* 28.51%ではラベル自体は正しかったが、証拠条件を満たさなかった
* 適切な証拠が得られた場合でも、13.84%では含意判定器が誤分類した
* 21.85%では、システムが正解アノテーションに含まれない新しい有効な証拠を取得していた
* 1.35%では、人手ラベルそのものに誤りがあった

特に21.85%の新規証拠は、正解証拠集合が完全ではないことを示している。自動評価上は誤りでも、実際には妥当な証拠である場合が存在する。([ACL Anthology][1])

## 6. 考察

本論文の重要な知見は、事実検証を単純な自然言語推論問題として扱うだけでは不十分だという点である。

正解証拠を与えればDecomposable Attentionは80%以上の精度を達成する一方、検索を含む完全パイプラインでは約32%に低下する。したがって、主要なボトルネックは以下にある。

**証拠検索の再現率**
複数ページ・複数文にまたがる証拠をすべて取得する必要があり、単純な語彙類似度では不十分である。

**証拠文選択と推論の相互依存**
文選択器は分類に不要な文を減らすが、推論に必要な補助文まで除外する可能性がある。検索と推論を独立した段階として最適化することには限界がある。

**NEI負例の設計**
完全に無関係なランダム文をNEIの訓練例にすると、モデルは「無関係かどうか」を学習するだけになる。現実には、関連しているが結論を出せない証拠をNEIとして扱う必要がある。

**証拠評価の不完全性**
同じ主張を支持する証拠は複数存在し得る。アノテータが選ばなかった妥当な証拠を誤りとして扱う完全一致型の評価には限界がある。

一方、データセット自体にも制約がある。

* 主張がWikipedia文から人為的に生成されており、実際の政治的発言や誤情報とは分布が異なる
* Wikipediaの導入部を中心とした百科事典的文章に限定される
* 特定時点のWikipediaに依存し、後日の編集や知識更新を扱わない
* 一部の反証例は、テンプレート的な否定やエンティティ置換の特徴から判別できる可能性がある
* NEIには証拠が付与されないため、関連するが不十分な情報を体系的に評価しにくい

それでも、著者らが示すオラクル分析、詳細なアノテーション検証、エラー分析は、後続研究が改善すべき箇所を明確にしている。([ACL Anthology][1])

## 7. まとめ

FEVERは、185,445件の主張と文単位の証拠を収録した、大規模なオープンドメイン事実検証データセットである。

本研究の主要な貢献は次の3点に集約できる。

1. 文書検索、証拠抽出、含意判定を一体化した事実検証課題を定義した
2. ラベルだけでなく、判断を正当化する証拠文を大規模に付与した
3. オラクル実験とパイプライン実験により、証拠検索が主要なボトルネックであることを示した

最終ベースラインは、ラベルのみでは50.91%、正解証拠を必須とすると31.87%であり、データセットが十分に難しく、改善余地の大きい課題であることが確認された。([ACL Anthology][1])

## 8. 総評

### 査読結果

**判定: 採択（Strong Accept寄り）**

公開情報上、本論文は実際に **NAACL-HLT 2018のLong Paperとして採択・掲載**されている。個別の査読コメントや点数はACL Anthology上では公開されていない。([ACL Anthology][2])

独立した査読評価としても、採択が妥当である。

**高く評価できる点**

* 当時の既存データセットを大幅に上回る規模
* 証拠付き事実検証という明確で重要な課題設定
* 複数文・複数文書推論を含む実用的な難しさ
* アノテータ間一致、専門アノテータ比較、著者による確認を組み合わせた品質管理
* ベースラインだけでなくオラクル実験と人手エラー分析まで行っている
* データ、アノテーションツール、ベースラインを公開する再現性の高さ

**弱点**

* 実世界の主張ではなく、Wikipedia文から人工的に生成された主張である
* 正解証拠集合が不完全で、妥当な別証拠を誤りと判定する可能性がある
* NEIの訓練データ生成が人工的であり、モデル性能が負例生成方法に強く依存する
* 情報源がWikipediaの特定スナップショットに限定される

これらの弱点は重要だが、データセット論文としての中心的貢献を損なうものではない。特に、モデル性能の高さを示すのではなく、検索・証拠選択・推論の各段階を分解して課題の難しさを明らかにした点が優れている。事実検証研究の共通評価基盤として十分な新規性、技術的妥当性、波及性を備えた論文と評価できる。
