# 論文情報

**題目:** *BEIR: A Heterogeneous Benchmark for Zero-shot Evaluation of Information Retrieval Models*
**著者:** Nandan Thakur, Nils Reimers, Andreas Rücklé, Abhishek Srivastava, Iryna Gurevych
**種別:** 情報検索モデルを評価するためのベンチマーク論文

## 1. はじめに

本論文は、新しい検索モデルそのものではなく、情報検索モデルの**未知ドメイン・未知タスクに対する汎化性能**を評価するためのベンチマーク「BEIR」を提案している。

従来のニューラル検索研究では、MS MARCOなど特定のデータセットで学習し、同じデータセット上で評価することが多かった。しかし、実運用では学習データを用意できないドメインに検索モデルを適用する場面が多い。そこで本論文は、9種類の検索タスク、18個のデータセットをまとめ、学習済みモデルを追加の教師あり学習なしで評価するゼロショット設定を整備した。主な結論は、単一データセット上で高性能なモデルが未知ドメインでも高性能とは限らず、古典的なBM25が非常に強いベースラインになる、というものである。

---

## 2. 背景

情報検索手法は、大きく次のように分類できる。

| 方式               | 概要                                |
| ---------------- | --------------------------------- |
| 語彙検索             | BM25など。クエリと文書に共通する単語を利用           |
| ニューラル疎検索         | 単語の重み付けや文書拡張をニューラルモデルで改善          |
| 密検索              | クエリと文書をそれぞれ一つの密ベクトルに変換            |
| Late Interaction | トークン単位のベクトルを保持し、検索時に相互作用を計算       |
| 再ランキング           | BM25などで候補を取得し、Cross-Encoderで並べ替える |

BM25は高速かつ安定しているが、同義語や言い換えのように表層語が一致しない「lexical gap」に弱い。一方、BERT系の密検索は意味的な対応を扱えるものの、学習ドメインとは異なる分野でどの程度機能するかが十分に検証されていなかった。([ar5iv][1])

既存のMultiReQAやKILTも複数データセットを扱っていたが、質問応答やWikipediaに偏っており、検索タスク、文書形式、コーパス規模、専門性を横断した評価にはなっていなかった。この問題がBEIRの出発点である。

---

## 3. 提案手法

### 3.1 BEIRベンチマーク

BEIRは以下の観点からデータセットを選択している。

* 検索タスクの多様性
* ニュース、Wikipedia、医学、金融などドメインの多様性
* 十分な難易度
* 専門家、クラウドワーカー、オンラインコミュニティなど、アノテーション方式の多様性

収録されたタスクとデータセットは次の通りである。

| タスク      | データセット                               |
| -------- | ------------------------------------ |
| 生物医学検索   | TREC-COVID、NFCorpus、BioASQ           |
| 質問応答     | Natural Questions、HotpotQA、FiQA-2018 |
| ニュース検索   | TREC-NEWS、Robust04                   |
| 議論・論証検索  | ArguAna、Touché-2020                  |
| 重複質問検索   | CQADupStack、Quora                    |
| ファクトチェック | FEVER、Climate-FEVER、SciFact          |
| ツイート検索   | Signal-1M                            |
| エンティティ検索 | DBPedia                              |
| 引用予測     | SCIDOCS                              |

コーパス規模は約3,600文書から約1,500万文書まで、平均クエリ長は約3語から193語までと大きく異なる。全データセットは英語である。

### 3.2 統一フォーマット

データセットごとに異なっていた形式を、以下の三要素に統一している。

* `corpus`: 文書集合
* `queries`: 検索クエリ
* `qrels`: クエリと文書の関連度ラベル

さらに、各種検索モデルを簡単に接続できるPythonフレームワークを公開し、Precision、Recall、MAP、MRR、nDCGなどを共通のインターフェースで計算できるようにした。([ar5iv][1])

### 3.3 評価指標

主評価指標には **nDCG@10** を採用している。nDCGは検索順位を考慮でき、二値関連度と多段階関連度の双方を扱えるためである。

ただし、用途によってはRecall@100やMRRの方が重要な場合もあるため、Recall@100も付録で報告されている。

## 4. 実験方法

### 4.1 比較モデル

10種類の検索システムを、5つの方式に分類して比較している。

| 方式               | モデル                      |
| ---------------- | ------------------------ |
| 語彙検索             | BM25                     |
| 疎検索              | DeepCT、SPARTA、docT5query |
| 密検索              | DPR、ANCE、TAS-B、GenQ      |
| Late Interaction | ColBERT                  |
| 再ランキング           | BM25 + Cross-Encoder     |

BM25にはAnseriniを使用し、Cross-Encoder方式ではBM25の上位100件をMiniLMモデルで再ランキングする。ColBERTはクエリと文書をトークン単位で表現し、MaxSimに相当する相互作用によってスコアを計算する。

大部分のニューラルモデルはMS MARCOで学習され、そのまま各BEIRデータセットに適用される。したがって、基本設定は「対象データセットの教師ラベルを使わない」という意味でのゼロショット評価である。

ただしGenQは、対象コーパスからT5で人工クエリを生成し、データセットごとにTAS-Bを追加学習する。このため、GenQは厳密な意味での完全なゼロショットというより、**教師なしドメイン適応**に分類する方が適切である。

### 4.2 文書入力

Transformer系モデルについては、すべての文書の先頭512 wordpiecesだけを使用する。この統一処理によって比較しやすくなる一方、重要情報が後半にある長文書では性能が過小評価される可能性がある。

### 4.3 効率評価

DBPediaから100万文書を抽出し、検索レイテンシとインデックスサイズを測定している。CPUには8コアIntel Xeon Platinum 8168、GPUにはTesla V100を使用した。

### 4.4 アノテーションバイアスの検証

検索評価データでは、すべてのクエリ・文書ペアを人手評価することはできない。そのため、既存システムが取得した文書をプールし、その文書だけを関連度判定することが多い。

BEIRでは、このプールがBM25などの語彙検索に偏っている可能性を調べるため、TREC-COVIDで未評価の上位文書980ペアを追加で人手判定した。

## 5. 結果

### 5.1 ドメイン内性能とゼロショット性能は一致しない

MS MARCO上では、BM25のnDCG@10が0.228であるのに対し、ANCEは0.388、TAS-Bは0.408、BM25+CEは0.413を達成している。

しかし、BEIRの未知データセット全体では、これらの順位関係が維持されなかった。特定の学習データ上の改善だけでは、モデルの汎化性能を判断できないことが示された。

### 5.2 最も高性能なのは再ランキング方式

BM25+CEは18データセット中16個でBM25を上回り、BM25比の平均性能は約11%高かった。失敗したのは、MS MARCOとはタスク構造が大きく異なるArguAnaとTouché-2020である。

ColBERTも9データセットでBM25を上回った。これらの結果から、クエリと文書を完全に独立して一つのベクトルに圧縮するよりも、トークン間の細かい相互作用を残す方式の方が、未知ドメインに頑健である可能性が示唆される。

### 5.3 文書拡張は単語重み付けより頑健

DeepCTとSPARTAは、MS MARCO内ではBM25より高性能だったものの、BEIRでは平均してそれぞれBM25を大きく下回った。

一方、docT5queryは文書から想定クエリを生成し、元文書に追加することで未知ドメインの語彙を補うため、平均ではBM25をわずかに上回った。単語の重みだけを学習するより、新しい検索語を生成する方がドメイン移行に強いという結果である。

### 5.4 密検索の汎化性能には大きなばらつきがある

DPR、ANCE、TAS-Bなどの密検索モデルは、Natural QuestionsやQuoraなど一部のデータでは強いが、BioASQやTouché-2020などではBM25を大きく下回った。

密検索の中ではTAS-Bが最も強く、ANCEに14/18、DPRに17/18のデータセットで勝っている。強い教師モデルからの知識蒸留、Margin-MSE、適切な負例サンプリングが寄与したと著者らは推測している。

### 5.5 性能と計算コストには明確なトレードオフがある

100万文書での代表的な結果は以下の通りである。

| モデル     | GPUレイテンシ | CPUレイテンシ | インデックス |
| ------- | -------: | -------: | -----: |
| BM25+CE |   450 ms | 6,100 ms | 0.4 GB |
| ColBERT |   350 ms |        ― |  20 GB |
| BM25    |        ― |    20 ms | 0.4 GB |
| TAS-B   |    14 ms |   125 ms |   3 GB |
| ANCE    |    20 ms |   275 ms |   3 GB |
| SPARTA  |        ― |    20 ms |  12 GB |

再ランキングとColBERTは高性能だが遅く、密検索はGPU上で14～20 ms程度と高速である。ColBERTは文書ごとに複数のトークンベクトルを保存するため、BioASQ規模では推定約900 GBのインデックスが必要になる一方、BM25は約18 GBとされる。

### 5.6 アノテーションバイアスは評価順位を変える

TREC-COVIDで未判定文書を追加評価すると、次の変化が生じた。

| モデル        | 追加判定前 | 追加判定後 |
| ---------- | ----: | ----: |
| BM25       | 0.656 | 0.668 |
| docT5query | 0.713 | 0.714 |
| ANCE       | 0.654 | 0.735 |
| ColBERT    | 0.677 | 0.735 |
| BM25+CE    | 0.757 | 0.760 |

語彙検索はほとんど変化しない一方、ANCEは0.654から0.735、ColBERTは0.677から0.735へ大きく上昇した。元の関連度プールには語彙検索が取得しやすい文書が多く、意味検索が新たに取得した文書は、関連していても未評価のため非関連と扱われていたことを示している。

## 6. 考察

### ベンチマークの最大の価値

BEIRの最も重要な貢献は、「どの検索モデルが最も高いか」を決めたことではなく、**単一のドメイン内評価だけではモデルの実用的な頑健性を測れない**ことを、広範な実験で示した点にある。

特にBM25は、新しいニューラル方式に比べて単純であるにもかかわらず、未知タスクでは安定していた。したがって、新しい検索手法を提案する場合、MS MARCOだけでなく、BM25とのゼロショット比較が不可欠になる。

### アーキテクチャに関する因果的結論には注意が必要

本実験で比較されるモデルは、アーキテクチャだけでなく、次の要因も異なる。

* 学習データ
* Backboneモデル
* 負例サンプリング
* 損失関数
* 知識蒸留の有無
* 類似度関数
* インデックスおよび近似探索方式

したがって、「Cross-Encoderだから汎化する」「密検索だから汎化しない」と断定することは難しい。論文の結果は、統制実験によるアーキテクチャの因果比較というより、**当時利用可能だった代表的検索システムの実証的比較**として解釈すべきである。

### 評価ラベル自体もモデル非依存ではない

TREC-COVIDの再判定実験は、評価データの正解ラベルが既存検索システムの影響を受けることを示している。新しいモデルほど、従来モデルが発見しなかった文書を取得するため、「未判定＝非関連」という処理によって不利になる。

この問題は、検索モデルの改良だけでは解決できない。異なる方式から候補を集める多様なpooling、継続的な関連度判定、未判定文書を考慮する評価方法が必要である。

### BEIR自体の限界

著者らが挙げている主な制約は、全データセットが英語であること、長文書検索が十分でないこと、PageRank・新しさ・ユーザークリックなどの非テキスト信号を扱わないこと、複数フィールドを持つ文書の評価が限定的であることなどである。

また、全タスクをnDCG@10で統一することは比較には便利だが、実利用上の目的は異なる。たとえばRAGの第一段階ではRecall@100が重要であり、重複質問検索ではMRRやPrecision@1がより直接的である。したがって、単一の平均値だけでモデルを選ぶべきではない。

---

## 7. まとめ

BEIRは、18データセット、9検索タスクを統一形式に整理し、情報検索モデルのゼロショット汎化を測定可能にしたベンチマークである。

主要な知見は次の通りである。

1. MS MARCOなどのドメイン内性能は、未知ドメイン性能を予測しない。
2. BM25は未知タスクに対して非常に強く安定したベースラインである。
3. Cross-Encoder再ランキングとColBERTは高性能だが、計算コストが高い。
4. 密検索は高速だが、ドメイン・タスク変化に弱い場合がある。
5. 文書拡張は、学習された単語重み付けより汎化しやすい。
6. 検索データのアノテーションプールには語彙検索バイアスが存在し得る。

## 8. 総評

### 長所

本論文の強みは、研究上の問題設定が明確で、ベンチマーク、コード、データ形式、モデル実装を一体として公開している点にある。18データセットを単に集約するだけでなく、性能、レイテンシ、インデックスサイズ、文書長への選好、アノテーション選択バイアスまで分析している。

とりわけ、未判定文書を再評価して検索モデルの順位が変化することを示した実験は、ベンチマーク論文として重要な洞察である。

### 短所

比較モデル間で学習条件が十分に統制されていないため、方式間の性能差をアーキテクチャだけに帰属することはできない。また、英語・テキスト検索中心であり、長文書、多言語、ユーザー行動、時間性を含む実検索を全面的に表現しているわけではない。

軽微な記述上の問題として、docT5queryがBM25を上回ったデータセット数について、本文では11/18と書かれている一方、表の数値および図では12勝と読める不整合がある。中心的結論には影響しないが、修正が望ましい。

### 査読結果

**公式結果:** NeurIPS 2021 Datasets and Benchmarks Trackに採択。NeurIPSの採択論文一覧にも掲載されている。([arXiv][2])

**模擬査読判定:** **採択（Accept、8/10）**

ベンチマークとしての新規性、研究コミュニティへの実用性、再現可能性、分析の幅が高い。比較条件の交絡や評価範囲の制約はあるものの、それらは論文の中心的貢献を損なうほどではない。特定モデルの優劣を決める論文というより、検索モデル研究の評価方法そのものを改善する基盤研究として高く評価できる。
