# P098 — MillWheel: Fault-Tolerant Stream Processing at Internet Scale

**著者:** Tyler Akidauほか、Google
**掲載:** *Proceedings of the VLDB Endowment*, Vol. 6, No. 11, 2013
**対象:** 大規模・低遅延・状態保持型ストリーム処理基盤

## 1. はじめに

MillWheelは、Google内部で利用されていた分散ストリーム処理フレームワークである。利用者は処理を有向グラフとして記述し、各ノードにアプリケーションロジックを実装する。データ配送、永続状態、障害復旧、負荷分散、重複排除などの分散システム固有の処理はフレームワーク側が担当する。

本論文の中心的な主張は、以下の三要素を一つのプログラミングモデルで実現できるという点にある。

* レコード単位の低遅延処理
* 状態を持つ処理に対する耐障害性とExactly-onceセマンティクス
* Out-of-orderデータを扱うための論理時刻とLow Watermark

ここでいうExactly-onceは、ネットワーク上でレコードが物理的に一度しか送信されないという意味ではない。配送自体は再試行され得るが、重複排除と原子的な状態更新によって、**ユーザーコードからは一度だけ処理されたように見える**という保証である。また、この保証はMillWheel内部に限定され、外部システムへの副作用については利用者が冪等性を確保する必要がある。

## 2. 背景

従来の大規模バッチ処理では、MapReduceのようなモデルによって分散実行の複雑さを隠蔽できた。一方、ストリーム処理では低遅延だけでなく、長期間維持される状態、マシン障害、データの遅延・順序逆転、重複配送を同時に扱う必要がある。

論文では、Google Zeitgeistの検索トレンド検出を動機例としている。このパイプラインでは、検索クエリを短い時間窓ごとに集計し、過去のモデルと比較して急増または急減を検出する。

急増の検出は、現時点の件数が既に予測値を超えていれば早期に出力できる。一方、急減の検出では、件数が少ない原因が「本当に検索されなかった」のか、「ネットワークや上流処理で遅延しているだけ」なのかを区別しなければならない。そこで、あるイベント時刻以前のデータが概ね到着済みであることを示すLow Watermarkが必要になる。

このユースケースから、論文はストリーム処理基盤に次の要件を設定している。

* 入力から出力までにシステム固有のバッチ境界を設けない
* ユーザーコードから耐障害性のある永続状態を利用できる
* Out-of-orderデータを自然に処理できる
* 単調増加するLow Watermarkを提供する
* システム規模を拡大しても平均遅延をほぼ一定に保つ
* レコード処理にExactly-onceセマンティクスを与える

特に課金処理や広告関連処理では、重複が金額の不整合に直結するため、フレームワークレベルの重複排除が重要とされている。

## 3. 提案手法

### 3.1 プログラミングモデル

MillWheelの処理は、ユーザー定義の**Computation**をノードとする有向グラフとして表現される。Computation間のデータ配送経路が**Stream**である。

各レコードは概念的に次の三つ組で表される。

[
(\text{key},\ \text{value},\ \text{timestamp})
]

`value`は任意のバイト列であり、`timestamp`は通常、イベントが発生した時刻を表す。`key`は状態管理と並列化の単位である。同一キーに対する処理は直列化され、異なるキーは複数マシン上で並列処理できる。これにより、ユーザーは明示的なロックや並行制御を実装する必要がない。

キー抽出関数は送信側ではなく、Streamを購読する**受信側**が指定する。同じ入力Streamに対し、あるComputationは検索語をキーにし、別のComputationはCookie IDや地域をキーにするといった再パーティショニングが可能である。

### 3.2 永続状態

各Computationはキーごとの永続状態を持てる。状態はBigtableやSpannerのような複製された永続ストレージに保存される。

状態の用途には、時間窓ごとのカウンタ、異常検知モデル、ストリーミングJoinのためのバッファなどがある。ユーザーは状態のシリアライズ形式を定義するが、保存、復旧、一貫性管理はMillWheelが担当する。

### 3.3 Low Watermark

Computation (A) のLow Watermarkは、概念的には次の最小値として定義される。

[
LW(A)=\min \left(
\text{A内の最古の未完了処理時刻},
{LW(C)\mid C\text{がAへ出力する}}
\right)
]

ここで未完了処理には、実行中、保存済み、配送待ち、転送中のレコードなどが含まれる。Low Watermarkが時刻 (t) を通過した場合、利用者は原則として「時刻 (t) 以前について、既知の未処理データは残っていない」と判断できる。

外部入力側が将来の到着を完全には予測できないため、Watermarkより古いLate Dataが到着する可能性は残る。Zeitgeistではそのようなデータを破棄して計測し、別のパイプラインでは過去の集計結果を訂正する。論文のZeitgeist事例では、破棄されるLate Dataは経験的に約0.001%だったと報告されている。

### 3.4 Timer

Timerはキー単位で登録され、以下のいずれかを契機に実行される。

* Wall-clock timeが指定時刻に達する
* Low Watermarkが指定イベント時刻を通過する

Timerは永続化され、マシン障害後も復旧できる。同一キーのTimerは時刻順に発火し、入力レコードと同様のExactly-onceセマンティクスで実行される。

例えば時間窓の終了時刻にWatermark Timerを設定すれば、その窓のデータが揃ったと判断できる段階で集計結果を出力できる。

### 3.5 Exactly-once処理

入力レコードを受信すると、MillWheelは概ね次の順序で処理する。

1. レコードIDを用いて重複を確認する
2. ユーザーの`ProcessRecord`を実行する
3. 状態、Timer、出力レコード、重複排除情報を永続ストレージへコミットする
4. 上流へACKを返す
5. 下流へ出力レコードを配送する

送信側はACKを受け取るまで配送を再試行するため、物理的な配送はAt-least-onceである。受信側は一意なレコードIDによって再送を検出し、重複処理を防ぐ。高速化のため、既知のレコードIDをBloom Filterで確認し、必要な場合だけ永続ストレージを参照する。

### 3.6 Strong ProductionとWeak Production

**Strong Production**では、下流へ出力するレコードを、状態更新と同じ原子的書き込みで事前にチェックポイントする。障害後には保存された出力を再送できるため、状態だけ更新された、または出力だけ送信されたという不整合を防げる。

一方、元から冪等な処理では、この保証は過剰な場合がある。その場合は**Weak Production**を選択し、永続化より先に下流へ楽観的に送信できる。ただし、下流のACKが遅れるとパイプライン全体が連鎖的に待たされるため、MillWheelは長時間残った出力だけを選択的にチェックポイントする。

### 3.7 Zombie Writer対策

キー範囲が障害や負荷分散によって別のワーカーへ移動した後、旧ワーカーが遅延した書き込みを行うと状態が破壊される可能性がある。

MillWheelは各キー範囲に**Sequencer Token**を付与し、所有権が移動した時点で旧Tokenを無効化する。永続ストレージは有効なTokenを持つ書き込みだけを受理するため、論理的に各キーを更新できるワーカーを一つに限定できる。これは単なるトランザクションだけでは防げない、Zombie Writerやネットワーク上に残った書き込みへの対策である。

### 3.8 システム構成

複製されたMasterが各Computationのキー空間を辞書順の区間に分割し、ワーカーへ割り当てる。CPU負荷やメモリ使用量に応じて区間を移動、分割、統合することで動的負荷分散を行う。

キーごとのユーザー状態、Timer、未配送出力は、永続ストレージ上の同一行に保存される。キー範囲が別ワーカーへ移動すると、新ワーカーは保存されたメタデータを走査してTimer Heapや出力Queueを復元する。

Low Watermarkはグローバルな管理機構で集約される。各ワーカーが未完了レコード、保存状態、Timer、未配送出力の最小時刻を報告し、中央管理機構がパイプライン全体のWatermarkを計算する。実装上は複数マシンへShardでき、論文では50万のキー区間まで性能低下なしに扱えたとしている。また、正確性を緩和して99%のレコード進捗を表す近似Watermarkを計算する機能も備える。

## 4. 実験方法

評価では、単一の総合ベンチマークではなく、MillWheelの主要な構成要素を個別に測定している。

**出力遅延評価**では、数値をバケット化・ソートする単段パイプラインを使用した。前後のComputationでキーが異なるMany-to-many Shuffleを発生させ、200 CPU上でレコード配送遅延を測定した。Exactly-onceとStrong Productionを無効にした構成と、両方を有効にした構成を比較している。

**スケーラビリティ評価**では、CPU数を20から2000まで増加させ、入力負荷もCPU数に比例して増加させた。これにより、システム規模の拡大に対してレコード遅延が増えるかを調べた。

**Watermark Lag評価**では、200 CPU上の3段パイプラインを使用し、各段のLow Watermarkを1秒ごとに取得した。

**キャッシュ評価**では、ワーカーごとのキャッシュサイズを変化させ、MillWheelワーカーと永続ストレージを合わせたCPU負荷を測定した。企業機密上、CPU負荷は絶対値ではなく正規化された値で報告されている。

## 5. 結果

| 評価項目                                    |                   結果 |
| --------------------------------------- | -------------------: |
| Exactly-once・Strong Production無効時の中央値遅延 |               3.6 ms |
| 同構成の95パーセンタイル遅延                         |                30 ms |
| Exactly-once・Strong Production有効時の中央値遅延 |              33.7 ms |
| 同構成の95パーセンタイル遅延                         |              93.8 ms |
| スケール評価                                  | 20～2000 CPUで中央値はほぼ一定 |
| Stage 1のWatermark Lag                   | 平均1795 ms、標準偏差159 ms |
| Stage 2のWatermark Lag                   | 平均1954 ms、標準偏差127 ms |
| Stage 3のWatermark Lag                   | 平均2081 ms、標準偏差140 ms |
| キャッシュ効果                                 |       CPU負荷を約2分の1に削減 |

Exactly-onceとStrong Productionを有効にすると、中央値は約9.4倍、95パーセンタイルは約3.1倍に増加する。すなわち、高い整合性保証には明確なレイテンシコストが存在する。一方、それでも95パーセンタイルは100 ms未満に収まっている。

CPU数を2000まで増加させても中央値はほぼ一定であり、入力負荷をリソースに比例して増加させる範囲では良好なScale-out特性を示した。ただし、99パーセンタイルは規模とともに悪化し、およそ100 ms程度まで増加した。

Watermark Lagは最初の段で約1.8秒あり、後続段では1段当たり200 ms未満ずつ増加した。フレームワークキャッシュは約550 MBまで効果が増加し、十分なキャッシュ容量ではワーカーとストレージを合わせたCPU負荷を半減させた。

実運用例として、広告データのStreaming Join、課金パイプライン、異常検知、ネットワークスイッチやクラスタの監視、Street View向け画像処理などが挙げられている。

## 6. 考察

### 優れている点

最大の貢献は、状態、イベント時刻、Timer、Exactly-once、動的負荷分散を、統一されたキー単位の処理モデルにまとめた点である。特に、以下の設計が強い。

**第一に、状態と出力を同じ整合性境界に含めている。**
状態更新だけでなく、Timer、重複排除情報、未配送出力まで原子的に保存するため、障害復旧後もアプリケーションレベルの整合性を維持できる。

**第二に、Low Watermarkを単なる入力上のマーカーではなく、システム内の未完了処理全体から計算している。**
転送中のレコード、保存済みデータ、Timer、未配送出力まで考慮するため、時間窓処理の正しさをフレームワークレベルで説明できる。

**第三に、保証を選択可能にしている。**
Strong ProductionとExactly-onceを常に強制せず、冪等な処理では無効化できる。実験結果からも、この選択がレイテンシに大きく影響することが示されている。

**第四に、実運用上の問題を正面から扱っている。**
Zombie Writer、遅延ACK、Hot Key、Late Data、Watermarkの停滞など、実際の大規模分散システムで発生する問題が具体的に議論されている。

### 制約・弱点

最も大きな弱点は、タイトルでFault-Tolerantを強調しているにもかかわらず、**明示的な障害注入実験が報告されていない**点である。マシン停止時の復旧時間、障害中のレイテンシ、状態サイズと復旧時間の関係、複数障害時の挙動などは評価されていない。

また、Throughput、入力レート、レコードサイズ、ネットワーク帯域、マシン構成などが十分に示されておらず、他システムとの定量比較もない。このため、性能評価の再現性は低い。

Exactly-once保証もMillWheel内部に限定される。外部DBへの書き込み、通知送信、課金API呼び出しなどをComputationから直接行う場合、その副作用のExactly-once性はユーザー側で確保しなければならない。

キー単位の並列化にも限界がある。特定キーに入力が集中するHot Keyでは、一つのワーカーがボトルネックになる。論文自身も、90%のトラフィックが一つのキーへ集中すれば、その負荷を一つのマシンが処理することになると述べている。

さらに、極端に遅れたデータや停止した入力源があるとLow Watermarkが進まず、時間窓の状態を解放できない。結果としてバッファやメモリ使用量が増大する。99% Watermarkはこの問題を緩和するが、完全性を犠牲にする近似的な解決策である。

## 7. まとめ

MillWheelは、レコード単位の低遅延ストリーム処理に対して、次の機能を統合したシステムである。

[
\text{Keyed State}
+\text{Event Time}
+\text{Watermark}
+\text{Persistent Timer}
+\text{Exactly-once Processing}
]

At-least-onceの再送、レコードIDによる重複排除、状態と出力の原子的チェックポイント、Sequencerによる単一Writer制御を組み合わせることで、障害や再送が発生してもユーザーコードからは一度だけ処理されたように見える。

実験では、Exactly-onceを有効にした場合でも100 ms未満の95パーセンタイル遅延を示し、20～2000 CPUの範囲で中央値遅延をほぼ一定に保った。ただし、障害復旧性能、Throughput、他システムとの比較、再現可能な実験条件については不足している。

## 8. 総評

### 査読結果

**実際の採否:** 採択。
本論文は*Proceedings of the VLDB Endowment*, Vol. 6, No. 11に掲載され、VLDB 2013での発表対象となっている。公開資料からは、個別査読者のスコアや査読コメントまでは確認できない。

**再査読判定:** **Accept（4 / 5）**

採択相当と判断する理由は、ストリーム処理における状態、論理時刻、耐障害性、Exactly-onceを一つの実用的な設計へ統合し、Google内部の複数サービスで運用されていることを示した点にある。特に、Low Watermarkと永続Timerをキー単位の状態管理および障害復旧機構へ組み込んだ設計は、論文の技術的価値が高い部分である。

一方で、Fault Toleranceを主要貢献とする論文としては、障害注入実験、復旧時間、可用性、Throughput、競合システムとの比較が不足している。したがってStrong AcceptではなくAcceptが妥当である。システム設計論文としての新規性と実用性は高いが、実験的検証の網羅性には改善の余地がある。
