# P084：Zero-Shot Listwise Document Reranking with a Large Language Model

**著者**：Xueguang Ma, Xinyu Zhang, Ronak Pradeep, Jimmy Lin
**発表年**：2023年
**邦題**：大規模言語モデルによるゼロショット・リストワイズ文書再ランキング

## 1. はじめに

本論文は、大規模言語モデルを文書検索の**リランキング器**として直接利用する手法、**LRL（Listwise Reranker with a Large Language Model）**を提案している。

従来のリランキングでは、各文書をクエリと個別に組み合わせて関連度スコアを計算する。これに対してLRLは、クエリと複数の候補文書を一度にLLMへ入力し、文書IDの並べ替え結果を直接生成させる。

中心的な研究仮説は次のとおりである。

> 各文書を独立に評価するよりも、複数文書を同時に比較させた方が、LLMは文書間の相対的な関連性を正確に判断できる。

タスク固有の関連度ラベルやファインチューニングを使用せず、プロンプトだけで高性能なリランキングを実現する点が本研究の特徴である。

---

## 2. 背景

一般的な情報検索システムは、次のような多段構成を採用する。

1. BM25やDense Retrieverが大量の文書から候補を取得する。
2. より高精度なモデルが候補文書を再評価する。
3. 上位文書を最終結果として返す。

従来のリランカーの多くは、クエリ (q) と文書 (D_i) から関連度

[
s_i=f(q,D_i)
]

を個別に計算し、(s_i) の降順に文書を並べる**ポイントワイズ方式**である。

この方式では、ある文書を評価するときに他の候補文書を参照できない。したがって、各文書が「単独で関連しているか」は判断できても、「候補集合の中でどれが最も関連しているか」という相対比較を直接行っていない。

教師ありランキングでは、pairwise lossやlistwise lossにより文書間の相対関係を学習できる。しかし、そのためには大量のクエリ–文書関連度ラベルが必要となる。

本論文以前の代表的なゼロショット方式であるUPRは、文書を条件としてクエリが生成される尤度を計算するが、これも各文書を独立に評価するポイントワイズ方式である。LRLは、LLMの自己注意機構を利用し、推論時に複数文書を同時比較させることでこの問題に対処する。

なお、本手法における「listwise」は、listwise lossを使ってモデルを学習するという意味ではなく、**推論時の入出力形式がリスト単位である**ことを指す。

---

## 3. 提案手法

### 3.1 Listwise Reranker with an LLM

LRLでは、複数の候補パッセージとクエリを以下のようなプロンプトに埋め込む。

```text
Passage1 = {passage_1}
...
Passage10 = {passage_10}

Query = {query}

Passages = [Passage1, ..., Passage10]
Sort the Passages by their relevance to the Query.
Sorted Passages = [
```

LLMには、次のような文書IDの順序を生成させる。

```text
Passage3, Passage1, Passage7, ...
```

先頭ほどクエリへの関連度が高いと解釈する。

`Sorted Passages = [` の角括弧を閉じずに入力を終えることで、LLMがその続きをランキング結果として生成しやすくしている。関連度の数値スコアを計算せず、最終的な順序を直接出力する点が重要である。

### 3.2 ポイントワイズ比較手法PRL

listwise化そのものの効果を確認するため、著者らは同じGPT-3をポイントワイズ方式で使用するPRLも構築した。

```text
Passage: {passage}
Query: {query}
Is this passage relevant to the query?
Please answer True/False.
Answer:
```

最初の出力トークンが`True`となる確率を関連度スコアとして使用し、文書を並べる。

この比較により、

* PRL：LLMを使うが文書は個別評価
* LRL：LLMを使い文書を集合として比較

という違いを検証している。

### 3.3 Progressive Reranking

GPT-3の入力長には上限があり、100文書を一度に入力することはできない。そのため、LRLではスライディングウィンドウ型の**Progressive Reranking**を導入する。

実験では以下の設定である。

* ウィンドウサイズ：10文書
* 移動幅：5文書
* 候補リストの末尾から先頭方向へ処理

まず末尾の10文書を並べ替え、次にウィンドウを5文書分だけ先頭側へ移動して再度並べ替える。この処理をリスト先頭まで繰り返す。

関連文書を段階的に上位へ「昇格」させるアルゴリズムである。ただし、リスト全体を厳密にソートするアルゴリズムではなく、上位順位の改善に特化した近似処理である。

また、効率を高める構成として、UPRまたはPRLで候補を先に絞り込み、その上位10件または20件だけをLRLで再ランキングする多段構成も評価している。

## 4. 実験方法

### データセット

英語検索では、TREC Deep Learning Passage Rankingの3データセットを使用している。

* DL19
* DL20
* DL21

DL19とDL20は約880万パッセージのMS MARCO v1、DL21は約1億3,800万パッセージのMS MARCO v2に基づく。

多言語評価では、18言語を含むMIRACLから以下の3言語を選択している。

* 中国語
* スワヒリ語
* ヨルバ語

中国語は比較的高資源、スワヒリ語とヨルバ語は低資源言語として選ばれている。

### 使用モデル

TREC DLでは、PRLとLRLの双方に`text-davinci-003`を使用する。

MIRACLでは、非英語テキストが過剰にトークン化される傾向を考慮し、より長い入力を扱える`code-davinci-002`を使用する。1回のLRL入力に含める候補文書数は10件であり、必要に応じてパッセージを切り詰めている。

### 比較手法

ゼロショット手法として以下を比較している。

* BM25
* Contriever／mContriever
* UPR
* PRL
* LRL

参考値として、MS MARCOなどで学習した教師ありモデルも掲載している。

* DPR
* TCT-ColBERT
* MonoBERT
* MonoELECTRA
* mDPR

### 評価指標

TREC DLでは、

* nDCG@10
* MRR@10

を使用する。

MIRACLではnDCG@10を使用する。

## 5. 結果

### 5.1 TREC Deep Learning

主要なnDCG@10の結果は以下のとおりである。

| 手法               |       DL19 |       DL20 |       DL21 |
| ---------------- | ---------: | ---------: | ---------: |
| BM25             |     0.5058 |     0.4796 |     0.4458 |
| UPR：BM25上位100件   |     0.5910 |     0.5958 |     0.5621 |
| PRL：BM25上位100件   |     0.5975 |     0.6088 |     0.5678 |
| LRL：BM25上位100件   | **0.6580** | **0.6224** | **0.5996** |
| 各データセットで最良のLRL構成 | **0.6650** | **0.6364** | **0.6260** |

LRLは、BM25上位100件を直接再ランキングする設定で、UPRとPRLをすべてのデータセット上で上回った。

同一構成である「BM25上位100件を再ランキング」で平均すると、

* LRL：0.6267
* UPR：0.5830
* PRL：0.5914

であり、LRLの改善量はUPR比で約4.4ポイント、PRL比で約3.5ポイントである。

一方、データセットごとに最良のLRL構成を選択した場合、UPRとの差は平均約6.0ポイントとなる。論文中の「UPRより約6ポイント高い」という表現は、直接比較する単一のLRL構成ではなく、各データセットで良い構成を選んだ場合に対応する。

### 5.2 最終段LRLの効果

PRLの上位20件をLRLで再ランキングすると、元のPRLに対し平均で、

* nDCG@10：約5.1ポイント向上
* MRR@10：約7.1ポイント向上

となった。

すなわち、候補100件すべてにProgressive Rerankingを適用しなくても、ポイントワイズモデルで絞り込んだ上位20件にLRLを適用することで、大部分の効果を得られることが示されている。

### 5.3 MIRACL

MIRACLのnDCG@10は以下のとおりである。

| 手法              |       中国語 |     スワヒリ語 |      ヨルバ語 |
| --------------- | --------: | --------: | --------: |
| BM25            |     0.180 |     0.383 |     0.406 |
| mContriever     |     0.212 |     0.368 |     0.187 |
| Hybrid          |     0.307 |     0.510 |     0.422 |
| Hybrid上位20件＋LRL | **0.344** | **0.542** | **0.533** |

LRLは入力元であるHybrid検索器に対して、それぞれ0.037、0.032、0.111改善しており、平均改善量は約0.060である。特にヨルバ語で大きな改善が得られている。

論文はBM25に対する平均約15ポイントの改善を強調している。ただしLRLが実際に再ランキングしているのはBM25単体ではなく、BM25とmContrieverを組み合わせたHybridの結果である。したがって、LRL自身の限界改善を測るにはHybridとの差である平均約6ポイントを見る方が適切である。

## 6. 考察

### 評価できる点

第一に、提案が非常に単純でありながら、従来のゼロショットリランカーを明確に上回っている。LLMを関連度分類器として使うのではなく、**順序を生成するモデルとして使う**という問題設定は自然である。

第二に、PRLとLRLで同じGPT-3を使用しており、少なくとも「高性能なLLMを使ったから改善した」という要因と、「listwiseにしたから改善した」という要因をある程度切り分けている。

第三に、LRLを候補100件全体に適用するだけでなく、ポイントワイズリランカーの上位10件または20件に適用する構成を示した点は実用的である。

第四に、中国語だけでなく低資源言語でも改善が得られており、プロンプトベースのリランキングが多言語検索に利用できる可能性を示している。

### 問題点

**統計的検定がない。**
TREC DLはクエリ数が比較的少ない評価セットであるが、信頼区間や有意差検定が報告されていない。そのため、特にDL20のような差が小さいケースで、改善が安定しているか判断しにくい。

**入力順序への依存を検証していない。**
listwise LLMは候補の提示順序に影響される可能性がある。しかし、候補をランダムに並べ替える実験、複数回実行による一貫性評価、逆順入力などが実施されていない。

**Progressive Rerankingが初期順位に強く依存する。**
処理は末尾から先頭への一方向であり、リスト全体の厳密な整合性を保証しない。ウィンドウサイズや移動幅に関するアブレーションもない。

**生成結果の妥当性が報告されていない。**
LLMが文書IDを重複させる、候補を欠落させる、存在しないIDを出力するなどの失敗率と、その修復方法が記述されていない。

**再現性が弱い。**
使用したモデルは外部API型のGPT-3であり、モデル内部や学習データが非公開である。また、temperature、top-p、停止条件、最大出力長などの生成設定、具体的な文書切り詰め規則が十分に示されていない。

**効率性の主張が定量化されていない。**
上位10件・20件だけをLRLで処理すれば効率化できることは直感的だが、API呼び出し回数、入力トークン数、金銭コスト、レイテンシの測定がない。

**データ汚染の問題が残る。**
MS MARCOやTRECの文書・クエリがGPT-3の事前学習データに含まれていた可能性を排除できない。著者自身もこの問題を認めており、「zero-shot」という表現には留保が必要としている。

**多言語一般化の根拠は限定的である。**
MIRACLの18言語中3言語だけを評価しており、英語実験とは異なるGPT-3モデルを使用している。そのため、改善が言語一般化によるものか、モデル変更によるものかを完全には切り分けられない。

---

## 7. まとめ

本論文は、クエリと複数の候補文書をLLMへ同時入力し、関連度順の文書ID列を直接生成させるLRLを提案した。

主要な知見は以下である。

* listwiseなLLMリランキングはpointwiseなLLMリランキングより高精度である。
* BM25上位100件の再ランキングでUPRとPRLを上回る。
* PRLやUPRの上位10～20件だけを処理する最終段リランカーとしても有効である。
* 中国語、スワヒリ語、ヨルバ語でもゼロショット改善を示す。
* 一方で、順序バイアス、統計的有意性、再現性、コスト、データ汚染に関する検証が不足している。

アイデアは簡潔かつ先駆的だが、論文としての実証は探索的な段階にとどまっている。

**査読判定：Weak Reject（不採択寄り）**

LLMによるlistwise rerankingという中心的アイデアは明快で、情報検索分野における重要な研究方向を提示している。特に、同じGPT-3を用いたPRLとの比較で一貫した改善を示した点は説得力がある。

一方、採択判断には次の不足が大きい。

* 統計的有意差検定がない
* 入力順序やプロンプトへの頑健性評価がない
* Progressive Rerankingのアブレーションがない
* APIコストとレイテンシが報告されていない
* 出力エラー率と修復処理が不明
* GPT-3の学習データ汚染を排除できない
* 多言語評価が3言語に限定されている
* 一部の改善値が、異なるLRL構成の事後的な最良値や、前段検索器の改善を含んでいる

したがって、着想は採択水準に近いものの、当該バージョンの実験的裏付けだけでは不十分と判断する。

なお、論文本文の著者注記には、本研究が2022年末に実施され、**ACL 2023ではRejectとなった**ことが明記されている。
