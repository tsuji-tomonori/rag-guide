# 1. はじめに

**論文名:** *KILT: a Benchmark for Knowledge Intensive Language Tasks*
**著者:** Fabio Petroni ほか
**採録先:** NAACL-HLT 2021

本論文は、外部知識を必要とする自然言語処理タスクを、単一のWikipediaスナップショット、共通データ形式、共通評価方法の下に統合したベンチマーク **KILT（Knowledge Intensive Language Tasks）** を提案する。新しい単独モデルの提案というよりも、検索・推論・生成を横断的に研究するための評価基盤を構築した論文である。([ACL Anthology][1])

# 2. 背景

オープンドメイン質問応答、ファクトチェック、エンティティリンキング、スロットフィリング、知識対話などでは、入力文だけでなく、大規模な外部知識源から必要な情報を検索する必要がある。

しかし従来のデータセットは、異なるWikipediaの版、異なる文書形式、異なる入出力仕様、異なる評価指標を採用していた。このため、タスクごとにWikipediaを再処理・再索引化する必要があり、同一の検索器や知識表現が複数タスクで再利用できるかを公平に比較しにくかった。

さらに、従来評価では最終回答の正誤だけが重視されることが多く、モデルがその回答を裏づける文書を取得できたかは必ずしも評価されない。KILTは、この問題を「回答」と「根拠、すなわちprovenance」の同時評価として定式化する。

# 3. 提案手法

## 3.1 共通知識源

KILTでは、**2019年8月1日時点のWikipedia英語版、約590万記事**を、全タスク共通の知識源として使用する。各データセットが参照していた元のWikipedia記事を、この固定スナップショット上の記事と対応づける。

対応づけは次のように行われる。

1. Wikipediaのリダイレクト情報を用いて記事を特定する。
2. 元データの証拠スパンと最もBLEUが高いスパンを、対象記事全体から探索する。
3. 元の証拠を、KILT版Wikipedia上のスパンに置き換える。
4. 開発・テストデータでは、BLEUが0.5未満となる対応を除外する。

この処理により、エンティティリンキングを除く開発・テストデータの平均約18%が除外された。一方、訓練データはすべて保持される。

## 3.2 タスクとデータセット

KILTは5種類のタスク、合計11データセットを統合する。

| タスク          | データセット                                   |
| ------------ | ---------------------------------------- |
| ファクトチェック     | FEVER                                    |
| エンティティリンキング  | AIDA CoNLL-YAGO、WNED-WIKI、WNED-CWEB      |
| スロットフィリング    | T-REx、Zero Shot RE                       |
| オープンドメイン質問応答 | Natural Questions、HotpotQA、TriviaQA、ELI5 |
| 知識対話         | Wizard of Wikipedia                      |

入力は、主張、質問、構造化クエリ、文章中のエンティティ、対話履歴など多様であり、出力も分類ラベル、エンティティ名、短い回答、長文回答、対話応答を含む。

すべてのデータはJSON Lines形式で表され、各レコードは概ね次の要素を持つ。

* `id`
* 自然言語形式の`input`
* 一つ以上の正解`output`
* 各出力を支持するWikipedia記事・スパンの`provenance`

したがって、モデルの内部構造に依存せず、「テキストを入力してテキストと根拠を出力する」共通インターフェースが実現される。

## 3.3 根拠情報の追加アノテーション

Natural QuestionsとELI5については、既存の根拠情報が十分でないため、Amazon Mechanical Turkを用いた追加アノテーションを実施した。

候補スパンはTF-IDF、DPR、RAG、BLINK＋flairなどで検索され、各スパンを3人が評価し、2人以上が「回答に十分な証拠を含む」と判断した場合に正解根拠へ追加される。評価者間一致度はCohenのκでNatural Questionsが0.3、ELI5が0.1と低く、特に長文質問応答における「十分な証拠」の曖昧さが表れている。

## 3.4 評価指標

KILTでは三つの側面を評価する。

**下流タスク性能**では、分類・エンティティ出力にAccuracy、短い質問応答にExact Match、ELI5にROUGE-L、Wizard of WikipediaにF1を用いる。

**検索性能**では、正解となるWikipediaページ集合を取得できたかを、ページ単位のR-PrecisionとRecall@kで評価する。

**KILTスコア**では、正解根拠集合が検索順位の最上位に完全に含まれ、すなわちR-Precisionが1である場合にのみ、Accuracy、EM、ROUGE-L、F1の得点を与える。正しい回答だけでなく、回答を支持する根拠も取得できなければ得点にならない。

# 4. 実験方法

データ規模は、訓練約313万件、開発51,460件、テスト50,736件である。テストデータの正解は公開せず、EvalAI上で予測を提出して評価する方式が採られた。

比較対象は以下の三系統である。

**タスク固有モデル**として、FEVER用NSMN、エンティティリンキング用BLINK、質問応答・スロットフィリング用DPR＋BERT reader、対話用Transformer Memory Networkを使用する。

**パラメトリック知識のみを使う汎用モデル**として、BARTとT5を検索なしで各タスクにファインチューニングする。ここでは、事前学習時にモデルパラメータへ記憶された知識だけで回答する。

**明示的検索を使う汎用モデル**として、BART＋DPRとRAGを使用する。BART＋DPRでは上位3パッセージを入力へ連結する。RAGはDPR検索器とBART生成器を組み合わせ、検索クエリエンコーダまで誤差逆伝播し、上位5パッセージを使って生成する。

検索器単体の比較には、TF-IDF、DPR、BLINK＋flair、複数KILTデータセットで共同学習したMulti-task DPRを用いる。Multi-task DPRでは、巨大なT-RExだけが学習を支配しないよう、各データセットを一様に再サンプリングしている。

# 5. 結果

## 5.1 明示的な検索の効果

多くのタスクで、検索を使用するBART＋DPRやRAGが、検索なしのBART・T5を上回った。

代表例は以下の通りである。

| データセット              |      主指標 |                   代表的な結果 |
| ------------------- | -------: | -----------------------: |
| FEVER               | Accuracy | BART＋DPR 86.74、RAG 86.31 |
| T-REx               | Accuracy |                RAG 59.20 |
| Zero Shot RE        |       EM |                RAG 44.74 |
| Natural Questions   |       EM |                RAG 44.39 |
| HotpotQA            |       EM |                RAG 26.97 |
| TriviaQA            |       EM |                RAG 71.27 |
| Wizard of Wikipedia |       F1 |           BART＋DPR 15.19 |

特にZero Shot REでは、検索なしBARTの9.14に対してRAGは44.74となり、外部知識検索が大きく寄与している。

ただし、明示的検索が常に有効なわけではない。長文回答のELI5では、検索なしBARTのROUGE-Lが20.55であるのに対し、BART＋DPRは17.41、RAGは14.05であった。検索結果に適切に条件づけて長文を生成することの難しさが示されている。

## 5.2 検索方式にはタスク依存性がある

エンティティリンキングではBLINK＋flairが強く、質問応答やファクトチェックではDPR系が強いなど、全タスクで最良となる検索器は存在しなかった。

一方、Multi-task DPRは、通常のDPRに対して11データセットすべてでR-Precisionを改善した。例えばFEVERでは55.33から74.48、T-RExでは13.26から69.46、Zero Shot REでは28.96から80.91へ向上している。これは異なる知識タスク間で検索学習を共有できることを示す。

## 5.3 回答性能と根拠提示性能の隔たり

KILTスコアは通常の下流指標より大幅に低い。

* FEVERのRAGはAccuracy 86.31に対してKILT-Accuracy 53.45
* HotpotQAのRAGはEM 26.97に対してKILT-EM 3.21
* Wizard of WikipediaのRAGはF1 13.11に対してKILT-F1 8.75
* ELI5のBART＋DPRはROUGE-L 17.41に対してKILT-ROUGE-L 1.90

モデルは正しい回答を出せても、その回答を完全に裏づける根拠集合を上位に検索できていない。

# 6. 考察

## 6.1 評価できる点

最大の貢献は、検索・生成・分類・エンティティ同定・対話という異質なタスクを、同一知識源と同一provenance形式へ統合した点である。これにより、タスクごとに別のWikipediaインデックスを構築する必要がなくなり、タスク横断的な検索器や知識表現を実験できる。

また、最終回答だけでなく、その根拠まで要求するKILTスコアは重要である。単なる正答率では区別できない「正しく答え、かつ正しい情報源を示す能力」を明示的に評価対象とした点は、本論文の中核的な新規性である。

データ、評価コード、モデル用ライブラリ、非公開テストによるリーダーボードを提供しており、再現性と研究基盤としての有用性も高い。

## 6.2 限界

第一に、KILTはすべての問題がWikipedia内で回答可能な**in-KB設定**であり、情報が存在しない問題や「回答不能」の判定を扱わない。FEVERでも根拠を持たないNot Enough Information事例が除外されている。実運用では知識源に答えがない場合も多いため、現実的な検索・生成システムの評価としては限定的である。

第二に、データセット変換によって元タスクの性質が変化している。開発・テストデータの約18%が対応失敗として除外され、Zero Shot REでは負例が削除され、Wizard of Wikipediaでは元の候補知識を与えない完全オープンドメイン設定へ変更されている。したがって、元データセット上の既存結果との直接比較には注意が必要である。

第三に、正解provenanceは必ずしも網羅的ではない。Wikipediaには同じ事実を説明する複数の記事があり、別の妥当なページを検索したモデルが不正解扱いになる可能性がある。著者自身も、根拠が別システムから後付けされているかを判定できず、提示された根拠が実際にモデルの回答生成に使われたという因果的忠実性までは保証できないと認めている。

第四に、NQとELI5のアノテーション一致度が低い。加えて、アノテーション対象の候補スパンを既存検索器で選んでいるため、それらの検索器が発見しやすい証拠に正解集合が偏る可能性がある。後者は論文の手順から導かれる評価上の懸念である。

最後に、ELI5のROUGE-Lや対話の単語F1は、長文回答の事実性、完全性、自然さを十分には測れない。ページ単位の根拠判定も、ページ中のどの記述が回答を支持するかを評価するには粗い。

# 7. まとめ

KILTは、5種類の知識集約型NLPタスク、11データセットを、一つのWikipediaスナップショットと共通インターフェースへ統合したベンチマークである。

実験では、DPRなどの密検索とBART系生成モデルを組み合わせる構成が、多くのタスクで検索なしモデルや一部のタスク固有モデルを上回った。特にMulti-task DPRの結果は、複数タスクを共有して検索器を学習する有効性を示した。

一方、通常の回答指標とKILTスコアの大きな差から、正しい答えを生成することと、その答えを適切な証拠で裏づけることは別の難しい問題であることが明確になった。

# 8. 総評

**独立査読判定: Accept（採択）**
**総合評価: 8.5 / 10**

新規モデルの性能向上だけを示す論文ではなく、分断されていた知識集約型NLP研究を統合し、回答と根拠を同時評価する研究基盤を提供した点を高く評価する。データ規模、タスクの多様性、ベースラインの広さ、コード・評価環境の公開も十分であり、ベンチマーク論文として採択に値する。

主要な懸念は、正解根拠の非網羅性、低いアノテーション一致度、静的なWikipediaかつ回答可能問題のみという設定、データ変換による元タスクからの乖離である。ただし、これらは主張全体を無効化するものではなく、むしろ後続研究が改善すべき課題を明確にしている。

実際の採録結果も、**NAACL-HLT 2021本会議への採択・掲載**である。([ACL Anthology][1])
