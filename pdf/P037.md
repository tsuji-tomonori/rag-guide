# 1. はじめに

**LongLLMLingua: Accelerating and Enhancing LLMs in Long Context Scenarios via Prompt Compression** は、長いプロンプトをそのままLLMに入力するのではなく、質問への回答に必要な情報を中心に圧縮することで、性能・推論コスト・遅延を同時に改善する研究である。

単に入力長を短くするのではなく、質問関連情報の密度を上げ、重要情報をLLMが認識しやすい位置へ移動させる点が特徴である。本論文はACL 2024の本会議Long Papersに採択され、論文集の1658–1677頁に収録されている。([ACL Anthology][1])

# 2. 背景

長文コンテキストを扱うLLMには、主に三つの問題がある。

第一に、入力トークン数が増えるほどAPI料金や計算量、推論時間が増加する。第二に、関連性の低い文章が大量に含まれると、重要情報の密度が低下し、回答性能が悪化する。第三に、LLMは長文の中央付近にある情報を見落としやすい、いわゆる **Lost in the Middle** または位置バイアスを示す。

既存手法のLLMLinguaは、小規模言語モデルで各トークンのパープレキシティを計算し、予測しやすいトークンを削除する。しかし、質問を考慮しない純粋な情報量ベースの圧縮では、質問と無関係だが珍しいトークンを残し、質問に必要だが一般的な表現を削除する可能性がある。

LongLLMLinguaは、この問題を「質問との関連性を考慮した圧縮」によって解決する。

# 3. 提案手法

LongLLMLinguaは、LLMLinguaを基盤とし、次の四つの主要機構を追加している。処理の流れは、文書単位の選別と並べ替え、トークン単位の圧縮、対象LLMによる回答生成、回答中の破損表現の復元である。

## 質問対応型の粗粒度圧縮

各文書について、文書自体のパープレキシティではなく、

[
p(\text{質問}\mid\text{文書})
]

に相当する条件付き尤度を計算する。すなわち、「この文書を読んだ小規模言語モデルが、質問文をどれだけ自然に生成できるか」を文書と質問の関連度として利用する。

さらに質問の後ろに、「この質問の答えは与えられた文書から得られる」という制約文を追加する。これは、小規模モデルが文書と無関係な知識から質問を説明してしまうことを抑える正則化として機能する。得られたスコアにより文書を順位付けし、重要度の低い文書を削除する。

## 質問対応型の細粒度圧縮

残された文書内の各トークンについて、通常のパープレキシティではなく、次の**対照パープレキシティ**を使用する。

[
s_i =
\operatorname{PPL}(x_i\mid x_{<i})
----------------------------------

\operatorname{PPL}(x_i\mid q,x_{<i})
]

質問 (q) を条件として与えたときに予測しやすくなるトークンほど、高い関連度を持つと判断する。この尺度は条件付きPointwise Mutual Informationに対応すると説明されている。

実験では、通常のパープレキシティは無関係文書にも高い値が分散する一方、対照パープレキシティは正解文書付近に集中している。

## 文書の並べ替え

粗粒度圧縮で得られた質問関連度を使い、重要な文書をLLMが認識しやすい位置に移動する。

これは、長文中央に置かれた情報の認識率が下がる位置バイアスを、モデルの再学習ではなく入力順序の変更によって緩和する方法である。

## 動的圧縮率

すべての文書を同じ割合で圧縮するのではなく、質問との関連度が高い文書には多くのトークン予算を割り当てる。

したがって、重要文書は比較的弱く圧縮され、補助的な文書は強く圧縮される。粗粒度の文書ランキングと細粒度のトークン削減を接続する役割を持つ。

## 部分列復元

トークン単位の圧縮では、人名や年号が破損することがある。例えば、論文中では「Wilhelm Conrad Röntgen」が圧縮後に「Wilhelmgen」となり、LLMがその破損した表記をそのまま出力する例が示されている。

そこで、LLMの出力中に圧縮プロンプト由来の部分列が現れた場合、元の非圧縮プロンプト内の対応部分列を検索し、元表記に置換する。これにより固有名詞や数値の正確性を改善する。

# 4. 実験方法

対象LLMとしてGPT-3.5-Turbo-0613、長い入力にはGPT-3.5-Turbo-16k-0613、およびLongChat-13B-16kを使用した。圧縮を担当する小規模モデルはLLaMA-2-7B-Chatである。生成はgreedy decoding、temperature 0に固定され、反復的トークン圧縮のセグメント長は200に設定された。

評価データは以下の五種類である。

* NaturalQuestions：20文書から正解を探す複数文書QA。正解文書を1、5、10、15、20番目に配置して位置バイアスを評価。
* LongBench：単一文書QA、複数文書QA、要約、few-shot学習、人工タスク、コード補完を含む16タスク。
* ZeroSCROLLS：要約、QA、感情分類、情報並べ替えを含む10タスク。
* MuSiQue：複数文書をまたぐマルチホップQA。
* LooGLE：約3万トークン規模の長距離依存QA。

比較手法にはBM25、Gzip、Sentence-BERT、OpenAI Embeddingなどの検索型手法と、Selective Context、LLMLinguaなどのプロンプト圧縮手法が含まれる。

遅延評価はV100 32GB上で実施され、圧縮処理と対象LLMへのリクエスト時間の両方を含むエンドツーエンド時間が測定された。

# 5. 結果

| ベンチマーク                    |               元プロンプト |               LongLLMLingua |
| ------------------------- | -------------------: | --------------------------: |
| NaturalQuestions、正解文書10番目 |  精度54.1、2,946 tokens |   精度71.2、748 tokens、約3.9倍圧縮 |
| LongBench                 | 平均44.0、10,295 tokens | 平均48.3、1,822 tokens、2.6倍高速化 |
| ZeroSCROLLS               |  平均32.5、9,788 tokens | 平均32.7、1,826 tokens、2.3倍高速化 |
| MuSiQue                   | F1 45.8、2,427 tokens |        F1 51.2、1,077 tokens |
| LooGLE                    | 平均22.6、30,546 tokens |  平均32.1、3,121 tokens、約10倍圧縮 |

NaturalQuestionsでは、正解文書が中央付近にある場合の改善が特に大きい。正解文書が10番目の場合、元プロンプトの54.1から71.2へ17.1ポイント向上した。文書再配置を使用した結果は75.5である。また、約4倍圧縮時のエンドツーエンド速度は2.0倍になった。

LongBenchでは、約6倍に圧縮しながら平均スコアが44.0から48.3へ上昇した。単に計算量を減らすだけでなく、不要情報の除去がタスク性能の向上にも寄与している。

LooGLEでは、約3万トークンを約3,100トークンに圧縮し、平均スコアを22.6から32.1に向上させた。論文当時のGPT-3.5-Turbo価格に基づくAPI費用では、1,000件当たり93.6ドルから5.6ドルとなり、94.0%の削減と報告されている。

アブレーションでは、質問対応型の粗粒度圧縮を除去した場合の性能低下が最も大きい。NaturalQuestionsの2倍圧縮条件では、完全モデルの70点台に対し、質問対応を除くとおおむね40点前後まで低下した。細粒度圧縮、動的圧縮率、部分列復元、文書再配置も、それぞれ性能向上に寄与している。

# 6. 考察

本研究の最大の長所は、長文処理の問題を単なる「入力長」の問題として扱わず、**情報密度、情報位置、文字列表記の破損**という三つの問題に分解した点である。特に、質問を条件とする文書・トークン重要度は、一般的な検索モデルとは異なる形で、小規模言語モデルの生成確率を直接利用している。

また、対象LLMを再学習せず、ブラックボックスAPIにも適用できる。既存のRAGに前処理として組み込みやすく、実用上の価値は高い。

一方、次の問題が残る。

第一に、質問ごとに圧縮結果が変わるため、同じ文書集合に複数の質問を行う場合でも再圧縮が必要である。文書表現をキャッシュできず、圧縮計算量はLLMLinguaの約2倍になると著者自身が指摘している。複雑で間接的な文書間関係では、粗粒度の質問関連度が重要文書を見落とす可能性もある。

第二に、評価対象はGPT-3.5-Turbo、LongChat-13B、およびLLaMA-2系圧縮モデルに限られる。異なるトークナイザー、より強い長文モデル、非英語データに対する一般化は、この実験からは確定できない。

第三に、費用削減表はGPT-3.5-Turboの入出力API料金を対象としており、LLaMA-2-7Bによる圧縮処理のGPU費用は金額換算されていない。遅延評価には圧縮時間が含まれているが、金銭的コストについては完全なシステム総費用ではない。

第四に、部分列復元は、回答が圧縮プロンプトの表現をコピーする場合には有効だが、強く言い換えた回答には適用しにくい。また、誤った元部分列を復元する可能性について、独立した適合率や誤置換率の評価は示されていない。

最後に、論文が主張するNaturalQuestionsでの「最大21.4%向上」は比較条件に注意が必要である。表1の同一位置・再配置なしの比較では54.1から71.2への**17.1ポイント向上**であり、21.4という差は54.1と再配置後の75.5を比較した値に対応すると読める。再配置の有無を含む比較条件を、より明示することが望ましい。

# 7. まとめ

LongLLMLinguaは、質問に関連する文書とトークンを優先して残すプロンプト圧縮手法である。

中心的な構成は、質問対応型の粗粒度・細粒度圧縮、位置バイアスを緩和する文書再配置、重要文書に多くの予算を与える動的圧縮率、破損した固有名詞や数値を戻す部分列復元である。

実験では、約4～10倍の圧縮を行いながら、複数の長文ベンチマークで元プロンプトと同等以上の性能を示し、最大2.6倍のエンドツーエンド高速化と、大幅なAPI料金削減を報告した。特に、質問対応型の粗粒度圧縮が性能向上の主要因である。

# 8. 総評

**査読結果として公開されている採否：ACL 2024 Main Conference、Long Papers採択。** ([ACL Anthology][1])

**模擬査読判定：Accept、7/10、確信度4/5。**

質問対応型の対照パープレキシティ、文書再配置、動的圧縮率を統合した設計は明快で、実験範囲も広い。性能だけでなく、トークン数、API料金、圧縮時間を含む遅延を評価している点も高く評価できる。

一方、質問ごとの再圧縮、圧縮モデルの計算費用、比較条件の一部の曖昧さ、使用モデルの限定、部分列復元の誤置換分析不足は弱点である。それでも、長文LLMをモデル側から改造するのではなく、入力側の情報配置と密度を最適化するという実用的な貢献は大きく、採択相当と判断する。
