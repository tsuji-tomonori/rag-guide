## 1. はじめに

**RAGChecker**は、RAGシステムの最終回答だけでなく、検索器と生成器を分離して「どこで、どのような失敗が起きたか」を診断する評価フレームワークである。回答を文書全体ではなく、検証可能な**主張（claim）単位**に分解して評価する点が中心的なアイデアである。

本論文は後に **NeurIPS 2024 Datasets and Benchmarks Track**へ採択され、Posterとして発表されている。([NeurIPS Proceedings][1])

## 2. 背景

従来のRAG評価には、主に三つの問題がある。

第一に、RAGは検索器と生成器からなるため、回答が悪かった場合に「必要な情報を検索できなかった」のか、「検索できた情報を生成器が使えなかった」のかを区別しにくい。

第二に、Recall@kやMRRは正解チャンクのアノテーションやチャンク分割方法に依存する。一方、BLEU、ROUGE、BERTScoreなどは、長文回答に含まれる個々の正しい主張、誤った主張、欠落した主張を十分に識別できない。

第三に、既存の自動評価指標が、人間の正しさ・完全性の判断と本当に整合するかが十分検証されていなかった。

## 3. 提案手法

RAGCheckerは、質問 (q)、文書集合 (D)、正解回答 (gt)、検索されたチャンク、生成回答を入力とする。

まず、正解回答と生成回答をそれぞれ原子的なclaimへ分解する。次に、各claimが正解回答、検索チャンク、生成回答のいずれに含意されるかを判定する。このclaim抽出と含意判定を組み合わせ、以下の指標を計算する。

| 対象  | 指標                           | 意味                               |
| --- | ---------------------------- | -------------------------------- |
| 全体  | Claim Precision              | 生成したclaimのうち正しいものの割合             |
| 全体  | Claim Recall                 | 正解claimのうち回答に含まれたものの割合           |
| 全体  | F1                           | PrecisionとRecallの調和平均            |
| 検索器 | Claim Recall                 | 正解claimのうち検索チャンクに含まれる割合          |
| 検索器 | Context Precision            | 検索チャンクのうち、少なくとも一つの正解claimを含む割合   |
| 生成器 | Context Utilization          | 検索できた正解claimを、実際の回答で利用した割合       |
| 生成器 | Faithfulness                 | 回答claimのうち検索コンテキストに支持される割合       |
| 生成器 | Relevant Noise Sensitivity   | 正解情報を含むチャンク内のノイズを誤って回答へ取り込む割合    |
| 生成器 | Irrelevant Noise Sensitivity | 無関係なチャンクの情報を誤って回答へ取り込む割合         |
| 生成器 | Hallucination                | 誤ったclaimのうち、検索コンテキストにも存在しないものの割合 |
| 生成器 | Self-knowledge               | 正しいが検索コンテキストには存在しないclaimの割合      |

Self-knowledgeは常に悪いわけではないが、「回答は検索根拠のみに依存すべき」という厳格なRAG設定では低い方が望ましいと定義されている。

重要なのは、**FaithfulnessとCorrectnessを分離している点**である。検索コンテキスト自体に誤情報やノイズが含まれている場合、生成器はコンテキストに忠実でありながら、誤った回答を出す可能性がある。RAGCheckerはこれをNoise Sensitivityとして捉える。

## 4. 実験方法

評価対象は、二つの検索器と四つの生成器を組み合わせた8種類のRAGシステムである。

* 検索器：BM25、E5-Mistral
* 生成器：GPT-4、Llama 3 8B、Llama 3 70B、Mixtral 8×7B

評価ベンチマークは、Wikipedia、Novel、Writing、Biomedical、Finance、Lifestyle、Recreation、Science、Technology、AI Scienceの10領域、合計4,162問からなる。既存の公開QAデータセットを再利用し、短答形式だった一部の領域ではGPT-4を使って長文正解回答を作成している。

標準設定では、文書を300トークン、オーバーラップ率0.2で分割し、上位20チャンクを生成器へ渡す。生成温度は0、最大生成長は2,048トークンである。Claim抽出器と含意判定器には、RefChecker上のLlama 3 70B Instructを使用している。

さらに、自動評価指標自体を評価するメタ評価を行っている。8システムから得た回答をペア比較し、10領域×28システムペアの280事例を作成した。各事例を2名のアノテータが、正しさ、完全性、総合評価について−2から2の5段階で判定した。比較対象にはRAGAS、TruLens、ARES、CRUD-RAG、BLEU、ROUGE-L、BERTScoreなどが含まれる。

## 5. 結果

### 人間評価との相関

RAGCheckerの人間評価との相関は、総合評価でPearson 0.619、Spearman 0.609となり、比較手法より全体として高い結果だった。正しさでは0.497／0.470、完全性では0.607／0.581である。人間同士の総合評価相関は0.701／0.689であり、自動評価と人間の間にはまだ差が残る。2名のアノテータの判断差が1以内となる一致率は90.95%だった。

ただし、全6相関値でRAGCheckerが最高というわけではない。完全性のSpearman相関では、RAGASの0.614がRAGCheckerの0.581を上回っている。したがって、「すべての観点・相関尺度で最良」という解釈は避けるべきである。

### 8種類のRAGシステム

最高の総合F1は **E5-Mistral＋GPT-4の52.7**だった。BM25＋GPT-4は50.3であり、生成器を固定した場合でも、BM25からE5-Mistralへの変更によってRecallやF1が一貫して改善した。

また、Llama 3 70Bは8Bよりも、Context Utilization、Noise Sensitivity、Hallucinationを含むほぼすべての生成器指標で優れていた。E5-Mistral＋Llama 3 70Bでは、Faithfulnessが95.9、Hallucinationが3.3だった。

主要な観察結果は次の通りである。

* 良い検索器は必要情報のRecallを高め、全体性能を改善する。
* 生成器のContext Utilizationが全体F1と強く関係する。
* 検索Recallが上がると、有益な情報だけでなく同一チャンク内のノイズも増える。
* GPT-4はオープンモデルよりも、正しい情報とノイズを区別する能力が高かった。
* オープンモデルはFaithfulnessが高い一方、検索コンテキストを過度に信用する傾向があった。

### RAG設定の変更

検索チャンク数を5から20へ増やすと、Claim Recallは61.5から77.6、Faithfulnessは88.1から92.2、F1は51.7から53.4へ向上した。一方、Noise Sensitivityも34.0から35.4へ悪化した。

FaithfulnessやContext Utilizationを明示的に要求するプロンプトでは、Faithfulnessが92.2から93.6、Context Utilizationが59.2から63.7へ改善したが、Noise Sensitivityも35.4から38.1へ増加した。チャンクのオーバーラップ率はContext Precisionをわずかに改善したものの、他の指標への効果は小さかった。

## 6. 考察

本研究の最大の強みは、単一の総合点を出すだけでなく、改善すべき箇所を具体化できる点にある。例えば、Claim Recallが低ければ検索器、Claim Recallは高いがContext Utilizationが低ければ生成器、Faithfulnessは高いがNoise Sensitivityも高ければ検索チャンクの品質や生成器の情報選別能力が問題だと判断できる。

一方、以下の制約がある。

第一に、完全な正解回答を必要とするため、正解が一意でない質問、最新情報を扱う質問、実運用上の自由回答をreference-freeで評価する用途にはそのまま適用しにくい。

第二に、claim抽出と含意判定をLlama 3 70Bに依存する。したがって、最終スコアには抽出漏れ、含意判定ミス、モデル固有のバイアスが含まれる可能性がある。

第三に、正解回答の一部をGPT-4で長文化しているため、評価対象となる回答との文体や内容粒度の相性が結果に影響する可能性がある。

第四に、メタ評価は280ペア、各2名のアノテータに基づく。主結果の表には信頼区間や明示的な有意差検定が示されておらず、「有意に優れる」という主張の統計的な強さは判断しにくい。

著者自身も、検索器指標が情報密度・多様性・一貫性を扱えていないこと、NeutralとContradictionを区別していないこと、ベンチマークが英語のテキストデータに限定されていることを制約として挙げている。

## 7. まとめ

RAGCheckerは、RAG回答をclaim単位に分解し、検索器については「必要情報を取得できたか」、生成器については「取得情報を使えたか」「ノイズを取り込んだか」「根拠なく生成したか」を個別に評価する。

人間評価との相関は既存手法より概ね高く、特にRAGの構成変更やプロンプト調整を診断する用途で有用である。一方、完全な正解回答とLLMベースの判定器を必要とするため、絶対的な品質判定器というより、**条件を固定したRAG開発における診断ダッシュボード**として利用するのが適切である。

**本要約における査読判定：Accept相当**

個々の指標すべてが新しいというより、claim-level entailmentを軸に、全体・検索器・生成器を一貫した体系で評価した統合設計に価値がある。人間評価とのメタ評価、複数領域・複数RAG構成での検証、実際のパラメータ調整に結び付く診断結果も評価できる。

ただし、正解回答への依存、LLM判定器への依存、限定的な人手評価規模、統計的検定の不足は重要な弱点である。それでも、研究上の明確さと実務的有用性がこれらを上回っており、RAG評価研究として採択に値する論文と判断する。
