# P105 Adaptive-RAG: Learning to Adapt Retrieval-Augmented Large Language Models through Question Complexity

**著者:** Soyeong Jeongほか
**採録:** NAACL 2024 Long Papers, pp. 7036–7050 ([ACL Anthology][1])

## 1. はじめに

本論文は、すべての質問に同じRAG処理を適用する「one-size-fits-all」の非効率性を問題としている。単純な質問に多段検索を行うと計算資源を浪費する一方、複雑な質問を検索なし、または1回の検索だけで処理すると回答精度が低下する。

そこで、質問の複雑さを事前に分類し、質問ごとに「検索なし」「単一ステップ検索」「複数ステップ検索」を切り替える **Adaptive-RAG** を提案する。重要な点は、単に「検索するか否か」を判断するのではなく、**必要な検索・推論の深さまで選択する**ことである。([arXiv][2])

## 2. 背景

検索拡張生成では、LLMのパラメトリック知識だけで回答する方法に加え、外部文書を検索して回答生成に利用する方法が用いられる。しかし、質問の性質によって適切な処理は異なる。

| 処理方式      | 特徴         | 主な問題               |
| --------- | ---------- | ------------------ |
| 検索なし      | 最も高速       | 最新情報、固有知識、複雑な質問に弱い |
| 単一ステップRAG | 質問を用いて1回検索 | 複数文書を結合する質問には不十分   |
| 複数ステップRAG | 検索と推論を反復   | 高精度だが計算コストが大きい     |

従来のAdaptive Retrievalは、質問内エンティティの出現頻度などから「検索するか否か」を決定する。Self-RAGも検索の必要性をモデル自身に判定させるが、本論文では、これらは主として検索の有無を扱うため、複数文書を必要とするマルチホップ質問への対応が不十分だと指摘している。一方、IRCoTのような多段手法を全質問に適用すると、単純な質問にも反復検索が発生する。([arXiv][2])

## 3. 提案手法

Adaptive-RAGは、質問 (q) を最初に小型言語モデルへ入力し、次の3クラスのいずれかに分類する。

| クラス      | 質問の判定          | 実行する戦略    |
| -------- | -------------- | --------- |
| A: No    | LLM内部知識のみで回答可能 | 検索なし      |
| B: One   | 外部知識が必要だが比較的単純 | 単一ステップRAG |
| C: Multi | 複数文書と逐次推論が必要   | 複数ステップRAG |

複数ステップ部分にはIRCoTを利用し、LLMによる中間推論とBM25検索を交互に実行する。ルータ以外のLLMや検索器の構造・パラメータを変更する必要がないため、既存のRAGパイプラインに追加しやすい構成である。([arXiv][2])

### 複雑度ラベルの自動生成

質問複雑度を人手で付与したデータセットが存在しないため、著者らは2種類の弱教師データを自動生成する。

第一に、同一質問を検索なし、単一ステップ、複数ステップの3方式で解き、**正解できた最も単純な方式**をラベルとする。例えば、検索なしで正解できればA、検索なしは失敗したが単一ステップで正解できればBとなる。

第二に、すべての方式が失敗してラベルを決められない質問には、データセットの作成上の性質を利用する。単一ホップデータセットの質問にはB、マルチホップデータセットの質問にはCを付与する。これらの自動生成ラベルを用いて、小型言語モデルを交差エントロピー損失で学習する。([arXiv][2])

## 4. 実験方法

評価には、単一ホップとマルチホップを合わせた6種類のオープンドメインQAデータセットを使用している。

単一ホップはSQuAD v1.1、Natural Questions、TriviaQA、マルチホップはMuSiQue、HotpotQA、2WikiMultiHopQAである。回答生成モデルにはFLAN-T5-XL（3B）、FLAN-T5-XXL（11B）、GPT-3.5-turbo-instructを使用した。検索器は全方式でBM25に統一され、単一ホップ用とマルチホップ用に前処理済みWikipediaコーパスが使用された。([arXiv][2])

複雑度分類器はT5-Largeで、学習率は (3\times10^{-5})、最適化手法はAdamWである。モデル予測結果から作るsilver labelと、単一ホップ／マルチホップというデータセットの帰納的バイアスを組み合わせて学習する。評価指標にはEM、F1、回答中に正解を含むかを測るAccuracyに加え、検索・生成ステップ数と単一ステップRAGを1とした相対実行時間が用いられた。([arXiv][2])

比較対象は、検索なし、常時単一ステップ、Adaptive Retrieval、Self-RAG、常時複数ステップ、および完全な複雑度分類を仮定したOracle版Adaptive-RAGである。計算には80GBのA100 GPUが使用されたが、計算コスト上の理由から実験は単一runで実施されている。([arXiv][2])

## 5. 結果

全6データセットの平均結果は以下の通りである。Timeは単一ステップRAGを1とした相対時間である。

| ベースLLM      | Adaptive-RAG F1 | 常時複数ステップ F1 | Adaptive-RAG Time | 常時複数ステップ Time |
| ----------- | --------------: | ----------: | ----------------: | ------------: |
| FLAN-T5-XL  |           46.94 |       48.85 |              3.60 |          8.81 |
| FLAN-T5-XXL |           48.62 |       50.09 |              2.00 |          3.80 |
| GPT-3.5     |       **50.91** |       50.87 |          **1.46** |          3.33 |

GPT-3.5では、Adaptive-RAGは常時複数ステップ方式とほぼ同等以上のF1を維持しながら、相対実行時間を約56%削減している。平均ステップ数も1.03回で、常時複数ステップ方式の2.81回より約63%少ない。FLAN-T5でもF1の低下を1.5～1.9ポイント程度に抑えながら、実行時間を約47～59%削減している。([arXiv][2])

データセット別では、FLAN-T5-XLを用いたMuSiQueでAdaptive-RAGのF1は31.80、常時複数ステップは31.90となり、ほぼ同等の精度で時間を7.58から6.61へ削減した。HotpotQAでは53.82対56.54、2WikiMultiHopQAでは49.75対58.85であり、特に2WikiMultiHopQAでは効率向上と引き換えに比較的大きな性能差が残っている。([arXiv][2])

分類器の全体精度は54.52%であり、クラス別ではAが30.52%、Bが66.28%、Cが65.45%だった。検索不要な質問の約47%を単一ステップ、約22%を複数ステップに誤分類しているため、不要な検索を十分には排除できていない。([arXiv][2])

一方、Oracle分類器を使用すると、GPT-3.5のF1は50.91から62.80へ向上し、相対時間も1.46から1.03へ低下した。この大きな差から、Adaptive-RAG全体の主要なボトルネックは、各RAG方式そのものではなく、質問複雑度を推定するルータにあることが分かる。([arXiv][2])

## 6. 考察

本研究の最大の長所は、RAGの精度だけでなく、**質問ごとに投入する計算量を変える問題としてRAGを定式化した点**にある。既存LLM、検索器、マルチステップ推論器をそのまま利用できるモジュール型の設計であり、精度とレイテンシのPareto改善を実験的に示している。6データセット、3種類の回答生成モデルで評価している点も、単一設定だけの結果より説得力が高い。

ただし、「質問複雑度」という用語には注意が必要である。分類ラベルは質問の言語的・論理的複雑さを直接測定したものではなく、特定のLLM、BM25、Wikipediaコーパスの組み合わせにおいて、**正解できる最小コストの処理方式**から生成されている。したがって分類器が学習しているのは、純粋な質問難易度というよりも、現在のシステム構成に対する戦略適合度だと解釈できる。LLMや検索器、文書コーパスを変更した際にラベルの意味が変化する可能性がある。([arXiv][2])

また、ラベル生成に単一ホップ／マルチホップというデータセット種別を利用しているため、未知ドメインへの適用ではデータセット固有のショートカットを学習する懸念がある。著者ら自身も、自動ラベルには誤りが含まれ得ること、Oracleとの性能差が大きいことを限界として挙げている。([arXiv][2])

比較実験にも制約がある。Self-RAGは異なるベースLLMで学習されており、論文も方式間の数値を完全に直接比較できないと認めている。さらに、検索器はBM25、対象はWikipediaベースの短答式QAに限定されているため、dense retrieval、hybrid search、企業内のノイズを含む文書、長文回答などへの一般化は未検証である。単一runのみであるため、分類器学習や生成結果の分散も評価できていない。([arXiv][2])

## 7. まとめ

Adaptive-RAGは、質問を事前に3段階へ分類し、検索なし、単一ステップRAG、複数ステップRAGを切り替えるルーティング方式である。常に複雑なRAGを実行する方式と比較して、大幅に計算時間を削減しつつ、平均的なQA性能をおおむね維持している。

本論文の本質的な貢献は、新しい検索アルゴリズムそのものではなく、**検索・推論パイプラインを質問単位で選択する計算資源配分の枠組み**を明確に提示したことにある。今後の性能向上には、ルータの教師データ、ドメイン外汎化、より連続的な計算量制御の改善が重要となる。

## 8. 総評

**査読結果: Accept（採択相当）**

実用上重要な「精度と計算コストの両立」を明確な問題として設定し、単純かつモジュール性の高い解決策を提示している。6データセットと複数のLLMを用いた実験から、常時複数ステップ方式に近い精度を、より低い計算コストで実現できることが示されており、研究上・実装上の価値は高い。

一方で、複雑度ラベルの循環性、分類精度の低さ、ベースLLMが異なる比較、単一run、BM25とWikipedia QAに限定された評価は弱点である。特に「複雑度分類器」というより「最小十分戦略のルータ」であることを、論文中でより厳密に位置付ける必要がある。それでも、Oracle実験によって研究仮説と今後の改善余地が明瞭に示されており、弱点は採択を覆すものではない。
