# P006 音声ガイド台本

対象: Leveraging Passage Retrieval with Generative Models for Open Domain Question Answering  
著者: Gautier Izacard, Edouard Grave / EACL 2021  
読み上げ: ずんだもん（VOICEVOX想定）  
想定尺: 約6-8分

## 原稿

こんにちは、ずんだもんなのだ。今回は、EACL 2021 の論文、Leveraging Passage Retrieval with Generative Models for Open Domain Question Answering を紹介するのだ。略して FiD、Fusion-in-Decoder の論文なのだ。

この論文のテーマは、オープンドメイン質問応答なのだ。質問に答えるとき、モデルの内部知識だけに頼るのではなく、Wikipedia のような大きな文書集合から関連 passage を検索し、その文書を読んで回答を生成する。いまの RAG にかなり近い問題設定なのだ。

当時の悩みは、検索で取ってきた複数の文書を、生成モデルへどう渡すかだったのだ。全部を一つに連結すると入力が長くなり、Transformer の計算が重くなる。かといって一つずつ読むだけでは、複数文書にまたがる手がかりを合わせにくい。FiD はこの問題に、かなりシンプルな設計で答えたのだ。

FiD の手順はこうなのだ。まず、質問に対して BM25 や DPR で上位 K 件の passage を検索する。次に、それぞれの passage を、質問、文書タイトル、本文という形で T5 のエンコーダに入れる。ただし、ここで大事なのは、各 passage をまとめて一つの長文にするのではなく、個別にエンコードすることなのだ。

そして、各 passage から得られたエンコーダ出力を連結し、デコーダがその全体へクロスアテンションしながら回答を生成する。つまり、文書ごとの読み取りはエンコーダで別々に行い、情報の融合はデコーダ側で行う。これが Fusion-in-Decoder という名前の意味なのだ。

この設計が効く理由は二つあるのだ。一つ目は、各 passage を独立に処理するので、長い入力を一気に自己注意へ入れるより扱いやすいこと。二つ目は、回答を作るデコーダが、すべての passage の表現を横断して参照できることなのだ。だから、一つの文書だけでは弱い手がかりでも、複数の検索結果を合わせて答えに近づけるのだ。

実験では、Natural Questions、TriviaQA、SQuAD Open が使われているのだ。生成モデルは T5-base、約2億2千万パラメータと、T5-large、約7億7千万パラメータ。多くの設定で上位100件の passage を使い、評価は Exact Match、つまり生成回答が正解文字列と一致したかで見るのだ。

結果は強力なのだ。FiD-large、100 passages の設定では、Natural Questions で 51.4、TriviaQA Open で 67.6、TriviaQA Hidden で 80.1、SQuAD Open で 56.7 の EM を出しているのだ。当時の RAG や DPR と比べても、NQ や TriviaQA では大きく上回る。巨大な closed-book モデルへ知識を全部押し込むのではなく、比較的小さな生成モデルと外部検索を組み合わせて高精度を出した点が重要なのだ。

もう一つの重要な観察は、passage 数を増やすほど性能が上がりやすいことなのだ。10件、25件、50件、100件と候補を増やすと、多くのデータセットで EM が改善する。検索上位だけでは答えを含む文書を取り逃がすことがあり、候補を広げると正解根拠に当たる確率が上がるからだと読めるのだ。

ただし、「文書を増やせば何でも解決」ではないのだ。100件の passage を T5-large で処理するには、学習でも推論でも大きな計算資源が必要になる。実務で使うなら、検索件数、再ランキング、コンテキスト圧縮、キャッシュ、応答時間のバランスを考えないといけないのだ。

読み取りで注意したい点もあるのだ。FiD は、複数 passage をデコーダで融合すると強いことを示している。一方で、性能向上の理由が、検索 recall が上がったからなのか、本当に複数証拠を統合できたからなのかは、完全には分離されていないのだ。100件の中に、単独で答えられる passage が入る確率が上がっただけ、という可能性もあるのだ。

また、FiD は根拠提示そのものを保証する仕組みではないのだ。デコーダは多くの passage を見て回答するが、その答えがどの passage に基づくのか、矛盾する根拠をどう扱ったのかは、標準の FiD だけでは十分に説明されない。現代の RAG で必要な citation、faithfulness、監査可能性は、別途設計が必要なのだ。

RAG エンジニアの観点では、持ち帰る教訓は三つなのだ。第一に、検索結果の渡し方は性能を大きく左右する。同じ検索器でも、生成モデルにどう読ませるかで差が出るのだ。第二に、少数の上位文書だけに頼ると、答えを含む根拠を落とすことがある。第三に、候補を増やすほど計算コストも増えるので、検索品質、生成品質、レイテンシを一緒に設計する必要があるのだ。

まとめるのだ。FiD は、検索した複数 passage を個別にエンコードし、デコーダ側で融合して回答を生成する方法なのだ。Natural Questions や TriviaQA で強い結果を出し、小さめの生成モデルでも外部文書をうまく使えば高精度 QA が可能だと示した。一方で、計算コストは大きく、根拠統合の中身を完全に説明したわけではない。だから実務では、検索 recall、候補数、圧縮、根拠提示、レイテンシをセットで考えるのが大事なのだ。

以上、ずんだもんによる FiD の音声ガイドだったのだ。
