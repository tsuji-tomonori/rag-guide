# 1. はじめに

**論文名:** *Dense Passage Retrieval for Open-Domain Question Answering*
**著者:** Vladimir Karpukhin ほか
**採録:** EMNLP 2020 Main Conference

本論文は、オープンドメイン質問応答における文書検索を、BM25のような疎な語彙一致ではなく、ニューラルネットワークが生成する**密ベクトル表現だけで実現するDense Passage Retriever（DPR）**を提案している。

中心的な主張は、複雑な追加事前学習や検索器・回答器の同時学習を行わなくても、質問と正解パッセージの組を使って単純なdual-encoderを適切に学習すれば、BM25を大幅に上回れるというものである。

# 2. 背景

オープンドメイン質問応答は、一般に次の2段階で構成される。

1. **Retriever**がWikipediaなどの巨大コーパスから関連しそうなパッセージを絞り込む
2. **Reader**が取得したパッセージを詳しく読み、回答スパンを抽出する

従来のRetrieverにはTF-IDFやBM25が広く使われていた。しかし、これらは基本的に単語の一致を利用するため、「bad guy」と「villain」のように意味は近いが表記が異なる場合に検索できないことがある。

先行手法ORQAは密検索がBM25を超え得ることを示したが、Inverse Cloze Taskによる大規模な追加事前学習や、複雑な学習方法を必要としていた。本論文はこれに対し、**通常のBERTと質問・パッセージ対だけで十分なのか**を検証している。([ar5iv][1])

# 3. 提案手法

DPRは、質問用とパッセージ用に独立した2つのBERTを使用するdual-encoderである。

質問 (q) とパッセージ (p) をそれぞれ、

[
E_Q(q),\quad E_P(p)
]

という768次元ベクトルに変換し、関連度を内積で計算する。

[
\mathrm{sim}(q,p)=E_Q(q)^\top E_P(p)
]

両エンコーダにはBERT-base-uncasedを使用し、`[CLS]`トークンの表現を検索ベクトルとする。パッセージのベクトルは事前計算してFAISSに格納できるため、質問時には質問ベクトルだけを計算し、Maximum Inner Product Searchによって近いパッセージを取得できる。

## 学習方法

各質問に対して、正例パッセージのスコアが負例より高くなるように、softmaxに基づく負の対数尤度を最小化する。

特に重要なのが**in-batch negatives**である。バッチ内に (B) 個の質問と正解パッセージがある場合、ある質問の正解パッセージ以外の (B-1) 個を負例として再利用する。これにより、追加のエンコード計算をほぼ増やさず、大量の質問・負例対を作れる。

最良の構成では、次の2種類を併用する。

* 同じミニバッチ内の他の正解パッセージ
* BM25が上位に返したものの回答文字列を含まない、1個のhard negative

つまり、DPRの性能向上の中心は、新しい複雑なアーキテクチャではなく、**負例の設計とバッチ内再利用**にある。

## 回答器

検索後は、取得した最大100パッセージにBERT Readerを適用する。Readerは各パッセージについて、

* 回答スパンの開始位置
* 回答スパンの終了位置
* そのパッセージ自体が正解を含む確率

を推定し、最終回答を選択する。RetrieverとReaderは基本的に別々に学習される。

# 4. 実験方法

検索対象には、2018年12月20日時点の英語Wikipediaを使用している。表、情報ボックス、リストなどを除去した後、記事本文を**100語単位の重複しないパッセージ**に分割し、約2,101万パッセージを構築した。各パッセージにはWikipedia記事のタイトルも付与される。

評価データセットは以下の5種類である。

| データセット            | 特徴                   |
| ----------------- | -------------------- |
| Natural Questions | 実際のGoogle検索質問を基にしたQA |
| TriviaQA          | Web上のトリビア質問          |
| WebQuestions      | Google Suggest由来の質問  |
| CuratedTREC       | TRECなどから収集された事実質問    |
| SQuAD v1.1        | 提示された文章を見て作成された読解質問  |

Retrieverの評価指標は、上位 (k) 件のどこかに回答文字列を含むパッセージが存在する割合である**Top-(k) retrieval accuracy**で、主にTop-20とTop-100を使用する。

エンドツーエンド質問応答では、予測回答が正解文字列と完全一致する割合である**Exact Match（EM）**を用いる。比較対象にはLucene-BM25、ORQA、REALM、GraphRetrieverなどが含まれる。

# 5. 結果

## 検索性能

単一データセットで学習したDPRとBM25のTop-20検索精度は次の通りである。

| データセット            |     BM25 |      DPR |     差 |
| ----------------- | -------: | -------: | ----: |
| Natural Questions |     59.1 | **78.4** | +19.3 |
| TriviaQA          |     66.9 | **79.4** | +12.5 |
| WebQuestions      |     55.0 | **73.2** | +18.2 |
| CuratedTREC       |     70.9 | **79.8** |  +8.9 |
| SQuAD             | **68.8** |     63.2 |  −5.6 |

SQuADを除く4データセットで、DPRはBM25を約9～19ポイント上回った。特にNatural Questionsでは、Top-20精度が59.1%から78.4%へ大きく向上している。

SQuADでBM25が優位なのは、質問作成者が対象パッセージを見ながら質問を書いており、質問と文書の語彙的重複が大きいためと説明されている。また、SQuADの出典が500件強のWikipedia記事に偏っていることも影響している。

## 負例学習の効果

Natural Questions開発セットでは、通常の負例学習からin-batch negativesに変更するだけで性能が向上した。さらにBM25 hard negativeを1個追加した構成では、Top-5、Top-20、Top-100精度がそれぞれ**65.8、78.0、84.9**となった。

また、わずか1,000件の質問・パッセージ対で学習したDPRでも、Natural Questions上ではBM25を上回った。これは、事前学習済み言語モデルを用いれば、密検索の学習に必ずしも巨大な教師データが必要ではないことを示している。

## エンドツーエンド質問応答

Natural Questionsでは、BM25を用いたシステムの32.6 EMに対し、DPRは**41.5 EM**を達成した。ORQAの33.3、REALMの39.2～40.4も上回っている。

TriviaQAでは52.4から56.8、複数データセットで学習した場合のWebQuestionsでは29.9から42.4、CuratedTRECでは24.9から49.4へ向上した。BM25とDPRを組み合わせたCuratedTRECの結果は50.6であった。一方、SQuADではDPRが29.8、BM25が38.1であり、検索性能の傾向と同様にDPRが劣った。論文発表時点では、5データセット中4つで既存最高水準を更新したと報告されている。

## 計算効率

FAISSのインメモリ索引を使用したDPRは、Top-100検索で毎秒995質問を処理した。比較されたLucene-BM25は、CPUスレッド当たり毎秒23.7質問だった。

ただし、索引構築はDPRの方が重い。約2,100万パッセージの埋め込み計算に8 GPUで約8.8時間、FAISS索引構築に約8.5時間を要した一方、Luceneの転置索引構築は約30分だった。

# 6. 考察

本論文の最大の成果は、密検索の成功に必要なのは必ずしも複雑なモデルではなく、**適切な教師信号、in-batch negatives、hard negative**であることを明確に示した点である。Retrieverの改善が最終的なQA精度にもつながることを、複数データセットで実証した点も重要である。

一方で、いくつかの制約がある。

第一に、評価は「回答文字列を含むか」で判定されるため、そのパッセージが実際に質問への根拠として適切かどうかを完全には評価していない。同じ文字列を偶然含むだけのパッセージが正解扱いされる可能性がある。

第二に、一部データセットの正例は、BM25が取得した「回答を含む最上位パッセージ」から作られている。この遠隔教師あり学習は実用的だが、教師データ自体にBM25の検索バイアスが入る。

第三に、DPRは意味的な言い換えには強いものの、珍しい固有表現や特徴的な文字列の厳密な一致には弱い。論文の定性分析でも、BM25が希少なキーフレーズを正確に捉える一方、DPRが取り逃がす例が示されている。

第四に、純粋なDPRが常に最良というわけではない。TriviaQAやCuratedTRECなどでは、BM25とDPRのスコアを組み合わせたハイブリッド検索が最高性能となっている。この結果は、疎検索と密検索が代替関係というより、**相補的な関係**にあることを示している。

また、実験は英語Wikipediaの事実質問と抽出型QAを中心としている。表や情報ボックスはコーパス構築時に除去され、回答は本文中に明示的に存在する必要がある。そのため、多言語検索、更新頻度の高いWeb検索、複数文書推論、生成型回答への一般化は、この論文単独では十分に検証されていない。

# 7. まとめ

本論文は、質問とパッセージを別々のBERTでベクトル化し、内積によって検索するDPRを提案した。in-batch negativesとBM25由来のhard negativeを用いることで、追加の検索向け事前学習や複雑な同時学習を行わず、BM25を大きく上回る検索性能を実現した。

特に重要な結論は次の3点である。

* 単純なdual-encoderでも、負例設計が適切なら高性能な密検索が可能
* RetrieverのTop-(k)精度向上は、最終的なQAのExact Match向上につながる
* 密検索は意味的対応に強いが、BM25との組み合わせが有効な場合もある

この構成は、その後の検索拡張型質問応答や生成モデルと検索器を組み合わせるシステムにとって、扱いやすい基本形となった。論文自身も、DPRが生成モデルと組み合わせられることを後続研究との関係で論じている。

# 8. 総評

本論文の新規性はdual-encoderそのものではなく、**単純な教師あり学習だけで密検索がBM25を安定して上回ることを、精密なアブレーションと大規模QA実験によって示したこと**にある。提案は簡潔で再利用しやすく、検索性能、QA性能、標本効率、ドメイン間転移、実行速度まで検証されており、実証研究として非常に完成度が高い。

主な弱点は、Wikipedia上の抽出型QAに評価範囲が限定されること、回答文字列ベースの検索評価に粗さがあること、索引構築コストとメモリ要求が大きいこと、純粋な密検索がSQuADや希少語検索では疎検索に劣ることである。それでも、主張と実験結果の対応は明確であり、結論も概ね十分に裏付けられている。
