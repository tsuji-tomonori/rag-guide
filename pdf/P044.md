# 1. はじめに

**論文名:** *Chain-of-Verification Reduces Hallucination in Large Language Models*
**著者:** Shehzaad Dhuliawalaほか
**概要:** 大規模言語モデルが生成した回答を、同じモデル自身に段階的に検証させることで、事実誤認・ハルシネーションを減らす **Chain-of-Verification（CoVe）** を提案した論文である。2023年にarXivで公開され、後に **Findings of ACL 2024** に採録された。([arXiv][1])

本論文の中心的な主張は、LLMが長い回答の中では誤った事実を生成しても、その事実を独立した短い質問として改めて尋ねると、正しく回答できる場合が多いというものである。CoVeはこの性質を利用し、回答生成後に「検証質問の作成」「独立した回答」「最終修正」を行う。

# 2. 背景

LLMは、もっともらしいが事実とは異なる情報を生成することがある。特に、学習データ中で出現頻度の低い人物・出来事・属性などの情報や、多数の事実を含む長文生成では、誤りが発生しやすい。

従来の対策は、おおむね次の三つに分類される。

* 学習時にモデルの重みを調整する方法
* 生成時に自己批判・再推論・一貫性確認を行う方法
* 検索や知識ベースなどの外部ツールを利用する方法

本研究は二番目に位置し、モデルの再学習や外部検索を必要とせず、推論時のプロンプト処理だけで事実性を向上させる。通常のChain-of-Thoughtは推論問題には有効でも、事実を知っているかどうかに由来するハルシネーションを必ずしも減らさないため、「何を検証すべきか」を明示的に分解する必要があると考えている。

# 3. 提案手法

CoVeは、以下の4段階で構成される。

### ① 初期回答の生成

ユーザーの質問に対して、通常どおりLLMに回答させる。この回答には誤った事実が含まれる可能性がある。

### ② 検証質問の計画

初期回答に含まれる事実主張を確認するための質問を生成する。

たとえば、「ヒラリー・クリントンはニューヨーク生まれである」という初期回答に対しては、「ヒラリー・クリントンはどこで生まれたか」という検証質問を作る。

### ③ 検証質問の実行

各検証質問にLLM自身が回答する。重要なのは、検証回答を生成する際に、可能な限り初期回答をコンテキストから除外することである。

初期回答を見せたまま検証させると、モデルが誤った記述にアンカリングされ、同じ誤りを繰り返す可能性がある。

### ④ 最終回答の生成

初期回答と検証結果を照合し、矛盾した事実を修正または削除した最終回答を生成する。

## CoVeの変種

**Joint**

検証質問の作成と回答を一つのプロンプト内で連続して行う。処理は簡単だが、検証回答が初期回答を参照するため、誤りをコピーしやすい。

**2-Step**

検証質問の作成と回答を別々のプロンプトで実行する。回答段階では初期回答を与えず、検証質問だけを入力する。

**Factored**

検証質問を一つずつ、互いに独立したプロンプトで回答する。初期回答だけでなく、他の検証回答による干渉も防止する。

**Factor＋Revise**

Factored方式に加え、各事実を「一致」「不一致」「部分的一致」に明示的に分類してから最終回答を作る。長文生成ではこの方式が最も高い性能を示した。

# 4. 実験方法

基盤モデルには主として、命令チューニングされていない **Llama 65B** を使用し、3-shot程度のfew-shotプロンプトとgreedy decodingを利用している。比較対象には、Llama 2 70B Chatのzero-shot、通常のChain-of-Thought、ならびに既報のInstructGPT、ChatGPT、PerplexityAIの結果が含まれる。

評価タスクは次の四種類である。

| タスク                  | 内容                          | 規模・評価指標                    |
| -------------------- | --------------------------- | -------------------------- |
| Wikidata             | 「ボストン生まれの政治家を挙げよ」のようなリスト生成  | 56問、micro precision、正解・誤答数 |
| Wiki-Category List   | Wikipediaカテゴリに基づく、より難しい集合生成 | 55問、micro precision、正解・誤答数 |
| MultiSpanQA          | 複数の短い答えを要求する閉じた知識QA         | 418問、F1・適合率・再現率            |
| Biography Generation | 人物の伝記を生成する長文タスク             | FActScore、生成した事実数          |

伝記生成のFActScoreは、回答を原子的な事実に分解し、検索拡張型の評価モデルを用いて各事実を検証する自動評価指標である。

# 5. 結果

主要な結果は以下のとおりである。

| タスク                     | Llama 65B Few-shot |             最良のCoVe |     変化 |
| ----------------------- | -----------------: | ------------------: | -----: |
| Wikidata Precision      |               0.17 |        0.36（2-Step） |  約2.1倍 |
| Wiki-Category Precision |               0.12 |      0.22（Factored） |  約1.8倍 |
| MultiSpanQA F1          |               0.39 |      0.48（Factored） | 約23%向上 |
| Biography FActScore     |               55.9 | 71.4（Factor＋Revise） | 約28%向上 |

Wikidataでは、1回答当たりの誤ったエンティティ数が **2.95から0.68** に減少した。一方、正しいエンティティ数も0.59から0.38へ減っており、CoVeは誤答を大幅に削除する代わりに、一部の正答も削除している。

伝記生成では、Factored方式のFActScoreが63.7、Factor＋Revise方式が71.4となった。単純なFactored方式よりも、検証結果との整合性を明示的に判定する工程が有効だった。ただし、平均事実数はベースラインの16.6から12.3へ減少している。

また、Wikidataのリスト質問では、初期回答中のエンティティが正しい割合は約17%だったのに対し、各エンティティを独立した短い質問として確認した場合、約70%を正しく判定できた。この結果がCoVeの基本的な動機を支持している。

そのほか、次の傾向が報告されている。

* Jointより2-StepやFactoredの方が一貫して高性能
* 通常のChain-of-Thoughtはハルシネーションを改善せず、タスクによっては悪化
* テンプレート的な検証質問より、LLMが生成した検証質問の方が高性能
* 「この記述は正しいか」というYes／No質問より、「実際には何か」を尋ねる開放型質問の方が有効

Yes／No形式では、質問中に含まれた誤った前提にモデルが同意してしまう傾向が観察されている。

# 6. 考察

## 評価できる点

最大の貢献は、**同じモデルでも質問の提示方法とコンテキストを変えることで、保持している知識をより正確に引き出せる**ことを明確に示した点である。

特に、検証時に初期回答を見せないFactored方式は、単なる「もう一度考えてください」ではなく、誤答へのアンカリングを構造的に回避している。学習やモデル内部へのアクセスを必要とせず、ブラックボックス型LLMにも適用しやすい点も実用的である。

Joint、2-Step、Factored、Factor＋Reviseを比較したアブレーションにより、「処理を増やしたから良くなった」だけではなく、**独立した検証と明示的な整合性判定が重要である**ことを示している。

## 問題点・限界

### 精度向上の一部は回答の短縮による

CoVeは誤った情報を削除する一方、正しい情報も削除している。伝記では事実数が16.6から12.3、Wikidataでは正解数が0.59から0.38へ減少した。

したがって、「より多くの正しい情報を生成できた」というより、主として「不確実な情報を削除し、精度を高めた」と解釈すべきである。実運用では、正確性だけでなく網羅性・回答有用性とのトレードオフを評価する必要がある。

### 同一モデルによる自己検証の上限

初期回答と検証回答は同じモデルの内部知識に依存する。モデルが一貫して誤った知識を持っている場合や、そもそも必要な知識を持っていない場合、独立して質問しても訂正できない。

実際、非常に希少な事実では、外部検索を利用するPerplexityAIがCoVeを上回る場合が示されている。著者自身も、検索拡張を検証工程へ導入することを今後の方向性としている。

### 実験範囲が限定的

CoVeの中心的な実験はLlama 65Bに依存しており、異なるモデル規模や複数の命令チューニング済みモデルに対する再現性は十分に検証されていない。

リストタスクも56問、55問と比較的小規模で、MultiSpanQAも短い事実回答に限定して抽出されている。統計的有意差、複数プロンプト、複数乱数シードによる頑健性評価も提示されていない。

### 評価器への依存

長文評価には自動評価器であるFActScoreを使用しており、人間による全面的な事実確認ではない。評価モデルの検索失敗や判定誤りが、実験結果に影響する可能性がある。

### 計算コスト

Factored方式は、事実や検証質問ごとに別々のLLM呼び出しを必要とする。並列化は可能だが、通常回答と比べてトークン数、API呼び出し回数、料金、レイテンシが増える。論文ではこのコストが定量的に評価されていない。著者も計算量増加を制約として認めている。

# 7. まとめ

本論文は、LLMの回答を次の流れで再検証するCoVeを提案した。

> 初期回答 → 検証質問の作成 → 独立した回答 → 整合性確認 → 最終修正

最も重要な発見は、**検証回答を初期回答から分離するほど、同じ誤りの反復を防ぎやすい**という点である。

複数の事実質問、閉じた知識QA、伝記生成で一貫した改善が確認され、特にFactor＋Revise方式は長文の事実精度を大きく向上させた。一方で、外部知識を取得する手法ではなく、モデル内部に存在する知識をより適切に引き出す手法であるため、知識欠落そのものは解決できない。

# 8. 総評

## 査読結果

**実際の掲載結果:** **Findings of ACL 2024に採録**。論文は同会議録の3563–3578頁に掲載されている。([ACL Anthology][2])

**独自査読判定:** **Weak Accept／採録推奨**

### 判定理由

手法自体は単純なプロンプト分解であり、アルゴリズム上の新規性は極めて高いわけではない。しかし、初期回答を検証コンテキストから切り離すという設計原理が明確で、複数タスクと変種比較によって有効性が説得的に示されている。再学習を必要とせず、実際のLLMシステムへ組み込みやすい点も大きい。

一方で、精度向上と回答量減少の切り分け、複数モデルでの一般化、統計的検定、計算コスト、人手評価が不足している。したがって強い採録判定までは難しいものの、ハルシネーション抑制に関する実用的かつ重要な知見を提供しており、採録に値する論文である。
