## 1. はじめに

**Toolformer: Language Models Can Teach Themselves to Use Tools** は、言語モデル自身に、外部APIを「いつ・どれを・どの引数で呼び出し、その結果をどう利用するか」を自己教師ありで学習させる手法を提案した論文です。基盤モデルはGPT-J 6.7Bで、電卓、質問応答、Wikipedia検索、翻訳、カレンダーを扱います。arXiv版は2023年2月に公開され、その後NeurIPS 2023でOral論文として採択されました。([arXiv][1])

一言でまとめると、**「API結果を与えたときに後続トークンの予測損失が下がるか」を教師信号として、ツール利用データをモデル自身に生成・選別させる研究**です。

---

## 2. 背景

大規模言語モデルは幅広いタスクに対応できますが、正確な計算、最新・希少な事実の参照、低資源言語の理解、現在日時の認識などでは、専用ツールより劣ります。単純にモデル規模を拡大しても、外部知識へのアクセスや厳密な計算能力が直接得られるわけではありません。

既存のツール利用研究には、次の問題がありました。

* 大量の人手によるツール利用アノテーションが必要
* 特定タスク用のfew-shot例やプロンプトに依存
* あらかじめ使用するツールを指定する必要がある

Toolformerは、少数のAPI使用例だけを人間が用意し、その後の学習データ生成と選別を言語モデル自身に行わせることで、これらの問題を軽減しようとしています。([ar5iv][2])

---

## 3. 提案手法

### 3.1 API呼び出し候補の生成

通常のテキストコーパスを (C)、基盤言語モデルを (M) とします。各APIについて数個の利用例をプロンプトとして与え、モデルに次のようなAPI呼び出しをテキスト中へ挿入させます。

```text
400人中1400人 → [Calculator(400 / 1400) → 0.29] 29%
```

モデルはまず、各位置でAPI開始トークンを生成する確率を計算し、閾値 (\tau_s) を超えた位置から呼び出し候補をサンプリングします。専用の構造化出力層は使わず、既存の文字列トークンでAPI呼び出しを表現します。([ar5iv][2])

### 3.2 APIの実行

生成した候補を実際に実行し、結果を取得します。APIは、ニューラルモデル、検索システム、Python処理など、入出力を文字列として表現できるものであれば利用できます。

本論文で使用するツールは次の5種類です。

| ツール         | 実装                          |
| ----------- | --------------------------- |
| 質問応答        | Natural Questionsで学習したAtlas |
| Wikipedia検索 | KILT版Wikipediaに対するBM25検索    |
| 電卓          | 四則演算、結果は小数第2位まで             |
| 機械翻訳        | NLLB 600M、英語への翻訳            |
| カレンダー       | 現在の日付を返すAPI                 |



### 3.3 予測損失によるフィルタリング

Toolformerの中心的アイデアです。

位置 (i) 以降のトークンに対する重み付きクロスエントロピーを、

[
L_i(z)
======

-\sum_{j=i}^{n}
w_{j-i}
\log p_M(x_j\mid z,x_{1:j-1})
]

と定義します。ここで (z) はモデルへ追加するAPI情報です。

API呼び出し (c_i) とその結果 (r_i) を与えた場合を、

[
L_i^{+}=L_i(e(c_i,r_i))
]

APIを使わない場合、または呼び出しだけを与えて結果を与えない場合のうち良い方を、

[
L_i^{-}=

\min\left(
L_i(\epsilon),
L_i(e(c_i,\epsilon))
\right)
]

とします。そして、

[
L_i^{-}-L_i^{+}\geq \tau_f
]

を満たす呼び出しだけを残します。

つまり、**APIの返答が実際に後続テキストを予測しやすくした場合のみ、それを有用な学習例とみなす**仕組みです。単にAPI形式の文字列が挿入されたことによる効果とも比較する点が重要です。

### 3.4 ファインチューニングと推論

選別されたAPI呼び出しを元のコーパスへ挿入し、拡張コーパス (C^*) を構築します。その後、通常の言語モデリング目的でGPT-Jをファインチューニングします。

推論中にモデルがAPI結果を要求する記号を生成したら、生成を一時停止してAPIを実行し、返答を文脈へ追加して生成を再開します。これにより、ツール選択、呼び出し位置、引数生成、返答の利用を、すべて次トークン予測の枠組みで扱えます。

## 4. 実験方法

基盤モデルにはGPT-J 6.7B、学習コーパスにはCCNetの一部を使用しています。計算量を削減するため、例えば電卓用データでは数字を3個以上含む文書だけを候補にするなど、ツールごとのヒューリスティックを適用しています。各APIにつき最大25,000例、最大系列長1,024、実効バッチサイズ128、学習率 (10^{-5})、最大2,000ステップで学習し、8基のNVIDIA A100 40GBを使用しています。

比較対象は以下です。

| モデル                 | 内容                           |
| ------------------- | ---------------------------- |
| GPT-J               | 未調整の基盤モデル                    |
| GPT-J + CC          | APIなしのCCNetで追加学習             |
| Toolformer disabled | Toolformer学習済みだが推論時のAPI利用を禁止 |
| Toolformer          | API利用を許可                     |
| OPT 66B             | 大規模比較モデル                     |
| GPT-3 175B          | davinci系の大規模比較モデル            |

評価は、タスク固有のfew-shot例を与えないzero-shot設定です。

* 知識補完：LAMA
* 数学文章題：ASDiv、SVAMP、MAWPS
* 質問応答：WebQuestions、Natural Questions、TriviaQA
* 多言語質問応答：MLQA
* 時間推論：TempLAMA、独自のDateset
* 言語モデリング：WikiText、CCNet
* スケーリング：GPT-2の124M、355M、775M、1.6BとGPT-J 6.7B

主実験では、API開始トークンが次トークン確率の上位10位以内に入ればAPI呼び出しを開始する、通常のgreedy decodingより積極的な設定を採用しています。また、無限にツールを呼び続けることを防ぐため、1入力につきAPI呼び出しは最大1回に制限されています。

## 5. 結果

代表的な結果は以下の通りです。値は論文独自のzero-shot評価基準に基づきます。

| タスク          | GPT-J | Toolformer | GPT-3 175B |
| ------------ | ----: | ---------: | ---------: |
| LAMA T-REx   |  31.9 |   **53.5** |       39.8 |
| ASDiv        |   7.5 |   **40.4** |       14.0 |
| SVAMP        |   5.2 |   **29.4** |       10.0 |
| MAWPS        |   9.9 |   **44.0** |       19.8 |
| WebQuestions |  18.5 |       26.3 |   **29.0** |
| TriviaQA     |  43.9 |       48.8 |   **65.9** |
| Dateset      |   3.9 |   **27.3** |        0.8 |

Toolformerは、事実補完と数学文章題でGPT-Jを大幅に上回り、66BのOPTや175BのGPT-3も超えています。数学タスクでは97.9%の例で電卓を利用し、LAMAでは98.1%で質問応答APIを利用しました。一般質問応答では99.3%でWikipedia検索を利用し、同規模モデルには勝つもののGPT-3には届きませんでした。

多言語質問応答では、APIを無効にしたToolformerよりは全言語で改善しましたが、通常のGPT-Jを一貫して上回ることはできませんでした。著者らは、CCNetによる追加学習の分布シフトが原因の可能性を指摘しています。

時間推論ではDatesetを3.9から27.3へ改善しました。一方、TempLAMAでカレンダーAPIが使用された割合は0.2%にすぎず、改善の多くは質問応答や検索によるものでした。

言語モデリングのperplexityは、GPT-J + CCとAPI無効化ToolformerでともにWikiText 10.3、CCNet 10.5でした。この意味では、API用データを追加しても、通常テキストのモデリング性能は追加で悪化していません。ただし未調整GPT-JのWikiText perplexityは9.9であり、「基盤モデルから全く性能低下していない」という意味ではありません。

モデル規模の実験では、ツールを有効に活用する能力はおおむね775Mパラメータ付近から現れています。小規模モデルではAPIの有無による差が小さく、言語モデル側に一定以上の文脈理解能力が必要であることが示唆されます。

## 6. 考察

### 評価できる点

最大の貢献は、ツール利用を人手アノテーションや強化学習の問題ではなく、**自己生成データと次トークン予測損失による選別問題**として定式化したことです。API出力がモデルにとって有益かを、モデル自身の損失差で判定する設計は単純でありながら汎用性があります。

また、性能向上がタスクとツールの対応関係に沿っています。数学では電卓、知識補完では質問応答、一般QAでは検索、他言語質問では翻訳を選択しており、単なる追加事前学習だけでは説明できない改善が確認されています。

### 注意すべき点

第一に、「いつAPIを呼ぶかを自律的に判断する」という主張には留保が必要です。主結果ではAPIトークンが上位10位以内なら強制的に呼び出す設定です。通常のgreedy decodingに相当する (k=1) では、WebQuestionsのAPI利用率は8.5%、性能は19.3ですが、(k=3) にすると利用率99.3%、性能26.3となります。性能の一部は、モデル自身の自然な判断というより、デコード時にツール利用を促進した結果です。

第二に、API呼び出しが最大1回であるため、次のような処理はできません。

* カレンダーで現在日付を得た後、その日付を質問応答APIへ渡す
* 検索結果が悪い場合にクエリを書き換える
* 複数の検索結果を比較する
* 計算と検索を組み合わせる

著者らも、ツール連鎖、対話的検索、プロンプト感度、学習データ生成の非効率性、APIコストを考慮しない点を限界として挙げています。([ar5iv][2])

第三に、LAMAでは正解が最初の5語以内、質問応答では最初の20語以内に含まれれば正解とするなど、評価基準は通常の完全一致より寛容です。全モデルに同一基準を適用しているため相対比較には意味がありますが、絶対値を他研究の標準スコアと直接比較するのは適切ではありません。

総じて、Toolformerは完成された自律エージェントというより、**言語モデルが自己教師ありでAPI呼び出しを獲得できることを示した強力な概念実証**と評価するのが妥当です。

---

## 7. まとめ

Toolformerは、以下の処理によって言語モデルへツール利用能力を付与します。

1. 少数のデモからAPI呼び出し候補を自己生成する
2. APIを実行する
3. API結果が後続トークンの予測損失を下げる例だけを残す
4. そのデータで通常の言語モデリングを行う

6.7BのGPT-Jでも、計算、知識補完、時間推論などでは、はるかに大きい66B・175Bモデルを上回りました。一方で、複数ツールの連鎖、検索の反復、API利用コスト、自然な呼び出し判断などは未解決です。
