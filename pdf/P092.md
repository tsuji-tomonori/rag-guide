# P092　FlumeJava: Easy, Efficient Data-Parallel Pipelines

**著者:** Craig Chambersほか、Google
**掲載:** ACM PLDI 2010, pp. 363–375 ([Google Research][1])

## 1. はじめに

本論文は、複数のMapReduceジョブを連結した大規模データ処理パイプラインを、通常のJavaプログラムに近い形で記述できるライブラリ「FlumeJava」を提案する。

単一のMapReduceは比較的簡単に記述できるが、実用的な処理では複数のMapReduceを連結する必要がある。その場合、ジョブの起動順序、ステージ間の一時ファイル、障害処理、データ形式の変換などの管理コードが増え、本来の計算ロジックが見えにくくなる。FlumeJavaは、論理的な処理を高水準APIで記述させ、実行直前にデータフロー全体を最適化することで、記述容易性と実行効率の両立を目指す。([UWコンピュータサイエンスユーザーページ][2])

## 2. 背景

MapReduceは、処理をMap、Shuffle、Reduceの3段階に分割し、並列化、ワーカ配置、通信、障害回復などを実行基盤に委ねる。しかし、集計、結合、フィルタリングなどを組み合わせた処理は、複数のMapReduceジョブとして手作業で構成しなければならない。

従来の実装では、開発者が次の物理的詳細まで扱う必要があった。

* 各MapReduceジョブの起動と依存関係
* 中間結果の保存先と削除時期
* 論理演算をMapperやReducerへ分割する方法
* 小規模テスト用コードと本番分散実行コードの差異

この結果、論理的には単純な処理でも、コードが特定の実行ステージ構成に強く依存し、修正や再利用が困難になる。

## 3. 提案手法

### 高水準データ抽象

FlumeJavaの中心は、巨大な不変コレクションを表す次の2クラスである。

| 抽象               | 内容                     |
| ---------------- | ---------------------- |
| `PCollection<T>` | 型`T`の要素からなる不変の並列コレクション |
| `PTable<K,V>`    | キーと値の組からなる不変の並列マルチマップ  |

主な基本演算は以下の4種類に絞られている。

| 演算                | 役割                                |
| ----------------- | --------------------------------- |
| `parallelDo()`    | 各要素を独立に処理する。map、filter、flatMapに相当 |
| `groupByKey()`    | 同じキーの値をまとめる。Shuffleに相当            |
| `combineValues()` | 結合則を満たす関数でキーごとに集約                 |
| `flatten()`       | 複数のコレクションを1つの論理コレクションとして統合        |

`count()`、`join()`、`top()`などは、これらの基本演算を組み合わせた派生演算として提供される。APIを少数の合成可能な演算に限定することで、ユーザーによる拡張性と、最適化器による解析の容易さを両立している。([UWコンピュータサイエンスユーザーページ][2])

### 遅延評価と実行計画

`parallelDo()`などを呼び出しても、その場では計算を実行しない。代わりに、入力、演算、出力の関係を有向非巡回グラフとして記録する。

最後に`FlumeJava.run()`が呼ばれると、システムは次の順序で処理する。

1. データフローグラフを完成させる。
2. グラフ全体を最適化する。
3. 最適化後の演算をトポロジカル順に実行する。
4. 不要になった中間ファイルを自動的に削除する。

この動的な実行計画構築により、条件分岐やループ、Javaのメソッド呼び出しを含む複雑なプログラムでも、実際に生成されたパイプライン全体を最適化できる。

### 演算融合

最適化器は、複数の論理演算を少数の物理MapReduceジョブへ融合する。

`ParallelDo`に対しては、主に次の融合を行う。

* **producer-consumer fusion:** 連続する関数$f$と$g$を、$g\circ f$として1回の走査に統合
* **sibling fusion:** 同じ入力を読む複数の演算を、複数出力を持つ1回の走査に統合

これにより、中間結果の生成、書込み、再読込みを削減する。

### MSCRへの変換

最適化の中間表現として、**MSCR（MapShuffleCombineReduce）**を導入する。MSCRは複数の入力チャネルと出力チャネルを持ち、各出力についてShuffle、Combine、ReduceまたはMapからの直接出力を指定できる。

表現力は通常のMapReduceより高いが、1つのMSCRは1つのMapReduceジョブとして実行される。共通入力を持つ集計や複数の出力をまとめることで、ジョブ起動回数と中間I/Oを削減する。([UWコンピュータサイエンスユーザーページ][2])

最適化器の主要パスは、Flattenの移動、CombineValuesの持ち上げ、融合境界の決定、ParallelDo融合、MSCR融合である。SiteDataの例では、当初16個あった並列演算が最終的に2個のMSCRへ縮約されている。([UWコンピュータサイエンスユーザーページ][2])

### 実行系

最適化後のMSCRについて、入力が小さければローカルの逐次処理、大きければ分散MapReduceを自動選択する。これにより、同じプログラムを小さなテストデータではIDEやデバッガ上で実行し、本番データでは多数のマシン上で実行できる。

さらに、一時ファイル管理、標準出力の転送、遠隔ワーカで発生した例外の再送出、前回の中間結果を再利用するキャッシュ実行も提供する。([UWコンピュータサイエンスユーザーページ][2])

## 4. 実験方法

評価は、次の3観点から行われた。

| 評価項目  | 方法                              |
| ----- | ------------------------------- |
| 実利用状況 | Google内部のFlumeJava実行ログから利用者数を集計 |
| 最適化効果 | 各パイプラインについて最適化前後のステージ数を比較       |
| 実行性能  | 実運用パイプラインを基に4種類のベンチマークを作成       |

性能実験では、Ads Logs、IndexStats、Build Logs、SiteDataを、原則として以下の3方式で実装した。

1. モジュール化されたFlumeJava
2. モジュール化されたJava MapReduce
3. 手動最適化されたJava MapReduce

Ads LogsとSiteDataについては、手動最適化されたSawzall版も比較対象とした。入力規模は550 MBから3.3 TB、使用マシン数は4台から200台である。各方式に同等の資源を割り当て、5回実行したうちの最短時間を採用した。コントローラ起動、バイナリ配布、一時ファイル削除などの調整時間は測定から除外されている。

## 5. 結果

### 利用実績

2010年3月時点で、1日アクティブユーザー62人、7日で106人、30日で176人、累計319人が利用していた。数百本のプログラムが作成され、数千台のマシン上でペタバイト規模のデータを処理した実績が報告されている。([UWコンピュータサイエンスユーザーページ][2])

### 最適化効果

実運用パイプラインでは、最適化によってステージ数が平均約5分の1になり、30分の1を超える縮約例もあった。207ステージが1ステージに融合された例や、最大規模のパイプラインで820ステージが149ステージになった例も報告されている。

ベンチマークでは、FlumeJavaが自動生成したステージ数は、手動最適化されたMapReduce版とすべて一致した。

| ベンチマーク     | FlumeJava最適化前→後 | 手動最適化MapReduce |
| ---------- | --------------: | -------------: |
| Ads Logs   |          14 → 1 |              1 |
| IndexStats |          16 → 2 |              2 |
| Build Logs |           7 → 1 |              1 |
| SiteData   |          12 → 2 |              2 |



### コード量

すべてのベンチマークで、FlumeJava版はモジュール化されたJava MapReduce版より短かった。例えば、IndexStatsでは176行対296行、Build Logsでは276行対476行である。一方、専用言語であるSawzallはJavaよりさらに短かった。

### 実行時間

モジュール化された未最適化MapReduceは、ジョブ起動、中間データI/O、エンコード・デコードの増加により遅くなった。FlumeJavaは、論理的にモジュール化された記述を維持しながら、手動最適化MapReduceに近い実行時間を達成した。

特にBuild LogsとSiteDataでは手動最適化版との差が小さく、FlumeJavaの自動融合が有効に機能している。一方、Ads Logsでは手動最適化MapReduceの方がFlumeJavaより高速であり、完全に同等ではない。

## 6. 考察

本論文の最大の強みは、**プログラミング抽象と物理実行計画を分離しながら、後から融合によって性能を回復する**という設計にある。開発者は保守しやすい論理演算を記述し、最適化器がジョブ境界を決定する。ライブラリとして既存のJavaへ組み込んだ点も、既存のIDE、デバッガ、型、ライブラリを利用できるため、独自言語より導入障壁が低い。

また、静的解析でソースコードから実行計画を推測するのではなく、Javaプログラムを実行して具体的なグラフを構築するため、複雑な条件分岐やデータ構造を利用したパイプラインにも対応できる。この「ホスト言語でグラフを構築し、実行前に最適化する」という方式は、実用上きわめて重要な設計判断である。([UWコンピュータサイエンスユーザーページ][2])

一方、以下の限界がある。

* ユーザー関数`DoFn`の内部を解析せず、融合後のコード生成やインライン化も行わない。
* 共通部分式除去や不要な`groupByKey()`の削除が未実装である。
* 論文時点ではバッチ処理と単一パイプラインが中心で、継続処理や複数パイプライン横断最適化は未対応である。
* 高水準抽象がワーカやMapper／Reducerの詳細を隠すため、熟練者による低水準最適化を表現しにくい。
* Java当時の匿名関数や異種タプルの扱いが弱く、記述が冗長になる。([UWコンピュータサイエンスユーザーページ][2])

評価方法にも注意が必要である。実験はGoogle内部の基盤とワークロードに限定され、再現可能性は低い。また、5回中の最短時間を用い、調整時間を除外しているため、平均的なエンドツーエンド性能より有利な数値になっている可能性がある。利用者満足度についても、厳密なユーザースタディではなく、利用ログと定性的報告が中心である。

## 7. まとめ

FlumeJavaは、少数の不変並列コレクションと演算、遅延評価、データフロー最適化、MapReduceへの自動変換を組み合わせたJavaライブラリである。

論文の主要な成果は次の3点に集約できる。

1. 複数段MapReduceを通常のJavaプログラムに近い形で記述可能にした。
2. 演算融合とMSCR変換により、手動最適化版と同数程度の物理ステージを自動生成した。
3. 数百人規模の内部利用とペタバイト規模の実運用によって、研究プロトタイプを超えた実用性を示した。

高水準API、遅延評価、DAG最適化、ローカル・分散実行の切替えという設計は、その後のデータ並列処理システムを考えるうえでも重要な原型と評価できる。

## 8. 総評

**模擬査読結果：Strong Accept（採択）**

問題設定は実務上明確であり、解決策も少数のAPI、遅延評価、演算融合、MSCRという一貫した設計にまとめられている。特に、単なるAPI提案ではなく、最適化器、実行系、開発支援機能まで実装し、実運用データとTB級ベンチマークで評価した点は非常に強い。

評価の再現性、統計的厳密さ、エンドツーエンド時間の扱いには弱点があるものの、実システム論文としての新規性、有用性、完成度がそれらを上回る。実際に本論文はPLDI 2010へ採録されているが、個別のレビュースコアや査読コメントは公開されていない。
