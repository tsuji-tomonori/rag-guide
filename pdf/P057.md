# 1. はじめに

本論文は、LLMの回答を細かな事実単位に分解し、参照文書と照合することでハルシネーションを検出するフレームワーク **RefChecker** を提案している。中心的なアイデアは、回答中の主張を `(主語, 関係, 目的語)` という **claim-triplet** で表し、個々の三つ組を検証することである。リンク先は2024年5月公開のarXiv初版であり、その後、改訂版が **“Knowledge-Centric Hallucination Detection”** という題名でEMNLP 2024本会議に採択されている。以下では主にリンク先のarXiv版を説明する。([arXiv][1])

論文の主な貢献は、①claim-tripletという検証単位、②3種類の文脈条件を含む人手アノテーション済みベンチマーク、③商用・オープンソースモデルの両方を利用できる抽出・検証パイプラインの三点である。

# 2. 背景

従来のハルシネーション検出には、回答全体、文、あるいは文より短いフレーズを検証単位とする方法がある。しかし、回答全体では一部分だけに含まれる誤りを見逃しやすく、文単位では複数の事実が混在する。一方、FActScoreなどが用いるサブセンテンスは境界が曖昧で、主張同士が重複する場合がある。そこで本論文は、知識グラフで用いられる三つ組を、構造が明確な事実単位として採用する。

また、論文はハルシネーション検出を次の3条件に分けている。

* **Zero Context（ZC）**：回答生成時に外部文脈を与えず、LLM内部の知識だけで答える。
* **Noisy Context（NC）**：RAGのように、ノイズを含み得る検索文書を与える。
* **Accurate Context（AC）**：要約や文書ベースQAのように、基本的に正確な入力文書を与える。

ここでいうZero Contextは「回答を生成するLLMに文脈を与えない」という意味であり、検出器側はNaturalQuestionsの正解段落を参照情報として使用する。したがって、完全な参照不要型検出ではない。

# 3. 提案手法

RefCheckerは、次の2段階を中心とするモジュール型パイプラインである。

まず **Extractor (E)** がLLM回答をclaim-tripletへ分解する。例えば、

> 「I Ran (So Far Away) はA Flock of Seagullsによって演奏され、1983年に発売された」

という回答から、

* `(I Ran (So Far Away), originally performed by, A Flock of Seagulls)`
* `(I Ran (So Far Away), released in, 1983)`
* `(A Flock of Seagulls, is, English new wave band)`

のような事実を抽出する。

次に **Checker (C)** が各三つ組を参照文書と比較し、以下の3クラスに分類する。

* **Entailment**：参照文書から支持される。
* **Contradiction**：参照文書と矛盾する。
* **Neutral**：参照文書だけでは真偽を判断できない。

Neutralを明示的に分ける点は、単純な「正しい／誤り」の二値分類より実用的である。ただしNeutralは「誤り」ではなく、あくまで参照情報不足である。

ExtractorにはGPT-4、Claude 2、Mixtral、Mistralなどを利用できる。著者らはMixtral 8x7Bを教師として、10,000件の応答でMistral 7Bを蒸留・教師あり微調整した **Mistral-SFT extractor** も構築している。CheckerにはGPT-4やClaude 2のほか、AlignScore、RoBERTa-NLI、Mistralの内部表現にSVMやMLPを接続する **RepC** が利用される。長い参照文書は小型Checkerの入力制限に合わせて約200トークン単位に分割し、判定を集約する。

最後に、claim-tripletごとの結果を集約し、回答中のEntailment、Neutral、Contradictionの割合や、回答全体のラベルを求める。厳しい集約規則では、一つでもContradictionがあれば回答全体を矛盾ありと判定する。([ar5iv][2])

# 4. 実験方法

ベンチマークは計300問で、各文脈条件から100問ずつ構成される。

| 条件               | データ                  | 主なタスク         |
| ---------------- | -------------------- | ------------- |
| Zero Context     | NaturalQuestions     | クローズドブックQA    |
| Noisy Context    | MS MARCO             | RAG           |
| Accurate Context | databricks-dolly-15k | 要約、閉じたQA、情報抽出 |

GPT-4、GPT-3.5-Turbo、InstructGPT、Claude 2、Llama 2 70B Chat、Falcon 40B Instruct、Alpaca 7Bの7モデルから回答を収集したため、合計は2,100回答となる。約11,000個のclaim-tripletを6名のNLP専門家が判定し、23%を二重アノテーションした結果、一致率は95.0%だった。

ただし、単純なランダム抽出ではない。各条件について1,000候補を用意し、4モデルの回答をFalcon-40B-Instructで事前評価して、ハルシネーションが多いと推定された上位100件を採用している。これは識別困難な事例を多く含む一方、実世界における通常のハルシネーション発生率を表す標本ではない。

主な評価は次のように行われる。

1. 回答・文・サブセンテンス・tripletの各粒度を比較し、最終的に回答単位へ集約した3クラスMacro-F1を測定する。
2. SelfCheckGPT、FActScore、FacToolと比較し、人間が算出したハルシネーション率とのPearson・Spearman相関を測定する。
3. Extractorは適合率・再現率・F1・処理速度、Checkerは3クラスAccuracyとMacro-F1で評価する。

# 5. 結果

## 粒度の比較

claim-triplet単位の検証は、回答全体を直接検証する場合より平均で約10ポイント高いMacro-F1を示した。文単位は回答単位より約5ポイント改善したが、サブセンテンス単位は文単位より約3.5ポイント低下した。7種類のCheckerのうちClaude 2を除く多くの設定でtripletが最良となっており、構造化された細粒度主張の有効性を支持している。

## 既存手法との比較

arXiv版における、人間評価との相関は以下の通りである。数値はPearson／Spearmanである。

| 条件               | RefChecker：Claude 2 + GPT-4 |   各指標で最良の既存手法 |
| ---------------- | --------------------------: | ------------: |
| Zero Context     |               83.69 / 82.99 | 59.78 / 62.57 |
| Noisy Context    |               53.14 / 47.89 | 46.35 / 38.69 |
| Accurate Context |               60.99 / 58.96 | 40.23 / 32.82 |

この結果から、RefCheckerは最良の比較手法を6.8～26.1ポイント上回ったと報告されている。完全なオープンソース構成であるMistral-SFT + AlignScoreも、ZCで75.81／74.16、NCで53.88／45.09、ACで46.34／43.22を記録している。

## 各コンポーネント

Mistral-SFT extractorは自動評価でF1 86.4、1反復1.7秒であり、GPT-4 extractorのF1 89.3、8.7秒にかなり近い精度を、より高速に実現した。

Checker単体では、RepC-LEの2層MLP構成が平均Accuracy 81.27、Macro-F1 60.80で最良だった。ただしGPT-4のMacro-F1 59.80、AlignScoreの59.45との差は小さく、Checker自体の絶対性能には改善余地が残る。特にNoisy Contextでは、長い参照を分割する必要があり、全モデルの性能が低めである。

人手評価では、Contradictionの割合がZCの約25%からNCの約13%、ACの約6%へ低下した。これは、適切な文脈を与えることがLLMの事実性を大きく改善することを示している。

# 6. 考察

本研究の最大の長所は、検証単位を明確に定義した点である。claim-tripletは文より細かく、自由形式のサブセンテンスより構造的であり、「回答のどの関係が参照と矛盾したか」を示しやすい。また、Extractor、Checker、集約規則を交換可能にした設計と、オープンソース実装を提供した点も実用上の価値が高い。

一方、主な課題は以下である。

**第一に、tripletは複雑な意味を単純化し過ぎる。** 時点、条件、数量、否定、因果関係、複数文をまたぐ推論などを、単純な三つ組だけで正確に表すのは難しい。論文自身も「ある人物が米国大統領である」という主張は時点によって真偽が変わる例を挙げている。推論過程や長距離依存に起因するハルシネーションにも弱い。

**第二に、NeutralとContradictionの扱いに注意が必要である。** Neutralは「参照に書かれていない」だけで、実世界では正しい可能性がある。しかし既存手法との相関比較では、NeutralとContradictionの比率を合わせてハルシネーション率としている。この定義は「参照に根拠がない生成」を評価するRAGでは妥当だが、「客観的に誤っている内容」を測る指標とは一致しない。

**第三に、ベンチマークの代表性が限定的である。** 300問は人手による細粒度評価としては有用だが、ハードケース選択によって意図的に難しい例へ偏っている。そのため、文脈条件別のContradiction率を一般的なLLM利用環境の発生率として解釈することはできない。また、アノテータ間一致は高いものの、二重アノテーションは23%に限定され、κ係数などの偶然一致を補正した指標は示されていない。

**第四に、評価にLLM判定が多く介在する。** Extractor評価の妥当性確認は30サンプルであり、その後の大規模評価にはGPT-4 Turboを用いている。強力なモデルを評価器として利用する実用性はあるが、同種モデルに有利な評価バイアスや、少数の人手検証から一般化してよいかという問題が残る。

さらに、参照文書中にある重要情報を回答が欠落させた場合は評価対象外であり、完全性や有用性は測定しない。事実だけを短く並べた回答が高得点になりやすい可能性があり、実運用ではhelpfulnessやcoverageとの併用が必要である。

# 7. まとめ

RefCheckerは、LLM回答をclaim-tripletへ分解し、参照文書との関係をEntailment・Contradiction・Neutralの3クラスで判定する、細粒度ハルシネーション検出フレームワークである。従来の回答・文・サブセンテンス単位より高い性能を示し、人間によるLLMランキングやハルシネーション率とも高い相関を得た。特に、検証粒度の定式化、複数の文脈条件を含むベンチマーク、商用モデルに依存しない構成の提示が重要な成果である。

ただし、tripletでは表しにくい時間・条件・推論関係、Neutralの意味、参照取得の信頼性、ハードケース中心のベンチマーク、Checkerの絶対性能といった課題が残っている。RefCheckerは万能な真偽判定器というより、**参照文書に対する回答の根拠性を細粒度に診断する基盤** と位置付けるのが適切である。

# 8. 総評

**実際の査読結果：採択。** 本研究の改訂版は、題名を **“Knowledge-Centric Hallucination Detection”** として、EMNLP 2024のMain Conferenceに採択・出版された。ACL Anthologyでは本会議論文、6953–6975頁、DOI付きで収録されている。個々の査読者スコアや査読コメントは、確認できる公開情報には含まれていない。([ACL Anthology][3])

**独立した査読判定：Weak Accept／採択推奨。**

採択を支持する理由は、研究課題の重要性、claim-tripletという分かりやすい定式化、2,100回答・約11,000主張に対する人手評価、オープンソース構成を含む広い実験、既存手法を上回る一貫した結果である。一方、triplet表現の限界、Neutralと誤情報の混同、ハードケース選択による標本偏り、30サンプルに基づくExtractor評価、Noisy ContextでのChecker性能などは、強い採択ではなくWeak Accept相当と判断する理由になる。

なお、リンク先のarXiv初版は既存手法に対する改善を6.8～26.1ポイントと報告する一方、EMNLP採録版では構成更新後の結果として18.2～27.2ポイントと報告している。再現や引用の際には、arXiv初版と査読後版のどちらを対象とするかを明記すべきである。([arXiv][1])
