# P118：Text Embeddings by Weakly-Supervised Contrastive Pre-training

## 1. はじめに

本論文は、検索・クラスタリング・分類・意味類似度計算などに共通して利用できる汎用テキスト埋め込みモデル **E5**（EmbEddings from bidirEctional Encoder rEpresentations）を提案している。

中心的なアイデアは、少量の人手ラベルや単純なデータ拡張に依存せず、Web上の自然な対応関係から得られる大規模テキストペアを弱教師信号として利用することである。著者らは約13億件の候補ペアから約2.7億件を選別した **CCPairs** を構築し、大規模な対照学習を行う。E5は単一ベクトル表現を出力するため、文書を事前にベクトル化でき、高速な近似最近傍探索にも適用できる。([arXiv][1])

---

## 2. 背景

BERTなどの事前学習済み言語モデルは転移可能な表現を生成できるが、通常の事前学習目的は、検索や文対応で必要となる「文章全体を表す単一ベクトル」の品質を直接最適化していない。

従来手法は主に次の二つに分かれる。

* NLIやMS MARCOなどの人手ラベル付きデータで学習する教師あり手法
* ICT、ランダムクロッピング、隣接文などから疑似正例を生成する自己教師あり手法

教師あり手法は高性能だが、利用できるラベルが限定され、特定タスクへの偏りが生じやすい。一方、自己教師あり手法は大量のデータを生成できるものの、正例ペアの意味的品質が低く、追加の教師あり学習なしではBEIR上でBM25を超えられないことが多かった。

そこで本論文は、単純に人工的な正例を作るのではなく、質問と回答、投稿とコメント、論文と引用先など、Web上に元から存在する半構造的な関係を弱教師信号として利用する。

---

## 3. 提案手法

### 3.1 CCPairsの構築

CCPairsは、多様な半構造化データからテキストペア ((q,p)) を収集したデータセットである。フィルタ後の主な内訳は以下の通りである。

| データ源           | ペアの例                    |     件数 |
| -------------- | ----------------------- | -----: |
| Wikipedia      | エンティティ名・節タイトル／本文        | 2,400万 |
| Reddit         | 投稿／高評価コメント              | 6,000万 |
| Common Crawl   | ページタイトル／本文              | 6,900万 |
| Stack Exchange | 質問／回答                   | 1,900万 |
| S2ORC          | 論文タイトル／要旨、引用関係          | 9,000万 |
| ニュース           | タイトル・要約／本文              |   300万 |
| その他            | WikiHow、Yahoo Answersなど |   600万 |

単純なヒューリスティックによる前処理後には約13億ペアが残るが、その中には、表面的には対応していても意味的関連性が低いペアが多数含まれる。

そこで、まず13億ペアで暫定モデルを学習し、各正例候補を100万件のランダム文書とともに順位付けする。元の相手文書が上位 (k=2) に入った場合のみ、そのペアを保持する。この **consistency-based filter** により、最終的に約2.7億ペアへ絞り込む。

### 3.2 対照事前学習

E5は、クエリと文書を同一のTransformerで符号化する共有パラメータ型のbi-encoderである。各トークンの最終層表現を平均プーリングし、固定次元ベクトル (E_q,E_p) を得る。

類似度は、温度付きコサイン類似度で定義される。

[
s_\theta(q,p)=\frac{\cos(E_q,E_p)}{\tau}
]

実験では (\tau=0.01) である。学習目的にはInfoNCE型の対照損失を用い、同一ミニバッチ内の他の文書を負例とする。

入力には役割を明示するため、クエリ側に `query:`、文書側に `passage:` という接頭辞を付ける。この非対称化により、質問と回答のように役割が異なる検索タスクへの適合を図っている。

### 3.3 教師ありファインチューニング

弱教師あり事前学習後、必要に応じて以下のラベル付きデータで追加学習する。

* MS MARCO：文書検索
* Natural Questions：質問応答検索
* NLI：含意・矛盾関係

MS MARCOとNQでは、マイニングしたhard negativeとcross-encoder教師モデルのスコアを利用する。目的関数は、教師モデルからのKLダイバージェンスと対照損失の線形結合である。

[
D_{\mathrm{KL}}(p_{\mathrm{CE}},p_{\mathrm{student}})
+\alpha L_{\mathrm{cont}}
]

NLIでは矛盾文をhard negativeとして利用する。これにより、検索性能だけでなく、意味類似度や分類性能も補強する。

## 4. 実験方法

モデルは3規模で構築されている。

| モデル      | 層数 |  隠れ次元 | パラメータ数 |
| -------- | -: | ----: | -----: |
| E5-small | 12 |   384 | 3,300万 |
| E5-base  | 12 |   768 |   1.1億 |
| E5-large | 24 | 1,024 |  約3.3億 |

事前学習ではバッチサイズを32,768とし、20,000ステップ、約2.5エポック学習する。small、base、largeの学習には、それぞれ16、32、64基のV100 GPUを使用し、largeでは約2日を要した。ファインチューニングは8 GPU、バッチサイズ256、3エポックで行われる。事前学習時の最大入力長は128、ファインチューニング時は192、評価時は最大512トークンである。

評価には次を用いる。

**BEIR**では、公開取得可能な15検索データセットを使用し、主指標をnDCG@10とする。比較対象はBM25、SimCSE、LaPraDoR、Contriever、ANCE、ColBERT、GTRなどである。

**MTEB**では、英語部分の56データセットを使用する。分類、クラスタリング、ペア分類、再ランキング、検索、意味類似度、要約評価という7タスク群を含み、タスクごとにAccuracy、V-measure、Average Precision、MAP、nDCG@10、Spearman相関などを用いる。

## 5. 結果

### 5.1 主要結果

| 設定       | モデル             |   BEIR平均 |   MTEB平均 |
| -------- | --------------- | -------: | -------: |
| 疎検索      | BM25            |     41.7 |        ― |
| 教師なし     | SimCSE-BERT     |     20.3 |     45.5 |
| 教師なし     | E5-PT-base      |     42.9 |     55.6 |
| 教師なし     | E5-PT-large     | **44.2** | **56.6** |
| 教師あり     | GTR-large       |     47.0 |     58.3 |
| 教師あり     | E5-base         |     48.7 |     60.4 |
| 教師あり     | E5-large        | **50.0** | **61.4** |
| 48億パラメータ | GTR-xxl         |        ― |     59.0 |
| 48億パラメータ | Sentence-T5-xxl |        ― |     59.5 |

E5-PT-baseは、BEIR平均でBM25を1.2ポイント上回った。著者らによれば、ラベル付きデータを利用しない単一のdense embeddingモデルがBEIR平均でBM25を上回った初めての結果である。E5-PT-largeでは差が2.5ポイントまで拡大した。

ファインチューニング後のE5-largeはBEIRで50.0、MTEBで61.4を記録した。特にE5-baseは1.1億パラメータでMTEB 60.4を達成し、約48億パラメータのGTR-xxlおよびSentence-T5-xxlを上回っている。

### 5.2 タスク別の傾向

E5-largeのMTEB結果は、分類73.1、ペア分類85.9、再ランキング56.5、検索50.0、STS 82.1など、多くのタスクで高い。ただし、クラスタリングでは事前学習のみのE5-PT-largeが44.3であるのに対し、ファインチューニング後は43.3へ低下した。要約評価も32.6から31.0へ低下しており、教師あり学習がすべての用途を一様に改善するわけではない。

SST-2のゼロショット感情分類では、通常のBERT-baseが58.9%であるのに対し、E5-largeは85.3%を達成した。ただし、タスク専用に完全ファインチューニングしたBERT-largeの94.9%には及ばない。

### 5.3 アブレーション

バッチサイズを1,000から8,000、32,000へ増加させると、6つのBEIRデータセットの平均は45.8、50.2、51.6と継続的に改善した。大量のin-batch negativeが性能上重要であることを示している。

データフィルタリングも大きな効果を持つ。100万ペアの実験では平均が34.9から40.7へ改善した。また、未フィルタの全データはフィルタ後データの約4倍以上あるにもかかわらず、平均50.0にとどまり、フィルタ後の51.6を下回った。

負例数を増やすpre-batch negativeやMoCoは、それぞれ43.3、45.5であり、単純な32K in-batch negativeの51.6より悪かった。したがって、単に負例数を増やすのではなく、現在のモデルで生成された新鮮な負例を使うことが重要と考えられる。

## 6. 考察

本論文の最も重要な知見は、複雑な新規損失関数ではなく、**対照学習に投入する正例ペアの品質と多様性が汎用埋め込み性能を決定する**という点である。モデル構造自体は共有bi-encoder、平均プーリング、InfoNCEという比較的単純な構成であり、性能向上の中心はCCPairsと大規模in-batch negativeにある。

一方、consistency-based filterには、暫定モデルが高く評価できるデータだけを残すため、初期モデルの偏りを再生産する危険性がある。どの種類のペアが除外されやすいか、少数ドメインや難しい意味関係が失われていないかについては十分に分析されていない。

また、評価データとの重複は完全一致で除外されているが、言い換えや近重複まで排除できているとは限らない。Common Crawl、Reddit、Wikipediaなどを大規模に利用する以上、意味的なデータ汚染をより厳密に検証する余地がある。

教師ありファインチューニングには明確なトレードオフがある。MS MARCOとNQは検索を、NLIはSTSと分類を改善するが、クラスタリングや一部のドメイン外検索では性能が低下する。したがって、「汎用埋め込み」を単一の総合平均だけで評価するのではなく、実運用タスクに応じて事前学習版と教師あり版を使い分ける必要がある。

さらに、32Kバッチと最大64基のV100を必要とするため、学習再現性は計算資源に強く依存する。評価コードやモデルチェックポイントは実用上有用だが、CCPairsと同等のデータを収集・選別して事前学習全体を再現することは容易ではない。長文についても、評価時に512トークンへ制限され、長い文書を単一ベクトルに圧縮する限界は残る。

---

## 7. まとめ

E5は、Web上の自然なテキスト対応関係を弱教師信号として収集し、品質フィルタリングと大規模対照学習を組み合わせた汎用テキスト埋め込みモデルである。

主要な成果は以下である。

* 約13億ペアから約2.7億件の高品質CCPairsを構築
* 単純なbi-encoderとin-batch negativeだけで高いゼロショット性能を実現
* 教師なしE5がBEIR平均でBM25を上回る
* 1.1億パラメータのE5-baseが、約48億パラメータの既存モデルをMTEBで上回る
* データ品質、バッチサイズ、教師データの多様性が性能を大きく左右する

アルゴリズム上の新規性よりも、データ中心の設計と大規模な実証によって、後続の汎用埋め込み研究に大きな影響を与えた研究と位置付けられる。

---

## 8. 総評

**査読判定：Weak Accept（採択寄り）**
**評点：7／10、確信度：4／5**

実験規模、評価範囲、実用性は非常に高く、BEIRとMTEBの双方で一貫した改善を示している。特に、モデル規模ではなく、弱教師データの品質と多様性によって大規模モデルを上回った点は重要な貢献である。データフィルタリング、バッチサイズ、教師データ構成、負例方式に関するアブレーションも、提案の有効性を適切に裏付けている。

一方で、手法そのものは既存のInfoNCE型対照学習を基盤としており、アルゴリズム上の新規性は限定的である。CCPairsの完全再現に必要なデータと計算資源、近重複による評価汚染、Webデータの社会的偏り、統計的有意差や複数乱数種の報告不足も弱点となる。これらを踏まえるとStrong Acceptではなく、実証的価値を高く評価したWeak Acceptが妥当である。

なお、arXiv本文には「Work in progress」と記載されており、ここでの判定は公式会議の採否ではなく、本要約における独立した模擬査読結果である。([arXiv][1])
