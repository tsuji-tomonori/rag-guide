## 1. はじめに

**EnrichIndex: Using LLMs to Enrich Retrieval Indices Offline**は、Peter Baile Chen、Tomer Wolfson、Michael Cafarella、Dan Rothらによる検索技術の研究で、**COLM 2025採択論文**である。

本研究の中心的な発想は、LLMを検索時に毎回呼び出すのではなく、文書や表をデータベースへ登録する段階で一度だけ利用し、検索インデックス自体を意味的に拡張することである。これにより、専門用語や表構造のためにクエリとの表面的な類似度が低い対象でも、検索候補に入りやすくする。([OpenReview][1])

---

## 2. 背景

BM25のような語彙ベース検索は、クエリと文書の単語が一致する場合には強い。一方、Dense Retrievalは意味的な類似性を捉えられるものの、次のような検索には依然として弱い。

* 専門用語、略語、コード、数式を含む文書
* 列名だけでは用途が分かりにくいデータベース表
* 文書の「内容」ではなく「用途」を推論する必要がある検索
* クエリと正解文書の表現が大きく異なる検索

LLMによるクエリ拡張、文書拡張、再ランキングはこの問題を緩和できる。しかし、クエリが来るたびに多数の候補文書をLLMへ入力するため、レイテンシーと推論費用が大きい。また、再ランキングは第1段階の検索で取得された候補しか処理できないため、正解文書が候補集合から漏れると回復できない。

EnrichIndexは、この処理を**クエリ依存のオンライン処理から、文書依存のオフライン処理へ移す**ことで、第1段階検索のRecallを直接改善する。論文中の例では、通常のDense Retrieverで152表中36位だった正解表が、EnrichIndexによって2位まで上昇している。

---

## 3. 提案手法

### オフラインでのマルチビュー生成

各文書または表を、論文ではまとめて「object」と呼ぶ。EnrichIndexは、元の内容に対してGPT-4o-miniを用い、次の3種類の表現を生成する。

| 拡張表現         | 役割                                |
| ------------ | --------------------------------- |
| **Summary**  | 専門用語や表構造を自然言語で言い換え、内容を簡潔に表現する     |
| **Purpose**  | その文書・表が何のために使われ、どのような問題に役立つかを記述する |
| **QA pairs** | その対象から回答可能な質問と回答を最大20組生成する        |

Summaryは主に「内容の翻訳・平易化」、Purposeは「潜在的な利用目的の明示」、QA pairsは「将来入力されそうなクエリの事前生成」に相当する。

### 独立した4つのインデックス

生成結果を元文書へ単純に連結するのではなく、以下の4インデックスを独立して保持する。

1. 元の内容
2. Purpose
3. Summary
4. QA pairs

クエリ (q) と対象 (o) の最終スコアは、各インデックスから得られる類似度の重み付き和として計算する。

[
S(q,o)=
\alpha_1S(q,o_b)+
\alpha_2S(q,o_p)+
\alpha_3S(q,o_s)+
\alpha_4S(q,o_{qa})
]

ここで、(o_b) は元内容、(o_p) はPurpose、(o_s) はSummary、(o_{qa}) はQA pairsである。

この設計には、元データを変更しない、新しい拡張表現を後から追加できる、BM25・Dense Retriever・Hybrid Retrieverのいずれにも適用できるという利点がある。

## 4. 実験方法

### データセット

文書検索、暗黙的推論、表検索を含む5種類のデータセットで評価している。

| データセット            | 主な性質                           |                 規模 |
| ----------------- | ------------------------------ | -----------------: |
| BRIGHT            | StackExchange、コード、定理などの推論的文書検索 | 1,384問、1,145,164文書 |
| Spider2           | 複雑なデータベースの表検索                  |        258問、5,088表 |
| Beaver            | 企業データを対象とした表検索                 |          209問、463表 |
| Fiben             | 金融領域の表検索                       |          300問、152表 |
| Natural Questions | 一般的な文書検索                       |   3,452問、500,000文書 |

表検索では、データベース名、表名、列名、ランダムに選択した5行をMarkdown形式に変換した。また、検索を十分難しくするため、データセット内の個別データベースを統合した中央データベースから検索する設定を採用している。

### 比較手法

第1段階Retrieverとして、BM25、UAE-Large-V1、GTE-multilingual-base、Snowflake-arctic-l、およびBM25とDense Retrieverを組み合わせたHybrid方式を使用した。

強力なオンラインLLM方式との比較には**Judgerank**を用いた。JudgerankはBM25による候補取得後、クエリ拡張とクエリ依存の文書拡張を行い、Llama-3.1-8B-Instructで再ランキングする方式として再実装されている。元実装コードが公開されていなかったため、著者による再現実装である。

評価指標はRecall、Precision、F1、NDCGで、主結果ではRecall@10とNDCG@10を重視している。各データセットの20%を検証用、80%をテスト用とし、検証集合でスコア統合係数を調整した。

効率性については、実測時間や料金ではなく、オンライン処理時のLLM入力・出力トークン数を代理指標としている。オフライン拡張に要したトークンは効率比較に含めていない。

## 5. 結果

### 全体的な検索性能

論文全体の集計では、EnrichIndexを既存Retrieverへ追加することで、平均して次の改善が報告されている。

* Recall@10：**4.75ポイント向上**
* NDCG@10：**4.50ポイント向上**

特に表検索で効果が大きく、BM25に適用した場合、Recall@10が平均14.6ポイント、NDCG@10が11.8ポイント向上した。一方、一般的なNatural Questionsでは、それぞれ1.5ポイント、1.8ポイントの改善にとどまった。したがって、本手法は通常の自然言語文書よりも、表、コード、数式、専門用語など、**クエリとの意味的ギャップが大きい対象で強い**。

### オンラインLLM方式との比較

EnrichIndexとオンラインのクエリ拡張を組み合わせ、Judgerank型の文書別再ランキングを使用しない構成では、Judgerankに対して以下の差となった。

| タスク群         | Recall@10 |   NDCG@10 |
| ------------ | --------: | --------: |
| BRIGHT       |      +3.0 |      +2.7 |
| 表検索3データセット平均 |     +20.4 |     +18.4 |
| 上記2タスク群の平均   | **+11.7** | **+10.6** |

論文要旨に記載された11.7ポイント、10.6ポイントという改善値は、BRIGHT群と表検索群という2群の差を平均した値に対応している。全データセットをサンプル数で重み付けした改善値ではない点には注意が必要である。([OpenReview][1])

オンラインLLMトークン数は、Judgerankと比べてBRIGHTで平均351.0分の1、表検索で235.5分の1となり、両タスク群を平均した削減率が**293.3倍**と報告されている。ただし、これはオンライン処理のみの比較であり、インデックス構築時のGPT-4o-mini利用は含まれない。([OpenReview][1])

### アブレーション分析

BRIGHTでは、単独の拡張表現としてPurposeが最も有効で、Recall@10を2.0ポイント、NDCG@10を1.6ポイント改善した。暗黙的推論では、単なる要約よりも「この文書が何に使えるか」を明示することが重要と考えられる。

一方、表検索ではSummaryが最も有効で、Recall@10を5.77ポイント、NDCG@10を5.10ポイント改善した。表を自然言語へ変換することで、自然言語中心に学習されたRetrieverが処理しやすくなるためである。いずれのタスクでも、Purpose、Summary、QA pairsをすべて組み合わせた構成が最も高い性能を示した。

## 6. 考察

本研究の重要な知見は、LLMを必ずしも検索時の推論器として使用する必要はなく、**Retrieverが検索しやすい表現を事前に生成する装置として利用できる**ことである。LLMの知識や推論結果をインデックス側へコンパイルしておく発想と捉えられる。

特にPurpose表現は、元文書に明示されていない利用可能性を言語化する。QA pairsは、未知のクエリを完全に予測することはできないものの、検索対象が回答可能な質問空間を近似する。Summaryは、表、コード、専門用語などを、Retrieverが扱いやすい自然言語空間へ写像する。それぞれ異なる意味的ギャップを補完しているため、複数表現の併用が有効だったと考えられる。

一方、以下の制約がある。

**第一に、費用比較がオンライン処理に限定されている。** 約115万文書を含むBRIGHT全体について、Summary、Purpose、最大20組のQAを生成するには相当量の初期推論費用が必要となる。高いクエリ量を持つ静的コーパスでは償却できるが、文書が頻繁に更新される検索システムでは優位性が小さくなる。

**第二に、保存容量と構築時間の評価が不足している。** 元インデックスに加えて3インデックスを保持するため、埋め込み保存量、インデックス構築時間、メモリ消費が増える。しかし論文では、これらを含むエンドツーエンドのシステムコストは十分に報告されていない。

**第三に、効率性はトークン数による推定である。** 実際のレイテンシー、スループット、API料金、インデックス検索時間を直接測定したわけではない。293.3倍という値を、システム全体が293.3倍高速または安価になるという意味に解釈すべきではない。

**第四に、比較対象の再現性に不確実性がある。** Judgerankは公式コードではなく著者による再実装であり、計算資源の制約からLlama-3.1-8Bのみが使用されている。より大きな70Bモデルや、最適化された公式実装との差は検証されていない。

そのほか、拡張LLMがGPT-4o-miniのみに限定されていること、生成内容の幻覚や誤った用途記述を評価していないこと、最終的なRAG回答精度ではなく検索指標のみを評価していることも今後の課題である。

---

## 7. まとめ

EnrichIndexは、各文書・表についてSummary、Purpose、QA pairsをオフライン生成し、元内容を含む4つの独立インデックスのスコアを統合する手法である。

最大の利点は、LLMの意味理解能力を使いながら、文書ごとのオンラインLLM再ランキングを削減できる点にある。特に、専門文書、コード、数式、企業データベース、表検索など、クエリと検索対象の表現が一致しにくい状況で大きな効果を示した。

適用先としては、更新頻度が比較的低く、同じコーパスに多数の検索が発生する企業内検索、技術文書検索、データカタログ、Text-to-SQL用の表検索などが有望である。一方、ニュースやリアルタイムデータのように更新頻度が高い環境では、再拡張費用を含めた判断が必要となる。

## 8. 総評

### 査読結果

**COLM 2025に採択され、会議論文として公開されている。** OpenReview上でもCOLM 2025論文として掲載されているため、公開上確認できる査読結果は「採択」である。個別査読者の評点や詳細コメントは、取得できた公開ページでは確認できなかったため、具体的なスコアは記載しない。([OpenReview][1])

### 論文としての評価

本論文は、全く新しい検索アルゴリズムを提案したというより、文書拡張、合成質問生成、マルチインデックス検索という既存要素を、**オフライン処理とオンライン費用削減という明確な設計原則の下で統合した研究**である。したがって、理論的新規性は中程度だが、システム設計上の実用性は高い。

実験面では、文書と表、複数Retriever、クエリ拡張、再ランキング、アブレーション、共有重みの評価まで行っており、提案手法の有効性は比較的強く裏付けられている。特に、単純な手法で表検索を大幅に改善した点は説得力がある。コードと生成済み拡張データも公開されており、再現性への配慮も見られる。([GitHub][2])

一方、オフライン生成費用、追加ストレージ、実時間レイテンシー、生成内容の信頼性が十分に評価されておらず、「低コスト」という主張はオンラインフェーズに限定して読む必要がある。

**総合的には、採択は妥当である。** 新規性よりも、明快な問題設定、方式の単純さ、表検索での大きな改善、既存Retrieverへ追加可能なモジュール性を評価すべき実用志向の研究である。
