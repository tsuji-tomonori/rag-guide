# P115: Query Rewriting for Retrieval-Augmented Large Language Models

**著者:** Xinbei Ma, Yeyun Gong, Pengcheng He, Hai Zhao, Nan Duan
**掲載:** EMNLP 2023 Main Conference、Regular Long Paper、pp. 5303–5315

## 1. はじめに

本論文は、一般的なRAGの「検索してから回答する」という **Retrieve-then-Read** 構成に、検索クエリを書き換える処理を追加した **Rewrite-Retrieve-Read** を提案している。

中心的な問題意識は、ユーザーが入力した質問文と、検索エンジンが適切な文書を取得するために必要とするクエリとが必ずしも一致しないことである。特に、曖昧な質問、複数の条件を含む質問、マルチホップ質問は、そのまま検索しても関連文書を取得しにくい。

そこで本論文は、検索器や回答LLM自体を変更するのではなく、その間に位置する「クエリ」を最適化する。さらに、ChatGPTのような大規模モデルを毎回クエリ生成に使用する方法に加え、小型のT5モデルを強化学習で訓練する方法を示した。([ACL Anthology][1])

## 2. 背景

従来のRAGは、入力質問 (x) を検索器に与え、取得文書を質問とともにLLMへ入力する。

[
x \rightarrow \text{Retriever} \rightarrow \text{Documents}
\rightarrow \text{LLM Reader} \rightarrow \text{Answer}
]

しかし、自然言語の質問は人間に意図を伝えるための文章であり、検索エンジン向けに最適化された文字列ではない。たとえば、マルチホップ質問には複数の人物、関係、年代などが含まれ、検索上重要なエンティティや制約が埋もれる可能性がある。

従来研究の多くは、検索器を回答モデルに適応させる、回答LLMのプロンプトを工夫する、検索と推論を複数回反復するといった方向を採っていた。一方、ChatGPTのようなモデルはAPI経由でしか利用できず、パラメータを直接更新できない。そのため、本論文は、凍結された検索器とLLMの間を調整するモジュールとしてクエリ書換え器を導入する。

## 3. 提案手法

### Rewrite-Retrieve-Read

提案パイプラインは次の3段階で構成される。

[
x
\rightarrow
\underbrace{\text{Rewriter}}*{\tilde{x}\text{を生成}}
\rightarrow
\underbrace{\text{Retriever}}*{\text{文書検索}}
\rightarrow
\underbrace{\text{Reader}}_{\text{回答生成}}
]

ここで、(x) は元の質問、(\tilde{x}) は書き換えられた検索クエリである。書換え器は、不要な表現を除去したり、重要な固有名詞を強調したり、複雑な質問を複数の検索クエリに分解したりする。検索が不要と判断した場合は、クエリを生成しないことも許容される。

### 凍結LLMによる書換え

最も単純な構成では、ChatGPTにfew-shotプロンプトを与え、必要な検索クエリを生成させる。オープンドメインQAでは、「段階的に考え、必要な知識を検索するためのクエリを出力する」という指示を用いる。

この方式は追加学習を必要としないが、各質問について大規模LLMを呼び出すため、APIコストや推論時間が大きくなる。

### 学習可能な書換え器

より低コストな構成として、著者らは770MパラメータのT5-largeを書換え器として訓練する。訓練は次の2段階である。

**第1段階：ウォームアップ**

ChatGPTに訓練用質問を書き換えさせ、そのクエリを使って検索と回答を実行する。そのうち、最終回答が正解したクエリだけを擬似正解データとして残し、T5を教師あり学習する。

[
D_{\mathrm{warm}}
=================

{(x,\tilde{x})\mid \hat{y}=y}
]

つまり、「実際に最終回答を正解に導いた書換え」を模倣させる。

**第2段階：強化学習**

ウォームアップ後のT5を初期方策とし、PPOで追加学習する。生成したクエリを実際に検索器とLLMへ入力し、最終的なQA性能を報酬とする。

オープンドメインQAの主要な報酬は、概略として次の形である。

[
R_{\mathrm{task}}
=================

EM+\lambda_f F_1+\lambda_h Hit
]

ここで、`Hit`は取得文書中に正解文字列が含まれるかを表す。さらに、方策が初期モデルから過度に変化しないよう、KLダイバージェンスによる罰則を加える。

[
R
=

R_{\mathrm{task}}
-\beta D_{\mathrm{KL}}(\pi_\theta\Vert\pi_0)
]

この設計により、クエリの表面的な自然さではなく、「最終的に検索と回答が成功するか」を直接最適化する。

## 4. 実験方法

| 要素         | 設定                                        |
| ---------- | ----------------------------------------- |
| 書換え器       | ChatGPT、またはT5-large                       |
| 検索器        | Bing Search API                           |
| 文書構成       | 検索結果スニペット、またはWebページを解析後にBM25で段落選択         |
| 回答モデル      | ChatGPT（gpt-3.5-turbo）、Vicuna-13B         |
| オープンドメインQA | HotpotQA、AmbigNQ、PopQA                    |
| 多肢選択QA     | MMLUのHumanities、STEM、Other、Social Science |
| 評価指標       | EM、F1、取得文書のHit ratio                      |

比較対象は、検索を行わない`Direct`、元質問をそのまま検索する`Retrieve-then-read`、ChatGPTによる`LLM rewriter`、学習済みT5による`Trainable rewriter`である。

HotpotQAはマルチホップ質問、AmbigNQは曖昧性を解消した質問、PopQAは低頻度・ロングテール知識を多く含む。MMLUは訓練用と評価用に80対20で分割され、既知の汚染文書リストに含まれる検索結果は除外された。

T5のウォームアップでは学習率 (3\times10^{-5})、強化学習では学習率 (2\times10^{-6}) が用いられ、生成時には4-beam searchが使用された。

## 5. 結果

### オープンドメインQA

数値は `EM / F1` である。

| データセット   |        Direct | Retrieve-then-read |      LLM rewriter | Trainable rewriter |
| -------- | ------------: | -----------------: | ----------------: | -----------------: |
| HotpotQA | 32.36 / 43.05 |      30.47 / 41.34 |     32.80 / 43.85 |  **34.38 / 45.97** |
| AmbigNQ  | 42.10 / 53.05 |      45.80 / 58.50 |     46.40 / 58.74 |  **47.80 / 60.71** |
| PopQA    | 41.94 / 44.61 |      43.20 / 47.53 | **46.00 / 49.74** |      45.72 / 49.51 |

HotpotQAでは、元質問をそのまま検索すると、検索なしよりも性能が低下した。複雑なマルチホップ質問が検索クエリとして不適切で、ノイズを含む文書を取得したためと考えられる。一方、学習可能な書換え器は通常検索に対してEMを3.91ポイント、F1を4.63ポイント改善した。

AmbigNQでも、学習可能な書換え器が通常検索に対してEMを2.00ポイント、F1を2.21ポイント改善した。

PopQAではChatGPT書換えが最高性能であり、T5はわずかに下回った。PopQAでは「検索しない」という判断も重要であり、大規模LLMの方が検索要否を適切に判断できたことが一因と分析されている。

### 検索性能

AmbigNQにおけるBM25構成では、通常検索のHit ratioが76.4%だったのに対し、学習可能な書換え器では82.2%に上昇した。

| 手法                      |        EM |        F1 | Hit ratio |
| ----------------------- | --------: | --------: | --------: |
| Retrieve-then-read＋BM25 |     45.80 |     58.50 |      76.4 |
| LLM rewriter＋BM25       |     46.40 |     58.74 |      77.5 |
| Trainable rewriter＋BM25 | **47.80** | **60.71** |  **82.2** |

Hit ratioの改善幅に比べて最終QA性能の改善幅は小さい。この結果は、関連文書を取得できても、回答LLMがその情報を必ずしも十分に利用できないことを示している。

### MMLU

ChatGPTを回答器にした場合、LLM書換えはHumanities、STEM、Otherで通常検索を上回ったが、Social Scienceでは通常検索の78.2に対して76.4へ低下した。

Vicuna-13Bでは効果がより明確で、学習可能な書換え器は通常検索に対して、Humanitiesで40.2から43.2、Otherで55.2から59.3へ改善した。著者らは、能力の低い回答器ほど内部知識が少ないため、適切な外部知識による補完効果が大きいと解釈している。

## 6. 考察

本論文の最も重要な知見は、RAGの性能が検索モデルと回答モデルだけでなく、両者を接続するクエリ表現に強く依存することを実験的に示した点である。特に、マルチホップ質問では、質問をそのまま検索するよりも、エンティティや条件を抽出・分解した方が有効である。

また、小型モデルを「凍結された外部ツールとLLMを整合させるインターフェース」として学習する考え方は、検索以外のAPIやツール利用にも応用可能である。検索器と回答器のパラメータにアクセスできない場合でも、間にある書換え器だけを最適化できる点は実用上の長所である。

一方、以下の課題が残る。

**再現性:** Bing検索結果は時間、地域、検索エンジンの更新によって変化し、ChatGPTもバージョンやAPI設定に依存する。そのため、同じ実験を後から完全に再現することは難しい。著者自身も、Web検索は有料であり、取得内容が不整合、冗長、有害になり得ると述べている。

**統計的検証:** 結果は単一の点推定として報告されており、複数seedの標準偏差、信頼区間、有意差検定は提示されていない。特にPopQAのLLM書換えとT5書換えの差は非常に小さく、統計的に意味のある差かは判断できない。

**教師信号への依存:** 強化学習には正解回答、EM、F1、正解文字列のHitが必要である。したがって、正解ラベルが存在しない実運用環境へそのまま適用できるわけではなく、タスクごとの再学習が必要になる。

**アブレーション不足:** ウォームアップのみ、各報酬項の除去、KL項の除去、異なる書換え器サイズなどの比較が限定的であり、改善のどの程度がPPOそのものによるものかは十分に分離されていない。

**単発検索の限界:** 本手法は原則として一度の書換えと検索で回答する。最初の検索結果を読んで次の検索クエリを生成する反復型エージェントに比べ、複雑な多段推論への適応力は限定される。著者らも、汎化性とタスク特化のトレードオフ、および反復型LLMエージェントとの差を限界として挙げている。

## 7. まとめ

本論文は、従来のRAGにクエリ書換え段階を追加したRewrite-Retrieve-Readを提案した。ChatGPTを直接書換え器として使う方法と、ChatGPTの成功例でウォームアップしたT5-largeをPPOで最適化する方法を評価している。

実験では、特にHotpotQAやAmbigNQで検索精度と最終回答性能が改善し、「ユーザー質問をそのまま検索クエリとして使うこと」がRAGの明確なボトルネックになり得ることを示した。一方、検索要否の判断、Web検索の再現性、教師ラベルへの依存、統計的検証の不足などは未解決である。

## 8. 総評

### 公式の査読結果

本論文は、**EMNLP 2023 Main ConferenceにRegular Long Paperとして採択**されている。OpenReviewではQuestion Answeringトラックおよび「Large Language Models and the Future of NLP」テーマトラックへの投稿として登録されている。今回参照できた公開メタデータでは、個別査読者の数値スコアや詳細コメントまでは確認できなかった。([OpenReview][2])

### 再査読判定：Accept

採択を支持する。アイデア自体は単純だが、RAGの改善対象を検索器や回答器ではなく「検索前のクエリ」に置いた点は明快であり、複数データセットと複数回答モデルで一貫した改善を示している。ブラックボックスLLMと外部検索エンジンの間に小型の学習可能モジュールを配置する設計も、実用性と拡張性が高い。

ただし、「スケーラブルである」という主張は、実験したモデル、タスク、検索器の範囲に対してやや強い。厳密な再現実験、複数seedでの統計評価、APIコストと遅延の測定、報酬設計のアブレーション、より強い反復検索ベースラインが追加されれば、主張はさらに強固になる。

総合すると、**RAGにおけるクエリ書換えの重要性を広く認識させた先駆的研究であり、採択は妥当**である。現在の基準では実験統計と再現性に改善余地があるものの、研究上の着眼点と後続研究への影響という点で価値の高い論文である。
