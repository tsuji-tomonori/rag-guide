# P116　SPLADE v2

**論文名:** *SPLADE v2: Sparse Lexical and Expansion Model for Information Retrieval*
**著者:** Thibault Formal, Carlos Lassance, Benjamin Piwowarski, Stéphane Clinchant
**公開年:** 2021年、arXiv:2109.10086 ([arXiv][1])

## 1. はじめに

本論文は、ニューラル情報検索における**第一段階検索器**を、密ベクトルではなく語彙空間上の疎ベクトルとして構成するSPLADEを改良した研究である。

SPLADEは、BERTのMasked Language Modelヘッドを利用して、クエリと文書をWordPiece語彙上の疎ベクトルに変換する。これにより、ニューラルモデルによる意味的な語彙拡張を行いながら、従来の転置インデックスを利用できる。

SPLADE v2の主要な改良点は、次の三つである。

1. トークン方向の集約をsum poolingからmax poolingへ変更する
2. クエリ側のニューラル処理を省略したSPLADE-docを導入する
3. Cross-Encoderからの知識蒸留とhard negative miningを導入する

最終モデルのDistilSPLADE-maxは、MS MARCOだけでなく、未学習ドメインを含むBEIRでも高い性能を示した。([arXiv][2])

## 2. 背景

BM25などのBag-of-Words型検索は、転置インデックスによる高速な検索と、固有名詞などの完全一致に強い。一方、クエリと文書で異なる語が使われる**語彙ミスマッチ問題**に弱い。

密検索モデルは、クエリと文書を低次元の密ベクトルに変換することで意味的な類似性を捉えられるが、明示的な単語一致を失う可能性がある。また、検索には近似最近傍探索用のインデックスが必要となる。

SPLADEはこの中間に位置する。BERTの語彙数である30,522次元の空間を使用するが、実際に非ゼロとなる次元を少数に抑える。各次元がWordPieceに対応しているため、ニューラルモデルが推定した拡張語も、転置インデックス上の語として検索できる。([arXiv][2])

入力中のトークン (i) から語彙中のトークン (j) に対する重要度を

[
w_{ij}
======

\mathrm{transform}(h_i)^\top E_j+b_j
]

として計算する。元のSPLADEでは、最終的な語彙次元 (j) の重みを

[
w_j
===

\sum_{i\in t}
\log\left(1+\mathrm{ReLU}(w_{ij})\right)
]

によって集約していた。クエリと文書のスコアは、得られた疎ベクトル同士の内積である。([arXiv][2])

学習時にはランキング損失に加えて、頻繁に活性化される語彙次元を抑制するFLOPS正則化を使用する。クエリと文書には異なる正則化係数 (\lambda_q,\lambda_d) を設定でき、特にクエリ側を強く疎化することで検索時の計算量を抑える。([arXiv][2])

## 3. 提案手法

### 3.1 Max pooling

最も単純かつ重要な変更は、sum poolingを次のmax poolingに置き換えることである。

[
w_j
===

\max_{i\in t}
\log\left(1+\mathrm{ReLU}(w_{ij})\right)
]

つまり、語彙次元 (j) に対して、入力中のすべてのトークンからの寄与を足し合わせるのではなく、最も強い根拠だけを残す。

直観的には、sum poolingでは多数の入力トークンが弱く予測した語が過剰に強調される可能性がある。max poolingは、ある拡張語を最も明確に示唆するトークンの情報を保存し、弱い予測の累積を防ぐ。論文では詳細な原因分析までは行っていないが、実験上はほぼすべての疎性設定で、元のSPLADEより良好な性能・効率トレードオフを示している。([arXiv][2])

### 3.2 SPLADE-doc

SPLADE-docは、文書だけをSPLADEでエンコードし、クエリにはニューラルモデルを使用しない方式である。検索スコアは

[
s(q,d)=\sum_{j\in q}w_j^d
]

となる。

クエリ拡張やクエリ語のニューラル重み付けを行わず、入力クエリ中の語に対応する文書側の重みを合計する。文書ベクトルは事前計算できるため、オンライン検索時にTransformerを実行する必要がない。([arXiv][2])

### 3.3 DistilSPLADE-max

性能向上の中心となるのが、知識蒸留とhard negative miningである。

最初に、既存の蒸留用tripletを用いてSPLADEとCross-Encoder型rerankerを準備する。次に、学習済みSPLADEを使ってBM25より難しい負例を検索し、Cross-Encoderによって正例と負例のスコアを付ける。そのスコア差をMargin-MSE損失で模倣するように、新しいSPLADEを最初から学習する。

Cross-Encoderには `ms-marco-MiniLM-L-12-v2` が使用されている。この二段階の結果がDistilSPLADE-maxである。([arXiv][2])

### 3.4 全体の学習目的

通常のSPLADEでは、ランキング損失とクエリ・文書それぞれの疎性正則化を組み合わせる。

[
\mathcal{L}
===========

\mathcal{L}*{\mathrm{rank\text{-}IBN}}
+
\lambda_q\mathcal{L}*{\mathrm{reg}}^q
+
\lambda_d\mathcal{L}_{\mathrm{reg}}^d
]

ランキング損失では、明示的なhard negativeに加えて、同一バッチ内の別クエリの正例文書も負例として利用する。([arXiv][2])

## 4. 実験方法

主な評価条件は以下の通りである。([arXiv][2])

| データセット           | 条件                           | 評価指標                  |
| ---------------- | ---------------------------- | --------------------- |
| MS MARCO Passage | 約880万passage、dev 6,980 query | MRR@10、Recall@1000    |
| TREC DL 2019     | 人手による詳細な関連度を持つ43 query       | NDCG@10、Recall@1000   |
| BEIR             | 利用可能だった14データセット              | NDCG@10、平均zero-shot性能 |

モデルはDistilBERT-baseで初期化され、Adam、学習率 (2\times10^{-5})、6,000ステップのwarmup、バッチサイズ124で学習された。通常モデルは150,000 iteration、SPLADE-docは50,000 stepで学習され、入力長の上限は256である。

FLOPS正則化係数は、学習開始直後の影響を弱めるため、最初の50,000 stepまで二次関数的に増加させている。モデル学習には32GBのTesla V100を4基使用し、検索部分はPython arrayとNumbaによる独自実装で評価された。([arXiv][2])

比較対象にはBM25、DeepCT、doc2query-T5、SparTerm、COIL-tok、DeepImpactなどの疎検索モデルと、ANCE、TCT-ColBERT、TAS-B、RocketQAなどの密検索モデルが含まれる。ただし、比較モデルの一部は各原論文で報告された数値を引用している。([arXiv][2])

## 5. 結果

MS MARCOとTREC DL 2019における主要結果は以下の通りである。([arXiv][2])

| モデル              | MS MARCO MRR@10 | MS MARCO R@1000 | TREC NDCG@10 | TREC R@1000 |
| ---------------- | --------------: | --------------: | -----------: | ----------: |
| SPLADE           |           0.322 |           0.955 |        0.665 |       0.813 |
| SPLADE-max       |           0.340 |           0.965 |        0.684 |       0.851 |
| SPLADE-doc       |           0.322 |           0.946 |        0.667 |       0.747 |
| DistilSPLADE-max |       **0.368** |       **0.979** |    **0.729** |   **0.865** |

### Max poolingの効果

SPLADE-maxは元のSPLADEに対し、MS MARCOのMRR@10を0.322から0.340、TRECのNDCG@10を0.665から0.684へ改善した。絶対値では約1.8～1.9ポイントの改善である。

また、FLOPS正則化を変化させた比較でも、SPLADE-maxはほぼ一貫してsum pooling版より上側に位置しており、単に疎性を弱めて性能を上げただけではなく、同程度の計算量でより高い性能を示した。([arXiv][2])

### SPLADE-docの効果

SPLADE-docのMRR@10は0.322で、元のSPLADEと同じである。クエリ側のニューラル処理を完全に省いても、doc2query-T5の0.277を上回った。

さらに、文書あたり平均19個程度の非ゼロ次元という強い疎性でも、MRR@10は約0.296を維持した。ただしRecall@1000は0.946、TRECでは0.747まで低下しており、候補集合の網羅性には明確な代償がある。([arXiv][2])

### 蒸留の効果

DistilSPLADE-maxは、元のSPLADEに対しMS MARCO MRR@10を約14.3%、TREC NDCG@10を約9.6%相対改善した。

MS MARCOではRocketQAの0.370にほぼ並び、TCT-ColBERTの0.359やTAS-Bの0.347を上回っている。TREC DL 2019でもNDCG@10が0.729で、TCT-ColBERTの0.719、TAS-Bの0.717を上回った。([arXiv][2])

### BEIRのzero-shot性能

BEIRの平均zero-shot NDCG@10は次の通りである。

| モデル              | 平均zero-shot NDCG@10 |
| ---------------- | ------------------: |
| ColBERT          |               0.457 |
| BM25             |               0.456 |
| TAS-B            |               0.437 |
| SPLADE           |               0.451 |
| SPLADE-max       |               0.464 |
| DistilSPLADE-max |           **0.506** |

DistilSPLADE-maxは、論文中の集計で11データセットにおいて最高性能を記録した。特にHotpotQA、SciFact、TREC-COVID、FEVERなどで大きな改善が見られる。([arXiv][2])

## 6. 考察

第一に、SPLADE v2の結果は、**疎検索モデルではネットワーク構造だけでなく、語彙次元の集約方法が重要である**ことを示している。sumからmaxへの変更は非常に単純だが、複数トークンから生じる弱い拡張語の累積を抑え、各語彙次元に対する最も強い文脈的根拠を残していると解釈できる。

第二に、最大の性能向上はmax pooling単体ではなく、蒸留とhard negative miningによって得られている。したがって、SPLADE v2の成功はモデル表現だけでなく、強い教師モデル、負例選択、学習目的の設計に大きく依存している。

第三に、SPLADE-docは実運用上重要な派生モデルである。クエリ処理にTransformerを必要としないため、厳しいレイテンシ制約がある環境に適している。一方、Recall@1000の低下から、後段rerankerに十分な候補を渡す用途では慎重な調整が必要である。([arXiv][2])

本論文の主な弱点は、効率性の評価がFLOPSという代理指標と非ゼロ次元数に偏っている点である。実際のレイテンシ、QPS、インデックス容量、メモリ使用量を共通の検索エンジン上で比較していないため、「実環境でどの程度高速か」は十分に確定できない。

また、BEIR評価は当時利用できなかったCQADupStack、BioASQ、Signal-1M、TREC-NEWS、Robust04を除いた部分集合である。TREC DL 2019も43クエリと小さく、統計的有意差、複数seedによる分散、信頼区間は報告されていない。蒸留、Margin-MSE、hard negative miningの効果も個別に分離されておらず、どの要素が改善を生んだのかは完全には特定できない。([arXiv][2])

## 7. まとめ

SPLADE v2は、ニューラルな意味表現と転置インデックスによる疎検索を組み合わせたSPLADEを、以下の三方向から強化した研究である。

* max poolingによる表現集約の改善
* 文書側のみをニューラル処理するSPLADE-doc
* 知識蒸留とhard negative miningによるDistilSPLADE-max

特にDistilSPLADE-maxは、密検索モデルに匹敵するin-domain性能と、それらを上回るBEIR zero-shot性能を示した。単純なアーキテクチャ変更と強力な学習戦略によって、learned sparse retrievalの有効性を明確に示した論文である。([arXiv][2])

**査読結果：Weak Accept（採択寄り）**

max poolingや蒸留そのものは既存技術であり、理論的な新規性は限定的である。一方、疎検索においてそれらを適切に組み合わせ、MS MARCO、TREC、BEIRで明確かつ大きな改善を示した点は高く評価できる。コード公開、主要な学習条件、性能・疎性のトレードオフも提示されている。

採択を強く支持するには、実レイテンシとインデックス容量の比較、統計的検定、蒸留とhard negative miningの詳細なアブレーションが不足している。しかし、learned sparse retrievalを有力な第一段階検索方式として定着させた実証的価値は大きく、総合的には採択相当と判断する。
