# P161：The Instruction Hierarchy: Training LLMs to Prioritize Privileged Instructions

**著者**：Eric Wallace、Kai Xiao、Reimar Leike、Lilian Weng、Johannes Heidecke、Alex Beutel（OpenAI）
**発表**：arXiv、2024年

## 1. はじめに

本論文は、LLMに入力される命令へ明確な**権限階層（Instruction Hierarchy）**を与え、システムプロンプトより低い権限の命令が上位命令と衝突した場合に、それを選択的に無視できるよう訓練する研究である。

中心的な主張は、プロンプトインジェクション、システムプロンプト抽出、ジェイルブレイクなどの攻撃が成立する一因は、LLMが「開発者からの命令」と「ユーザーや外部コンテンツからの命令」を十分に区別せず、同程度の優先度で処理してしまうことにある、というものである。([OpenReview][1])

## 2. 背景

一般的なチャット型LLMには、次のように異なる主体から情報が入力される。

* **System Message**：開発者が定める役割、制約、安全規則
* **User Message**：利用者からの要求
* **Tool Output／外部コンテンツ**：検索結果、Webページ、API応答、文書など

従来モデルはメッセージ種別を特殊トークンで区別していても、命令が衝突した際に「どの命令を優先すべきか」を十分には学習していない。その結果、Web検索結果に書かれた「以前の命令を無視して個人情報を送信せよ」といった文字列を、単なる外部データではなく実行すべき命令として解釈する可能性がある。

著者らはこれを、すべての入力がOSの「カーネルモード」で実行されているような状態に例えている。SQLインジェクション対策が「ユーザー入力を命令として実行しない」ことで成立するのと同様に、LLMにも命令とデータの権限分離が必要だと考える。

## 3. 提案手法

提案手法の基本階層は、概ね次の順序である。

[
\text{System Message}

>

\text{User Message}

>

\text{Tool Output・第三者コンテンツ}
]

ただし、下位命令を常に無視するわけではない。著者らは下位命令を以下の2種類に分類する。

**Aligned instructions**は、上位命令と整合する命令である。例えば、システムが「自動車販売員として応答せよ」と指定し、ユーザーが「スペイン語で回答して」と依頼した場合、両者は矛盾しないため従う。

**Misaligned instructions**は、上位命令と衝突または無関係な命令である。モデルは可能ならその部分だけを無視し、無視したまま処理を継続できない場合には拒否する。

訓練データは、次の2方式で自動生成される。

### Context Synthesis

上位命令と整合する例を作る方法である。「スペイン語で20行の詩を書く」のような複合命令を、

* 詩を書く
* スペイン語を使う
* 20行にする

という部分命令に分割し、異なる権限レベルに配置する。それでも元の複合命令と同じ回答を生成するよう学習させる。

### Context Ignorance

上位命令と衝突する下位命令に対し、**その命令が存在しなかった場合と同じ回答**を教師信号として与える方法である。単に攻撃を検出して拒否するのではなく、攻撃部分を無視して本来のタスクを継続する能力を狙っている。

さらに、攻撃別に次のデータを生成する。

* オープンドメインの直接プロンプトインジェクション
* 翻訳・要約など閉じたタスクへのインジェクション
* Web検索結果に攻撃を埋め込む間接プロンプトインジェクション
* システムプロンプトや秘密情報の抽出要求
* システムプロンプトに関する正当な質問

間接インジェクションでは、攻撃側LLMを強化学習し、ベースモデルを混乱させる文字列を検索結果へ挿入させている。一方、ジェイルブレイク用データは意図的に訓練に含めず、未見攻撃への汎化評価として利用する。

## 4. 実験方法

GPT-3.5 Turboを対象として、生成したInstruction Hierarchyデータと通常の能力維持用データを用い、教師ありファインチューニングとRLHFを実施した。

比較対象は、同様の能力維持用データだけで訓練され、Instruction Hierarchyデータを使用していないモデルである。評価は以下の3群に分けられる。

1. 訓練時に想定した攻撃
2. ツール経由攻撃、パスワード抽出、ジェイルブレイクなどの未見攻撃
3. 正常な命令まで拒否しないかを測る過剰拒否評価

TensorTrust、Gandalf、Jailbreakchatなどの既存データに加え、合成データや手作業の間接インジェクションを使用している。多くの評価では、回答が攻撃に従ったか、拒否したかをGPT-4に判定させている。TriviaQA、LAMBADA、HellaSwagでは、通常能力がベースラインと同程度であることも確認している。

## 5. 結果

主要結果と未見攻撃への汎化結果は以下の通りである。数値は高いほど堅牢である。

| 評価項目                              | ベースライン | 提案モデル |    改善 |
| --------------------------------- | -----: | ----: | ----: |
| Prompt Injection：Hijacking        |   59.2 |  79.2 | +20.0 |
| Prompt Injection：New Instructions |   89.6 |  93.7 |  +4.1 |
| ユーザーの競合命令                         |   62.2 |  92.6 | +30.4 |
| 間接攻撃：Webブラウジング                    |   77.5 |  85.0 |  +7.5 |
| システムメッセージ抽出                       |   32.8 |  95.9 | +63.1 |
| 未見のツール経由攻撃                        |   77.6 |  87.0 |  +9.4 |
| TensorTrustパスワード抽出                |   53.8 |  84.2 | +30.4 |
| Gandalfパスワード抽出                    |   51.8 |  73.7 | +21.9 |
| Jailbreakchatの危険要求                |   83.8 |  89.2 |  +5.4 |
| 既知のChatGPTジェイルブレイク                |   37.4 |  71.2 | +33.8 |

特にシステムプロンプト抽出では、堅牢性が32.8%から95.9%へ大幅に向上した。訓練に含めていないパスワード抽出やジェイルブレイクでも改善しており、個別の攻撃文字列を暗記しただけではなく、命令の優先順位という一般的規則をある程度学習した可能性が示されている。

一方、過剰拒否には明確な悪化が見られた。

* システムプロンプトに関する正当な質問：85.2% → 75.0%
* ジェイルブレイク風だが無害な要求：83.1% → 60.4%
* 通常の非競合命令：78.9% → 77.7%
* 境界的だが許可可能な要求：87.4% → 86.2%

特に「攻撃に見えるが実際には安全な入力」の処理が弱く、安全性と有用性の境界が保守的になっている。

## 6. 考察

本研究の最も重要な貢献は、複数の攻撃を個別問題としてではなく、**アクセス制御または権限分離の失敗**として統一的に捉えた点である。この考え方は、その後のLLMエージェント設計やモデル仕様にも直接つながる重要な枠組みである。

また、単にシステムプロンプトへ「上位命令を優先せよ」と記述するだけでは十分でないことも示されている。例えばHijackingでは、ベースライン59.2%に対し、階層を文章で説明するだけでは55.5%に低下したが、提案データで訓練すると79.2%となった。競合命令でも、プロンプトだけでは55.1%、訓練モデルでは92.6%だった。つまり、命令階層は推論時の注意書きではなく、学習段階でモデルの行動規範として定着させる必要がある。

ただし、以下の限界がある。

第一に、対象が独自のGPT-3.5 Turboであり、学習データ量、詳細なRLHF設定、モデルチェックポイントなど、完全再現に必要な情報が十分に公開されていない。

第二に、評価の多くがGPT-4判定に依存している。評価器が対象モデルと似た拒否傾向を持つ場合、堅牢性や過剰拒否の測定に系統的偏りが入る可能性がある。

第三に、固定ベンチマーク上の改善であり、提案モデルを観察しながら攻撃を最適化する適応的攻撃への耐性は明らかでない。

第四に、本実装ではWebページやツール出力内の命令を原則として無視している。しかし実用エージェントでは、Webページに書かれた操作手順など、外部情報中の正当な命令を実行すべき場合もある。

著者ら自身も、テキスト以外の画像・音声インジェクション、メッセージ種別専用埋め込みなどのアーキテクチャ改善、より強力な適応的攻撃への対策が今後必要であり、現モデルは依然として強力な攻撃に脆弱であると認めている。

## 7. まとめ

本論文は、LLMへの命令を「すべて同等な自然言語」として扱うのではなく、送信主体に応じた権限階層を設ける手法を提案した。

Context SynthesisとContext Ignoranceによる自動データ生成を通じて、GPT-3.5のプロンプトインジェクション、秘密情報抽出、未見ジェイルブレイクへの堅牢性を大幅に改善した。一方、攻撃に似た無害な入力を拒否する問題、再現性、適応的攻撃に対する検証不足が残されている。

## 8. 総評

本論文は、プロンプトインジェクションを単なる入力フィルタリング問題ではなく、**信頼境界と権限管理の問題として定式化した先駆的研究**である。考え方は単純だが一般性が高く、現在のLLMエージェントやツール利用型システムを考える上で非常に重要である。

ただし、トップ会議論文として見ると、独自モデルへの依存、再現可能な学習情報の不足、GPT-4評価への依存、適応的攻撃の不在が大きな弱点である。私の評価は、アイデアと実用的影響は高評価だが、公開された証拠だけでは安全性を一般化するには不十分なため、**Borderline～Weak Reject**である。

**査読結果**としては、ICLR 2025のSubmission #11878として公開査読を受けている。少なくとも確認できる公式レビュー1件では、**Soundness 1、Presentation 4、Contribution 3**であり、実験の興味深さは評価された一方、技術的妥当性には非常に厳しい点数が付けられた。([OpenReview][2])
