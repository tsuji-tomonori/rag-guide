# 1. はじめに

本論文 **「DIN-SQL: Decomposed In-Context Learning of Text-to-SQL with Self-Correction」** は、自然言語の質問とデータベーススキーマからSQLを生成するText-to-SQLタスクを扱う。著者は Mohammadreza Pourreza と Davood Rafiei である。

中心的な主張は、LLMにいきなりSQL全体を生成させるのではなく、**スキーマ対応付け、クエリ構造の判定、SQL生成、自己修正**へ問題を分解することで、ファインチューニングなしでも大幅な精度向上が得られるというものである。([arXiv][1])

# 2. 背景

従来のText-to-SQLでは、Spiderなどのデータセットで学習した専用モデルが高い性能を示していた。一方、GPT-4やCodexのfew-shot promptingは、学習を必要としないという利点があるものの、複数テーブルのJOINやネスト、スキーマの曖昧性を含む複雑な質問で精度が低下していた。

著者らはSpider訓練セットから500問を抽出し、Codex Davinciによる単純なfew-shot promptingの失敗を分析した。その結果、主な原因を次の6種類に分類している。

* スキーマリンキングの誤り
* JOIN対象・結合条件の誤り
* GROUP BYの誤り
* ネスト・集合演算の誤り
* 実行不能なSQL
* 条件、DISTINCT、DESC、集約関数などのその他の誤り

特に、関連するテーブル・列・値を正しく同定できない**スキーマリンキング**が最大の失敗要因であり、次いでJOINの誤りが多かった。

# 3. 提案手法

DIN-SQLは、以下の4モジュールを順番に実行するパイプラインである。

### 3.1 スキーマリンキング

質問中の表現を、対応するテーブル名、列名、条件値へ変換する。プロンプトにはSpider訓練セットから選んだ10例を含め、段階的な推論によって関連スキーマを抽出させる。

例えば「平均収容人数」という表現に対し、`average`という名前の列を誤選択するのではなく、`capacity`列に`AVG`を適用すべきことを明示的に判断させる。

### 3.2 クエリ分類・分解

質問を次の3クラスに分類する。

* **Easy**：単一テーブルで、JOINやネストが不要
* **Non-nested complex**：JOINは必要だが、サブクエリは不要
* **Nested complex**：JOIN、サブクエリ、UNION・INTERSECT・EXCEPTなどを含み得る

同時に、必要なテーブル集合と、ネストされた質問を構成するサブクエリを抽出する。分類結果は、後段で使用するプロンプトの複雑さを切り替えるルーターとして機能する。

### 3.3 SQL生成

難易度ごとに生成方法を変える。

* Easyでは、スキーマリンクを含む通常のfew-shot生成を行う。
* Non-nested complexでは、自然言語とSQLの間に**NatSQL**という中間表現を挿入する。
* Nested complexでは、サブ質問ごとのSQLを先に生成し、それらとNatSQL表現を用いて最終SQLを組み立てる。

すべての質問に長いChain-of-Thoughtを適用せず、複雑な質問だけに多段推論を用いる点が重要である。

### 3.4 自己修正

生成済みSQLをLLMへ再入力し、DISTINCT、DESC、集約関数、結合条件などを点検させる。

* **Generic prompt**：「このSQLはバグを含む」と仮定して修正させる
* **Gentle prompt**：誤りがあると断定せず、問題がないか確認させる

Codexではgeneric prompt、GPT-4ではgentle promptの方が有効だった。強いモデルに誤りの存在を断定すると、正しいSQLまで変更してしまう可能性がある。

# 4. 実験方法

使用モデルはGPT-4、Codex Davinci、Codex Cushmanで、いずれもOpenAI API経由で利用された。temperatureは0、最大出力トークン数は自己修正で350、その他のモジュールで600に設定された。

評価データは次の2つである。

* **Spider**：200データベース、10,181質問を含むクロスドメインText-to-SQLデータセット
* **BIRD**：95の大規模データベースと12,751の質問・SQLペアを含み、外部知識や実データ値を必要とする、より実践的なデータセット

Spiderでは実行結果の一致率である**Execution Accuracy（EX）**とSQL構造の一致率である**Exact Set Match（EM）**を使用する。BIRDではEXに加え、正しいSQLの実行効率も考慮する**Valid Efficiency Score（VES）**を使用する。BIRDのプロンプトにはテーブルのサンプル行と、データセットが提供する外部知識も入力された。

# 5. 結果

主要結果は以下の通りである。

| データセット      | 手法                |        EX |        EM／VES |
| ----------- | ----------------- | --------: | ------------: |
| Spider test | DIN-SQL＋GPT-4     |  **85.3** |       EM 60.0 |
| Spider test | 従来最高のRESDSQL系     |      79.9 |       EM 72.0 |
| Spider dev  | DIN-SQL＋GPT-4     |  **74.2** |       EM 60.1 |
| Spider dev  | 通常のfew-shot＋GPT-4 |      67.4 |       EM 54.3 |
| BIRD dev    | DIN-SQL＋GPT-4     | **50.72** | VES **58.79** |
| BIRD dev    | GPT-4 baseline    |     46.35 |     VES 49.77 |
| BIRD test   | DIN-SQL＋GPT-4     |  **55.9** |     VES 59.44 |
| BIRD test   | GPT-4 baseline    |     54.89 | VES **60.77** |

Spider testでは、DIN-SQL＋GPT-4が論文執筆時点の従来最高EX 79.9を5.4ポイント上回る85.3を記録した。ただしEMは60であり、構造的一致では一部の専用モデルに劣っている。BIRD testでもEXは向上したが、VESは通常のGPT-4 baselineをわずかに下回った。

Spider devの難易度別EXでは、GPT-4のextra hardが31.9から43.4、hardが59.2から64.9へ向上した。単純なfew-shotが苦手としていた複雑な質問ほど、分解の効果が大きい。一方、extra hardでも正解率は半分未満であり、複雑なSQL生成が解決済みになったわけではない。

# 6. 考察

本研究の優れた点は、一般的な「Chain-of-Thoughtを使う」という主張にとどまらず、事前のエラー分析からText-to-SQL固有の失敗要因を特定し、それぞれを独立した処理段階へ対応付けたことである。特に、質問の複雑さに応じてプロンプトを切り替える設計は、単純な質問に過剰な推論を適用して精度を落とす問題を回避している。

アブレーションでは、Codex DavinciのEXが完全版の69.9に対し、自己修正なしで67.3、スキーマリンキングなしで65.9、分類なしで単純few-shotを全問に用いると63.1まで低下した。各段階が性能に寄与していることは確認できる。GPT-4ではgentle self-correctionが74.2、自己修正なしが73.3であり、自己修正の寄与は比較的小さい。一方、generic self-correctionでは70.0まで低下し、自己修正が常に有効とは限らないことも示された。

主な問題点は以下である。

* 固定された人手設計のデモ例に依存しており、データベースや質問分布が変化した場合の頑健性が不明である。
* GPT-4の具体的なモデルスナップショットが明示されておらず、閉鎖APIの更新を含めた厳密な再現性に課題がある。
* NatSQL、中間推論、分類ルーターの効果を完全に独立させたアブレーションにはなっていない。
* EX 85.3に対してEMが60と差が大きく、意味的に正しい別解だけでなく、特定のデータ内容で偶然同じ結果になるSQLが含まれる可能性も考慮する必要がある。
* 論文執筆時点では、GPT-4による1問当たりの処理費用が約0.5ドル、遅延が約60秒と報告されており、対話的・大規模な利用には重い。

# 7. まとめ

DIN-SQLは、Text-to-SQLを単一の生成問題として扱わず、**スキーマリンキング、構造分類、難易度別SQL生成、自己修正**へ分解する手法である。

最大の知見は、モデル自体を追加学習しなくても、Text-to-SQL固有の推論工程を明示して適切なプロンプトへ振り分けることで、通常のfew-shot promptingを大きく改善できる点にある。特に複数JOINやネストを含む難しい質問で効果が大きく、後続のLLMベースText-to-SQL研究におけるモジュール型パイプラインの基礎となる研究である。

# 8. 総評

**公開された査読結果：NeurIPS 2023 Main Conferenceのposter論文として採択。** ([OpenReview][2])

**本要約における査読判定：Accept（採択）**

採択と判断する理由は、問題設定が重要で、エラー分析と提案モジュールの対応が明確であり、SpiderとBIRDの双方で大きな実証的改善を示しているためである。特に2023年時点で、ファインチューニングを伴わないLLM promptingが専用モデルの実行精度を上回った点は研究上のインパクトが大きい。

一方、閉鎖モデルへの依存、固定デモ、コストと遅延、評価指標間の大きな差、アブレーションの不完全な分離は弱点である。したがって、理論的新規性よりも、**Text-to-SQLの失敗構造を実用的なLLMワークフローへ落とし込んだ設計と実証結果**を高く評価する論文である。
