# P085 — Cedar: A New Language for Expressive, Fast, Safe, and Analyzable Authorization

## 1. はじめに

本論文は、アプリケーションの認可判断を記述するためのドメイン固有言語 **Cedar** を提案している。Cedarの設計目標は、次の4要件を同時に満たすことである。

* **Expressive**：RBAC、ABAC、ReBACを表現できる
* **Fast**：オンライン処理に適した低遅延の認可判定
* **Safe**：誤った認可や実行時エラーを抑制する
* **Analyzable**：型検査やSMTソルバによる静的・記号的解析が可能

Cedarは、単なるポリシー記述構文ではなく、認可エンジン、スキーマ検証器、SMTコンパイラ、Leanによる形式モデルを一体として設計している。Rust実装はAWSの複数サービスで利用されていると報告されている。([arXiv][1])

## 2. 背景

認可処理を通常のアプリケーションコードに直接埋め込むと、判断ロジックが各所に分散し、監査、変更、テスト、バージョン管理が難しくなる。このため、認可ロジックを独立した「ポリシー」として管理する **policy-as-code** が用いられる。

しかし、既存のポリシー言語には次のようなトレードオフがある。

* 高い表現力を与えると、停止性や解析可能性を失いやすい
* 汎用的な探索や再帰を許すと、判定時間を予測しにくい
* 静的型検査を強くすると、現実的な属性ベースのポリシーを書きにくい
* 外部データベースへの問い合わせを前提にすると、純粋な認可評価性能を保証しにくい

Cedarは、任意の計算や一般再帰を許すのではなく、認可で頻出するエンティティ、属性、集合、階層関係をプリミティブとして提供することで、表現力を確保しつつ解析可能性を維持する。

## 3. 提案手法

### 3.1 ポリシーモデル

Cedarの認可要求は、概ね次の4要素から構成される。

* `principal`：操作主体
* `action`：実行しようとする操作
* `resource`：操作対象
* `context`：時刻、ネットワーク、認証状態などの追加情報

ポリシーは、`permit`または`forbid`という効果、principal・action・resourceに対するスコープ制約、`when`または`unless`による条件式から構成される。([arXiv][1])

認可の最終結果は次の規則で決まる。

1. 適用可能な`permit`が少なくとも1件必要
2. 適用可能な`forbid`が1件でもあれば拒否
3. どの`permit`にも該当しなければ拒否

したがって、Cedarは **deny by default** と **forbid overrides permit** を採用している。権限の付与は明示的であり、禁止規則を例外なく優先できる。

### 3.2 統一されたエンティティ表現

各エンティティは、型と識別子、属性、親エンティティとの関係を持つ。親関係の推移閉包を言語処理系が扱うため、以下のアクセス制御方式を同一言語で記述できる。

* **RBAC**：ユーザーが所属するグループやロールで判定
* **ABAC**：ユーザーやリソースの属性で判定
* **ReBAC**：所有者、メンバー、親組織などの関係で判定

エンティティ関係の探索を言語組み込み機能として限定的に提供することで、汎用プログラムのような予測不能な再帰を避けている。([arXiv][1])

### 3.3 ポリシースライシング

大規模なポリシー集合では、要求ごとにすべてのポリシーを評価するとコストが増大する。Cedarは各ポリシーのprincipalおよびresourceスコープに基づいて索引を構築し、現在の要求に関係するポリシーだけを抽出する。

例えば、要求のprincipal自身だけでなく、その所属グループや上位組織に対応する索引を検索する。スコープに制限がないポリシーも候補に含める。この抽出処理が認可結果を変えないこと、すなわちスライシングの健全性はLean上で証明されている。([arXiv][1])

### 3.4 スキーマ検証器

Cedarでは、エンティティ型、属性、親子関係、各actionで許されるprincipal・resource・contextの組合せをスキーマとして定義できる。

検証器はポリシーをactionごとに特殊化し、そのactionで発生し得る要求環境に対して型検査する。特に、存在しない可能性がある属性へのアクセスを安全に扱うため、次の仕組みを組み合わせる。

* 真偽値のシングルトン型である`True`と`False`
* 属性の存在が確認済みであることを表す静的ケイパビリティ
* `&&`や`||`の短絡評価を考慮したフロー依存の検査

例えば、属性へのアクセスに先立って「その属性が存在する」と確認されていることを追跡できる。このため、単純な型検査より誤検出を抑えながら、実行時エラーとなるポリシーを事前に発見できる。([arXiv][1])

### 3.5 SMTによる記号的解析

CedarポリシーはSMT式へ変換され、次のような問い合わせに利用できる。

* 2つのポリシーが等価か
* あるポリシーが別のポリシーを包含するか
* リファクタリング前後で認可結果が変化するか
* 特定の認可結果を生む反例が存在するか

エンティティ階層の推移性や非循環性を量化論理で直接記述すると、決定不能性やソルバのタイムアウトが問題となる。そこでCedarは、ポリシー中に現れるエンティティ型部分式の有限集合を **footprint** として計算し、その要素についてのみ基底化された制約を生成する。

これにより、量化子を避けながら、ポリシー等価性および包含関係に対するSMT符号化の健全性と完全性を維持している。なお、これらの証明は論文上で与えられており、発表時点ではSMTコンパイラ全体のLeanによる機械検証は進行中であった。([arXiv][1])

### 3.6 形式検証と実装テスト

Leanモデルでは、主に次の性質が証明されている。

* `forbid`が`permit`に優先する
* 明示的な許可がなければ拒否される
* 適用可能な許可があり禁止がなければ許可される
* ポリシースライシングが認可結果を保存する
* 検証済みポリシーが型に起因する実行時エラーを起こさない

実運用されるRust実装は完全に機械検証されているわけではない。代わりに、実行可能なLeanモデルとRust実装へ同一のランダム入力を与える差分テストとファジングを行い、両者の一致を確認している。この過程で約20件を超える実装上の不具合が発見された。([arXiv][1])

## 4. 実験方法

評価では、異なるアクセス制御パターンを持つ3つのアプリケーションモデルを使用している。

* **gdrive**：ファイル・フォルダ共有モデル
* **github**：組織、リポジトリ、チームを含むモデル
* **TinyTodo**：属性や関係を組み合わせるタスク管理アプリケーション

比較対象はCedar 3.0.1、Rego 0.61.0、OpenFGAの指定コミットである。実験環境はAmazon EC2のm5.4xlarge、Amazon Linux 2で、データとポリシーはメモリ上に保持した。

各データ点について200個のエンティティストアを生成し、それぞれ500件の要求を評価している。したがって、1データ点当たりの要求数は10万件である。各処理系の許可・拒否結果が一致することも確認された。

測定対象は純粋な認可評価であり、以下は除外されている。

* 永続ストレージからの読み込み
* ポリシーやデータの構文解析
* エンジン初期化
* HTTP通信

このほか、ポリシースライシングの有無を比較する実験と、正しい・誤ったポリシーリファクタリングをSMTで判定する実験を実施している。([arXiv][1])

## 5. 結果

### 5.1 認可評価性能

入力サイズ全体について集計した中央値の平均では、次の高速化が報告されている。

| アプリケーション | OpenFGA比 | Rego比 |
| -------- | -------: | ----: |
| gdrive   |    28.7倍 | 60.4倍 |
| github   |    34.4倍 | 80.8倍 |
| TinyTodo |    35.2倍 | 42.8倍 |

gdriveでは、エンティティ型当たりの要素数を5から50へ増加させても、Cedarの中央値は約4.0マイクロ秒から5.0マイクロ秒への増加にとどまった。githubでは中央値が概ね11マイクロ秒で安定していた。p99レイテンシも、gdriveで10マイクロ秒未満、githubで20マイクロ秒未満であった。([arXiv][1])

### 5.2 ポリシースライシング

テンプレートから多数のポリシーを生成する構成では、スライシングを行わない場合に比べ、認可判定が平均で約10～18倍高速化した。

ただし、最初から少数の静的ポリシーとして記述したCedarとの比較では結果が異なる。

* githubモデルでは、テンプレートとスライシングの組合せが約18%高速
* gdriveモデルでは、同構成が静的ポリシー版より約94%低速

したがって、スライシングは常に静的ポリシーより高速なのではなく、大量の具体化済みポリシーを扱う場合に特に有効である。([arXiv][1])

### 5.3 SMT解析性能

正しいリファクタリングの等価性確認に要した、符号化時間とSMT求解時間の合計中央値は次の通りである。

| モデル      |      合計時間 |
| -------- | --------: |
| gdrive   |  約45.3 ms |
| github   |  約60.9 ms |
| TinyTodo | 約119.6 ms |

3モデルの平均は約75ミリ秒であり、対話的なポリシー編集支援やCI上の検査に利用できる水準である。また、誤ったリファクタリングについては、等価でないことを示す反例を生成できた。([arXiv][1])

## 6. 考察

本研究の最大の貢献は、新しい構文そのものよりも、**言語制限、認可意味論、型検査、索引、SMT変換、形式モデルを一貫して共同設計した点**にある。認可に必要な演算を限定されたプリミティブとして提供することで、高速性と解析可能性を両立している。

一方、性能結果には次の留意点がある。

第一に、測定対象はインメモリの認可コアであり、ネットワーク、永続化、キャッシュミス、ポリシー更新などを含むエンドツーエンドのサービス性能ではない。

第二に、評価は3つのモデルと合成データに限定されている。実運用トレース、大規模な組織構造、頻繁なポリシー更新を用いた評価ではないため、報告された数十倍という比率をそのまま一般化することはできない。

第三に、比較条件には各処理系の性質による差がある。OpenFGAのインメモリデータストアは主にデバッグ用途であり、本番向けに最適化されたバックエンドではない。また、Regoでは推移閉包をポリシー内で計算しており、事前計算すれば性能を改善できるが、その場合は認可ロジックの一部をポリシー外へ移すことになる。([arXiv][1])

安全性についても境界がある。検証器が防止するのは型の不一致や存在しない属性へのアクセスなどであり、「本来許可すべきでない利用者を許可している」といった仕様意図の誤りを自動的に除去するものではない。また、スキーマ検証は任意である。

形式保証については、言語意味論と検証器、スライシングに強い保証がある一方、Rust実装全体と形式モデルとの対応は差分テストに依存している。したがって、「Cedarの実装全体が形式検証済み」と解釈するのは正確ではない。

表現力についても意図的な制限があり、汎用的な文字列処理、任意探索、一般再帰などには適さない。これは欠点であると同時に、予測可能な性能と決定可能な解析を成立させるための設計判断である。

## 7. まとめ

Cedarは、認可ポリシー言語に必要な次の要素を一つの体系に統合した。

* RBAC、ABAC、ReBACを扱えるエンティティ中心の言語
* deny-by-defaultと禁止優先による明確な意味論
* スキーマに基づく静的検証
* 健全性が証明されたポリシースライシング
* SMTによる等価性・包含関係の解析
* LeanモデルとRust実装の差分検証
* マイクロ秒単位のインメモリ認可評価

特に、「表現力を最大化する」のではなく、「実際の認可で必要な表現力に限定して高速性・安全性・解析可能性を確保する」という設計方針が明確である。

## 8. 総評

### 査読結果

**採択。** 本論文は査読を経て **OOPSLA 2024** に採択され、*Proceedings of the ACM on Programming Languages*, Volume 8, OOPSLA1, Article 118, pp. 670–697として出版されている。公開情報で確認できるのは採択・掲載結果であり、個別査読者のコメントや点数は公開されていない。([ACM Digital Library][2])

### 独自評価：Strong Accept相当

採択に値する完成度の高い研究である。特に評価できるのは、言語設計だけで終わらず、形式意味論、静的検証、SMT解析、実装テスト、性能評価、実サービスでの利用までを一貫して提示している点である。

弱点は、性能評価の外的妥当性、ベースライン間の実装条件差、ユーザビリティ評価の欠如、Rust実装全体が機械検証済みではない点にある。しかし、これらを考慮しても、認可システムにおける理論と実用の橋渡しとして技術的・実務的価値が高い。特に、形式手法を実運用可能な認可言語へ落とし込んだ点は、本論文の強い新規性である。
