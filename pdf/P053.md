# 1. はじめに

**G-Eval**は、GPT-4を「評価者」として利用し、要約や対話応答などの自然言語生成結果を自動採点する手法です。ここでいう “Human Alignment” は、安全性アラインメントではなく、**自動評価スコアと人間の評価との相関を高めること**を指します。

論文はYang Liuらによるもので、後に**EMNLP 2023 Main Conference**へ採択されています。([arXiv][1])

# 2. 背景

従来のNLG評価では、BLEUやROUGEのように生成文と参照文の単語・n-gram重複を測る指標が広く利用されてきました。しかし、これらには次の問題があります。

* 表現が異なっていても内容的には正しい文章を低く評価する
* 一貫性、読みやすさ、面白さなどを十分に捉えられない
* 新しいタスクでは参照文を人手で用意する必要がある

BERTScoreやMoverScoreは意味的類似度を扱えますが、文章全体の品質を評価基準に沿って判断するわけではありません。また、BARTScoreやGPTScoreなどの生成確率を用いる手法も、人間評価との対応が十分ではありませんでした。そこで本論文は、LLMに文章そのものを生成させるのではなく、**明示的な採点タスクを実行させる**ことを提案します。

# 3. 提案手法

G-Evalは、主に次の3要素から構成されます。

### 3.1 評価プロンプト

プロンプトに以下を記述します。

* 評価対象となるタスク
* 評価観点
* 評価尺度

例えば要約の一貫性では、「文章が適切に構成され、単なる情報の羅列ではなく、文から文へ論理的につながっているか」を1～5で評価させます。

### 3.2 Auto Chain-of-Thought

タスク説明と評価基準だけをLLMに与え、具体的な**Evaluation Steps**を自動生成させます。

要約の一貫性であれば、

1. 原文の主題と重要点を確認する
2. 要約が重要点を含んでいるか比較する
3. 情報の順序が明確か確認する
4. 基準に従って点数を付ける

という採点手順を生成します。

これは各事例について長い推論を出力させるというより、**抽象的な評価基準を実行可能なルーブリックへ展開する処理**に近いものです。

### 3.3 Form-fillingと確率加重

原文、生成文、評価基準、評価手順をLLMへ入力し、評価欄を埋める形式で1～5などの点数を出力させます。

ただし、整数だけを使うと「3」に集中したり、同点が大量に発生したりします。そこで各点数トークンの確率を使い、

[
\mathrm{score}=\sum_i p(s_i)s_i
]

として期待値を計算します。

例えば1～5点の確率がそれぞれ与えられれば、最終値を2.59のような細粒度のスコアにできます。

# 4. 実験方法

評価用LLMとして以下を使用しています。

* GPT-3.5：`text-davinci-003`、temperature 0
* GPT-4：当時トークン確率を取得できなかったため、temperature 1、top-p 1で**20回サンプリング**し、点数の確率分布を近似

このため、G-Eval-4は1事例・1評価観点につき複数回のAPI呼び出しを必要とします。

実験対象は3つです。

| ベンチマーク       | タスク         | 評価観点                                            |
| ------------ | ----------- | ----------------------------------------------- |
| SummEval     | 要約          | coherence、consistency、fluency、relevance         |
| Topical-Chat | 対話応答        | naturalness、coherence、engagingness、groundedness |
| QAGS         | 要約の幻覚・事実整合性 | CNN/DailyMail、XSum                              |

比較対象にはROUGE、BLEU、METEOR、BERTScore、MoverScore、BARTScore、FactCC、USR、UniEval、GPTScoreなどが含まれます。評価指標自体の性能は、人間評価とのPearson、Spearman、Kendallの相関によって測定しています。

# 5. 結果

主要な平均相関は次の通りです。

| ベンチマーク                            |                  G-Eval-4 |                      主な従来最良手法 |
| --------------------------------- | ------------------------: | ----------------------------: |
| SummEval：Spearman / Kendall       |         **0.514 / 0.418** |         UniEval：0.474 / 0.377 |
| Topical-Chat：Pearson / Spearman   |         **0.575 / 0.588** |         UniEval：0.552 / 0.417 |
| QAGS：Pearson / Spearman / Kendall | **0.599 / 0.611 / 0.525** | UniEval：0.571 / 0.575 / 0.465 |

G-Eval-4は、3ベンチマークの平均値では従来手法を上回りました。特にSummEvalでは、relevanceが0.547、consistencyが0.507となり、UniEvalの0.426、0.446から比較的大きく改善しています。一方、coherenceとfluencyでの改善幅は小さく、全評価観点で一様に大差がついたわけではありません。

アブレーションでは、SummEvalの平均Spearman相関が、

* 完全版：0.514
* 確率加重なし：0.502
* CoTなし：0.500

となりました。したがって、CoTと確率加重はいずれも有効ですが、GPT-4自体の能力と比べると改善量は比較的小さいと解釈できます。

また、人間作成要約とGPT-3.5作成要約を比較した分析では、**人間が人間要約を好んだ場合でさえ、G-Eval-4は平均的にGPT-3.5要約へ高い点を付けました**。著者らは、LLM評価器がLLM特有の文章や評価基準を好む可能性を指摘しています。

# 6. 考察

本研究の重要性は、単にGPT-4を評価へ使ったことではなく、現在の「LLM-as-a-Judge」の基本形となる以下の設計を明確に示した点にあります。

* 評価観点を明文化する
* 評価観点を手順へ分解する
* 構造化された形式で採点させる
* 離散点ではなく確率加重スコアを使う

一方で、査読上は次の点が問題になります。

**再現性とモデル依存性**
GPT-4は閉鎖モデルであり、モデル更新によって同じプロンプトでも結果が変化する可能性があります。厳密なモデル版、実行日時、試行間分散も十分に示されていません。

**計算コスト**
GPT-4では確率分布を得るため20回サンプリングしています。評価観点ごとにこれを行うため、大規模なモデル比較ではコストと待ち時間が大きくなりますが、費用・実行時間の分析はありません。

**統計的有意性**
相関係数は報告されていますが、信頼区間や有意差検定がありません。特にSummEvalのcoherenceやfluencyでは従来最良手法との差が小さく、差が安定しているか判断しにくいです。

**モデル能力と手法の交絡**
G-Eval-4の改善が、form-fillingやCoTによるものなのか、単にGPT-4が従来ベースラインより強力だからなのかを完全には分離できていません。同一バックボーンを用いた比較がより必要です。

**評価対象へのバイアス**
LLM生成文をLLM評価器が好む傾向は、自己強化を引き起こす危険があります。評価スコアをRLHFやモデル選択の報酬として直接使うと、人間に好まれる文章ではなく、評価モデルに好まれる文章へ最適化される可能性があります。

**適用範囲**
実験は英語の要約と対話が中心です。翻訳、コード、長文、専門文書、多言語、創作文章などへの一般化は実証されていません。

また、Topical-Chatについて、本文と表の記号はPearson (r) とSpearman (\rho) を示している一方、表キャプションにはSpearmanとKendallと書かれており、記述上の不整合があります。

# 7. まとめ

G-Evalは、LLMを評価器として使うための実用的な枠組みを提案しました。

中心的なアイデアは、

1. 人間が評価基準を定義する
2. LLMが評価手順を生成する
3. LLMがフォーム形式で点数を出す
4. 点数確率の期待値を最終スコアとする

というものです。

GPT-4を用いたG-Evalは、要約、対話、事実整合性の各ベンチマークで人間評価との相関を改善しました。一方で、その相関はSummEvalで0.514にとどまり、人間評価を完全に代替できる水準ではありません。実務では、**大量候補の一次評価や人手評価の補助**として使い、最終判断には人間評価、複数評価器、バイアス監査を組み合わせるのが妥当です。

# 8. 総評

**査読判定（私見）：採択／Weak Accept、7点相当**

採択理由は、LLM評価器という重要な研究方向に対して、単純で再利用可能な設計を提示し、複数タスク・複数ベンチマークで明確な改善を示したためです。特に、LLM生成文への偏りを成功例だけで終わらせず、自ら問題として報告した点は高く評価できます。

ただし、手法上の新規性は主にプロンプト構成と確率加重にあり、性能向上の一部はGPT-4のモデル能力に依存しています。統計的有意性、費用、頑健性、モデル更新への再現性、異なるLLM間のバイアス検証が不足しているため、Strong AcceptではなくWeak Acceptが適切です。

**実際の査読結果は、EMNLP 2023 Main Conferenceへの採択です。**
