# P097：Manu: A Cloud Native Vector Database Management System

## 1. はじめに

本論文は、大規模な埋め込みベクトルを管理するクラウドネイティブ型ベクトルデータベース **Manu** を提案するシステム論文である。著者らは、従来型DBMSの複雑なトランザクション機能をそのまま継承するのではなく、ベクトル検索用途では不要な機能を削減し、その代わりに「長期的な拡張性」「調整可能な一貫性」「細粒度の弾力性」「高性能」を実現する方針を採用した。

中核となるアイデアは、WALとbinlogをシステムの共通基盤とし、書き込み処理をログの発行者、検索・索引構築・分析処理を独立した購読者として構成する **log-as-data型アーキテクチャ** である。Manuは、Milvusの利用者1,200以上から得たフィードバックを基に設計されている。論文はPVLDB 15巻12号、3548–3561頁に掲載された。

## 2. 背景

従来の関係データベースとベクトルデータベースでは、要求が大きく異なる。

ベクトル検索では、画像、文章、商品、ユーザなどの複雑な意味情報が単一の埋め込みベクトルに集約される。そのため、複数表JOINや複数行をまたぐ複雑なトランザクションよりも、単一エンティティ単位のACID特性が重要になる。

一方で、一貫性に対する要求はアプリケーションごとに異なる。ウイルス検索などでは最新データの即時反映が重要だが、推薦システムでは新商品が数秒遅れて検索可能になっても問題がなく、検索応答を高速化する方が有益な場合がある。そこで、強一貫性と結果整合性の中間に位置する、鮮度を時間で指定できる仕組みが必要となる。

さらに、ベクトル検索、索引構築、データ挿入、ログ永続化では、CPU、GPU、メモリ、ストレージに対する負荷特性が異なる。このため、単純なread/write分離ではなく、機能単位で独立にスケールできる構成が求められる。著者らは、長期的拡張性、調整可能な一貫性、弾力性、高可用性、高性能、異なる実行環境への適応性を設計目標に設定した。

## 3. 提案手法

### システム構成

Manuは、次の4層から構成される。

**アクセス層**はステートレスなProxyであり、リクエスト受付、ルーティング、部分検索結果の統合を担当する。
**Coordinator層**には、スキーマ管理を行うRoot Coordinator、データ配置を扱うData Coordinator、検索ノードを管理するQuery Coordinator、索引構築を管理するIndex Coordinatorが存在する。
**Worker層**は、Query Node、Data Node、Index Nodeに分かれ、それぞれ検索、ログ変換・保存、索引構築を独立して実行する。
**ストレージ層**では、メタデータにetcd、ベクトル・索引・binlogにS3やMinIOなどのオブジェクトストレージを利用する。

データはCollection、Shard、Segmentによって管理される。Shardは更新ログのチャネルに対応し、実際のデータ配置と処理の単位にはSegmentを用いる。Segmentには、更新を受け付けるgrowing状態と、読み取り専用のsealed状態がある。標準では512 MBに到達するか、一定期間更新がなければsealed状態へ移行する。

### ログバックボーン

Manuでは、すべての状態変更を論理ログとして記録する。WALはKafkaやPulsarなどのメッセージキュー上に構築された低遅延の行指向ストリームであり、Data Nodeがこれをフィールド単位の列指向binlogへ変換してオブジェクトストレージに保存する。

Query NodeやIndex NodeはWALまたはbinlogを独立に購読する。このため、書き込み、検索、索引構築、ログ保存が直接同期せず、各機能を別々に増減できる。コンポーネント間の制御通知もログ経由で行われるため、送信側が個々のWorkerの障害や受信状態を直接管理する必要がない。

### Delta Consistency

各更新と検索リクエストには、Hybrid Logical Clockに基づくLSNが割り当てられる。また、各ログチャネルには、処理済み時刻を表すtime-tickが定期的に挿入される。

検索リクエストの時刻を (L_r)、Query Nodeが処理済みの最新time-tickを (L_s)、ユーザが許容する陳腐化時間を (\tau) とすると、鮮度条件を満たすまで検索を待機させる。著者らの定義では、(\tau=0) が強一貫性、(\tau=\infty) が結果整合性に対応し、その中間を連続的に指定できる。

### 索引構築と検索

Manuは、PQ、OPQ、SQなどの量子化方式、IVF-FLAT、IVF-PQ、IVF-HNSWなどの転置索引、HNSW、NSG、NGTなどの近傍グラフをサポートする。

索引構築はSegment単位で行われ、バッチ構築とストリーミング構築の両方に対応する。新しいSegmentがsealedになると、Index Nodeが非同期に索引を構築する。削除データはbitmapで管理し、削除率が高くなった段階で索引を再構築する。

検索では、各Query Nodeが担当Segmentの局所top-(k)を求め、Node内、Proxy内の2段階で結果を統合する。更新中のgrowing segmentについてはWALを直接取り込み、10,000件程度のsliceごとに一時索引を作成する。この一時索引により、著者らの実測ではgrowing segmentの検索が最大10倍高速化した。

このほか、属性フィルタ、複数ベクトル検索、Python・Java・Go・C++・REST API、管理GUIのAttu、BOHBによる索引パラメータ探索、ログ再生を用いたtime travel、CPU・GPU・SSD向け最適化を備える。

## 4. 実験方法

評価には、128次元のSIFTと96次元のDEEPから抽出したデータセットを使用した。標準構成はQuery Node 2台、Data Node 1台、Index Node 1台で、各WorkerにはAmazon EC2 m5.4xlargeを使用した。索引はIVF-FLATとHNSW、検索はtop-50とし、明示されない実験では平均Recallが0.8以上となるように設定した。

実験は主に次の観点で行われた。

1. **混合ワークロード**
   空のCollectionに毎秒2,000、3,000、4,000件でベクトルを挿入しながら検索を行い、旧Milvusとの検索遅延を比較した。

2. **EC推薦シナリオ**
   4種類の匿名化されたオープンソースシステムと、単一ノードでRecall–QPSを比較した。さらにSIFT100Mと実際のECサイトに近い日内トラフィック変動を用い、検索遅延に応じたQuery Nodeの増減を評価した。

3. **動画重複検出シナリオ**
   Query Node数を2台から10台へ変更した場合、およびデータ量を2,000万件から1億件へ変更した場合のスケーラビリティを測定した。

4. **ウイルス検索シナリオ**
   許容鮮度時間とtime-tick間隔を変化させ、検索遅延との関係を測定した。また、データ量に対する索引構築時間を評価した。

## 5. 結果

混合ワークロードでは、挿入率が高くなるほどMilvusの検索遅延が大きく増加したのに対し、Manuは比較的低い遅延を維持した。Milvusでは書き込みと索引構築が同一ノードの資源を競合する一方、Manuでは専用Index Nodeが索引を迅速に構築できることが主因と説明されている。

単一ノードのRecall–QPS比較では、SIFTのユークリッド距離、DEEPの内積の双方で、ManuのHNSWが全般的に最も高いスループットを示した。著者らはCPUキャッシュとSIMD向けの実装最適化が寄与したと推測している。ただし比較対象はSystem A〜Dと匿名化されている。

弾力性評価では、検索遅延が100 ms未満の場合にQuery Nodeを半減し、150 msを超えた場合に倍増する制御を行った。日内のトラフィックが大きく変化しても、検索遅延は概ね目標範囲に維持された。

Query Node数を増やすとManuのQPSは明確に向上し、System Cより良好な水平スケーラビリティを示した。データ量を増やした場合は、固定Segmentサイズの下でQPSがおおむねデータ量の逆数に従って低下した。索引構築時間はデータ量にほぼ比例して増加した。

Delta Consistencyについては、許容鮮度時間を長くすると検索リクエストが更新待ちをする確率が下がり、遅延が急速に減少した。また、time-tickの発行間隔が短いほど、同じ許容鮮度時間でも検索遅延が短くなった。これにより、一貫性と応答性能をユーザが調整可能であることが示された。

## 6. 考察

本論文の最大の貢献は、新しい近似最近傍探索アルゴリズムではなく、**ベクトルデータベース全体をどのようにクラウドネイティブ化するか**を具体化した点にある。

特に、WALを単なる障害復旧用データとしてではなく、全コンポーネントを接続するイベントストリームとして扱った設計は明快である。検索、索引構築、ログ保存を独立した購読者にすることで、機能追加、障害分離、独立スケーリングを同一原理で説明できている。Delta Consistencyも、ベクトル検索サービスで重要となる「最新データの可視性」と「応答遅延」のトレードオフを、時間という理解しやすいパラメータで表現している。

一方、次の制約がある。

* 比較対象がSystem A〜Dと匿名化されており、製品名、バージョン、詳細設定が分からないため、性能比較の再現性が低い。
* 主評価の最大規模は1億ベクトルであり、論文が動機として掲げる十億規模以上を十分に直接検証しているわけではない。
* 高可用性、障害復旧時間、time travel、BOHB自動設定、属性フィルタ、複数ベクトル検索については、機能説明はあるものの定量評価がほとんどない。
* コスト削減を重要な目標としているが、EC2料金や性能当たりコストは示されていない。
* エラーバー、試行回数、分散などが報告されておらず、測定の安定性を判断しにくい。
* Query Nodeは低遅延化のため原則として検索対象データを計算側にロードする。この設計は高速だが、超大規模環境ではメモリコストが問題となり得る。
* JOINや複数行トランザクションを排除する判断は対象用途には合理的だが、ベクトルと複雑な業務データを同一DBで扱いたい用途には適さない。

また、顧客事例は実際の要求を基にしているものの、評価自体は公開ベンチマークデータを用いた模擬実験である。したがって、本番環境での長期間の運用安定性まで証明した評価とは区別する必要がある。

## 7. まとめ

Manuは、ベクトル検索用途では複雑な関係モデルとトランザクションを緩和できるという前提に立ち、その余力をクラウド上の弾力性、一貫性制御、機能拡張性へ振り向けたベクトルDBMSである。

主な設計要素は以下の3点に集約できる。

1. WALとbinlogを中心に据えたpublish/subscribe型のログバックボーン
2. 検索、索引構築、データ保存を独立させた機能単位のスケーリング
3. time-tickとLSNを用いた、時間指定可能なDelta Consistency

評価では、旧Milvusに対する混合ワークロード性能、他システムに対するRecall–QPS、負荷変動への弾力性、ノード数とデータ量に対するスケーラビリティ、一貫性と遅延のトレードオフが示された。

## 8. 総評

**長所**は、実利用者の要求から設計目標を導き、ログ中心の疎結合アーキテクチャ、調整可能な一貫性、Segment単位の分散処理として一貫したシステム設計に落とし込んだ点である。特定のANNアルゴリズムだけでなく、データ更新、索引構築、検索、スケーリング、管理機能まで含むDBMS全体を論じており、システム論文としての実用的価値は高い。

**弱点**は、匿名化されたベースライン、限定的な再現情報、主要機能の一部に対する評価不足である。特に、可用性やコストを設計目標に掲げながら、それらの定量評価がない点は査読上の主要な懸念となる。
