# P109：Document Expansion by Query Prediction

**著者**：Rodrigo Nogueira、Wei Yang、Jimmy Lin、Kyunghyun Cho
**発表年**：2019年
**手法名**：Doc2query
**概要**：文書から「その文書を探す際に利用されそうな検索クエリ」を生成し、そのクエリを文書に追加してから索引化する文書拡張手法である。([arXiv][1])

## 1. はじめに

本論文は、情報検索における**語彙不一致問題**を扱う。語彙不一致とは、ユーザーが検索クエリで用いる語と、関連文書に書かれている語が異なるため、意味的には関連している文書を単純なキーワード検索では取得できない問題である。

提案手法Doc2queryの中心的な発想は、文書 (d) から、その文書が関連文書となりそうなクエリ (q) を

[
q \sim p_\theta(q\mid d)
]

として事前に生成し、その生成クエリを元の文書に連結することである。拡張後の文書は通常どおり転置索引に登録でき、検索時にはBM25などの高速な語彙ベース検索をそのまま利用できる。つまり、ニューラルモデルの計算を検索時ではなく**索引構築時に移す**ことが本研究の重要な特徴である。

## 2. 背景

従来、語彙不一致問題には主に**クエリ拡張**が利用されてきた。クエリ拡張では、初期検索結果や関連フィードバックを使って、元のクエリに同義語や関連語を追加する。

一方、ニューラル情報検索では、BERTなどを使ってクエリと文書の意味的関連性を判定できる。しかし、大規模コーパスの全文書にニューラルモデルを適用することは計算量の面で困難である。このため、一般的な構成は次の二段階となる。

1. BM25などで候補文書を高速に取得する。
2. 取得した少数の候補をニューラルモデルで再ランキングする。

この構成では、第一段階のBM25が関連文書を候補集合に含められなければ、第二段階の高性能な再ランキングモデルでも救済できない。

文書拡張は、クエリではなく文書側に関連語を追加する方法である。従来研究では、コーパス全体の再索引化が必要で実験コストが高いことや、クエリ拡張より一貫して有効とは限らなかったことから、比較的注目されていなかった。本論文は、この文書拡張をニューラルなクエリ生成によって実現する。

## 3. 提案手法

### Doc2queryの処理

Doc2queryは、次の流れで動作する。

1. 「クエリと関連文書」のペアを学習データとして用意する。
2. 文書を入力、クエリを出力とするSequence-to-Sequence Transformerを学習する。
3. 各文書について複数のクエリを生成する。
4. 生成クエリを元の文書末尾に連結する。
5. 拡張文書をBM25用の転置索引に登録する。
6. 必要に応じて、取得結果をBERTで再ランキングする。

元の文書と生成クエリを区別する特別なタグは挿入されない。生成された単語は通常の文書語として扱われるため、文書に元から存在した重要語を再生成すれば、その語の頻度を高める**項重み付け**として働く。文書に存在しない関連語を生成すれば、通常の意味での**語彙拡張**として働く。

例えば、文書に「cinnamon reduces blood sugar levels」と書かれている場合、モデルが「does cinnamon lower blood sugar?」というクエリを生成する。「reduce」と「lower」のような表現差を吸収できるため、元の文書だけでは一致しにくかった検索クエリから取得されやすくなる。

### 前処理と生成

文書とクエリはMoses tokenizerでトークン化した後、BPEでサブワード分割する。メモリ使用量を抑えるため、文書は最大400トークン、クエリは最大100トークンに切り詰められる。

推論時にはtop-(k) random samplingを用いて、各文書について10個のクエリを生成する。付録実験では、ビームサーチよりtop-(k) random samplingの方がわずかに良く、生成数は10個で性能が最大になった。生成数が少なすぎると意味的多様性が不足し、多すぎるとノイズが増えて元文書の寄与が弱くなると考察されている。

### Transformerの構成

モデルはTransformer Base相当で、以下の構成である。

| 項目              |        設定 |
| --------------- | --------: |
| Encoder／Decoder |       各6層 |
| 隠れ次元            |       512 |
| Attention head  |         8 |
| Feed-forward層   |     2,048 |
| バッチサイズ          | 4,096トークン |
| 最大epoch数        |        30 |
| Optimizer       |      Adam |
| 学習率             | (10^{-3}) |
| Dropout         |       0.1 |
| Warmup          | 8,000ステップ |
| 学習GPU           |  V100 × 4 |

学習・開発データ間のBLEU差が4ポイントを超えた場合に学習を停止する。実装にはOpenNMTが使用されている。

## 4. 実験方法

### データセット

| データセット                   | 内容                               | 学習・評価規模                              | 評価指標   |
| ------------------------ | -------------------------------- | ------------------------------------ | ------ |
| MS MARCO Passage Ranking | 実際の検索クエリとWebパッセージ                | 約880万パッセージ、学習約50万ペア、開発・テスト各約6,900クエリ | MRR@10 |
| TREC-CAR                 | Wikipediaの記事名・節名をクエリとし、該当節の段落を取得 | 学習約300万クエリ、検証約70万、テスト約2,250          | MAP    |

MS MARCOでは、1つのクエリ当たり平均約1件の関連パッセージが存在する。TREC-CARでは、英語Wikipediaの概要部分を除いた段落が検索対象となる。

### 比較手法

実験では、以下の検索構成が比較されている。

* **BM25**：拡張されていない文書を検索する基本条件
* **BM25 + RM3**：従来型のクエリ拡張
* **BM25 + Doc2query**：提案する文書拡張
* **BM25 + Doc2query + RM3**：文書拡張とクエリ拡張の併用
* **BM25 + BERT**：BM25候補をBERTで再ランキング
* **BM25 + Doc2query + BERT**：文書拡張後にBERT再ランキング

BM25の実装にはAnseriniを使用し、再ランキング対象として上位1,000件の候補文書を取得する。

## 5. 結果

主要結果は以下のとおりである。数値は論文の表記に合わせており、例えばMRR@10の18.4は0.184に相当する。

| 手法                          | TREC-CAR MAP | MS MARCO Test MRR@10 | MS MARCO Dev MRR@10 |     検索時間 |
| --------------------------- | -----------: | -------------------: | ------------------: | -------: |
| BM25                        |         15.3 |                 18.6 |                18.4 |    50 ms |
| BM25 + RM3                  |         12.7 |                    ― |                16.7 |   250 ms |
| **BM25 + Doc2query**        |     **18.3** |             **21.8** |            **21.5** |    90 ms |
| BM25 + Doc2query + RM3      |         15.5 |                    ― |                20.0 |   350 ms |
| BM25 + BERT                 |         34.8 |                 35.9 |                36.5 | 3,400 ms |
| **BM25 + Doc2query + BERT** |     **36.5** |             **36.8** |            **37.5** | 3,500 ms |

Doc2query単独の効果を見ると、BM25に対してTREC-CARではMAPが15.3から18.3、MS MARCO開発セットではMRR@10が18.4から21.5へ改善した。絶対値で約3ポイント、相対値で約17～20%の改善である。

BERT再ランキングを用いた場合も、Doc2queryを加えることでTREC-CARのMAPが34.8から36.5、MS MARCO開発セットのMRR@10が36.5から37.5へ向上した。これは、Doc2queryと再ランキングが代替関係ではなく、**相補的に機能する**ことを示している。当時の比較では、TREC-CARで既知の最高性能を達成し、MS MARCOでも最高水準に近い結果だった。

### 生成語の分析

生成クエリ中のストップワードを除くと、単語の69%は元文書からコピーされた語、31%は元文書に存在しない新しい語だった。

| 拡張内容     | MS MARCO Dev MRR@10 |
| -------- | ------------------: |
| 拡張なし     |                18.4 |
| 新規語のみ    |                18.8 |
| コピー語のみ   |                19.7 |
| コピー語＋新規語 |            **21.5** |

コピー語は重要語の重み付け、新規語は同義語・関連語による意味的拡張に対応する。両方を利用した場合に最も高い性能となり、二つの効果が相補的であることが示された。

また、MS MARCO開発セットのRecall@1000は、BM25の85.3からDoc2query適用後の89.3へ上昇した。したがって、BERTの性能向上は単に候補内の順位が変わったためではなく、第一段階で取得できる関連文書自体が増えたことによる。

RM3によるクエリ拡張は、両データセットで性能を低下させた。著者らは、MS MARCOとTREC-CARが上位精度を重視するタスクであり、不適切な拡張語によるクエリドリフトの影響を受けやすいためと説明している。

## 6. 考察

### 有効である理由

Doc2queryは、一つの処理で二つの役割を実現している。

**第一に、重要語の再重み付けである。**
生成モデルが文書中の重要語をクエリ内で再生成すると、その語が拡張文書内で複数回出現する。その結果、BM25における語の寄与が強くなる。

**第二に、語彙不一致の解消である。**
文書中にない同義語や上位概念、質問表現を生成することで、元文書と語彙的に一致しないクエリからも検索できる。

さらに、文書は通常クエリより長く、生成に利用できる情報量が多い。短いクエリから関連語を推測するクエリ拡張より、長い文書から想定クエリを生成する方が豊富な手掛かりを利用できるという説明には合理性がある。

### 実運用上の利点

ニューラル推論は索引構築時に一度だけ行えばよく、検索要求ごとに生成モデルを実行する必要がない。Doc2queryのみの場合、検索時間はBM25の50 msから90 msに増えるものの、Single Duet v2の約650 msより約7倍高速である。低遅延が重要な環境では、精度と速度の有力な折衷案となる。

### 課題

本論文には次の制約がある。

1. **索引サイズと事前計算コストが十分に報告されていない。**
   各文書に10個のクエリを追加するため、生成時間、索引サイズ、索引構築時間は大幅に増える可能性がある。

2. **更新頻度の高いコーパスには適用コストがある。**
   文書が追加・更新されるたびにクエリ生成と再索引化が必要になる。

3. **生成ノイズを制御する機構がない。**
   生成クエリは元文書に直接連結され、信頼度に応じた重み付けやフィルタリングは行われない。実際、20クエリまで増やすと性能が低下しており、過剰な拡張がノイズになることが確認されている。

4. **評価範囲が限定されている。**
   実験は英語のパッセージ検索を中心とした2データセットであり、長文書検索、多言語検索、ドメイン外データへの一般化は検証されていない。

5. **統計的有意差の検定がない。**
   数値上の改善は一貫しているが、特にBERT併用時のMS MARCO Testにおける0.9ポイントの差が統計的に安定しているかは判断できない。

6. **生成モデルの学習目的と検索目的が一致していない。**
   モデルはクエリ生成として学習・評価されるが、最終目的は検索指標の最大化である。BLEUによる早期終了が、MRRやMAPにとって最適とは限らない。

## 7. まとめ

本論文は、文書から想定検索クエリを生成し、そのクエリを文書に追加するDoc2queryを提案した。

Doc2queryの主要な成果は、以下の三点である。

* ニューラルモデルを検索時ではなく索引構築時に使用した。
* BM25のような既存の高速な転置索引検索をそのまま強化した。
* コピー語による重要語の再重み付けと、新規語による語彙拡張を同時に実現した。

BM25単独に対して明確な改善を示し、BERT再ランキングと組み合わせた場合にも追加の性能向上を得ている。第一段階検索のRecallを改善することで、後段の高性能な再ランキングモデルに、より良い候補集合を渡せることを実証した点が特に重要である。

## 8. 総評

本研究の最大の価値は、非常に単純な構成で「生成モデルの意味理解能力」と「転置索引の高速性」を接続した点にある。複雑なニューラル検索器を新たに構築するのではなく、生成クエリを通常の文書語として追加するだけで既存のBM25基盤を強化しており、実装・導入の観点でも優れている。

一方で、索引サイズ、オフライン生成時間、文書更新コスト、統計的有意差が報告されていない点は弱い。生成クエリを無条件に連結するため、誤った関連語や過剰な語の反復が検索性能を損なう可能性も残っている。

### 査読結果

**判定：Weak Accept（弱い採択）**

**理由**：新規性、実用性、実験結果の一貫性は採択に十分である。特に、Doc2query単独とBERT併用の双方で改善を確認し、Recall分析やコピー語・新規語のアブレーションによって改善要因を説明している点を高く評価する。ただし、計算・ストレージコストの分析、統計的有意差、多様な検索タスクでの検証が不足しているため、強い採択ではなくWeak Acceptと判断する。

**総合評価：7／10**
**査読確信度：4／5**
