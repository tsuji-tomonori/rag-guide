# P104：ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data

## 1. はじめに

本論文は、**ベクトル類似検索と構造化条件を組み合わせたハイブリッド検索**を高速化する手法「ACORN（ANN Constraint-Optimized Retrieval Network）」を提案している。著者は Liana Patel、Peter Kraft、Carlos Guestrin、Matei Zaharia で、SIGMOD 2024に採択され、*Proceedings of the ACM on Management of Data* に掲載された。([ACM Digital Library][1])

対象となる問い合わせは、クエリベクトル (q) と述語 (p) が与えられたとき、

[
{x\mid p(x)=\mathrm{true}}
]

を満たすデータの中から、(q) に最も近い上位 (K) 件を求めるものである。たとえば「2024年以降に公開された医療論文の中から、質問文に意味的に近い文書を検索する」といった処理に相当する。

ACORNの特徴は、検索対象の属性、値、演算子をインデックス構築時に固定せず、**未知の述語や高カーディナリティ属性にも単一のグラフインデックスで対応する**点にある。([arXiv][2])

## 2. 背景

既存のハイブリッド検索には、主に次の二つの方法がある。

**事前フィルタリング**では、最初に述語を満たすデータだけを抽出し、その集合に対して総当たりのベクトル検索を行う。正確ではあるが、述語を通過する割合を選択率 (s)、全データ数を (n) とすると、概ね (O(sn)) 件の距離計算が必要になる。

**事後フィルタリング**では、HNSWなどの近似最近傍探索を全データに対して行い、その後で述語に合わない結果を削除する。しかし、最終的に (K) 件を得るには平均して (K/s) 件程度を探索する必要があり、選択率が低いほど非効率になる。さらに、述語を満たすデータがクエリベクトルから遠い位置に偏る「負のクエリ相関」では、多数の無効候補を探索してしまう。([arXiv][3])

Filtered-DiskANNやNHQなど、フィルタ付きANN専用手法も存在する。しかし、これらはインデックス構築時に既知の属性値ごとの接続を作るため、少数のカテゴリ値や等価条件には強い一方、正規表現、範囲条件、リスト包含、任意の論理式、非常に多くの属性値を扱うことが難しい。

理想的には、考え得る各述語 (p) について、条件を満たす部分集合 (X_p) 専用のHNSWを構築すればよい。論文はこれを「Oracle Partition Index」と呼んでいる。しかし、述語の種類が未知または無数の場合、述語ごとのインデックスを事前構築することは現実的ではない。([arXiv][3])

## 3. 提案手法

### ACORNの基本アイデア

ACORNは、単一のグラフから、任意の述語を満たすノードだけで構成される**述語部分グラフ（predicate subgraph）**を検索時に取り出す。

通常のHNSWでは、各ノードは近傍の少数ノードとだけ接続される。この状態で述語によるフィルタを行うと、条件を満たす隣接ノードが存在せず、探索経路が途中で切れる可能性がある。ACORNは各ノードの近傍候補を増やし、どのような述語を適用しても、条件を満たすノード間に十分な接続が残るようにする。([arXiv][3])

### ACORN-(\gamma)

ACORN-(\gamma) は、各ノードについて通常のHNSWより多い (M\gamma) 個の近傍候補を収集する。

対象とする最小選択率を (s_{\min}) とした場合、

[
\gamma \approx \frac{1}{s_{\min}}
]

と設定する。すると、(M\gamma) 個の近傍のうち、平均して少なくとも (M) 個が述語を満たすことが期待できる。

検索時には、近傍をまず述語でフィルタし、その中から最大 (M) 個を探索候補に追加する。これにより、無効ノードを経由する事後フィルタリングとは異なり、主として述語を満たすノード上を移動できる。

単純にHNSWのパラメータ (M) を増やすのではなく、階層構造を決定する本来の (M) と、近傍候補の拡張率 (\gamma) を分離している点も重要である。単に (M) を増やすと、HNSWの上位層が薄くなり、階層探索の性質が変わってしまう。([arXiv][3])

### グラフ圧縮

すべての (M\gamma) エッジを保存すると、メモリ使用量と構築時間が大きくなる。そこでACORN-(\gamma) は、レベル0のグラフに対して述語非依存の圧縮を行う。

各ノードについて、近い (M_\beta) 本のエッジは直接保存する。それより遠いノードについては、保存済みの近傍を経由した2ホップで到達できる場合、直接エッジを削除する。検索時に必要に応じて2ホップ近傍を展開することで、削除された接続を近似的に復元する。

この圧縮では属性値や述語の内容を利用しないため、ACORNの「predicate-agnostic」という性質が維持される。([arXiv][3])

### ACORN-1

ACORN-1は (\gamma=1) とした軽量版である。構築時には通常のHNSWに近いグラフを作り、検索時に1ホップおよび2ホップ近傍を展開して述語部分グラフを近似する。

ACORN-(\gamma) より検索性能は低いものの、インデックス構築時間を大幅に削減できるため、頻繁にデータを更新する環境や、検索頻度が比較的低い用途に適している。([arXiv][3])

理論解析では、ACORN-(\gamma) の期待検索計算量は概ね

[
O!\left((d+\gamma)\log(sn)+\log(1/s)\right)
]

とされ、述語ごとの理想的なインデックスの (O(d\log(sn))) に近づく。ただし、この解析は理想化された近傍グラフを仮定しており、任意の実データに対する部分グラフの連結性を厳密に保証するものではない。([arXiv][3])

## 4. 実験方法

実験では、低カーディナリティの人工的な属性から、正規表現やリスト包含を含む高カーディナリティの実用的な属性までを評価している。

| データセット    |         規模・次元 | 主な述語                |
| --------- | ------------: | ------------------- |
| SIFT1M    |  約100万件・128次元 | ランダム属性の等価条件         |
| Paper     |  約203万件・200次元 | ランダム属性の等価条件         |
| TripClick |  約106万件・768次元 | 臨床分野の包含、公開日の範囲      |
| LAION-1M  |  約100万件・512次元 | キャプションの正規表現、キーワード包含 |
| LAION-25M | 約2465万件・512次元 | LAION-1Mと同様         |

LAIONでは、述語を満たすベクトルがクエリ付近に集まる正の相関、相関なし、クエリから遠ざかる負の相関を用意し、クエリ相関の影響も調べている。([arXiv][3])

比較対象は、HNSW事後フィルタリング、総当たり事前フィルタリング、FilteredVamana、StitchedVamana、NHQ、Milvus、および述語ごとに専用HNSWを構築するOracle Partitionである。HNSWには単に (K) 件を検索させるのではなく、約 (K/s) 件を過剰探索させており、比較ベースラインとしては比較的強い設定になっている。([arXiv][3])

主な評価指標はRecall@10に対するQPS、インデックス構築時間、インデックスサイズである。実験はAWS m5d.24xlargeのメモリ常駐環境で行われ、QPSは複数試行の平均として測定された。([arXiv][3])

## 5. 結果

Recall@10を0.9付近に固定した場合、ACORN-(\gamma) は適用可能な既存手法に対し、概ね次の性能向上を示した。

| 評価条件                           |    QPS向上 |
| ------------------------------ | -------: |
| SIFT1M・Paperの低カーディナリティ条件       |   約2～10倍 |
| TripClick・LAION-1Mの高カーディナリティ条件 |  約30～50倍 |
| LAIONの異なるクエリ相関                 | 約28～100倍 |
| LAION-25M                      |   1000倍超 |
| 全実験の範囲                         | 約2～1000倍 |

特に負のクエリ相関では、HNSW事後フィルタリングが条件不適合ノードを大量に探索する一方、ACORNは述語部分グラフ上を探索するため、大きな差が生じた。([arXiv][3])

距離計算回数についてもOracle Partitionに近い。Recall@10が0.8のSIFT1Mでは、Oracleが398回、ACORN-(\gamma) が611回、ACORN-1が約1000回、HNSW事後フィルタリングが約1838回であった。Paperデータセットでも同様の傾向が確認されている。([arXiv][3])

一方で、ACORN-(\gamma) のインデックス構築時間は大きい。LAION-25Mでは、HNSWの1,147.2秒に対してACORN-(\gamma) は38,007.5秒であり、表の数値から計算すると約33.1倍となる。ACORN-1はACORN-(\gamma) よりQPSが約1.5～5倍低いが、構築時間は約9～53倍短い。インデックスサイズについては、ACORN-(\gamma) がHNSWの最大約1.3倍、ACORN-1が最大約1.25倍に抑えられている。([arXiv][3])

## 6. 考察

ACORNの性能向上は、単に探索候補数を増やしたためではない。**述語を満たすノード間の局所的な接続を検索時に維持する**ことで、通常のHNSWを条件適合データだけに構築した場合に近い探索経路を再現している点が本質である。

また、述語ごとのエッジやインデックスを保持しないため、属性値の種類が数億～数千億通りに達する場合や、正規表現、範囲、リスト包含のように結果集合を事前列挙しにくい条件にも適用できる。これは、カテゴリ値ごとのインデックスを構築する従来手法に対する明確な優位点である。

ただし、以下の課題が残る。

第一に、ACORN-(\gamma) の構築コストは無視できない。論文本文にはHNSW比で「最大11倍」とする記述がある一方、掲載された表からはLAION-25Mで約33倍、LAION-1Mで約25倍となる。この数値的不整合は修正が必要である。([arXiv][3])

第二に、(\gamma) は想定する最小選択率 (s_{\min}) に依存する。実際の選択率がこれを大きく下回る極端に選択的な問い合わせでは、ACORNは事前フィルタリングへの切り替えを必要とする。

第三に、「述語非依存」とはグラフ構造が述語に依存しないという意味であり、正規表現や複雑な条件の評価自体が無料になるわけではない。公式実装例でも、条件を満たすIDを表す `filter_ids_map` を検索処理に与えているため、条件評価やビットマップ生成を含むエンドツーエンド遅延は別途検証する必要がある。([GitHub][4])

第四に、実験は主としてCPU・メモリ常駐の単一マシン環境であり、ディスク常駐、分散配置、オンライン更新、削除、並行ワークロード、DBMSへの統合コストは評価されていない。また、SIFT1MとPaperの属性はランダム生成であり、すべての実験が実運用の属性分布を再現しているわけではない。

さらに、TripClickの平均選択率について、データセット表では0.17と0.36、本文では0.17と0.26と記載されており、軽微な報告上の不整合が存在する。([arXiv][3])

## 7. まとめ

ACORNは、フィルタ付きベクトル検索を「検索後に結果を削除する問題」ではなく、**述語を満たす部分グラフ上で近似最近傍探索を行う問題**として捉え直した手法である。

ACORN-(\gamma) は、構築時間との交換で非常に高い検索性能を実現する。ACORN-1は検索速度を一部犠牲にしつつ、HNSWに近い構築コストを提供する。したがって、読み取り中心の大規模サービスではACORN-(\gamma)、更新や再構築が多い用途ではACORN-1という使い分けが考えられる。

任意の高カーディナリティ述語を扱いながら、専用インデックスに近い探索性能を得るという設計は明快であり、ベクトルデータベースのハイブリッド検索に対する重要な貢献である。

## 8. 総評

本論文の最大の長所は、述語ごとのインデックスを作ることなく、任意のフィルタに対して良好な探索グラフを生成するという、単純かつ汎用性の高い着想である。選択率、クエリ相関、データ規模、述語カーディナリティを変えた評価も充実しており、Oracle Partitionとの距離計算回数の比較によって、速度向上の理由も分析されている。FAISSベースのC++実装が公開されている点も再現性を高めている。([GitHub][4])

一方、ACORN-(\gamma) の大きな構築コスト、理論解析の理想化された仮定、エンドツーエンドの述語評価コストが不明な点、および構築時間・選択率に関する数値的不整合は弱点である。特に構築時間の「最大11倍」という主張は、表の値に合わせて修正すべきである。

**査読判定：採択（Accept、軽微修正を要求）**

新規性と実験的有効性は採択に十分である。ただし、数値記載の訂正、述語評価時間を含めた測定範囲の明確化、更新・ディスク常駐環境に関する制約の追記を求めたい。実際の掲載結果もSIGMOD 2024採択・PACMMOD掲載済みである。([ACM Digital Library][1])
