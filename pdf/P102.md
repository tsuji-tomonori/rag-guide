# P102：The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction

## 1. はじめに

本論文は、機械学習システムが本番運用に耐えられる状態かを評価するための指標として、**ML Test Score**を提案したものである。著者はEric Breck、Shanqing Cai、Eric Nielsen、Michael Salib、D. Sculleyで、2017年のIEEE International Conference on Big Dataに掲載された。

中心的な貢献は、MLシステム固有のリスクを28個の具体的なテスト項目に整理し、その実施状況を数値化した点にある。モデル精度だけでなく、データ品質、再現性、デプロイ、ロールバック、監視、公平性などを含む**システム全体の運用品質**を評価対象としている。

## 2. 背景

一般的なソフトウェアでは、期待する動作をコードとして比較的明確に記述できる。一方、MLシステムの動作は、コードだけでなく、学習データ、特徴量、モデル構成、外部環境の変化によって決まる。そのため、通常の単体テストや結合テストだけでは、システムの信頼性を十分に保証できない。

論文では、モデル学習をコンパイル処理になぞらえている。この見方では、ソースコードに相当するものは「プログラムと学習データ」の両方であり、データにもコードと同様のテストが必要となる。また、学習済みモデルには、実行バイナリと同様に、デバッグ可能性、バージョン管理、監視、ロールバックが求められる。

本論文が主に想定するのは、継続的に再学習され、サーバ上で低遅延推論を行う教師あり学習システムである。ただし、多くの項目はバッチ推論やクライアント側推論にも適用できるとしている。

## 3. 提案手法

### 3.1 28個のテスト項目

ML Test Scoreは、次の4領域にそれぞれ7項目、合計28項目を設定する。

| 領域          | テスト項目                                                                                                                                  |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| **特徴量・データ** | ①特徴量の期待値をスキーマ化する、②全特徴量が有益である、③特徴量の計算・保守コストが高すぎない、④特徴量がポリシーや上位要件を満たす、⑤プライバシー制御がある、⑥新特徴量を迅速に追加できる、⑦特徴量生成コードをテストする                        |
| **モデル開発**   | ①モデル仕様をレビュー・バージョン管理する、②オフライン指標とオンライン指標が相関する、③ハイパーパラメータを調整する、④モデル陳腐化の影響を把握する、⑤単純なベースラインより複雑なモデルが有益である、⑥重要なデータスライスで品質を保証する、⑦公平性・包摂性を検査する |
| **MLインフラ**  | ①学習を再現可能にする、②モデル仕様を単体テストする、③パイプライン全体を結合テストする、④配信前にモデル品質を検証する、⑤個別事例を追跡してデバッグできる、⑥カナリアリリースを行う、⑦安全かつ迅速にロールバックできる                          |
| **本番監視**    | ①依存先の変更を通知する、②入力データが不変条件を満たす、③学習時と推論時の特徴量にずれがない、④モデルが古すぎない、⑤数値的に安定している、⑥速度・遅延・メモリなどが劣化していない、⑦本番予測品質が劣化していない                            |

これらは単なる確認事項ではなく、「その条件が真であることを継続的に検証し、偽になった場合に修正する」というアサーションとして定義されている。

### 3.2 スコア計算

各テストの得点は次のように定められる。

* 未実施：0点
* 手動で実施し、結果を文書化・共有：0.5点
* 自動化され、反復的に実行：1点

4領域についてそれぞれ7項目の合計点を求め、最終スコアはその最小値とする。

[
\text{ML Test Score}
====================

\min
\left(
S_{\text{Data}},
S_{\text{Model}},
S_{\text{Infra}},
S_{\text{Monitoring}}
\right)
]

最小値を採用するのは、一部の領域だけが成熟していても本番信頼性は保証できず、最も弱い領域がシステム全体のボトルネックになるという考え方による。

スコアの解釈は、おおむね以下である。

* 0点：研究プロジェクトに近い
* 0超～1点：重大な信頼性上の穴が存在する可能性
* 1超～2点：基本的な本番化は実施済み
* 2超～3点：一定のテストはあるが、自動化の余地がある
* 3超～5点：ミッションクリティカル用途にも適した自動テスト・監視
* 5点超：例外的に高い成熟度



## 4. 実験方法

本論文の「実験」は、モデル精度を比較する一般的なベンチマーク実験ではなく、ルーブリックを実システムに適用する**フィールド評価**である。

著者らは、Google内の多様な製品領域から36チームを対象に構造化インタビューを行い、各システムについて28項目の実施状況を評価した。項目ごとの平均得点を集計するとともに、チェックリストが発見した問題、導入を妨げる要因、チームからの意見を定性的に分析している。

また、TFXの標準的な機能を利用する仮想システムについてもスコアを算出した。ただし、当時は実際のTFX利用システムを対象とした評価までは行われていない。

## 5. 結果

36システムの項目別平均点は全般に低く、図4から概算すると、領域別平均は次の程度である。

| 領域      | 1項目当たりの平均点 |
| ------- | ---------: |
| 特徴量・データ |      約0.36 |
| モデル開発   |      約0.40 |
| MLインフラ  |      約0.39 |
| 本番監視    |      約0.29 |

特に本番監視の成熟度が低い。個別項目では、モデル仕様のコードレビューは比較的普及していた一方、包摂性・公平性の検査や、パイプライン全体の結合テストは低水準だった。

さらに、次の知見が得られている。

* 28項目のうち、80％を超えるチームが実施していた項目は一つもなかった。一方、多くの項目は少なくとも半数のチームで部分的には実施されていた。
* 結合テストは特に実施率が低く、導入済みでも推論系だけを対象とし、学習パイプラインを含まない場合が多かった。
* カナリアリリースは比較的普及していたが、既存のリリース基盤が整っているチームほど実施しやすかった。
* 学習時と推論時の特徴量の不一致である**training/serving skew**は、多数の本番障害につながる重要な問題であるにもかかわらず、十分に監視されていなかった。
* チェックリストにより、約1,000行に及ぶ未テストの特徴量生成コード、国別の性能劣化を検出できない監視設計、音声認識における方言や評価者構成への配慮不足などが発見された。

## 6. 考察

本論文の最大の長所は、曖昧だった「本番運用準備度」を、具体的かつ実行可能な項目に分解した点である。特に、モデルの平均精度だけでなく、データ依存関係、再現性、カナリアリリース、ロールバック、データスライス別性能、運用監視まで含めたことは重要である。

また、最終得点を4領域の最小値とした設計は、監視が欠落しているのにモデル開発だけで高得点を得る、といった状態を防ぐ。これは、MLシステムの信頼性が直列的なパイプライン全体によって決まるという考え方をよく表している。

一方、以下の限界がある。

第一に、評価対象がGoogle内部の36システムに限定されており、業種、企業規模、規制環境の異なる組織への一般化は検証されていない。評価もインタビューを中心としているため、自己申告や評価者判断の影響を受ける可能性がある。

第二に、全項目が同一の重みである。広告推薦におけるモデル陳腐化と、医療システムにおけるプライバシー侵害を同じ1点として扱うため、実際のリスクや損失の大きさを反映できない。

第三に、自動化すれば1点となるため、**テストの質、カバレッジ、アラートの有効性**までは評価されない。誤検知ばかりの自動監視でも形式上は高得点となり得る。

第四に、スコアの向上が障害件数、保守費用、復旧時間、技術的負債を実際に削減することを、長期的・因果的に検証していない。したがって、本スコアは厳密な信頼性指標というより、改善を促す成熟度チェックリストとして理解するのが適切である。

実務で利用する場合は、28項目をそのまま適用するのではなく、システムの危険度に応じた重み付け、証跡へのリンク、必須ゲートの設定、障害指標との時系列比較を追加するのが望ましい。

## 7. まとめ

本論文は、MLシステムの本番運用準備度を、データ、モデル、インフラ、監視の4領域・28項目で評価するML Test Scoreを提案した。

その本質は、モデル精度の向上ではなく、**MLシステムを継続的に安全に変更・配信・監視・復旧できる状態にすること**にある。Google内部の36システムへの適用では、専門性の高いチームであっても、結合テスト、training/serving skew監視、公平性評価などに大きな不足があることが示された。

### 評価

実務的重要性、問題設定の明確さ、28項目の具体性、36システムへの適用実績を高く評価する。特に、後続のMLOps実践に必要となる考え方を、簡潔なルーブリックとして整理した点は優れている。

ただし、科学的検証としては、単一企業内の観察研究であること、スコア設計の妥当性比較がないこと、スコア改善と障害削減との因果関係を示していないことが弱点である。したがって、厳密な測定理論としてではなく、**MLシステムの弱点を発見し、組織的な改善を開始するための実践的フレームワーク**として高く評価できる。
