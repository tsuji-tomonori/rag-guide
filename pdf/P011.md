# 1. はじめに

**論文名:** *Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks*
**著者:** Yunfan Gao, Yun Xiong, Meng Wang, Haofen Wang
**公開日:** 2024年7月26日

本論文は、RAGを単純な「検索してから生成する」直線的な処理ではなく、複数の機能部品を組み替えられる**計算グラフ型システム**として捉え直す論文である。

中心的な主張は、現在のRAGには検索器、クエリ変換、再ランキング、検証、ルーティング、反復制御など多数の機構が導入されており、従来のNaive RAG／Advanced RAGという分類だけでは表現しきれない、というものである。そこで著者らは、RAGをモジュール、サブモジュール、オペレータの3階層に分解する**Modular RAG**を提案している。新しい検索アルゴリズムそのものというより、既存および将来のRAG手法を整理・設計するためのアーキテクチャ論、あるいはポジションペーパーに近い。 ([arXiv][1])

# 2. 背景

初期のNaive RAGは、概ね以下の3段階で構成される。

1. 文書をチャンク化して索引を作る
2. クエリと類似したチャンクを検索する
3. 検索結果とクエリをLLMに入力して回答を生成する

しかし、この方式には、クエリと回答文書の表面的な意味類似度に依存しすぎること、無関係・冗長・矛盾したチャンクがLLMに混入すること、複雑な質問を十分に分解できないことなどの問題がある。

Advanced RAGでは、検索前にクエリを書き換えるPre-retrieval処理や、検索後に再ランキング・圧縮を行うPost-retrieval処理が導入された。一方、実運用ではさらに以下の要求が生じる。

* テキスト、表、SQLデータベース、知識グラフなど異種データの統合
* モジュール単位のデバッグ、説明可能性、保守性
* タスクや計算資源に応じた検索器・生成器の選択
* 条件分岐、並列処理、反復検索、終了判定を含むワークフロー制御

このため、RAGを固定パイプラインとしてではなく、動的に構成可能なシステムとして定義する必要がある、というのが論文の問題設定である。著者らは、Naive RAGはAdvanced RAGの特殊例、Advanced RAGはModular RAGの特殊例と位置付けている。

# 3. 提案手法

## 3.1 3階層の構造

Modular RAGは、次の3階層から構成される。

* **L1: Module**
  RAGにおける大きな処理段階。
* **L2: Sub-module**
  モジュール内部の機能分類。
* **L3: Operator**
  実際のモデル、関数、アルゴリズム、プロンプトなどの具体的実装。

例えばPost-retrievalがモジュール、Rerankがサブモジュール、Cross-EncoderやLLMによる再ランキングがオペレータに相当する。

モジュールとオペレータの接続関係を著者らは**RAG Flow**と呼び、システム全体を有向計算グラフとして表現する。これにより、オペレータの交換だけでなく、処理順序、分岐、並列実行、ループも記述できる。

## 3.2 6つの主要モジュール

| モジュール          | 役割              | 代表的な処理                                                                        |
| -------------- | --------------- | ----------------------------------------------------------------------------- |
| Indexing       | 文書を検索可能な形に整理する  | チャンクサイズ調整、メタデータ付与、Small-to-Big、階層インデックス、知識グラフ化                                |
| Pre-retrieval  | 検索前にクエリを改善する    | Multi-Query、サブクエリ分解、Query Rewrite、HyDE、Step-back Prompting、Text-to-SQL/Cypher |
| Retrieval      | 関連情報を検索する       | Sparse／Dense／Hybrid検索、検索器選択、教師あり微調整、LM-supervised Retrieval                   |
| Post-retrieval | 検索結果を整理する       | 再ランキング、プロンプト圧縮、不要チャンク選択、LLM Critique                                          |
| Generation     | 回答を生成・検証する      | Generator Fine-tuning、強化学習、Retrieverとの共同学習、外部知識・モデルによる検証                      |
| Orchestration  | RAG Flow全体を制御する | Routing、Scheduling、Judge、Fusion                                               |

特に重要なのが**Orchestrationモジュール**である。従来のRAGでは処理順序が固定されていたが、Modular RAGではクエリの種類や途中結果に応じて、次に実行するモジュールを変える。

Routingでは、メタデータ、キーワード、意味的意図などに基づいて適切なRAG Flowを選択する。Schedulingでは、再検索が必要か、回答を終了してよいか、別の経路に戻るべきかをRule JudgeやLLM Judgeが判定する。Fusionでは、複数経路の検索結果や生成結果をLLM、重み付きアンサンブル、Reciprocal Rank Fusionなどで統合する。

## 3.3 RAG Flowのパターン

本文で整理される主要なFlowは次の通りである。

**Linear Pattern**
クエリ変換、検索、再ランキング、生成を固定順序で実行する。RRRなどが例である。

**Conditional Pattern**
Routingモジュールが複数のパイプラインから1つを選択する。例えば、一般質問は軽量な検索を使い、法律・医療など高信頼性が必要な質問には検証付きフローを使う。

**Branching Pattern**
複数の経路を並列に実行して最後に統合する。

* Pre-retrieval branching: 複数のサブクエリを生成して並列検索
* Post-retrieval branching: 各検索文書について個別に生成し、回答を統合

**Loop Pattern**
検索と生成を複数回実行する。

* Iterative retrieval: 前回の出力を使って固定回数反復
* Recursive retrieval: 前段の結果から新しい質問を作り、木構造的に深掘り
* Adaptive／Active retrieval: 必要なときだけ検索し、十分な情報が得られた時点で終了

**Tuning Pattern**
Retrieverのみ、Generatorのみ、または両者を共同で学習する。共同学習では、Retrieverの文書選好とGeneratorの文書選好を整合させる。

なお、論文中では、要旨で「linear、conditional、branching、looping」の4種、本文ではTuningを含む5系統、導入部では「6つの典型パターン」と記述されている。Pre/Post branchingを別々に数えると6つになるが、分類単位の表現には若干の不統一がある。

# 4. 実験方法

本論文では、著者らが新しいRAGシステムを実装し、ベンチマーク上で性能を比較するような**独自の定量実験は行われていない**。

代わりに、以下の方法で提案フレームワークを説明している。

* 既存のRAG研究をモジュールとオペレータに分解する
* 各Flowを数式および疑似コードで形式化する
* 代表的な既存手法をFlowパターンへ対応付ける
* 新しい手法が既存モジュールの再結合、新Flowの追加、新オペレータの追加として説明できることを示す

分析対象例には、RRR、REPLUG、ITER-RETGEN、Tree of Clarifications、FLARE、Self-RAG、RA-DIT、PlanRAGなどが含まれる。例えばFLAREは低確率トークンが現れた場合のみ再検索するAdaptive Flow、Self-RAGは特殊トークンによって検索・批評・生成を制御するTuning-based Adaptive Flowとして整理される。

したがって、通常の実験論文で期待されるデータセット、評価指標、ベースライン、アブレーション、統計検定、推論時間、APIコストなどは提示されていない。

# 5. 結果

本論文の成果は、数値的な性能向上ではなく、主として次の概念的成果である。

第一に、複雑化したRAGを、**Module–Sub-module–Operator**という共通の粒度で記述できる枠組みを示した。

第二に、既存手法をLinear、Conditional、Branching、Loop、TuningというFlowの組合せとして整理した。これにより、従来は個別の手法として扱われていたRRR、REPLUG、FLARE、Self-RAGなどを、共通の設計空間上に位置付けられる。

第三に、今後のRAG研究を以下の3種類の拡張として整理した。

1. 既存モジュールの再結合
2. 既存オペレータを使った新しいFlowの設計
3. 新しいオペレータと、それに伴う新Flowの導入

論文ではDR-RAG、PlanRAG、Multi-Head RAGをそれぞれの例として挙げ、Modular RAGが将来の手法も取り込めることを論じている。

ただし、「Modular RAGが従来方式より精度、レイテンシ、コスト、堅牢性で優れる」という定量的結果は示されていない。

# 6. 考察

## 良い点

最も評価できる点は、RAGをモデル単体ではなく、**ソフトウェアアーキテクチャおよびワークフロー設計の問題**として扱ったことである。

特にRouting、Scheduling、Fusionを独立したOrchestration機能として明示した点は重要である。実運用のRAGでは、検索器やLLMの選択以上に、どの条件で検索するか、いつ再検索するか、どの結果を採用するかがシステム性能を左右する。本論文はこの制御部分をRAGの主要構成要素として明確に位置付けている。

また、モジュール単位の交換やデバッグという視点は、保守性、再利用性、観測可能性を重視する実務上の設計と整合する。既存研究を俯瞰するための語彙としても利用価値が高い。

## 制約・問題点

最大の弱点は**実証評価の欠如**である。モジュール化によって保守性や拡張性が高まるという主張は妥当だが、モジュール数の増加は、レイテンシ、APIコスト、障害点、設定探索空間も増大させる。こうした負の側面は定量的に検証されていない。

また、各モジュールの入出力契約、状態管理、エラー処理、キャッシュ、並列実行時の整合性、タイムアウト、監視指標など、実際に再構成可能なRAG基盤を実装するための仕様は十分に定義されていない。したがって、論文の「LEGOのように交換可能」という表現は設計理念としては分かりやすいものの、RetrieverとGeneratorの分布整合など、モジュール間の強い依存関係を単純化している。

さらに、サーベイとして見ると、文献検索方法、採用・除外基準、対象期間などの体系的レビュー手順が示されていない。分類の網羅性や再現性は保証されておらず、Flowパターンの数にも前述の表記揺れがある。

# 7. まとめ

本論文は、現代のRAGを固定された「Index–Retrieve–Generate」パイプラインではなく、交換可能な機能部品と制御ロジックからなる**再構成可能な計算グラフ**として定式化した。

主要な貢献は以下である。

* RAGをModule、Sub-module、Operatorの3階層に分解
* IndexingからOrchestrationまで6つの主要モジュールを整理
* 線形、条件分岐、並列分岐、反復、学習を含むFlowパターンを形式化
* 既存手法と将来手法を共通の設計空間で説明

新しい最高性能モデルを提示する論文ではないが、複雑なRAGシステムを設計・比較・説明するための概念フレームワークとして有用である。
