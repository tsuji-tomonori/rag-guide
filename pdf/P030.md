# 1. はじめに

本論文は、関連度ラベルを一切用いずに高性能なゼロショット密検索を実現する **HyDE（Hypothetical Document Embeddings）** を提案する。

中心的なアイデアは、ユーザーの短い検索クエリをそのまま埋め込むのではなく、まずLLMに「この質問への回答文」を仮想的に生成させ、その文章を検索用ベクトルへ変換することである。生成文には誤情報が含まれてもよく、最終的には実在する文書コーパスから検索結果を取得する。([arXiv][1])

**論文名:** *Precise Zero-Shot Dense Retrieval without Relevance Labels*
**著者:** Luyu Gao、Xueguang Ma、Jimmy Lin、Jamie Callan
**主題:** 教師なし密検索、LLMによるクエリ拡張、ゼロショット検索

# 2. 背景

一般的な密検索では、クエリ (q) と文書 (d) をそれぞれエンコーダでベクトル化し、その内積を関連度として用いる。

[
\mathrm{sim}(q,d)
=================

\langle \mathrm{enc}_q(q), \mathrm{enc}_d(d) \rangle
]

この方式では、クエリベクトルと文書ベクトルが同じ空間上で「関連するもの同士が近くなる」ように学習する必要がある。そのため通常は、大量のクエリ–関連文書ペアが必要となる。

一方、Contrieverのような教師なし対照学習モデルは文書間の意味的類似性を学習できるものの、短く質問形式で書かれたクエリと、説明的な文書との形式的・意味的ギャップを十分に解消できない。実際、実験では未調整のContrieverがBM25を下回るケースも確認されている。

本論文は、この難しい「クエリと文書の対応関係の学習」を直接行う代わりに、LLMでクエリを文書らしい表現へ変換するという方針を採用した。

なお、著者はHyDEを教師なしに近い手法として位置付けているが、厳密にはLLMの指示追従学習には教師データが使われている。HyDE自身が新たな関連度ラベルやタスク固有の学習を必要としない、という意味での教師なしである。([ar5iv][2])

# 3. 提案手法

HyDEは次の処理を行う。

### ① 仮想文書の生成

クエリとタスク固有の指示をInstruction-following LLMへ入力する。

例えばWeb検索では、次のような指示を与える。

> 質問に回答する文章を書いてください。

SciFactでは科学論文風、FiQAでは金融記事風、ArguAnaでは反論文、Mr.TyDiでは検索対象言語の文章を生成するよう指示する。

LLMが生成する文章を仮想文書 (\hat d) とする。これは検索対象コーパスに実在する必要はなく、事実誤認やハルシネーションを含んでも構わない。

### ② 仮想文書の埋め込み

生成した仮想文書をContrieverなどの文書エンコーダ (f) でベクトル化する。

複数の仮想文書を生成する場合、検索用クエリベクトルは概念的に次の平均で表される。

[
\hat v_q
========

\frac{1}{N}
\sum_{k=1}^{N} f(\hat d_k)
]

論文では、元のクエリ自体の埋め込み (f(q)) を平均に加える形式も提示している。

### ③ 実文書の検索

コーパス中の各文書も同じエンコーダ (f) で事前にベクトル化しておく。仮想文書ベクトルとの内積が大きい実文書を検索結果として返す。

したがってHyDEは、

[
\text{クエリ}
\rightarrow
\text{仮想的な関連文書}
\rightarrow
\text{文書ベクトル}
\rightarrow
\text{実文書}
]

という経路で検索する。

著者は、Contrieverの固定長ベクトルが情報を圧縮する「dense bottleneck」として働き、生成文中の細かな誤情報を捨てつつ、主題や意味的方向だけを残すと説明している。ただし、これは本論文で十分に因果検証された機構というより、性能を説明する仮説である。

# 4. 実験方法

実装では、生成モデルに `text-davinci-003`、英語検索にContriever、多言語検索にmContrieverを使用している。生成温度は0.7で、検索と評価にはPyseriniを利用した。

評価データは以下の構成である。

* Web検索：TREC DL19、TREC DL20
* BEIR：SciFact、ArguAna、TREC-COVID、FiQA、DBPedia、TREC-NEWS
* 多言語検索：Mr.TyDiのスワヒリ語、韓国語、日本語、ベンガル語

教師なし比較対象としてBM25、Contriever、mContrieverを用い、参考上限としてMS MARCOなどの関連度データで学習したDPR、ANCE、ContrieverFT、mContrieverFTなどを比較した。

評価指標は、Web検索ではMAP、nDCG@10、Recall@1000、BEIRではnDCG@10とRecall@100、多言語検索ではMRR@100である。

さらに生成モデルをFlan-T5 11B、Cohere 52B、GPT 175Bに変更し、生成モデル規模の影響も調べている。

# 5. 結果

## Web検索

| データ  | 手法         |      MAP |  nDCG@10 | Recall@1000 |
| ---- | ---------- | -------: | -------: | ----------: |
| DL19 | Contriever |     24.0 |     44.5 |        74.6 |
| DL19 | BM25       |     30.1 |     50.6 |        75.0 |
| DL19 | **HyDE**   | **41.8** | **61.3** |    **88.0** |
| DL20 | Contriever |     24.0 |     42.1 |        75.4 |
| DL20 | BM25       |     28.6 |     48.0 |        78.6 |
| DL20 | **HyDE**   | **38.2** | **57.9** |    **84.4** |

DL19では、HyDEはContrieverに対してnDCG@10を **16.8ポイント**、DL20では **15.8ポイント** 改善した。

DL19では、教師ありContrieverFTのnDCG@10が62.1であるのに対し、HyDEは61.3であり、関連度ラベルなしでほぼ同等の性能を示した。DL20ではHyDEの57.9に対してContrieverFTは63.2であり、精度面では差が残るが、Recall@1000は84.4対85.8と比較的近い。

## BEIR

HyDEは6データセットすべてで未調整Contrieverを上回った。BM25との比較でも、TREC-COVIDで0.2ポイント下回った以外はHyDEが優れている。

特にArguAnaではnDCG@10がContrieverの37.9から46.6へ、TREC-NEWSでは34.8から44.0へ向上した。一方、FiQAやDBPediaでは教師ありContrieverFTがHyDEを上回っており、専門領域やエンティティ検索では改善余地が残る。

## 多言語検索

MRR@100は次のように改善した。

* スワヒリ語：38.3 → 41.7
* 韓国語：22.3 → 30.6
* 日本語：19.5 → 30.7
* ベンガル語：35.3 → 41.3

特に日本語では **11.2ポイント** の大幅な向上である。ただし、教師ありmContrieverFTは日本語で32.4を記録しており、HyDEが常に教師ありモデルを上回るわけではない。

## 生成モデルの影響

TREC DL19のnDCG@10は、Flan-T5が48.9、Cohereが53.8、GPTが61.3だった。DL20でもすべての生成モデルが元のContrieverを改善しており、概して高性能・大規模な生成モデルほど検索性能も高くなる傾向が示された。

# 6. 考察

本手法の最大の長所は、**クエリを文書の表現空間へ直接移すという単純かつ強力な発想**にある。検索モデルを再学習せず、既存のLLMと既存の密検索インデックスを組み合わせるだけで大幅な改善を得ている。特にラベルを収集できない新規ドメインや、検索システムの立ち上げ初期に有用である。

また、ハルシネーションを完全に防ぐのではなく、「最終回答として使わず、検索方向を示す意味的プロトタイプとして使う」という設計が興味深い。生成された事実が誤っていても、トピック、語彙、文体、関連概念が適切であれば検索に役立つ可能性がある。

一方、以下は重要な弱点である。

**計算コストと遅延:** クエリごとにLLM生成が必要であり、通常のBM25やContrieverよりオンライン処理が重い。しかし論文では、APIコスト、生成時間、検索全体のレイテンシが評価されていない。

**プロンプト依存性:** データセットごとに科学論文、金融記事、反論文などの指示を人手で設定している。このため「完全にそのまま使えるゼロショット手法」という主張には注意が必要である。著者自身もFiQAやDBPediaの低性能について、指示の具体性不足が原因かもしれないと述べている。

**ハルシネーション除去の検証不足:** dense bottleneckが誤情報を除去するという説明は直感的だが、生成文中の誤情報量と検索性能の関係、数値・固有名詞の誤りが検索に与える影響などは分析されていない。

**再現性:** 主結果が当時のプロプライエタリモデルである`text-davinci-003`に依存し、確率的生成も使用している。生成回数、温度、生成長、乱数による分散について、より詳細なアブレーションが望まれる。

**曖昧なクエリ:** 数式上は仮想文書ベクトルの分布が単峰的、すなわちクエリの意図が一つであると仮定している。複数解釈を持つクエリでは、平均ベクトルによって異なる検索意図が混ざる可能性がある。

# 7. まとめ

HyDEは、検索クエリからLLMで仮想的な関連文書を生成し、その文書を教師なし密検索モデルで埋め込む手法である。

関連度ラベルを用いず、モデルの追加学習も行わないにもかかわらず、Web検索、科学的主張検証、金融検索、多言語検索などでContrieverを一貫して改善し、一部では教師ありモデルに匹敵した。

本論文の本質は、クエリと文書の関連度を直接学習するのではなく、LLMに「関連文書の典型例」を生成させることで、関連性モデリングを生成問題へ置き換えた点にある。

# 8. 総評

**模擬査読判定：採択（Weak Accept）**

発想の新規性、手法の単純さ、複数タスク・複数言語にわたる一貫した改善は高く評価できる。特に、LLMの生成能力と教師なし文書エンコーダを役割分担させ、追加学習なしで強い検索性能を実現した点は、情報検索とRAGの後続研究に影響を与え得る重要な貢献である。

一方で、ハルシネーションが実際に埋め込み段階で除去されていることの実証、LLM呼び出しのコスト・遅延、プロンプト感度、確率的生成による分散など、実運用上重要な分析が不足している。そのためStrong AcceptではなくWeak Accept相当と判断する。
