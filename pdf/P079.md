# P079　Corrective Retrieval Augmented Generation

## 1. はじめに

本論文は、検索拡張生成（Retrieval-Augmented Generation: RAG）において、**検索結果が誤っている場合に生成性能が悪化する問題**を扱う。通常のRAGは取得文書をほぼ無条件に生成モデルへ渡すため、無関係・不正確な文書が検索されると、かえって幻覚や誤答を誘発する。

著者らはこの問題に対し、検索結果を生成前に評価し、その品質に応じて利用方法を切り替える **Corrective Retrieval Augmented Generation（CRAG）** を提案した。CRAGの本質は、「検索結果を常に信用する」のではなく、**検索結果を評価し、必要なら修正・置換する制御層**をRAGへ追加することである。

---

## 2. 背景

標準的なRAGは、質問 (X) に対してコーパスから上位 (K) 件の文書 (D) を検索し、生成モデルが (X) と (D) をもとに回答 (Y) を生成する。

この構造では、検索器と生成器が強く結合している。そのため、生成器が高性能であっても、検索された文書が無関係であれば回答品質が低下する。さらに、取得文書全体をそのまま入力すると、回答に必要な部分だけでなく、大量の周辺情報やノイズまでコンテキストへ含まれてしまう。

Self-RAGなどの先行研究は、主として「検索すべきか」「どの文書を参照すべきか」を生成モデル自身に判断させる。一方、CRAGは、**検索が実施された後、その検索結果が信頼できなかった場合にどう修正するか**に焦点を置いている。

本論文が対象とする問題は、次のように整理できる。

* 関連文書が検索されない
* 一部だけ関連する長い文書が検索される
* 誤った文書が生成モデルを強く誘導する
* 静的コーパス内に回答に必要な情報が存在しない

---

## 3. 提案手法

CRAGは、主に「検索評価器」「アクション選択」「知識精緻化」「Web検索」の4要素で構成される。

### 3.1 検索評価器

質問と各検索文書のペアを、fine-tuningした **T5-large（約0.77Bパラメータ）** に入力し、文書の関連度を (-1) から (1) の範囲で評価する。

通常、質問ごとに約10文書を個別に評価し、それらのスコアから検索結果全体の信頼度を判定する。Self-RAGで用いられた7B規模のcriticモデルより小さく、生成LLMとは独立して動作する点が特徴である。

### 3.2 3種類のアクション

評価結果に基づき、以下のいずれかを選択する。

| 判定            | 条件                | 処理                   |
| ------------- | ----------------- | -------------------- |
| **Correct**   | 少なくとも1文書が上側閾値を超える | 取得文書を精緻化して利用         |
| **Incorrect** | 全文書が下側閾値を下回る      | 取得文書を破棄し、Web検索へ切り替える |
| **Ambiguous** | 上記以外              | 精緻化した取得文書とWeb検索結果を併用 |

二値的にCorrect／Incorrectだけを選ぶと、検索評価器の誤判定がシステム全体へ直接影響する。そこで中間状態のAmbiguousを設け、評価器の確信度が低いときには内部知識と外部知識を併用する。

### 3.3 Knowledge Refinement

CorrectまたはAmbiguousの場合、検索文書をそのまま生成器へ渡さず、以下の処理を行う。

1. 文書を数文程度の短い「knowledge strip」に分割する。
2. 検索評価器を再利用して、質問との関連度をstrip単位で計算する。
3. 無関係なstripを除外する。
4. 関連するstripを元の順序で再結合する。

これは論文中で **decompose-then-recompose** と呼ばれる。文書全体ではなく、回答に必要な断片だけをコンテキストへ残す処理である。

### 3.4 Web検索

IncorrectまたはAmbiguousの場合、外部知識を取得する。

質問をChatGPTで最大3個程度の検索キーワードに書き換え、Google Search APIを利用してWeb検索を行う。上位URLのうちWikipediaなどの比較的管理された情報源を優先し、ページ内の段落を検索評価器で選別する。選択された段落を外部知識 (k_{\mathrm{ex}}) として生成器へ渡す。

最終的な入力は次のようになる。

* Correct：質問＋精緻化された内部知識
* Incorrect：質問＋Webから取得した外部知識
* Ambiguous：質問＋内部知識＋外部知識

## 4. 実験方法

### 4.1 データセット

4種類の異なる生成タスクで評価している。

| データセット        | タスク              | 指標        |
| ------------- | ---------------- | --------- |
| PopQA         | 長尾エンティティに関する短文QA | Accuracy  |
| Biography     | 人物伝記の長文生成        | FactScore |
| PubHealth     | 医療・健康分野の真偽判定     | Accuracy  |
| ARC-Challenge | 科学常識の多肢選択問題      | Accuracy  |

PopQAでは、Wikipediaの月間閲覧数が100未満の希少エンティティからなる1,399問をテストセットとしている。

### 4.2 モデルと検索結果

生成器には主に以下を使用した。

* LLaMA2-hf-7b
* Self-RAG用にfine-tuningされたSelfRAG-LLaMA2-7b

検索結果には、Self-RAGと比較条件をそろえるため、Contrieverによる同一の取得結果を使用している。

検索評価器はPopQAの残りの学習データでfine-tuningし、そのままBiography、PubHealth、ARC-Challengeへ転用した。ただし、Correct／Incorrect判定に用いる上下の閾値はデータセットごとに経験的に設定されている。内部知識とWeb知識はいずれも上位5件を利用する。

### 4.3 比較手法

以下と比較している。

* 検索を使わないLLaMA2、Alpaca、ChatGPT
* 標準RAG
* SAIL
* Self-RAG
* Ret-ChatGPT、Ret-LLaMA2
* Web情報を常時追加するRAG／Self-RAG

また、CRAGを標準RAGへ組み込んだ構成を **CRAG**、Self-RAGへ組み込んだ構成を **Self-CRAG** として評価した。

コード、検索評価器の重み、データ準備用スクリプトは公式リポジトリで公開されている。([GitHub][1])

## 5. 結果

### 5.1 主要結果

同一生成器内での比較結果は次の通りである。括弧内は絶対改善幅、すなわちパーセントポイントまたはFactScoreポイントを表す。

| 生成器・比較                      |          PopQA |       Biography |       PubHealth |             ARC |
| --------------------------- | -------------: | --------------: | --------------: | --------------: |
| LLaMA2-hf-7b：RAG            |           50.5 |            44.9 |            48.9 |            43.4 |
| LLaMA2-hf-7b：CRAG           | **54.9（+4.4）** |  **47.7（+2.8）** | **59.5（+10.6）** | **53.7（+10.3）** |
| SelfRAG-LLaMA2-7b：RAG       |           52.8 |            59.2 |            39.0 |            53.2 |
| SelfRAG-LLaMA2-7b：CRAG      | **59.8（+7.0）** | **74.1（+14.9）** | **75.6（+36.6）** | **68.6（+15.4）** |
| SelfRAG-LLaMA2-7b：Self-RAG  |           54.9 |            81.2 |            72.4 |        **67.3** |
| SelfRAG-LLaMA2-7b：Self-CRAG | **61.8（+6.9）** |  **86.2（+5.0）** |  **74.8（+2.4）** |      67.2（−0.1） |

標準RAGにCRAGを追加した場合は、すべてのデータセットで改善している。特に、SelfRAG-LLaMA2-7bを生成器としたPubHealthでは、39.0から75.6へ大きく向上した。

Self-RAGとの比較でも多くの条件で改善したが、すべてではない。SelfRAG-LLaMA2-7b上のARC-Challengeでは67.3から67.2へわずかに低下している。

### 5.2 検索評価器

PopQAにおける検索品質判定精度は以下であった。

| 評価器                      | Accuracy |
| ------------------------ | -------: |
| T5ベースの検索評価器              | **84.3** |
| ChatGPT                  |     58.0 |
| ChatGPT＋Chain-of-Thought |     62.4 |
| ChatGPT＋few-shot         |     64.7 |

専用にfine-tuningした小型評価器が、プロンプトのみで評価したChatGPTを上回っている。ただし、fine-tuning済みモデルとzero／few-shotモデルの比較であり、条件が完全に対称ではない。

### 5.3 アブレーション

Correct、Incorrect、Ambiguousのいずれを除去しても性能は低下した。また、以下の処理を除去した場合にも性能が低下した。

* 文書の分割・精緻化
* 検索クエリの書き換え
* Web検索結果の選別

特に文書精緻化と外部知識選別の寄与が大きく、単にWeb検索結果を追加するだけでは十分でないことが示された。

### 5.4 検索性能低下に対する頑健性

正しい検索文書を意図的に除去し、検索精度を低下させた実験では、Self-RAGよりSelf-CRAGの方が生成精度の低下が緩やかだった。

また、Web情報を全質問へ一律に追加したRAGよりもCRAGの方が高性能であった。したがって、改善は単なるWeb知識量の増加ではなく、検索品質に応じた選択的な処理によるものと考えられる。

## 6. 考察

### 評価できる点

**第一に、問題設定が実用的である。**
現実のRAGでは、検索器が常に正しい文書を返すとは限らない。CRAGは生成器だけでなく、「検索失敗後の復旧」に焦点を当てている。

**第二に、制御規則が単純で解釈しやすい。**
Correct、Incorrect、Ambiguousの3分岐により、どの知識が回答へ使われたかを追跡しやすい。Ambiguousは評価器の不確実性を吸収する安全弁として機能する。

**第三に、生成モデルから分離されている。**
特別なcritic tokenの出力能力を生成器へ学習させるSelf-RAGとは異なり、CRAGは外部評価器として追加できる。このため、既存RAGへの導入や生成モデルの交換が比較的容易である。

### 問題点・限界

**検索評価器への依存が大きい。**
評価器はPopQAで学習され、他の3データセットへ転用されている一方、判定閾値はデータセットごとに経験的に調整されている。未知ドメインへそのまま導入した場合の性能や、閾値調整に必要な検証データ量は明確でない。論文自身も、外部検索評価器のfine-tuningが必要である点を主要な限界として挙げている。

**関連性評価と事実性評価は異なる。**
CRAGの評価器は、主として質問と文書の関連性を判定する。関連性が高くても、文書内容が古い、誤っている、相互に矛盾している可能性は残る。したがってCRAGは検索結果の「有用性」を改善するが、厳密なfact verifierではない。

**外部サービスへの依存がある。**
クエリ書き換えにChatGPT、検索にGoogle Search APIを使うため、APIコスト、待ち時間、検索順位の時間変化、プライバシー、再現性が問題になり得る。また、Webページの信頼性判断はWikipedia優先などのヒューリスティックに依存し、偽情報やprompt injectionへの体系的な対策は示されていない。

**計算量評価は限定的である。**
報告された平均実行時間はRAGの0.363秒に対しCRAGは0.512秒で、約41%増加している。さらに、この測定には検索とデータ処理が含まれていない。Web検索が発生する実運用時のend-to-end遅延を考えると、「軽量」という主張は検索評価器のモデルサイズに限って解釈すべきである。

**統計的な検証が不足している。**
信頼区間、複数seed、統計的有意差は報告されていない。また、PubHealthやARCでは生成器が指定されたTrue／Falseや選択肢記号を正しく出力できないことがあり、検索品質以外のinstruction-following性能がAccuracyへ混入している。

---

## 7. まとめ

CRAGは、RAGの検索結果を無条件に生成モデルへ渡すのではなく、

1. 検索結果を評価し、
2. 信頼できれば不要部分を除去し、
3. 信頼できなければWeb検索へ切り替え、
4. 判断が曖昧なら両者を併用する、

という自己修正機構を導入した手法である。

特に、検索器の精度が低い、コーパスが限定的、取得文書にノイズが多い、といった条件で有効性が高い。一方で、評価器の学習、閾値調整、外部検索サービス、Web情報の信頼性管理が新たなシステム依存要素となる。

---

## 8. 総評

本論文の最も大きな価値は、RAGを「検索して生成する固定パイプライン」から、**検索品質を監視しながら処理経路を変更する適応的システム**へ拡張した点にある。手法は比較的単純だが、問題設定との対応が明確で、後続のCorrective／Adaptive RAG設計へつながる実用的な着想である。

一方、データセット別閾値、PopQAで学習した評価器への依存、商用Web検索、end-to-endコストの未評価、統計的検証の不足から、実験結果だけで一般的なRAG環境への普遍的有効性を断定することは難しい。

### 査読結果

**判定：弱採択（Weak Accept、6/10）**
**確信度：3/5**

採択寄りとする理由は、問題の重要性、手法の明快さ、標準RAGとSelf-RAGの双方へ適用した実験、検索劣化実験とアブレーションがそろっているためである。ただし、厳密な採択には、未知ドメインでの閾値固定評価、商用APIを使わない再現実験、検索・Web処理を含むend-to-endコスト、統計的有意差、誤情報・矛盾文書に対する検証が必要である。

なお、公式OpenReview上のICLR 2025投稿ステータスは **Withdrawn Submission** であり、これは採択または査読による不採択を意味する最終判定ではない。([openreview.net][2])
