# P074：RepoCoder: Repository-Level Code Completion Through Iterative Retrieval and Generation

**書誌情報**：Fengji Zhangほか、EMNLP 2023 Main Conference、Regular Long Paper、pp. 2471–2484。([ACL Anthology][1])

## 1. はじめに

本論文は、編集中ファイルだけでなく、リポジトリ内の別ファイルに散在するAPI定義・使用例・命名規則などを利用してコードを補完する「リポジトリレベル・コード補完」を扱う。

提案手法 **RepoCoder** の核心は、最初に生成した暫定的なコードを、次の検索クエリに組み込むことである。つまり、

> 検索 → 生成 → 生成結果を使って再検索 → 再生成

という反復処理を行う。これにより、未完成コードだけでは検索できなかった、実際の補完対象に近いコード断片を取得する。

また、行単位、リポジトリ内API呼び出し、関数本体という3種類の粒度を評価するベンチマーク **RepoEval** も提案している。

## 2. 背景

通常のIn-Fileコード補完は、現在編集中のファイル内の前方コンテキストだけを入力とする。しかし実際のリポジトリでは、次のような情報が別ファイルに存在する。

* 独自APIの定義と使用例
* 共通ユーティリティや設定
* リポジトリ固有の変数名・命名規則
* 同一ディレクトリ内の類似実装

静的解析を利用する手法は正確な依存関係を取得できる一方、対象言語や補完形態への依存が強い。リポジトリごとにモデルを再学習する手法も、データ作成と学習コストが問題となる。

単純なRAGでは、未完成コード (X) を検索クエリにする。しかし、検索したいのは本来「これから書かれるコード」であり、そのAPI名や識別子は (X) にまだ現れていない。この**検索クエリと補完対象のずれ**が主要な問題である。

論文中の例では、1回目の生成が誤った引数を含むCOLMAP API呼び出しを予測する。しかし、その予測には `COLMAP` などの重要な識別子が含まれるため、それを使った2回目の検索では正しい使用例を取得でき、最終的な補完が修正される。

## 3. 提案手法

### 3.1 リポジトリの検索インデックス

リポジトリ内のソースコードを、(S_w) 行のスライディングウィンドウに分割する。ウィンドウは (S_s) 行ずつ移動し、得られたコード断片を検索対象集合 (C_{\mathrm{repo}}) とする。

主実験ではコードをトークン集合に変換し、Jaccard係数によってクエリとの類似度を計算する。

### 3.2 1回目の検索と生成

最初の検索では、未完成コード (X) の末尾 (S_w) 行をクエリとする。

[
q^{(1)}=\mathrm{suffix}_{S_w}(X)
]

[
C_{\mathrm{ret}}^{(1)}
=R(C_{\mathrm{repo}},q^{(1)})
]

検索コード断片と未完成コードをプロンプトに含め、言語モデル (M) により暫定補完を生成する。

[
\hat{Y}^{(1)}
=M(C_{\mathrm{ret}}^{(1)},X)
]

この1回目だけを実施したものが、通常のリポジトリレベルRAGに相当する。

### 3.3 反復検索・生成

2回目以降は、未完成コードの末尾と、直前の予測コードの先頭を連結して新しい検索クエリを作る。

[
q^{(i)}
=======

\mathrm{suffix}*{S_w-S_s}(X)
\oplus
\mathrm{prefix}*{S_s}(\hat{Y}^{(i-1)})
]

[
C_{\mathrm{ret}}^{(i)}
=R(C_{\mathrm{repo}},q^{(i)}),\qquad
\hat{Y}^{(i)}
=M(C_{\mathrm{ret}}^{(i)},X)
]

生成結果が完全に正しくなくても、API名や変数名などが検索ヒントとして機能する可能性がある。検索器と生成モデルのパラメータは更新されないため、RepoCoderは追加学習を必要としない。

検索されたコードには元ファイルのパスを付与し、未完成コードとともにプロンプトへ配置する。プロンプト長の範囲内で、最大 (K) 個のコード断片を利用する。

## 4. 実験方法

### 4.1 RepoEval

RepoEvalは、2022年以降に作成された14個のPythonリポジトリから構築されている。選定条件は、オープンソースライセンス、非fork、100スター以上、Pythonファイル比率80%以上、明示的な単体テストの存在などである。

| タスク             | リポジトリ数 | サンプル数 | 評価指標                        |
| --------------- | -----: | ----: | --------------------------- |
| 行補完             |      8 | 1,600 | Exact Match、Edit Similarity |
| リポジトリ内API呼び出し補完 |      8 | 1,600 | Exact Match、Edit Similarity |
| 関数本体補完          |      6 |   373 | 単体テストのPass Rate             |

行補完ではコメントを除き、5トークン以上の非重複行を選択する。API補完ではリポジトリ内で定義されたAPI呼び出しを対象とする。関数補完では、単体テストで実行される3～30行の関数本体を使用する。

### 4.2 比較手法とモデル

比較対象は以下である。

* **In-File**：編集中ファイルの前方コンテキストのみ
* **RepoCoder Iter-1**：検索と生成を1回行う通常のRAG
* **RepoCoder Iter-2～4**：反復検索・生成
* **Oracle**：正解コードの先頭 (S_s) 行を検索クエリに使用する参考上限

生成モデルにはGPT-3.5-Turboと、CodeGen-Monoの6B、2B、350Mを使用する。主検索器はJaccard係数による疎検索で、付録ではUniXcoderによる密検索も評価している。

行・API補完では (S_w=20)、(S_s=10)、最大出力100トークン、関数補完では (S_w=50)、(S_s=10)、最大出力500トークンとし、取得断片数は最大10個である。

## 5. 結果

以下はExact Match、関数補完のみPass Rateであり、単位はパーセントポイントである。

| タスク・モデル          | In-File |  RepoCoder最高値 |    改善量 | Oracle |
| ---------------- | ------: | ------------: | -----: | -----: |
| 行・GPT-3.5        |   40.56 | 57.00（Iter-3） | +16.44 |  57.75 |
| 行・CodeGen 6B     |   34.56 | 47.75（Iter-3） | +13.19 |  48.81 |
| 行・CodeGen 2B     |   33.63 | 47.13（Iter-4） | +13.50 |  47.31 |
| 行・CodeGen 350M   |   29.56 | 43.94（Iter-3） | +14.38 |  45.19 |
| API・GPT-3.5      |   34.06 | 49.56（Iter-4） | +15.50 |  50.13 |
| API・CodeGen 6B   |   26.19 | 39.31（Iter-4） | +13.12 |  40.25 |
| API・CodeGen 2B   |   25.44 | 38.44（Iter-3） | +13.00 |  39.44 |
| API・CodeGen 350M |   22.19 | 33.88（Iter-2） | +11.69 |  34.88 |
| 関数・GPT-3.5       |   23.32 | 42.63（Iter-2） | +19.31 |  42.63 |

すべてのモデルについて、行・API補完のExact MatchがIn-Fileより11.69～16.44ポイント向上した。また、2回以上の反復は1回のみのRAGをほぼ一貫して上回り、Oracleに近い性能に到達した。

特に関数本体補完では、Pass Rateが23.32%から42.63%へ上昇し、Iter-2が平均値でOracleと同等になった。一方、3回目以降は必ずしも改善せず、最適反復数が固定ではないことも示されている。

API使用例を取得できた割合も、GPT-3.5ではIter-1の86.04%からIter-2の90.34%に上昇した。CodeGen 6Bでも76.27%から82.92%に改善しており、生成結果が次の検索を実際に改善していることを支持する。

UniXcoderによる密検索でも同様の傾向が得られ、単純なJaccard検索と概ね同等の性能であった。

## 6. 考察

RepoCoderの重要な点は、生成コードを最終回答としてだけでなく、**検索クエリを補強する中間表現**として利用したことである。これは質問応答におけるクエリ拡張や疑似関連フィードバックに近い発想であり、コード中のAPI名・識別子が強い検索キーになるという特性を活用している。

有効な検索断片は、対象ファイルと同じディレクトリ、共通のimportを持つファイル、類似したファイル名を持つファイルに多く存在した。ただし、これらの場所だけに検索範囲を限定すると性能は低下しており、リポジトリ全体を単純に検索できる構成にも利点がある。

一方で、以下の問題が残る。

**誤りの自己増幅**：最初の生成が誤ったAPI名や引数を含む場合、次の検索も誤った方向へ誘導される。同じAPIでもファイルごとに引数形式が異なり、取得例がかえって誤解を招くケースが報告されている。

**反復回数の不安定性**：各反復では新たに正解するケースがある一方、それまで正解だったケースが不正解になる。したがって「反復を増やせば常に改善する」とはいえず、自動停止条件が必要である。

**コード重複への依存**：類似実装が多いリポジトリほど検索が有効になりやすく、重複が少ないリポジトリでは改善が限定される。

**実用時の遅延**：反復ごとに検索とモデル推論が追加されるため、リアルタイム補完ではレイテンシとAPIコストが増える。本論文では精度を中心に評価しており、実測レイテンシや費用対効果は示されていない。

**評価範囲の限定**：対象はPythonのみで、関数補完は373件に限られる。また、行・API補完のEMは機能的に正しい別解を不正解と判定する。著者自身も、実際には正しいコードがEMでは誤りと評価されるケースを確認している。

比較対象も主にIn-File、通常RAG、Oracleであり、学習型・静的解析型の強力なリポジトリ補完手法との体系的比較や統計的有意差検定は不足している。著者も、ベースライン、プロンプト設計、生成モデル、検索モデルの探索が限定的であることを認めている。

## 7. まとめ

RepoCoderは、最初のコード生成を次の検索クエリに利用することで、未完成コードと補完対象との検索上のずれを縮小する手法である。

追加学習や静的解析を必要とせず、疎検索・密検索、複数規模の生成モデルで一貫した改善を示した。特に、小規模なCodeGenモデルでもリポジトリ検索を組み合わせることで、より大規模なモデルのIn-File補完に近い性能を得られる点は実用的に重要である。

一方で、検索結果による誤誘導、最適反復数、レイテンシ、コード重複への依存が主要な課題として残る。

## 8. 総評

| 評価項目   | 評価                            |
| ------ | ----------------------------- |
| 新規性    | 高い。生成結果を検索クエリへフィードバックする設計が明快  |
| 技術的妥当性 | 概ね高い。検索品質向上の分析も手法の仮説を支持       |
| 実験の充実度 | 中～高。複数モデル・3粒度・実行ベース評価を含む      |
| 再現性    | 中。コードとデータは公開されているが、商用モデル依存がある |
| 実用性    | 高い可能性があるが、遅延・費用評価が不足          |
| 主な弱点   | 比較手法、言語数、統計検定、リアルタイム性能の不足     |

**本査読判定：弱い採択（Weak Accept、7/10）**

中心的なアイデアは単純だが有効であり、モデル規模と補完粒度をまたいで一貫した大幅な改善が確認されている。RepoEvalの構築と、単体テストによる関数補完評価も研究上の貢献が大きい。一方、強力な既存手法との比較、統計的検証、レイテンシ評価、Python以外への一般化が不足しているため、無条件の高評価ではなくWeak Acceptが妥当である。

**実際の査読結果はEMNLP 2023 Main Conferenceへの採択**である。([ACL Anthology][1])
