# Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection

## 1. はじめに

本論文は、Akari AsaiらによるSelf-RAGの提案論文である。従来のRetrieval-Augmented Generation（RAG）が常に固定数の文書を検索するのに対し、Self-RAGは、言語モデル自身が検索の要否を判断し、取得文書の関連性、生成文の根拠性、回答全体の有用性を評価しながら生成する。

核心は、RAGを「検索してから生成する固定パイプライン」から、「必要時に検索し、証拠と生成内容を逐次評価する制御可能な生成過程」へ変更した点にある。論文は2023年10月にarXivへ投稿され、その後ICLR 2024に採択された。([arXiv][1])

---

## 2. 背景

大規模言語モデルは、モデル内部のパラメータに保存された知識だけで回答すると、事実誤認や古い情報を生成することがある。RAGは外部文書を取得してプロンプトに追加することでこの問題を緩和するが、一般的なRAGには次の問題がある。

第一に、創作文や個人的経験の記述など、検索を必要としない質問にも文書を取得してしまう。第二に、検索上位の文書が質問に関連しているとは限らず、無関係な文書が生成を悪化させる可能性がある。第三に、取得文書が正しくても、生成された主張がその文書によって支持されている保証はない。

Self-RAGは、検索の要否、文書の関連性、生成文の根拠性をモデルに明示的に判断させることで、これらを同時に解決しようとする。図1では、通常のRAGが固定的に文書を渡すのに対し、Self-RAGが必要時だけ検索し、複数候補を評価して最良の生成を選択する流れが示されている。([ar5iv][2])

## 3. 提案手法

### 3.1 Reflection Token

Self-RAGでは、通常の語彙に以下の特殊トークンを追加する。

| トークン       | 出力値                            | 役割                        |
| ---------- | ------------------------------ | ------------------------- |
| `Retrieve` | Yes / No / Continue            | 新たな検索が必要か、既存証拠を引き続き使うかを判断 |
| `IsRel`    | Relevant / Irrelevant          | 取得文書が質問や生成内容に関連するかを評価     |
| `IsSup`    | Fully / Partially / No support | 生成文が取得文書によってどの程度支持されるかを評価 |
| `IsUse`    | 1～5                            | 回答全体の有用性を評価               |

実験では原則として一文を一つの「セグメント」とし、各セグメントの前後にこれらのトークンを生成する。つまり、自然言語による長い自己批評を生成するのではなく、構造化された少数の評価ラベルを通常の次トークン予測として出力する。([ar5iv][2])

### 3.2 学習方法

学習は二段階で行われる。

最初にGPT-4へ質問、取得文書、生成文を提示し、各Reflection Tokenの正解ラベルを作らせる。そのラベルを教師としてLlama 2-7BベースのCriticモデルを学習する。CriticのGPT-4ラベル再現精度は、`Retrieve`が93.8%、`IsSup`が93.5%、`IsRel`が80.2%、`IsUse`が73.5%である。

次にCriticとRetrieverを用いて、約14.6万件の指示・回答データへ、取得文書とReflection Tokenをオフラインで挿入する。そのデータを用いて、最終的なGeneratorであるLlama 2-7Bまたは13Bを通常の次トークン予測で学習する。取得文書そのものは損失計算からマスクされ、Generatorは回答文とReflection Tokenを予測する。

したがって推論時には独立したCriticを動かす必要がない。一方で、外部文書を取得するRetrieverは引き続き必要である。([ar5iv][2])

### 3.3 推論方法

各セグメントの生成時に、まず`Retrieve=Yes`の確率を計算する。確率が設定された閾値を超えた場合のみ検索を実行する。

検索した場合は、複数文書について並列に生成候補を作り、それぞれに対して`IsRel`、`IsSup`、`IsUse`を予測する。最終候補は、通常の生成確率と各Reflection Tokenの確率を重み付きで組み合わせたセグメント単位のビーム探索によって選ばれる。

この重みは推論時に変更できる。例えば`IsSup`の重みを大きくすれば証拠に強く裏付けられた回答を優先できるが、回答が短くなり、流暢性や情報量が低下する可能性がある。再学習なしで「根拠性重視」「有用性重視」などの動作を調整できる点が重要である。([ar5iv][2])

## 4. 実験方法

評価は推論時にfew-shot例を与えないzero-shot設定で、次の6タスクを用いている。

| 分類       | データセット                  | 主な評価指標                            |
| -------- | ----------------------- | --------------------------------- |
| 短文QA     | PopQA、TriviaQA          | 正解文字列を含む割合                        |
| 閉集合問題    | PubHealth、ARC-Challenge | Accuracy                          |
| 長文生成     | Biography Generation    | FactScore                         |
| 引用付き長文QA | ALCE-ASQA               | EM、ROUGE、MAUVE、引用Precision・Recall |

GeneratorはLlama 2-7Bと13B、CriticはLlama 2-7Bである。Retrieverには主としてContriever-MS MARCOを使い、通常は上位5件を利用する。BiographyとOpen-domain QAでは、Web検索由来の文書も追加している。学習にはOpen-Instruct、Natural Questions、FEVER、ASQAなどから約14.6万件を使用した。

比較対象にはLlama 2、Alpaca、Llama 2-chat、ChatGPT、それらへ単純に検索文書を追加したRAG、同一データでReflection Tokenなしに学習したLlama2-FT、SAIL、Toolformer、CoVEなどが含まれる。学習は4基のA100 80GBで3エポック実施されている。([ar5iv][2])

## 5. 結果

主要結果は以下の通りである。

| モデル          |    PopQA | TriviaQA | PubHealth |    ARC-C | Bio FactScore |
| ------------ | -------: | -------: | --------: | -------: | ------------: |
| Self-RAG 7B  |     54.9 |     66.4 |      72.4 |     67.3 |      **81.2** |
| Self-RAG 13B | **55.8** | **69.3** |  **74.5** | **73.1** |          80.2 |

ASQAではSelf-RAG 13Bが、EM 31.7、ROUGE 37.0、MAUVE 71.6、引用Precision 70.3、引用Recall 71.3を記録した。7Bは引用Precision 66.9、Recall 67.8だが、MAUVEは74.3と13Bを上回った。

非プロプライエタリモデルとの比較では、Self-RAGは総合的に最上位である。ただし、すべての個別指標で最高というわけではない。例えばASQAのEMでは、検索付きAlpaca 13Bの34.8がSelf-RAG 13Bの31.7を上回る。一方、通常の検索付きLlama 2やAlpacaの引用Precision・Recallは概ね2～7程度であり、Self-RAGの約67～71は大幅な改善である。

ChatGPTに対しても、全面的に勝っているわけではない。Self-RAG 13BはChatGPTよりPopQA、PubHealth、Biographyなどで高いが、TriviaQAでは69.3対74.3、ARCでは73.1対75.3で下回る。検索付きChatGPTとの比較でも、PopQA、TriviaQA、PubHealth、引用Precisionでは上回る一方、ARCやASQAのEM、ROUGE、MAUVE、引用Recallでは検索付きChatGPTが高い。したがって、論文の結果は「全指標でChatGPTを超えた」ではなく、「小規模な公開モデルとして、複数の知識タスクと引用精度で競争力を示した」と解釈するのが適切である。

アブレーションでは、推論時に検索を完全に無効化するとPopQAが45.5から24.7へ低下した。また、Criticなしの学習ではASQAが32.1から18.1へ低下し、常に上位1文書を利用する通常型RAGでもASQAは28.6に低下した。`IsSup`を候補選択から除く場合も性能が落ちるため、適応的検索だけでなく、根拠性評価が重要であることが示されている。

人手評価では、回答が妥当かつ証拠で支持される割合がPopQAで92.5%、Biographyで70.0%、`IsRel`判断の一致率が95%・90%、`IsSup`判断の一致率が90%・85%だった。ただし、これは50サンプル規模の小規模評価である。([ar5iv][2])

## 6. 考察

**最大の貢献は、検索・生成・検証を一つの言語モデルのトークン生成として統合した点である。** 外部のNLIモデルやReward Modelを推論時に逐次呼び出すのではなく、Generator自身が評価トークンを出すため、実装上の統一性が高い。また、評価トークンの重みを変えるだけで、根拠性と流暢性のトレードオフを操作できる。

一方、論文中の「自己反省」は、人間のように独立して事実を再検証する仕組みではない。実態はGPT-4の判断を蒸留した特殊トークン分類である。Criticの精度も真の正解に対する精度ではなく、基本的にはGPT-4ラベルとの一致率である。GPT-4ラベルに対する著者の確認も各分類20例であり、教師モデルの誤りや偏りを十分に検証したとは言いにくい。([ar5iv][2])

**再現性と計算量にも課題がある。** 学習ラベルにGPT-4、検索にGoogle Programmable Searchを用い、タスクごとに異なるWikipediaダンプやRetrieverを使っている。さらに、セグメントごとに複数文書を処理し、ビーム探索を行うため、通常の一回検索型RAGより推論コストが高くなると考えられるが、レイテンシ、スループット、総検索回数などの比較は十分に報告されていない。([ar5iv][2])

**zero-shotという表現にも注意が必要である。** 本論文のzero-shotは、推論時に例示を与えないという意味である。一方、Generatorの学習データにはASQAが3,897件含まれているため、少なくともASQAは未知タスクへの完全なゼロショット汎化ではない。また、ASQAではベースラインとSelf-RAGに異なる指示文を使用しており、アーキテクチャ上の必要性はあるものの、プロンプト差の影響は残る。([ar5iv][2])

Self-RAG自身の判断も誤る。論文中には、質問対象とは別の類似タイトルの書籍を取得し、誤った著者名を「Fully Supported」と判定した例がある。これは、Retrieverが誤った文書を取得し、その文書と生成文が表面的に整合している場合、自己評価も同時に誤る可能性を示している。([ar5iv][2])

---

## 7. まとめ

Self-RAGは、必要時だけ検索を実行し、取得文書の関連性、生成文の根拠性、回答の有用性をReflection Tokenとして生成するRAGフレームワークである。

特に、固定的な上位文書追加よりも高いQA性能と引用精度を示し、推論時の重み変更によって生成方針を制御できる点が強い。後続の適応的RAGや自己評価型RAGに大きな影響を与えた、実用面と研究面の双方で重要な研究と評価できる。

一方で、GPT-4由来の教師信号、検索品質への依存、推論コスト、限定的な人手評価、同一タスク分布を含む学習データなどを考慮すると、「自己反省によって事実性が保証される」とまでは言えない。

## 8. 総評

### 公式の査読結果

**ICLR 2024 Oralとして採択**されている。OpenReview上でも最終ステータスが「ICLR 2024 oral」と明記されている。([OpenReview][3])

### 本稿に対する査読判定

**判定：Strong Accept相当（8/10、確信度4/5）**

提案の新規性は高く、適応的検索、取得文書評価、生成文評価、推論時制御を一つの枠組みにまとめた設計は明快である。複数種類のタスク、強いベースライン、アブレーションによって主要構成要素の有効性も示されている。

減点要因は、GPT-4教師への依存、Critic評価の循環性、推論コストの定量比較不足、ASQAを含む学習・評価設計、人手評価の小ささである。採択判断は妥当だが、より強い結論を主張するには、未知タスク・未知ドメインでの評価、RetrieverとGeneratorを分離したコスト分析、より大規模な第三者人手評価が必要である。
