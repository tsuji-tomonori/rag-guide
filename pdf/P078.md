# P078：ArchRAG: Attributed Community-based Hierarchical Retrieval-Augmented Generation

## 1. はじめに

ArchRAGは、知識グラフを用いるGraph-based RAGにおいて、**回答精度とオンライン推論コストを同時に改善する**ことを目的とした手法である。著者はShu Wang、Yixiang Fang、Yingli Zhou、Xilin Liu、Yuchi Maで、AAAI-26のProceedingsに正式掲載されている。以下ではAAAI掲載版を中心に、2026年5月11日更新のarXiv拡張版に含まれる追加実験も参照する。([arXiv][1])

中心的なアイデアは、次の三つである。

1. グラフ構造だけでなく、ノードの意味的属性も考慮した**Attributed Community**
2. エンティティから高次の概念までを多層化する**LLM-based hierarchical clustering**
3. 各階層を高速に検索する**C-HNSW**と、取得情報を整理する**adaptive filtering**

これにより、個別事実を尋ねる具体的質問と、文書集合全体の傾向を尋ねる抽象的質問を、同一の検索基盤で扱う。

---

## 2. 背景

代表的なGraphRAGは、文書から知識グラフを構築し、Leiden法によってコミュニティを検出した後、各コミュニティをLLMで要約する。質問に対しては、局所的なエンティティを検索するLocal Searchと、コミュニティ要約を広く調べるGlobal Searchを使い分ける。

著者らは、既存のGraphRAGに三つの問題があると指摘する。

**第一はコミュニティ品質である。** Leiden法は主にグラフの接続構造を利用するため、意味的には異なるエンティティが同一コミュニティに含まれることがある。結果としてコミュニティ要約のテーマが拡散し、質問に関連する情報を見つけにくくなる。

**第二は検索粒度の固定である。** Local Searchは具体的情報、Global Searchは抽象的情報を対象とするが、それぞれがほぼ単一の粒度を検索する。そのため、詳細な事実と上位概念の両方を必要とする質問への対応が難しい。

**第三はオンラインコストである。** Global Searchでは大量のコミュニティ要約をLLMに評価させる。論文の例では、Multihop-RAGから2,984コミュニティが生成され、100問への回答だけで約1億600万トークン、約650ドルを要したと報告されている。([AAAI Publications][2])

ArchRAGは、これらを「意味と構造を併用したコミュニティ」「多粒度の階層索引」「必要な情報だけをLLMに分析させる検索生成方式」によって解決しようとする。

## 3. 提案手法

ArchRAGは、**オフライン索引構築**と**オンライン検索・生成**の二段階から構成される。全体の処理は次のように表せる。

> 文書コーパス
> → 知識グラフ
> → Attributed Communityの階層
> → C-HNSW索引
> → 多層検索
> → 適応的フィルタリング
> → 最終回答

### 3.1 知識グラフの構築

入力文書をチャンクに分割し、LLMによって各チャンクからエンティティと関係を抽出する。得られた部分グラフを統合し、重複するエンティティや関係の記述をまとめる。

ここで各ノードとエッジには、単なるIDや接続関係だけでなく、エンティティの説明などの**テキスト属性**が付与される。([AAAI Publications][2])

### 3.2 Attributed Communityの階層化

通常のコミュニティ検出では構造的な接続だけを用いるのに対し、ArchRAGでは次の処理を反復する。

1. 属性埋め込みが近いノード間にKNNなどでエッジを追加する。
2. 意味的類似度に基づいてエッジを重み付けする。
3. Weighted Leidenなどでコミュニティを検出する。
4. 各コミュニティについてLLMで要約を生成する。
5. 一つのコミュニティを一つの上位ノードとみなし、次の階層のグラフを構築する。

下位層には個別エンティティや細かな事実が残り、上位層ほど複数のコミュニティをまとめた抽象的テーマとなる。これにより、同一の知識グラフを異なる意味粒度で表現できる。([AAAI Publications][2])

### 3.3 C-HNSW索引

各階層を個別のベクトルデータベースとして検索すると、時間・メモリの両面で非効率になる。そこで著者らは、Attributed Communityとエンティティを一つの階層型近似近傍索引に統合した**Community-based HNSW、C-HNSW**を提案する。

C-HNSWには二種類のリンクがある。

* **層内リンク**：同じ階層内で意味的に近いノードを接続
* **層間リンク**：隣接する階層の関連ノードを接続

通常のHNSWでは同一ベクトルが複数階層に現れるが、C-HNSWでは各エンティティまたはコミュニティは一つの階層だけに存在する。検索時には上位層から開始し、そこで得た近傍を下位層の探索開始点として再利用するため、多層を個別に検索する場合の重複計算を削減できる。([AAAI Publications][2])

### 3.4 Adaptive filtering-based generation

各階層から取得した情報をそのまま一つの長いプロンプトに入れるのではなく、LLMに階層ごとの**分析レポート**を作成させる。各レポートには質問との関連度を付け、関連度順に統合・整列し、コンテキスト長を超える部分を除外してから最終回答を生成する。

この部分の設計思想は、LLMを大量の候補から情報を探し出す「retriever」としてではなく、限定された候補を整理する「analyzer」として用いることである。これにより、無関係情報や“lost in the middle”の影響を抑制する。([AAAI Publications][2])

## 4. 実験方法

### 4.1 データセット

| データセット       | 文書・パッセージ数 |       トークン数 |    質問数 | 主な評価指標                   |
| ------------ | --------: | ----------: | -----: | ------------------------ |
| Multihop-RAG |       609 |   1,426,396 |  2,556 | Accuracy、Recall          |
| HotpotQA     |     9,221 |   1,284,956 |  1,000 | Accuracy、Recall          |
| NarrativeQA  |     1,572 | 121,152,448 | 43,304 | BLEU-1、METEOR、ROUGE-L F1 |

Multihop-RAGとHotpotQAは複数の情報を組み合わせる具体的QA、NarrativeQAは長文に対する生成型QAとして使用された。また、抽象的QAではMultihop-RAGのコーパスを基に、LLMで文書集合全体の理解を必要とする質問を生成している。([AAAI Publications][2])

### 4.2 比較手法

比較対象には以下が含まれる。

* 検索なし：Zero-shot、Chain-of-Thought
* 文書検索型：BM25、Vanilla RAG
* 階層・グラフ型：RAPTOR、HippoRAG、GraphRAG、LightRAG

GraphRAGについてはLocal SearchとGlobal Search、LightRAGについてはlocal、global、hybridの各構成を評価している。([AAAI Publications][2])

### 4.3 評価設定

具体的QAのAccuracyは完全一致ではなく、**生成文に正解文字列が含まれているか**を基準とする。抽象的QAは、Comprehensiveness、Diversity、Empowerment、Overallについて、LLM evaluatorによる二回答間の比較を行う。

基本設定は、生成モデルがLlama 3.1-8B、埋め込みモデルがnomic-embed-text、グラフ拡張がKNN、クラスタリングがWeighted Leiden、各層の検索件数が (k=5) である。([AAAI Publications][2])

実験環境はIntel Xeon 2.0 GHz、メモリ1,024 GB、24 GBのRTX A5000を8基という比較的大規模な構成で、C-HNSWはC++とカスタムFAISSで実装されている。([arXiv][3])

## 5. 結果

### 5.1 具体的QA

各指標について、ArchRAGと比較手法中の最高値を比較すると次のようになる。

| データセット・指標              |  ArchRAG | 最高ベースライン |       改善幅 |
| ---------------------- | -------: | -------: | --------: |
| Multihop-RAG Accuracy  | **68.8** |     59.1 |  +9.7ポイント |
| Multihop-RAG Recall    | **37.2** |     34.1 |  +3.1ポイント |
| HotpotQA Accuracy      | **65.4** |     51.3 | +14.1ポイント |
| HotpotQA Recall        | **69.2** |     56.8 | +12.4ポイント |
| NarrativeQA BLEU-1     | **11.5** |      8.0 |  +3.5ポイント |
| NarrativeQA METEOR     | **15.6** |     12.5 |  +3.1ポイント |
| NarrativeQA ROUGE-L F1 | **17.6** |      9.1 |  +8.5ポイント |

特にHotpotQAでの改善が大きい。GraphRAG-GlobalはNarrativeQAの検索を3日以内に完了できず、RAPTORもHotpotQAでは索引構築を完了できなかったと報告されている。([AAAI Publications][2])

### 5.2 抽象的QA

Overallのhead-to-head勝率は、Vanilla RAGに対して96%、Hybrid LightRAGに対して78%、GraphRAGの二つのコミュニティ階層に対してそれぞれ69%、70%だった。

GraphRAGに対するComprehensivenessの勝率は82～84%と高い。一方で、DiversityとEmpowermentでは54～62%であり、抽象的QAのすべての観点で圧倒的というより、**特に情報の網羅性で優れている**と解釈するのが適切である。([AAAI Publications][2])

### 5.3 コストと検索速度

HotpotQAにおけるオンラインのトークン使用量は、GraphRAG-Globalの1,394MトークンからArchRAGの5.1Mトークンへ減少し、論文では約250倍の削減とされている。([AAAI Publications][2])

1,000万ノードを含む合成階層データでのC-HNSW実験では、階層ごとに独立したHNSWを検索する方式と比べて、最大5.4倍、平均3.5倍高速だった。ただしRecallはC-HNSWが0.5537、比較方式が0.6058であり、速度向上には一定の検索精度低下が伴っている。([arXiv][3])

### 5.4 アブレーション

HotpotQAのAccuracyは完全版の65.4に対し、

* 単一階層化：60.1
* エンティティのみ：59.9
* 通常のLeiden：61.7
* 適応的フィルタリングを使わないDirect Prompt：40.7

となった。特にAdaptive filteringを除いた場合の低下が大きく、検索結果を単純にプロンプトへ投入するだけでは、階層検索の利点を十分に活用できないことが分かる。

また、通常のGraphRAGのコミュニティをAttributed Communityに置き換えただけでも、HotpotQA Accuracyは33.5から50.6へ上昇した。したがって、性能向上はC-HNSWだけでなく、コミュニティ自体の意味的一貫性にも由来する。([AAAI Publications][2])

## 6. 考察

### 優れている点

本研究の最大の貢献は、GraphRAGの「コミュニティを作る」「コミュニティを検索する」「取得情報から回答する」という三段階を個別に改善するのではなく、**意味的コミュニティ、階層索引、回答生成を一体として設計した点**にある。

特に、下位層のエンティティと上位層の要約を同時に検索できるため、「誰が何をしたか」のような具体的質問と、「文書集合ではどのような傾向が見られるか」のような抽象的質問を、別々の検索パイプラインに分けず処理できる。Attributed Community単独でもGraphRAGを改善し、各構成要素を除くと性能が低下するため、提案設計には一定の実験的裏付けがある。([AAAI Publications][2])

### 限界

第一に、削減されているのは主に**オンライン検索時のコスト**である。オフラインではLLMによる知識グラフ抽出と各階層のコミュニティ要約が必要であり、索引構築コストはGraphRAGと同程度で、HippoRAGより高い。更新頻度の高いコーパスで階層をどのように差分更新するかも十分には検証されていない。([arXiv][3])

第二に、性能は上流の知識グラフ品質に依存する。公開実装もMicrosoft GraphRAGによるKG抽出を前提としているため、エンティティの欠落、重複統合の失敗、誤った関係抽出は、その後のクラスタリングと検索に伝播する。さらにC-HNSWにはカスタムFAISSが必要であり、コードは公開されているものの導入負荷は小さくない。([GitHub][4])

第三に、評価方法には注意が必要である。具体的QAのAccuracyは正解文字列の包含判定なので、冗長な回答が有利になる可能性がある。抽象的QAもLLMが生成した質問をLLM evaluatorで評価しており、人手評価や客観的な正解集合に基づくものではない。したがって、抽象QAの勝率をそのまま人間の有用性評価とみなすべきではない。([AAAI Publications][2])

第四に、C-HNSWの大規模速度評価はランダムベクトルからなる合成データであり、実文書の埋め込み分布を完全には再現しない。また、実験値ではBase-HNSWよりRecallが約0.052低い。今後は実データ上のレイテンシ、検索Recall、回答精度の関係を統合的に示す必要がある。([arXiv][3])

---

## 7. まとめ

ArchRAGは、グラフの接続構造と意味属性を組み合わせたAttributed Communityを階層的に構築し、それをC-HNSWで多粒度検索するGraph-based RAGである。さらにAdaptive filteringによって、取得した情報をLLMが関連度付きで整理してから回答する。

実験では、Multihop-RAG、HotpotQA、NarrativeQAのすべてで高い性能を示し、特にHotpotQAのAccuracyでは最高ベースラインを14.1ポイント上回った。また、GraphRAG-Globalに対するオンラインのトークン消費を約250分の1に削減している。

したがって本論文は、GraphRAGの「高品質だが高コスト」という問題に対し、単なるプロンプト改善ではなく、**知識表現、索引構造、検索、生成を横断した解決策**を提示した研究と位置付けられる。

## 8. 総評

**査読結果は採択である。** 本論文は査読を経て、AAAI-26のProceedings、Vol. 40、No. 19、pp. 15868–15876に正式掲載された。公開情報から確認できるのは採択・掲載の事実であり、個別レビュアーの点数や査読コメントは確認できないため、それらについては推測しない。([arXiv][1])

研究内容としては、**新規性と実用上の効果がともに高く、採択は妥当**と評価できる。Attributed Community自体の効果、階層検索の効率、Adaptive filteringの寄与がアブレーションで分離して示されている点は強い。一方、オフライン索引コスト、LLM-as-a-judgeへの依存、合成データ中心の検索速度評価、重い実験環境などから、一般的な環境での再現性と運用性には課題が残る。

総合すると、ArchRAGはGraphRAG系研究における有力な発展形であり、特に**静的で大規模な文書集合に対して、具体質問と抽象質問を低いオンラインコストで処理したい用途**に適した手法である。
