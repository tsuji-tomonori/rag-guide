# 1. はじめに

**FrugalGPT**は、複数の大規模言語モデル（LLM）APIを使い分けることで、**回答品質を維持・向上させながらAPI利用料金を削減する**手法である。著者はLingjiao Chen、Matei Zaharia、James Zouで、査読済み最終版は2024年12月にTransactions on Machine Learning Research（TMLR）で公開された。([arXiv][1])

本論文の中心的な主張は、単一の高性能・高価格モデルをすべての質問に使う必要はないという点にある。簡単な質問は低価格モデルで処理し、低価格モデルでは十分な品質を得られない質問だけを高価格モデルへ回すことで、コストと性能の双方を改善できる。

---

# 2. 背景

LLM APIには、モデルごとに以下の違いがある。

* 入出力トークン単価
* タスクごとの正解率
* 得意・不得意な質問
* 応答速度

高価格なモデルが常にすべての質問で優れているとは限らない。あるタスクでは安価なモデルが高価なモデルを上回る場合もあり、同じデータセット内でも、質問ごとに適切なモデルが異なる。

論文では、LLM利用コストを削減する方法を大きく三つに整理している。

1. **Prompt adaptation**
   不要な例や説明を除いてプロンプトを短縮したり、複数の質問を一つのプロンプトにまとめたりする。

2. **LLM approximation**
   過去の回答をキャッシュする、または高価なLLMの回答を教師データとして安価なモデルを学習する。

3. **LLM cascade**
   安価なモデルから順番に問い合わせ、回答品質が不十分と判断された場合だけ次のモデルを呼び出す。

FrugalGPTは、このうち主に**LLM cascadeを具体化したシステム**である。論文では、1回の問い合わせコストが小さく見えても、大量利用では無視できないことを指摘している。例として、月36万件の問い合わせを当時のGPT-4料金で処理すると、条件によっては月額約2.12万ドルになると試算している。

# 3. 提案手法

## 3.1 FrugalGPTの基本構成

FrugalGPTは、次の三つの要素から構成される。

### LLM Router

利用可能なLLMの中から、呼び出す順番を決定する。

例えば、

[
\text{GPT-J} \rightarrow \text{J1-Large} \rightarrow \text{GPT-4}
]

のように、原則として安価なモデルから高性能モデルへ段階的に問い合わせる。

本論文の実装では、モデルの順序は個々の質問に応じて変更されるのではなく、学習データから決めた**固定の順序**を使用する。

### Generation Scorer

各LLMが生成した回答について、

[
P(\text{回答が正しい}\mid \text{質問, 回答})
]

を推定する。

実装では、質問と生成回答を入力とする**DistilBERTベースの分類器**を使用し、回答が正解か不正解かを教師ラベルとして学習する。

### Stop Judger

Generation Scorerのスコアが、モデルごとに設定された閾値を上回れば、その回答を最終回答として採用する。

下回った場合は回答を棄却し、カスケード内の次のLLMを呼び出す。

## 3.2 最適化問題

FrugalGPTは、平均APIコストが予算 (B) 以下になるという制約の下で、期待正解率を最大化する。

概念的には、次の最適化問題である。

[
\max_{\pi,,\boldsymbol{\tau}}
\mathbb{E}[\text{回答品質}]
]

subject to

[
\mathbb{E}[\text{APIコスト}] \le B
]

ここで、

* (\pi)：LLMを呼び出す順序
* (\boldsymbol{\tau})：各段階の回答受理閾値

である。

モデル順序と閾値を同時に探索する問題は、完全な回答評価器が存在したとしてもNP困難である。

---

## 3.3 近似的な探索方法

論文では探索量を削減するため、**Maximum Performance Improvement（MPI）**という指標を導入する。

MPIは、あるモデルが誤答した質問のうち、別のモデルなら正答できる割合を表す。現在のモデルを補完しやすいモデルを、カスケードの次段候補として優先する。

実装では、各段階についてMPI上位3モデルに候補を絞って順序を探索する。閾値については、グリッド上で性能とコストを評価し、二次関数で補間した後、二次計画法を用いて予算制約を満たす値を求める。

# 4. 実験方法

## 4.1 使用モデル

主要実験では、次の6事業者が提供する計14種類のLLM APIを使用している。

* OpenAI
* AI21
* Cohere
* TextSynth
* Databricks
* ForefrontAI

料金やAPI性能は、主として2023年3月時点の条件に基づく。カスケードで呼び出せるモデル数の上限は3に設定された。

査読済み最終版では追加実験として、GPT-4o、GPT-4o-mini、GPT-4 Turbo、Claude 3.5 Sonnet、Gemini 1.5、Llama 3、Jamba 1.5など、より新しいモデルも評価している。

## 4.2 データセット

| データセット     | 分野   | 問題形式              |   データ数 |
| ---------- | ---- | ----------------- | -----: |
| HEADLINES  | 金融   | 金価格の方向を予測する4値分類   | 10,000 |
| OVERRULING | 法律   | 判例が覆されたかを判定する2値分類 |  2,400 |
| COQA       | 複数分野 | 読解に基づく自由回答        |  7,982 |
| AGNEWS     | ニュース | 記事カテゴリの4値分類       |  7,600 |
| SCIQ       | 科学   | 科学知識に関する質問応答      | 11,680 |

各データセットを50%の学習用データと50%の評価用データに分割する。学習部分は、モデル順序、回答評価器、受理閾値の決定に使用する。

各モデルにはfew-shot形式でプロンプトを与え、主要実験ではtemperatureを0.1、最大生成トークン数を50に設定している。したがって、実験の中心は分類や短い質問応答であり、長文生成ではない。

## 4.3 比較方法

主に次の方式と比較している。

* 各LLMを単独で利用
* 安価なモデルからGPT-4へ切り替える単純な信頼度ベース方式
* ChatGPTを複数回呼び出し、多数決を取るアンサンブル
* 異なる予算制約下で学習したFrugalGPT

評価指標は、タスク性能、API料金、遅延などである。

# 5. 結果

## 5.1 最高性能の単一モデルと同等の性能を得るためのコスト

以下は、各データセットで最も高い精度を持つ単一LLMと、その精度に到達したFrugalGPTのAPI費用を比較した結果である。

| データセット     | 最良の単一モデル | 単一モデル費用 | FrugalGPT費用 |   削減率 |
| ---------- | -------- | ------: | ----------: | ----: |
| HEADLINES  | GPT-4    |   $33.1 |        $0.6 | 98.3% |
| OVERRULING | GPT-4    |    $9.7 |        $2.6 | 73.3% |
| COQA       | GPT-3    |   $72.5 |       $29.6 | 59.2% |
| AGNEWS     | GPT-4    |   $64.6 |       $15.9 | 75.4% |
| SCIQ       | GPT-3    |  $132.4 |       $63.1 | 52.3% |

すべてのデータセットにおいて、FrugalGPTは最良の単一モデルと同程度の精度を、より低い費用で達成した。最大削減率はHEADLINESの**98.3%**である。ただし、金額は論文実験時のAPI料金、プロンプト長、評価データ数に依存する。

## 5.2 HEADLINESの具体例

予算を6.5ドルに設定した場合、FrugalGPTは次のカスケードを学習した。

[
\text{GPT-J} \rightarrow \text{J1-Large} \rightarrow \text{GPT-4}
]

結果は以下の通りである。

| 方式        |   正解率 |    費用 |
| --------- | ----: | ----: |
| GPT-4単独   | 85.7% | $33.1 |
| FrugalGPT | 87.2% |  $6.5 |

FrugalGPTでは、最終段のGPT-4まで到達した質問は16.6%にすぎなかった。つまり、多くの質問を安価なモデルで処理しつつ、難しい質問だけをGPT-4へ送ることで、**約80%のコスト削減と1.5ポイントの正解率向上**を同時に達成した。

## 5.3 同一予算での性能向上

論文全体では、最良の単一モデルと同等の性能を維持しながら最大約98%のコスト削減、または同程度の予算で最大約4ポイントの精度向上が報告されている。

これは、モデル間の誤答が完全には重ならないためである。高価なモデルが誤答する質問に別の安価なモデルが正答する場合があり、適切な選別によって単一モデル以上の性能を実現できる。([OpenReview][2])

## 5.4 追加評価

新しいモデルを用いた追加実験でも、HEADLINESとSCIQでは、最良モデルと同程度の性能を保ちながら90%以上のコスト削減を達成した。

また、モデルの価格と性能は必ずしも単調な関係にならず、特定タスクではGPT-4o-miniが、より高価なGPT-4oを上回る例も確認された。これは、価格だけを基準にモデルを選択するのではなく、タスク別の実測性能を利用すべきことを示している。

# 6. 考察

## 6.1 優れている点

第一に、FrugalGPTはモデル内部のパラメータやトークン確率を必要とせず、**APIから得られる回答だけで動作する**。そのため、GPT-4やClaudeのようなブラックボックス型の商用モデルにも適用できる。

第二に、単純な「安いモデルを使う」という方式ではなく、質問ごとに回答を受理するか、次のモデルへ送るかを判断する。その結果、単なる費用削減にとどまらず、複数モデルの相補性を利用して単一モデル以上の正解率を得られる。

第三に、モデル価格、要求精度、利用可能予算に合わせて閾値を変更できるため、実運用におけるコスト・性能トレードオフを明示的に制御できる。

## 6.2 回答評価器の重要性

FrugalGPTの性能はGeneration Scorerの精度に強く依存する。

追加実験では、約1,100万パラメータのALBERTから約6,700万パラメータのDistilBERTへ評価器を変更すると、性能が約8ポイント向上した。一方、DistilBERTから約1億1,000万パラメータのBERTへ拡大した場合の改善は小さかった。

したがって、極端に小さい評価器では回答品質を十分に判定できないが、一定規模を超えれば、評価器をさらに大型化しても効果は限定的と考えられる。

## 6.3 制約と課題

### タスク固有の学習データが必要

回答評価器を学習し、モデルの順序と閾値を決めるには、正解ラベル付きデータと、候補LLMから事前に収集した回答が必要である。

したがって、データの少ない新規タスクに直ちに適用できるわけではない。論文では約500件の学習例でも有効性を確認しているが、事前評価にかかるAPI費用や運用工数も総コストに含めて検討する必要がある。

### 分布変化の影響

実運用の問い合わせ内容が学習データから変化すると、回答評価器の確率推定や、最適なモデル順序が不正確になる可能性がある。

論文の分布変化実験では比較的安定した結果を示したものの、価格改定、モデル更新、利用者層の変化があれば、モデル順序と閾値を定期的に再学習する必要がある。

### 固定されたモデル順序

本方式のルータは、質問内容ごとに最初のモデルを直接選ぶのではなく、基本的に固定されたカスケード順序を使う。

例えば、法律質問には法律が得意なモデル、コード質問にはコードが得意なモデルを最初から割り当てるような、意味内容に基づく動的ルーティングにはなっていない。この点は、現代的なMixture-of-Experts型ルータやタスク認識型ルータによって改善できる余地がある。

### 長文・主観的生成への適用

評価は分類、読解、短い質問応答が中心である。要約、創作、対話、コード生成など、複数の妥当な回答が存在するタスクでは、単純な正解・不正解ラベルで回答品質を学習することが難しい。

LLM-as-a-judgeを利用する方法も考えられるが、評価モデル自身のバイアスや追加料金が発生する。著者らも、長文・自由生成やリスクの高い用途における不確実性評価を今後の課題としている。

### 遅延

複数モデルを逐次呼び出すため、最後のモデルまで到達した質問では待ち時間が増える可能性がある。

一方、論文の測定例では、多くの質問が安価な初期モデルで終了するため、FrugalGPTの90%の問い合わせが0.9秒以内に完了し、GPT-4単独の90パーセンタイルである1.1秒超より短かった。したがって、平均的な遅延は改善できても、最悪時遅延は別途管理する必要がある。

---

# 7. まとめ

FrugalGPTは、複数のLLMを安価な順または相補性の高い順に並べ、各回答の推定品質に応じて処理を終了するか、より高性能なモデルへエスカレーションする手法である。

主要な成果は次の通りである。

* 最良の単一LLMと同等の性能を維持しながら、最大約98%のAPI費用を削減した。
* 同程度の予算では、単一モデルより最大約4ポイント高い精度を示した。
* 高性能モデルを全件に使わず、難しい質問だけに限定することで費用を抑えた。
* モデル間の相補的な誤答パターンを利用し、単一の高価格モデル以上の精度を実現した。
* ブラックボックス型APIにも適用可能であり、既存モデルを再学習する必要がない。([OpenReview][2])

# 8. 総評

## 査読結果

**TMLR採択。2024年12月20日公開。Featured Certificationを取得。** ([OpenReview][2])

本論文は、LLMの性能競争だけでなく、実運用で重要となる**推論費用の最適化**を、予算制約付き最適化問題として明確に定式化した点で高く評価できる。特に、「最も高性能なモデルを常に使う」という単純な運用方針が必ずしも費用対効果や精度の面で最適ではないことを、具体的な実験で示している。

FrugalGPTの強みは、既存の商用APIをそのまま組み合わせられる実用性と、コスト削減だけでなく精度向上も達成できる点にある。一方で、効果はタスク固有の回答評価器、学習データの代表性、API料金、候補モデルの構成に左右される。長文生成、主観評価、継続的に分布が変化するサービス、医療・法律などの高リスク用途については、追加検証が必要である。

総合的には、**LLMルーティングおよびモデルカスケード研究の基礎となる、実用性と研究的意義の双方が高い論文**である。特定の料金削減率を普遍的な数値として捉えるのではなく、「各問い合わせに必要十分なモデルだけを割り当てる」という設計原理が、本論文の最も重要な貢献である。
