# P096：M3-Embedding

## 1. はじめに

本論文は、単一モデルで次の「3つのM」を実現する検索用テキスト埋め込みモデル **M3-Embedding** を提案する。

* **Multi-Linguality**：100以上の言語による多言語・言語横断検索
* **Multi-Functionality**：密検索、疎検索、マルチベクトル検索
* **Multi-Granularity**：短い文から最大8,192トークンの長文まで

従来は用途ごとに別モデルを用意することが多かったが、本研究は一つのXLM-RoBERTa系エンコーダから三種類の検索表現を生成し、単独でもハイブリッドでも利用できるようにする。論文は **Findings of ACL 2024** に掲載された。([ACL Anthology][1])

## 2. 背景

既存の検索用埋め込みモデルには、主に三つの問題がある。

第一に、多くの高性能モデルが英語中心であり、低資源言語や言語横断検索では性能が不安定である。第二に、密検索、語彙検索、ColBERT型マルチベクトル検索はそれぞれ異なる長所を持つものの、通常は別々のモデルとして学習される。第三に、長文を入力するとGPUメモリ消費が増え、対照学習に必要な大規模バッチと長い入力系列を両立しにくい。([arXiv][2])

各検索方式の特徴は次のように整理できる。

| 方式        | 長所               | 主な弱点                 |
| --------- | ---------------- | -------------------- |
| 密検索       | 言い換えや意味的類似性に強い   | 固有名詞や完全一致情報を落とす場合がある |
| 疎検索       | 単語一致、固有名詞、専門語に強い | 異なる言語間では単語の共起がほぼない   |
| マルチベクトル検索 | トークン単位の細粒度な照合が可能 | 計算量とインデックス容量が大きい     |

M3-Embeddingは、これらを競合する別方式としてではなく、相互補完する複数の検索ヘッドとして共同学習する。

## 3. 提案手法

### 三種類の検索表現

共通エンコーダの出力から、以下の三種類の関連度を計算する。

**密検索**では、クエリと文書の正規化済み`[CLS]`表現の内積を関連度とする。

**疎検索**では、各トークンの隠れ状態をスカラー重みに変換し、クエリと文書の双方に現れるトークンについて重みの積を加算する。単なる出現頻度ではなく、文脈を考慮して単語の重要度を学習する点がBM25との違いである。

**マルチベクトル検索**では、全トークン表現を射影し、ColBERTと同様に、各クエリトークンと最も類似する文書トークンとのMaxSimを平均する。最終的なハイブリッドスコアは、三方式のスコアの重み付き和となる。

### Self-Knowledge Distillation

各検索ヘッドを単純にInfoNCE損失で共同学習すると、目的関数が競合する。特に、意味類似性を重視する密表現と、語彙一致を重視する疎表現の間で負の転移が起こり得る。

そこで、三方式のスコアを重み付きで統合した

[
s_{\mathrm{inter}}
=w_1s_{\mathrm{dense}}
+w_2s_{\mathrm{sparse}}
+w_3s_{\mathrm{multi}}
]

を教師信号とする。統合スコアをsoftmaxで確率分布に変換し、それぞれの検索ヘッドがこの分布を再現するよう蒸留損失を加える。最終損失は通常の対照学習損失と蒸留損失の平均である。外部の巨大教師モデルを必要とせず、同じモデル内の異種ヘッドをアンサンブル教師として用いる点が特徴である。

### 多段階学習とデータ

基盤モデルには、最大位置長を8,192へ拡張したXLM-RoBERTa Largeを用い、RetroMAEで検索向けに事前学習する。その後、概ね次の順序で学習する。

1. 大規模な教師なしペアを用いた密検索の対照学習
2. 教師付きデータと合成長文データを用いた三方式の初期学習
3. Self-Knowledge Distillationによる統合学習

教師なしデータはWikipedia、S2ORC、xP3、mC4、CC-News、NLLB、CCMatrixなどから構築され、合計約12億ペア、194言語を含む。教師付きデータにはMS MARCO、NQ、MIRACL、Mr. TyDiなどを使用する。また、長文検索データが不足しているため、Wikipedia、Wudao、mC4の長文から段落を抽出し、GPT-3.5で質問を生成した **MultiLongDoc** を作成している。([arXiv][2])

### Efficient Batching

長文と大規模バッチを両立するため、以下を導入する。

* 系列長ごとのバケット化によるパディング削減
* 長文バッチを小さなサブバッチに分割するSplit-batch
* Gradient Checkpointing
* GPU間での埋め込み共有による大規模なin-batch negative
* GPU間の負荷を揃える固定サンプリング

8,192トークンの場合、Split-batchによって1デバイス当たりの最大バッチサイズが6から130へ増加しており、20倍以上の拡大となる。また、学習資源がない場合の代替として、256トークンごとに`[CLS]`を挿入し、それらを平均するMCLSも提案されている。

## 4. 実験方法

評価対象は以下の四つである。

| ベンチマーク      | 評価内容                    | 指標         |
| ----------- | ----------------------- | ---------- |
| MIRACL      | 18言語の同一言語内検索            | nDCG@10    |
| MKQA        | 25言語の質問から英語Wikipediaを検索 | Recall@100 |
| MLDR        | 13言語の長文検索               | nDCG@10    |
| NarrativeQA | 英語の長文検索                 | nDCG@10    |

比較対象はBM25、mDPR、mContriever、mE5、E5-Mistral、Jina Embeddings、OpenAI埋め込みモデルなどである。

密検索と疎検索はそれぞれ上位1,000件を検索する。計算コストの高いマルチベクトル方式は、密検索の上位200件を再ランキングする用途に限定される。ハイブリッド時のスコア重みはベンチマークごとに調整されている。([arXiv][2])

学習規模も大きく、RetroMAE段階に32基のA100、教師なしペア学習に96基のA800、最終ファインチューニングに24基のA800を使用している。ファインチューニングではクエリ当たり7件の負例を用い、ANCE型のハードネガティブも導入する。([arXiv][2])

## 5. 結果

主要結果は次の通りである。数値は論文発表時の比較設定に基づく。

| ベンチマーク      |            最良の比較モデル | M3 Dense |   M3 All |    改善 |
| ----------- | ------------------: | -------: | -------: | ----: |
| MIRACL      |            mE5：66.6 |     69.2 | **71.5** |  +4.9 |
| MKQA        |            mE5：70.9 |     75.1 | **75.5** |  +4.6 |
| MLDR        |           BM25：53.6 |     52.5 | **65.0** | +11.4 |
| NarrativeQA | OpenAI-3-Large：51.6 |     48.7 | **61.7** | +10.1 |

MIRACLとMKQAでは、密検索単独でも既存モデルを上回り、マルチベクトルやハイブリッドによってさらに改善する。一方、長文検索では密検索単独は必ずしも強くなく、MLDRでは疎検索が62.2、全方式の統合が65.0となる。長文では、文書全体を一つのベクトルに圧縮するよりも、重要語や局所的対応を保持する方式が有効である。

アブレーションではSelf-Knowledge Distillationの効果が明確である。

* Dense：68.7 → 69.2
* Sparse：36.7 → 53.9
* Multi-vector：69.3 → 70.5

特に疎検索で **17.2ポイント** 改善しており、自己蒸留が異種検索目的間の競合を緩和している。また、MIRACLのDense性能は、単純なファインチューニングの60.5から、RetroMAE追加で66.1、教師なしペア学習追加で69.2まで向上した。

## 6. 考察

本論文の最大の貢献は、新しいエンコーダ構造そのものよりも、**複数の検索方式、巨大な多言語データ、長文対応、効率的学習を一つの実用的モデルに統合したこと**にある。特に自己蒸留は、単なる三ヘッドのマルチタスク学習ではなく、ヘッド間の合意分布を形成する共同正則化として機能している。

結果からは、タスクごとに有効な検索方式が異なることも分かる。言語横断検索では単語の直接一致が少ないためSparseは45.3に留まり、DenseやMulti-vectorが中心となる。一方、長文検索ではSparseがDenseを約10ポイント上回る。したがって、「常にDenseが最適」ではなく、単一モデルから適切な表現を選択できること自体がM3の価値である。

ただし、以下の限界がある。

**言語評価の範囲**
100以上の言語をサポートすると主張する一方、主要評価はMIRACLの18言語とMKQAの25言語である。低資源言語、異なる文字体系、言語族ごとの性能差は十分に分析されていない。この点は著者自身も限界として認めている。

**計算資源と再現性**
モデルとコードが公開されても、原論文と同等の事前学習には多数のA100/A800と12億ペア規模のデータが必要である。完全再現は限られた組織にしか現実的でない。

**検索時コスト**
マルチベクトル方式は高コストであるため、実験でも上位200件の再ランキングに限定される。また、ハイブリッド運用では密・疎インデックスの双方が必要で、データセットごとにスコア重みも調整されている。したがって「単一モデル」であっても、検索基盤全体が単純になるとは限らない。([arXiv][2])

**BM25比較の注意点**
主表のBM25はM3と同じXLM-Rトークナイザを使っている。付録では一般的なLucene Analyzerを使うと、MLDRのBM25が53.6から64.1へ向上する。M3 Allは65.0であるため、標準的なBM25との長文検索性能差は11.4ポイントではなく、実質0.9ポイントまで縮小する。この付録結果は、長文検索ではBM25が依然として非常に強いことを示している。([arXiv][2])

## 7. まとめ

M3-Embeddingは、100以上の言語、三種類の検索方式、最大8,192トークンの入力を一つのモデルで扱う汎用検索埋め込みモデルである。中核技術は、異なる検索ヘッドの統合スコアを教師とするSelf-Knowledge Distillation、大規模多言語データの段階的学習、長文向けの効率的バッチングである。

実験では多言語・言語横断検索で密検索単独が強く、長文検索では疎検索とハイブリッドが大きく寄与した。これにより、RAGや企業内検索など、言語、文書長、検索要件が混在する環境に適した基盤モデルとなっている。

## 8. 総評

**総合評価：採択が妥当な、実用性の高い検索基盤研究である。**

自己蒸留の考え方は比較的簡潔だが、疎検索での大幅な改善によって有効性が明確に示されている。さらに、多言語、長文、複数検索方式を同時に高水準で実現した点は、情報検索システムとしての価値が高い。一方、性能向上の一部は大規模データと計算資源に依存しており、検索時のコストやベースライン設定には慎重な解釈が必要である。
