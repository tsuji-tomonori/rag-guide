# P121「Compressing Context to Enhance Inference Efficiency of Large Language Models」

## 1. はじめに

本論文は、LLMに入力するコンテキストから情報量の低い部分を削除することで、回答品質を大きく損なわずに推論コストを削減する手法 **Selective Context** を提案している。著者はYucheng Li、Bo Dong、Frank Guerin、Chenghua Linであり、論文は **EMNLP 2023 Main Conference** に採択・掲載された。([ACL Anthology][1])

Selective Contextの特徴は、対象LLMを追加学習せず、比較的小さな言語モデルで各単語・句の「自己情報量」を計算し、予測しやすい部分を削除する点にある。著者らは、入力コストを約50％削減した条件でも、完全なコンテキストを用いた出力とのBERTScoreの低下が0.023にとどまると報告している。([arXiv][2])

---

## 2. 背景

LLMの推論コストは、モデルサイズだけでなく入力コンテキスト長にも依存する。長い入力は、アテンション計算、KVキャッシュ、応答遅延、APIの入力トークン料金を増加させる。また、モデルのコンテキストウィンドウを超える文章は、そのまま入力できない。

一方、自然言語には次のような冗長性が含まれる。

* 文法的に予測しやすい機能語
* 前後の内容から容易に復元できる表現
* 同じ意味の繰り返し
* LLMが事前学習ですでに獲得している一般知識

本論文は、このような「LLMにとって予測しやすい部分」は、下流の推論に不可欠でない場合が多いという仮説に立つ。

各トークン (x_i) の自己情報量は、次式で定義される。

[
I(x_i)=-\log_2 P(x_i\mid x_0,\ldots,x_{i-1})
]

言語モデルが高い確率で予測できるトークンほど自己情報量が小さく、予測困難なトークンほど大きい。Selective Contextは、自己情報量が小さい部分を冗長とみなして削除する。([arXiv][2])

## 3. 提案手法

### 3.1 Selective Contextの処理手順

手法は次の3段階で構成される。

1. 小規模な因果言語モデルを用いて、入力中の各トークンの自己情報量を計算する。
2. トークンを語彙単位にまとめ、単位ごとの自己情報量を集約する。
3. 自己情報量が低い単位を、指定した割合だけ削除する。

削除後も、残された単位の元の出現順序は維持される。対象となるLLM本体のパラメータ、推論アルゴリズム、プロンプト形式を変更する必要はない。([arXiv][2])

### 3.2 語彙単位

論文では以下の3種類を比較している。

| 単位       | 概要              |
| -------- | --------------- |
| Token    | 個々のサブワードトークンを削除 |
| Phrase   | 名詞句などのまとまりを削除   |
| Sentence | 文単位で削除          |

Phrase単位ではspaCyのnoun chunkを用い、Sentence単位ではNLTKの文分割を用いる。利用できる安定した動詞句抽出器がなかったため、主として名詞句が用いられている。実験ではPhrase単位が最も安定した結果を示した。([arXiv][2])

### 3.3 圧縮率

論文における圧縮率 (p) は、厳密には「削除する語彙単位の割合」である。例えば (p=0.5) なら、自己情報量が低い方から50％の語彙単位を削除する。

ただし、句ごとのトークン数は異なるため、句を50％削除しても、必ずしもトークン数が50％減るわけではない。論文中の例では、語彙単位の50％を除去した結果、残存トークンは57.2％で、実際のトークン削減率は42.7％となっている。([arXiv][2])

### 3.4 情報量の計算方法

全文を一度にスコアリングすると、文章後半のトークンほど前方文脈が多くなり、自己情報量が系統的に低くなる問題が生じた。そのため、実験では各文を独立に言語モデルへ入力し、文ごとに自己情報量を計算している。([arXiv][2])

## 4. 実験方法

### 4.1 データセット

事前学習データとの重複を抑えるため、原則として2023年3月以降に作成されたデータを用いている。

| データセット   |  件数 |  平均文数 |   平均句数 | 平均トークン数 |
| -------- | --: | ----: | -----: | ------: |
| arXiv    | 408 | 28.20 | 514.55 |  864.85 |
| ShareGPT | 470 | 27.35 | 389.42 |  689.32 |
| BBC News | 294 | 25.63 | 523.96 |  732.54 |

arXivについては論文全体ではなく最初の2節だけを使用し、BBC Newsでは記事本文、ShareGPTでは対話履歴を利用した。また、比較対象モデルの入力長制限をそろえるため、2,048トークンを超える事例は除外された。([arXiv][2])

### 4.2 評価対象モデル

以下のモデルが評価された。

* GPT-3.5、GPT-4
* LLaMA 7B、13B、30B
* Vicuna 7B、13B

自己情報量を計算する補助モデルには、LLaMA・Vicuna系ではLLaMA-7B、OpenAI系ではGPT-3 Curieが用いられた。対象モデルより小さいモデルで圧縮対象を選択できるかを調べる設計である。([arXiv][2])

### 4.3 タスク

次の4種類のタスクで評価している。

* 元のコンテキストの再構成
* 要約
* 質問応答
* 会話応答

圧縮率は0.2、0.35、0.5、0.65、0.8の5段階で設定された。主な比較対象は、同じ割合の語彙単位を無作為に削除するRandom Deletionである。([arXiv][2])

### 4.4 評価指標

自動評価には以下が使用された。

* BLEU
* METEOR
* ROUGE-1、ROUGE-2、ROUGE-L
* BERTScore

重要な点として、正解データや人手作成回答ではなく、**圧縮前の完全なコンテキストを同じLLMに入力した際の出力**を参照回答としている。したがって、この評価が直接測っているのは、圧縮後も元のモデル出力を再現できるかという「出力の安定性」であり、回答そのものの正しさではない。([arXiv][2])

## 5. 結果

### 5.1 自動評価

温度0.7における、全モデル・全タスク平均のBERTScore F1は次のようになった。

|  削除率 | BERTScore F1 | 完全コンテキストとの差 |
| ---: | -----------: | ----------: |
|    0 |        0.909 |           ― |
| 0.20 |        0.902 |      −0.007 |
| 0.35 |        0.897 |      −0.012 |
| 0.50 |        0.887 |      −0.023 |
| 0.65 |        0.877 |      −0.032 |
| 0.80 |        0.863 |      −0.046 |

50％の語彙単位を削除した条件でも、BERTScore F1は0.887であり、完全コンテキストとの差は0.023だった。一方、同じ条件でBLEUは0.17、ROUGE-1は0.12程度低下しており、表層的な語句の一致は意味的類似度より大きく損なわれている。([arXiv][2])

### 5.2 無作為削除との比較

Greedy decodingにおける削除率0.5の結果では、BERTScore F1が次のようになった。

* Random Deletion：0.873
* Selective Context：0.900

ROUGE-1もRandom Deletionの0.576に対し、Selective Contextは0.642だった。単に入力を短くするのではなく、自己情報量に基づいて削除対象を選択することに効果がある。いくつかの指標では、50％削除したSelective Contextが、20％しか削除していないRandom Deletionを上回った。([arXiv][2])

### 5.3 タスク別傾向

元の文章を再現する再構成タスクは、圧縮率の上昇に伴って最も急速に性能が低下した。これは、削除した表現そのものを復元する必要があるため妥当な結果である。

一方、質問応答と要約は比較的頑健だった。これらのタスクでは、入力のすべてを再現する必要がなく、主要情報が残っていれば出力を生成できるためと考えられる。会話タスクではスコアが不安定だったが、自由度の高い会話応答を単一の参照出力との類似度で測ること自体にも限界がある。([arXiv][2])

### 5.4 語彙単位の比較

総合的には次の順で良好だった。

[
\text{Phrase} > \text{Token} > \text{Sentence}
]

Token単位の削除では語や文の構造が壊れやすい。Sentence単位では一度に失われる情報量が大きく、結果のばらつきが増えた。Phrase単位は意味的なまとまりを維持しながら細粒度に削除できるため、最も安定した。([arXiv][2])

### 5.5 モデルサイズとInstruction Tuning

LLaMA 7B、13B、30Bの比較では、モデルを大きくしても圧縮入力に対する頑健性が明確に向上する傾向は確認されなかった。

一方、Instruction TuningされたVicunaは、ベースモデルであるLLaMAより圧縮入力に対して強かった。LLaMAでは、圧縮によって不自然になった入力に対して指示自体を適切に実行できない場合が多かったのに対し、Vicunaはより安定してタスクを遂行した。([arXiv][2])

### 5.6 忠実性評価

GPT-3.5による質問応答について、合計1,000件を人手評価した結果は以下の通りである。

|  削除率 | 回答拒否数 | 平均回答長 |  非忠実率 |
| ---: | ----: | ----: | ----: |
| 0.20 |     0 | 156.5 | 0.027 |
| 0.35 |     6 | 136.0 | 0.050 |
| 0.50 |     4 | 140.2 | 0.038 |
| 0.65 |    19 | 131.2 | 0.051 |
| 0.80 |    27 | 103.7 | 0.086 |

削除率0.5では非忠実率が0.038に抑えられた。ただし、圧縮率が高くなると、根拠不足による回答拒否が増え、回答も短くなる傾向がある。なお、「分からない」と回答したケースは非忠実率の算出から除外されている。([arXiv][2])

### 5.7 推論効率

ケーススタディでは、完全コンテキストと削除率0.5の条件が次のように比較された。

| 条件                    |   CUDAメモリ | 1トークン当たり時間 |
| --------------------- | --------: | ---------: |
| 完全コンテキスト              | 77,695 MB |   110.8 ms |
| Selective Context 0.5 | 61,885 MB |    76.3 ms |

応答生成時間は約31～32％短縮されている。コンテキスト圧縮処理には一度だけ46.1 msを要した。著者はメモリ使用量を約36％削減したと報告しているが、表に示された総CUDAメモリ値を直接比較すると削減率は約20.3％であり、固定的なモデルメモリを除いて計算した可能性があるものの、論文中では算出方法が十分説明されていない。([arXiv][2])

## 6. 考察

### 長所

最大の長所は、追加学習を必要とせず、対象LLMの内部へアクセスしなくても使用できる点である。オープンモデルだけでなく、GPT-4のようなAPI型モデルにも適用できる。また、圧縮用モデルを対象LLMより小さくできるため、同一コンテキストに対して複数回質問する用途では、前処理コストを償却しやすい。

アルゴリズムも単純であり、モデル固有のアテンション重み、勾配、ファインチューニングデータを必要としない。LLM入力前の前処理器として導入できる点は、実用上大きな利点である。

### 課題1：自己情報量はタスク関連度ではない

自己情報量が低いことと、下流タスクに不要であることは同義ではない。例えば「ない」「唯一」「以前」といった予測しやすい表現でも、否定、数量、時系列を決定する重要語になり得る。

逆に、珍しい固有名詞や数字は高い自己情報量を持ちやすいが、現在の質問には無関係かもしれない。本手法は質問内容を見ずにコンテキストを圧縮するため、特に質問応答では、**query-awareな重要度評価**を組み合わせる余地が大きい。

### 課題2：正確性ではなく出力の再現性を評価している

主評価では、完全コンテキストを与えた同じモデルの出力を参照としている。そのため、完全コンテキスト時の回答が誤っていても、その誤りを圧縮後に再現すれば高いスコアになる。

したがって、本実験から強く主張できるのは「圧縮してもモデル出力が大きく変化しにくい」という点であり、「圧縮しても正解率が維持される」とまではいえない。人手正解を持つQAデータセットや、事実検証を伴う評価が必要である。

### 課題3：比較手法が弱い

主なベースラインは無作為削除であり、Selective Contextがそれを上回ること自体は十分予想される。より説得力のある比較には、以下が必要だった。

* TF-IDFやBM25による重要文選択
* 質問との類似度に基づく抽出
* Extractive summarization
* Attentionやembeddingを使った選択
* 学習型プロンプト圧縮手法

したがって、自己情報量方式が他の情報選択方式より優れているかは、本実験だけでは判断できない。

### 課題4：「長文コンテキスト」の検証としては限定的

2,048トークンを超える事例を除外しており、平均入力長も700～900トークン程度である。arXiv論文も最初の2節しか使用していない。そのため、数万～数十万トークン規模のLong-context LLMに対する有効性は未検証である。([arXiv][2])

### 課題5：前処理コストの評価が不十分

Selective Contextは、LLM推論前に別の言語モデルで入力全体を読み込む必要がある。論文では一つのケースについて46.1 msと報告しているが、文書長、補助モデル、ハードウェア、繰り返し利用回数を変えた体系的な評価はない。

一度しか利用しない短い入力では、圧縮による節約より前処理コストが大きくなる可能性がある。反対に、同じ文書に何度も質問するRAGや文書分析では、圧縮結果を再利用できるため有利になる。

### 課題6：アノテーションと統計的検証

要約の人手評価には3名の大学生が参加し、1,150件を二値判定しているが、評価者間一致度は明示されていない。また、多くの結果について信頼区間や統計的有意差検定が示されていない。([arXiv][2])

## 7. まとめ

本論文は、トークンの自己情報量を利用して、LLM入力から予測しやすい語句を削除するSelective Contextを提案した。

主要な知見は以下である。

* 追加学習なしでさまざまなLLMに適用できる。
* TokenやSentenceよりPhrase単位の圧縮が安定している。
* 語彙単位を50％削除しても、BERTScore F1の低下は0.023だった。
* 無作為削除より明確に高い品質を維持した。
* Instruction Tuningされたモデルは、圧縮された不自然な入力に比較的強い。
* 高圧縮率では回答拒否と事実的不整合が増える。
* 一例では生成時間を約32％短縮した。

ただし、本手法が評価しているのは主に完全コンテキスト時の出力との一致であり、タスク正解率の維持を直接保証するものではない。また、質問との関連性を考慮しないこと、強力な比較手法が不足していること、真に長大なコンテキストで検証されていないことが主要な課題である。

## 8. 総評

### 査読結果

本論文は **EMNLP 2023 Main Conferenceに採択**された。メタレビューでは、著者のrebuttalが査読者の懸念の大部分に対応したと判断され、最終推薦は **4：Sound and Moderately Exciting—Main ConferenceまたはFindingsへの採択相当**だった。([OpenReview][3])

一方、一部の査読者からは、既存技術を組み合わせた漸進的な研究であり、さらに改訂が必要との批判も示されていた。([OpenReview][3])

### 総合評価

**採択は妥当である。**

本研究の価値は、理論的に高度な新規性よりも、単純で導入しやすいコンテキスト圧縮法を提示し、複数のLLMと複数のタスクで広く検証した点にある。特に、閉鎖的なAPIモデルにも適用でき、モデルの再学習を必要としないことは実用上重要である。

ただし、論文の主張は次のように限定して理解すべきである。

> Selective Contextは、入力を大幅に短縮しても元のLLM出力を比較的よく維持できる、実用的な汎用ベースラインである。しかし、回答の正確性やタスク固有の重要情報を保証する圧縮法ではない。

研究としては有用な出発点だが、評価設計、比較手法、長文条件、コスト算定の透明性には改善余地がある。現在の水準では、完成された最適解というより、後続のquery-aware圧縮、RAG向け圧縮、学習型プロンプト圧縮へつながる基礎的研究と評価するのが適切である。
