# P160「StruQ: Defending Against Prompt Injection with Structured Queries」

## 1. はじめに

本論文は、LLMアプリケーションにおける**プロンプトインジェクション**を、信頼できる命令と信頼できないデータを同一の文字列としてLLMへ渡すことに起因する「制御とデータの混同」と捉え、両者を明示的に分離する **StruQ（Structured Queries）** を提案する。

著者はSizhe Chen、Julien Piet、Chawin Sitawarin、David Wagnerで、UC Berkeleyの研究である。論文は34th USENIX Security Symposium（USENIX Security 2025）に採択され、発表区分はShort Presentation、掲載ページは2383–2400である。([USENIX][1])

なお、本論文でいう「構造化クエリ」はJSONなどの構造化出力ではなく、**LLMへの入力を、命令領域とデータ領域に権限分離する仕組み**を意味する。

## 2. 背景

一般的なLLMアプリケーションでは、開発者が作成した命令と、ユーザー文書や検索結果などの外部データを連結してLLMへ入力する。ところがデータ中に「以前の指示を無視して別の処理を行え」といった文が含まれると、LLMはそれを単なるデータではなく、新しい命令として解釈する可能性がある。

本論文の脅威モデルでは、攻撃者はデータ部分を任意に変更でき、アプリケーションのプロンプトや入力形式も知っている。一方、開発者のプロンプト、フロントエンド、モデル自体は変更できない。LLMがデータ中の隠れた命令に従えば、元の命令も同時に実行したかどうかに関係なく攻撃成功と判定する。

著者らはこの問題を、SQLインジェクションなどと同じ古典的なインジェクション問題として整理する。すなわち、制御情報とデータを同一チャネルで送る設計が根本原因であり、プロンプトに注意書きを追加するだけでは十分な防御にならないという立場である。

また、プロンプトインジェクションは、モデル提供者の安全方針を回避する「ジェイルブレイク」とは区別される。プロンプトインジェクションでは、モデル提供者とアプリケーション開発者が信頼主体であり、外部データの供給者が攻撃者となる三者構造を想定している。

## 3. 提案手法

StruQは、次の三つの要素から構成される。

### 3.1 プロンプトとデータの分離

アプリケーションのAPIでは、開発者の命令である`prompt`と、ユーザー由来の`data`を別々の引数として受け取る。実際のTransformer入力では両者を単一のトークン列へ変換するが、その際に予約済み特殊トークンを用いて境界を明示する。

例えば、従来のAlpaca形式における`### instruction:`などを、次の特殊トークンに置き換える。

* `[MARK]`：`###`
* `[INST]`：instruction
* `[INPT]`：input
* `[RESP]`：response
* `[COLN]`：コロン

これらは通常のテキストとしては生成されない、モデル専用の予約トークンとして扱われる。

### 3.2 セキュアフロントエンド

フロントエンドは、ユーザーデータに特殊トークンの文字列表現が含まれていないか確認し、`[MARK]`、`[INST]`、`[INPT]`、`[RESP]`、`[COLN]`および`##`を再帰的に除去する。

これにより、攻撃者が正規の区切りを偽装し、「モデルが既に回答を終え、次の命令を受け取った」と誤認させるCompletion攻撃を防ぐ。正規の区切りはフロントエンドだけが生成できるため、データ側から命令領域へ脱出することが困難になる。

新しい特殊トークンの埋め込みは完全なランダム初期化ではなく、対応する通常トークンの埋め込み、例えば`[MARK]`なら`###`、`[INST]`なら`instruction`の埋め込みから初期化する。著者らは、この初期化がモデルの有用性維持に重要だったと報告している。

### 3.3 Structured Instruction Tuning

通常のinstruction tuningは、入力中のどこに現れた命令でも従うようモデルを学習させる。StruQではこれを変更し、**命令領域の指示だけに従い、データ領域の指示を無視する**よう学習させる。

訓練データは概ね次の割合で構成される。

* 50%：元のinstruction tuningデータをそのまま使用
* 25%：データ中に別サンプルの命令を追加するNaïve攻撃
* 25%：偽の回答と偽の区切り、その後に別命令を追加するCompletion-Other攻撃

攻撃付きサンプルでも教師出力は元の正規命令に対する回答のままとする。したがって、モデルは命令の内容ではなく、**命令が置かれた位置とチャネル**を基準に従うべきかを学習する。注入命令には、正規命令と同じinstruction tuningデータ分布から抽出した命令を利用しており、特定の攻撃文言だけを暗記することを避けている。

## 4. 実験方法

対象モデルは、非instruction-tunedのLlama-7BとMistral-7Bである。cleaned Alpacaデータセットを用いてモデル全体を3エポックfine-tuningし、学習率はLlamaが (2\times10^{-5})、Mistralが (2.5\times10^{-6}) である。

有用性はAlpacaEval 1.0の805サンプルに対するwin rateで評価する。セキュリティ評価では、データ入力を持つAlpacaFarmの208サンプルを使用し、注入命令として主に「Print exactly Hacked!」を末尾へ追加する。出力中の“Hacked”は手作業でも確認し、誤検知を除外してAttack Success Rate（ASR）を計算する。

攻撃は15系統以上を対象としている。単純な追加命令、Ignore攻撃、改行・削除文字を使うEscape攻撃、偽の回答と区切りを使うCompletion攻撃、HackAPromptの人手作成攻撃に加え、最適化型攻撃として以下も評価している。

* **TAP**：攻撃用LLMがサンプルごとの注入文を生成
* **GCG**：モデルの勾配を利用して敵対的なsuffixを最適化する白箱攻撃

さらに、防御方式を知る攻撃者が区切り文字の大文字・小文字、ハッシュ数、空白、タイプミスなどを変更する適応的攻撃も実施している。

## 5. 結果

主要結果は以下の通りである。ASRは低いほどよく、Utilityは高いほどよい。

| 指標・攻撃                  | Llama：無防御 → StruQ | Mistral：無防御 → StruQ |
| ---------------------- | ----------------: | ------------------: |
| Completion-Real ASR    |          96% → 0% |            96% → 0% |
| Completion-RealCmb ASR |          71% → 0% |            83% → 2% |
| HackAPrompt ASR        |          52% → 0% |            38% → 0% |
| TAP ASR                |          97% → 9% |          100% → 36% |
| GCG ASR                |         97% → 58% |           99% → 56% |
| AlpacaEval Utility     |     67.2% → 67.6% |       80.0% → 78.7% |

手作業で設計された攻撃やCompletion攻撃については、StruQ導入後のASRがほぼ0～2%になった。一方、サンプルごとに最適化するTAPとGCGには完全には対処できず、特にGCGでは約56～58%の攻撃が依然として成功した。

有用性については、Llamaでは0.4ポイント上昇し、Mistralでは1.3ポイント低下した。著者らは影響を小さいと評価しているが、Mistralの低下については統計的有意性の境界付近であるとも述べている。

区切りを少し変更する適応的Completion攻撃では、無防御モデルのASRが61～93%程度だったのに対し、StruQでは概ね0%、最大でもLlamaの1%だった。正規の区切りはフロントエンドが除去し、類似しているだけの区切りは通常トークンとして扱われるため、structured instruction tuningによって無視されたと分析されている。

アブレーションでは、Naïve攻撃とCompletion-Other攻撃の両方を訓練データへ加える構成が、Utility 66.0%を維持しながら、対象とした非最適化攻撃の最大ASRを0%にした。特殊トークンをすべての区切り要素へ使用する構成が、有用性とセキュリティの両面で最も良かった。

## 6. 考察

本研究の最大の貢献は、プロンプトインジェクションを「より強い注意書きで防ぐ問題」ではなく、**APIとモデル学習の両方で制御とデータを分離する問題**として再定義した点にある。単なる防御プロンプトではなく、インターフェース、トークナイザ、入力フィルタ、fine-tuningを一体として設計している点は技術的に明快である。

特にCompletion攻撃への対応は重要である。区切り文字を置くだけでは、攻撃者が同じ区切りをデータ中へ記述できる。StruQは、予約トークンとフロントエンドのフィルタを組み合わせることで、区切りの生成権限をアプリケーション側へ限定している。

一方、SQLのprepared statementのような機械的な分離とは異なり、StruQの命令・データ境界は最終的にはモデルが学習した挙動である。そのため、形式的な安全保証はなく、意味的に正規命令へ適応するTAPやGCGでは境界が崩れる。GCGのASRが50%を超えていることから、「プロンプトインジェクションを解決した」というより、**広範な手作業攻撃を大幅に抑制する第一段階の防御**と評価すべきである。

また、以下の適用上の制約がある。

第一に、命令とデータを明確に分けられるAPI型アプリケーションが対象であり、ユーザー発話自体が命令にもデータにもなり得るマルチターン・チャットには直接適用できない。第二に、ジェイルブレイク、学習データ抽出、機密情報漏えいなどは防御対象外である。第三に、非instruction-tunedモデルからモデル全体を再学習する必要があり、API経由でしか利用できない商用モデルには導入しにくい。

実験面でも、7BモデルとAlpaca系データが中心であり、長文RAG、ツール実行型エージェント、大規模モデルに同じ結果が一般化するかは不明である。また、“Hacked”という明確な文字列を出力する攻撃成功判定は再現性が高い反面、部分的な命令逸脱、出力操作、データ流出などのより微妙な攻撃を過小評価する可能性がある。

BIPIAとの比較ではStruQが優れているものの、BIPIAはVicuna-7B、StruQはLlama-7Bを使っており、完全に同条件の比較ではない点にも注意が必要である。

## 7. まとめ

StruQは、信頼できる命令と信頼できないデータを構造的に分離し、データ中の命令を無視するようLLMを再学習する防御方式である。セキュアフロントエンド、予約済み特殊トークン、structured instruction tuningを組み合わせることで、単純攻撃、Ignore攻撃、Completion攻撃、既知形式の適応的攻撃をほぼ無効化しながら、モデルの一般的な有用性を維持した。

ただし、TAPやGCGなどのタスク依存・最適化型攻撃は依然として有効である。したがって、本論文の価値は完全防御の達成ではなく、**プロンプトインジェクション防御を入力構造と権限設計の問題として扱う、実用的かつ発展性の高い設計原則を提示したこと**にある。

## 8. 総評

**公式査読結果：採択。**
USENIX Security 2025のCycle 1 Accepted Papersに掲載され、発表区分は**Short Presentation**である。最終論文はUSENIX Security 25の論文集に収録されている。公開ページでは、個別の査読スコアや査読コメントまでは確認できない。([USENIX][1])

**独自査読判定：Weak Accept。**

制御とデータの分離というセキュリティ上妥当な原則をLLMへ適用し、フロントエンドとモデル学習を一体化した点は新規性・実用性ともに高い。攻撃種類、適応的攻撃、アブレーション、既存防御との比較も充実しており、結果の再現性も比較的高い。

一方で、強力な最適化攻撃に対するASRが高く、形式的保証もない。対象アプリケーション、モデル規模、評価指標にも制約がある。それでも、限界を明確に認めたうえで、後続研究や実システムの安全なAPI設計につながる重要な基盤を提示しており、採択相当と判断する。
