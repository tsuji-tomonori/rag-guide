# P101：Role-Based Access Control Models

**著者**：Ravi S. Sandhu、Edward J. Coyne、Hal L. Feinstein、Charles E. Youman
**掲載誌**：*IEEE Computer*, Vol. 29, No. 2, pp. 38–47, 1996

## 1. はじめに

本論文は、当時統一的な定義が存在しなかった**ロールベースアクセス制御（Role-Based Access Control：RBAC）**を整理し、機能の異なる4種類の参照モデルとして体系化した論文である。

RBACの基本的な考え方は、権限をユーザへ直接付与するのではなく、まず組織上の職務や責任を表す「ロール」に権限を割り当て、そのロールをユーザへ割り当てることである。ユーザの異動や職務変更時にはロール割当だけを変更すればよく、権限管理を簡素化できる。

本論文の主要な貢献は、RBACを **RBAC₀、RBAC₁、RBAC₂、RBAC₃** に分類した点と、通常のアクセス制御とは別に、RBAC自身を管理するための管理モデルを提示した点にある。([NIST Computer Security Resource Center][1])

## 2. 背景

従来のアクセス制御では、任意アクセス制御（DAC）や強制アクセス制御（MAC）が中心であった。しかし、実際の企業や行政組織では、個人単位よりも「医師」「監査担当者」「プロジェクト管理者」といった職務単位で権限を管理する方が自然である。

論文では、ロールとユーザグループの違いも明確にしている。一般的なグループは主として「ユーザ集合」を表すのに対し、ロールは次の二つを結び付ける媒介要素である。

[
\text{ユーザ集合} \longleftrightarrow \text{ロール} \longleftrightarrow \text{権限集合}
]

したがって、ロールについては、その所属ユーザだけでなく、割り当てられた権限も容易に確認できることが重要である。

また、RBACは特定のセキュリティ方針に限定されたモデルではなく、**ポリシー中立的な仕組み**として位置付けられる。適切に構成すれば、最小権限、職務分離、データ抽象化を支援できる。ただし、RBACを導入するだけでこれらの原則が自動的に保証されるわけではなく、不適切なロール設計や権限割当によって原則を破ることも可能である。([NIST Computer Security Resource Center][1])

## 3. 提案手法

### RBAC₀：基本モデル

RBAC₀は、RBACシステムが最低限備えるべき構成を定義する。

* (U)：ユーザ集合
* (R)：ロール集合
* (P)：権限集合
* (S)：セッション集合
* (UA \subseteq U \times R)：ユーザ・ロール割当
* (PA \subseteq P \times R)：権限・ロール割当

ユーザはセッションを開始すると、自分に割り当てられているロールの一部だけを有効化できる。セッション (s) における有効権限は、おおむね次のように表される。

[
Permissions(s)
==============

\bigcup_{r \in roles(s)}
{p \mid (p,r)\in PA}
]

強力な管理ロールを常時有効化せず、必要な作業時だけ有効化できるため、セッションの概念は最小権限の実現に有効である。また、本モデルでは権限を原則として肯定的な許可として扱い、拒否は「負の権限」ではなく制約として表現する。([NIST Computer Security Resource Center][1])

### RBAC₁：ロール階層

RBAC₁はRBAC₀に**ロール階層 (RH)** を追加する。上位ロールは下位ロールの権限を継承する。

例えば、

[
\text{医療従事者}
<
\text{医師}
<
\text{専門医}
]

という階層では、専門医は医師および医療従事者の権限を継承する。階層は反射的、推移的、反対称的な半順序として定義され、複数の下位ロールから権限を継承する多重継承も認められる。

一方、上位者に継承させたくない権限については「プライベートロール」を追加し、継承範囲を構造的に分離する方法が示されている。([NIST Computer Security Resource Center][1])

### RBAC₂：制約

RBAC₂はRBAC₀に制約を追加する。制約は、ユーザ割当、権限割当、セッション、ロール有効化などの状態が組織のセキュリティ方針上「許容可能か」を判定する述語である。

代表的な制約は次のとおりである。

* **相互排他制約**：同一人物を購買担当と支払承認担当の両方に割り当てない
* **カーディナリティ制約**：あるロールの所属人数を最大1名などに制限する
* **前提ロール制約**：プロジェクト所属者だけをプロジェクト内テスト担当に割り当てる
* **セッション制約**：二つのロールへの所属は認めるが、同一セッションでの同時有効化は禁止する

ユーザ割当時の相互排他は現在の静的職務分離に、セッション上の同時有効化禁止は動的職務分離に対応する考え方である。論文では、ユーザ割当だけでなく、同じ強力な権限が相互排他的な複数ロールへ付与されないようにする、権限割当側の制約も論じている。([NIST Computer Security Resource Center][1])

### RBAC₃：統合モデル

RBAC₃は、ロール階層を持つRBAC₁と、制約を持つRBAC₂を統合したモデルである。

[
RBAC_3 = RBAC_1 + RBAC_2
]

ただし、階層と制約を単純に組み合わせるだけではない。例えば、テスト担当とプログラマを相互排他にした場合、両者の権限を継承するプロジェクト管理者を許容するかという問題が生じる。したがって、制約が直接割当だけに適用されるのか、継承された所属・権限にも適用されるのかを明確にする必要がある。([NIST Computer Security Resource Center][1])

### RBAC管理モデル

論文はさらに、RBAC自身をRBACで管理するモデルを提案している。

通常ロール (R)・通常権限 (P) とは別に、

* 管理ロール (AR)
* 管理権限 (AP)
* 管理ロール階層 (ARH)
* 管理権限割当 (APA)

を設ける。

管理権限には、ユーザのロール追加・削除、ロールへの権限割当、ロール階層の変更などが含まれる。管理者ごとに管理可能なサブ階層や操作を限定できるため、大規模組織における管理権限の委譲を表現できる。([NIST Computer Security Resource Center][1])

## 4. 実験方法

本論文はシステム実装や性能測定を行う実験論文ではなく、**概念モデル・参照モデルを構築する理論的論文**である。

評価方法は、主に以下で構成されている。

1. 集合と関係を用いたRBAC構成要素の形式的定義
2. 医療、会計、ソフトウェア開発プロジェクトなどの事例による説明
3. 基本機能、階層、制約を分離したモデル間の構造比較
4. 通常のアクセス制御と管理操作を分離した管理モデルの検討

プロトタイプ実装、処理時間、管理コスト、誤設定率、ユーザ調査などの定量評価は行われていない。そのため、論文が示すのは性能上の優位性ではなく、RBACを記述・比較するための概念的妥当性と整理能力である。

## 5. 結果

本論文によって、RBACの機能を次のように段階的かつ直交的に整理できるようになった。

| モデル   | 主な機能                   |
| ----- | ---------------------- |
| RBAC₀ | ユーザ、ロール、権限、セッション       |
| RBAC₁ | RBAC₀＋ロール階層            |
| RBAC₂ | RBAC₀＋制約               |
| RBAC₃ | ロール階層＋制約               |
| 管理モデル | 管理ロールと管理権限によるRBAC自身の管理 |

特に、RBAC₁とRBAC₂を単純な上下関係ではなく、独立した拡張として整理したことが重要である。これにより、階層だけを必要とするシステム、制約だけを必要とするシステム、両方を必要とするシステムを共通の枠組みで比較できる。

その後、この枠組みはNISTの統一RBACモデルへ取り込まれ、2004年のANSI/INCITS標準、さらに2012年版へとつながった。したがって、本論文の成果は単なる概念整理にとどまらず、RBAC標準化の基礎の一つとなった。([NIST Computer Security Resource Center][2])

## 6. 考察

本論文の最大の長所は、複雑になりやすいRBACを、基本機能・階層・制約という独立性の高い構成要素へ分解した点である。特に、セッションを通じたロールの選択的有効化、権限割当側にも適用される職務分離、管理操作を通常操作と分離する設計は、アクセス制御モデルとして非常に整理されている。

また、RBACを特定の組織方針そのものではなく、各組織が方針を実装するためのポリシー中立的機構として扱った点も妥当である。これにより、RBACはDACやMACと排他的なものではなく、必要に応じて併用できる。

一方、次の限界がある。

* 制約は主として非形式的に説明され、統一的な制約言語や競合解決規則がない
* ロールや権限を実組織からどのように設計・抽出するかというロールエンジニアリングを扱っていない
* プライベートロールによる継承制御は、複雑な組織ではロール数を増大させる可能性がある
* 時刻、場所、端末状態、対象データの属性など、ロール以外の文脈条件を直接扱わない
* 業務手順や操作順序の制御は対象外であり、ワークフロー制御を別途重ねる必要がある
* 管理ロール階層そのものの最終管理を単一の最高セキュリティ責任者へ残しており、複数組織間の管理関係は未解決である
* 定量的・実装的な評価がなく、運用コストやスケーラビリティは実証されていない

著者自身も、制約の分類、形式表現、強制の難易度、ロール階層と管理モデルを統合した設計・分析手法を今後の研究課題として挙げている。([NIST Computer Security Resource Center][1])

## 7. まとめ

本論文は、ユーザに権限を直接割り当てる方式から、組織上の職務を表すロールを中心としたアクセス制御への移行を、明確な参照モデルとして定式化した。

中核となる成果は、RBACを基本モデル、階層モデル、制約モデル、統合モデルへ整理し、さらにRBAC自身の管理を管理ロールによって制御する枠組みを提示したことである。実証評価はないものの、その後の研究・製品比較・標準化に利用可能な共通語彙と構造を提供した点で、アクセス制御研究における基礎的論文と評価できる。

## 8. 総評

**査読結果：採択（Accept）**

実際に本論文は査読を経て、1996年に*IEEE Computer*へ採択・掲載されている。原稿にも掲載受理済みであることと、匿名査読者の指摘によって論文が改善されたことが記載されている。([NIST Computer Security Resource Center][1])

採択と判断する主な理由は、RBACについて統一されていなかった概念を、比較可能で拡張可能な参照モデルとして体系化した新規性と実用性である。特にRBAC₀～RBAC₃という分解は明快であり、管理モデルまで含めた射程も広い。

一方、制約の形式化や実装評価が不足しているため、完成された包括的理論というよりは、後続研究の基盤となる参照フレームワークである。それでも、論文の目的は十分に達成されており、当時のアクセス制御研究および実務に対する貢献は大きい。
