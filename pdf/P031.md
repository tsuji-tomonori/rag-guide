## 1. はじめに

**論文名:** *Query2doc: Query Expansion with Large Language Models*
**著者:** Liang Wang, Nan Yang, Furu Wei
**発表先:** EMNLP 2023 Main Conference

本論文は、大規模言語モデルに検索質問への回答らしい文章を生成させ、その文章を元の検索クエリに追加することで、情報検索性能を改善する手法 **Query2doc** を提案している。複雑なモデル構造を新たに導入するのではなく、LLMが内部に保持する知識を「疑似文書」として取り出し、既存のBM25やDense Retrieverに入力する点が特徴である。論文はEMNLP 2023の本会議論文として正式に採択・掲載されている。([ACL Anthology][1])

---

## 2. 背景

情報検索には、大きく分けて次の二つの方式がある。

**疎検索（Sparse Retrieval）**では、BM25のようにクエリと文書に含まれる単語の一致を利用する。高速で解釈しやすく、未知ドメインにも比較的強いが、クエリと文書で異なる表現が使われる「語彙的不一致」に弱い。

**密検索（Dense Retrieval）**では、クエリと文書をニューラルネットワークでベクトル化し、意味的な近さを測定する。十分な学習データがあれば高い性能を示す一方、学習ドメインから外れたデータではBM25が依然として競争力を持つ場合がある。

従来、この問題にはクエリ拡張が利用されてきた。RM3などの疑似適合フィードバックでは、最初の検索結果から関連語を取り出してクエリに追加する。しかし、初回検索結果が不正確だと、無関係な単語を追加してしまう。

一方、doc2queryやdocT5queryのような文書拡張では、各文書に対して想定クエリを事前生成する。ただし、生成モデルの学習やインデックスの再構築が必要になる。

本論文の発想は、初回検索結果から語を拾うのではなく、LLMに直接「このクエリに答える文書」を生成させることである。LLMの知識を、検索に有効な語彙を供給する外部知識源として扱っている。([ar5iv][2])

## 3. 提案手法

### 3.1 疑似文書の生成

入力クエリを (q) とする。LLMには以下の形式のプロンプトを与える。

> Write a passage that answers the given query:

この指示に加え、MS MARCOの学習データからランダムに選んだ4組の「クエリ―回答文書」の例をFew-shot exampleとして提示する。その後、対象クエリに対する疑似文書 (d') を生成する。

例えば、

> when was pokemon green released

というクエリに対し、LLMは発売日やゲームの背景を説明する文章を生成する。この文章には、検索対象文書にも現れやすい固有名詞、年月日、関連作品名などが含まれる。

### 3.2 疎検索への適用

BM25では、元のクエリを5回繰り返した後に疑似文書を連結する。

[
q^{+}=\operatorname{concat}({q}\times 5,d')
]

疑似文書は元のクエリより長いため、そのまま連結すると生成文書中の単語が過剰に重視される。元クエリを繰り返すことで、ユーザーが実際に入力した語の重みを維持する。

作成した (q^{+}) を通常のBM25に入力するため、BM25自体を学習し直す必要はない。

### 3.3 密検索への適用

Dense Retrieverでは、元クエリと疑似文書を特殊トークン `[SEP]` で連結する。

[
q^{+}=\operatorname{concat}(q,[SEP],d')
]

この拡張クエリをクエリエンコーダに入力する。論文では、次の二つの学習条件を調べている。

第一は、BERT-baseを初期値とするDPRをBM25由来のHard Negativeと対照学習損失で学習する設定である。

第二は、SimLMやE5などの強力なRetrieverを初期値とし、Cross-Encoder教師から知識蒸留する設定である。蒸留損失と対照学習損失を組み合わせている。

### 3.4 従来の疑似適合フィードバックとの違い

従来の疑似適合フィードバックは、初回検索の上位文書に依存する。Query2docは初回検索を行わず、LLMが生成する疑似文書を拡張情報として使う。

したがって、初回検索結果が不適切でも影響を受けない反面、LLMの幻覚や事実誤認をそのまま検索クエリへ混入させる危険がある。

## 4. 実験方法

### 4.1 データセット

ドメイン内評価には以下を使用している。

* MS MARCO Passage Ranking
* TREC Deep Learning 2019
* TREC Deep Learning 2020

ゼロショットのドメイン外評価には、BEIRから以下の5データセットを選択している。

* DBpedia
* NFCorpus
* SciFact
* TREC-COVID
* Touché 2020

評価指標は、MS MARCOではMRR@10、Recall@50、Recall@1000、TRECおよびBEIRでは主としてnDCG@10である。

### 4.2 比較手法

疎検索ではBM25、BM25+RM3、docT5queryと比較している。

密検索ではDPR、ANCE、HyDE、RocketQAv2、AR2、SimLM、E5などを比較対象としている。

### 4.3 LLMと生成条件

主実験ではOpenAIのtext-davinci-003を使用している。

* Few-shot例数: 4
* 最大生成長: 128トークン
* Temperature: 1
* Dense Retrieverの最大クエリ長: 144

Few-shot例はAPI呼び出しごとにMS MARCOの学習集合からランダムに選ばれる。疑似文書生成には約55万回のAPI呼び出しを行い、当時の費用は約5,000ドルだったと報告されている。

また、LLM規模の影響を調べるため、babbage、curie、davinci-001、davinci-003、GPT-4も比較している。

## 5. 結果

### 5.1 BM25

主要結果は以下の通りである。

| 評価設定                 | BM25 | BM25 + Query2doc |     差 |
| -------------------- | ---: | ---------------: | ----: |
| MS MARCO MRR@10      | 18.4 |             21.4 |  +3.0 |
| MS MARCO Recall@50   | 58.5 |             65.3 |  +6.8 |
| MS MARCO Recall@1000 | 85.7 |             91.8 |  +6.1 |
| TREC DL 2019 nDCG@10 | 51.2 |             66.2 | +15.0 |
| TREC DL 2020 nDCG@10 | 47.7 |             62.9 | +15.2 |

特にTREC DLでは15ポイント以上の改善が見られる。著者らは、TREC DLに長尾のエンティティ指向クエリが多く、LLMが生成した固有名詞や関連語との完全一致がBM25に有効だったと分析している。

論文要旨では「3%から15%の改善」と表現されているが、主結果表で示されている加算値は、厳密には3.0から15.2の絶対ポイント差である。

RM3はMS MARCOのRecall@1000をわずかに改善したものの、MRRやTREC DLではほとんど効果がなかった。docT5queryはMS MARCOではQuery2docより高い性能を示したが、ラベル付きデータによる生成モデルの学習を必要とする。

### 5.2 Dense Retriever

DPRでは一貫して比較的大きな改善が得られた。

| 指標                   |  DPR | DPR + Query2doc |
| -------------------- | ---: | --------------: |
| MS MARCO MRR@10      | 33.7 |            35.1 |
| TREC DL 2019 nDCG@10 | 64.7 |            68.7 |
| TREC DL 2020 nDCG@10 | 64.1 |            67.1 |

一方、もともと強力なSimLMやE5では改善幅が小さくなる。

* SimLMのTREC DL 2020: 69.7 → 71.6
* E5+KDのTREC DL 2020: 70.7 → 72.5

つまりQuery2docは既存Retrieverが弱いほど大きな情報を補えるが、強力な事前学習や知識蒸留によって既に豊富な情報を持つRetrieverでは限界効用が小さい。

### 5.3 ドメイン外評価

BM25では5つのBEIRデータセットすべてで改善している。例えばDBpediaは31.3から37.0、TREC-COVIDは65.6から72.2へ向上した。

Dense Retrieverでは結果が混在している。SimLMはDBpedia、TREC-COVID、Touché 2020で改善した一方、NFCorpusでは0.6ポイント、SciFactでは2.9ポイント低下した。E5+KDもSciFactで70.4から67.5へ低下している。

### 5.4 LLMの規模

TREC DLにおけるBM25+Query2docの性能は、LLMが強くなるほど向上した。

| 生成モデル            | TREC DL 2019 | TREC DL 2020 |
| ---------------- | -----------: | -----------: |
| babbage 1.3B     |         52.0 |         50.2 |
| curie 6.7B       |         55.1 |         50.1 |
| davinci-001 175B |         63.5 |         58.2 |
| davinci-003 175B |         66.2 |         62.9 |
| GPT-4            |         69.2 |         64.5 |

同じ175B規模でもdavinci-003がdavinci-001を上回っており、単なるパラメータ数だけでなく、学習データやInstruction Tuningの品質も重要であることが示唆される。

### 5.5 アブレーション

TREC DLにおける結果は次の通りである。

| 入力        | TREC DL 2019 | TREC DL 2020 |
| --------- | -----------: | -----------: |
| 元クエリ＋疑似文書 |         66.2 |         62.9 |
| 元クエリのみ    |         51.2 |         47.7 |
| 疑似文書のみ    |         48.7 |         44.5 |

疑似文書だけを検索に使うと、元クエリだけの場合よりも悪い。したがって、疑似文書は元クエリの代替ではなく、元クエリを補完する情報として働いている。

## 6. 考察

Query2docが有効な主な理由は、短いクエリを、検索対象文書に現れそうな語を含む長い表現へ変換できる点にある。特に人物名、地名、年月日、別名、関連組織などを生成できれば、BM25における語彙的不一致を直接解消できる。

また、生成文書を単独で使わず元クエリと連結する設計が重要である。元クエリが検索意図のアンカーとなり、疑似文書が補助的な語彙や背景情報を提供する。この構造により、LLMが余計な説明を生成した場合にも、元の検索語を一定程度維持できる。

一方、LLMの事実誤認は重要な問題である。論文中では、テレビ番組『Monk』のテーマ曲について、曲名と歌手は正しいものの、使用開始時期を誤って生成した例が示されている。事実誤認を含む語が検索クエリに追加されれば、関連文書ではなく、誤った内容を含む文書を上位に押し上げる可能性がある。SciFactにおける性能低下は、この問題が科学的事実検索で特に深刻になり得ることを示している。

実用上、最大の問題は計算コストと遅延である。通常のBM25では検索に約16ミリ秒だったのに対し、Query2docではLLM呼び出しに2秒以上、拡張後のインデックス検索にも177ミリ秒を要している。生成処理だけでなく、長いクエリによってBM25自体も10倍以上遅くなる。

再現性にも制約がある。Few-shot例のランダム選択とLLMの確率的生成という二つの乱数要因があり、BM25について3回実行した結果、TREC DL 2019では標準偏差1.14、2020では1.63が報告されている。分散評価がBM25の少数試行に限定されているため、Dense Retrieverを含む改善の安定性については、より体系的な反復実験が望まれる。

さらに、本手法はLLMが事前学習中に記憶した知識を性能源として積極的に利用している。そのため、将来的な評価では、LLMの学習データと評価コーパスとの重複や、質問・回答の記憶による性能上昇を区別する監査も重要になる。

---

## 7. まとめ

Query2docは、検索クエリからLLMに疑似文書を生成させ、元クエリと連結して検索する手法である。

最大の特徴は、BM25やDense Retrieverの構造を変更せずに導入できる単純さにある。BM25ではTREC DLのnDCG@10を約15ポイント改善し、DPR、SimLM、E5でも概ね性能向上を確認している。

一方で、効果はLLMの能力に強く依存し、強力なRetrieverでは改善幅が小さくなる。また、幻覚、API費用、2秒以上の生成遅延、確率的生成による再現性など、オンライン検索への導入には大きな課題が残る。

## 8. 総評

**公式の査読結果: 採択**

本論文は、**EMNLP 2023 Main Conferenceに採択され、正式な会議録に掲載されている**。([ACL Anthology][1])

**独立した査読判定: Accept**

アルゴリズムそのものの新規性は比較的単純であり、「LLMに回答らしい文章を生成させてクエリに追加する」という発想に集約される。しかし、疎検索と密検索の双方で効果を確認し、強いベースライン、ドメイン外評価、モデル規模、学習データ量、入力構成、遅延、乱数分散まで検証した実証的価値は高い。

特に、学習不要のBM25に対して大幅な改善を示した点と、「元クエリと疑似文書は相補的であり、疑似文書単独では性能が下がる」ことを明確に示した点は重要である。

一方、LLMの幻覚、特定の商用LLMへの依存、生成コスト、評価データの記憶可能性、強いDense Retrieverでの限定的な改善は明確な弱点である。それでも、LLMを情報生成器ではなく「検索表現を強化する知識源」として利用する方向性を簡潔かつ説得力のある形で示しており、採択に値する研究と評価できる。
