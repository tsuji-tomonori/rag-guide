# 1. はじめに

**MRKL Systems**は、大規模言語モデル単体ですべての知識処理や推論を行うのではなく、言語モデルを「自然言語インターフェース」として用い、計算機、データベース、検索API、専門モデルなどに処理を分担させるニューロシンボリック・アーキテクチャを提案した論文である。

MRKLは **Modular Reasoning, Knowledge and Language** の略で、「miracle」と発音する。論文はAI21 Labsから2022年に公開され、同社による実装を **Jurassic-X** と呼んでいる。中心的な主張は、LLMを万能な推論器として扱うのではなく、適切な専門モジュールへ処理を振り分ける「システムの一部」として扱うべきだ、というものである。

---

# 2. 背景

著者らは、大規模言語モデルには次の四つの構造的な問題があると整理している。

第一は、**最新情報へのアクセス不足**である。為替、株価、天気、日付などは事前学習後にも変化するため、静的なモデル内部のパラメータだけでは追随できない。

第二は、**非公開・組織固有情報へのアクセス不足**である。企業内データベース、顧客情報、ゲーム状態などは、通常の事前学習データには含まれない。

第三は、**離散的・厳密な推論の弱さ**である。LLMは小規模な算術問題に正答できても、桁数が増えると不安定になり、電卓のような決定的な正確性を保証できない。

第四は、**モデル爆発**である。タスクごとに巨大モデルをファインチューニングすると、保存・運用コストが増大する。新しいタスクを追加するたびにマルチタスクモデル全体を再学習する方式も、破滅的忘却と計算コストの面で現実的ではないと論じている。

これらの問題に対し、著者らは「モデルをさらに巨大化すればすべて解決する」という立場ではなく、LLMと外部システムの役割を分離するシステム設計を採用する。

---

# 3. 提案手法

MRKLシステムは、主に次の要素から構成される。

## 3.1 ルータ

入力された自然言語を解析し、どの専門モジュールに処理させるべきかを決定する。対象となる専門モジュールがない場合には、汎用LLMへ入力を送る。

## 3.2 エキスパートモジュール

エキスパートには二種類がある。

* **ニューラルモジュール**：汎用LLM、小規模な専門言語モデル、分類器など
* **シンボリックモジュール**：電卓、通貨換算器、データベース、検索システム、外部APIなど

一つのモジュールの出力を別のモジュールへ送ることも想定されており、複数段階の処理を組み合わせられる。

## 3.3 ニューラル–シンボリック間のインターフェース

最も重要な技術課題は、自然言語から、外部モジュールが受け取れる厳密な引数を抽出することである。

例えば、

> 「58に12を足すといくつか」

という入力から、電卓に渡す

```text
58 + 12
```

を正しく生成しなければならない。計算自体は電卓が正確に実行するため、LLMの役割は計算ではなく、**演算子とオペランドの抽出**に限定される。

## 3.4 期待される利点

著者らは、MRKLに以下の利点があるとしている。

* 新しいモジュールを追加しても、既存モジュールを再学習する必要がない
* 最新情報や組織内情報を外部データ源から取得できる
* 「電卓がこの値を返した」など、回答の由来を部分的に説明できる
* 複数モジュールを組み合わせた多段階処理が可能
* 専門モジュールがない入力は汎用LLMへフォールバックできる

したがってMRKLは、LLMを知識・推論の保管場所ではなく、自然言語と各種計算資源を接続する制御層として位置づけている。

# 4. 実験方法

論文の実験はMRKL全体ではなく、**自然言語から電卓用の数式を正確に抽出できるか**に焦点を当てている。

使用モデルは70億パラメータの **J1-large** であり、モデル本体を全面的に更新するのではなく、10個のプロンプトトークンを学習するprompt tuningを用いている。学習率は0.3、バッチサイズは32で、各実験を3～5回実施し、平均値と標準偏差を報告している。学習・テスト間では、同じ数式を別表現にしたデータも重複しないように分割されている。

訓練データはテンプレートから合成され、次の要素を変化させる。

* 数値を数字で書くか、英単語で書くか
* オペランドの桁数：1～9桁
* 四則演算：加算、減算、乗算、除算
* 演算回数：1回または2回
* 括弧と演算順序
* 質問文の表現形式

評価指標は最終回答の正解率である。実際の計算は電卓が行うため、誤答はすべて、LLMが誤った演算子またはオペランドを電卓へ渡したことに起因する。

# 5. 結果

| 実験           | 主な結果                                            |
| ------------ | ----------------------------------------------- |
| 桁数の汎化        | 1桁だけで学習しても、加算は1～9桁ですべて100%。乗算も6桁の98%を除いて100%    |
| 数字と英単語間の汎化   | 英単語で学習すると数字への正解率98.7%だが、数字だけで学習して英単語を評価すると15.6% |
| 質問形式の汎化      | 多くの形式でほぼ100%。ただし非疑問文形式では減算86.3%、除算72.7%まで低下     |
| 異なる演算への汎化    | 同じ演算内では99%以上だが、除算を含む演算間転移は不安定                   |
| 2演算式への汎化     | 29種類の式のうち22種類で90%以上。ただし一部の構文では30～70%程度          |
| 1演算から2演算への汎化 | 多くの組合せで高精度だが、加算→乗算は1%、減算→乗算は20.2%、減算→除算は22.4%   |

### 桁数

GPT-3による直接計算の既報値は、2桁で100%、3桁で80.4%、4桁で25.5%、5桁で9.3%と急激に低下した。一方、MRKL方式ではLLMが計算を行わないため、桁数が増えてもほぼ完全な精度を維持した。

### 数字表記と単語表記

数字表記で訓練し、英単語で書かれた数値を評価すると正解率は15.6%にとどまった。一方、英単語表記で訓練したモデルは、数字表記に98.7%、英単語表記に98.8%の精度を示した。より言語的に複雑な表現で学習する方が、単純な数字表記へ汎化しやすいことを示している。

### 表現形式

疑問文として提示されたテンプレート間ではほぼ完全に汎化した。ただし、「XとYの比は……」のような疑問文ではない形式では、特に除算で平均72.7%、標準偏差46.5%となり、学習の不安定性が確認された。

### 複数演算

未知の2演算式に対する汎化は全体として良好だったが、構文依存性が大きい。例えば ((A+B)\times C) は28.8%、(A+B\times C) は40.6%にとどまる一方、単純な加減算や連続乗算では100%を達成している。

### Few-shotとの比較

10例を与えるfew-shot方式は、数字表記の桁数変更には対応できたが、数値を英単語で記述すると桁数の増加に伴って性能が崩れ、9桁では0%となった。prompt tuningでは同条件で98%を維持した。また、非疑問文形式ではfew-shotが22%、prompt tuningが99.3%だった。

# 6. 考察

本論文の重要な点は、「LLMに計算を学習させる」ことと「LLMに計算式を抽出させ、電卓に実行させる」ことを明確に分離した点である。後者では、計算精度をモデルサイズや学習データ中の数値パターンに依存させず、決定的な外部モジュールに委譲できる。

現在の観点では、MRKLはツール呼び出し型LLMやエージェント・アーキテクチャの原型として理解できる。特に、以下の設計原則は現在でも有効である。

* 言語理解と厳密な計算を別コンポーネントに分ける
* モデルのパラメータを知識データベースとして使用しない
* ツール単位で機能を追加・更新する
* モジュール境界で入出力を検査する

一方、論文の実証範囲は限定的である。最大の問題は、MRKLの中心要素である**ルータのツール選択性能が実験されていない**ことである。論文が評価しているのは、電卓を使うと決定された後の引数抽出であり、「電卓、検索、データベース、汎用LLMのどれを呼ぶか」という本質的な問題は対象外となっている。

また、データは少数の合成テンプレートから生成されている。実際のユーザー入力に含まれる曖昧性、不要情報、省略、複数の解釈、誤字、会話文脈などに対する頑健性は確認されていない。特定の構文で精度が急落していることから、「自然言語全般に対するほぼ完全な信頼性」が示されたわけではない。

さらに、論文が挙げる「safe fallback」は厳密な意味で安全とはいえない。専門モジュールを選択できなかった場合に汎用LLMへ送ることは可用性を高めるが、そのLLMが誤情報を生成する可能性は残る。これは安全機構というより、**機能上のフォールバック**と表現する方が適切である。

---

# 7. まとめ

本論文は、LLMの弱点をモデル内部だけで解決するのではなく、外部知識源や決定的推論器を組み合わせることで補完するMRKLアーキテクチャを提案した。

実験では、自然言語から電卓用の引数を抽出するモデルをprompt tuningで構築し、桁数、表記、質問形式、演算種類、演算数に関する分布外汎化を評価した。桁数増加に対してはほぼ完全な精度を示し、LLMによる直接計算よりも外部電卓への委譲が有効であることを明確に示した。

ただし、検証されたのはMRKL全体の一部分であり、ルーティング、複数ツールの統合、実データへのアクセス、多段階推論、API障害、誤ったツール出力への対処などは未評価である。したがって本論文は、完成された汎用システムの実証というより、**ツール拡張型LLMの設計原理を提示したビジョン兼概念実証論文**と位置づけるのが妥当である。
