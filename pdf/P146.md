# P146：Efficient Memory Management for Large Language Model Serving with PagedAttention

## 1. はじめに

本論文は、LLM推論サーバにおける**KVキャッシュのメモリ管理**を改善するため、OSの仮想記憶から着想を得た **PagedAttention** と、それを実装した推論サーバ **vLLM** を提案する。SOSP 2023に採択され、論文集の611–626頁に掲載された。([ACMデジタルライブラリ][1])

中心的な主張は、LLMサービングのボトルネックが単なるGPU演算性能ではなく、動的に増減するKVキャッシュの断片化と重複保存にもある、という点である。PagedAttentionはKVキャッシュを固定長ブロックに分割し、物理的に連続していないGPUメモリ上でもAttentionを計算できるようにする。著者らは、同程度のレイテンシを維持しつつ、従来システムに対して概ね**2～4倍のスループット**を実現したと報告している。([arXiv][2])

## 2. 背景

LLMの推論は、プロンプト全体を並列処理する**prefill段階**と、トークンを1個ずつ生成する**decode段階**に分かれる。decodeでは、それまでの全トークンのKey・Valueを参照する必要があるため、過去のKey・ValueをKVキャッシュとして保持する。decodeは逐次的で、行列ベクトル積が中心になるため、GPUの演算性能よりもメモリ帯域やメモリ容量に制約されやすい。([arXiv][2])

例えばOPT-13Bでは、1トークンのKVキャッシュが約800KBを必要とし、最大2048トークンでは1リクエスト当たり最大約1.6GBになる。A100 40GBで13Bモデルを動かす例では、モデルパラメータが約65%を占め、KVキャッシュにも30%前後のメモリが使われるため、同時処理できるリクエスト数はKVキャッシュの管理効率に強く依存する。([arXiv][2])

従来方式では、各リクエストのKVキャッシュを連続領域に置くため、以下の無駄が生じる。

* 最大出力長を見越して事前確保する**予約領域**
* 確保領域を使い切れない**内部断片化**
* 空き領域が分散して大きな連続領域を確保できない**外部断片化**
* parallel samplingやbeam searchで共通するKVキャッシュの**重複保存**

著者らのプロファイリングでは、従来方式で実際のトークン状態に使われていたKVキャッシュ領域は、全体の20.4～38.2%にとどまった。

## 3. 提案手法

### PagedAttention

PagedAttentionは、各系列のKVキャッシュを、一定数のトークンを格納する固定長の**KVブロック**に分割する。論理的には連続した系列でも、対応する物理ブロックはGPUメモリ内で連続している必要がない。

| OSの仮想記憶       | vLLM            |
| ------------- | --------------- |
| プロセス          | リクエスト／系列        |
| 仮想ページ         | 論理KVブロック        |
| 物理ページ         | 物理KVブロック        |
| ページテーブル       | ブロックテーブル        |
| Copy-on-Write | 共有KVブロックの分岐時コピー |

Attentionカーネルはブロックテーブルを参照し、分散しているKey・Valueブロックを順番に取得してAttentionを計算する。計算式自体は通常のAttentionと等価であり、近似や量子化によって精度を犠牲にする方式ではない。([arXiv][2])

### オンデマンド割り当て

vLLMは最大出力長分を最初から予約せず、生成が進んで既存ブロックが満杯になった時点で、新しい物理ブロックを割り当てる。論理ブロックと物理ブロックの対応はブロックテーブルで管理される。

このため外部断片化は基本的に解消され、内部断片化も各系列の**最後の未充填ブロック1個分以内**に抑えられる。リクエストが完了すれば、そのブロックは直ちに他のリクエストへ再利用できる。([arXiv][2])

### KVキャッシュの共有

parallel samplingでは、複数の出力系列が同じプロンプトを共有する。vLLMは共有部分を同じ物理ブロックへマッピングし、ブロックごとに参照カウントを保持する。系列が分岐して共有ブロックへ書き込む必要が生じた場合だけ、**Copy-on-Write**で1ブロックを複製する。([arXiv][2])

beam searchでは候補系列の祖先部分が動的に共有される。vLLMは候補の追加・削除に応じて参照カウントを更新するため、系列全体を頻繁にコピーせず、共有できない末尾ブロックだけを複製すればよい。事前定義された共通システムプロンプトについても、複数リクエスト間で物理KVブロックを共有できる。([arXiv][2])

### スケジューリングと実装

vLLMは中央スケジューラ、KVキャッシュマネージャ、複数のGPUワーカから構成される。基本方針はFCFSで、メモリ不足時には新しいリクエスト側を優先的にプリエンプトする。退避方法として、KVブロックをCPUメモリへ移す**swap**と、復帰時にKVキャッシュを作り直す**recomputation**をサポートする。([arXiv][2])

GPU側では、ブロック書き込み、ブロック読み出しとAttention、Copy-on-Write時の複数ブロックコピーをそれぞれ融合したCUDAカーネルを実装している。また、Megatron-LM型のtensor parallelismにも対応し、各GPUワーカが共通のブロック対応表を使用する。([arXiv][2])

## 4. 実験方法

評価には以下の構成が用いられた。

* OPT-13B：A100 40GB × 1
* OPT-66B：A100 × 4、合計160GB
* OPT-175B：A100 80GB × 8、合計640GB
* LLaMA-13B：共通prefixを利用する翻訳実験

ワークロードはShareGPTとAlpacaの入出力長に基づいて生成された。平均長はShareGPTが入力161.31、出力337.99トークン、Alpacaが入力19.31、出力58.45トークンである。元データに到着時刻がないため、リクエスト到着はPoisson分布を使って合成している。([arXiv][2])

比較対象はFasterTransformerとOrcaである。ただしOrcaの実装は公開されていなかったため、著者らによる再実装が使用された。Orcaについては、実際の出力長を事前に知る非現実的な上限性能の「Oracle」、2の累乗単位で確保する「Pow2」、常に2048トークン分を確保する「Max」の3条件を評価した。([arXiv][2])

主要指標は、各リクエストのエンドツーエンド遅延を出力トークン数で割った**平均正規化レイテンシ**と、低レイテンシを維持できる最大リクエストレートである。基本sampling、parallel sampling、beam search、共有prefix、チャットボットに加え、ブロックサイズやswap／recomputationのアブレーションが行われた。([arXiv][2])

## 5. 結果

**メモリ利用率**では、vLLMはKVキャッシュ領域の96.3%を実際のトークン状態に使用した。従来方式の20.4～38.2%と比較して、予約領域と断片化が大幅に削減されている。

**基本sampling**では、ShareGPT上でOrca Oracleより1.7～2.7倍、Orca Maxより2.7～8倍高いリクエストレートを同程度の遅延で処理した。FasterTransformerに対しては最大22倍となったが、この差にはKV管理だけでなく、FasterTransformer側がiteration-level schedulingを備えていないことも影響している。([arXiv][2])

**parallel samplingとbeam search**では、共有可能な系列数が増えるほど効果が大きくなった。OPT-13B／Alpacaにおいて、Orca Oracleに対する改善は基本samplingの1.3倍から、beam width 6では2.3倍へ拡大した。KVブロック共有によるメモリ削減率は、Alpacaでparallel samplingが6.1～9.8%、beam searchが37.6～55.2%、ShareGPTではそれぞれ16.2～30.5%と44.3～66.3%であった。([arXiv][2])

**共有prefix**では、80トークンの1-shot prefixでOrca Oracleの1.67倍、341トークンの5-shot prefixで3.58倍のスループットを達成した。チャットボットワークロードでも、Orca系ベースラインの約2倍のリクエストレートを維持した。([arXiv][2])

一方、PagedAttention単体のAttentionカーネルは、ブロックテーブル参照などのためFasterTransformerより20～26%遅い。ブロックサイズには、GPU並列性と断片化のトレードオフがあり、著者らは16トークンを標準値に設定した。短い系列と十分なGPUメモリにより処理がcompute-boundになる条件では、vLLMの優位性も縮小した。([arXiv][2])

## 6. 考察

本研究の最も重要な点は、Attention演算そのものを高速化するだけでなく、**より多くのリクエストをGPUメモリへ収容することで、バッチサイズとシステム全体のスループットを引き上げた**ことである。個々のAttentionカーネルにはオーバーヘッドがあるにもかかわらず、エンドツーエンドでは大幅に高速化されており、局所的なカーネル性能とシステム性能が必ずしも一致しないことを示している。([arXiv][2])

また、論理ブロックと物理ブロックを分離したことで、基本sampling、beam search、parallel sampling、共有prefixといった異なる処理を、同じ抽象化で扱える。単なるメモリアロケータではなく、Attentionカーネル、スケジューラ、共有機構、プリエンプションを一体設計した点が強い。

一方、評価上の主な制約は次の通りである。

* Orcaが著者らによる再実装であり、実システムとの完全な比較ではない。
* 到着時刻がPoisson分布による合成で、実サービスのバーストや日内変動を反映していない。
* 主指標が平均正規化レイテンシであり、TTFT、TPOT、p95・p99遅延などのSLO指標は主要評価に含まれていない。
* 評価は主にA100、OPT／LLaMA、最大2048トークンの環境であり、さらに長いコンテキストや異なるAttention構造への一般化は本論文だけでは確認できない。
* 共通prefixの共有は、サービス提供者が事前登録したprefixを中心に扱っており、任意のリクエスト間で自動的に共通部分を探索する方式ではない。
* 「ほぼゼロの無駄」であって完全なゼロではなく、末尾ブロックの未使用領域、ブロックテーブル、参照カウントなどの管理コストは残る。([arXiv][2])

## 7. まとめ

PagedAttentionは、LLMのKVキャッシュを固定長ブロックとして管理し、論理的に連続した系列を非連続な物理メモリへ配置可能にする。vLLMはこの機構を利用して、オンデマンド割り当て、Copy-on-Write、prefix共有、beam search、プリエンプション、分散実行を統合した。結果として、KVキャッシュの断片化と重複を削減し、同時バッチ数を増やすことで、従来方式に対して概ね2～4倍のスループット改善を実現した。([arXiv][2])

**独自査読判定：Strong Accept**

新規性、システム設計の完成度、実運用上の重要性はいずれも非常に高い。特に、OSのページングとCopy-on-WriteをLLM固有のKVキャッシュへ適用し、Attentionカーネルからスケジューラまで一貫して設計した点は明確な技術的貢献である。実験もモデル規模、データセット、decoding方式を広くカバーしており、主張を十分に裏付けている。

Orcaの再実装、合成到着過程、tail latency評価の不足は査読上の懸念となるが、中心的な結論を覆すほどではない。**査読評価はStrong Accept、確信度は4/5**とする。
