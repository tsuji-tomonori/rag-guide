# P107: Anserini Gets Dense Retrieval: Integration of Lucene’s HNSW Indexes

**著者:** Xueguang Ma, Tommaso Teofili, Jimmy Lin
**掲載:** CIKM 2023, pp. 5366–5370。([ACM Digital Library][1])

## 1. はじめに

本論文は、情報検索ツールキット **Anserini** にLuceneのHNSWベクトル索引を統合し、従来の疎検索と密検索を単一のLuceneベースのソフトウェアスタックで扱えるようにする研究である。

従来のPyseriniでは、BM25などの疎検索にはLucene、密検索にはFaissを利用していた。そのため、疎検索と密検索を組み合わせるハイブリッド検索では、二つの異なる索引、実行環境、更新処理を管理する必要があった。本研究はLucene 9で導入されたHNSW索引をAnseriniから利用可能にし、この複雑性を軽減することを目的としている。さらに、LuceneとFaissについて、検索精度、索引構築時間、検索スループット、索引サイズを比較している。([arXiv][2])

## 2. 背景

従来型の情報検索では、BM25に代表される語彙ベースの検索モデルと転置索引が長く利用されてきた。一方、近年の密検索では、Bi-encoderによってクエリと文書を密ベクトルに変換し、ベクトル空間上の近傍探索として検索を実行する。

密検索は意味的な類似性を捉えやすい反面、全ベクトルとの厳密比較は高コストである。この問題に対し、HNSWは多層グラフを用いて近傍候補を絞り込むことで、高い再現率を保ちながら高速な近似最近傍探索を実現する。

ただし、密検索だけで疎検索を完全に代替できるわけではない。密検索はドメイン外データへの一般化に弱い場合があり、BM25やSPLADEなどの疎検索とのスコア融合が、単独方式より高性能になることが多い。そのため、疎検索と密検索を同じ基盤で扱えることには、研究上・運用上の価値がある。([arXiv][2])

## 3. 提案手法

### 3.1 AnseriniへのHNSW統合

著者らは、Anseriniに次の二つのクラスを追加した。

* `IndexHnswDenseVectors`: 密ベクトルからLucene HNSW索引を構築する
* `SearchHnswDenseVectors`: クエリベクトルを用いてHNSW索引を検索する

索引構築では、以下のパラメータを指定できる。

| パラメータ            | 役割                                              |
| ---------------- | ----------------------------------------------- |
| `M`              | 各ノードが持つ双方向リンク数。増やすと精度が上がる傾向があるが、索引サイズと構築時間も増加する |
| `efConstruction` | 新規ノード挿入時に探索する候補数。増やすとグラフ品質が向上するが、構築時間が増える       |
| `threads`        | 索引構築に使用するスレッド数                                  |
| `optimize`       | Luceneの複数セグメントを単一セグメントへマージするかどうか                |

検索時には、探索候補数を表す `efSearch` を指定する。値を大きくすると検索精度は向上する一方、QPSは低下する。検索の並列化は、複数クエリを同時処理するクエリ間並列として実装されている。([arXiv][2])

### 3.2 cosDPR-distilモデル

Luceneの当時のHNSW実装では、内積類似度を使用するベクトルについてL2ノルムを1にする必要があった。そこで著者らは、コサイン類似度に適した新しい密検索モデル **cosDPR-distil** を訓練した。

訓練手順は次の通りである。

1. MS MARCOとBM25のハードネガティブを用いて初期の密検索モデルを訓練する。
2. そのモデルの検索上位200件から、正解でない文書をクエリ当たり30件抽出する。
3. BM25と密検索のハードネガティブを統合し、ELECTRA-largeベースのCross-encoder教師モデルを訓練する。
4. Co-Condenserで初期化したBi-encoder学生モデルへ知識蒸留する。

教師モデルは6エポック、学生モデルは10エポック訓練され、実験には8台のA6000 GPUが使用された。([arXiv][2])

## 4. 実験方法

### データセット

**MS MARCO Passage**

* 文書数: 約880万パッセージ
* 評価クエリ数: 6,980
* 指標: MRR@10、Recall@1k、QPS
* 主目的: Lucene HNSWとFaissの精度・効率比較

**BEIR**

* 公開利用可能な13データセットを使用
* 指標: nDCG@10
* 主目的: MS MARCOで訓練したモデルのゼロショット・ドメイン外性能の評価

### 比較方式

検索精度ではBM25、DPR、ANCE、TAS-B、Co-Condenserなどと比較している。cosDPR-distilについては、全ベクトルを比較する厳密検索のFlat索引と、近似検索のHNSW索引を評価した。

効率実験には、16 vCPU、64 GB RAM、1 TB SSDを備えたGoogle Cloudのe2-standard-16が用いられた。Luceneは9.4.2、FaissはPyserini経由で実行された。クエリと文書は事前にベクトル化されており、QPSや索引構築時間にはニューラルモデルの推論時間は含まれない。([arXiv][2])

## 5. 結果

### 5.1 MS MARCOでの検索精度

| 手法                 |    MRR@10 | Recall@1k |
| ------------------ | --------: | --------: |
| BM25               |     0.184 |     0.853 |
| ANCE               |     0.330 |     0.959 |
| TAS-B              |     0.344 |     0.977 |
| Co-Condenser       |     0.382 | **0.984** |
| cosDPR-distil Flat | **0.390** |     0.980 |
| cosDPR-distil HNSW |     0.389 |     0.974 |

cosDPR-distilの厳密検索は比較方式中で最高のMRR@10を示した。HNSW化による低下はMRR@10で0.001、Recall@1kで0.006にとどまり、近似検索でも厳密検索の性能をほぼ維持している。([arXiv][2])

### 5.2 BEIRでのゼロショット性能

13データセット平均のnDCG@10は以下の通りである。

| 手法                 | 平均nDCG@10 |
| ------------------ | --------: |
| BM25               | **0.448** |
| DPR                |     0.263 |
| ANCE               |     0.413 |
| TAS-B              |     0.416 |
| cosDPR-distil Flat |     0.417 |
| cosDPR-distil HNSW |     0.415 |

cosDPR-distilは既存の密検索モデルと同程度だったが、平均ではBM25を下回った。これは、MS MARCOで訓練された密検索モデルのドメイン外一般化が必ずしも安定しないことを示している。FlatとHNSWの差は0.002であり、BEIRでも近似探索による損失は小さい。([arXiv][2])

### 5.3 索引構築性能

`M=16`、`efConstruction=100`、16スレッドの条件では、Luceneの非最適化索引の構築時間が2.9時間だったのに対し、Faiss HNSWは0.3時間だった。索引サイズは両者とも約27 GBである。

Luceneではセグメントマージを行うと検索QPSが大幅に向上したが、追加の構築時間が必要だった。例えば1スレッド、`efConstruction=100`では、非最適化時の13.6時間・3.6 QPSに対し、最適化時は21.6時間・35.8 QPSとなった。一方で、最適化後にはMRR@10とRecall@1kが若干低下している。([arXiv][2])

### 5.4 検索性能

`efSearch=1000`の結果は次の通りである。

| 実装           | 検索スレッド | MRR@10 |       QPS |
| ------------ | -----: | -----: | --------: |
| Lucene HNSW  |      1 | 0.3842 |      33.0 |
| Lucene HNSW  |     16 | 0.3842 | **373.0** |
| Faiss HNSW   |      1 | 0.3806 |      60.5 |
| Faiss HNSW   |     16 | 0.3806 |     164.5 |
| Faiss FlatIP |      1 | 0.3896 |       0.4 |
| Faiss FlatIP |     16 | 0.3896 |      24.8 |

単一スレッドではFaiss HNSWがLuceneより約1.8倍高速だった。一方、16スレッドではLuceneが373 QPS、Faissが164.5 QPSとなり、Lucene側が約2.3倍高いスループットを示した。著者ら自身も、この差の原因について実装内部まで十分に調査できていないと認めている。([arXiv][2])

## 6. 考察

### 評価できる点

最大の貢献は、アルゴリズム上の新規性よりも、**密検索と疎検索を実用的な単一基盤に統合した点**にある。Anseriniは再現可能な情報検索研究で広く利用されるツールキットであり、LuceneだけでBM25、学習疎表現、密ベクトル検索を扱えることは、研究コードと運用システムの双方を簡素化する。

また、検索精度だけでなく、索引構築時間、索引サイズ、検索QPS、スレッド数、HNSWパラメータを比較しており、運用上のトレードオフが明確である。コマンド例、ハードウェア、モデル訓練条件も比較的詳しく記載されている。

### 問題点・限界

第一に、研究の主要な動機は疎検索と密検索のハイブリッド化であるが、実験では実際のハイブリッド検索を評価していない。単一Luceneスタックにしたことで、融合処理、索引更新、運用コストがどれだけ改善するかは定量化されていない。

第二に、システム性能の比較はMS MARCO、単一マシン、単一ベクトルモデルを中心としている。複数のコーパス規模、ベクトル次元、同時実行数、レイテンシ分位点、メモリ使用量、ウォームアップ、キャッシュ状態などは評価されていない。また、LuceneはJavaから直接実行される一方、FaissはPyserini経由であるため、マルチスレッド性能差がHNSW本体によるものか、言語バインディングや並列化方式によるものか切り分けられていない。著者らも原因を未解明としている。([arXiv][2])

第三に、表間の再現性に疑問がある。本文では表1のHNSW索引を「`M=16`、`efConstruction=100`、1スレッド、最適化あり」と説明し、MRR@10を0.389、Recall@1kを0.974と報告している。しかし同じ設定に相当する表3の行では0.3842、0.9600であり、表4でも`efSearch=1000`のMRR@10は0.3842である。実験条件または表記のどちらかに説明不足がある。([arXiv][2])

第四に、Luceneへの適合のため新モデルを訓練しているが、既存の内積モデルを事後的にL2正規化するベースラインは示されていない。そのため、新規訓練がどの程度必要だったかは十分に検証されていない。

なお、本文には16スレッド化が「effectivenessを改善する」とあるが、表4ではMRRは変化せず、改善しているのはQPSである。これは「efficiency」の誤記と考えられる。([arXiv][2])

## 7. まとめ

本論文は、Lucene HNSWをAnseriniへ統合し、疎検索と密検索を同じJava・Lucene基盤で実行できるようにした。新たに訓練したcosDPR-distilとHNSW索引は、厳密検索とほぼ同等の検索精度を維持しながら、検索を大幅に高速化した。

Faissは索引構築と単一スレッド検索で優位だった一方、論文の実験条件ではLuceneがマルチスレッド検索で高いQPSを示した。ただし、この性能差の原因や実験結果の一部不整合は未解決である。研究の価値は新しい検索アルゴリズムというより、検索研究・検索システムのソフトウェア基盤を統合した実装上の貢献にある。
