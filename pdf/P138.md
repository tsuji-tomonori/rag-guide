# 1. はじめに

本論文は、Retrieval-Augmented Generation（RAG）の**評価方法**に焦点を当てたサーベイである。著者らは、検索器と生成モデルからなるRAGを単一の正解率だけで評価するのは不十分だと指摘し、評価対象・データセット・評価指標を統一的に整理する枠組み **Auepora: A Unified Evaluation Process of RAG** を提案する。([arXiv][1])

書誌情報としては、Hao Yuらによる論文で、BigData 2024の会議論文として採択され、SpringerのCCIS第2301巻、102–120ページに収録されている。([Springer][2])

---

# 2. 背景

RAGは、質問に関連する外部文書を検索し、その文書をコンテキストとしてLLMに回答を生成させる。大きく次の処理から構成される。

1. 文書の索引化
2. 質問に対する文書検索・再ランキング
3. 検索結果を含むプロンプトの構築
4. LLMによる回答生成

通常の情報検索ならPrecision、Recall、MRRなどで検索順位を評価でき、通常の文章生成ならBLEUやROUGEなどで正解文との重なりを評価できる。しかしRAGでは、検索結果が不完全でもLLMの内部知識から正解できる場合や、正しい文書を取得しても生成モデルが誤読する場合がある。そのため、検索器と生成器を個別に評価するだけでなく、両者の相互作用を評価する必要がある。([arXiv][1])

著者らは、RAG評価の主要な困難を次の三つに分けている。

**検索側の課題**は、知識ベースが巨大かつ時間とともに変化し、無関係文書・古い文書・誤情報も取得され得ることである。
**生成側の課題**は、回答の事実性、取得文書への忠実性、質問との関連性、文章品質を同時に評価しなければならないことである。
**システム全体の課題**は、検索が最終回答にどの程度寄与したかを分離して測定しにくく、さらに遅延や複雑な質問への対応能力も考慮する必要があることである。([arXiv][1])

# 3. 提案手法

## Auepora

Aueporaは、RAG評価を次の三つの問いに分解する。

* **Target：何を評価するか**
* **Dataset：どのようなデータで評価するか**
* **Metric：どの指標で定量化するか**

中心的な考え方は、RAG内部で得られる「評価可能な出力（Evaluable Outputs: EO）」と「正解・基準情報（Ground Truths: GT）」の組合せから評価対象を定義することである。([arXiv][1])

## 評価対象の分類

| 評価部分 | 比較するもの         | 評価軸       |
| ---- | -------------- | --------- |
| 検索   | 取得文書と質問        | 関連性       |
| 検索   | 取得文書と正解文書・候補集合 | 検索精度      |
| 生成   | 生成回答と質問        | 回答関連性     |
| 生成   | 生成回答と取得文書      | 忠実性・根拠整合性 |
| 生成   | 生成回答と模範回答      | 正解性       |

ここで特に重要なのが、**忠実性と正解性を区別している点**である。

* 忠実性が高い：取得文書に書かれている内容を正しく反映している
* 正解性が高い：現実世界または模範回答に照らして正しい

取得文書自体が誤っている場合、回答は文書に忠実でも事実としては誤っている。この区別により、検索失敗と生成失敗をより細かく診断できる。([arXiv][1])

## 追加的な評価要件

著者らは、検索・生成品質以外にも次の評価が必要だとしている。

* **Latency**：検索から回答生成までの処理時間
* **Diversity**：取得文書や回答内容の多様性
* **Noise Robustness**：無関係な文書が混入した場合の頑健性
* **Negative Rejection**：根拠不足の場合に回答を拒否できるか
* **Counterfactual Robustness**：誤情報・反事実情報を無視できるか
* 可読性、毒性、網羅性などのユーザー指向評価

これにより、単なるQA正解率ではなく、実運用時の信頼性まで評価範囲に含めている。([arXiv][1])

# 4. 実験方法

本論文はサーベイ論文であり、独自のRAGモデルを訓練して性能比較する実験は行っていない。代わりに、既存の評価ツール、ベンチマーク、関連研究をAueporaの三軸で分類している。

改訂版の表1には、TruEra RAG Triad、LangChain Benchmark、Databricks Evaluationなどのツール、RAGAs、ARES、RGB、MultiHop-RAG、CRUD-RAG、MedRAG、DomainRAG、ReEvalなどのベンチマーク、および遅延・多様性に関する研究が整理されている。([arXiv][1])

## データセットの分類

調査対象のデータセットは、主として二種類に分けられる。

**既存データセットの再利用**では、Natural Questions、HotpotQA、FEVER、MultiRC、ReCoRDなどが用いられる。これらは比較可能性や再現性に優れる一方、モデルの事前学習データに含まれている可能性があり、最新情報を検索する能力を評価しにくい。

**新規・動的データセットの生成**では、ニュース記事や大学の入試情報など、時間とともに更新される情報から質問と正解を作る。RGB、MultiHop-RAG、CRUD-RAG、CDQA、DomainRAGなどがこの方針を採用している。([arXiv][1])

## 指標の分類

検索については、Accuracy、Precision、Recall@kなどの非順位指標と、MRR、MAP、Hit@kなどの順位指標を整理している。

生成については、BLEU、ROUGE、F1などの参照回答ベースの指標、人手評価、LLM-as-a-Judgeを比較している。さらに、ノイズ耐性にはMisleading Rate、誤情報耐性にはError Detection Rate、回答拒否にはRejection Rateなど、RAG固有の指標も扱っている。([arXiv][1])

# 5. 結果

本論文から得られる主要な調査結果は、次のように整理できる。

第一に、既存ベンチマークの評価対象は統一されていない。生成回答の評価は広く実施されている一方、検索器の品質を明示的に測定しないベンチマークも多い。したがって、最終回答の改善が検索器によるものか、LLM自体の能力によるものかを判定できない場合がある。([arXiv][1])

第二に、単純なPrecisionやRecallだけでは、RAGに必要なノイズ耐性、反事実耐性、回答拒否、多様性を十分に評価できない。検索順位が良好でも、取得文書中の誤情報を生成器が採用すれば、システム全体としては危険だからである。

第三に、静的なQAデータセットだけでは「外部情報を実際に検索して利用したか」を確認しにくい。最新ニュースや更新頻度の高い情報を用いた動的データセットは、LLMの記憶だけによる回答を防ぐうえで有効だが、生成・検証・保守のコストが高い。([arXiv][1])

第四に、LLM-as-a-Judgeは、関連性、忠実性、流暢性、網羅性などを柔軟に評価できるため有望である。しかし、使用するモデル、プロンプト、採点基準、例示方法によって結果が変化し、人間評価との整合性や評価の再現性が未解決である。([arXiv][1])

# 6. 考察

本論文の最大の意義は、RAG評価を「指標の一覧」としてではなく、**どの出力を、どの基準情報と比較するか**という関係として整理した点にある。特に、回答と質問、回答と取得文書、回答と正解回答を区別することで、関連性・忠実性・正解性の混同を減らしている。

また、RAGの品質を単一スコアに集約せず、検索品質、生成品質、頑健性、遅延、拒否能力などの多次元問題として捉えた点は、実用システムの評価設計に有用である。

一方、査読上は以下が弱点となる。

**文献調査手順の再現性が低い。** 検索データベース、検索式、採択・除外基準、重複処理、品質評価といったシステマティックレビューの手続きが示されていない。そのため、「包括的」という主張を第三者が再現しにくい。

**Auepora自体の妥当性検証がない。** 提案分類が人間評価との相関を改善するか、評価者間一致度を高めるか、システム障害の診断に有効かといった実証分析は行われていない。

**検索指標と最終回答品質の因果関係が未整理である。** 著者自身も、検索中間出力の変化と最終生成結果の関係を分析する研究が必要だと認めている。([arXiv][1])

**調査件数の記述に軽微な不整合がある。** 序論では12の評価フレームワークを収集したと記述されている一方、改訂版表1にはツール、ベンチマーク、関連研究を合わせて16件が掲載されている。改訂時に本文の件数表記が更新されなかった可能性がある。([arXiv][1])

# 7. まとめ

本論文は、RAG評価を以下の三段階で設計すべきだと主張する。

1. **評価対象を決める**
   検索関連性、検索精度、回答関連性、忠実性、正解性、頑健性などを選ぶ。

2. **目的に合ったデータセットを構築する**
   静的QAだけでなく、最新情報、マルチホップ、複数文書、ノイズ、反事実、根拠不足の事例を含める。

3. **評価対象に対応した指標を選ぶ**
   検索順位指標、参照回答ベース指標、人手評価、LLM-as-a-Judge、運用指標を組み合わせる。

結論として、RAGを一つの正解率で評価することはできず、検索器・生成器・両者の相互作用・実運用要件を分解して測定する必要がある。([arXiv][1])

# 8. 総評

## 査読結果

**判定：Weak Accept／弱採択**

| 観点     |      評価 |
| ------ | ------: |
| 新規性    | 3.5 / 5 |
| 重要性    | 4.0 / 5 |
| 技術的厳密性 | 2.5 / 5 |
| 明瞭性    | 3.0 / 5 |
| 再現性    | 2.0 / 5 |
| 総合     |  6 / 10 |

RAG評価が急速に断片化していた時期に、評価対象・データセット・指標を一つの枠組みに整理した意義は大きい。特に、関連性・忠実性・正解性を比較対象の違いから定義する考え方は、評価パイプラインを設計する実務者にとって有用である。

一方で、サーベイの収集方法が体系化されておらず、提案分類の実証的な妥当性検証もないため、強い採択とするには技術的根拠が不足している。採択する場合は、検索・除外手順の明示、分類基準の形式化、既存評価指標の人間評価との相関比較、表と本文の件数整合性の修正を求めたい。

実際の掲載結果としては、本論文は**BigData 2024に採択済み**であり、Springerの会議録として刊行されている。([Springer][2])
