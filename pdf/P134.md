# P134：Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use

著者はFrancisco Javier ArceoとVarsha Prasad Narsing。ACM Conference on AI and Agentic Systems 2026（ACM CAIS ’26）に採択・収録された11ページの論文である。([arXiv][1])

## 1. はじめに

本論文は、企業向けRAGおよびAIエージェントを複数の組織・顧客で共有する際のデータ分離問題を扱う。

中心的な主張は、通常の検索システムが最適化しているのは「問い合わせとの関連度」であり、「その利用者が文書を閲覧してよいか」という認可ではない、という点である。そのため、意味的に類似しているという理由だけで、別テナントの機密文書が検索結果やLLMのコンテキストに混入し得る。著者らはこれを**relevance–authorization gap（関連度と認可のギャップ）**として定式化する。([arXiv][1])

この問題に対し、文書取り込み、検索、推論、ツール実行、会話状態管理をサーバ側の信頼境界内で統制する多層アーキテクチャを提案し、オープンソース基盤OGX上に実装している。([arXiv][1])

## 2. 背景

企業環境では、単一利用者を想定した一般的なLLM APIとは異なり、複数テナント、異種データ、細粒度のアクセス制御、監査、規制対応、オンプレミス運用、ベンダーロックイン回避が必要になる。一方で、テナントごとにベクトルDB、LLM、ツール実行環境を完全複製すると、インフラ費用と運用負荷がテナント数に比例して増加する。([arXiv][1])

著者らは、既存のエージェント構成に次の4つの問題があると整理する。

1. **関連度と認可の分離**：ベクトル検索やBM25は認可を考慮しない。
2. **ツール経由の情報漏えい**：エージェントの資格情報でツールを呼ぶと、エンドユーザーの権限を超える可能性がある。
3. **会話コンテキストの蓄積**：過去ターンの検索結果やツール出力が、権限変更後も残存し得る。
4. **クライアント側制御の迂回**：認可処理をクライアントに任せると、改変されたクライアントが認可付きエンドポイントを回避できる。([arXiv][1])

脅威モデルには、他テナントのデータを狙う悪意ある利用者、認可処理を迂回する改変クライアント、テナント間状態を漏らす不具合ツールが含まれる。一方、サーバプロセス自体の侵害、インフラ管理者、タイミングなどのサイドチャネル、モデル抽出、学習済みモデルのパラメトリック記憶からの漏えいは対象外である。([arXiv][1])

## 3. 提案手法

提案は、**データパスの3層分離**と、**制御パスのサーバ側オーケストレーション**から構成される。

**第1層：Policy-Aware Ingestion**

文書登録時に、所有テナント、ロール、チーム、プロジェクト、名前空間などの認可属性を付加する。文書分割後の各チャンクにも同じ属性を継承させ、所有者情報が後付けや推測に依存しないようにする。

**第2層：Retrieval Gating**

検索前にベクトルストアなどのリソース自体へのABAC認可を行い、検索時または検索後にチャンク単位の属性フィルタを適用する。検索バックエンドがpredicate pushdownをサポートする場合、認可条件をベクトル検索内部へ押し込み、許可された領域だけを対象にランキングする。([arXiv][1])

**第3層：Shared Inference**

LLMはテナント間で共有するが、LLMへ渡すコンテキストは第1・第2層で認可済みの情報だけに限定する。モデル自体をテナントごとに複製するのではなく、コンテキスト構築を分離することでコストを抑える。ただし、モデルが事前学習で記憶した情報までは保護対象にならない。([arXiv][1])

**サーバ側オーケストレーション**

推論、検索、ツール実行、入出力安全検査、会話状態保存をサーバ側で実行する。これにより、クライアントは「何を依頼するか」は決められるが、「どの認可処理を通すか」「どの資格情報でツールを実行するか」は変更できない。

実装にはOGXを使用する。OGXはOpenAI互換のResponses API、ベクトルストア、会話、ファイル、プロンプト、ツールなどをテナント単位のAPIリソースとして扱う。認可はAPIミドルウェア、プロバイダ解決、ストレージ読み出しの3段階で適用される。推論・検索プロバイダは差し替え可能で、Kubernetes上の共有、テナント別、ハイブリッド構成をサポートする。([arXiv][1])

## 4. 実験方法

実験は、オーケストレーション方式と検索認可の有無を組み合わせた2×2構成で行われた。

| 構成 | オーケストレーション | 検索     |
| -- | ---------- | ------ |
| A  | クライアント側    | 認可なし   |
| B  | クライアント側    | ABACあり |
| C  | サーバ側       | 認可なし   |
| D  | サーバ側       | ABACあり |

ワークロードは、金融、技術、法務の3テナント、合計300文書で構成される。各構成について、正規クエリ300件、他テナントの情報を狙うクエリ300件、プロンプトインジェクション90件を実行した。インジェクションは命令上書き、ロール偽装、デバッグ機能悪用、コンテキスト操作の4種類である。([arXiv][1])

主な実験環境は、推論が`gpt-4o-mini`、埋め込みが`text-embedding-3-small`、ベクトルストアがsqlite-vecである。別途、NVIDIA T4上のvLLMとLlama-3.2-1B-Instructを使用して、ローカルGPU環境でのオーバーヘッドも測定した。([arXiv][1])

主要指標は、他テナントクエリのうち1件以上の不正チャンクを返した割合であるCTLRと、全API呼び出しに占める認可違反の割合であるAVRである。加えて、レイテンシ、QPS、Precision@5、Recall@5、MRR、ABAC判定精度を評価した。([GitHub][2])

## 5. 結果

### セキュリティ

| 構成               |   CTLR |   AVR |
| ---------------- | -----: | ----: |
| A：Client＋Ungated | 100.0% | 50.0% |
| B：Client＋Gated   |   0.0% |  0.0% |
| C：Server＋Ungated |  98.3% | 49.5% |
| D：Server＋Gated   |   0.0% |  0.0% |

認可のない構成では、ほぼすべての他テナント検索が情報漏えいを起こした。一方、ABACを適用したBとDではCTLR、AVRともに0%となった。サーバ側オーケストレーションだけでは漏えいを防げず、**検索ゲーティングが実際の情報漏えい防止機構**であることが示されている。([GitHub][2])

プロンプトインジェクション90件では、Aが80.0%、Cが62.2%の漏えい率だったのに対し、BとDは0%だった。ただし、これはLLMが攻撃命令を意味的に見破ったためではなく、検索層が不許可文書を返さなかったためである。([GitHub][2])

### 性能

認証、ABAC評価、テナント別ストア解決を含む検索ゲーティングの追加コストは約19msだった。T4環境では、OGXのルーティングが4.7ms、メタデータフィルタが5.5msの追加となった。([arXiv][1])

一方、サーバ側オーケストレーションは、非ストリーミングのResponses APIでツール実行往復が増えるため、クライアント側より約3秒遅かった。同時実行数25では、QPSはAが5.4、Bが4.2、Cが2.2、Dが2.6であり、クライアント側はサーバ側のおよそ2倍のスループットとなった。([GitHub][2])

### 検索品質とスケーリング

チャンク単位のゲーティングにより、Precision@5は0.200から0.433へ約2.2倍、MRRは0.700から1.000へ向上した。他テナント文書というノイズを除去した結果、許可された正解文書の順位も改善している。([GitHub][2])

ただしpredicate pushdownを持たないsqlite-vecで検索後フィルタだけを行う場合、Recall@5は100チャンクで1.000だったものが、50,000チャンクでは0.002まで低下した。漏えいは防止できるものの、上位検索結果が他テナント文書で埋まり、フィルタ後に有用な文書が残らなくなるためである。([GitHub][2])

## 6. 考察

本論文の重要な点は、プロンプトによる「LLMへのお願い」ではなく、検索・認可境界で強制的にデータを遮断したことである。LLMを非信頼コンポーネントとみなし、そもそも不許可データをコンテキストへ入れない設計は、企業システムのセキュリティ原則として妥当である。

また、検索認可だけでなく、会話状態、ツール資格情報、監査ログをサーバ側のcontrol planeへ集約する方針も実運用上有用である。クライアント側ゲーティングでも実験上は漏えい率0%だが、悪意あるクライアントが認可なし検索を直接呼べるなら保証にならないため、サーバ側強制には意味がある。([arXiv][1])

一方、以下の限界がある。

第一に、実験の中心は検索であり、論文タイトルに含まれるツール利用、複数ターン状態、資格情報伝播についての攻撃実験は限定的である。ツールが不正な出力を返す場合や、過去の会話に不許可情報が残る場合の実証が必要である。

第二に、「プロンプトインジェクション耐性」は一般的なインジェクション防御を意味しない。示されているのは、インジェクション文があっても検索認可を越えられないことであり、許可された文書内の間接プロンプトインジェクション、危険なツール操作、データ改変までは評価していない。([GitHub][2])

第三に、実データ評価はなく、主要実験は3テナント、300文書の合成環境である。また、ベンダーニュートラル性はアーキテクチャ上は示されているが、複数のベクトルDB、認証基盤、推論プロバイダを横断した同一実験までは行われていない。([arXiv][1])

第四に、「オーバーヘッドは無視できる」という主張は検索ゲーティングには当てはまるが、サーバ側オーケストレーション全体には当てはまりにくい。約3秒の遅延と高負荷時の約半分のQPSは、対話システムでは無視できないトレードオフである。

## 7. まとめ

本論文は、企業向けマルチテナントRAGの本質的な問題を「関連度と認可は別物である」と明確化し、次の構成を提示した。

**取り込み時の所有属性付与 → 検索時のABACゲーティング → 認可済みコンテキストだけを共有LLMへ投入 → ツールと状態をサーバ側で統制**

実験では、認可なし構成で98～100%発生した他テナント漏えいを0%に抑え、検索認可自体の追加コストを約19msに維持した。一方、サーバ側オーケストレーションには遅延・スループットの代償があり、predicate pushdown非対応DBでは検索再現率が大きく低下する。([arXiv][1])

## 8. 総評

**公開されている査読結果：ACM CAIS 2026に採択・正式刊行済み。** 会議の採択論文一覧とACM Proceedingsの双方で確認できる。([CAIS Conference][3])

**独立査読判定：Weak Accept（弱い採択）**

問題設定の重要性、信頼境界の明確さ、再現可能なオープンソース実装、検索認可に関する一貫した実験結果は高く評価できる。特に「LLMに漏えいさせない」のではなく「LLMに不許可データを渡さない」という方針は、企業システムとして正しい。

ただし、技術的新規性はABAC、メタデータフィルタ、サーバ側オーケストレーションという既知技術の統合が中心である。著者ら自身も、新規性が個々の技術ではなく、マルチテナント・エージェント基盤への組み合わせにあると位置付けている。([arXiv][1])

また、ツール経由漏えい、長期会話状態、実際の企業データ、複数ベンダー構成についての検証が不足している。したがって、システム／アーキテクチャ論文としては採択に値するが、包括的なエージェントセキュリティ保証を示した論文として評価するには追加実験が必要である。
